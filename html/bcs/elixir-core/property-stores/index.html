B2.2 · Property Stores on ETS — manuscript chapter 2.2

The substrate becomes the database.

Three system tables under one tree — AST instruments, PRT balances, ORD orders — with the branded id as the only key, chronology as a property of the keyspace rather than a column, and exactly one amendment to the store module: window/3.

The chapter is content/bcs2.2.md; the rung behind it is bcs2.2 (bcs_rung_2_2_check.exs), its committed transcript closes PASS 5/5 — and the 2.1 rung re-runs green under the grown surface. Every figure on this module's pages is verbatim from those outputs.

§1

The desk's three questions

content/bcs2.2.md · Why
LATEST

A newest-first page

What are the latest orders is a page off the table tail — page_desc/2, a prev walk from the table's end, no timestamp column consulted.

BETWEEN

A window

What happened between 14:30 and 14:32 is a half-open window — two synthetic cursors and one gated select, window/3.

KNOWN NAME

A get by name

What is this instrument is get/2 — the branded id is the only key, so the name is the lookup.

A database that needs a timestamp column, an index on it, and a query planner to answer the first two has paid three times for what the keyspace already carries. The part preface's guideline — a table keyed by branded ids is a timeline, add no second clock — is cashed here as working reads on real stores.
