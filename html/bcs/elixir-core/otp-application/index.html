B2.1 · A System Is an OTP Application — manuscript chapter 2.1

The noun, given a runtime shape.

“System” was Part I's load-bearing noun, and OTP is where the noun acquires a runtime shape: a supervised process owning a table behind an export list. Chapter 1.1 built the skeleton and promised the full treatment; this chapter delivers it without changing a line of the skeleton — which is the claim.

The chapter is content/bcs2.1.md; the rung behind it is bcs2.1, and its committed transcript closes PASS 5/5. Boundary, tree, ownership, restart semantics — each a decision made explicit and gated on stage.

§1

Decisions, not accidents

content/bcs2.1.md · Why

The stakes are not abstract on a trading platform — when the position store crashes at 14:30 with the book open, what restarts, what survives, and what cascades is a design decision with money attached, and a design decision that was never written down is still a design, only an accidental one. This chapter writes the defaults down and proves each on stage, so that every Part II system after it inherits decisions rather than accidents.

OWNERSHIP

Data dies with the owner, by design

A private ETS table dies with its owning process; durability is a different chapter's job — the written default, gated by R2.

CHECKPOINTS

Checkpoints are rows

State that must survive a process lives in a store; recovery runs through the boundary, not the heap — gated by R3.

RADIUS

one_for_one is the default

Wider is a written claim that the named children die together — the strategy as a blast-radius statement, gated by R4.

BOUNDARY

The export list is the boundary contract

Six domain functions plus OTP callbacks, nothing else; adding an export is an architecture change — gated by R1 and R5.
