B1.3 · dive 3 of 4 — the candidate table

Every column filled.

Against the BCS requirements — discriminant in the value and the type, lexicographic mint order, in-contract placement, fixed printable width, cross-runtime canon — seven candidates enter the table. One row has every column filled.

The storage column is the measured table's; the remaining columns are the contract's properties. Nothing in this table is taste.

§1

The requirements

content/bcs1.3.md · The chooser

Five columns. Valkey B/key — the measured table's verdict at rest. Time-ordered — string order as mint order, the order theorem's demand. Type tag — the namespace discriminant, carried in the value. Placement fn — in-contract placement arithmetic any holder of the id may run. Coordination — what the scheme asks of the fleet to mint safely.

column · Select a requirement; the readout reads that column down all seven candidates and names where it goes empty.

§3

The strongest outsider, and the verdict

content/bcs1.3.md · The chooser

UUIDv7 is the strongest outsider — ordered without coordination — but pays two classes as text, surrenders readability as binary, and carries no namespace and no placement contract in either form. The branded snowflake is the only row with every column filled, and the table removed the last argument against it: the contract's wire form is also the cheapest printable one.

The verdict stands as the contract states it, and re-validation when the engine bumps re-checks the directional invariants on a dated new output — it does not reopen the chooser.
