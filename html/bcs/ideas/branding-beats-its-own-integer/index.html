B1.6 · Branding Beats Its Own Integer — manuscript appendix 1.1

Branding beats its own integer.

Chapter 1.3 measured a sentence worth promoting to its own appendix: the same snowflake rendered as decimal text (19 digits) costs 73 bytes per key on Valkey’s new table, eight more than its 14-byte branded base62 form at 65. At the storage layer the brand is a discount, not a tax. This module asks the follow-up the finding deserves: does the CPU agree? The encode path — u64 in, text out — is measured in all five runtimes of the canon, against the real implementations, with the decimal rendering as the opponent in each.

The module teaches content/bcs1.a1.md. The answer has texture: the compiled runtimes agree emphatically, the BEAM records a tie, and V8 records the one loss — in exactly the place the contract already prescribes a native crossing.

§1

The question the storage record earned

content/bcs1.a1.md · scope and method

The storage verdict is already committed: 65 bytes per key for the 14-byte branded form against 73 for its own 19-digit decimal rendering, on Valkey 8.1’s single-allocation entries — five fewer bytes pushed the embedded object down one allocator size class. A discount at rest says nothing about the clock. The appendix puts the encode path itself on the bench, runtime by runtime, and inherits the storage rows verbatim rather than re-measuring them.

Method, as recorded in the committed outputs: each runtime encodes snowflakes with the low 20 bits varied (to defeat constant folding), accumulates a byte of each output into a sink (to defeat dead-code elimination), and reports the best of three runs — five on Node. Toolchains as recorded under bench/branding-vs-decimal/: cc 13.3.0 at -O2, rustc 1.75.0, go 1.22.2, Elixir with the native codec loaded, and Node v22.22.2 running the TypeScript implementation directly.
