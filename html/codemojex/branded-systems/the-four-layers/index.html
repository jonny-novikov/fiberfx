The four layers.

Codemojex is four layers, and the layering is the architecture. A request enters at a thin Phoenix surface; durable work travels on the EchoMQ bus to a small set of authorities; and state rests in three storage tiers, each chosen by what its data needs. Each layer owns its tier, and the branded id is the one value that threads all of them — money, the secret, and the commitment rest on the durable floor, while everything in Valkey is derived and reconstructable.

Pick a layer below. The band it names lights up, and the readout says what that layer owns.

§1

Four layers, four owners

design §Fault tolerance

The surface is deliberately small; the facade delegates and never holds logic; the bus carries durable work to the authorities; and the storage tiers hold the truth. The order is the dependency order — the system of record first, then the fan-out and the wire, then the near-cache, then the consumers, and the HTTP surface last, so a live channel is the last thing to start and the last thing to fall.

The as-built Codemojex.Tables also starts a third table, :cm_sessions (cm.4) — the design's diagram predates the auth floor, so the code is the authority for the roster (see §3). Each consumer leases the job it is working, so a crashed consumer's in-flight job becomes visible again rather than lost, and its supervisor restarts it in place.

§2

Durable and derived

design §Storage tiers

The storage split is exact. Some state is on the Postgres floor and cannot be regenerated; the rest lives in Valkey and is rebuildable from the floor. Toggle the two halves — the cells that belong to each light up.

§3

The near-cache tier

Codemojex.Tables

Codemojex.Tables declares three L1-over-L2 caches, each keyed by a branded id: :cm_games (kind "GAM", coherence :none), :cm_emojisets (kind "EMS", :none), and :cm_sessions (kind "SES", cm.4 — the first mutable table, coherence :tracking, so a revoked session is evicted from every holder's L1 at once). A read is a caller-side :ets lookup; a miss coalesces onto one in-flight fill that checks L2 and falls through to a loader that reads Postgres, writing both layers under a TTL. The loader returns {:ok, term_to_binary(map), id} — the entity's own 14-byte id as the L2 version, so the frame carries the name the value belongs to. The two game caches are immutable for a game's life, so their coherence is :none and the cache never goes stale; the directory is supervised first under :rest_for_one, so the roster cannot silently empty out from under the readers.
