The branded-id gate

A job's id is a 14-byte branded Snowflake, and the wire checks it before a row exists.

A caller does not pick a job's id. It is a 14-byte branded id: a three-letter uppercase namespace, then eleven characters of Base62 Snowflake. For a job the namespace is JOB, and the @enqueue script checks it first — it reads the first three bytes and refuses anything that is not JOB, with the EMQKIND error class, before EXISTS, before HSET. Identity is enforced on the server, in Lua, as part of the same atomic step that writes the row. Decode a real id on the right, then read the gate that admits it.

Identity is not a number the caller picks

A job's id is a 14-byte branded id: a three-character uppercase namespace concatenated with an eleven-character Base62 encoding of a 63-bit Snowflake. The layout is fixed — three plus eleven, always fourteen — and the Snowflake itself packs a 41-bit timestamp, a 10-bit node, and a 12-bit sequence (ts << 22 | node << 12 | seq) against a fixed epoch (1704067200000). For a job, the namespace is JOB.

The name carries the weight a type used to — four properties, and the gate reads the first of them. It is typed: the namespace declares the domain class on the wire, so a USR id cannot enter the job keyspace. It is ordered: the text sorts as its mint instant sorts, so a table keyed by it is a timeline. It is placed: one function, EchoData.BrandedId.hash32/1, locates a row from the identity alone — placement("USR0KHTOWnGLuC") is 234878118, the same for every holder. And it is conformant: one source of truth makes encode and decode identical across the runtimes, so which language speaks the wire is a deployment detail.

The decoder above takes a real id apart: the namespace, the Snowflake, and the unpacked timestamp, node, and sequence — and the gate verdict. A JOB id is admitted; a USR id is the right shape but the wrong namespace, so the gate refuses it; a bare "job-42" is not a branded id at all. The branding is not a label — it is what the @enqueue script tests before the id may enter the job keyspace.

These are not aspirations: the runtime asserts them at boot. self_check!/0 in echo_data's branded_id.ex compares the native and pure code paths against the contract's vectors and refuses to start if they disagree — so the name is trustworthy because the system proves it before trusting it. The three lines below are those vectors, verbatim — source truths, not measurements.

The gate, in two beats

Beat one — the named handle. EchoMQ.Jobs.enqueue/4 builds the declared keys and runs the @enqueue script through the connector, then maps the server's verdict back to a typed result — including the EMQKIND refusal the gate produces, which becomes {:error, :kind}.

# echo_mq — EchoMQ.Jobs
# The host builds the declared keys and runs @enqueue. The server's verdict
# is mapped: 1 enqueued, 0 a duplicate, and an EMQKIND error reply (the
# branded-id gate firing) becomes the typed {:error, :kind}.
def enqueue(conn, queue, job_id, payload) when is_binary(job_id) and is_binary(payload) do
  keys = [Keyspace.job_key(queue, job_id), Keyspace.queue_key(queue, "pending")]

  case Connector.eval(conn, @enqueue, keys, [job_id, payload]) do
    {:ok, 1} -> {:ok, :enqueued}
    {:ok, 0} -> {:ok, :duplicate}
    {:error, {:server, "EMQKIND" <> _}} -> {:error, :kind}
    other -> other
  end
end

Beat two — the gate, in the script body. The first statement of @enqueue is the namespace check. ARGV[1] is the 14-byte id; string.sub(ARGV[1], 1, 3) is its first three bytes. Not 'JOB' → redis.error_reply('EMQKIND …'), and the script stops before the idempotency check, before the row write, before the pending insert.

-- the @enqueue script — the branded-id gate is the first statement
-- ARGV[1] is the 14-byte branded id. Its first three bytes are the
-- namespace. A job id must be JOB-namespaced; anything else is refused at
-- the wire, before a row exists.
if string.sub(ARGV[1], 1, 3) ~= 'JOB' then
  return redis.error_reply('EMQKIND job id must be JOB-namespaced')
end
-- only a JOB-namespaced id reaches the idempotency check and the row write
if redis.call('EXISTS', KEYS[1]) == 1 then
  return 0
end
redis.call('HSET', KEYS[1], 'state', 'pending', 'attempts', '0', 'payload', ARGV[2])
redis.call('ZADD', KEYS[2], 0, ARGV[1])
return 1

A second gate runs earlier, host-side: EchoMQ.Keyspace.job_key/2 calls EchoData.BrandedId.valid?/1 and raises on an ill-formed id, so a malformed id never even reaches the key builder. The Lua gate is the wire's own check — the one that holds for every speaker in any runtime, because it lives in the shared script, not in one runtime's host code. Two gates, one for shape (host) and one for namespace (wire), and the row is written only past both.

A job id is a 14-byte branded Snowflake under JOB, and the gate is the first line of @enqueue — three bytes compared, the wrong namespace refused with EMQKIND before any write.

The gate, byte by byte

Recap — the gate and the id

A job id is a 14-byte branded Snowflake under JOB: EchoData.BrandedId fixes the layout (a three-character namespace + eleven Base62 characters) and gives it four properties — typed, ordered, placed (hash32) and conformant, the contract self_check!/0 asserts at boot — and the @enqueue script's first line — string.sub(ARGV[1], 1, 3) ~= 'JOB' — reads the typed property to refuse the wrong namespace with EMQKIND before any write. Keyspace.job_key/2 gates the shape host-side as well. The next dive reads the third guard — how a whole connection is fenced before it can touch the store.
