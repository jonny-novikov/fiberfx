E2 · Movement I · The Core

The lifecycle, the
components & the runtimes

The as-built library in motion: the eight states a job moves through, the one atomic script behind each move, and the three runtimes that share them.

E1 froze the wire — the keys and the atomic Lua scripts. This chapter sets it in motion. A job is always in exactly one of eight states, and every move between them is one atomic Lua script — moveToActive-11, moveToFinished-14, moveToDelayed-8, moveStalledJobsToWait-8 — with a closed -1…-11 return vocabulary. Four modules build the picture: the lifecycle and its scripts, the components that drive it (the job model, the stateless queue, the worker fetch loop), the one-timer lock management and the supporting processes, and the three runtimes with their honest interop status. A workshop then runs a job through every state.

Why & when

A reader who walked the → EchoMQ door from the reliable-queue pattern (R3) has met states-as-locations and the atomic-move thesis. A reader who finished E1 holds the immutable wire — the keys and the scripts. This chapter answers the next question: which eight states a job occupies, which script performs each transition between them, and how the lock keeps a job owned while a worker runs it. Read it when you need the mechanism in motion — reading a real queue's state, debugging a stalled job, or building on the core (every extension in E3–E8 stands on the lifecycle this chapter lays out).

The chapter holds one rule throughout — the Golden Rule: the state's keys (L1) and the transition's Lua script (L2) are immutable and shared across runtimes; the executor (L3) and the API (L4) vary. The interactive below reads the eight real states EchoMQ.Keys builds. Pick a state to see its Redis location and the script that moves a job into it.

A job is always in exactly one state, and each is one key built by one function in EchoMQ.Keys. The state and the script that moves a job into it are the protocol — the same in Elixir, Go, and Node.js.

How it works

One transition makes the layering concrete. A worker picks up the next job: at L4 it calls a runtime API; at L3 the executor dispatches a script by SHA (EchoMQ.Scripts.move_to_active/4); at L2 the script moveToActive-11 runs atomically; at L1 it moves a job id from emq:{queue}:wait to emq:{queue}:active and sets the lock at emq:{queue}:{jobId}:lock with a UUID token and a PX TTL. The completion script moveToFinished-14 first verifies that token; a missing or mismatched lock returns -2 or -6 from the closed code set. L1 and L2 are byte-identical across runtimes; only L3 and L4 differ.

the protocol (immutable L1/L2)

The state keys (EchoMQ.Keys.wait/1, active/1, …) and the transition scripts (moveToActive-11, moveToFinished-14) — frozen, with the same closed -1…-11 codes in every runtime.

→
its three runtimes (variable L3/L4)

Elixir dispatches the moves with EchoMQ.Scripts over a Redix pool and renews locks with one EchoMQ.LockManager timer per worker; Go and Node.js drive the same scripts their own way. The wire underneath does not move.

The chapter teaches the lifecycle and the components in depth, and shows, on every transition, that the runtime above is free precisely because the protocol below is fixed.
