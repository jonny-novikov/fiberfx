The three runtimes
& interop

One wire, three runtimes — the reference, the port, and the origin — and a job round-trips between any two that share it.

EchoMQ's v1 line speaks the BullMQ wire — the emq: keyspace, meta.version = bullmq:5.65.1, the same 50 Lua scripts — and three runtimes share it: the Elixir reference (echo/apps/echomq v1.3.0), the Go port (apps/echomq-go, partial with named gaps), and Node.js, which is the BullMQ reference implementation itself — EchoMQ has no first-party Node runtime today. The Golden Rule holds across all three: L1 (the keyspace and field names) and L2 (the scripts) are byte-identical; only L3 (the executor) and L4 (the client API) vary. Three dives follow, and the chapter's place in the 2.0 fork closes the page.

How it fits

A runtime is free to drive the protocol however its host language drives best — OTP processes in Elixir, goroutines in Go, an event loop in Node — but every runtime writes the same keys and runs the same scripts. The emq: keyspace and the field names are L1; the 50 Lua scripts every transition runs are L2; both are byte-identical across runtimes. The executor above them is L3 and the client API is L4 — each runtime's own. That is why a job enqueued by one runtime can be processed by another: the wire underneath does not move. The course's honest cross-runtime status — Elixir reference, Go partial, no first-party Node — is the truth this module teaches.

The 2.0 fork — EchoMQ leaves the BullMQ wire

The shared wire above is the v1 line (frozen at 1.3.0) — the BullMQ wire. emq.1 ships EchoMQ 2.0, which leaves it: the emq: keyspace replaces emq:, every Lua key is declared in KEYS[], the per-queue {q} hashtag is applied transparently by the core, and meta.version becomes echomq:2.0.0. Interop then becomes the EchoMQ fleet's own — stock BullMQ clients cannot speak v2 by design, the Go port must implement 2.0 to stay in the fleet, and a first-party echomq-node is proposed (it does not exist) — and the payoff is Dragonfly-native multithreading.
