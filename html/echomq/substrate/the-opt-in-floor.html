The owned keyspace

EchoMQ 2.0 owns its keyspace. The emq: prefix replaces emq:, the per-queue hashtag is the core's job, and every queue is placed by construction.

EchoMQ 2.0 breaks from BullMQ on the first rung, and the most visible part of the break is the keyspace. emq.1 ships emq:{q}:<type> — the emq: prefix replacing emq:, the queue name brace-wrapped transparently by the core (a caller still writes Queue("orders")), and the braced base {emq}: reserved for the core's own cross-queue keys so the namespaces cannot collide. Every queue is slot-local by construction. The v1 line that interpolates the queue name verbatim under emq: freezes at 1.3.0. The rung is emq.1, drafted, taught from its spec.

emq.1 owns its keyspace

EchoMQ 2.0 drops BullMQ wire compatibility on the first rung, and the most visible part of that break is the keyspace. The v1 line — echo/apps/echomq v1.3.0, the line the fork freezes — speaks the emq: prefix and interpolates the queue name verbatim: the shipped EchoMQ.Keys.base/1 returns "#{prefix}:#{name}" (keys.ex:97-98), key_prefix/1 returns "#{prefix}:#{name}:" (keys.ex:85-86) — no wrapping, no validation, no placement. A queue named payments produces emq:payments:wait, emq:payments:active, and the rest: flat strings with nothing for a thread-per-shard datastore to balance on.

emq.1 ships the owned keyspace. Every per-queue key becomes emq:{q}:<type> — the emq: prefix replacing emq:, the queue name brace-wrapped transparently by the core (a caller still writes Queue("orders") and the core places emq:{orders}:*). The wrap is idempotent (an already-braced caller name is not double-wrapped) and validated (no colon inside a hashtag; emq is rejected as a queue name). The braced base {emq}: is reserved for the core's own cross-queue keys — so the core's namespace and any queue's cannot collide (INV3). Every queue is slot-local by construction.

The rung is drafted. lib/echomq/ext/ and the v2 script set do not exist yet, so every emq:{q}:* surface here is taught from specs/emq/emq.1.md, written "emq.1 ships …" — never asserted as shipped. The v1 anchors (EchoMQ.Keys, bullmq:5.65.1) are real and present.

The v1 → v2 key comparison, on the real grounding

Take the fixed queue payments. The v1 line, today, produces the emq: families — the verbatim-interpolated builders in EchoMQ.Keys: the LIST keys :wait / :active / :paused, the ZSET keys :delayed / :prioritized / :completed / :failed / :waiting-children, the SET :stalled, the STREAM :events, the HASH :meta, and the per-job emq:payments:<jobId>:lock via EchoMQ.Keys.lock/2. These are flat: no hashtag, nothing to place across threads. This is the line the fork freezes at 1.3.0.

emq.1 ships the v2 form of each. emq:payments:wait becomes emq:{payments}:wait, emq:payments:active becomes emq:{payments}:active, … emq:payments:<jobId>:lock becomes emq:{payments}:<jobId>:lock. The emq: prefix replaces emq:; the queue name carries its {…} hashtag, applied transparently by the core; the field set is otherwise the same. The hero figure shows each line's key set side by side — toggle it and watch the prefix flip and the hashtag appear.

Two slots, by construction

emq.1 ships the transparent hashtag so a queue's keys are placed precisely. A datastore that locks on the hashtag routes a key by the substring inside the first {…} — the hashtag. So every v2 key of the payments queue — emq:{payments}:wait, emq:{payments}:active, emq:{payments}:<jobId>:lock — carries the same tag, payments, and lands in the same slot. That is what lets a v2 Lua script touch several of a queue's keys in one atomic multi-key transition — the R2 atomicity floor, now placement-native.

The core's own cross-queue keys live under the reserved {emq}: base — {emq}:version, {emq}:nodes — and group into a second, distinct slot: the queue's and the core's. They cannot collide, because emq is rejected as a queue name (INV3). A property test over arbitrary queue names checks the rule holds: wrap°wrap = wrap, and every per-queue key shares exactly the queue's one hashtag (emq.1-D1). emq.1 ships the owned keyspace: emq:{q}:* with the brace applied by the core, every queue slot-local, and the core's own keys alone under the reserved {emq}: (INV3) — so a single hashtag per transition keeps it single-slot and atomic.

Same {tag} ⇒ same slot. emq.1 ships a keyspace placed by construction: one queue per slot, the core's own keys alone in their reserved slot — the foundation for locking on the hashtag instead of the whole store.

The 2.0 fork — EchoMQ leaves the BullMQ wire

The verbatim emq:payments:* keyspace is the v1 line's, frozen at 1.3.0 (E2 · the core teaches it). emq.1 ships the break: emq: replaces emq:, the per-queue {q} hashtag is applied transparently by the core, and {emq}: is reserved for the core's own keys — so DragonflyDB can lock on the hashtag precisely instead of the whole store.

Recap — emq.1 owns its keyspace

The v1 line interpolates the queue name verbatim under emq: (EchoMQ.Keys, keys.ex:85-86) — flat, unplaced, frozen at 1.3.0. emq.1 ships emq:{q}:<type>: the emq: prefix replacing emq:, the per-queue {q} hashtag applied transparently (idempotent, validated), the braced base {emq}: reserved for the core's own cross-queue keys, and emq rejected as a queue name so the namespaces cannot collide (INV3). Every queue is slot-local by construction, so a hashtag-locking datastore places one queue per thread and distinct queues across cores.
