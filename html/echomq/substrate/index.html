Why & when

The v1 wire carries two structural inheritances that cap it on any placement-aware store — wherever a key's {…} hashtag decides its placement. First, the undeclared-keys flaw: roughly half of the 50 inherited Lua scripts (about 24–26, pattern-dependent) build per-job keys inside the script (local jobKey = keyPrefix .. jobId — moveToActive-11.lua:148) and never declare them in KEYS[] — outside the engine scripting contract (Valkey's own guidance: every key a script accesses arrives as an input key argument). Second, the unplaced keyspace: EchoMQ.Keys interpolates the queue name verbatim — no hashtags, so nothing for the threads to balance on. Containing both under compatibility has a proven ceiling; resolving both means breaking the wire. So the wire breaks — once, versioned, at the first rung.

Read this chapter first if you follow the build into groups (E4), batches (E5), lifecycle controls (E6), or the cache (E7) — each rides the v2 protocol this break establishes. The interactive below shows the fork directly: the same queue's keys on the v1 line and under the 2.0 protocol emq.1 ships.

How it works

The break is one rung with one mechanism: own the keyspace, declare the keys, version the wire. emq.1 ships the key builders that write emq:{q}:wait, emq:{q}:active, emq:{q}:<jobId> — the per-queue hashtag inserted by the core, never by the caller — and a script set in which every key a script touches is listed in KEYS[] before the script runs. The v1 line concatenates per-job keys inside the script (local jobKey = keyPrefix .. jobId, moveToActive-11.lua:148), outside the engine scripting contract — Valkey's guidance: every key a script accesses arrives in KEYS[]. With every key declared and placed by its {q} hashtag, each call's key set is slot-local and known before it runs. meta.version records echomq:2.0.0; a two-way boot fence refuses cross-version contact; the migration path moves a v1 deployment over once, deliberately.

the pattern · ← redis-patterns R0, R2

A multi-key change is one indivisible Lua move over KEYS/ARGV with no foreign command (R2), dispatched once by SHA (R0) — and the KEYS[] declaration is load-bearing: it is how the store knows what a script will touch before it runs.

the implementation · ⇄ emq.1 (drafted)

emq.1 ships the emq:{q}:… keyspace and the declared-keys script set, so every move keeps the v1 line's atomicity and gains a true, declared key set — slot-local by construction, no escape hatch.

The break is the one rung that ships no feature — it ships the wire, so every later rung is a set of declared-keys scripts on an owned, placed keyspace, and "correct on the v2 wire" is a check that runs.

The 2.0 fork — EchoMQ leaves the BullMQ wire

This chapter is where the fork happens.
