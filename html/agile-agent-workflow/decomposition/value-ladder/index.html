A2.06 · Decomposition

The value ladder

A ladder is a pile of stories put in order: each rung adds one usable capability, depends only on the rungs below it, and leaves the system runnable.

The earlier modules made good stories one at a time — value not tasks, the Connextra form, the INVEST tests. A value ladder is what those stories become when they are ordered: a dependency-ordered stack where value accrues from the bottom up and every prefix can be demonstrated. It is the structure the rest of the course plans, specifies, and builds over. Three dives: compose the rungs, order them by dependency, and keep the system runnable at every one.

What a ladder is The three dives References

A pile becomes a ladder

A pile is a set of stories with no order; a ladder is the same stories arranged so they stack. Three properties make a stack a ladder, and each is one dive below. Every rung is a capability — a vertical slice a role can exercise, not a technical task. Every rung depends only on rungs below it — the stack is a topological order of the dependency graph. Every prefix runs — building the bottom rungs leaves a system that can be demonstrated. Lose any one and the stack stops being a ladder.

The three dives

Each dive isolates one property of a ladder, in the arc compose → order → keep runnable: assemble the rungs as capabilities, order them so each depends only on rungs below it, and confirm every prefix still runs. Read them in order — A2.06.1 starts.

A2.06.1built

Compose the ladder

Assembling individual stories into one ordered ladder: each rung a usable capability, not a task. The move from a pile of stories to a stacked ladder.
A2.06.2built

Dependency order

Each rung depends only on the rungs below it: a topological order. Value accrues bottom-up, and ordering by value reconciles with ordering by dependency.
A2.06.3built

Always runnable

Every rung leaves the system runnable and demoable. This is the structure the roadmap layer later plans delivery over — one prefix at a time.
