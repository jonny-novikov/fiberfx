Split and test

Split the too-big story

The first story set left one story flagged: “a learner tracks progress through a course.” It is valuable and testable in principle, but it fails Small — it spans every lesson, the web surface, the bot, and the dashboard at once — and a story that cannot be sized cannot be estimated either. A flag is not a rejection: the value is real, so the story is kept and split.

Split it vertically, by workflow step, so each slice is still a change a role can demonstrate. The parent becomes three slices: mark one lesson complete, see a course’s completion count, and see completed courses on the dashboard. The split is finished not when the story is smaller, but when each slice passes the six tests on its own — re-test, and each scores six of six where the parent scored four.

parent
“Track progress through a course” — INVEST 4/6, fails Small and Estimable. Real value, too large for one rung.
slice A
“Mark one lesson complete” — a learner finishes a lesson and it is recorded. A vertical cut; passes all six.
slice B
“See a course completion count” — a learner sees how many lessons are done. A vertical cut; passes all six.
slice C
“Completed courses on the dashboard” — the dashboard surface reads it back. A vertical cut; passes all six.

Re-testing each slice against the A2.03 INVEST gate is the move that ends the split. The number of slices does not matter; passing the six tests the parent failed does.

Vertical, not horizontal

There is a wrong way to split, and it is the tempting one. The same parent could be cut by technical layer: “add the progress table”, “write the progress context function”, “build the progress LiveView.” That is a horizontal split, and each piece fails the same tests a chore fails — Valuable and Testable — because no role can observe a schema or a context function on its own. A horizontal cut slices the work, not the value.

Switch the cut below. A vertical split yields three slices a learner can each demonstrate; a horizontal split yields three layers none of which can be demoed alone. The readout scores each cut against Valuable and Testable and reports how many slices are demoable on their own.

The split, written out

On the Portal, the parent and its slices read as below. The parent carries its INVEST verdict; each slice carries a role-want-reason card and scores six of six. The branded store and the event-sourced engine behind one facade hold the recorded progress; that machinery is built in the companion course. Here the work is the slicing, and the test that ends it is INVEST, not a line count.

# Parent (from the first set): too big — fails Small, Estimable.
track progress through a course   # INVEST 4/6 · fails S, E

# Split vertically, by workflow step. Re-test each slice against INVEST.
mark one lesson complete          # INVEST 6/6 · ready
  As a learner, I want a lesson marked complete, so that my progress is recorded.
  Given an open lesson, when the learner finishes it, then it is recorded as complete.

see a course completion count     # INVEST 6/6 · ready
  As a learner, I want a completion count, so that I can see how far I have come.

completed courses on the dashboard# INVEST 6/6 · ready · the dashboard surface
  As a learner, I want my completed courses listed, so that I can review what I finished.

# A horizontal cut (schema / context / view) would fail Valuable and Testable per slice.
the principle

A split is finished when every slice passes INVEST on its own, and a slice is a vertical cut of value — never a horizontal layer no role can demo.

Recap — three slices, all ready

The flagged story split into three vertical slices — mark one lesson complete, see a completion count, see completed courses on the dashboard — each re-tested to six of six where the parent scored four. The horizontal alternative was rejected because each layer fails Valuable and Testable. The first set’s three ready stories plus these three slices make six rung-sized stories, all passing INVEST. They are still unordered — the last dive turns them into a ladder.
