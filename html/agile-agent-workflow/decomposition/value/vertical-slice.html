A2.01 · Value, not tasks · dive 3

The vertical slice

Value cuts through the layers. A story is a thin vertical thread — store, domain, surface — that a role can use end to end.

Its opposite is a horizontal slice: one whole layer, finished in isolation, that no role can demo. Toggle the grid on the right between a vertical thread and a horizontal layer, and read whether a role-usable thread exists.

The definition Widen story by story On the Portal References

Value cuts down, not across

The Portal stacks three layers: a store (data and ids), a domain core (the framework-free rules), and a surface (the web app, the bot, the dashboard). Work can be cut through that stack two ways. A horizontal slice is one layer across the whole system — all the tables, the whole domain, the entire web layer. Each is large, each is real, and none is demoable on its own: there is no end-to-end thread a role can pull.

A vertical slice is one thin thread through every layer for a single story: enough store, enough domain, enough surface to let a role do one new thing. It is demoable, because it is end to end. This is the tracer-bullet shape — thin but complete, top to bottom, proving the path before it is widened. Cutting vertically is how value, not effort, becomes the unit of work.

A vertical slice is exactly the thin, provable rung the loop runs on: complete top to bottom, small enough for one pass, demoable the moment it lands.

Widen story by story

A system built from vertical slices grows differently from one built layer by layer. Each new story adds one thin thread through the layers, and each thread is demoable the day it lands; value accrues one thread at a time, not one layer at a time. The horizontal alternative demos nothing until the last layer crosses every story.

Add the Portal’s learner stories one at a time and read what a learner can do as the threads stack. Each step adds a complete vertical slice; the demoable surface grows downward-complete, never a half-built layer.

The system grows by stacking thin vertical threads — each demoable the day it lands — not by finishing one horizontal layer that demos nothing until the end.

The vertical slice on the Portal

“A learner browses the catalogue of courses” is a vertical slice. It touches the store (a few courses to list), the domain (the rule that only published courses appear), and the surface (the page that renders them) — a thread thin enough for one rung, complete enough to demo. The horizontal alternative — build every table, then the whole domain, then the whole UI — finishes three large chores and demos nothing until the last one lands.

# Vertical: one thread through every layer for one story → demoable now
browse the catalogue:  store — domain — surface   # spans 3/3 layers · a complete thread

# Horizontal: one whole layer across all stories → no thread, nothing to demo
store   across all stories                # spans 1/3 layers · no story end to end
domain  across all stories                # spans 1/3 layers · no story end to end
surface across all stories                # spans 1/3 layers · no story end to end

The vertical slice demos on day one and widens story by story; the same id authority, Portal.ID.generate/1, is touched by each thread without ever being a slice of its own. The Portal’s Elixir and OTP internals — how the store and domain are built — are taught in the companion Functional Programming course.

The principle

Value is a thin vertical thread through the layers, not a finished horizontal layer; a slice that no role can pull end to end is a chore.

Recap — cut vertically

Value cuts down through the layers, not across one of them. A story is a thin vertical thread a role can use end to end and therefore demo; a horizontal layer is real work with no thread and no demo. The system grows by stacking thin vertical slices, value accruing one thread at a time. This closes A2.01: a unit of work is a unit of demonstrable value — named for a role, ordered by value, cut vertically. The next module gives that slice a shared written form.
