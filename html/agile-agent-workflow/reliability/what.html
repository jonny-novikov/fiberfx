The four pillars

A6's scope is four pillars, each named at the course roadmap's altitude and each closing one failure a happy-path demo is blind to. They are not features; they are the same increment made to hold under conditions the build never exercised.

OTP supervision
A crash is isolated and recovered, not propagated. What it adds: a failing process is restarted clean instead of taking neighbours down.
boundaries
Untrusted input meets a single edge before it reaches the core. What it adds: the core trusts only values that crossed the boundary.
parse-don't-validate
The boundary turns an untrusted string into a typed value or rejects it there. What it adds: the typed value carries its own guarantee inward, with no downstream re-check.
property tests
An invariant is proven across generated inputs, not asserted on a chosen few. What it adds: a property holds over thousands of generated inputs, not three examples.

Each pillar's OTP mechanics — the supervision tree, the process model — are taught by the companion Elixir course. This chapter cites them and does not re-teach them.

Coverage is all four

Each pillar closes exactly one failure case, and the four cases are independent: a boundary that parse-don't-validates closes a malformed input, a supervisor closes a propagating crash, and a property test closes an unproven invariant. Coverage is not partial credit — an increment is production-grade only when every failure case it can hit is closed. Toggle the pillars below and the matrix reports how many of the four cases are covered; with no pillar, none are.

A6's scope is four pillars, each closing one case; coverage is production-grade only when every case is closed, not some.

The pillars on the Portal

The Portal lands all four pillars in one place. Its boundary parses an untrusted string into a typed value with Portal.ID.decode/1 — which returns a value carrying its .type and .timestamp, or rejects the input — so a malformed id never reaches the core. A supervisor isolates a crashed process. And a property test proves the round-trip invariant: a generated id from Portal.ID.generate/1 decodes back to itself across generated inputs, not three chosen ones. The supervision tree and the OTP process model are the companion Elixir course's — cited here, not re-taught.

The principle

Production-readiness is four named techniques — supervision, boundaries, parse-don't-validate, property tests — each closing a specific happy-path weakness, and complete only together.
