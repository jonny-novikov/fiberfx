Reliability and correctness

A5 leaves a built increment that passes its own demo. A6 makes it production-grade — it survives a malformed input, a crashed process, and an invariant proven across generated inputs rather than asserted on a chosen few.

“It builds and the demo passes” is not yet “it survives production.” A happy-path increment can still crash on a malformed input, leak a crashed process, or hold an invariant only by luck. This chapter closes that gap with the techniques that turn a built increment into a production-grade one: OTP supervision, boundaries that parse-don't-validate, and property tests — named here at the course roadmap's altitude and practised on the Portal.

The gap A5 leaves

A5 — the agent brief wraps an A4 spec in a machine-readable brief and runs a Claude Author to turn it into code. A5 produces a built increment: code that exists and passes its own acceptance. That is real, and it is not enough. A demo passes the inputs the author chose; production sends the inputs the author did not. Three failures separate “it builds” from “it survives”, and a happy-path demo is blind to all three.

An unhandled malformed input travels inward and corrupts state far from where it entered. A crashed process propagates and takes neighbours down with it. An invariant held by luck — asserted on three chosen examples — breaks on a fourth input no one imagined. A6 exists to close exactly this gap: it names what production-readiness A5 leaves unproven, and proves it.

The principle

A built increment that passes its demo is not yet production-grade. Production-readiness — supervision, boundaries, proven invariants — is separate work from the build, and it is the work A6 names.

On the Portal

The Portal's boundary parses untrusted input into a typed value — Portal.ID.decode/1 returns a struct or rejects the string — a supervisor isolates a crashed process, and a property test proves an id round-trips across generated inputs.

A5 says “it builds and the demo passes.” A6 says “it survives a malformed input, a crashed process, and an unproven invariant.”

What this chapter covers

A6's scope, named at the course roadmap's altitude, is four pillars: OTP supervision (a crash is isolated and recovered, not propagated), boundaries (untrusted input meets a single edge before the core), parse-don't-validate (that edge turns a string into a typed value or rejects it there), and property tests (an invariant proven across generated inputs, not asserted on a chosen few).
