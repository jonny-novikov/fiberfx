Tracer vs prototype

Three thin-looking artifacts, told apart on two axes: does it carry real feature value, and is it kept or thrown away.

A tracer is a real thin feature, kept and grown. A walking skeleton is the end-to-end structure with no feature in it yet. A prototype is built to learn one thing, then discarded. On the Portal, f6.1 is the tracer — kept and grown into the live platform. Pick one on the right to read its profile.

Three artifacts, two axes

The three are easy to confuse because all three look thin. They separate cleanly on two questions. Does it carry real feature value — a path a user could actually take? And is it kept — grown into the shipped product — or thrown away once it has done its job? A tracer answers yes and kept; a walking skeleton answers no and kept (the structure survives, the feature is still to come); a prototype answers no and thrown away.

tracer
A tracer bullet — a real thin feature, threaded end to end, kept and grown. It carries one real path through the system.
skeleton
A walking skeleton — the end-to-end structure with no feature in it yet: the layers wired together with a placeholder where the behaviour belongs.
prototype
A prototype — built to learn one thing (does this approach fit?), then discarded. Its value is the answer it gives, not the code it leaves.

The danger is the prototype that is never thrown away: a throwaway spike reused as the foundation makes a draft into the product by accident. A tracer is built to be kept; a prototype is built to be discarded — keeping that distinction is what stops the mix-up.

Kept vs thrown away

The sharpest difference is fate: how much of each survives into the shipped platform. A tracer is kept whole and grown — all of it lives on. A walking skeleton's structure survives, but it carried no feature to keep. A prototype is discarded once it has answered its question; reusing it is the mistake. The figure reports the kept fraction for each, over a fixed model.

The two interactives teach different moves: the hero places each artifact on the two axes, this one measures how much of each is kept. Pick an artifact and read the survival bar.

A tracer is kept whole; a prototype is kept at zero — built to be discarded once it has taught its one thing.

The tracer, on the Portal

On the Portal, rung f6.1 ships a real, if thin, feature: a request renders a real page from the real Portal facade. It is kept — every later rung grows it. A walking skeleton, by contrast, would wire the endpoint to the view with a placeholder where the facade call belongs: structure, no behaviour. A prototype would be a throwaway spike to answer one question and then be deleted. F6.1 is neither.

The distinction matters because the three are built differently. A tracer is built to production quality from the start, because it stays; a prototype is built fast and loose, because it goes. Treating a prototype as a tracer ships a draft; treating a tracer as a prototype throws away work that was meant to be kept. Every rung holds the master invariant: “The web layer calls only the Portal facade and renders only the closed %Portal.Error{} set.”

# The three artifacts, told apart on two axes.
#   artifact          | real feature? | kept or thrown away?
#   tracer (f6.1)     | yes (thin)    | kept and grown
#   walking skeleton  | no            | structure kept, no feature
#   prototype         | no            | thrown away after it teaches

artifact   = "tracer"
real_feature = true     # f6.1 renders a real page from the real facade
kept         = true     # grown by every later rung — not discarded
example      = "f6.1 — the engine served as a web app"

# A prototype would be kept = false: deleted once it answered its question.
