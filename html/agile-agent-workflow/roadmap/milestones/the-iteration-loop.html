The iteration loop

Inside a milestone, each rung is one iteration — a PR-sized increment sized over four columns: Ships, Demo, Harness, Feedback.

The per-iteration table on the Portal's real phoenix.roadmap.md sizes every rung that way. Pick a rung on the right to fill its row — the slice it ships, the demo, the harness, and the feedback it asks for — each accepted before the next begins.

One iteration, four columns

The milestones group the rungs; the iteration sizes each one. The per-iteration table on phoenix.roadmap.md gives every rung one row over four columns, verbatim: Rung, Ships (the slice), Demo, Harness, Feedback asked. The file calls each row “a PR-sized increment — a spec triad, the slice, a green harness, a demo, a feedback note”.

The columns hold the rung accountable. Ships names the vertical slice. Demo is the observable proof a real role can watch. Harness is the test that keeps it green. Feedback asked is the one question the demo puts to the reviewer. A rung is accepted — demo green, harness green, feedback taken — before the next begins.

Ships
the vertical slice the rung delivers — e.g. f6.1: “the engine served as a web app (endpoint, request → facade → render)”.
Demo
the observable proof — e.g. f6.1: “hit the root, see a page”.
Harness
the test that keeps it green — e.g. f6.1: a ConnTest GET smoke.
Feedback
the question the demo asks — e.g. f6.1: “shell and layout right?”.

The per-rung loop

Each row is run by a loop, stated verbatim in the file: sharpen → build → ship → demo → review → feedback → adapt. The Author sharpens the spec triad, builds the slice, ships it, demos it, the Operator reviews, feedback is gathered, and the plan adapts. The file's rule is the load-bearing part: feedback edits the spec, the single source of truth, never the shipped code.

Step the loop below to walk the seven phases. It teaches a different move from the hero: the hero fills one rung's table row; this walks the loop that produces and accepts it, and marks the two phases that edit the spec.

Run each rung through one iteration; feedback edits the spec, not the shipped code, and the next rung starts from the corrected spec.

The iteration loop, on the Portal

On the Portal, the per-iteration table is the operational heart of phoenix.roadmap.md. Take f6.1: it ships the engine served as a web app; its demo is “hit the root, see a page”; its harness is a ConnTest GET smoke; its feedback asks “shell and layout right?”. A PR-sized increment with a demo and a harness, accepted before f6.2 begins.

When the demo prompts a change — “the shell needs the nav” — that feedback edits f6.1's spec. The next rung is built from the corrected spec, not by patching shipped code. The roadmap holds order; the spec holds behaviour; feedback flows into the spec.

Recap — one iteration per rung

Each rung is one iteration over four columns — Ships, Demo, Harness, Feedback — a PR-sized increment run by the loop sharpen → build → ship → demo → review → feedback → adapt. Feedback edits the spec, never the shipped code, and a rung is accepted before the next begins.
