A3.1 · The roadmap layer

Agile, distilled

Working software in short cycles, responding to change, inspect-and-adapt. A handful of principles actually drive the workflow — separated from the ceremony that does not.

Most Agile process exists to coordinate a team of people. The Author/Operator pair is not a team, so a lot of that process is ceremony here — not wrong, only answering a problem the pair does not have. This module keeps the load-bearing few, each of which names a concrete move in the loop, and drops the rest. Three dives: the values, the inspect-and-adapt engine, and what we keep versus what we drop — all landed on the Portal.

Load-bearing or ceremony The three dives References

Load-bearing, or ceremony

There is a single test for whether an Agile principle earns its place in this workflow: does it name a move the Author/Operator loop already needs? The loop runs sharpen → build → ship → demo → review → feedback → adapt on every rung. A principle is load-bearing when it drives one of those moves, and ceremony when it adds process around the loop without adding a move.

Select a principle below. The readout reports its kind and, if it is load-bearing, the loop move it drives; the dataset holds eight principles, five load-bearing and three ceremony. The default selection is responding to change — the value that becomes the loop's adapt.

The principle

Keep only the principles that drive a move; drop the ceremony that does not. The test is whether a practice adds a move the one-agent / one-reviewer loop already needs.
