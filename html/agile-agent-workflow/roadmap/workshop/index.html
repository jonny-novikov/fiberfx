Workshop: roadmapping Portal

Four modules taught one move each. The capstone runs all of them on one real target and produces an artifact you can read — the F6 roadmap itself.

Five moves, one roadmap

The roadmap layer decides how we deliver the A2 backlog, separately from what we build and prove — which is the spec layer, A4. The workshop is that layer applied once, end to end, on the Portal's web surface. It runs in five stages, and each produces a named artifact.

Ladder to roadmap orders A2's value ladder into the nine-rung sequence f6.1…f6.9 (A3.3, A3.5). Group into milestones folds the rungs into the three F6 milestones — ship the catalog, make it live, ship to users. Fill the iteration table gives each rung its row: Ships, Demo, Harness, Feedback asked (A3.5). Choose the tracer picks the first end-to-end thread, F6.1, and names the seams deferred (A3.4, A3.7). Place in the program sets the chapter beside the bot as a parallel surface over the one facade (A3.6).

Over all five sits the master invariant, quoted verbatim from the file: “The web layer calls only the Portal facade and renders only the closed %Portal.Error{} set. No controller, LiveView, plug, or template names Portal.Engine, a repo, or GenServer.call.” The output is not invented — it reproduces the real phoenix.roadmap.md, line for line.

The three dives

Each dive carries one pass of the workshop on the real F6 chapter, in the arc ladder to roadmap → choose the tracer → the program view: first turn the value ladder into a chapter roadmap with milestones and the iteration table, then pick the tracer bullet and name the deferred seams, then place the chapter beside the bot in a program view over one facade. Read them in order — A3.8.1 starts.
