Continuous feedback

The third Extreme Programming practice: a review after every increment, not at the end. The loop closes between rungs, so a wrong assumption is caught one rung deep rather than nine.

Continuous feedback is the practice that makes the other two pay: small releases and incremental design lower risk only if a reviewer is in the loop at every release. Set how often a review fires on the right and watch the feedback rounds and worst-case rework move over the Portal’s F6 ladder.

What continuous feedback is

Continuous feedback is a review after every increment, not one review at the end. After a rung ships, it is demoed, reviewed, and the review returns feedback — before the next rung is planned. The full per-rung loop is sharpen → build → ship → demo → review → feedback → adapt. Closing the loop every rung means a wrong assumption surfaces with one rung built on it, not nine; the worst-case rework is the number of rungs between two reviews.

The practice carries one rule about where feedback lands: feedback edits the spec, the single source of truth, and the build follows the spec. Feedback never patches a shipped page in isolation. That rule is what keeps the source of truth from forking — the subject of the next section. Continuous feedback is what makes small releases and incremental design pay: both lower risk only because a reviewer meets the work at every release.

Continuous feedback only protects the system if the feedback lands in the right place; the next section renders what happens when it edits the spec versus when it patches the shipped page.

Where feedback lands

A review surfaces a wrong assumption, and there are two places the feedback can land. It can edit the spec — the single source of truth — and let the build follow, so every downstream artifact is regenerated from one place. Or it can patch the shipped page directly, leaving the spec describing a system that no longer exists. The first keeps the source of truth single; the second forks it. Switch the choice to render which artifact stays authoritative and what drifts.

Feedback that edits the spec keeps the source of truth single; feedback that patches the page forks it and the spec starts to lie.

Continuous feedback on the Portal

The F6 roadmap states the loop word for word: “sharpen → build → ship → demo → review → feedback → adapt. Feedback edits the spec, because the spec is the single source of truth; the build follows the spec.” Every rung carries one feedback question for the review to answer — F6.1 asks “shell and layout right?”, F6.5 asks “layout, UX, error wording?”, F6.7 asks “what should propagate; count semantics?” The Operator reviews the shipped increment and returns feedback asking for the next rung’s specs or a change to a shipped one.

The Author makes each round cheap. A Claude agent ships one PR-sized rung with a green harness, so the review meets working software at every release rather than once at the end — and the feedback it returns edits the spec, never the shipped page, so the source of truth never forks.

Recap — the module closes

Continuous feedback reviews every increment and lets feedback edit the spec, not the shipped page, so a wrong assumption is caught one rung deep and the source of truth stays single. It is the practice that makes small releases and incremental design pay. Together the three — small releases, incremental design, and continuous feedback — are the cadence the F6 ladder runs at: nine rungs, each grown behind the facade, shipped, demoed, and reviewed before the next. A Claude agent makes that batch of one cheap enough to be the default. That closes A3.2.
