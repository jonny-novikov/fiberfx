The contract between a spec and its stories

A spec answers what we build and how we know it is right. Its Deliverables section lists the concrete artifacts the rung produces. The .stories.md answers a neighbouring question — who wants each deliverable, and how we will know it works — as Connextra user stories (as a … I want … so that …) with Given/When/Then acceptance and an INVEST line. Two files, two questions, one rung.

The rule that binds them is plain: every deliverable is realized by at least one story, and every story names the invariants it encodes. A stories file records this with a single Coverage line at its end — a map from each deliverable to the stories that realize it. A deliverable that does not appear in the Coverage line has no covering story, and the rung cannot yet be accepted as done. The hero figure builds that map over the real F6.1 file; the dives read it line by line.

the principle

A list of deliverables is turned into a map of stories that realize them: every deliverable is covered, and every story names the invariants it exercises.

The three dives

Each dive takes one move, in the arc build → record → read: build the deliverable→story map, read the Coverage line that records it, then read what each story carries on its INVEST line. Read them in order — A4.4.1 starts.
