The Coverage line

The deliverable→story map is not left implicit. A stories file records it as one line at the end — the Coverage line, the audit that every deliverable is covered.

The line reads each deliverable once and names the stories that realize it. A scan of it tells whether the rung is covered, because a deliverable that has no covering story is visible by its absence from the line. One line, the whole map

The deliverable→story map is recorded once, as the last line of the stories file: the Coverage line. It reads each deliverable in order and names the stories that realize it, separated by a middle dot. There is one entry per deliverable, and that is the rule the line enforces by its shape: if a deliverable is missing from the line, its absence is the signal that no story realizes it.

The line is the audit, not the work. The work was done in the stories — the Connextra sentences, the Given/When/Then, the INVEST lines. The Coverage line is the one-glance summary that lets an Operator confirm the rung is fully covered before accepting it. It is also the thing a reviewer reads first: a closed line means every deliverable has somewhere to be proven, an open one points straight at the gap.

Coverage line
The last line of a .stories.md: one entry per deliverable, mapping it to the stories that realize it.
entry
A single D→US… pair, e.g. D4→US2,US3,US5.
closed
Every deliverable has an entry — the rung is fully covered and can be accepted.
open
A deliverable has no entry — a gap; the rung is not yet coverable as done.

The line is read, not computed by hand each time. Its value is that a single scan answers one question: is every deliverable covered.

An open line — reading the gap

The hero figure read the closed F6.1 line, entry by entry. The figure below shows what happens when an entry is missing. Pick a deliverable to drop its entry from the line; the readout reports that the line is no longer closed and names the deliverable with no entry. Choose none and the line is whole again — all seven entries present, closed.

This is the practical use of the line. A deliverable with no entry is a deliverable the rung produces but no story accepts; the open line surfaces it at a glance, before a single line of code is written to it. The two figures teach different moves: the hero reads the line that is there, this one reads the gap when one entry is not.

A closed line carries one entry per deliverable; drop an entry and the line names the deliverable no story realizes — the gap is read, not guessed.

On the Portal — the real F6.1 line

On the Portal — the learning platform this course builds — the file f6.1.stories.md ends with its Coverage line. It maps the seven deliverables of the bootstrap rung to the five stories that realize them. Every deliverable appears, so the line is closed and the rung is coverable as done. The markdown below carries that line verbatim, with the two stories whose entries frame it.

# f6.1.stories.md — the Coverage line, the last line of the file (markdown, not code).
#   D1 → US1            the endpoint, served (an operator boots the app)
#   D7 → US4, US5       verification (an operator restarts, a visitor sees a clean error)

# The line itself, verbatim:
Coverage: D1→US1 · D2→US1,US3,US4 · D3→US2 · D4→US2,US3,US5 · D5→US2 · D6→US1 · D7→US4,US5.

# Seven entries, one per deliverable D1…D7 — every deliverable present, so the line is closed.

Recap — the line is the audit

The Coverage line is the last line of a stories file: one entry per deliverable, mapping it to the stories that realize it. A closed line covers every deliverable and the rung can be accepted; an open line names the deliverable no story realizes, before a line of code is written. The hero read the closed F6.1 line entry by entry; the content figure read the gap when one entry is dropped.
