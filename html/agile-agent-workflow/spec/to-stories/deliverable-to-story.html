Deliverable to story

A spec lists the artifacts a rung produces; the stories file realizes each one with at least one story. A deliverable with no story is a gap.

The mapping is many-to-many: a story can implement several deliverables, and a deliverable can be realized by several stories. Each F6.1 story records the deliverables it implements verbatim. Pick a story on the right to read what it realizes; the content figure builds the map the other way and shows a gap when a story is removed.

The rule: every deliverable, at least one story

A spec's Deliverables section is a list of the concrete artifacts a rung produces — the endpoint, the router, the controller, the view, the liveness route, the verification. The .stories.md realizes each one with at least one user story: as a… I want… so that…, with Given/When/Then acceptance. The rule is one-directional and total: for every deliverable there is a story that realizes it.

A deliverable with no story is something the rung produces that no one asked for and that nothing accepts. It is built without a reason and shipped without a check — the kind of thing that drifts into a codebase and is never verified. Writing the stories so that each deliverable is realized is what keeps the rung honest: the Coverage line at the end of the file is the audit that proves no deliverable was left behind.

deliverable
A concrete artifact the rung produces, listed in the spec (F6.1 has seven: D1…D7).
realizes
A story realizes a deliverable when the deliverable appears on the story's "Implements deliverables" line.
many-to-many
A story can implement several deliverables, and a deliverable can be realized by several stories.
gap
A deliverable that no story realizes — visible in the Coverage line by its absence.

The mapping runs both ways: from a story, read the deliverables it implements; from a deliverable, read the stories that realize it. The two views are the same map, read from each end.

Covering every deliverable — and reading the gap

The hero figure read the map from the story side. The figure below reads it from the deliverable side: pick a deliverable and see the stories that realize it. F6.1's seven deliverables are all covered, so the map is closed. To see what a gap looks like, drop a story: removing US2 leaves D3 and D5 — realized only by US2 — with no covering story, and the covered count falls from seven to five.

That fall is the whole point. The map is not a decoration; it is a check. A deliverable that drops out of coverage is a deliverable the rung cannot accept as done, because nothing exercises it. The two figures teach different moves: the hero reads one story's reach, this one proves that every deliverable is reached — and shows the gap when one is not.

A closed map covers every deliverable; drop a story and the deliverables it alone realized fall out — the gap is read straight from the count.

#  so that I can see the courses I am enrolled in.

# Acceptance: Given a signed-in learner with enrollments, when GET /my/courses (protected),

Recap — the map, both ways

A spec lists deliverables; the stories file realizes each one with at least one story, and the mapping is many-to-many. From a story, read the deliverables it implements; from a deliverable, read the stories that realize it — the same map, read from each end. A deliverable with no story is a gap, visible the moment a covering story is removed. The next dive reads the single line that records this map for the whole rung.
