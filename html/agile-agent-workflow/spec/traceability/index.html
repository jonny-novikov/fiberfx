A4.6 · The spec layer

Correct by definition

Completion is not a claim a person makes — it is a closure a rule verifies. The traceability chain is the ledger that makes “done” checkable from the text alone.

Every deliverable is realized by a story; every story is accepted by a check and encodes an invariant; every requirement is satisfied; every invariant holds under test. Step the chain on the right over the Portal's real F6.1 ids; when every link resolves, the rung is correct by definition. Three dives: the chain, the closure, and a broken link.

What traceability is

A roadmap ordered the work; a spec defined it; stories accepted it; invariants pinned it. A4.6 closes the chapter's argument: completion is a closure, not a claim. Traceability is the property that every artifact carries a stable id and every id is reachable from intent to proof — so “done” can be read from the text, not asserted by a person.

The chain runs straight down. A deliverable is realized by a user story; the story is accepted by its Given/When/Then and encodes an invariant; the deliverable is also built by an agent story governed by a requirement, which is proven by an invariant or an acceptance test. This is the same idea A1.05 — Correct by definition named; A4.6 makes it operational over the Portal's real F6.1 ids.

# specs.approach.md — the traceability chain (verbatim).
Deliverable (fN.M.md  · FN.M-D#)
   └─ realized by → User story (fN.M.stories.md · FN.M-US#)
        ├─ accepted by → Acceptance criteria (Given/When/Then on the story)
        └─ encodes     → Invariant (fN.M.md · FN.M-INV#)
   └─ built by   → Agent story (fN.M.llms.md · FN.M-AS#)
        └─ governed by → Requirement (fN.M.llms.md · FN.M-R#)
             └─ proven by → Invariant (FN.M-INV#)  or  an acceptance test

Traceability turns “done” into a property you can read: every id reachable from intent to proof, the completion rule a closure rather than an opinion.

The three dives

Each dive takes one part of the closure, in the arc walk → close → break: the chain that links every id, the completion rule that closes it, and the broken link that opens it again and names the gap. Read them in order — A4.6.1 starts.

A4.6.1built

The chain

Walk every id from deliverable to story to acceptance and invariant to requirement, over the Portal's real F6.1 triad. The path is read from the text, not inferred.
A4.6.2built

The closure

The completion rule, clause by clause: deliverables covered, criteria pass, requirements satisfied, invariants hold — every gate run, not asserted. The closure A1.05 calls correct by definition.
A4.6.3built

A broken link

Remove one link — an uncovered deliverable, an unreachable invariant — and the rule reports not done and names the gap. Completion is verified, never asserted.

Break a link, read the gap

A walking chain is not yet a guarantee — the guarantee is that a missing link is caught. Two closures make the chain checkable from the text: the stories file ends with a Coverage line (D#→US# · …) mapping every deliverable to its stories, and each story's INVEST line ends with encodes FN.M-INV#, so every invariant is reachable from a story. Break either and the rung is not done.

Below is F6.1's intact Coverage line. Drop the stories that cover a deliverable, or a story's encodes, and the readout reports not done and names the exact hole. Completion is a closure: one broken link is enough to open it.

the principle

Completion closes by a rule, not a claim: every deliverable traces to a story, an acceptance check, and an invariant. No behavior is unpinned, and no gate is merely asserted.

A closure is only as strong as its links: remove one and the rule reports not done and names the gap — which is why completion is verified, never asserted.
