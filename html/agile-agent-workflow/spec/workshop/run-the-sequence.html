Six moves, accumulating

The spec layer is not one document written at once — it is six moves, each producing a named part, run in order until the rung is fully defined and provable. The workshop runs them over the F5.1 triad, and the parts accumulate: the abstract requirement collapses into worked examples, the triad splits the work across three files, the anatomy fills section by section, the stories derive from the deliverables, the invariants are stated, and the chain closes.

by-example
Replace “handle requests” with F5.1-US1's three worked cases — boot, POST /enroll → 422, an unknown path → 404. Each doubles as the check (A4.1).
the triad
Split the work across f5.1.md (what & why & done), f5.1.stories.md (who wants what), and f5.1.llms.md (how to build with proof gates) (A4.2).
the anatomy
Fill the six sections of f5.1.md — Goal, Rationale (5W), Scope, Deliverables D1–D7, Invariants, Definition of Done (A4.3).
the stories
Derive five Connextra stories US1–US5 from the deliverables, each with Given/When/Then acceptance and an INVEST line (A4.4).
the invariants
State the five — INV1 (the replaceable seam), INV2 (identity), INV3 (supervision), INV4 (pure id functions), INV5 (functional core) (A4.5).
traceability
Close the chain: the Coverage line maps every deliverable to a story, so completion is a closure, not a claim (A4.6).

The moves are ordered because each depends on the one before: there are no stories without deliverables, no traceability without stories. Run them out of order and the spec has gaps.

One deliverable through the chain

Stepping the whole pipeline shows the spec fill; this figure follows a single deliverable through it. Pick one of F5.1's seven, and watch it connect forward: the story that realizes it (read straight off the Coverage line), the acceptance that story earns (a Given/When/Then from f5.1.stories.md), and the invariant it touches where one applies. The two figures teach different moves — the hero steps the whole sequence once; this traces one item end to end.

The consequence is the lesson: a deliverable is not “done” because it is listed. It is done when a story realizes it and a check accepts it. A deliverable with no story is a hole in the spec — the gap A4.7.3 hunts for.

The sequence on the Portal

On the Portal — the learning platform this course builds — the six moves run over F5.1 — Portal's engine-facade rung (the engine chapter's first rung; the seed of the F6 master invariant). By-example fixes F5.1-US1's three cases; the triad splits the work; the anatomy fills f5.1.md; the stories derive US1–US5; the invariants state INV1–INV5; and traceability closes with the Coverage line. The output is not invented — it reproduces the real f5.1.md and f5.1.stories.md, the Coverage line included.

Every Portal reference here stays at the spec level — rungs, deliverables, stories, invariants. The one id contract the course names freely is Portal.ID.generate/1 / Portal.ID.decode/1; the engine itself is named only through the master-invariant seed. The framework that implements these ideas is taught in the companion course.

Recap — the spec, part by part

Six moves run in order over the F5.1 triad build the spec part by part: worked examples that double as checks, the triad that splits the work, the anatomy that fills the six sections, the stories that derive from the deliverables, the invariants that name what must always hold, and the traceability chain that closes. Each deliverable connects forward to a story and a check, so nothing in the spec is merely listed.
