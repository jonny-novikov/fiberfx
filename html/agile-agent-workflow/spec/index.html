The spec layer

A roadmap ordered the work; a spec defines it. The spec layer is what we build and how we accept it — examples that double as checks, invariants that must always hold, and a traceability chain that makes “done” a closure rather than an opinion.

A3 produced a delivery plan: thin-but-robust rungs, each pointing at a spec it does not contain. This chapter is that spec — how to write one precisely enough that a Claude Author can build it and an Operator can accept it by proof, correct by definition. Every technique lands on the Portal's real rung triads — a spec, its stories, and the brief an agent builds from.

The principle

A roadmap rung names a slice and points at a spec. It says deliver this next, but it does not define what the slice is, or how anyone would know it is done.

On the Portal

A rung's spec triad — f6.N.md, f6.N.stories.md, f6.N.llms.md — defines the slice by example, states the invariants it must never break, and traces every deliverable to an accepting check.

The roadmap answers when a slice ships; the spec answers what it is and how it is proven — and feedback edits the spec, never the built page.

The spec layer, in five questions

The spec layer is a small set of artifacts with a single job: define a slice precisely enough to be built by an agent and accepted by proof. Five questions fix what it is, why it exists, who owns it, when it changes, and how it operates — the deep overview the seven modules then unpack.

What
A spec (Goal, Rationale, Scope, Deliverables, Invariants, Definition of Done) plus its stories (.stories.md) — the single source of truth for what we build and prove, written as worked examples, not abstract requirements.
Why
An agent cannot build, and an Operator cannot accept, what is not specified precisely. The spec is what makes an increment buildable by proof rather than opinion — correct by definition (A1.05).
Who
The Operator writes and accepts the spec; the Author reads it to build and self-checks against its Definition of Done. The spec is the contract they meet on.
When
After the roadmap orders a rung, before the agent brief wraps it for building. The spec changes only by feedback — never silently, never by the build.
Where
It sits under the roadmap (which points at it) and over the agent brief and the code (which derive from it). The specs reference nothing above them.
How
Specification by example, invariants kept distinct from acceptance checks, and a traceability chain from deliverable → story → check & invariant that closes “done” from the text alone.

Each question is unpacked across the seven modules below: by-example and the-triad take the What; spec-anatomy the How of writing one; to-stories and invariants the What-and-Why of proof; traceability the closure; and the workshop runs the whole sequence on the Portal's engine.
