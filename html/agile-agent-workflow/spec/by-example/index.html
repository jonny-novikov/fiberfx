A4.1 · The spec layer

Specification by example

An abstract requirement admits many readings; a worked example pins one. The first move of the spec layer is to make the examples the specification.

“Handle bad input” could mean a redirect, a crash, a 404, or an empty page — the phrase does not decide. Replace it with worked examples and the readings collapse to one, before a line of code is written. On the right, contrast the abstract requirement with the examples that pin it. Three dives: examples as the specification, living documentation that cannot drift, and removing ambiguity.

Examples as specification The three dives References

Examples are the specification

An abstract requirement is a sentence the reader has to interpret. “The page must handle a bad user id” is fine until two people read it: one builds a 404, one builds an empty page, one lets it crash. Each is a defensible reading of the same words, and none of them is what the requirement meant — because the requirement never decided.

Specification by example replaces the sentence with worked cases. Each case is concrete, stated as Given/When/Then, and pins exactly one behaviour. Together the cases narrow the readings to one, and because they are concrete, the build can be held to them: a worked example doubles as the executable check. The ambiguity is removed before a line of code is written, and the documentation cannot later drift from the system — the example is both at once.

The three dives

Each dive takes one consequence of making examples the specification, in the arc be the spec → cannot drift → remove ambiguity: the example as the specification the harness reads, the living documentation that follows, and the mechanism underneath — readings collapsed from many to one. Read them in order — A4.1.1 starts.

A4.1.1built

Examples as specification

The concrete examples are not an illustration of the spec — they are the spec, and the acceptance harness reads its checks straight from them. Pinned on the catalog and /my/courses cases.
A4.1.2built

Living documentation

Because the examples are the checks the build runs against, the documentation cannot drift from the system: a behaviour change that breaks an example breaks a check.
A4.1.3built

Removing ambiguity

The mechanism the first two relied on: add worked examples until exactly one reading of the requirement survives — before any code is written.
