Removing ambiguity

An abstract requirement is a set of possible readings. The work of specifying by example is choosing the cases that narrow that set to one — before a line of code is written.

“Handle a bad user id” could mean a 404, a crash, an empty page, or a redirect. Each worked example rules out the readings it contradicts. Slide on the right to add examples and watch the surviving readings fall — F6.1 closes on one: the empty state, a clean 200.

A requirement is a set of readings

Ambiguity is not vagueness. “Handle a bad user id” is a perfectly clear English sentence; what it lacks is a decision. It admits a 404, a crash, an empty page, a redirect — each a legitimate reading of the same words. The requirement names a set of possible behaviours and stops, leaving the choice to whoever builds it.

Removing ambiguity is the act of adding worked examples until exactly one reading survives. Each example is a concrete case with a stated result, so it contradicts every reading that would have produced a different result — and rules them out. Add the case “an unknown id renders the empty state, a 200” and the 404, the crash, and the redirect are all gone. The set has been narrowed to one, and it was narrowed before a line of code was written.

ambiguity
A requirement that admits more than one reading. The words are clear; the decision is missing.
a reading
One behaviour the requirement could mean. "Handle a bad id" has at least four — a 404, a crash, a redirect, an empty state.
removing ambiguity
Adding worked examples until exactly one reading survives. Each example rules out the readings it contradicts.
before code
The narrowing happens in the spec, ahead of the build, so the agent has one reading to implement and no choice to invent.

An example removes the readings it contradicts; enough examples leave one. The skill is choosing cases that, together, cover the behaviour and converge — not piling on cases that all say the same thing.

Ambiguous, or a pinning example

The test of a statement in a spec is simple: does it leave many readings open, or does it pin one? An abstract phrase — “handle bad input,” “show something reasonable” — opens readings; a worked example — “an unknown id renders the empty state, a 200” — closes them to one. The same spec usually carries both: the prose frames the intent, the examples pin the behaviour.

On the Portal

On the Portal — the learning platform this course builds — F6.1's spec opens with the abstract intent: serve a course page and handle a bad user id gracefully. Left there, “gracefully” is a set of readings. The spec closes them with worked examples: Portal.courses_of/1 over a known user, a user with no enrollments, and an unknown id, each with a stated result. The unknown-id case pins the one reading that survives — the empty state, a clean 200, never a 500.

The fragment below is spec text: the abstract Definition-of-Done phrasing set against the pinning Given/When/Then from f6.1.stories.md. The contrast is the whole lesson — one line read many ways, three examples that leave one.

# f6.1.md — the abstract intent (one line, many readings until pinned):
#   "the page must handle a bad user id gracefully"
#   readings: a 404? a crash? a redirect? an empty page?  ← undecided

# f6.1.stories.md — F6.1-US5: the worked example that pins ONE reading:
#   Given an unknown or malformed user id,
#     when GET /courses/:user_id,
#     then it renders the empty state — a clean 200, never a 500.

surviving_reading = "empty state, a clean 200"   # the one the examples leave
removed           = :the_other_three          # 404, crash, redirect — ruled out

The principle

Choose worked examples until exactly one reading of the requirement survives. The examples remove the ambiguity in the spec, before the agent builds.

Recap — many readings to one

A requirement is a set of readings; ambiguity is the missing decision, not vagueness. Removing it is adding worked examples until exactly one reading survives, each example ruling out the readings it contradicts — and the narrowing happens in the spec, before a line of code is written. F6.1 closes “handle a bad user id” to the empty-state-200 reading. That closes the module: examples are the specification, they are living documentation that cannot drift, and they remove ambiguity.
