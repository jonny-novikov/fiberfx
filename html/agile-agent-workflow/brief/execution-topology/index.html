Execution topology

References named the sources; requirements numbered the checks. Those requirements imply a runtime shape and a build order — and the topology writes both down, with the exact files the agent touches.

The brief's third part answers the three things a spec leaves open about how the code is assembled: what runs (the supervision tree and the request flow), in what order it is built (the task DAG, each step compiling), and in which files (the touched-file list and the boundary the agent does not cross). With them fixed, a Claude Author assembles a system bottom-up without guessing. Select a part on the right; the three dives go deep on each. All grounded on the Portal's real f6.1.llms.md.

From requirements to a shape

A5.2 left a set of numbered, traced requirements — F6.1-R1–R8 on the Portal's f6.1.llms.md. Each names a source and a check, but on its own a requirement does not say where its code lives, in what order it is wired, or what process owns it at runtime. Those are the decisions the spec leaves open, and the execution topology closes them.

The topology has three pinned artifacts. The runtime shape names the supervision tree and how a request flows through it. The task DAG fixes the build order, with each step leaving the app compiling. The file list bounds the change to the exact files created, edited, and deleted. Together they let an Author assemble the increment bottom-up — the same request lifecycle the companion Phoenix request lifecycle chapter builds.

The principle

The topology fixes the runtime shape, the build order, and the files — the three things a spec leaves open about how the code is assembled.

On the Portal

f6.1.llms.md pins the two-supervisor tree (:portal + :portal_web), the T1→T7 order, and the touched-file list, so the agent assembles bottom-up.

Topology is the difference between “build this” and “build this, in this shape, in this order, here.”

Trace a request

The runtime shape is most concrete as a path. A request to f6.1.llms.md's Portal enters the endpoint and walks the plug stack to the router, which dispatches to the controller, which calls the domain through exactly one door: the Portal.courses_of/1 facade. Step through the hops below. Switch the route to /health and the path stops at the router — the one path that never reaches the facade.
