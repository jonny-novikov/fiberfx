Programming with an agent

A5.1–A5.6 built the brief and the practice of running it. This module names the thesis the whole course has argued: a Claude agent is a fast, tireless implementer of well-specified slices; the human owns decomposition, judgement, and acceptance.

The value is the product of two roles, not their sum. The agent supplies speed and implementation; the human supplies direction — decomposition, judgement, acceptance. Speed alone, pointed nowhere, is a fast walk off a cliff; direction alone, with no implementer, is a plan that never ships. The precise statement

The course has argued one thesis from A1 to here, and A5.7 states it plainly. Pragmatic programming with a Claude agent is a pairing of two roles. The agent supplies speed and implementation: it builds a specified slice fast and tirelessly, and it does not tire, drift, or skip steps across a long run. The human supplies direction: decomposition (cutting the work into thin slices), judgement (the calls a spec cannot make), and acceptance (reading the result against the spec's Definition of Done, never the agent's self-report).

The value of the pairing is the product of speed and direction, not their sum. Speed alone, pointed nowhere, is a fast walk off a cliff. Direction alone, with no implementer, is a plan that never ships. The two compound, and the rest of this module shows where the compounding helps and where it hurts.

The worked unit is f6.1.llms.md, the Portal's real brief for the Phoenix web bootstrap over the unchanged facade. It specifies the slice down to requirements F6.1-R1…R8, the task DAG T1→T7, and the acceptance gates per agent story F6.1-AS1…AS4. Its own closing line is the evidence the value is real on a specified slice: “each story is two-to-six files behind one command and one gate, and every branch … is already specified — so the agent implements rather than decides.” “So the agent implements rather than decides” is the thesis in one clause: when the slice is specified, the agent's speed converts directly into progress, because nothing is left for it to decide. That brief is built in the companion course at the Phoenix request lifecycle.

Speed times direction

The product has a sign. The agent's speed is high and roughly constant; what changes between a good run and a bad one is the direction the spec set. On a well-specified slice the direction is right, and high speed carries the work a large step toward done. On an under-specified slice a decision is left to the agent, which it fills by improvising — and high speed carries the work a large step the wrong way.

Toggle the specification below. The model fixes the agent's speed at a high constant and reads the direction from the brief: a well-specified slice scores +1, an under-specified one −1. The readout reports the signed outcome — the same magnitude, with the sign the spec chose. The lesson is upstream: the way to a large positive step is to specify the slice, not to slow the agent.
