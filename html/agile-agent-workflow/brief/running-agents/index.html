Running agents well

A5.1–A5.5 wrote the brief and folded it into one runnable prompt. The brief is complete; what remains is the practice of running it: brief, supervise, review — a loop the human owns around a prompt the agent runs.

The agent runs the prompt; the human owns the gate. Three moves frame the practice: brief the agent with a complete .llms.md so it implements rather than decides, supervise the run against the task gates, and review the output against the spec's Definition of Done — never the agent's self-report. Pick a practice on the right to read its role; the module grounds on the Portal's real F6 ship prompts, all practised on the Portal.

The practice around the prompt

The five parts of the brief assemble into one prompt an Author can run. Assembling it is the end of A5.1 through A5.5. Running it is a separate craft, and it is the human's. Three moves carry it. Brief the agent with a complete .llms.md, so every decision the spec fixed is restated and none is left for the agent to invent. Supervise the run against the task gates, letting a passing stage proceed and intervening on a failing one. Review the output against the spec's Definition of Done.

The third move carries the rule the whole module turns on. A Claude Author can report that it is done; a report is not a check. Review reads the work against the gates the spec named, not the message the agent sent. The value of the practice is in the gates, not in how many agents run in parallel: the Portal's F6 ship prompts run a lead-team Director-in-loop precisely so a human stands at each gate.

This hub frames the practice; the three dives below go deep on each move — briefing, supervising, and reviewing. The reviewing dive carries the liveness gate the content figure below introduces.

Self-report against the gate

The move that separates a good review from a credulous one is the source it checks against. The Portal's F6.7 ship prompt makes the point exactly. A rung's suite can be fully green and still leave the running server unproven, because the test endpoint runs with the server switched off. The F6.7 brief states it verbatim: “mix test runs server:false (config/test.exs), so a green suite does NOT prove the dev server boots.” The Director's ratify gate restates it: “mix test alone does NOT satisfy the liveness criterion (the endpoint runs server: false under test).”

So the reviewer runs the gate the report cannot. For F6.7 that gate is the liveness check: boot the node and confirm /health returns 200, then run the rung's live two-window update. The same shape recurs across the F6 prompts — the LiveView rung (F6.6) the ship prompt builds is interactive only when a booted server is serving it, which a green suite never demonstrates. The three dives

Each dive takes one move of the practice, in the order of a run: brief → supervise → review. Hand the agent a complete brief, watch the run against the gates, then review the output against the Definition of Done. Read them in order — A5.6.1 starts.

Briefing

Hand the agent a complete, unambiguous brief so it implements rather than decides — the spawn carries only the rung delta; the discipline is durable.
