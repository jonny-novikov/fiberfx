The agent brief

A4 wrote the spec — correct by definition. A5 wraps that spec in a brief a Claude Author can run: links first, every reference exact, the runtime shape and build order fixed, the proof gates named — and the practice of running that agent well.

A spec defines what and why and done; it is not, on its own, runnable by an agent. The agent brief is the layer that fixes every remaining how-to-build decision and turns the spec into code. It has five parts — references, requirements, topology, agent stories, the implementation prompt. Select one on the right to read its role, the module that teaches it, and the real .llms.md it lands on, all practised on the Portal.

Where we are

The course is one argument carried across eight parts. The first five are now in place: A0 framed the method, A1 made the case for thin, provable slices, A2 turned a vision into a backlog, A3 planned its delivery, and A4 defined and proved one rung as a spec. A5 is where that spec becomes a brief an agent can run. The remaining parts — reliability and the Portal exemplar — build on the increment A5 produces.

The arc below walks the eight parts. Select one and the readout reports its title, what it delivers, and whether it is built — the same orientation this landing exists to give. The default selection is A5, the part you are reading.

The journey so far

A0 — Foundations named the two roles the course runs on: the human Operator (judgement, decomposition, acceptance) and the Claude Author (fast, well-specified implementation). The Author/Operator loop is the cadence that runs one rung end to end, and A0.2.2 — The four artifacts named the per-rung documents a roadmap line points at.

A3 — The roadmap layer planned delivery without touching the spec, and A4 — The spec layer defined and proved the unit: a thin slice written down as a spec — acceptance criteria, Given/When/Then, invariants, traceability — so “done” is a closure over traced, executed checks, not an opinion. A4 produced a spec that is correct by definition. But a spec says what and why and done; it does not, on its own, become running code.

The principle

A spec defines the work; a brief tells an implementer how to build it — references first, every decision the spec fixed restated for the builder, and none left open for the implementer to invent.
