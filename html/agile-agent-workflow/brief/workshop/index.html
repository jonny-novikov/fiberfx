The full pass

A5 has built one brief, part by part, across seven modules. A5.8 runs the whole of it on a real rung, in three stages. Brief the rung assembles the five parts for one engine rung; a brief missing a part is not runnable. Run the agent runs the implementation prompt in task order, the tree compiling after each step, supervising each step's gate. Verify the increment reads the output against the spec's Definition of Done, not the agent's self-report.

Each stage closes on a gate, and the next stage starts only when the prior one's gate passes. Step the control below through the three stages; the readout reports how many gates have passed and what the workshop leaves at its end — a verified increment, gated at each stage. The default state shows the first stage gated; the remaining two are still pending.

The worked rung

The rung the workshop briefs is F5.2 — Model the Portal domain, the engine chapter's second rung. Its brief f5.2.llms.md is a real, complete .llms.md: References (Elixir structs and @enforce_keys, typespecs, the upstream Portal.Store and Portal.ID), requirements F5.2-R1…R6, the T1→T6 task topology and the touched-file list, agent stories F5.2-AS1…AS3, and the comprehensive implementation prompt in six numbered build steps.

Its requirements are the kind the workshop writes. F5.2-R1 names seven entity structs — User, Session, Course, Lesson, Page, Enrollment, Progress — each with @enforce_keys, a defstruct, and a @type t, so a partial record cannot be built. F5.2-R3 documents the public API per context, each function with an @spec, all others private: Accounts.user/1, Catalog.course/1, Learning.enroll/2, Learning.courses_of/1. The worked web rung this all leads to is built in the companion course at the Phoenix request lifecycle.

# f5.2.llms.md — the requirement the workshop writes (R3)
F5.2-R3  Learning.enroll/2 :: {:ok, %Enrollment{}} | {:error, atom}
         Learning.courses_of/1 :: [%Enrollment{}]

# the worked first write — one context function over the store + a minted id
u  = Portal.ID.generate("USR")
crs = Portal.ID.generate("CRS")
Portal.Learning.enroll(u, crs)   # => {:ok, %Enrollment{progress: 0}}
Portal.Learning.courses_of(u)     # => [%Enrollment{}]

The brief fixes the field lists, the id namespaces, and the return shapes, so the Author implements the rung rather than deciding it. Portal.ID.generate/1 mints a branded id; the real F5 surfaces (Portal.Store, Portal.ID.new/1) are cited only as f5.2.llms.md names them.
