R3.02 · Reliable Queues

At-least-once.

Guarantee at-least-once message delivery using LMOVE to atomically transfer messages to a processing list, enabling recovery if consumers crash before completing work.

R3.01 built the move that loses no job: a worker leases a job into an in-flight position, so a crash leaves it parked and recoverable. That move makes redelivery possible — and that is the point. The consequence is precise: a reliable queue delivers a job one or more times, never zero. A finish-then-crash before the acknowledgement makes a duplicate inevitable, so the responsibility for absorbing it moves to the consumer, which must be idempotent. The grounding is the real EchoMQ worker path in echo/apps/echo_mq: the branded JOB id is the idempotency key, and the worked consumer is codemojex.

Delivered N times, effect once

A reliable queue can hand the same job to a worker more than once. What separates a correct consumer from a broken one is whether the effect stays at one no matter how many times the job arrives. Delivered N times, effect once. A reliable queue is at-least-once; an idempotent consumer is what turns that into an exactly-once effect.

§2

Recovery from Failures

reliable-queue.md.txt · Recovery from Failures

A separate monitor process — a reaper — periodically scans the in-flight set for stalled jobs. For each job that has sat there longer than its lease, it treats the worker as dead and returns the job to the main queue. R3.01 grounded this in EchoMQ's stalled sweep. The recovery path is what makes the queue reliable, and it is also what makes redelivery inevitable.

The window is precise. A worker leases a job, applies the effect — charges a card, writes a row, sends a mail — and then crashes before it can acknowledge the job. The effect already happened; the acknowledgement did not. The reaper finds a job whose lease expired, returns it to the main queue, and a second worker runs it again. The effect runs twice.

This guarantees “at-least-once” delivery — messages may be processed multiple times but are never lost.

That sentence from the source is the whole bargain. The dependency chain is exact: R3.01's in-flight move makes redelivery possible (that is the design); a finish-then-crash before the acknowledgement makes it inevitable; therefore the consumer must be idempotent. There is no fourth option that removes the duplicate without reintroducing the loss.

The recovery path that never loses a job is the same path that can redeliver one. Keep the recovery, accept the duplicate, and move the work to the consumer.

§3

Delivery Guarantees

reliable-queue.md.txt · Delivery Guarantees

This pattern provides at-least-once delivery, which is two properties stated together. Messages are never lost — a job always exists in some place (the pending set, the active set), so no crash drops it. Messages may be processed multiple times — a worker that crashes after the effect but before the acknowledgement leaves the job leased, and recovery runs it again.

The three guarantees that bound the design space are worth naming together.

GuaranteeWhen the job leavesFailure outcome
At-most-oncebefore the work (RPOP)a crash loses the job — zero deliveries
At-least-onceafter the acknowledgement (lease, then complete)a crash redelivers — one or more deliveries
Exactly-once—not a deliverable guarantee — build it as an idempotent effect

A reliable queue must keep the job until it is acknowledged, so it is at-least-once — not a defect of the implementation, but the only point on the spectrum that loses no job.

For exactly-once semantics, make message handlers idempotent — processing the same message twice should produce the same result as processing it once.

This is where the work moves to the consumer. Idempotency converts at-least-once delivery into an exactly-once effect. Some effects are naturally idempotent — a SET key value overwrites to the same value however many times it runs. Others are not — an INCR, a charge, a row insert — and those need a guard: a marker claimed with SET … NX keyed by a stable id, so the second delivery finds the marker and skips.

A reliable queue must keep the job until it is acknowledged, so it is at-least-once — the only point on the spectrum that loses no job.

§4

In EchoMQ — the branded JOB id is the idempotency key

echo_mq/jobs.ex · enqueue

EchoMQ's idempotency starts at the identity, not at a side key. A job is a branded JOB entity, and its identity is the key: EchoMQ.Jobs.enqueue/4 runs one idempotent Lua script that admits by kind, refuses a duplicate id, and writes the row and the pending entry atomically. The duplicate refusal is a single line, and the host maps its 0 to {:ok, :duplicate}. Two enqueues that carry the same branded JOB id collapse to one row, because the id is the key, gated at EchoMQ.Keyspace.job_key/2.

# EchoMQ.Jobs.enqueue/4 — one idempotent script; the branded JOB id is the key (real)
@enqueue Script.new(:enqueue, """
  if string.sub(ARGV[1], 1, 3) ~= 'JOB' then
    return redis.error_reply('EMQKIND job id must be JOB-namespaced')
  end
  if redis.call('EXISTS', KEYS[1]) == 1 then
    return 0                                  -- duplicate id: refuse → host reads {:ok, :duplicate}
  end
  redis.call('HSET', KEYS[1], 'state', 'pending', 'attempts', '0', 'payload', ARGV[2])
  redis.call('ZADD', KEYS[2], 0, ARGV[1])
  return 1
  """)

That refusal is producer-side admission: it stops two enqueues of the same id from both creating work. It is not consumer exactly-once, because the recovery path can still redeliver a job that entered the queue once — a worker that leases a job, applies the effect, and crashes before the acknowledgement loses its lease, and the reaper returns the job to pending for another worker. The consumer absorbs that redelivery by being idempotent: the worker loop hands the handler the job and, on :ok, calls EchoMQ.Jobs.complete/5 with the attempts token; the token is the per-claim fence, so only the current lease-holder may settle the job. Producer admission and consumer idempotency are two disciplines protecting two moments — the enqueue and the effect.

the pattern

A reliable queue is at-least-once; the consumer must be idempotent. The exactly-once effect is the consumer's to build — a naturally idempotent effect, or a marker claimed at effect-time.

in EchoMQ & codemojex

The branded JOB id refuses a duplicate enqueue (Jobs.enqueue → :duplicate), while Codemojex.ScoreWorker absorbs a duplicate delivery — it scores a guess to the same GES and answers :ok for an unknown game, a drop, never a retry loop.

EchoMQ's branded id refuses a duplicate submission; an idempotent consumer absorbs a duplicate delivery. Two disciplines, two moments.
