R8.06.1 · Operating EchoMQ — dive 1

Cluster colocation.

A single-node Valkey lets a Lua script touch any keys at all. A Valkey Cluster does not — a command may only touch keys in one of 16384 slots, or it is refused. A queue's keys must all land on one slot, and the keyspace makes them.

Grounding: EchoMQ.Keyspace.slot/1 and hashtag/1 (echo/apps/echo_mq/lib/echo_mq/keyspace.ex) · the vector slot("123456789") == 12739 · the engine is Valkey 9. The braced keyspace figure is bcs.3.md.

§1

The hashtag and the slot

keyspace.ex · slot/1, hashtag/1

Valkey Cluster gives applications one lever over placement: a hashtag. If a key contains a {…} substring, the slot is computed from the bytes inside the braces alone, not the whole key. So emq:{orders}:pending and emq:{orders}:active both hash on orders — the same slot — even though the full keys differ. EchoMQ builds every queue key braced: the queue name is the hashtag, every key of one queue shares it, and so every key of one queue lands on one slot.

EchoMQ.Keyspace.slot/1 computes that slot client-side: hashtag/1 returns the substring inside the first {…} when present and non-empty, else the whole key; slot/1 is rem(crc16(hashtag(key), 0), 16384) — CRC16-XMODEM over the hashtag, modulo 16384, the cluster specification's exact algorithm. Computing it client-side means the connector can route a key to its node, and partition by slot, without a server round trip. The source pins one known vector: slot("123456789") == 12739 (CRC16 0x31C3).

Notes on ValkeyValkey Cluster splits the keyspace into 16384 hash slots, and a key is mapped to a slot by CRC16(key) mod 16384; when a key holds a {hashtag}, only the substring inside the braces is hashed, so an application can force related keys onto one slot — valkey.io/topics/cluster-spec.

A queue's keys are colocated by construction: every key carries the queue name as its {q} hashtag, so slot/1 places them all on one of 16384 slots from the same few bytes.

§2

Why the braced keyspace keeps Lua legal

CROSSSLOT · the atomic step

The braced keyspace is not decoration — it is what keeps a queue's multi-key Lua legal at cluster scale. A claim reads emq:{q}:pending, writes emq:{q}:active, and touches the job HASH emq:{q}:job:<JOB>. All three keys share the {q} hashtag, so all three hash to one slot, so the EVAL touches one slot and the cluster accepts it. Drop the brace and the three keys scatter across slots; the same EVAL answers CROSSSLOT and the operation cannot run atomically.

Select an operation below and read whether its keys share a slot — the same script is legal braced and rejected unbraced.

Notes on ValkeyIn cluster mode a single command, including a Lua script, may only access keys that hash to the same slot; touching two slots returns CROSSSLOT Keys in request don't hash to the same slot, so colocation is the precondition for an atomic multi-key script — valkey.io/topics/cluster-spec.

Colocation is what makes a multi-key script a single atomic step on a cluster. The brace pins a queue's keys to one slot, so a claim or a complete stays one EVAL — no CROSSSLOT, no manual sharding.

§3

Colocation and the gate are one rule

born braced, branded, declared

The braced key is the same key the keyspace requires for the gate. A JOB id is gated at the key builder, every key is born braced, and every key a Lua script touches is declared in its KEYS[]. Colocation and the gate are two readings of one rule: every key of a queue draws from one braced grammar. The slot is computed from the brace; the gate is enforced at the brace; the script declares the braced keys it touches. One discipline serves placement and safety at once.

# the queue name is the hashtag — every key of one queue shares it.
emq:{orders}:pending
emq:{orders}:active
emq:{orders}:job:JOB0N…        # the branded JOB id, gated at the key builder
# slot("orders") is the same for all three → one slot → one atomic EVAL.
§4

The bridge — hashtag to colocation

pattern → application
the pattern

Colocate related keys on one cluster slot: a hashtag forces a key family onto one of 16384 slots, so a multi-key command stays in one slot and runs atomically.

→
its EchoMQ application

EchoMQ.Keyspace.slot/1 is rem(crc16(hashtag(key), 0), 16384) over the {q} hashtag; every key of a queue shares the brace, so a queue's multi-key Lua EVAL is always CROSSSLOT-safe in cluster mode.

The hashtag is the shard key. EchoMQ braces the queue name into every key, so a queue's keys are one slot and its scripts are one atomic step — the colocation the bus depends on, by construction.

§5

The production angle

deploy on a cluster

Deploy the bus on a Valkey Cluster and nothing in the application changes. Each queue's keys already carry the {q} hashtag, so each queue already lands on one slot; the cluster spreads queues across slots and nodes for you, and each queue's scripts stay legal because they never leave their slot. The hashtag is the shard key, and the operator chose it the moment the queue was named.

the door · codemojex on the colocated keyspace

Where the colocation runs

For codemojex, the cm queue's keys share the {cm} brace — one slot — and the bot workers draining cm all address that one slot; the colocation is automatic, the same as any queue. The deeper protocol — the full Lua inventory, the lane and lease mechanics that run on those colocated keys — is the dedicated EchoMQ Queue pillar. For the braced keyspace as the architecture, see the bus; for where it runs in production, Codemojex on Fly.

Colocation is a deployment property the keyspace already guarantees. Name a queue and its keys are one slot on any cluster — the operator gets atomic multi-key scripts with no manual sharding.
