Pool sizing — pipeline depth, not cores.

A pool is the standard answer to serving many callers over a server. The standard intuition for sizing it — one connection per CPU core — is wrong for Valkey, and getting it wrong wastes sockets without buying throughput.

Grounding: EchoMQ.Pool (echo/apps/echo_mq/lib/echo_mq/pool.ex) · the real infra/valkey/conf/valkey.conf (io-threads 1, the single-thread comment) · the engine is Valkey 9. The same pool R6.05 introduces as the concurrency primitive — here it is sized for production.

§1

EchoMQ.Pool — N pipelined connectors

pool.ex · moduledoc + next/1

EchoMQ.Pool is a fixed pool of pipelined connectors with lock-free round-robin dispatch. Two design choices carry the production behaviour. First, each member is already a pipeline — a single EchoMQ.Connector keeps a FIFO of in-flight commands on one socket, so it does not block waiting for a reply before sending the next. Second, dispatch is lock-free — there is no checkout. next/1 advances an :atomics counter and a caller hits the next member directly. The default is size: 4.

Notes on ValkeyA single connection can hold many commands in flight at once — this is pipelining, and it is what lets one socket keep the server busy without waiting for each reply — redis.io/docs/…/pipelining.

A pool member is a pipeline, and dispatch is a lock-free atomic increment. A small pool multiplies throughput with no checkout ceremony — the question is only how many members the server can actually use.

§2

The single-threaded server

valkey.conf · io-threads 1

Here is the fact that decides how to size it. Valkey executes commands on a single thread. The real valkey.conf sets io-threads 1 and its comment reads “command execution is single-threaded.” No matter how many connections you open, the server runs one command at a time.

So a pool does not buy N× server parallelism. Opening four connections does not let the server run four commands at once — it runs them one after another on the same thread. What the pool buys is something else: it hides round-trip latency. With one connection, a caller that waits for each reply before sending the next leaves the command thread idle during every network round trip. With a pipeline, commands stack on the wire and the one command thread stays fed — it always has the next command ready when it finishes the last. The pool is several pipelines, so several callers can keep the thread fed concurrently without coordinating.

# valkey.conf — command execution is single-threaded.
# the second shared vCPU is for background work, not I/O threads.
io-threads 1

The distinction is the whole dive:

does NOT buy
N× server parallelism — the one command thread runs one command at a time regardless of pool size.
DOES buy
round-trip latency hidden — enough pipelines keep that single thread continuously fed across the network round trip.

Notes on ValkeyExtra I/O threads on a shared-CPU machine busy-wait and burn the CPU budget; the engine keeps one command thread and reserves the second vCPU for background helpers, so pool size is about feeding that one thread, not adding more — valkey.io/topics/admin.

§3

The sizing rule

depth, not cores

The rule that follows: size for pipeline depth and RTT-hiding, not for cores. Enough connectors to keep the single command thread continuously fed across the network round trip is enough. Beyond that, each extra connection adds a socket the single thread serializes anyway — more memory and more file descriptors, no more throughput. size: 4 is a small pool because a small pool is what RTT-hiding needs; over-sizing it to match core count is sizing for parallelism the server does not have.

the pattern

Size a connection pool for pipeline depth — enough to keep a single command thread fed across the round trip — not for cores the server does not use. Over-sizing adds sockets, not throughput.

→
its EchoMQ application

EchoMQ.Pool round-robins size: 4 pipelined connectors lock-free via :atomics; over a Valkey with io-threads 1, four pipelines hide RTT for the codemojex bot workers without contending for parallelism the server never had.

Pool sizing is a question about the network, not the CPU. A pool hides the round trip on a single-threaded server; size it for the depth that keeps the thread fed, and stop.

the door · the pool in the production posture

Where the pool meets the rest

The pool serves the load; the persistence policy (dive 1) keeps the data across a restart; the connector (dive 3) keeps each member reachable across a failover. Because a member of the pool is an EchoMQ.Connector, the pool inherits the connector’s reconnect discipline — a member whose socket drops fails its in-flight callers :disconnected and reconnects on its own, while the others keep serving. The EchoMQ protocol depth — lanes, leases, the Lua — is the dedicated /echomq course; this dive stays on the sizing. For where the whole pool runs against a Fly Valkey, see Codemojex on Fly.

The pool is one of three surfaces in one posture. Sized for RTT-hiding, it serves the load; as a set of connectors, it inherits the failover discipline of dive 3 for free.
