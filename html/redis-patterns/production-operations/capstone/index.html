R8.07 · Capstone — the door to the EchoMQ course

Thirty patterns, one thread.

The course capstone: nine chapters and thirty Redis patterns, each proven once in the real BCS build — EchoMQ backed by Valkey, EchoStore in front, codemojex the worked consumer — read back as one line. The thread that ran through all of it is the branded id.

Grounding: the real surfaces the course taught — the branded JOB id and the emq:{q}: keyspace (echo/apps/echo_mq), EchoMQ.Jobs.enqueue/claim, EchoMQ.Lanes.claim/3, EchoMQ.Keyspace.slot/1 (slot("123456789")==12739), AOF durability (infra/valkey/conf/valkey.conf, appendfsync everysec), EchoStore (echo/apps/echo_store), and the consumer Codemojex.Guesses.submit/3 (echo/apps/codemojex). The branded-id vectors are from echo/apps/echo_data branded_id self_check. The engine is Valkey.

A capstone is not a new pattern. It is the whole course read back as one line: thirty patterns, nine chapters, and one consistent claim — a pattern is worth learning only when you can apply it, so each one landed twice, once as the pattern and once as the real move it became.

§2

The through-line — one branded id, carried unchanged

echo_data/branded_id.ex

The course had a single thread, worth naming on its own: the branded id. One 14-character name — a 3-character uppercase namespace and 11 Base62 characters carrying a 63-bit snowflake — minted once at the consistent core and carried unchanged to the available periphery. Every chapter handled that one id under a different aspect, and it has four properties.

Notes on ValkeyThe placed property is a Valkey cluster fact: every key of a queue carries the queue's {q} hash tag, so they all hash (CRC16 of the brace bytes) to one of the 16384 cluster slots and co-locate. That is what keeps a multi-key claim or atomic Lua script legal (no CROSSSLOT) and lets EchoMQ.Keyspace.slot/1 route client-side — valkey.io/topics/cluster-spec.

That fourth property, conformant, is what makes a polyglot worker fleet possible: if every runtime agrees on the id and its placement, a Go worker and an Elixir worker can drain the same bus. The shipped bus is Elixir; the polyglot fleet is the conformance target the wire protocol and the id contract make reachable, demonstrated in the EchoMQ Proof pillar.

§3

What you can now do

the course · outcomes

The course's outcomes, stated plainly. After these thirty patterns you can:

  • Choose the right pattern and name its failure mode. Cache-aside risks staleness; at-least-once delivery risks a duplicate; a fixed-window limiter risks a burst at the boundary. Knowing the trade is the skill — the pattern is the easy half.
  • Read a real atomic Lua move and say why the multi-key change is one script. The keys share a slot (the {q} hashtag), so the change is atomic and CROSSSLOT-safe — one EVAL, not a transaction that can interleave.
  • Build a reliable, scheduled, observable, rate-limited queue. The claim-with-lease, the retry with a fencing token, the delay set, the telemetry tree, the token bucket — each a pattern you saw applied, and Codemojex.Guesses.submit/3 the worked consumer that composes them.
  • Operate the tier as a system of record. AOF durability (appendfsync everysec bounds worst-case loss to about a second), a pool sized for round-trip-hiding rather than cores, a reconnect discipline that fails in-flight callers rather than replay them, cluster colocation by hashtag, and a telemetry surface a Prometheus or OpenTelemetry exporter can read.

The skill the course built is judgement, not recall: a pattern, the failure it guards against, and the one real move it becomes — from a single enqueue to the durability dial that keeps it across a restart.
