R1.03.2 · Write-behind — dive 2

The durability window.

Between a write and its flush, the cache holds the only copy. That window is the price of the throughput — the flush interval sets its width, and durable storage sets whether a crash in it loses anything.

Grounding: echo_cache/journal.ex (the SQLite WAL, replay/2, compact/1) · content/bcs4.4.md (the measured price) — verbatim. Engine: Valkey.

§1

The unflushed window

write-behind.md.txt · The Durability Trade-off

The trade is the inverse of write-through. Write-through writes the database on the request, so the source is always current and a cache loss costs nothing. Write-behind defers that write, so between a write and the next flush the database is behind and the cache holds the only copy of those changes. If the cache is lost in that window — a crash, a failover, an eviction — the unflushed writes are gone.

The window’s width is the flush interval. With a one-second interval and a steady stream, roughly one second of writes is at risk at any moment; with a five-second interval, five seconds is. Narrowing the interval narrows the window but shrinks the batches, which is exactly the throughput the pattern was bought for. The choice is a dial, not a default.

durability window
The span from a write landing in the cache to its flush. In it, the cache holds the only copy of that write.
at-risk writes
Everything written but not yet flushed. A crash in the window loses exactly these.
flush interval
The dial. Shorter narrows the window and the loss; longer batches more and risks more.
RPO
Recovery point objective — how much recent data a system may lose on a crash. Write-behind’s RPO is bounded by the window.

A second lever shrinks the window without touching the interval: flush on shutdown. A clean stop drains the buffer first, so a planned restart loses nothing — only an unclean crash hits the window.

§2

Persistence bounds the loss

write-behind.md.txt · Mitigating Data Loss

The flush interval bounds how many writes the window holds. Durable storage bounds whether a crash actually loses them. The cache can run purely in memory, snapshot to disk on a schedule (RDB), or append every write to a log (AOF). Each mode changes what survives a process restart, and so changes the real risk of the unflushed window. Pick a persistence mode to classify what a crash loses for the same buffered writes.

Two dials bound the risk: the flush interval sizes the window, and durable storage sets whether a crash in it actually loses anything.

§3

The window on EchoCache — the lane that remembers

journal.ex:103 · :125-126 · content/bcs4.4.md

EchoCache does not put an append log on the cache. The bus stays volatile by decision; durability for the job lane’s obligations lives in a per-group SQLite WAL standing beside the bus (journal.ex:125-126 — PRAGMA journal_mode=WAL, synchronous=NORMAL). The outbox closes the window by construction: the intent is recorded before the bus hears it, and replay/2 re-enqueues every intent not yet covered by the applied memory after a bus restart. The bus already deduplicates a re-enqueued job id and newer-wins makes a re-applied version harmless, so replay is at-least-once and harmless by comparison.

The committed record prices the memory exactly. The writer’s edge costs 143 us per record-and-mark pair; the remembered-lane median is 524 us against the bare lane’s 148; and after a staged bus loss, replay re-enqueued exactly thirty uncovered intents in seq order.

on EchoCache

The intent is recorded in a SQLite WAL before the bus hears it, so replay/2 rebuilds the lane after a bus restart — 143 us per record-and-mark, 524 us end to end, the outbox the cost buys.

Notes on Valkeythe journal’s SQLite runs at synchronous=NORMAL, where the checkpoint is the only sync barrier; the trade is that a machine power loss may trim the unsynced WAL tail — every process and bus crash this part stages is fully covered. The plain-cache analogue is SET … PX with persistence configured (valkey.io/commands/set).

Write-behind is honest only when the window is named and bounded: a known interval, a known persistence story, and a change that can afford the loss.

§4

Recap — name the window, bound it

the window → the coalescing

The throughput of write-behind is paid for by a window where the cache holds the only copy of the unflushed writes. The flush interval sets the window’s width and the count at risk; durable storage — in-memory, RDB, AOF, or EchoCache’s outbox plus replay — sets whether a crash in the window loses them. Buffer only changes that can afford the loss, and the trade is sound. The next dive shows how to make a burst of writes to one key cost a single flush.
