R4 · Time, Delay & Priority — orientation dive

Score as meaning.

The sorted set is one mechanism; the score is the whole semantic axis. Score a job by a future millisecond and the set is a delayed queue; flatten the score and the mint-ordered id serves as the queue; score it by a next-run time and the set is a scheduler registry.

Grounding: echo/apps/echo_mq · EchoMQ.Jobs.enqueue_at/6 · EchoMQ.Repeat.register/6 — three real sets, emq:{q}:schedule, emq:{q}:pending, emq:{q}:repeat, each carrying one number per member.

§1

The score is a number that sorts

delayed-queue · sorted-sets

A sorted set stores members ordered by a numeric score. The textbook delayed queue uses the simplest possible score: a millisecond timestamp. The member is the job id; the score is when it should run; ZRANGEBYSCORE reads back everything due.

ZADD schedule 1706649000000 "JOB0Nb1VTbfnu4"
ZRANGEBYSCORE schedule -inf 1706648500000 LIMIT 0 10
ZREM schedule "JOB0Nb1VTbfnu4"

The member is the job id, the score is the fire-time, and the set stays ordered by fire-time with no extra bookkeeping. Nothing in Valkey is told what the score "means" — it is a number, and the set sorts by it. The meaning lives in how the writer chooses the number and how the reader interprets the order. That freedom is the lever: because the score is an arbitrary number, the member can carry meaning too. EchoMQ uses this on its pending set — it flattens every score to 0 and lets the branded job id, which sorts as its mint instant, carry the order.

The score is a number, and the set sorts by it; the meaning is the writer's choice, not the engine's.

§2

One number, three real sets

jobs.ex · repeat.ex

EchoMQ never packs two facts into one score. It keeps each reading on its own set and lets each set carry exactly one number per member.

  • Fire-time — emq:{q}:schedule. EchoMQ.Jobs.enqueue_at/6 writes the caller's absolute run-at millisecond; enqueue_in/6 computes now + delay_ms from the server clock inside the @schedule Lua. The score is the millisecond.
  • Order — emq:{q}:pending. EchoMQ.Jobs.promote/3 ZADDs a due job with score 0. Every member shares score 0, so the set sorts by the branded JOB id; the claim script's ZPOPMIN serves the oldest.
  • Next-run — emq:{q}:repeat. EchoMQ.Repeat.register/6 ZADDs a name scored by its next-run millisecond, and the @register Lua upserts — it guards on EXISTS the record hash and returns 0 without re-adding when the name is already live, so a reboot adds no duplicate.

Three sorted sets, one mechanism. The commands stay constant — ZADD to write, ZRANGEBYSCORE or ZPOPMIN to read — and the score is the whole semantic axis.

§3

The writer encodes the meaning

echo_mq/jobs.ex · @schedule

The textbook reaches the same "next job to serve" order by packing two fields into one score — a priority tier in the high 32 bits, an arrival counter in the low 32 bits (score = priority × 2³² + arrival), read by ZPOPMIN. That is a valid encoding, and it is the contrast worth knowing: it puts the order in the score. EchoMQ chose the opposite — it puts the order in the member. A score-0 pending set ordered by a mint-ordered branded id needs no arrival counter to break ties, because the id already is one.

Notes on ValkeyThe @register upsert reads EXISTS on the per-name record hash before ZADD, so a re-register is idempotent; ZADD would otherwise overwrite the score and could double-schedule a name — valkey.io/commands/zadd.

The number is the meaning, and the writer encodes it. EchoMQ's choice is to keep each number simple and let the branded id do the ordering work a packed counter would otherwise carry.
