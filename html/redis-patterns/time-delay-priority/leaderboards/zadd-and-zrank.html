ZADD and ZREVRANK

A player's score goes in with ZADD; the rank comes out with ZREVRANK. The score is the ranking key — the set orders by it, the rank is a position in that order, read on the fly.

A leaderboard rests on one structure: a sorted set whose score is the player's points. Two commands carry it. ZADD writes a member and a score, placing the member in the order. ZREVRANK reads a member's position in that order from the top, where the highest score is rank 0. No rank is stored; the rank is the position, recomputed each time it is read. Set a score on the right and watch the rank fall out of the order.

ZADD — the score is the write

A sorted set holds members, each with a numeric score, and keeps them ordered by score. The whole write is ZADD: a score and a member.

ZADD leaderboard 1500 "player:alice"

Run ZADD on a member that is already on the board and its score is overwritten, and the member is re-placed in the order. The score is the entire index — there is no separate ranking column. Reading the set high-to-low gives the leader first; reading it low-to-high gives the trailing player. A player's own score is read with ZSCORE, which is O(1).

The whole write is ZADD: a score and a member; re-running it overwrites the score and re-places the member. The score is the entire index, no separate ranking column.

ZREVRANK — the rank is a position

A rank is a player's position in the order, counted from one end. ZREVRANK counts from the top, so the leader is rank 0:

ZREVRANK leaderboard "player:bob"   # 0 if bob has the highest score

The rank is 0-based: rank 0 is first place, rank 1 is second. The set computes it by walking the order to the member, which is O(log N) — the cost does not grow with how the board is read, only with its size. Crucially, the rank is not stored anywhere. It is derived from the score order on each read, so it is always exactly the current order; there is no snapshot to fall out of date.

ZREVRANK reads a 0-based position from the top in O(log N); the rank is not stored, it is derived from the order on every read, so it is never stale.

ZRANK — the mirror from the bottom

ZRANK is the same position counted from the other end — low-to-high, so the lowest score is rank 0:

ZRANK leaderboard "player:bob"      # 0 if bob has the lowest score

For a board where a higher score is better, ZREVRANK is the rank a player wants to see; ZRANK answers the opposite question — how far up from the bottom. The two are mirrors of one position in the same order. If the board holds N players, a member's ZRANK and ZREVRANK add up to N − 1. Reading either end leaves the set unchanged.

ZRANK and ZREVRANK are mirrors of one position in the same order; on an N-player board they sum to N − 1, and reading either leaves the set unchanged.

In Portal — the percent is the score the rank would read

The pattern is a clean sorted-set leaderboard: score in, rank out. Portal does not run one. What Portal does record is a learner's progress through a lesson as a percent — Portal.Enrollment.Progress, the field percent :: 0..100, namespace PRG, a row in an in-memory store. Portal has no progress ranking and no Redis sorted set; the percent is a plain stored value.

# Portal.Enrollment.Progress — the real struct (echo/apps/portal), percent 0..100, namespace PRG
%Portal.Enrollment.Progress{id: "PRG…", enrollment_id: "ENR…", lesson_id: "LSN…", percent: 72}

# the ranking view a leaderboard would add over that percent (the applied pattern — Portal runs no sorted set)
ZADD board 72 "learner:dana"      # score = the stored percent
ZREVRANK board "learner:dana"      # where dana stands — read on the fly

Ranking learners by that percent is this dive applied to that data. ZADD board <percent> <learner> writes each learner's percent as a score; ZREVRANK board <learner> then reads where that learner stands. The percent is the score the rank would read. State it plainly: the percent exists in Portal's store today, and the sorted-set rank is the ordinary structure to add when a ranked view of that percent is wanted — the rank is never stored, only computed from the percent order on read.

Where this is heading — EchoMQ 2.0 A leaderboard is a generic sorted-set ranking, not part of the EchoMQ queue protocol. Where EchoMQ ranks with a sorted set is its priority queue (R4.03's emq:{queue}:prioritized); the settled EchoMQ 2.0 design (the protocol break on the first EMQ rung, emq.1) renames it to emq:{queue}:prioritized — emq: replacing emq:, every Lua key declared in KEYS[], meta.version bumped to echomq:2.0.0 — with the ranking math unchanged. A sorted-set ranking is a property of the score order, independent of the prefix or the version.
