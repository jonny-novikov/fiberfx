Chapter C3

F# for Elixir developers

You already think functionally. This chapter maps what you know onto a typed runtime.

If you write Elixir, F# will feel familiar long before it feels foreign: immutability by default, pattern matching, and the very same |> pipe. This chapter is the Elixir-community parallel to C2 · F# for C# developers — it takes each habit you carry from the BEAM and shows its idiomatic F# form on the CLR. The genuine shift is the type system: F# infers a static type for everything and checks your matches for exhaustiveness. Nine modules, three dives each.

The nine modules → The Elixir course

The same ideas, a typed home

Elixir and F# are both functional-first languages with immutable data, first-class functions, and pattern matching at the centre. Elixir runs on the BEAM and stays dynamically typed; F# runs on the .NET CLR and is statically typed with full inference. So most of the vocabulary transfers directly — and where it changes, the change is usually the compiler doing more for you.

Select a concept below to see it both ways: the Elixir you would write, beside its idiomatic F# form. Both snippets are real and runnable. The full module directory follows.

Coming from Elixir, the surprise is how little is new — and that the new part, static types, mostly shows up as the compiler catching what you would have caught in a test.
