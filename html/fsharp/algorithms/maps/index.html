Module C4.04

Maps, sets & hashing

A key reaches a value two ways on .NET — a hash to a bucket, or a comparison down a tree. F# defaults to the tree, and the choice is the whole module.

The Elixir course teaches this in E4.04 as one idea: a hash map looks a value up in O(1) average, and a set is the same machinery answering membership. F# re-derives it on .NET with a fork that has to be taught honestly. System.Collections.Generic.Dictionary is a mutable hash table — O(1) average, no order. F#'s Map and Set are immutable balanced search trees — O(log n), ordered by comparison, and a new value every write. Both are real and both have their place; this module builds all three across three dives. The running example is the F# EchoMQ port's queue state, which holds a Map<string, Job> of rows and a Set<string> of pending ids at once. Every snippet was produced by dotnet fsi.

One key, three structures

A keyed collection answers one of three questions: what value sits at this key (a map), is this element present (a set), and how does the key reach its place at all (hashing). On .NET the third question has two answers. A hash table runs the key through GetHashCode(), picks a bucket, and reads in O(1) average — that is Dictionary. A balanced search tree compares the key down O(log n) levels and reads in order — that is F#'s Map and Set.

The first dive states hashing and the cost model, and draws the Dictionary vs Map fork in full. The second takes the immutable map: Map.add, Map.tryFind, and the rule that every write returns a new map sharing the old one. The third takes the set: membership, idempotent dedup, the ordered minimum, and set algebra. Each closes with how the F# EchoMQ port uses the structure.

The same key, three jobs: a map resolves it to a value, a set answers whether it is present, and hashing or comparison is what gets it there. Read the cost from which road the structure takes.

The rows map and the pending set, side by side

Select a structure. The diagram marks the rows it touches, the code shows the F# (verified in dotnet fsi), and the readout names what the operation costs. The fixed data is the F# EchoMQ port's queue state: a Map of three job rows keyed by branded id, and a Set of pending ids.

The rows are a tree keyed by id, so they read in order and share on every write; the pending ids are the same tree without values, answering membership. A Dictionary would trade that order for a smaller constant.

EchoMQ in F#

One queue's state is a record. Two of its fields are this module's structures — quoted verbatim from echo_fs/EchoMQ/Store.fs:

/// One queue's state.
type QueueState =
    { Rows: Map<string, Job>
      Pending: Set<string>
      // … Schedule, Active, Dead, Paused, Completed, Failed }

Admission writes both at once — refuse a duplicate id, else add the row to the map and the id to the set (verbatim, echo_fs/EchoMQ/Store.fs):

if st.Rows.ContainsKey id then
    st, Admission.Duplicate
else
    let row = Job.create id payload
    { st with
        Rows = st.Rows.Add(id, row)
        Pending = st.Pending.Add id },
    Admission.Enqueued

The in-memory Store keeps the rows and the pending set today (echo_fs/EchoMQ/Store.fs, Job.fs — built and tested). The Valkey transport (EchoMQ.Connector) lands next; until then Connector.eval raises "EchoMQ Valkey connector: coming soon".
