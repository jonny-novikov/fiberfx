Lab: build a branded CHAMP store

The chapter built the parts in isolation. The lab wires them into one working queue you can keep.

This is the chapter's hands-on lab. The persistent Map and Set of C4.04–C4.06 become a queue's registry; the branded JOB id of C4.07–C4.08 is the key, time-ordered so it serves itself in mint order; the MailboxProcessor of C4.09 is the single writer. Assembled, they are a mini echo_fs — the F# port of the EchoMQ bus. Three dives build it: the data model the queue needs, the actor that owns it, and the assembled queue run end to end. Every F# snippet on these pages was verified with dotnet fsi.

The three dives → Back to C4 · Algorithms

What the lab assembles

Nothing in the lab is new. Each piece is an earlier lesson, named once and wired in. The registry is a persistent map keyed by branded id (C4.04–C4.06); the key is a branded JOB Snowflake that is time-ordered and dates itself (C4.07–C4.08); the writer is an F# MailboxProcessor (C4.09). The lab's job is to put them together into a queue with three operations — admit a job, claim the oldest, complete it — and to run it.

The result mirrors the bus. The echo_fs solution at the repo root is the F# port of the Elixir echo/apps/echo_mq bus; its in-memory core — a Store and a single-writer Broker — is built and tested. The lab rebuilds that core in miniature. The one part genuinely still ahead is the Valkey transport, which the port names EchoMQ.Connector and marks coming soon.

Three operations, one writer, one persistent store. That is the whole queue — and every piece is a lesson you have already met.

EchoMQ in F#

Status — echo_fs is an in-memory model. The EchoMQ protocol — the Lua transitions executing on a Valkey wire — is not implemented (there is no Lua and no Valkey client in the code); that is the coming-soon transport (EchoMQ.Connector). This block shows how the model uses the structure, not a running bus.

The lab is not a teaching sketch alongside the bus — it is the bus's core, in miniature.

runs today echo_fs/EchoMQ/Store.fs · Broker.fs · Job.fs · README.md

The echo_fs solution ships an in-memory EchoMQ.Store (a QueueState of a Rows map plus a Pending set) and a single-writer EchoMQ.Broker (a MailboxProcessor) — both built and under test. The lab rebuilds exactly this: the same two types, the same enqueue / claim / complete transitions, the same mailbox-serialized writer. By the end of ship it you have, in miniature, what echo_fs is today.

coming soon The echo_fs README states the in-memory Store "stands in for the branded CHAMP / Valkey rows behind the same enqueue/claim/complete/… surface, so the durable backing swaps in with no caller change." The Valkey transport — EchoMQ.Connector — is the one part the README marks coming soon; it is what turns the lab's in-memory queue into the production bus.

The store is a Map / Set today, not a CHAMP and not Valkey. The branded CHAMP (C4.06) and the Valkey wire are the production forms the same Store surface admits later — taught as the target, not claimed as built.
