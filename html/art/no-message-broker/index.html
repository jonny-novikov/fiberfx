A4 · Why You Don’t Need a Message Broker — The Art of BCS

A message broker is a separate process
for work a library already holds.

Part IV is the third of the four arguments. A message broker — Apache ActiveMQ, and its successor Artemis — is a separate daemon a production exchange hands its queues, topics, routing, and delivery guarantees to. A fair lane system does that same work as a library on the database the platform already runs, and the bus the exchange uses for fan-out is coherence on one connection. The broker’s two jobs become a library and a broadcast, not a process beside the application.

The senior continuation of /bcs, whose as-built journal the broker’s own would duplicate. Each figure here is verbatim from the manuscript (art4.md) or a cited source. The claim is narrow on purpose: a broker is a real, capable system, and the chapter prices it precisely — a runtime to operate, a protocol seam, a journal that duplicates the platform’s own — before declining it.

§1

State the message broker fairly

art4.md · what this chapter argues

A message broker is a real and well-made thing. Apache ActiveMQ is a standalone messaging daemon — a Java process implementing the JMS API and a range of wire protocols, including AMQP, STOMP, MQTT, and OpenWire — that carries queues and topics with delivery guarantees, routing each message through a destination. The chapter describes it as its makers do before declining it.

Its successor, Artemis, built from the HornetQ codebase donated to the project, is architecturally distinct: a non-blocking Netty I/O layer, a protocol-agnostic address model, and its own ultra-high-performance append-only journal for persistence, with the JMS layer kept separate because the core server is JMS-agnostic. The claim of this chapter is not that queues, routing, and delivery guarantees are unnecessary — an exchange needs all three — but that a separate process is not how this platform acquires them.

§2

The runtime’s answer — a library and a broadcast

art4.md · the two dives

The work a message broker does splits, like the log broker’s, into shapes the runtime already holds. The broker carries two jobs; naming where each goes keeps the refusal from leaving a hole. Each is the subject of one half of this chapter’s middle dive.

The first job

Retriable work → a fair lane system

Work that must be queued, routed, and delivered is the job of per-group lanes with at-least-once delivery, idempotent handlers, pause and resume, per-group ceilings, and observable depth — done as a library inside the database the platform already runs rather than as a separate process to deploy and tune. The exchange’s settlement, notification, and reporting work travels those lanes.

the broker’s queueing, held by a library, not a daemon

The second job

Fan-out → coherence on one connection

Fan-out of market data to many readers is not a broker’s queue at all; it is a coherence broadcast with client tracking on the same connection, carrying small claims rather than objects, each-to-all without a second daemon to run, monitor, or drill. The claim-check bus delivers the reference; the reader resolves the payload it already has.

the broker’s topics, held by a broadcast, not a daemon

Read together, the two close every job the broker did: retriable work a fair lane system carries as a library, and fan-out the claim-check bus carries as a broadcast on one connection — so neither remains for a separate process beside the application. The lanes and the bus are doored to /echomq in depth.
