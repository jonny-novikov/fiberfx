A2 · Why You Don’t Need a Coordinator — The Art of BCS

A coordinator coordinates a system
that cannot coordinate itself.

Part II is the first of the four arguments. A coordinator — Apache ZooKeeper as the canonical instance — is a separate service a distributed system runs to reach the agreements it cannot reach on its own: leader election, membership, and configuration. The BEAM already provides those agreements, so for a single-organization, soft-real-time system the separate service is what falls away. Coordination is a runtime property, not an acquired daemon.

The senior continuation of /bcs, whose audited placement hash this chapter computes against. Each figure here is verbatim from the manuscript (art2.md) or a cited source. The claim is narrow on purpose: distributed Erlang is itself coordination, and the regulated ledger’s consensus stays a database’s, outside the runtime by design.

§1

State the coordinator fairly

art2.md · what this chapter argues

A coordinator is a real and well-made thing. ZooKeeper, the canonical instance, is a centralized service that supplies the agreements a distributed system cannot reach on its own — leader election, membership, configuration, group services, and the distributed locks built on them. For more than a decade it was the service a generation of distributed systems leaned on for exactly those agreements. The claim of this chapter is not that the agreements are unnecessary; it is that the BEAM already provides them, so the separate service is what falls away.

The strongest evidence is the industry’s own movement, and it leads here because it is not the runtime’s partisans making the case. Kafka removed its dependency on ZooKeeper, folding the coordinator’s metadata and election into an internal Raft metadata quorum (KRaft), so that the only thing needed to run Kafka is Kafka. Consolidating coordination into the system rather than a separate coordinator is the BEAM’s posture, and the runtime has held it since long before the systems now converging on it existed.

§2

The runtime’s answer, in three substitutions

art2.md · the three dives

What replaces the coordinator is not one thing but three, and naming each keeps the refusal from leaving a hole. Each substitution is the subject of one dive of this chapter.

Substitution one

Membership & discovery → distributed Erlang

The runtime’s built-in model lets a program run across machines, with processes on different nodes addressing each other by the same message-passing they use locally. The service that told a cluster who its members were is the runtime’s own distribution layer.

the role a coordinator’s membership protocol filled, held by the runtime

Substitution two

Election → removed by the single-writer principle

The hardest coordinator job — electing a writer per shard — is removed rather than replaced. The single-writer principle gives each aggregate exactly one owning process, so there is no contest to resolve and nothing to elect.

a job that does not happen needs no election to settle it

Substitution three

Configuration & placement → the audited hash32

Which node owns which instrument is the canon’s audited hash — a function any holder of the branded id computes, not a routing table a coordinator hands out. It closes the seam the snowflake’s own history left when it chose worker numbers through a coordinator.

placement is computed, doored to /bcs for the audit

Read together, the three close every job the coordinator did: membership the runtime supplies, election the single-writer principle removes, placement the audited hash computes — so nothing is left for a separate service to hold.

§4

The line this chapter draws

art2.md · the line, and the named exit

The claim is narrow on purpose, and candor about its edge is part of the argument. This chapter does not say a distributed system needs no coordination at all: distributed Erlang is itself a coordination mechanism, with its own membership protocol and failure modes, and consensus still lives where it must — the regulated ledger’s transactional guarantees, drawn outside the runtime in a later chapter, remain a database’s. What the chapter retires is the separate coordinator service for a single-organization, soft-real-time system whose sharded state has one owner per shard and whose placement is computed.

And it states a price plainly rather than hide it. Under a network partition the architecture is designed for consistency in matching, so a book whose owning node is unreachable refuses rather than risk a second writer. The exit is named too: a system that needs cross-shard consensus, multi-writer agreement, or dynamic configuration distributed to many independent consumers has reached the envelope where an external coordinator earns its place.
