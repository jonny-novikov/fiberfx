A3 · No Log Broker — The Art of BCS

Why you don’t need a
log broker. Three shapes, one log.

Part III is the second of the four arguments. The previous chapter retired the coordinator; this one takes the log broker — the partitioned, replicated, retained log an exchange is conventionally built to produce its event stream to. The deeper move is to notice the broker is asked to do three jobs that are not the same shape, and that conflating them is what makes it look necessary.

The claim is narrower than the coordinator’s, because a log broker is excellent at one end of its envelope, and the course says so. State the tool fairly, name what replaces it, draw the line, and name the exit. Each figure here is verbatim from the manuscript or its cited sources. The senior continuation of /bcs.

§1

The log broker, stated fairly

art3.md · kafka.apache.org/#design

State the tool fairly first. Apache Kafka is a partitioned, replicated, retained log: append-only segments organized for linear reads, retention expressed as a per-topic policy, and consumers that own their offsets rather than a broker tracking them. It also offers keyed log compaction, which retains the latest value for each message key and deletes a key with a null-valued tombstone — turning a topic from an append-only stream of events into something closer to a table of current states. These are the features that make Kafka the right tool at the far end of its envelope, and the course does not dispute that end.

The feature

A partitioned, replicated, retained log

Append-only segments organized for linear reads, replicated for durability, with retention expressed as a per-topic policy and consumers that track their own offsets into each partition.

the shape a log broker is, at the scale it is built for

The feature

Keyed log compaction

Retain at least the latest value for each message key while discarding older values; a null-valued tombstone deletes a key. An append-only event log becomes a table of current states.

a backing store for many independent consumers

The far end

Where it is the right tool

Sustained multi-team fan-out, years-deep retention, and keyed compaction as a backing store for many independent consumers. An exchange whose log is per-instrument, with a small consumer count, is not at it.

the course names the end where it belongs

§2

Three jobs, not one shape

art3.md · the three shapes

The deeper move is to notice that a log broker is asked to do three different jobs that are not the same shape. Work that must be done once and retried is a queue. Fan-out of a volatile value to many readers is a broadcast. A durable, replayable history a consumer reads at its own pace is a log. The foundational series draws this boundary precisely and gives each shape its own primitive — the selector in the hero walks the schematic; the three cards below state each verdict. Conflating the three is what makes a single broker look necessary.

Work · once, retried

→ fair lanes

A worker takes an item, acknowledges it, and the next worker takes the next, at-least-once with observable depth. A partitioned log keeps history — a category mismatch for work that wants to be taken, done, and gone.

Fan-out · a volatile value

→ the claim-check bus

A value whose only interesting version is the latest, sent to many readers on one connection. Durable, ordered, retained segments are the wrong shape for it — a category mismatch, the second of the two.

Replay · a durable history

→ a reader-owned cursor

A history a consumer reads at its own pace, resuming from the last id it saw. A log broker fits this shape — but it is an oversized answer when the stream is per-instrument and the consumers are few.

The candor the chapter keeps

Two of the three shapes are not a log at all, so a log broker is the wrong tool by category, not by degree. The third is a log — and here the objection is size, not kind: a separate replicated cluster for a per-instrument stream with a small consumer count is more machine than the job needs. The runtime answers all three on the substrate it already runs.

§3

The exchange’s log, in the runtime’s substrate

art3.md · the durable spine

The exchange’s durable spine is then per-instrument streams on the database it already runs, with the branded id serving as the offset and retention declared as policy — the log shape, in the runtime’s own substrate rather than a separate cluster. The branded snowflake is monotonic and time-ordered, so it is the offset; a reader resumes from the last id it saw. Retention is a per-instrument policy, the same control a log broker exposes per topic, set on the substrate that already holds the rest of the system’s state.

This is the log shape served by a smaller primitive: not a denial that a log is needed, but a placement of it where the system’s state already lives. The Valkey streams beneath it are taught in depth on /redis-patterns; the stream tier of the bus on /echomq.

§4

The line, and the named exit

art3.md · the named exit

The chapter is candid about where the log broker comes back. If a real consumer crosses the named envelope, a log broker earns its place — and the discipline is to move only that tier while the lanes, the bus, the cache, and the regulated record stay where they are. The exit is named in advance rather than discovered late: a refusal that cannot say where it would be reversed is a slogan, not an argument. Mark which conditions hold to read the verdict.
