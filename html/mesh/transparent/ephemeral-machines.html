M7.2 · Transparent Infrastructure

A million ephemeral machines.

Modular work runs on short-lived machines that boot the whole application, not on a bespoke serverless rewrite. This dive reads the FLAME pattern — treat the entire app as a lambda whose parts run on temporary copies of itself — and the Fly Machine that is its grain, a microVM that launches and stops at subsecond speeds and shuts itself down when its program exits.

No measurement here: this dive reads a pattern and its grain, not a benchmark. Two primary sources carry it (McCord's FLAME pattern and the Fly Machine as the ephemeral compute unit); the pattern and the grain are cited, and how EchoMesh's worker tier would use them is named as design. The same-code property that makes a whole-app runner possible is M7.1; the placement of these machines is M7.3.

Proposed · not shipped

Running a whole app on short-lived infrastructure is the settled FLAME pattern, and the fast-launching, self-stopping ephemeral microVM is the shipped Fly Machine. EchoMesh's proposal is the use — draining its worker tier, the Go workers included, onto ephemeral machines that absorb work off the bus and vanish when it clears — separated from the shipped pattern and platform at M8.2. Magnitudes named below are qualitative, not measured.

§1

The tax FLAME removes

mesh.7.2.md · McCord, FLAME

Elastic scale has usually been bought at a steep price. The serverless and function-as-a-service path asks you to rewrite the work into a function dialect, target a bespoke runtime, and surround it with extra queues, storage, and glue code to talk back to the rest of the application — and the development, test, and continuous-integration complexity grows alongside the bill.

FLAME's premise is that none of that is necessary if the work can run on a short-lived copy of the application you already have. The tax FLAME removes is the reimplementation: instead of porting a piece of work into a foreign environment and wiring it back, you run it where it already lives, briefly, on infrastructure that appears and disappears. The compare below sets the two paths side by side.

Wrap existing code. A modular piece of work is wrapped in a function. There is no porting into a function dialect and no foreign environment — the work stays the code it already is, called where it already lives.
Run on a copy of the whole app. That function runs on a temporary copy of the entire application: a remote, ephemeral node is brought up running the same program. Because the runner is the whole app, the wrapped code can write the ledger, append to a stream, or broadcast on the bus — those capabilities are already present, with no extra plumbing.
Tear down when done. The work executes on the remote node, the result returns, and the node is torn down. There is no rewrite and no bespoke runtime — the same program runs, briefly, somewhere else, then stops. The pattern has a Fly backend but is not tied to one cloud; any provider with an API to launch an instance running the app code can host it.

§2

The FLAME pattern

mesh.7.2.md · McCord, FLAME

FLAME — Fleeting Lambda Application for Modular Execution — treats the entire application as a lambda whose modular parts execute on short-lived infrastructure. A piece of work is wrapped in a function, and that function runs on a temporary copy of the whole app: a remote, ephemeral node is brought up running the same application, the work executes there, the result returns, and the node is torn down.

Because the runner is the whole app rather than an isolated function, the wrapped code can do anything the application can — write the ledger, append to a stream, broadcast on the bus — without extra plumbing, since those capabilities are already present. There is no rewrite and no bespoke runtime, which is the same transparency the chapter has been building: the work runs as the same program, briefly, somewhere else. The pattern has a Fly backend but is not tied to one cloud; any provider with an API to launch an instance running the app code can host it.

§3

Fly Machines as the grain

mesh.7.2.md · Fly Machines

The grain FLAME launches onto is the Fly Machine, a fast-launching virtual machine built for exactly this. A Fly Machine starts and stops at subsecond speeds, on Firecracker microVMs of the kind that back large serverless platforms, and it stops automatically when the program inside it exits, so a runner that finishes its work shuts itself down. It provides ephemeral storage — a blank slate on each startup — and can be placed in any region through a simple REST API, which is what lets work run close to where it is needed.

The economic shape is scale-to-zero: nothing runs when nothing is happening, and machines appear only to absorb work. The cost belongs in view: placement is best-effort and can fail when a region is at capacity, so a system launching machines must be prepared to retry. The grain is small, fast, disposable, and placeable — which is what makes a million of them a reasonable thing to imagine.

§4

Draining the worker fleet onto ephemeral compute

mesh.7.2.md · M5 · M6.2 · proposed

EchoMesh would use this for its worker tier, and the Go workers are part of it. Work arrives on the bus of M6.2 as a claim check — the branded identity that references a payload held in a store — and rather than keep a standing pool of workers idling against the next burst, EchoMesh would launch ephemeral machines to absorb the work as it appears.

A machine comes up running the app, redeems the claim check to fetch the payload, does the work, writes its result back through the surfaces the app already has, and shuts down when it exits. The fleet's size tracks the work: many machines under a burst, none when the queue is empty. Draining the workers onto ephemeral compute is how the bursty, parallel part of the system would scale without paying for capacity it is not using, and it does so without the workers becoming a separate service with its own deployment — they are the same app, running briefly elsewhere.

§5

Why this fits the architecture

mesh.7.2.md · M5 · M7.1

The worker tier is one of the segmented surfaces of M5, and its trade is its own: the work is bursty and parallel, and it tolerates being done on infrastructure that did not exist a moment ago and will not exist a moment later. Ephemeral execution fits that trade precisely — it adds no idle cost, scales to the burst, and falls to zero between bursts.

And because the runner is the whole app, by the FLAME pattern, and the app is the same program everywhere, by M7.1, the worker is not a thing to maintain apart from the system; it is the system, instantiated for seconds where the work is. That is the chapter's transparency seen from the worker's side: the infrastructure the work runs on is summoned and dismissed by the design, not operated as a standing fleet.

§6

Real and proposed, candidly

mesh.7.2.md · the line at M8.2

The pattern and the grain are external and settled. Running a whole app on short-lived infrastructure is the FLAME pattern; the fast-launching, self-stopping ephemeral microVM is the Fly Machine. Both are shipped and cited, not invented here.
