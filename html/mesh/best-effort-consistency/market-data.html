M3.2 · Best Effort Consistency

Market data on EchoCache.

The availability-first read path in front of the consistency-first write path. A feed must always answer and may answer slightly behind, so EchoMesh serves it from a near-cache of recent values kept coherent by an invalidation broadcast — reading from local memory in the common case, and marking a value stale under partition rather than withholding it.

No measurement here: this dive reads a mechanism, not a benchmark. Two primary sources carry it (Gilbert & Lynch 2012, and a Valkey-class store's CLIENT TRACKING protocol). The mechanism is external and settled; EchoCache is named as the EchoMesh component built on it, and the feed that serves recent values from it is the proposal this dive develops. The staleness budget the trade asks for is the next chapter, M4.

Proposed · not shipped

Best-effort consistency and server-tracked client-side caching with invalidation pushes are shipped, settled mechanisms. EchoCache is EchoMesh's near-cache built on that protocol; the market-data feed that serves recent values from it and marks them stale under partition is the proposed design, with the shipped-versus-proposed line drawn in full at M8.2.

§1

The near-cache

mesh.3.2.md · Gilbert & Lynch 2012

The feed's read path is a near-cache: each reader keeps a local cache of recent values and serves a read from it directly, with no round-trip to a remote store on the common path. The authoritative value behind the cache is produced by the consistency-first book's single owner of M2 — one writer per instrument, one ordered history — so the near-cache is an availability-first read surface sitting in front of a consistency-first write surface.

The division of labor is the point: the book owner is the one place a value is decided, and the near-cache is the many places it is read, close to the readers and out of contention with the writer. A reader takes the value from local memory at memory speed, and the only question the rest of the dive answers is how that local copy is kept current.

§2

The coherence broadcast

mesh.3.2.md · Redis · CLIENT TRACKING

A near-cache is only useful if it is kept coherent with the authority, and EchoMesh does that with an invalidation broadcast rather than polling. The client-side-caching protocol works by tracking: when a connection requests a key it intends to cache, the server remembers that the connection holds it, and when that key is later modified the server sends an invalidation message for it, which tells the client to remove its now-stale local copy.

With the upgraded protocol that message arrives as an out-of-band push on the connection, so data reads and invalidations share one channel; a broadcasting mode is also available in which a client subscribes to key prefixes and receives every matching invalidation regardless of which keys it has read. EchoCache is EchoMesh's near-cache built on this protocol: the authority's writes to an instrument's value trigger invalidations to every reader caching it, and the readers flush and refetch. The cache stays coherent not by asking repeatedly whether a value changed but by being told the moment it does.

Two ways to be told. The hover panels read the two tracking modes the protocol offers — remember-per-connection against subscribe-to-a-prefix — and how the per-instrument key scopes an invalidation to one instrument's readers rather than the whole feed.

Default mode · the server remembers. The server remembers what keys each connection requested during its lifetime and sends invalidation messages only when those same keys are modified. This costs memory on the server side, but the reader is told only about keys it might actually hold.
Broadcasting mode · the reader subscribes. The server does not remember what a connection accessed; instead the reader subscribes to key prefixes such as instrument: and receives a notification every time a matching key is touched. It costs no server memory, at the price of notifications for keys the reader may not hold.
One channel or two. On the upgraded protocol, the data queries and the invalidation messages travel on the same connection as out-of-band push messages. On the older protocol the reader keeps two connections — one for data, one redirected to receive invalidations — reaching the same effect.
The key scopes the blast radius. The instrument's branded identity is what keys the cached value and the invalidation that targets it, so a write to one instrument invalidates only the readers of that instrument rather than the whole feed. The identity contract itself is the /bcs course, summarized here rather than restated.

§3

Serving the most recent value

mesh.3.2.md · Redis · client-side caching

In the common case the path is short: a read hits the local cache and returns the most recent value with no network round-trip. When the book owner writes a new value, the server pushes an invalidation to every reader holding the old one; each reader drops its cached entry, and its next read fetches the new value from the authority and caches that.

So the feed is fresh to within one invalidation — a reader is never more than the invalidation's propagation behind the authority, and usually exactly current. The instrument's branded identity is what keys the cached value and the invalidation that targets it, so a write to one instrument invalidates only the readers of that instrument rather than the whole feed. The result is a feed that reads at local-memory speed and is kept current by the authority pushing changes to it.

§4

Marked stale under partition

mesh.3.2.md · the deliberate opposite of M2.3

The availability-first choice shows when the invalidation channel breaks. If a partition cuts a reader off from the authority, the reader can no longer be told when its cached value changes, so it can no longer be sure the value is current. The consistency-first move would be to refuse — to stop serving rather than risk a stale answer — but this is an availability-first surface, so it does the opposite: it keeps answering and marks the value stale, making the uncertainty visible to the consumer rather than hiding it or withholding the value.

A consumer that can tolerate a stale quote uses it; one that cannot sees that it is stale and decides accordingly. When the connection is restored, resetting the local cache and refetching is the safe recovery, since invalidations that arrived during the gap may have been missed — the protocol guidance is plain: if the connection is lost, flush the local cache. This is the precise counterpart to the ledger of M2.3: where the ledger blocks rather than diverges, the feed diverges within a marked, bounded window rather than blocks — the same theorem, the opposite deliberate choice.

The ledger · consistency-first (M2.3)

Block rather than diverge

Under a partition the single-owner ledger stops rather than risk an answer that is not the latest. Silence, chosen deliberately, over a wrong value.

under partition: refuse to answer · cost: a withheld answer, never a wrong one

The feed · availability-first (this dive)

Diverge within a marked bound

Under a partition the feed keeps answering and marks the value stale, so the staleness is visible and the consumer chooses. Recovery is a reset and a refetch.

under partition: answer, marked stale · cost: bounded, visible staleness

The same wall, the opposite door. Both surfaces answer to the proof of M1.2: under a partition a correct register must give up consistency or availability. The ledger gives up availability; the feed gives up consistency within a marked bound. Neither defeats the theorem — each places its cost on purpose.

§5

The trade, named

mesh.3.2.md · the bound is M4

The win is a read path that answers from local memory with no round-trip and never stops, even when the authority is unreachable. The loss is the mirror image and must be named: a reader can serve a value that is behind the authority — during the brief gap between a write and its invalidation in the common case, and for the duration of a partition in the bad case.

What makes the loss bounded rather than open-ended is that the staleness is visible and capped: an invalidation closes the common-case gap promptly, and a partition's staleness is marked so a consumer can refuse it for itself. How stale a given consumer is willing to accept — the budget that turns “bounded” into a specific number per surface — is the subject of M4.
