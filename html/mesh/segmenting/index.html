M5 · Segmenting Consistency and Availability · the heart

One system, many trades.

The dominant strategy, and the heart of the course: do not pick one trade for a whole system — split it into subsystems, each with its own trade, so the whole guarantees neither strong consistency nor full availability, yet each part guarantees exactly what it should.

The three chapters before this each described one surface's trade in isolation; this chapter composes those surfaces into a single coherent system. It takes Gilbert & Lynch's five dimensions of segmentation in turn and maps EchoMesh onto each, then shows the move that makes a segmented system cohere rather than fragment: the branded identity as the seam. "Guarantees neither" is a feature, not a failure.

Proposed · not shipped

The strategy and its five dimensions are Gilbert and Lynch's; the fine-grained per-part choice is Brewer's; the time-ordered, coordination-free identity is the Snowflake lineage. What EchoMesh proposes is the mapping — which surface takes which trade — and the one branded identity that holds them together, separated from the shipped substrate at M8.2.

§1

The dominant strategy

mesh.5.md · Gilbert & Lynch 2012

When a system has no single uniform requirement — some parts need strong consistency, some need high availability — the natural response to the CAP limit is to redesign the system into components that provide different guarantees. Gilbert and Lynch present this as a third path beyond sacrificing availability or sacrificing consistency for the whole: segment the system, let each component make its own trade, and accept that from the system's outside it may look as if the architects gave up both properties.

The outcome is the opposite of giving up. A segmented system responds well to most requests even under bad network conditions and still provides strong consistency exactly where consistency is required, because each part was built for its own stakes. This is the design real systems are built with, and the culmination of the menu of M1.3: the choice the impossibility forces is not one trade imposed on everything, but the right trade chosen per part.

The first dive, M5.1, reads segmentation as the dominant strategy and argues why "guarantees neither" is a feature, not a failure of the design.

§2

The five dimensions

mesh.5.md · Gilbert & Lynch 2012

Segmentation can run along several dimensions, and Gilbert and Lynch name five. Each maps onto an EchoMesh surface, and each sits at its own point on the CAP spectrum. The precise guarantees a segmented whole provides remain application-specific — a candor this chapter keeps.

Data partitioning splits by what the data is — a cart highly available, checkout records strongly consistent. EchoMesh: the ledger and the feed. data
Operation partitioning splits by what the operation does — reads always answered, certain writes allowed to block. EchoMesh: the commit and query paths. operation
Functional partitioning splits by subservice — a lock service for coordination, a cache for naming, each with its own trade. EchoMesh: the ledger, cache, streams, and store. function
User partitioning splits by who and where — users served from a nearby data center for availability, with consistency within it. EchoMesh: regional placement. user
Hierarchical partitioning nests these — better connectivity, so easier trades toward the leaves and looser ones toward the global root. EchoMesh: per-instrument-to-global layering. hierarchy

The second dive, M5.2, takes each dimension in turn and maps it to an EchoMesh surface — the ledger and feed as a data split, the commit and query paths as an operation split, and so on.

§3

The branded identity as the seam

mesh.5.md · Snowflake · /bcs

Segmentation creates a problem in the same motion that it solves one. Once a fill is recorded in the consistency-first ledger, cached in the availability-first feed, appended to a per-instrument stream, retained in the global object store, and processed by a worker, those are five representations of one entity on five surfaces with five different trades. If each surface had its own notion of which fill it held, the system would fragment into disconnected parts that could not be reconciled.

The seam that prevents this is a single branded identity for the fill — minted once and used verbatim everywhere — whose properties are exactly what a seam needs. It is time-ordered, so the stream's order and the ledger's order agree; it is generated without coordination, so the consistent core can mint it and the available periphery can carry it without a round-trip; and it is location-independent, so any surface holds which entity it names without resolving it against a registry.

Time-ordered

Orderings agree

The timestamp in the high bits makes the id sortable, so the stream's append order and the ledger's commit order agree rather than conflict.

stream order = ledger order · by construction

Coordination-free

No round-trip to name

The core mints the id and the periphery carries it. No surface waits on a central sequence, so naming never re-introduces the coordination availability exists to avoid.

minted once at the core · the seam

Location-independent

Self-contained

The value carries which entity it names. Any surface interprets it without a registry, so a partition that severs a lookup cannot break the seam.

survives where consensus stalls · by design

Identity flows from the consistent core to the available periphery, never the reverse, so the seam holds under a partition exactly where a consensus-on-identity would stall. The identity contract itself is the /bcs course; this chapter builds its role as the seam. The third dive, M5.3, follows one fill's branded id across all five surfaces.
