M6.1 · The Stack

BEAM and the owned runtime.

EchoMesh's runtime is the BEAM, and the choice is load-bearing. A unit of work is a lightweight process that owns its own state, shares no memory, and communicates only by message passing — so the single-writer owner of M2 is the runtime's native shape, not a pattern bolted on. This dive reads what the BEAM and OTP give: coordination, an ordered mailbox, supervision, and location-transparent distribution.

No measurement here: this dive reads a runtime, not a benchmark. Two primary sources carry it (the BEAM's process model, and an account of OTP supervision and distributed Erlang's reliability model); the runtime is external and settled, and how EchoMesh uses it is named as design. Out of frame: the surfaces that run on it (M6.2, M6.3), the placement that distributes it (M7), and the language in depth, which is the /elixir course.

Proposed · not shipped

The isolated, message-passing process model is the BEAM's; OTP supervision with a global namespace and node-spanning trees, and location-transparent distribution, are the Erlang platform's — all external and settled. EchoMesh's proposal is the use: hosting the single-writer owner as a supervised process, writing coordination and ordering as language properties rather than separate services, and relying on location transparency for placement. That use is the proposal, separated from the shipped runtime at M8.2.

§1

The process owns its state

mesh.6.1.md · Erlang Solutions

The BEAM's unit of work is a lightweight process, and its defining property is isolation: each process owns its own memory, shares no state with any other, and communicates only by sending messages. This is the exact shape the consistency-first surface needs. The single-writer owner of M2 — one writer per instrument, one ordered history — is, on the BEAM, one process that holds one instrument's book in its own state and processes requests for it one at a time.

There is no shared mutable structure for several writers to contend over, because there is only one owner and its state is unreachable except through messages it handles in order. The single-writer principle, which on other runtimes is a discipline imposed against the grain of shared memory, is here the path of least resistance: owning state in a process is what the runtime is for.

§2

Coordination, log, broker, orchestrator — as language

mesh.6.1.md · Erlang Solutions

A distributed system usually assembles four things from separate infrastructure — something to coordinate access, something to keep an ordered log, something to broker messages, and something to orchestrate restarts — and the BEAM supplies all four as properties of the runtime rather than services to operate. A process that owns state and handles one message at a time is a coordination point, serializing access without a lock. A process's mailbox is an ordered queue, delivering messages in the order they arrived, which is an ordered log and a broker in one. Supervision, the subject below, is the orchestrator.

The consequence for EchoMesh is that the coordination core is written in the language rather than stood up alongside it: the owner, the ordering, and the restart logic are processes and supervisors in the same codebase, not a separate cluster to deploy and watch. Fewer moving parts to operate is itself a reliability property.

Coordinator → the owning process. A process that owns state and handles one message at a time is a coordination point: it serializes access without a lock, because only one message is processed at once and no other process can reach its state. The usual separate coordination service is, here, the shape of the process.
Ordered log → the mailbox. A process's mailbox delivers messages in the order they arrived. That ordering is an ordered log: requests are recorded and handled in arrival order, with no separate log service to operate alongside the owner.
Broker → the same mailbox. The mailbox that orders is also the queue that brokers: a message sent to a process is enqueued and delivered to it. The ordered log and the broker are one structure — the process's own inbox — not two services to deploy.
Orchestrator → supervision. Supervision is the orchestrator: a supervisor monitors a process and restarts it on a crash. The restart logic is a supervisor in the same codebase, not an external orchestration cluster to stand up and watch.

Hover or focus a service to read which runtime property supplies it; click to pin. Without scripting, all four read in sequence. The point: the coordination core is written in the language, not stood up alongside it.

§3

Supervision and the survivable owner

mesh.6.1.md · Chechina et al. 2017

A single owner has an obvious failure mode, and M2.2 named it: one process holding one book has no quorum to fail over to, so if it dies, its instrument is unserved. The BEAM's answer is supervision rather than replication. A supervisor monitors the owner and, when it crashes, replaces it with a fresh process that rebuilds its state, and a global namespace mapping process names to identifiers lets the clients that address the owner by name keep addressing it across the restart — the name outlives the process.

Supervision trees nest and can span nodes and hosts, so a supervisor can restart an owner on another node when the first is lost. This is how the single-writer design survives without becoming a quorum: it answers the failure with a fast, local restart under a stable name, not with a vote among replicas. The owner stays single — one writer, one history — and supervision makes single survivable.

§4

Distributed Erlang and location transparency

mesh.6.1.md · BEAM & Erlang platform

The BEAM extends a process's reach across machines, and it does so transparently: sending a message to a process is the same code whether the process is local or on another node in the cluster, so a process can be situated anywhere and be addressed the same way. For EchoMesh this is the property that makes placement a deployment decision rather than a code change — an owner or a surface can be situated in a particular region without rewriting how it is reached, which is exactly what the transparent infrastructure of M7 builds on.

The transparency is not free, and the cost belongs in view: every message between nodes is copied rather than shared, and the default fully-connected cluster limits how many nodes can join before the connectivity itself becomes the constraint. EchoMesh accepts those costs because what it buys — the same code addressing a process wherever it runs — is what lets one architecture span regions.

What it buys

The same code, anywhere

Sending a message is the same code whether the process is local or on another node. A process can be situated in a particular region without rewriting how it is reached — placement becomes a deployment decision, not a code change.

builds on: the transparent placement of M7

What it costs

§5

Why this runtime for this architecture

mesh.6.1.md · M5 segmentation

The segmented surfaces of M5 each need somewhere to live, and the BEAM fits all of their needs at once. A runtime that gave only one of these would force the others to be bolted on; the BEAM gives all three as native properties, which is why EchoMesh coordinates on it rather than treating coordination as external infrastructure. The owned runtime is the ground the rest of the stack stands on.

Consistency-first owner

Single, yet survivable

The consistency-first owner needs to be a single point that is nonetheless survivable. Supervision provides exactly that — a fast restart under a stable name, not a quorum.

single, kept survivable · supervision

Availability-first surfaces

Cheap, abundant concurrency

The availability-first surfaces need cheap, abundant concurrency to serve many readers. The BEAM runs an enormous number of lightweight processes at low cost.

many readers, cheaply · lightweight processes

The whole system

Span regions, one codebase

The whole system needs to span regions without forking its code. Location-transparent distribution provides that — the same code addressing a process wherever it runs.

one code, many regions · location transparency
