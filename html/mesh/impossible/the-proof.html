M1.2 · Doing the Impossible

The proof, briefly.

The argument that forces the choice, in brief but in full shape. The 2002 construction makes a correct, always-responding register return a wrong answer; a corollary shows a literal partition is not even required; and the consensus impossibility is the same wall with a steeper face.

This is the proof at orientation depth, not a formal reproduction. Three primary sources carry it (Gilbert & Lynch 2002, Fischer–Lynch–Paterson 1985, Gilbert & Lynch 2012); the constructions are external and settled, and EchoMesh's single-owning-writer response to them is named as design and built later. The menu of strategies it points to is the next dive, M1.3.

Proposed · not shipped

The impossibilities here are settled external results. EchoMesh's single-writer answer — correctness from ownership rather than a quorum vote — is the course's proposal, built on shipped technology and composed into the mesh at M2; the shipped-versus-proposed line is drawn in full at M8.2.

§1

The construction

mesh.1.2.md · Gilbert & Lynch 2002

The proof assumes, for contradiction, an algorithm that provides all three of atomicity, availability, and partition tolerance, and builds an execution it cannot survive. Take a network of at least two nodes and divide it into two non-empty groups, the first and the second; assume every message between them is lost, and let the register's initial value be the prior value.

Now run a write of a new value entirely within the first group; by the availability requirement that write must complete and acknowledge, even though nothing it sends reaches the second group. Then run a read entirely within the second group; by availability that read must return some value. The second group has received no message from the first, so its execution is indistinguishable from one in which the write never happened, and it must return the prior value — which is not the most recent write, and so is not atomic. The single assumed algorithm has produced a non-atomic response in an admissible execution, so no such algorithm exists.

That is the wall, stated once: under a partition, a correct register must sometimes refuse to answer, because answering from the wrong side returns the wrong value. The emulator above walks exactly this execution.

§2

A partition is not even required

mesh.1.2.md · Gilbert & Lynch 2002 · corollary

The construction drops messages, which in the asynchronous model is stronger than necessary, and the corollary makes the weaker version bite. Because there is no bound on message delay, the second group cannot distinguish a message that was dropped from one that is merely slow; delaying the cross-group messages produces the same indistinguishable execution as losing them. So the impossibility does not need a real partition — it needs only delay.

The corollary states the consequence precisely: no algorithm guarantees availability in all fair executions and atomic consistency in the fair executions in which no messages are lost. Take the offending execution, let the read return its stale value, and only then deliver every outstanding cross-group message; the result is a fair execution in which all messages are eventually delivered and the system is still inconsistent. The wall stands even when no message is ultimately lost.

What EchoMesh carries from the corollary: the cost of strong consistency is not confined to the rare partition. A surface that refuses to answer until it is certain pays in latency on the common, healthy network too — which is the standing reason EchoMesh's consistency-first surfaces are built to be correct cheaply rather than by waiting, the design M2 takes up.

§3

The harder sibling — consensus

mesh.1.2.md · Fischer–Lynch–Paterson 1985

The register impossibility has a steeper relative in the consensus literature, and seeing the two together is what the safety-versus-liveness frame is for. Consensus asks an asynchronous set of processes, each starting with a value, to agree on one, under agreement and validity — which are safety — and termination — which is liveness. Fischer, Lynch, and Paterson proved that no deterministic protocol guarantees all three when even one process may crash: for any purported protocol there is an execution, with no actual failure, in which it never decides.

The proof runs by a bivalency argument: there is an initial configuration whose outcome is not yet determined — a bivalent one — and an adversary that controls message scheduling can, from any bivalent configuration, always steer to another, keeping the system perpetually undecided. The engine of the argument is the same fact that drives CAP: no process can distinguish a crashed peer from a slow one, so the protocol can never safely commit to deciding. Agreeing is strictly harder than reading a register, so the consensus wall stands behind the CAP wall.

Set the two impossibilities side by side to see why one stands behind the other — the register argument needs a partition or a delay to bite, while consensus fails on scheduling alone, with no failure at all.

The register impossibility · CAP

Reading one value back

A write then a read across a split forces a stale answer or no answer. The bad case needs a partition, or in the corollary a delay, before it bites.

what fails: atomic read/write under a partition · cost: answer wrong, or never

The consensus impossibility · FLP

Agreeing on one value

Every process must agree, validly, and decide. No deterministic protocol holds all three under one crash: there is always an execution, with no actual failure, that never decides.

what fails: deterministic agreement on scheduling alone · cost: never decides

The register asks the smaller question. Return the most recent value of one shared register. The 2002 proof breaks it only with a partition, or in the corollary with delay; on a healthy, synchronous network a single copy answers correctly with no coordination.
Consensus asks the harder one. Every process, each starting with a value, must agree on one, under agreement and validity (safety) and termination (liveness). FLP: no deterministic protocol holds all three when even one process may crash — for any protocol there is an execution, with no actual failure, that never decides.
One engine drives both. No process can distinguish a crashed peer from a slow one — the same indistinguishability that lets a partitioned reader not tell dropped from delayed. The bivalency argument uses it to steer from one undecided configuration to another, perpetually.
So agreeing is strictly harder. The register can be answered without agreement; consensus cannot. Because a quorum vote is a way of solving consensus, reaching for one does not escape the trade — the consensus wall stands behind the CAP wall, which is why M2 removes the vote rather than tuning it.

§4

Quorums relocate the trade

mesh.1.2.md · proposed answer

The instinct on meeting the register impossibility is to reach for a quorum — a replicated vote that stays correct as long as a majority agrees. The consensus impossibility is the reason that instinct does not buy an exit. A quorum protocol is a way of solving consensus, and consensus is unsolvable in the asynchronous model with one crash; what real quorum systems do is lean on partial synchrony, failure detectors, or randomization to make progress in practice, accepting that in the worst-case asynchronous corner they may stall.

So a vote does not remove the trade — it relocates it: from answering wrong or not at all, to making progress unless the network is bad enough, while paying a coordination round-trip every time the network is good. EchoMesh's consistency-first surfaces decline the relocation by removing the vote: when a single writer owns an instrument, there is no agreement to reach, no quorum to assemble, and no per-request round-trip to pay; correctness comes from ownership rather than consensus.

That surface is M2, and its ledger composes several writes into one all-or-nothing transaction rather than coordinating replicas. The single-writer claim is asserted here and built there, where its own failure and recovery behavior is examined rather than assumed — carried under a Proposed · not shipped note until the substrate-versus-proposal line is drawn at M8.2.

§5

What the proof does and does not say

mesh.1.2.md · Gilbert & Lynch 2012

The result is an impossibility of a guarantee, not of working systems. It forbids exactly one region of the design space — perfect availability and consistency in the presence of partitions — and that region is small, because partitions are occasional rather than constant. Everywhere else the design has room, and the corollary's everyday cost is a price to be placed deliberately, not a verdict.

Reading the proof this way turns it from a wall to scale into a question to answer for each surface: what does this subsystem do during a partition, and how does it recover after. The menu of answers that working systems have settled on is the next dive, M1.3.
