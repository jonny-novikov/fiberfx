Lab: build a branded CHAMP store

Trace the stack

Every write passes through the same four layers, each one a piece built earlier in the chapter. The id is checked, its three-letter namespace routes it to a partition, the entry lands in that partition’s CHAMP, and the Snowflake embedded in the id is its timestamp. Pick an entity to trace.

One id carries everything the store needs: it validates itself, names its own partition, and dates itself. The four layers are the four earlier lessons, stacked into one write path.

Advanced: the assembled module

The whole store is a GenServer holding one map from namespace to that namespace’s CHAMP — %{"USR" => champ, "LSN" => champ, ...}. A write routes on the three-letter prefix and updates only that partition; because the partition is an immutable CHAMP, the update returns a new map that shares everything it did not touch, so a snapshot of the whole store is a single cheap reference. The GenServer serialises writes through its mailbox while reads run lock-free against the last published map — the split from F4.09. Nothing here is new; it is the chapter’s parts named once and wired together.

The three operations below are the public surface. put/2 validates and routes; get/1 resolves an id to its entry and its embedded timestamp; range/2 turns a time window into id bounds and scans a partition. The dives build each one in turn.

# the assembled store — a GenServer over %{namespace => CHAMP}
defmodule Portal.Store do
  use GenServer

  def start_link(_), do: GenServer.start_link(__MODULE__, %{}, name: __MODULE__)

  def put(id, value), do: GenServer.call(__MODULE__, {:put, id, value})  # write — serialised
  def get(id), do: GenServer.call(__MODULE__, {:get, id})              # resolve by id
  def range(ns, {from, to}), do: GenServer.call(__MODULE__, {:range, ns, from, to})

  def init(state), do: {:ok, state}            # state = %{ns => champ}; built by the dives
end
