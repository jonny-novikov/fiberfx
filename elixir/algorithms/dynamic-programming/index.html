Dynamic programming & advanced problems

Dynamic programming is the move you reach for when a problem breaks into overlapping subproblems: solve each one once, remember the answer, and reuse it. Two styles do this — memoisation caches a recursion top-down, tabulation fills a table bottom-up. The Portal leans on both wherever it would otherwise recompute the same thing: the depth of a prerequisite chain, the fewest modules to reach a credit target, the closest catalog entry to a misspelled search.

A bottom-up table makes the idea concrete: each cell f[i] is a stored subproblem, computed once from cells already filled to its left, then read back instead of recomputed. The figure fills a Fibonacci memo table cell by cell, drawing the two dependencies into each new entry.

The cost of recomputing

A learner works through a track of n lessons, taking one or two per session. The number of distinct pacings follows ways(n) = ways(n-1) + ways(n-2) — the same overlapping recurrence as Fibonacci. Computed naively it re-derives the lower terms over and over; computed with a cache it touches each once. Pick a length to compare the work.

The answer is the same either way; the cost is not. Naive recursion pays for every repeat of a subproblem, and the repeats grow exponentially. Solving each subproblem once turns that into one evaluation per lesson.

Advanced: when DP applies

Dynamic programming earns its keep under two conditions. The problem has optimal substructure — an optimal answer is built from optimal answers to subproblems — and those subproblems overlap, so a naive recursion solves the same ones repeatedly. Where both hold, solving each subproblem once and storing it turns exponential work into work proportional to the number of distinct subproblems. Where the subproblems do not overlap, plain divide-and-conquer is enough and a cache only adds overhead.

The two styles are the same idea from opposite ends. Memoisation keeps the natural recursion and adds a cache, so it is lazy — it only ever evaluates the subproblems the answer actually needs — at the cost of recursion depth and a map lookup per call. Tabulation rewrites the recursion as a loop that fills a table from the smallest subproblem up, so it is eager and iterative, usually with better constant factors and no deep stack, at the cost of computing every cell whether the answer needs it or not. The cache, in both, is one of the maps from earlier in this chapter.

# the same recurrence, both ways — pacings of an n-lesson track
def memo(n, cache \\ %{}) do                 # top-down: recurse + cache
  case cache do
    %{^n => v} -> {v, cache}
    _ when n <= 2 -> {n, cache}
    _ ->
      {a, c1} = memo(n - 1, cache)
      {b, c2} = memo(n - 2, c1)
      {a + b, Map.put(c2, n, a + b)}
  end
end

def tab(n) do                              # bottom-up: fill a table
  Enum.reduce(3..n//1, %{1 => 1, 2 => 2}, fn i, t ->
    Map.put(t, i, t[i - 1] + t[i - 2])
  end)[n]
end
