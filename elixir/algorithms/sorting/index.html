Sorting & searching

Sorting and searching are two halves of one bargain. Put a sequence in order once — at a cost of about n log n comparisons — and every later lookup drops from an O(n) scan to an O(log n) binary search. The two best comparison sorts, merge sort and quicksort, are the same divide-and-conquer recursion you met as a tree in F4.02, and binary search is the same halving as a balanced tree’s descent.

The running sequence is the sorted output of F4.02’s search tree, [5, 8, 10, 12, 20, 30, 42]. Three angles open the module: sorting a sequence, searching it, and the cost model that says n log n is the floor.

Order, once

An unordered sequence answers “is 20 in here?” only by checking every element. A sorted sequence answers it by halving: look at the middle, then the half that could contain the target. Select an angle to see the sort, the search, and the cost.

Sorting is work you do once so that searching is cheap forever after. The rest of the module is how each half is built — and why the price of sorting cannot drop below n log n for a comparison sort.

Advanced: sort once, search many — and the comparison floor

Sorting is an investment. If you will search a sequence once, a linear scan at O(n) beats paying n log n to sort it first. If you will search it many times, sorting once and then binary-searching at O(log n) each wins quickly — the break-even is only a handful of lookups. The same logic later picks a data structure: a map (F4.04) pays a hash up front to make every lookup O(1) on average, no ordering required.

There is a hard limit on the sorting half. Any sort that works only by comparing pairs of elements is, in effect, walking a decision tree whose leaves are the n! possible orderings; a binary tree with that many leaves is at least log₂(n!) ≈ n log n deep. So no comparison sort beats O(n log n) in the worst case — merge sort meets that floor, and only sorts that exploit the keys (counting, radix) can go lower. The cost dive proves it.

# one sort, then many fast searches:
sorted = Enum.sort(data)         # O(n log n), once
Search.binary(sorted, 20)     # O(log n), each lookup

# comparison-sort floor:  log2(n!) ~ n log n  (no sort beats it)
