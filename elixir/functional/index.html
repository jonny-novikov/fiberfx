Chapter F2 · nine modules

Functional Programming

F1 showed that functions are an algebra. This chapter takes that algebra off the page and makes it the working style of Elixir: pure transformations of immutable data, composed into programs.

Every idea here has a root in F1 — substitution becomes purity, immutable binding becomes persistent data, composition becomes the pipe, and the higher-order operators become the everyday verbs of the language. The path runs from what a functional value is, through the operators that act on collections, to composing whole programs, and ends in a lab where a dataset flows through a pipeline you build.

The arc of the chapter

Nine modules in four movements. Foundations fix what a functional value is — pure and persistent. The operators are the verbs that act on collections: higher-order functions, recursion, and folds. Abstraction builds larger structures — closures, algebraic data types, and composition. The lab puts it together over a real dataset.

Select a stop on the path to see what it covers.

The chapter is a single line of reasoning: settle what a value is, learn the verbs that act on it, then compose those verbs into programs.

The shape of a functional program

Most functional programs have one shape: data enters, flows through a series of pure transformations, and a result comes out. Each stage is a function; the pipe threads them together. Toggle the stages and watch a list move through the pipeline — the destination this chapter builds toward.

F2 · Functional

A program is a pipeline of pure transformations — data |> map |> filter |> reduce.

If each stage is pure, the whole pipeline is pure: the same input always gives the same output, and you can read it one stage at a time.
