Modeling the Portal domain

With the Portal running behind a thin server, the engine needs a shape before it needs behavior. That shape is the domain model, and it is built in three layers. Each entity — a user, a lesson, an enrollment — is a plain struct with a typespec. Entities are grouped into bounded contexts — Accounts, Catalog, Learning — modules that own their data and reference one another only by branded id. And each context exposes a small public API that validates input and keeps its internals private. No Ecto, no database rows yet; the model is data and the functions that guard it.

A context, seen from the inside out: structs at the core, a boundary around them, a public API on top. Select a layer to see what it is and which dive builds it.

Anatomy of a context Three deep dives

Anatomy of a context

Take the Learning context as the example. Inside it are the structs it owns; the module itself is the boundary that holds them; and the handful of public functions on its surface are the only way another part of the system may touch it. Build the model in that order — data, boundary, interface — and each later module has an obvious home for every rule.

The domain is plain data with a guarded edge. Get the structs, the boundary, and the API right and the engine has somewhere to put every command, query, and rule that follows.

Three deep dives

In order: the structs and typespecs that give an entity its shape; the bounded contexts that group them and keep them decoupled; and the small public API that is the only way into a context.
