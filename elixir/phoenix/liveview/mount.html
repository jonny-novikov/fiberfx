mount & assigns

A LiveView begins life in mount/3, and the state it sets there lives in the socket — the process’s memory for as long as the page is open. You assign data with assign(socket, key: value), read it in the template as @key, and that is the whole state model: a map carried by a long-lived process, exactly like a GenServer’s state from F5. There is one subtlety worth internalizing early: mount/3 runs twice. The first time is a regular HTTP request that renders static HTML so the page paints instantly and is indexable; the second is over the WebSocket, when the live process actually starts and stays. connected?(socket) tells the two apart, which matters because side effects — subscribing to updates, starting a timer, hitting an external service — should happen only on the connected mount, not on every first paint. The data itself still comes from the same place as everywhere else in F6: a call to the Portal facade, which reaches the contexts. mount is where you load it into the socket.

Four parts: the socket and its assigns, the two-stage mount, a real LiveView in code, and using connected?/1 to run side effects once.

The socket and assigns

Three things make up mounting. The socket is the process and its state. assign/3 puts a value into that state. connected?/1 reports whether this is the live connection or the first HTTP paint. Select one.

An assign is not a variable that vanishes after render — it is part of the process state and is still there for the next event. That persistence is the difference between a controller and a LiveView.

The two-stage mount

The first request to a live route is ordinary HTTP: mount/3 runs, the LiveView renders static HTML, and the browser paints immediately. Then the JavaScript client opens a WebSocket and mount/3 runs a second time — now the process starts for real and holds state until the socket closes. Both renders use the same code, so what you see on first paint matches the live version.

Loading data in mount is safe to do on both passes — it is read-only. It is the effects (subscriptions, timers) you gate behind connected?/1, so they do not fire on a paint that will be thrown away.

A real LiveView

A catalog LiveView is a module with mount/3 and render/1. mount loads the courses from the facade into the socket; render returns HEEx that reads @courses — reusing the course_card component from F6.05.2. The @impl true marks the callbacks as implementing the LiveView behaviour.

defmodule PortalWeb.CatalogLive do
  use PortalWeb, :live_view

  @impl true
  def mount(_params, _session, socket) do
    {:ok, assign(socket, courses: Portal.list_courses(), query: "")}
  end

  @impl true
  def render(assigns) do
    ~H"""
    <h1>Courses</h1>
    <.course_card :for={course <- @courses} course={course} />
    """
  end
end

It reads like a GenServer with a view: mount is the initial state, render is how that state looks. No controller, no separate template file — the LiveView is both, and it stays alive to receive events.

Side effects, once

Because mount runs twice, anything with a side effect belongs behind connected?/1. Subscribing to updates is the common case — you want one subscription on the live process, not a stray one on the throwaway first paint. The read of list_courses/0 stays outside the guard because it is harmless to run on both passes.

def mount(_params, _session, socket) do
  if connected?(socket) do
    Portal.subscribe("courses")        # only on the live connection (F6.07 uses this)
  end

  {:ok, assign(socket, courses: Portal.list_courses())}
end
