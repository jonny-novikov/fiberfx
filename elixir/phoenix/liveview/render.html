render & diffs

The reason LiveView can be chatty without being slow is the same HEEx property from F6.05.1: a template compiles into static segments and dynamic holes, kept separate. On the first render the whole thing is sent once. After that, when an event changes the assigns, LiveView tracks which dynamic values changed and sends only those — the static markup is never transmitted again, and an unchanged assign is skipped even if render/1 ran. So a counter ticking from 41 to 42 sends a few bytes, not the page. This is automatic; you write render/1 as a plain function of assigns and the framework computes the minimal diff. The one thing you do choose is how to hold large or append-only collections: keeping a thousand-row list in socket assigns means the server holds it in memory and re-checks it each render, so LiveView offers streams — the rows live in the DOM, the server tracks only their ids, and you append or update without retaining the whole list. Streams are what make a live feed or the F6.09 dashboard scale to many rows and many connected clients.

Four parts: what change tracking distinguishes, how the diff is computed, a render that shows what re-sends, and streams for large collections.

Static, dynamic, diff

Change tracking rests on three ideas. The static parts are the fixed markup, sent once. The dynamic parts are the interpolated values, tracked per assign. The diff is the set of dynamic values that actually changed, and it is all that travels on an update. Select one.

This is why HEEx insists you interpolate assigns rather than compute strings: the compiler can only track a value it can see as a distinct hole. A value hidden inside a string concatenation cannot be diffed.

How the diff is computed

Picture the compiled template as a shell with numbered holes. The first render sends the shell and every hole’s value. On an event, LiveView re-runs render/1, compares each hole to its previous value, and sends only the ones that differ — by their position, not their markup. The client already has the shell, so it patches the changed holes in place.

A re-render is cheap, so you re-render freely. The cost is proportional to what changed, not to the size of the page — which is exactly inverted from a full request that re-sends everything every time.

What re-sends

This render has two dynamic holes: {@count} and the course titles. If an event changes only @count, only that hole’s value is sent; the <h1> and <ul> markup and the list of titles are not. Change @courses and the list holes update instead. You do not manage any of this — you write the function.

def render(assigns) do
  ~H"""
  <h1>Courses ({@count})</h1>
  <ul>
    <li :for={course <- @courses}>{course.title}</li>
  </ul>
  """
end
# change @count only  -> the diff carries only the new count
# change @courses     -> the diff carries the changed list holes, not the shell

Splitting a view into smaller function components (F6.05.2) sharpens tracking further: a component whose assigns did not change is skipped wholesale, so a busy page only recomputes the parts that moved.

Streams for big lists

Holding a large collection in assigns keeps it in the process’s memory and re-checks it each render. stream/3 avoids that: the rows live in the DOM, the server tracks only their ids, and stream_insert/3 appends or updates one row without retaining the list. The template marks the container phx-update="stream" and iterates @streams.courses.

def mount(_params, _session, socket) do
  {:ok, stream(socket, :courses, Portal.list_courses())}
end

def handle_event("add", %{"course" => params}, socket) do
  {:ok, course} = Portal.create_course(params)
  {:noreply, stream_insert(socket, :courses, course)}   # one row, not the whole list
end

<%!-- the rows live in the DOM, not in socket memory --%>
<ul id="courses" phx-update="stream">
  <li :for={{dom_id, course} <- @streams.courses} id={dom_id}>{course.title}</li>
</ul>
