handle_event & state

Interaction in LiveView is a message, not a request. You annotate an element with a binding — phx-click, phx-change, phx-submit — naming an event; the client sends that event over the socket; and handle_event/3 receives the name, a map of params, and the socket, transforms the assigns, and returns {:noreply, socket}. The state change re-renders the view automatically, so you never touch the DOM. This is the same callback discipline as a GenServer’s handle_cast: a message arrives, you compute new state, you return it. A search box that filters a list, a button that toggles a panel, a form that creates a record — each is one handle_event clause that calls the same context from F6.04 and, for writes, the same changeset from F6.03. Nothing about the domain changes; it is reached over a socket instead of an HTTP request, and the result is a UI that updates without a page reload and without hand-written front-end code.

Four parts: the three bindings, the event-to-state-to-diff loop, a live search box in code, and a live create form that reuses the context and changeset.

Three bindings

A binding ties a DOM event to a named server event. phx-click fires on a click; phx-change fires when an input changes (every keystroke); phx-submit fires when a form is submitted. Each sends its event name to handle_event/3. Select one.

The event is only a name and a map of params. The server decides what it means, which keeps all the behaviour in Elixir — the markup only declares that an event happens, never what it does.

The event loop

One turn of the loop: the browser sends a named event over the socket; handle_event/3 matches it, calls a context if needed, and writes new assigns; returning {:noreply, socket} re-renders; the diff travels back. The process stays put the whole time — only its assigns change.

{:noreply, socket} is the usual return: handle the event, keep the connection. {:reply, map, socket} exists for when the client’s JavaScript hook needs a direct answer, but most events only update state and re-render.

A live search box

A search field with phx-change="search" sends its value on every keystroke; handle_event("search", ...) filters through the context and re-assigns @courses. The list updates as you type, with no page reload and no front-end code — the same course_card re-renders with fewer items.

def handle_event("search", %{"q" => q}, socket) do
  {:noreply, assign(socket, courses: Portal.search_courses(q), query: q)}
end

<%!-- the search box sends "search" on every keystroke --%>
<form phx-change="search">
  <input type="text" name="q" value={@query} placeholder="Search courses" />
</form>
<.course_card :for={course <- @courses} course={course} />

The value comes back in the params map keyed by the input’s name, and value={@query} keeps the box in sync with state. Filtering lives in the context’s search_courses/1 — the LiveView only wires the event to it.

A live create form

A form with phx-submit="create" hands its params to handle_event, which calls the same Portal.create_course/1 from F6.04. On success it re-assigns the list and flashes; on a bad changeset it re-assigns to_form(changeset) so the F6.05 inputs show errors — now without a full request.

def handle_event("create", %{"course" => params}, socket) do
  case Portal.create_course(params) do
    {:ok, _course} ->
      {:noreply,
       socket
       |> put_flash(:info, "Course created")
       |> assign(courses: Portal.list_courses())}

    {:error, %Ecto.Changeset{} = changeset} ->
      {:noreply, assign(socket, form: to_form(changeset))}
  end
end
