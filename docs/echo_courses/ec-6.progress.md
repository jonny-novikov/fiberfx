## {ec-6-learnings} Learnings

### L-1 — The installed flyctl is `fly v0.4.6` (darwin) and it has NO one-command rollback: `fly releases` lists only (no `rollback` child), and `fly deploy` has `-i/--image <ref>` but NO `--image-from-release`/`--rollback`. So the grounded, present-in-this-CLI rollback is two steps: `fly releases -a jonnify --image` (read the prior release's Docker image ref) then `fly deploy -a jonnify --image <prior-ref>` (re-deploy that exact image). Other/newer flyctl builds expose a single `fly releases rollback` (or `fly apps rollback`) — the runbook FLAGS that one-command form for Operator/Apollo confirmation against their authed CLI and does NOT assert it. Also present: `fly image show -a jonnify` (current image) and `fly apps restart`. NO-INVENT lesson: a rollback runbook must be pinned to the actual installed flyctl's command surface, with the version stamped, not to a remembered subcommand.

### L-2 — CONFIRMED (Director-flagged): the cross-app canonical/og:url defect. The echo `/` render carries `<link rel="canonical" href="https://jonnify.fly.dev/courses">` and `<meta property="og:url" content="https://jonnify.fly.dev/courses">` (curled live on 127.0.0.1:1323, lines 8+11). Grounded in source: indexPath="/courses" (internal/handler/courses.go:43 — the index's stable canonical ON ECHO, since echo serves both / and /courses and the sitemap lists /courses), canonical = canBase+indexPath (courses.go:90), canBase default "https://jonnify.fly.dev" (cmd/server/main.go:37). CORRECT on echo. But the cutover (mechanism A) installs these STATIC bytes at jonnify's `/`, where there is NO /courses route (T-3: grep href="/courses" over the golden master = 0). Result = a canonical + og:url that 404 — an INVISIBLE SEO/social defect (the page renders perfectly; only crawlers/unfurls see the broken self-reference), which the status+fingerprint smoke cannot catch. Distinct from the §2.A.2 asset finding (that one is VISIBLY broken = unstyled page). Full URL inventory of the `/` render: 2 echo assets (/static/app.<hash>.{css,js}, §2.A.2 handles), href="/" + 5 deep-root card links (/elixir /redis-patterns /echomq /bcs /course/agile-agent-workflow — jonnify-served), 2 external Google Fonts (resolve fine), and the 2 broken /courses meta tags. The visible body has NO clickable /courses link (an earlier grep false-positive: href= inside the <link>/<meta> tags). BOUNDED: the echo sitemap also lists /courses (seo.go:52) but is index-only-out-of-scope (jonnify serves its own /sitemap.xml) — does NOT reach jonnify under mechanism A.

### L-3 — UN-PROMPTED finding (gate gap, my own): the smoke battery resolves the CSS fingerprint asset to 200 but NEVER resolves the JS. MODE=echo: contains / '…\.js' checks the JS LINK is present (ec.6.smoke.sh:87), then resolves only the CSS (lines 99-101); MODE=jonnify checks the CSS fingerprint present (line 109) and resolves NEITHER asset. So a mechanism-A cutover that vendors/rewrites the CSS but ships a 404 JS bundle (/static/app.<hash>.js missing on jonnify) PASSES both phases — a broken interactive page both gates bless. On echo both resolve (824ffa72.js→200, 6c6c8cbd.css→200) so it is invisible today; the gap surfaces only post-cutover when the two assets are handled asymmetrically (exactly the §2.A.2 hazard, which lists BOTH .{css,js} but the gate only verifies one resolves). Low-cost fix: in MODE=echo resolve the advertised JS path too (mirror the CSS_PATH block for JS), and in MODE=jonnify resolve BOTH the advertised CSS and JS on jonnify (not just assert-present), since post-cutover asset-serving is the whole §2.A.2 risk. ATTACK-THAT-HELD #1: tried to make mechanism A (rebuild html/index.html) shadow a deep-course root — impossible BY CONSTRUCTION: go/Dockerfile COPYs html/index.html (line 28) and each deep root as a DISJOINT tree (elixir/, html/{redis-patterns,echomq,bcs,fsharp,art,mesh,agile-agent-workflow}/, lines 37-58); replacing one COPY source cannot affect a disjoint one. The no-shadow invariant is structural, not behavioral — the runbook's claim is sound. ATTACK-THAT-HELD #2: tried to find a cutover artifact the image-restore rollback would NOT undo for mechanism A — none: html/index.html and any vendored /static asset are both image-baked (COPY), so restoring the prior image is the COMPLETE rollback unit. Runbook §3 claim correct + complete for mechanism A.

### L-4 — Mechanism-B rollback gap (un-prompted) + agile-duality RESOLVED by ground truth + rollback pre-stage sound (attack-that-held #3). (a) ROLLBACK GAP for mechanism B: §3 opens "the cutover is a normal jonnify deploy, so rollback is restoring the prior jonnify release" — TRUE for mechanism A (html/index.html + any vendored /static asset are image-baked COPYs) and for mechanism B ONLY IF the proxy/`GET /` handler is shipped as a redeploy of the jonnify binary (image-baked). If an operator instead toggles proxy-`/` via runtime config/env (not a redeploy), image-restore would NOT revert it — §3 never states mechanism B's rollback also requires the change be image-baked. Low-severity (B is the non-recommended path) but a real omission; surfaced to Director, not blocking. (b) AGILE-DUALITY RESOLVED (was THE flagged Operator-confirm item): jonnify ALREADY serves /course/agile-agent-workflow — proven by jonnify's OWN html/sitemap.xml (line 700: <loc>https://jonnify.fly.dev/course/agile-agent-workflow</loc> + ~dozens of sub-pages), AND the golden master html/index.html (today's live `/`) already links href="/course/agile-agent-workflow" — identical to the echo render's card. So the smoke's `status /course/agile-agent-workflow 200` is correct and will pass; the duality is benign. Downgrade from "confirm before cutover" to "already served; the smoke verifies survival." Lesson: a static server's committed sitemap is authoritative ground truth for its route surface when main.go is out-of-VCS. (c) ATTACK-THAT-HELD #3: tried to break the rollback by mis-capturing "prior" — the runbook §3 PRE-STAGE step (`fly releases -a jonnify --image | head -3` immediately BEFORE the flip) captures the soon-to-be-prior ref while it is unambiguously current, the correct mitigation for "pick the pre-cutover release from a list that now includes the cutover at top." Sound.
