## 1 ¬∑ Time-travel ‚Äî a mint-instant window read over `XRANGE`, the window read == the id-filtered truth (INV-TT, the bus-side half)

emq3.6 builds a bus-side read surface on `EchoMQ.Stream` that takes a `DateTime` window and returns the stream entries whose branded `EVT` mint-instant falls in it. The contract, as precondition / postcondition / invariant:

| Step | What happens | The surface (real or forward-tense) |
|---|---|---|
| **1 ¬∑ the lower bound** | A window's start `t0` ‚Üí the `XRANGE` `from` `"<ms>-0"` via the SHIPPED `minid_floor/1` (the smallest entry id at-or-after `t0` ‚Äî the half-open `[t0, ‚Ä¶)` floor `trim/4` MINID already proves). | `EchoMQ.Stream.minid_floor/1` (`stream.ex:239`, SHIPPED) ‚Äî reused, byte-frozen. |
| **2 ¬∑ the upper bound** | A window's end `t1` ‚Üí the `XRANGE` inclusive `to` `"<ms>-<maxseq>"` (the LARGEST entry id at-or-before `t1`) ‚Äî the NEW bound-math this rung adds (forward-tense; the LOWER floor exists, the UPPER does not ‚Äî `trim/4` uses only `MINID`). The inclusive/exclusive edge at the exact `ms` is the Arm-2 fork (a closed `[t0, t1]`, a half-open `[t0, t1)`, or a half-open `[t0, ‚àû)` `read_since`). | The NEW `EchoMQ.Stream.<upper-bound-fn>/1` (forward-tense ‚Äî the inverse of `minid_floor/1`; Arm 2 settles the surface). |
| **3 ¬∑ the read** | `read/6` with the computed `from`/`to` (and an optional `COUNT`), parsing `{branded, fields_map}` in mint order. | `EchoMQ.Stream.read/3..6` (`stream.ex:270`, SHIPPED) ‚Äî DELEGATED to with mint-instant bounds. ZERO new Lua. |
| **4 ¬∑ the receipt** | `{:ok, [{branded, fields_map}]}` in mint order ‚Äî the window's entries; `{:error, term}` (any connector/server fault verbatim). | The NEW `EchoMQ.Stream.<read-window-fn>` (forward-tense; Arm 2 names it). |

**INV-TT ‚Äî the window read equals the id-filtered truth.** A mint-time window read over `[t0, t1]` returns EXACTLY the entries whose branded `EVT` mint-instant (`Snowflake.to_datetime/1` of the id's snowflake) falls in `[t0, t1]` ‚Äî i.e. it EQUALS the result of reading the full stream (`read/6` with `"-"`/`"+"`) and filtering each entry by its id's mint instant. *Check (POSITIVE, never a no-op):* append N records across KNOWN, distinct mint instants (some inside the window, some outside both edges); a window read returns EXACTLY the in-window entries, in mint order; the SAME set as `Enum.filter(full_read, &(mint_instant(&1) in window))`. A window that contains ALL records (degenerate to `"-"`/`"+"`) and a window that contains NONE (empty result) are BOTH asserted ‚Äî but the load-bearing case is a window that STRADDLES the data so the bounds actually filter. A read whose window never excludes anything proves nothing about the bounds.

**INV-BOUND ‚Äî bounds via `Snowflake.min_for/1`, never a hand-rolled instant; the half-open edge exact.** The lower bound is the SHIPPED `minid_floor/1` (`Snowflake.unix_ms(min_for(dt))` ‚Üí `"<ms>-0"`); the upper bound is the new inverse (the largest id at-or-before `t1`). A raw `min_for/1` integer is NEVER handed to the wire (the F-1-class discipline ‚Äî the wire wants `ms-seq`, not a snowflake integer). *Check:* the edge at the EXACT `ms` is asserted ‚Äî a record minted at `t0` is IN a `[t0, ‚Ä¶)` window, a record minted at `t0 - 1ms` is OUT (the half-open floor `minid_floor/1` already proves on the trim path, `stream.ex:230-242`); the upper-bound edge per the Arm-2 inclusive/exclusive ruling.

## 2 ¬∑ Hydration ‚Äî fold a stream tail into an `EchoStore.Table` by newer-wins, no compactor (INV-HYDRATE, the store-side half)

emq3.6 builds a store-side path that folds a stream tail (or the merge-read deep source) into an `EchoStore.Table`, holding per key the value carried by the NEWEST mint id ‚Äî *changelog semantics without a compactor*. The contract:

| Step | What happens | The surface (real or forward-tense) |
|---|---|---|
| **1 ¬∑ read the tail** | `read/3..6` the stream's tail (or `StreamArchive.merge_read/5` the deep source per Arm 4) ‚Üí `{:ok, [{branded, fields_map}]}` in mint order. Each record carries a KEY (the entity id, in the claims-only fields) + a VALUE + the branded `EVT` mint id (the version). | `EchoMQ.Stream.read/3..6` (`stream.ex:270`, SHIPPED) / `EchoStore.StreamArchive.merge_read/5` (`stream_archive.ex:191`, SHIPPED ‚Äî Arm 4). |
| **2 ¬∑ fold by newer-wins** | For each record in mint order, write the Table row for its key with the branded `EVT` id AS the version (`put/4`); because the tail is mint-ordered and `newer?/2` compares the mint id, the LAST write per key (the newest mint id) wins. NO background compaction ‚Äî the fold is a one-pass read + per-key versioned write. | `EchoStore.Table.put/4` (`table.ex:97`, SHIPPED ‚Äî the versioned write) keyed on the record's entity id, versioned by the branded `EVT` id; the NEW `EchoStore.Table.<hydrate-fn>` / a NEW hydrator module wraps the fold (Arm 3 names the surface ‚Äî a one-shot fn vs a supervised process). |
| **3 ¬∑ then the fence keeps it fresh** | After the hydrate, a subsequent admission (a fresh `put/4` with a newer version, or a `:tracking` invalidation push) WINS over a stale hydrated value ‚Äî *hydrate-then-fence equals loader truth*. The hydrate seeds the Table; the staleness fence (`:tracking`) keeps it current. | `EchoStore.Table` `:tracking` lane (`table.ex:256`, SHIPPED ‚Äî the staleness fence); newer-wins admission (`put/4` / `apply_coherence/4`). |

**INV-HYDRATE ‚Äî the Table holds per key the newest-mint-id value.** Hydrating a tail of K records across D distinct keys leaves the Table holding, for each key, the value of the record with the MAXIMUM branded `EVT` mint id (newer-wins by mint order). *Check (POSITIVE):* append K records for D keys (each key receiving ‚â•2 records at distinct, KNOWN mint instants, in a SHUFFLED-but-mint-ordered tail), hydrate, and assert `Table.fetch(name, key)` for each key returns the value of its newest record ‚Äî NOT an earlier one, NOT a later phantom. A hydrate of a single record per key proves nothing about newer-wins (no key receives a second, newer write); the load-bearing case is ‚â•2 records per key so the comparison actually fires.

**INV-NOCOMPACTOR ‚Äî hydration reads + folds; no background compaction; payloads claims-only.** The hydrate is a one-pass tail read + per-key versioned write; there is NO compactor process rewriting the log, NO log-as-database (the architecture keeps its databases ‚Äî Tables, the journal, Postgres ‚Äî BESIDE the log, [`../../emq.streams.md`](../../emq.streams.md) ¬ßThe needs). *Check:* the hydrate touches only the read path (`read/6` / `merge_read/5`) + the Table write path (`put/4`); a `grep` over the rung diff shows NO new compaction loop, NO `XADD`/`XTRIM` of the source stream (hydration READS, never mutates the source), and the payloads stay claims-only flat string pairs. The source stream is read-only to the hydrator.

## 3 ¬∑ The fence composition ‚Äî hydrate-then-fence equals loader truth (INV-FENCE)

The hydrate SEEDS the Table from the stream tail; the `:tracking` staleness fence (emq3.5/BCS 4.2 substrate) keeps it CURRENT thereafter. The composition is the tier's *"hydrate-then-fence equals loader truth"* ([`../../emq.streams.md`](../../emq.streams.md):73):

1. **Hydrate seeds.** The fold writes each key's newest-mint-id value into the Table (¬ß2). At this point the Table is a CHANGELOG snapshot of the stream tail ‚Äî every key at its latest stream value.
2. **The fence keeps fresh.** A subsequent write to a key ‚Äî through `put/4` with a newer branded version, or a `:tracking` invalidation push from a Valkey write to the matching `ecc:{table}:` key ‚Äî admits the newer value and the stale hydrated one loses (newer-wins / the L1 drop). The fence is the staleness mechanism `EchoStore.Table` already arms (`arm_tracking/2` `table.ex:564`); hydration does not add a new fence ‚Äî it SEEDS the table the fence then guards.
3. **Loader truth.** *Hydrate-then-fence equals loader truth* ‚Äî the Table after a hydrate + the fence keeping it fresh holds the SAME per-key latest value a fresh loader (a `fetch/3` miss-fill from L2/the source) would compute. The hydrate is a warm-start; the steady-state truth is the fence's.

**INV-FENCE ‚Äî a post-hydrate admission through the fence wins over a stale hydrated value.** After a hydrate, writing a NEWER version for a hydrated key (a fresh `put/4`, OR a `:tracking` invalidation that drops the stale L1 row so the next `fetch` reloads) leaves the Table holding the NEWER value, never the hydrated one. *Check (POSITIVE):* hydrate a key to value V1 (mint id m1); admit V2 (mint id m2 > m1) via the fence path (`put/4` or an `apply_coherence/4` drop + reload); `fetch` returns V2. A STALE admission (V0 with mint id m0 < m1) loses ‚Äî `fetch` still returns V1 (newer-wins refuses the older). The fence is exercised, not asserted.

**The R-1/L-6 two-phase-write hazard (flagged ‚Äî an INV iff Arm 3 rules a supervised hydrator).** IFF the hydrator is a SUPERVISED process that advances a CURSOR (a "hydrated up to record X" frontier) AFTER the Table write, the two-phase-write atomicity re-fires (the emq3.5 R-1 class ‚Äî `:arc_seq` advanced ONLY on a clean commit, one CubDB txn, so a failed write never leaves the cursor ahead). The discipline: the Table write must precede the cursor advance, AND a replay (a re-read of an already-hydrated record) must be HARMLESS ‚Äî newer-wins makes it so (a re-applied tail record with an equal-or-older mint id loses, mirroring `journal.ex:14`'s *"newer-wins makes a re-applied version harmless"*). A ONE-SHOT hydrate (Arm 3 Arm-A) sidesteps the hazard entirely (no standing cursor ‚Äî the caller folds a bounded tail once). A SUPERVISED hydrator (Arm 3 Arm-B) MUST carry INV-CURSOR (below). The Director's verify reads the cursor-advance branch IFF Arm 3 rules the supervised shape.

**INV-CURSOR (conditional ‚Äî binds ONLY if Arm 3 rules a supervised hydrator).** A supervised hydrator advances its cursor ONLY after the Table write for that record succeeds; a crash between the write and the advance re-hydrates the record on restart, and newer-wins makes the re-hydrate harmless (the record's value is unchanged, its mint id equal, so the re-write is a no-op-by-comparison). *Check:* fold a tail, crash before the cursor advance, restart ‚Äî the Table holds the same per-key latest value (no loss, no double-count, no phantom). Idempotent by the newer-wins comparison. **If Arm 3 rules the ONE-SHOT shape, INV-CURSOR is VACUOUS and dropped (no cursor exists).**
