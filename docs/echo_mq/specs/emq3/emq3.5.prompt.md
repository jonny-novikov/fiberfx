## The rung in one paragraph

emq3.5 builds **the archive** — the tier's stated "deep history without resident memory" need. A store-side fold consumer (the `EchoStore.Graft.Committer` shape) folds trimmed stream segments into the native `EchoStore.Graft` engine's CubDB at a reserved high page range (`@archive_base`, DISJOINT from the outbox's `@obx_base`), the engine streaming those pages to Tigris natively (one knob `remote_cfg`, not a sidecar). Deep reads merge the archive (in the engine) with the live tail (on the wire), split on an engine-derived watermark `W` = the branded `EVT` id of the highest-folded record (NOT the integer `head_lsn` — F-1). Box-loss restore is the engine's lazy `segments/{SEG}` fetch from Tigris. **The load-bearing invariant is fold-BEFORE-trim** (INV1): because `XTRIM` returns only a removed-count and never the deleted entries (F-2), the consumer must read the about-to-expire slice, commit it, advance `W`, and ONLY THEN trim — the consumer OWNS both calls, the ordering enforced in ONE store-side process, never split across a bus event the store cannot observe (no injected bus callback, EBV3-3). The net-new code lands STORE-SIDE (`echo_store`); the bus-side stream surface is byte-frozen (READ for the slice + the trim); at most an OPTIONAL `:archived` cache key (Arm 4) lands bus-side. The native engine is COEXIST-canonical and UNTOUCHED (the fold targets the PUBLIC `commit/3`, EBV3-2/9). All under the v2 master invariant (binding any bus-side surface UNCHANGED — braced keys, branded ids the writer minted, declared keys [vacuous — no new bus script], the server clock [no bus lease — the fold ordering is store-side], inline `Script.new/2` [none added], `@wire_version` frozen `echomq:2.4.2`).

## Risk tier — HIGH (a new cross-app surface + a new supervised fold-consumer process + a durable at-rest engine write + the FIRST stream rung to reach store-side)

emq3.5 crosses THREE HIGH-risk triggers — the gate is shaped to ALL THREE (not emq3.4's single destructive-op trigger):

- **A durable AT-REST engine write** — the fold commits records into the native engine's CubDB at `@archive_base` (and streams them to Tigris). The hazard is **data LOSS** (a record trimmed before it is folded is irrecoverable — F-2) and **page COLLISION** (`@archive_base` overlapping `@obx_base` corrupts an outbox intent). So the gate is the **fold-then-trim mutation battery** (REORDER the steps [trim-before-fold]; COLLIDE `@archive_base` with `@obx_base`; mutate the `W` split [a branded id → the integer `head_lsn`] — each mutation a CAUGHT loss/collision) + the POSITIVE INV1/INV3/INV4 conformance proofs (the store-side suite proves no-loss + no-gap/no-overlap + disjointness, never a no-op). **The no-loss gate is the load-bearing one this rung.**
- **A NEW supervised process** (Arm 1 — the fold consumer, supervised or host-wired) **+ id-minting test setup** — the suite SETUP mints `EVT` ids (to build a slice), so the same-millisecond branded-id mint hazard IS live. So the **≥100 determinism loop IS RUN** over the store-side fold suite (the program law's process + at-rest-write + id-minting default — NOT a multi-seed sweep; running only a sweep would UNDER-test a rung that mints ids in setup and writes at rest). The loop OWNS the machine (no concurrent liveness server, no sibling heavy I/O).
- **A NEW cross-app surface** (the FIRST stream rung to reach store-side) — the diff spans `echo_mq` + `echo_store`. So the gate adds the **two-app boundary grep** (the diff is purely `{echo_mq, echo_store}`; the engine internals EMPTY-diff; the Rust `echo_graft` untouched; `mix.lock` unchanged; no third app) + BOTH app gate ladders run independently.

But the rung touches **NO frozen line** in the sense that breaks the wire — ZERO `echo_wire` edit (the bus reads ride the shipped `read/3..6`/`trim/4`; `@wire_version` frozen `echomq:2.4.2`), adds **NO new bus script** (`grep -c redis.call` on the bus-side `lib/` diff = 0; any optional `:archived` cache is a stock `SET`/`GET`), edits **NO shipped stream verb** (the byte-freeze — the archive READS/TRIMS via the existing public surface), edits **NO engine internal** (the fold targets the PUBLIC `commit/3` — the engine is a COEXIST consumer-of, EBV3-2), and adds **NO new wire class** (the closed registry `{EMQKIND, EMQSTALE}` stands).

**The gate is the fold-then-trim mutation battery + the POSITIVE INV1/INV3/INV4 proofs + the ≥100 loop on the store-side fold suite + the two-app boundary grep + a box-loss adversarial probe + Apollo mandatory** — the three HIGH triggers (at-rest write · new process · new cross-app surface) each carried by its own gate component.
