## 1 · The order theorem under a consumer group — where it holds, where it cannot (D-6, the rung's canon point)

The writer earned the order theorem BY CONSTRUCTION (emq3.2 §1): for a single writer per stream of one namespace, stream order == id sort == mint order. A consumer group adds a SECOND ordering axis the writer never had — **per-consumer delivery order under re-claim**. The body NAMES the two regimes exactly:

| | Holds (id order preserved) | Cannot hold (re-claim re-orders) |
|---|---|---|
| **The stream's own order** | `XRANGE <key> - +` and `XREADGROUP GROUP g c … >` hand NEW entries in **mint order** — the writer's theorem is untouched (the consumer reads, it does not re-append). | — |
| **A consumer's delivery order** | First delivery of each entry is in mint order (`>` reads the stream tail in id order). | A **re-claimed** entry (recovered via `XAUTOCLAIM` or a PEL-drain after another newer entry was already delivered) returns to a consumer AFTER newer entries it has already seen — **out of real-time delivery order**. |

**Why it cannot hold, stated honestly.** At-least-once with crash re-delivery means an entry that was delivered-but-not-acked (its consumer died mid-handle) is re-delivered LATER — after the consumer has moved on to newer entries. The branded id of the re-claimed entry is OLDER (lower) than entries already handled, so its *delivery* arrives out of *mint* order. This is the irreducible cost of at-least-once: the only way to preserve real-time delivery order under crash would be to block the whole group on every un-acked entry (exactly-once-style head-of-line blocking), which the tier explicitly does NOT claim (`emq.streams.md:89`: *"Exactly-once — not claimed; at-least-once with idempotent handlers is the gated posture"*).

**The consequence for the handler contract.** Because a re-claimed entry can arrive out of delivery order, the handler MUST be **idempotent** (the tier's stated contract, `emq.streams.md:32`/`:89`): handling the same entry twice (the second time on re-claim) must be safe, and handling an older entry after a newer one must not corrupt state keyed by the branded id (the id IS the dedup key — the BCS newer-wins discipline). The `.stories.md` states this as the consumer's standing precondition. **A spec asserting "order preserved" without this PEL exception is the false-green this rung exists to retire (D-6).**

## 2 · The recovery design — two complementary mechanisms, both named (D-4)

emq3.3 recovers un-acked work two ways, each with a NAMED job — naming only one under-specifies recovery:

- **PEL-drain-on-(re)start recovers SELF.** On every start the consumer reads its own PEL with `XREADGROUP GROUP g <self> … 0` (the un-acked backlog keyed to its OWN consumer name) to exhaustion, settles each, THEN switches to `>`. A crashed consumer that restarts with the same name recovers its own held work **the instant it restarts** — not "eventually, after min-idle + a beat." A clean cold start has an empty PEL → `0` returns nothing → straight to `>` (one code path covers both). This makes the at-least-once guarantee **structural at the consumer's own boundary** — a reviewer points at one line and says "this is why no entry the consumer held is lost."
- **The `XAUTOCLAIM` beat recovers dead PEERS.** On each beat the consumer reclaims entries idle past the min-idle threshold via `XAUTOCLAIM <key> <group> <self> <min-idle-ms> <start-cursor>` — the entries held in the PEL by OTHER consumers that died and NEVER restarted (so their PEL is never self-drained). The min-idle threshold is the single tunable for "how long before a dead peer's work is re-delivered." This is the fold of S-2 (crash re-delivery into the consumer's own beat — no separate sweep module; the `EchoMQ.Stalled` separate-sweep precedent is weighed-against and NOT taken).

The two are **complementary, not redundant**: PEL-drain handles a consumer that DID restart (its own backlog); the beat handles a consumer that did NOT (a peer's orphaned backlog). The spec names both, or recovery is under-specified.

## Invariants (the runnable checks emq3.3 carries)

- **EMQ3.3-INV1 — at-least-once grouped delivery (every entry delivered ≥ 1, crash → re-delivery).** A `StreamConsumer` reading a group delivers every appended entry to a handler AT LEAST ONCE; an entry whose consumer dies mid-handle (un-acked) is RE-DELIVERED — by the consumer's own PEL-drain on restart (SELF) OR by the `XAUTOCLAIM` beat (a dead PEER). *Check:* the `stream_group` conformance scenario POSITIVELY proves re-delivery (append → group read → `XACK` one, LEAVE one un-acked → an idle-window/forced `XAUTOCLAIM` → assert the SAME entry returns to a consumer); the deep integration proof appends N entries, crash-injects at deterministic points (a handler that raises / a process kill between read and ack), and asserts every entry was delivered ≥ 1 time (no entry lost). A scenario that `XACK`s every entry and never exercises a re-delivery is a no-op — a LOUD failure (the false-green guard).
- **EMQ3.3-INV2 — drain-PEL-first then `>` (recovery is structural at the consumer's own boundary).** On (re)start the consumer drains its OWN PEL (`XREADGROUP GROUP g <self> … 0`) to exhaustion BEFORE reading new entries (`>`); a clean cold start (empty PEL) passes straight to `>`. *Check:* a consumer is killed holding un-acked entries, restarts with the same name, and the FIRST entries it handles are its OWN prior un-acked backlog (asserted by the entries' branded ids — they are the killed-but-not-acked ones, recovered before any new tail entry); the cold-start path reads `>` with no PEL round-trip wasted (the `0` read returns empty). The PEL-drain and the `XAUTOCLAIM` beat are BOTH present (the recovery design names two complementary mechanisms — §2).

## Closed error set (the typed surfaces emq3.3 may meet — grounded, NO new typed wire result)

- **a `:ok` / `{:error, reason}` handler verdict** — the per-entry verdict the handler returns; `:ok` → `XACK`, `{:error, reason}` → LEFT un-acked (re-deliverable). A raising handler converts to `{:error, reason}` and the loop survives (the `consumer.ex:148-153` rescue/catch). This is the consumer's contract surface, NOT a new wire class.
- **`BUSYGROUP` (swallowed) vs any other `XGROUP CREATE` error (LOUD)** — `BUSYGROUP` (the group already exists) is the ONE swallowed reply (idempotent start); a `WRONGTYPE` (a non-stream key collision) or any other error is surfaced LOUD (the consumer fails to start — the gate-liveness discipline, INV4). emq3.3 maps `BUSYGROUP` to a no-op start; it adds no typed refusal.
- **`{:error_reply, msg}`** — any other server-side group error (e.g. `NOGROUP` against a missing group — impossible under lazy-ensure, but surfaced if it ever arrives) arrives as a RESP error VALUE, surfaced verbatim; the caller decides severity (the shipped `resp.ex` convention).
- **`{:error, :overloaded}` / `:disconnected` / `:closed`** — the connector's shipped backpressure + socket-loss discipline (`connector.ex` moduledoc), inherited unchanged by the consumer's blocking `XREADGROUP … BLOCK` on its own lane; the loop's lane dying takes the loop with it for the supervisor to restart (the `consumer.ex` discipline).
- **An ill-formed queue/stream name** — `Keyspace.queue_key/2` builds the braced key; an ill-formed queue raises at the key builder (the shipped keyspace gate, wellformedness only). The stream key inherits this gate.

There is **NO new `EMQ*` wire class** — `BUSYGROUP` is a host-side swallow of a server reply, and the handler verdict is a host-side contract (no script to issue a wire class). The closed registry stands `{EMQKIND, EMQSTALE}` (`emq.design.md` §the closed wire-class registry).
