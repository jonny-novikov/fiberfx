### T-3 — The cross-queue ADR context + the INV7 honesty statement (Stage 1, authored solo)

THE SLOT-BOUNDARY CONSTRAINT (the forcing function).
Under the v2 braced keyspace (S-1; design.md:298-324 §6), every key of queue `q` carries the hashtag `q` between the first `{` and `}` — `emq:{q}:...`. Redis Cluster routes a key to its slot by that hashtag, so EVERY key of one queue answers ONE slot, by grammar (design.md:46, the co-location law). The reserve `{emq}:...` is first-byte-disjoint (design.md:315). Consequence: a parent in queue P and a child in queue C have keys on DIFFERENT slots (`slot({P}) ≠ slot({C})` in general). A Lua script may touch keys of ONE slot only (the cross-slot-error rule). Therefore **no single Lua script can atomically (a) ZREM the child from `emq:{C}:active` AND (b) DECR `emq:{P}:job:<parent>:dependencies`** — the single-queue fan-in (`@complete` jobs.ex:181-188) is impossible cross-slot. This is the WHOLE reason emq.3.3 exists as its own rung (emq.3.md:243-249 INV7; Fork A RULED, emq.3.md:259-263).

THE MECHANISM FAMILY (Fork A RULED — the completion-signal hop).
The cross-slot decrement is delivered by a SECOND mechanism in the v2 grain: work crosses a boundary by a SWEEP, never a cross-slot transaction (the `Pump.sweep`/`promote` precedent, pump.ex:91). A cross-queue child's completion EMITS a durable signal on a slot a sweep can reach; a per-queue sweep CONSUMES the signal and applies the decrement on the parent's slot. The four forks shape this mechanism: (1) the signal-key shape, (2) the sweep cadence/trigger, (3) the crash-recovery/idempotency model, (4) the shipped-script touch.

THE INV7 HONESTY STATEMENT (stated EXPLICITLY, never softened).
The cross-queue fan-in is **EVENTUALLY-CONSISTENT, NOT atomic across queues** (INV7, emq.3.md:243-249). Concretely:
- A cross-queue child completing does NOT synchronously release its parent. The parent's `:dependencies` is decremented, and at-zero the parent is released to `pending`, ONLY on the NEXT sweep tick of the consuming queue (latency bounded by `:tick_ms`, default 1000ms — pump.ex:44). Never "immediately", never "on completion".
- The single-queue carve (emq.3.1/3.2) REMAINS fully atomic (one slot, one EVAL) — emq.3.3 does not regress it. The honesty bound is scoped to the cross-queue path: emq.3.3's contract states its consistency model explicitly; no page, story, or doc may claim "atomic across queues".
- Delivery is AT-LEAST-ONCE (a signal may be re-delivered after a sweep crash) made EFFECTIVELY-ONCE by an idempotency guard on the parent's slot (Fork 3). The parent is released EXACTLY once; a re-delivered completion is a no-op.

THE GRAMMAR CONSEQUENCE (a wire-shaping fact the ADR must carry).
The §6 grammar's `job:<id>:<sub>` set is the CLOSED `{lock, logs, dependencies, processed, failed, unsuccessful}` (design.md:307-308) and the `suffix := type` registry is CLOSED. ANY new signal-key (Fork 1) is EITHER a new per-queue `type` registry member (e.g. `emq:{q}:flow:outbox`) OR a new `job:<id>:<sub>` member — and EITHER requires an ADDITIVE registration to the §6 closed registry, exactly as `pending`/`schedule`/`dead` joined at emq.2 without touching the grammar SHAPE (design.md:323-324). This is an additive minor (S-3), registered with its conformance probe in the same change. The ADR names the new registry member; the build adds it to `EchoMQ.Keyspace`'s registry + grammar.

THE A-1/SLOT-SOUNDNESS OBLIGATION (every candidate key).
Every key in the new emit-script and deliver-script is declared in KEYS[] or grammar-rooted from a declared KEYS[n] (S-6, the `@extend_locks` `base..'job:'..id` form ratified 2026-06-14, design.md:102-112). The emit-script's keys are all on the CHILD's slot `{C}` (the child row + the outbox); the deliver-script's keys are all on the PARENT's slot `{P}` (the parent's `:dependencies`/`:processed`/row). No script mixes slots; no key is read out of a data value in Lua (the v1 `parent_key` data-value form, flow_producer.ex:354/327, is NOT lifted — the host reads the child's `parent`/`parent_queue` fields HOST-SIDE and passes declared keys, the emq.3.1 pattern extended).

## {emq-3-3-alternatives} Alternatives

### V-1 — Fork 1: the signal-key shape (how a cross-queue child's completion becomes a parent-slot decrement)

THE QUESTION. A cross-queue child in queue C completes; the parent's `:dependencies` counter is on slot {P} ≠ slot({C}). How is the decrement carried across the slot boundary, A-1-clean and slot-sound, with emission not lost relative to completion?

ARM A — an OUTBOX on the CHILD's slot (`emq:{C}:flow:outbox`). [RECOMMENDED]
Shape: a NEW per-queue §6 `type` registry member — a HASH or LIST `emq:{C}:flow:outbox`. The cross-queue child's `@complete` (on slot {C}) writes, IN THE SAME SCRIPT as the ZREM that completes it, one outbox entry carrying `(parent_id, parent_queue, child_id, result)` — emission is ATOMIC WITH COMPLETION (one EVAL, one slot {C}). The CHILD-queue's sweep drains the outbox: for each entry it issues a SECOND, separate EVAL on the parent's slot {P} (`@flow_deliver`, keys `[emq:{P}:job:<parent>:dependencies, :processed, parent-row]`, all grammar-rooted from the parent id+queue the entry carries) that records the child in `:processed` + DECRs + at-zero releases, then removes the drained outbox entry (on {C}).
Grounding: the `Pump.sweep/1` drain grain (pump.ex:91, idempotent over its set); the `@enqueue_flow` child `parent`-field write (flows.ex:53) extended with a `parent_queue` field; the new `type` member joins the §6 registry the emq.2 way (design.md:323-324).
Steelman: emission is atomic with completion → NO signal can be lost relative to a completion (a completed child ALWAYS has an outbox entry; they are one EVAL). The sweep cost is O(completions since last tick), NOT O(N children) — it touches only what actually completed. The outbox lives on the child's own slot, so its write is local to the completing script (no cross-slot in the emit). The deliver is a clean separate EVAL on the parent's slot. This is the strict v2 grain (the durable signal + the sweep), the legacy v1 outbox/`moveToWaitingChildren` precedent rewritten A-1-clean.
Cost: (a) the child row + `@enqueue_flow` must additionally carry `parent_queue` (a new data field + the host passes it) so the outbox entry — and the deliver script — knows WHICH parent slot to target; (b) the outbox is a NEW per-queue key type (a §6 registry addition + its conformance probe); (c) it adds the outbox to the N1 subkey-lifecycle picture (its cleanup disposition must be named — but an outbox is drained-to-empty by its own sweep, so it is self-clearing in steady state, unlike `:dependencies`/`:processed`).

ARM B — a parent-side WAITING-ROSTER (`emq:{P}:flow:waiting`). [NOT recommended]
Shape: a NEW per-queue SET/ZSET `emq:{P}:flow:waiting` of parent ids awaiting cross-queue children, written by `@enqueue_flow` at add (on the parent's slot {P}). The parent-queue's sweep reads the roster; for each waiting parent it must discover which children completed by READING each child's row CROSS-SLOT (the child rows are on {C₁},{C₂},… — different slots from {P} and from each other), then reconcile `:dependencies` from that. To know which children to check, it ALSO needs a child-roster subkey on the parent (`emq:{P}:job:<parent>:children`, a new §6 subkey holding the child (id, queue) list).
Steelman: the add-side write is on the parent's slot (no child-side script edit to the emit — `@complete` stays closer to byte-frozen); the roster is the natural "which parents are waiting" introspection; it mirrors v1's parent-centric `moveToWaitingChildren`.
Cost: (a) NO atomic emission point — the sweep POLLS child state cross-slot, so it is O(N children) cross-slot reads PER waiting parent PER tick (vs Arm A's O(completions)); (b) the cross-slot reads are themselves non-atomic and racy (a child completes between the sweep's read of `:active` and its read of `:processed`); (c) it needs TWO new keys (the waiting roster + the child-roster subkey) vs Arm A's one; (d) it pulls the hard part (deciding "has this child completed?" by remote polling) into every tick forever, instead of recording it once at completion. Arm B front-loads cost onto the steady-state sweep; Arm A pays once per completion.

RECOMMENDATION: ARM A (outbox-on-child's-slot). The decisive property is EMISSION ATOMIC WITH COMPLETION: a completed child cannot fail to be signalled, because the outbox write and the ZREM are one EVAL on one slot. That collapses the entire "dropped signal" failure mode (Fork 3) to "re-delivered signal", which the idempotency guard handles cheaply. The sweep is O(completions), not O(N). I AGREE with the Director's prior here, and add the one under-priced cost: the child row + `@enqueue_flow` must carry `parent_queue` (not just `parent`), and the outbox is a §6 registry addition — both named, both additive-minor. The cost is real but bounded; Arm B's O(N)-cross-slot-poll-per-tick is strictly worse on the robustness axis the Operator's directive prioritizes.

### V-2 — Fork 2: the sweep cadence/trigger (which process drains the outbox and on what beat)

THE QUESTION. The outbox (Fork 1A) on each queue's slot must be drained and its decrements delivered. What process does the drain, and on what cadence? This sets the eventually-consistent LATENCY (INV7).

ARM A — piggyback `EchoMQ.Pump.sweep/1` (a 3rd pass). [RECOMMENDED]
Shape: extend the existing `Pump.sweep/1` (pump.ex:91) with a third pass after promote + fire_repeats: `deliver_flow_completions(conn, queue, batch)` drains `emq:{queue}:flow:outbox` (LIMIT :batch), issuing one `@flow_deliver` EVAL per entry on the entry's parent slot, returning `{:ok, %{promoted: p, fired: f, delivered: d}}`. The Pump already beats per-queue on `:tick_ms`, is `:transient`, idempotent over its sets, owner-started/opt-in.
Steelman: it INHERITS the proven cadence machinery wholesale — the :transient restart (a crash re-sweeps whole, pump.ex:31-38 child_spec), the :tick_ms beat, the :batch LIMIT, the per-queue ownership, the no-`mod:` opt-in law. Zero new process type, zero new supervision shape, zero new config. The outbox drain is idempotent over its set exactly as promote is (a re-run re-drains remaining entries, the emit-once guard makes redelivery safe). The latency is the SAME `:tick_ms` the deployment already chose for promote — one cadence knob, not two. It matches the runbook's Fork-A framing ("a per-queue promote-style sweep").
Cost: a queue that runs NO pump gets no cross-queue delivery (its outbox accumulates undrained). BUT this is correctness-SAFE: the outbox is durable (it survives on slot {C}), so when a pump IS later started on that queue, the backlog drains and every waiting parent is released — eventual consistency holds even across a pump-absent window. The operational REQUIREMENT (named, not hand-waved): the CHILD queue (the queue whose outbox holds the entry — the queue that drains its OWN outbox) must run a pump for its cross-queue children's completions to be delivered. This is the same opt-in contract scheduled/repeatable work already has (pump.ex moduledoc: "a deployment that wants scheduled and repeatable work released" runs a pump). The emq.3.3 contract states: a deployment using cross-queue flows runs a pump on every queue that hosts cross-queue children. The recovery if it doesn't: the durable outbox drains on the next pump start (no parent lost, only delayed).

ARM B — a dedicated flow sweep (a separate cadence/process). [NOT recommended]
Shape: a new `EchoMQ.FlowSweeper` GenServer (its own :tick_ms, its own supervision child) that drains outboxes independently of the Pump.
Steelman: separation of concerns — the cross-queue fan-in cadence is decoupled from the promote cadence, so a deployment can tune flow-delivery latency independently of schedule-promotion latency; a deployment that wants flows but not scheduled-work-promotion runs only the FlowSweeper.
Cost: (a) a WHOLE new process type + supervision shape + config surface + its own :transient/idempotent/batch reasoning to re-derive and re-test (the Pump already has all of this proven); (b) two cadences to operate and reason about instead of one; (c) it duplicates the Pump's machinery for marginal independent-tuning value the common deployment does not need; (d) more code on the riskiest rung. The independent-tuning benefit is real but speculative — no present requirement asks for a flow-latency knob distinct from the promote-latency knob.

RECOMMENDATION: ARM A (piggyback Pump.sweep). It inherits the proven :transient/idempotent/batch/per-queue/opt-in cadence at near-zero new surface, gives ONE latency knob (:tick_ms), and the durable outbox makes the pump-absent window correctness-safe (delayed, never lost). I AGREE with the Director. The one thing the contract MUST name (and I flag for the build runbook): the operational requirement — the queue hosting cross-queue children runs a pump — and its recovery (the durable outbox drains on the next pump start). The eventually-consistent latency is `:tick_ms` (default 1000ms), stated in INV7 and every story's observable.

### V-3 — Fork 3: the crash-recovery window (the at-least-once → effectively-once keystone — the correctness centre)

THE QUESTION. The deliver path is two steps on TWO slots: (1) drain the outbox entry on {C}, (2) DECR the parent's `:dependencies` on {P}. These cannot be one transaction (cross-slot). So a sweep can crash AFTER applying the parent DECR but BEFORE removing the outbox entry → the SAME completion is re-delivered next tick. How is delivery made idempotent so a re-delivered completion does NOT double-decrement (which would release the parent EARLY, before its other children finish)?

ARM A — the `:processed` guard (HSETNX-class check-then-act on the parent's slot). [RECOMMENDED]
Shape: `@flow_deliver` (on the parent's slot {P}) does, atomically in one EVAL: `if HSETNX KEYS[processed] child_id result == 1 then local left = DECR KEYS[dependencies]; if left <= 0 then ZADD pending 0 parent; HSET parent-row state pending end end`. The HSETNX SETS the field only if the child is NOT already recorded; the DECR fires ONLY on that first-record. A re-delivered completion finds child_id already in `:processed` (HSETNX returns 0) → NO DECR, NO double-count, NO early release. The guard is atomic on {P} (one EVAL, one slot) — the check-and-decrement cannot interleave.
Grounding: this EXACTLY mirrors the single-queue fan-in's record-then-decrement (jobs.ex:181-188: `DECR KEYS[3]; HSET KEYS[4] ARGV[1] ARGV[5]`), promoted to an HSETNX-guarded form and moved to the parent's slot. The single-queue branch already records-then-decrements; the cross-queue deliver makes the record the IDEMPOTENCY KEY. `:processed` is already the completed-children HASH (flows.ex:135 reads it via `children_values/3`), so the guard reuses an existing §6 subkey — no new key, and the guard's by-product (the recorded result) is exactly what emq.3.2's read API already serves.
Steelman: at-least-once delivery → EFFECTIVELY-ONCE fan-in, PROVABLY: the parent is released exactly once (the DECR-to-zero fires on exactly one delivery per child, the first). Atomic on one slot, so no check-then-act race. Reuses `:processed` (zero new key, and it doubles as the result store the read API already uses). O(1) per delivery. It is the single-queue mechanism's own shape, lifted across the slot with the minimal addition (HSETNX instead of HSET) — the lowest-surprise, most-auditable form.
Cost: `:processed` must hold EVERY cross-queue child (it already does for the single-queue case — it is the completed-children record), so its at-rest size is the child count (bounded, and the N1 lifecycle carry already owns its cleanup). The guard assumes child ids are unique within a flow (they are — each is a fresh branded JOB id, flows.ex gates `Keyspace.job_key/2`). No genuine downside.

ARM B — a re-derivation reconcile (recompute outstanding deps from authoritative child-row state). [NOT recommended]
Shape: the sweep does not trust the outbox count; each tick it re-reads ALL child rows of each waiting parent (cross-slot), recomputes how many have completed, and SETs `:dependencies` to the recomputed outstanding count (an absolute SET, not a DECR — so re-running is naturally idempotent).
Steelman: naturally idempotent (an absolute SET converges regardless of how many times it runs); self-healing against ANY lost/duplicated signal (it derives truth from child-row state, not from the signal); needs no HSETNX guard.
Cost: (a) O(N children) CROSS-SLOT reads per waiting parent per tick, FOREVER (vs Arm A's O(1) per delivery) — the steady-state cost Fork 1B was rejected for, re-imported here; (b) the cross-slot child-row reads are non-atomic (a child completes mid-reconcile → the recomputed count is a torn snapshot, corrected next tick but transiently wrong); (c) it needs the child-roster subkey (which children to re-read) — a new key Arm A avoids; (d) it is a fundamentally different (polling) mechanism from the single-queue fan-in, so it shares no shape with the shipped code — more to design, test, and audit on the riskiest rung.

THE "SIGNAL DROPPED ENTIRELY" ANALYSIS (the runbook's explicit ask — is a reconcile fallback needed?).
Question: what if a signal is never emitted — a crash BEFORE the in-script emit? Answer: WITH Fork 1A (outbox), this window DOES NOT EXIST. Emission is ATOMIC WITH COMPLETION — the outbox write and the ZREM-that-completes are ONE EVAL on slot {C}. So either the child completed AND has an outbox entry, or it did not complete at all (and is still claimable/in-flight, to be completed and emitted later). There is NO state where a child completed but produced no signal. Therefore the common path needs NO reconcile fallback — the durable outbox + the `:processed` guard is complete. (Contrast: a HOST-orchestrated two-call fan-in — Fork A's rejected Arm B at the family level, emq.3.md:292-302 — DOES have a drop window, because the child complete and the parent decrement are two separate host calls with a crash gap; that is precisely why the in-script atomic emit is chosen.) The ONE residual at-rest concern — an outbox entry on a queue that NEVER runs a pump — is a delivery DELAY, not a drop: the entry is durable; it drains when a pump starts (Fork 2A's named recovery). The honest bound to state: cross-queue delivery is eventually-consistent CONDITIONAL on the child queue eventually running a pump; with a pump running, latency is bounded by :tick_ms.

RECOMMENDATION: ARM A (the `:processed` HSETNX guard). It makes at-least-once delivery effectively-once with a one-EVAL atomic check-then-act on the parent's slot, mirroring the shipped single-queue fan-in's record-then-decrement, reusing the existing `:processed` subkey (which also IS the result store), at O(1) per delivery. Combined with Fork 1A's atomic emit, there is NO genuine signal-drop window — the recovery is PROVABLE, not hand-waved: every completed cross-queue child has a durable outbox entry; every delivery is idempotent; the parent is released exactly once; a pump-absent queue delays (never drops) delivery. I AGREE with the Director, and I have NAMED the drop-window analysis the directive demands: the atomic emit closes the drop window, so the `:processed` guard suffices with no reconcile fallback for correctness. (A reconcile fallback could be a defensive emq.3.4 ADD for the pathological pump-never-runs operator error, but it is NOT needed for emq.3.3's correctness — I flag it as an optional honest bound, not a requirement.)

## {emq-3-3-decisions} Decisions

### D-1 — Fork 1 (signal-key shape) RULED: the OUTBOX on the child's slot

`emq:{C}:flow:outbox` — a NEW §6 queue-level registry key on the CHILD's queue (additive-minor registration, the way pending/schedule/dead joined the grammar; design.md §6 closed-set is extended additively, the prior set byte-unchanged). The cross-queue child's @complete EMITS the completion to it ATOMICALLY with the active-ZREM (one EVAL on slot {C}) → a completed child cannot fail to be signalled → the drop window provably does NOT exist (Venus's sharpened argument). The child row + the cross-queue enqueue carry `parent_queue` (in addition to the emq.3.1 `parent` field) so the deliver rebuilds the parent key HOST-SIDE via `Keyspace.job_key(parent_queue, parent_id)`. A-1-clean: the outbox is declared in KEYS[]; the parent key is host-built + gated (the v1 data-value `parent_key` is NOT lifted). REJECTED Arm B (parent-side waiting-roster): no atomic emission point → O(N) cross-slot polling per waiting parent per tick + torn-snapshot race + two new keys. Robustness basis: emission atomic with completion is the property that makes the whole fan-in provable.

### D-2 — Fork 2 (sweep cadence) RULED: piggyback EchoMQ.Pump.sweep/1

Add a 3rd pass `deliver_flow_completions(conn, queue, batch)` to `EchoMQ.Pump.sweep/1` (pump.ex:91) after promote + fire_repeats → `{:ok, %{promoted, fired, delivered}}`. Inherits the proven :transient / idempotent-over-restart / :batch-limited / per-queue / opt-in cadence at near-zero new surface; ONE latency knob (:tick_ms). The eventually-consistent latency = :tick_ms — the parent releases on the NEXT sweep tick after the last child completes, NEVER synchronously (INV7). OPERATIONAL REQUIREMENT (stated in the contract): the queue HOSTING cross-queue children must run a pump to drain its outbox; if it does not, the durable outbox drains on the next pump start — delayed, NEVER dropped (the recovery, not a loss). REJECTED Arm B (a dedicated FlowSweeper): a whole new process + supervision + config shape duplicating the Pump's machinery for speculative independent-tuning value no requirement asks for.

### D-3 — Fork 3 (crash recovery, the keystone) RULED: the :processed HSETNX guard on the parent's slot

The deliver script `@flow_deliver` runs on the PARENT's slot (KEYS = the parent's :dependencies, :processed, row — host-built declared keys, gated): `if redis.call('HSETNX', KEYS[2]=:processed, ARGV[1]=child_id, ARGV[2]=result) == 1 then local left = redis.call('DECR', KEYS[1]=:dependencies); if left <= 0 then release the parent to its pending + HSET row state pending end end`. Atomic check-then-act in one EVAL → at-least-once delivery is EFFECTIVELY-ONCE: a re-delivered token (sweep crash AFTER the deliver, BEFORE the outbox-clear) finds child_id already a :processed field → HSETNX returns 0 → no DECR → no-op. Reuses the EXISTING :processed subkey (the emq.3.2 result store); O(1) per delivery; mirrors the shipped single-queue record-then-decrement (jobs.ex:181-188), now gated. THE DROP WINDOW IS PROVABLY ABSENT (D-1's atomic emission): every completed child has a durable outbox entry → recovery is provable, not hand-waved; the residual (a queue whose pump never runs) is a DELAY not a drop. REJECTED Arm B (re-derivation reconcile): O(N) cross-slot child-row reads per parent per tick + torn-snapshot. A defensive reconcile is an OPTIONAL emq.3.4 add for pathological operator error — an honest bound, NOT an emq.3.3 requirement.

### D-4 — Fork 4 (the shipped-script touch, HIGH-risk) RULED DELIBERATELY (per Venus's flag, not defaulted): the additive cross-queue BRANCH in @complete, single-queue branch BYTE-FROZEN
