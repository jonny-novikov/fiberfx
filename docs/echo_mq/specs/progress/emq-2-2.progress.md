## {emq-2-2-decisions} Decisions

### D-1 — module placement: NEW `EchoMQ.Admin` for the queue-scope verbs; the job-mutation verbs fold onto `EchoMQ.Jobs`

Adopts the carve (emq.2.design.md ADR-1: emq.2.2 = the operator plane, second). EMQ.2.2-D1(a) / AS-1.

**Decision.** A NEW module `EchoMQ.Admin` (echo/apps/echo_mq/lib/echo_mq/admin.ex) carries the four queue-scope verbs that operate on the queue as a whole — `pause/2`, `resume/2`, `drain/3`, `obliterate/3`. The six job-mutation verbs that extend the state machine fold onto the existing `EchoMQ.Jobs` — `update_data/4`, `update_progress/4`, `add_log/4`, `get_job_logs/3`, `remove_job/3`, `reprocess_job/3` — beside `enqueue`/`claim`/`complete`/`retry`/`promote`/`reap`/`browse`. This is the spec body's recommended placement (emq.2.2.md D1(a); llms.md Execution topology) and matches the as-built grain: `EchoMQ.Metrics` (emq.2.1) is already a cohesive read-plane module sibling to `Jobs`/`Lanes`; `Admin` is its mutation-plane twin for queue-scope ops, and the per-job transitions live with the row state machine they mutate (the same place `claim`/`complete` live). Both modules call only the existing `EchoMQ.{Connector,Keyspace,Script}` surface (and `Repeat` for the drain guard); no new transport, no new process, no facade change.

**Why not the alternatives (V-1, V-2 logged):** all-verbs-on-`Jobs` (V-1) bloats the state-machine module with queue-scope concerns that are not row transitions and reads awkwardly next to `claim`/`complete`; a single new `EchoMQ.Operator` holding all ten (V-2) splits the job-mutation verbs away from the `Jobs` state machine they extend, duplicating the `Keyspace.job_key/2` gating idiom and diverging from the `Metrics`-beside-`Jobs` precedent the read plane set. The chosen split keeps each verb where its concern lives (queue-scope → Admin; row transition → Jobs) and keeps the per-job mutations next to `get_job`/`get_job_state` callers exercise them with.

### D-2 — the queue-wide pause mechanism: a `paused` field on the `meta` HASH, gated by a SEPARATE check the public `claim/3` reads first (FORM b). `@claim` is NOT edited.

EMQ.2.2-D1(b) / AS-2 / the spec's pause-mechanism ruling (emq.2.2.md D1(b)/D2; runbook §design-make item 2).

**The flag.** A `paused` field on the per-queue `meta` HASH (`emq:{q}:meta`, HSET paused 1 / HDEL paused). `meta` is an as-built §6-registered key type (the rate scenario writes `meta.concurrency`; `Metrics.get_global_rate_limit` reads `meta.max`), so the paused flag adds NO new key type and needs NO `keyspace.ex` change (INV1: "the paused flag is a §6-registered key or a meta field … never a new type"). This re-derives v1 `pause-7`'s `meta.paused` flag honestly (v1 ALSO sets `HSET meta paused 1`), dropping v1's `wait↔paused` LIST RENAME (the bus has no LISTs — INV3).

**The gate — FORM (b), a separate gate the public claim path reads first; `@claim` is byte-UNCHANGED.** `EchoMQ.Admin.pause/2` runs one inline `@pause` script: `HSET meta paused 1`, returns ok. `resume/2` runs `@resume`: `HDEL meta paused`, returns ok. The claim path honors it: `EchoMQ.Jobs.claim/3` reads the paused flag FIRST (one `HGET meta paused` via a tiny `@claim_gate`-style guard, OR a guarded wrapper that short-circuits to `:empty` when paused) and only evals the existing `@claim` when not paused — so a paused queue claims `:empty` even with a non-empty pending, and the pending set is unmutated (emq.2.1's `get_counts` reads the same pending depth before/after — AS-2). `Lanes.claim/3` honors the same flag (the queue-wide pause gates BOTH the flat and the grouped claim — a queue-wide pause stops all work).

**Why FORM (b) over FORM (a) (V-3, V-4 logged).** Form (a) — the gate INSIDE `@claim` — is atomic but EDITS emq.1's shipped `@claim` script, forcing the `claim` conformance scenario to re-run byte-for-byte identical to prove INV1, and a single byte drift in that shipped script is exactly the elevated risk the runbook's risk tier names ("the pause gate may touch the shipped `@claim` transition … escalates to an Apollo charter if it does not [stay byte-identical]"). Form (b) touches NO shipped script, so the `claim` scenario's bytes are untouched BY CONSTRUCTION — it cannot endanger INV1. The cost of (b) is one extra `HGET` on the claim path and a benign boundary-instant race (a pause landing between the flag-read and the claim serves at most one more job at the pausing instant — no job lost, no state corrupted, operationally identical to "the pause took effect one claim later"); the pause invariant is "claim returns empty", and pause never mutates pending, so there is no correctness property the two-step can violate. The runbook recommends (a) IFF the claim scenario stays byte-identical, ELSE (b), and names (b) the smaller-blast-radius fallback that cannot endanger INV1 — Mars-1 takes the fallback deliberately to keep the shipped `@claim` and its conformance scenario byte-for-byte untouched. **Consequence: `@claim` is NOT edited; the `claim` conformance scenario is byte-identical by construction (no re-run-and-diff needed); no `echo_wire` delegate is added.**

### D-3 — the `EMQ*` refusal class words: TWO new classes — `EMQLOCK` (held/locked job) and `EMQSTATE` (wrong-state precondition). Missing-job uses a return-value sentinel → `{:error, :gone}` (no new class).

EMQ.2.2-D1(c) / AS-1 / INV6. Spelled against design.md §5 (the closed wire-class registry; additive minor registered WITH its probe; the five-code fence union UNEXTENDED). Joins the existing `EMQKIND`/`EMQSTALE`/`EMQRATE`.

**The two new wire classes (each `redis.error_reply('EMQ<WORD> <msg>')` server-side → `{:error, {:server, "EMQ<WORD>" <> _}}` → a typed atom client-side; each registered with a conformance probe in the SAME change — the additive-minor law):**

1. **`EMQLOCK`** — a held/locked job refusal. Emitted by `remove_job/3` when the job's `:lock` subkey (`emq:{q}:job:<id>:lock`) is present (the worker-side lock plane writes it at emq.2.3; emq.2.2 only READS it). Client mapping: `{:error, :locked}` (US7). The refusal is the script's first effect after the existence check and CHANGES NOTHING (INV6).

2. **`EMQSTATE`** — a state-precondition refusal, used by the three "wrong current state" failures, each carrying a distinct message tail and mapped client-side to a DISTINCT typed atom by the calling function (the class word is the wire token; the atom is the function's precise mapping):
   - `obliterate/3` on a NOT-paused queue → `EMQSTATE not paused` → `{:error, :not_paused}` (US3).
   - `obliterate/3` on a paused queue with LIVE active jobs and no force → `EMQSTATE active jobs present` → `{:error, :active}` (US3).
   - `reprocess_job/3` on a job NOT in `dead` → `EMQSTATE not dead` → `{:error, :not_dead}` (US8).
   Each refusal is the script's first act and changes nothing (INV6).

**Missing-job on update/log — a return-value sentinel, NOT a wire class.** `update_data/4`, `update_progress/4`, `add_log/4`, `get_job_logs/3` refuse a MISSING job (`EXISTS job == 0`) by returning a sentinel (the script returns `-1`, the v1 convention), mapped client-side to `{:error, :gone}` — consistent with the as-built `EchoMQ.Jobs.complete/4`, which already maps a vanished row `{:ok, 0} -> {:error, :gone}`. A missing job is an EXISTENCE check, not a policy refusal, so it earns no new wire class (INV6 permits "mapped from an `EMQ*` class OR a typed absent" — the typed-absent arm). This keeps the wire-class registry minimal (only the genuine policy refusals — locked, wrong-state — get a class) and reuses the existing `:gone` vocabulary.

**Why two classes, not one or four (V-5, V-6 logged).** One class for everything (V-5) loses the lock-vs-state distinction a caller wants to branch on; per-refusal class words (V-6 — `EMQNOTPAUSED`/`EMQACTIVE`/`EMQNOTDEAD`) over-grows the registry for refusals that share one concern (a precondition on current state) and that the client atom ALREADY distinguishes. Two classes — one for the lock plane's held-job refusal (a distinct subsystem, emq.2.3's), one for state preconditions (distinguished client-side by atom) — is the minimal honest carve. The five-code fence union (the connector's version fence) is untouched — these are wire-class registry minors, not fence codes (INV1).

Declares `KEYS[1]` = the queue base `emq:{q}:` (the slot root, per the @counts/@lane_counts precedent), `KEYS[2]` = `pending`, optional `KEYS[3]` = `schedule`. Empties `pending` (ZRANGE all → for each id DEL `base..'job:'..id` + `base..'job:'..id..':logs'`, then DEL the whole `pending` set). With the schedule flag, also empties `schedule` the same way. **`active` is NOT touched** (in flight — INV3/US2). Each job key is DERIVED in-script from the declared `KEYS[1]` root by the §6 grammar (`base..'job:'..id`) — slot-sound, A-1-clean (INV4).
  - **Repeat-guard re-derivation (REALIZATION-OVER-LITERAL — flagged):** the spec says drain "keeps repeat-owned scheduled jobs". v1 guards id-encoded `repeat:<sched>:<millis>` members; but the as-built `EchoMQ.Repeat` mints a FRESH branded JOB id per occurrence and enqueues it as an ORDINARY job (no job→repeat backref is stored on the row), so there is NO id pattern distinguishing a repeat-produced scheduled job from a hand-scheduled one. The honest re-derivation: **drain leaves the repeat REGISTRY intact — it NEVER deletes `emq:{q}:repeat` or `emq:{q}:repeat:<name>`** — so a drain does not cancel a registered repeatable and future occurrences keep minting (the operationally meaningful guarantee). Individual already-enqueued scheduled occurrences ARE drained like any scheduled job when the schedule flag is set (they are just jobs). Recorded for Venus to fold into D3/INV3 and for the Director to ratify.

**OBLITERATE scope (`Admin.obliterate/3`, D4) — paused-only, bounded.** ONE inline `@obliterate` script, declares `KEYS[1]` = `meta`, `KEYS[2]` = the queue base root. **Refuses with `EMQSTATE not paused` if `HGET meta paused` is absent (D-3), changing nothing.** Then (the bounded iterative capability — returns 1/"call again" while work remains, 0 when done): sweeps up to ARGV[1] ids from `active` FIRST — if `active` is non-empty and no force flag, refuses `EMQSTATE active jobs present` (D-3) changing nothing; with force, deletes them. Then sweeps `pending`/`schedule`/`dead` (ZRANGE bounded → DEL each `job:<id>` + `:logs` → ZREM), and when all sets are empty, a final DEL of the auxiliary §6 keys: `metrics:completed`, `metrics:completed:data`, `metrics:failed`, `metrics:failed:data`, `de:*` (the dedup strings — see note), the lane structures (`gactive`/`glimit`/`ring`/`wake`/`paused`-SET + every `g:<g>:pending`), `repeat` + every `repeat:<name>`, `limiter`, and finally `meta` (the paused flag with it). Every job key DERIVED from the declared base root (INV4). NO `completed`/`failed` SET is touched (none exists — the metrics HASHes ARE the throughput record, deleted as §6 keys — US3).
  - **The `de:*`/`g:*`/`repeat:*` family deletion under declared-keys:** obliterate must clear key FAMILIES whose exact members it cannot enumerate from a declared root by a fixed grammar (the dedup ids, the group ids, the repeat names are open). The honest realization: obliterate derives the FIXED-name keys from the base root (the four sets, `meta`, `limiter`, `metrics:*`, `gactive`/`glimit`/`ring`/`wake`/`paused`-SET, `repeat`) directly; for the OPEN families (`g:<g>:pending` per group, `de:<did>` per dedup, `repeat:<name>` per name) it reads the members it CAN from a declared structure (the `ring`/`paused` SET names the live groups; `repeat` ZSET names the registrations) and derives each family key from those — every derived key shares the declared base slot (slot-sound). A `de:<did>` with no surviving backref is swept by a bounded `SCAN`-free approach is NOT available under declared-keys; the de: strings reachable from removed job rows are released at remove-time (D8) and at drain-time, and obliterate clears the de: keys discoverable from the live structures — recorded as the bounded-completeness honest limit (a dedup key orphaned with no live referrer is not individually discoverable under declared-keys; this is the same structural honesty the read plane took). Flagged for the Director/Venus.

**REMOVE dedup-release re-derivation (REALIZATION-OVER-LITERAL — flagged, D8).** v1 `removeJob` reads `HGET jobKey "deid"` to find the dedup key to release; the as-built three-field row has NO `deid`/`dedup` field, so remove CANNOT discover the dedup id from the row. Honest re-derivation: **`remove_job/3` accepts an OPTIONAL `dedup_id` argument**; when supplied, the script releases `de:<did>` IFF its value `== this job id` (the v1 guard, re-derived under declared keys — `de:<did>` derives from the declared base + the passed did); when omitted, no dedup key is released (nothing to find). This keeps declared-keys (no scan) and is honest about the as-built row shape. Recorded for Venus to fold into D8/US7.

### D-5 — the progress-event payload encoding (R-1 / AS-6): a `cjson.encode`d JSON OBJECT `{"job": <id>, "progress": <value>}` on channel `emq:{q}:events`, event name carried as the message itself under a `"progress"`-typed envelope

EMQ.2.2-D6 / AS-6 / the Stage-2 REMEDIATE R-1. This is the event CONTRACT emq.2.3's subscription inherits, so it is locked deliberately here.

**Decision.** `@update_progress`, after the `HSET … 'progress' ARGV[1]`, issues `redis.call('PUBLISH', base..'events', cjson.encode({event='progress', job=ARGV[2], progress=ARGV[1]}))` — a single JSON object message on the per-queue events channel `emq:{q}:events`. The channel derives from the declared queue base root (the same `base` idiom the drain/obliterate scripts use); a pub/sub CHANNEL is not a slot-routed keyspace key, so it needs NO §6 keyspace change, NO `KEYS[]` declaration (the declared-keys law governs keyspace keys a script reads/writes, not the transient channel name), and NO new transport (the connector's RESP3 `subscribe/2`→`push_to` seam is the existing wire — ADR-4, design §12.3's no-new-transport ruling). PUBLISH to a subscriber-less channel returns 0 — a forward-compatible no-op until emq.2.3 subscribes.

**The wire form (the inherited contract):** the message is the JSON object `{"event":"progress","job":"<branded-id>","progress":"<value>"}`. The event NAME (`"progress"`) is carried IN the payload as the `event` field rather than encoded in the channel name (one channel `emq:{q}:events` per queue carries all lifecycle events, the v1 `QueueEvents` shape — events are distinguished by the `event` field, not by a channel-per-event-type), so emq.2.3's `EchoMQ.Events` subscribes ONCE to `emq:{q}:events` and dispatches on the `event` field. `job` carries the branded job id; `progress` carries the caller's progress string verbatim.

**Why a JSON object via `cjson` (not a delimited string, not a JSON array).** (1) **cjson is the engine-native Lua serializer** (built into the Valkey/Redis embedded Lua — no dependency, the standard serialization the v1 scripts' `XADD`-event payloads would have used). (2) **Self-describing under content** — a progress value can itself contain `:`/`|`/whitespace, which a delimited string (`"progress:<id>:<value>"`) cannot survive unambiguously; a JSON object is robust to any payload bytes. (3) **A keyed object over a positional array** — `{"event","job","progress"}` keys let emq.2.3's subscriber read fields by name and lets the contract GROW (a future event field) without breaking positional decoding. The cost (a few bytes over a delimited string) is irrelevant for a fire-and-forget at-most-once notification. **Recorded in the `update_progress/4` @doc as the registered event contract** emq.2.3 inherits.

**The deterministic emit-proof (no ≥100 flake).** The `:update_progress` conformance scenario and `jobs_ops_test.exs` assert the emit through the SAME bounded-receive seam the shipped `resubscribe` scenario already uses (conformance.ex:359 — a second RESP3 connection `subscribe`s to `emq:{q}:events` with `push_to: self()` and a `receive … after <ms>` bound), NOT an unbounded receive. The subscribe happens BEFORE the update_progress call (no lost-wakeup race), and the receive is bounded (a missed message fails fast as a test failure, never hangs). This is the proven-deterministic mechanism; it survived the resubscribe scenario across every prior ≥100 loop.

## {emq-2-2-alternatives} Alternatives

### V-1

V-1 (for D-1) — all ten verbs fold onto `EchoMQ.Jobs`

**Steelman.** `Jobs` already holds every transition script and the `Keyspace.job_key/2`/`queue_key/2` idiom; folding pause/resume/drain/obliterate there too means ONE module owns every wire mutation, no new file, and every operator verb sits beside the state machine it touches. The pause gate (which honors the claim path) is arguably most coherent right next to `@claim` itself. Fewer modules is fewer things to wire and import.

**Why rejected.** The queue-scope verbs (pause/drain/obliterate) are NOT row transitions — they gate or sweep the whole queue, a distinct concern. Folding them into the 347-line state-machine module mixes "one job's lifecycle" with "the whole queue's lifecycle", and reads poorly: `obliterate` (destroy every set) next to `claim` (pop one id) obscures both. The read plane already set the precedent — `EchoMQ.Metrics` is a SEPARATE cohesive module beside `Jobs`, not folded in; the mutation plane's queue-scope twin should mirror that. Chose the split (D-1): queue-scope → `Admin`, row-mutation → `Jobs`.

### V-2

V-2 (for D-1) — a single new `EchoMQ.Operator` module holding all ten verbs

**Steelman.** One new module named for the whole concern (the operator plane), holding every emq.2.2 verb, leaves `Jobs` byte-untouched (zero regression risk to the shipped state machine + its conformance scenarios) and gives the rung one clear home a reader opens to find "the operator surface". The whole emq.2.2 diff lands in one new file plus the conformance registration — maximally reviewable as one unit.

**Why rejected.** It severs the six job-mutation verbs from the `Jobs` state machine they extend: `update_data`/`update_progress`/`add_log`/`remove_job`/`reprocess_job` all mutate the same three-field row + four sets that `enqueue`/`claim`/`complete`/`retry` own, and `reprocess_job` is literally a `dead`→`pending` transition — a sibling of `retry`'s `→dead`. Housing them in a separate `Operator` duplicates the row/set/key idioms and makes a future reader chase the row's mutators across two files. The cleaner boundary follows the concern, not the rung: queue-scope ops cohere in `Admin`; row transitions belong with the row machine in `Jobs`. The "zero `Jobs` change" benefit is illusory under the chosen pause mechanism anyway (D-2 keeps the gate OUT of `@claim` for form-(b), so `Jobs` gains the six mutation verbs but `@claim` itself is byte-unchanged).

### V-3

V-3 (for D-2) — FORM (a): the pause gate INSIDE the as-built `@claim` script (atomic)

**Steelman.** Putting `if redis.call('HGET', meta, 'paused') then return {} end` as the first line of `@claim` makes the pause check and the `ZPOPMIN` ONE atomic server step — no window at all between "is the queue paused?" and "pop the head". This is the strictly-strongest realization of "a paused queue claims empty": there is no instant, however brief, where a claim races a pause. v1's own design folds the flag into the claim/move scripts for the same atomicity. It also keeps the flag's enforcement in exactly one place (the script), so a second claim entrypoint cannot forget to honor it.

**Why rejected.** It EDITS emq.1's shipped `@claim` — the single most conformance-load-bearing script in the bus (the `claim` scenario asserts its exact transition; `rotate`/`pause`/`limit`/`reap`/`counts`/`state`/`metrics` all drive claim). Editing it forces a byte-for-byte re-run of the `claim` conformance scenario to PROVE INV1 (the 24-prior-byte-unchanged law applies to the SCENARIO body, but the script those scenarios drive must remain behavior-identical, and the runbook's risk tier flags this as the rung's elevated risk + an Apollo-escalation trigger if the scenario does not stay byte-identical). The atomicity it buys is over a race that is benign in (b) — pause never mutates pending, so the worst (b) outcome is one extra job served at the pausing instant, no correctness property violated. The blast radius (touching the shipped @claim + needing `@gclaim` edited too, since a queue-wide pause must gate the grouped claim as well — doubling the shipped-script edits) is not worth eliminating a benign boundary race. Chose FORM (b) (D-2): the separate gate cannot endanger INV1.

### V-4

V-4 (for D-2) — the paused flag as a DEDICATED §6 key `emq:{q}:paused-queue` (not a meta field)

**Steelman.** A dedicated key (e.g. `emq:{q}:paused` … but that COLLIDES with the Lanes per-group `paused` SET, so call it `emq:{q}:qpaused`) makes the queue-wide pause flag a first-class, greppable structure independent of the `meta` HASH's other fields, and a single `EXISTS qpaused` is the cheapest possible claim-path read (no HGET field decode). It separates "the queue is paused" from "the queue's config" cleanly.

**Why rejected.** It introduces a NEW key TYPE into the §6 grammar, which INV1 explicitly fences ("never a new type") unless registered — forcing a `keyspace.ex` change + a §6-grammar amendment + a registry update, a larger blast radius than a meta field for a one-bit flag. The `meta` HASH is ALREADY a registered §6 type carrying exactly this kind of queue-scope config (concurrency, max), and v1's own `pause-7` uses `meta.paused` — so a `meta.paused` field is the honest re-derivation that adds zero new type and zero keyspace change. (The naming-collision risk this alternative tries to avoid — colliding with the Lanes `paused` SET — is moot for a meta FIELD: `HSET meta paused 1` is a field on the meta hash, structurally disjoint from the `SADD emq:{q}:paused <group>` SET, which is a different key entirely.) Chose the meta field (D-2).

### V-5

V-5 (for D-3) — ONE new class `EMQSTATE` for every precondition refusal (lock + wrong-state alike)

**Steelman.** A single `EMQSTATE` class for all five precondition refusals (locked, not-paused, active, not-dead, and even missing) is the smallest possible registry growth — one additive minor, one probe. The client atom (`:locked`/`:not_paused`/`:active`/`:not_dead`/`:gone`) already carries the precise meaning, so the wire word need only say "a precondition failed" and let the function map the detail. Fewer classes = a smaller closed registry to reason about.

**Why rejected.** A held/locked job is a refusal from a DIFFERENT subsystem — the worker-side lock plane (emq.2.3's `:lock` subkey), not the queue/job state machine. Collapsing it into `EMQSTATE` erases the boundary between "a worker still holds this" (a liveness/ownership condition that resolves when the worker releases) and "this job is in the wrong set for your op" (a state condition that resolves by moving the job). A polyglot reader or a dashboard wants to distinguish "retry later, a worker has it" from "you asked to reprocess a non-dead job" at the WIRE level, not only via an Elixir atom mapping. `EMQLOCK` for the lock plane is the honest separate class; `EMQSTATE` for the state preconditions. Chose two (D-3).

### V-6

V-6 (for D-3) — a distinct class word per refusal: `EMQLOCK`/`EMQNOTPAUSED`/`EMQACTIVE`/`EMQNOTDEAD`/`EMQGONE`

**Steelman.** A bespoke class word per precondition gives the wire its own self-describing token for every failure — `EMQNOTPAUSED` reads exactly as "the queue is not paused" with no message-tail parsing, and a polyglot client can branch on the class word alone with zero ambiguity. Maximal wire-level precision.

**Why rejected.** It over-grows the closed wire-class registry: design §5 keeps the registry SMALL and additive, and three of these words (`EMQNOTPAUSED`/`EMQACTIVE`/`EMQNOTDEAD`) all name one concern — a precondition on the current state — that the client atom already distinguishes precisely. Adding five words where two concerns exist (lock vs state) inflates the registry the additive-minor law is meant to keep disciplined, and each new word is a new probe to register and a new token every port must learn. `EMQGONE` as a wire class for a missing job is also heavier than the existing typed-absent `{:error, :gone}` the bus already uses (`complete/4`). The minimal honest carve is two classes (D-3): `EMQLOCK` (the lock plane) + `EMQSTATE` (state preconditions, atom-distinguished), with missing-job staying a return-value sentinel → `:gone`.

### V-7

V-7 (for D-4, drain repeat guard) — store a job→repeat backref on the row so drain can guard individual repeat-produced scheduled occurrences (literal v1 parity)

**Steelman.** To honor the spec's "keeps repeat-owned scheduled jobs" LITERALLY, `EchoMQ.Repeat`/`Pump` could stamp a `repeat=<name>` field on each occurrence's row at mint; then drain's schedule sweep could `HGET job:<id> repeat` and skip any with a repeat owner — exact v1-parity behavior (the v1 `scheduledJobs[jobId]` guard, re-expressed on the as-built row). The operator gets the precise v1 guarantee: a drain leaves the not-yet-due repeat occurrences AND the registry.

**Why rejected.** It changes the as-built ENQUEUE/repeat surface (a new row field written by `EchoMQ.Repeat`/`Pump`, which emq.2.2's scope is "out" of — the repeat plane is emq.1's, frozen for this rung; touching it widens the blast radius past the operator plane and would need its own conformance re-proof). It also adds a field to the three-field row contract the 24 prior scenarios assert (`mint` asserts EXACTLY `state`/`attempts`/`payload`) — a row-shape change is exactly what INV1's byte-unchanged law guards against. The chosen re-derivation (D-4: guard the REGISTRY, not individual occurrences) gives the operationally meaningful guarantee (a drain doesn't cancel your repeatables) WITHOUT touching the row shape or the emq.1 repeat surface — the honest scope-respecting choice. Surfaced as a realization-over-literal for Venus to fold the spec's wording to.

### V-8

V-8 (for D-4, remove dedup release) — scan the `de:*` family to find the dedup key whose value is this job id (no caller argument)

**Steelman.** To release a held dedup key WITHOUT a caller-supplied dedup_id, `remove_job/3` could find the `de:<did>` whose value equals this job id by scanning the `de:*` family — giving the caller a zero-argument remove that still cleans up the dedup linkage automatically, matching the v1 "find deid from the row" convenience without needing the row to carry the backref.

**Why rejected.** A scan over `de:*` VIOLATES the declared-keys law (S-6 / INV4): a key the script touches must be in `KEYS[]` or derived from a declared `KEYS[n]` root by the registered grammar — a `KEYS`/`SCAN` over an open family is neither, and is the exact v1 anti-pattern (rooting key operands in runtime discovery) the v2 laws forbid. It is also not slot-safe under cluster (a scan crosses slots). The chosen re-derivation (D-4: an optional `dedup_id` argument, `de:<did>` derived from the declared base + the passed did, released IFF its value == this id) keeps declared-keys, stays slot-sound, and is honest that the as-built row stores no backref. The caller that parked a dedup key knows its did and passes it; a caller that did not, has nothing to release.

## {emq-2-2-learnings} Learnings

### L-1 — the as-built bus stores NO job→repeat backref and NO job→dedup backref, so two v1 operator capabilities re-derive rather than port literally

Building emq.2.2's drain (D3) and remove (D8) surfaced that the as-built three-field row (`state`/`attempts`/`payload`) deliberately stores none of the v1 linkage fields the v1 operator scripts depend on: v1 `drain-6` guards repeat occurrences by id-pattern (`repeat:<sched>:<millis>`) but the as-built `EchoMQ.Repeat` mints fresh ordinary branded JOB ids per occurrence (no `rjk` on the row); v1 `removeJob-12` finds the dedup key via `HGET jobKey "deid"` but the as-built row has no `deid` field. **Why it matters:** the spec's literal wording ("keeps repeat-owned scheduled jobs"; "releases a held dedup key whose value is this id") was written against the v1 capability and reads as if the backref exists. The honest re-derivations (D-4): drain guards the repeat REGISTRY not individual occurrences; remove takes an optional `dedup_id` arg. **How to apply:** when porting a v1 operator capability that reads a linkage FIELD off the job row, re-probe the as-built row shape FIRST — the v2 row is intentionally minimal, and a capability that depends on a stored backref must either re-derive against what the bus actually stores (a registry, a caller argument) or be flagged as needing a row-shape change (which INV1's byte-unchanged scenarios forbid mid-rung). The 24 prior scenarios pin the EXACT three-field row (`mint`), so adding a backref field is out of scope for an operator rung.

### L-2 — `mix format` is NOT this app's gate; running it would BREAK INV1's byte-unchanged law

`echo/apps/echo_mq` has NO `.formatter.exs` (the umbrella's declares `subdirectories: ["apps/*"]`, expecting a per-app file that does not exist), so `mix format` from the app dir uses bare defaults (line_length 98) the app's code was never written to.
