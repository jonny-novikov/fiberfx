### T-3 — the subtlest interaction: threading awaiting_children WITHOUT breaking the shipped unknown_state verdict (the D4 precision pin)

THE COLLISION. @state_lookup (metrics.ex:98-106) is a strict precedence ladder: ZSCORE KEYS[1..4] (pending/active/scheduled/dead) checked in order; ONLY when all four miss does it reach `if EXISTS KEYS[5]==1 then return 'unknown' end; return 'absent'`. The shipped `unknown_state` conformance scenario (conformance.ex:853-870) constructs EXACTLY this: enqueue → claim (active holds it) → ZREM active WITHOUT a transition → the row SURVIVES, no set holds it → asserts {:ok, :unknown}, with get_job(...) showing state="active" still in the row.

A flow PARENT is ALSO a row-in-no-set: @enqueue_flow writes the parent row state=awaiting_children and WITHHOLDS it from pending (the fan-in gate). So both a waiting parent and an unknown-state in-flight job are rows that exist and sit in no set.

THE NAIVE D4 (UNSOUND). "Return awaiting_children for any row in no set" would make @state_lookup return 'awaiting_children' for the unknown_state scenario's row too — silently flipping its verdict, breaking INV3 (byte-unchanged shipped surface) + INV7 (prior conformance byte-unchanged). The unknown_state row's `state` FIELD is "active", but a row-existence-only check (EXISTS KEYS[5]) cannot tell the two apart.

THE SOUND MECHANISM (the binding D4 statement). After the four set checks miss AND the row exists, @state_lookup must READ the row's `state` field and return `awaiting_children` ONLY when that field equals exactly the string "awaiting_children"; EVERY OTHER row-in-no-set still returns 'unknown'. Concretely the script's tail changes from:
  if redis.call('EXISTS', KEYS[5]) == 1 then return 'unknown' end
to:
  local st = redis.call('HGET', KEYS[5], 'state')
  if st == 'awaiting_children' then return 'awaiting_children' end
  if st then return 'unknown' end   -- a row exists (any other state field) but no set holds it
  return 'absent'                    -- no row
This preserves unknown_state BYTE-FOR-BYTE (its row's state field = "active" ≠ "awaiting_children" → still 'unknown') while threading the new state honestly. The host wrapper get_job_state/3 (metrics.ex:117-124) ALSO extends its @set_states/typespec: add :awaiting_children to the {:ok, ...} union (line 116) so String.to_existing_atom("awaiting_children") resolves (the atom must exist at runtime — it does once the spec names it / a module references it). @set_states (~w(pending active schedule dead), metrics.ex:23) is the SET-membership list for get_counts; awaiting_children is NOT a set (it is a row-field state, no ZSET), so it does NOT join @set_states — only the @state_lookup row-field branch + the typespec. This keeps the `counts` scenario (which asserts {:error,{:unknown_state,"wait"}} for an unregistered NAME) byte-unchanged.

THE INV3 GUARD on @complete. The same byte-unchanged discipline: the fan-in branch is REACHED ONLY when the completing child carries a parent reference (a declared KEYS[n] = the parent's :dependencies key passed in KEYS[]); a child with no parent ref takes the existing flat/grouped branches unchanged. The shipped @claim is UNTOUCHED (the gate is the parent's absence from pending — emq.2.2-D2 separate-gate discipline).

## {emq-3-1-learnings} Learnings

### L-1 — a new row-FIELD state must be threaded by a FIELD read, never a row-existence check, when a shipped scenario already asserts the row-in-no-set verdict

THE CRAFT FINDING (reusable beyond emq.3.1). When a rung adds a new job state that is a ROW FIELD with NO backing set (here awaiting_children — a flow parent withheld from every ZSET), and the read plane already has a "row exists but sits in no set" verdict that a shipped conformance scenario PINS (here @state_lookup's `EXISTS KEYS[5] → 'unknown'`, asserted by the unknown_state scenario), the new state CANNOT be threaded by widening the row-existence branch — that silently captures the OTHER row-in-no-set case (the in-flight unknown job) and flips its byte-unchanged verdict. The sound mechanism is a FIELD read (HGET KEYS[5] 'state') that returns the new state ONLY for its exact field value, leaving every other extant-row state on the original verdict. The discriminator is the row's own `state` field, never row existence.

WHY IT'S SUBTLE: the spec's natural-language D4 ("thread awaiting_children into the state-set membership") READS as "add it to @set_states" or "report it for a waiting parent" — both of which a naive build satisfies by the existence branch, passing every test EXCEPT the one shipped scenario whose row also sits in no set. The collision is invisible until you notice TWO distinct conditions (a waiting parent / an in-flight unknown job) both reduce to "row exists, no set holds it." Pinning the FIELD-read mechanism in the body as a binding D-n (not leaving it to the builder) is what makes Mars build it deliberately and Apollo able to verify the unknown_state byte-unchanged regression.

GENERALIZES TO: any additive read-plane state on a substrate where a precedence-ladder lookup already terminates in a catch-all that a conformance scenario asserts. The same class as "a gate counts only if it RUNS" — here "a new verdict must be keyed on its OWN discriminator, not a broadened catch-all." Fold into echo-mq-architect (the read-plane-additive-state craft) at the mentoring loop if the Operator grants.

### L-2 — the fan-in idempotency is doubly-guaranteed: the row-retire (DEL) makes a re-completion :gone BEFORE the was_active==1 gate is reached, so a double-complete returns :gone (not :stale) regardless of token

THE SURPRISE (caught by the build's own test, twice — flow_fanin_test:108 + conformance flow_fanin). The directive + Venus D-2 specify the DECR gated INSIDE the was_active==1 branch (the Fork-B double-complete guard). I built exactly that. But the shipped @complete does `DEL KEYS[2]` (the row) on EVERY successful completion (jobs.ex:168). So once a child completes, its row is GONE, and a second @complete hits the FIRST guard `att = HGET KEYS[2] 'attempts'; if not att then return 0` → {:ok,0} → complete/4 maps to {:error, :gone} — BEFORE it ever reaches the was_active==1 fan-in branch. So a double-complete of an already-completed child returns :gone, NOT :stale, and the token is never even compared (the row's gone).
WHY IT MATTERS: my first test/probe asserted {:error, :stale} for the double-complete (the natural reading of "a stale-token retry"). Both failed with right: {:error, :gone}. The CORRECT semantics is stronger than the spec's natural phrasing: the idempotency is doubly-guaranteed — (1) the row-retire makes the DECR structurally unreachable for an already-completed child (any token → :gone), and (2) the was_active==1 gate additionally guards the redelivery-of-a-still-active case (where ZREM returns 0 but the row still exists, e.g. a reap re-claim). The fix: assert :gone + the count holds at 1 (proving no second decrement) — the stronger property. INV5 is satisfied either way (count drops by exactly 1).
THE CRAFT FINDING (reusable): when folding an idempotent side-effect into a script that ALREADY retires its primary key (DEL), the retire is the FIRST idempotency layer — a re-invocation short-circuits at the existence guard before any new branch runs. A test asserting the new branch's guard (was_active==1) by a double-complete will instead exercise the row-gone path and see the existing :gone/:absent verdict, not the new branch's. Assert the OBSERVABLE invariant (the count dropped by exactly 1), not a presumed error class. Generalizes the "realization over literal" rule to error semantics: the spec's natural-language "a stale-token retry is refused" READS as EMQSTALE, but the as-built retire makes it :gone — build + test to the observable, flag the deviation.

### L-3 — REMEDIATE O1 (the :processed placeholder): an honest bound, NOT a fix — :processed records child_id→child_id (a presence marker), the real child-result write + read is emq.3.2

THE DIRECTOR'S O1. complete/4's @complete fan-in does HSET KEYS[4] ARGV[1] ARGV[5] where ARGV[5]=job_id (the child's own id) — so the parent's :processed subkey records child_id→child_id, a PRESENCE MARKER, not a result payload. To record a real result would require either a result-bearing complete arity (a public-surface change emq.3.1 is scoped to AVOID) or a host read of the child payload before the DEL retires it (another round-trip). emq.3.2 adds both the result-passing AND the read API (children_values/dependencies over :processed). RESOLUTION: the bound is NAMED in the parent_of/3 comment ("a DELIBERATE presence marker, not a result payload ... the real child-result write + the read API are emq.3.2's, which this honest bound names rather than fakes"), sitting alongside the dead-child limit (INV9) as emq.3.1's two honest bounds. ### L-4 — REMEDIATE O2 (parent_of HGET per completion): a carried emq.3.2 perf follow-up, deliberately NOT folded now (it would expand the surface past the rung)

### L-5 — Apollo's un-prompted finding: a rung that ADDS a new per-job subkey owns that subkey's LIFECYCLE (the @complete row-retire AND obliterate's del_job sweep), or the subkey leaks — emq.3.1's :dependencies/:processed survive both the parent's completion and the queue-wipe

THE FINDING (surfaced by an un-prompted probe, not a prompted check). emq.3.1 introduces two new per-job subkeys on the flow parent: emq:{q}:job:<parent>:dependencies (the outstanding-child counter) and :processed (the child-completion record). Probing the full lifecycle:
- @complete does `DEL KEYS[2]` (the parent ROW) on every success (jobs.ex:189) but does NOT touch the parent's :dependencies/:processed subkeys → after the parent completes, the spent counter (value 0) and the :processed hash REMAIN.
- @obliterate's del_job helper (admin.ex:150-153) enumerates a FIXED subkey set: `redis.call('DEL', jk, jk..':logs', jk..':lock')` — it has no knowledge of the NEW :dependencies/:processed subkeys, so they survive the queue-wipe too.
Net: the flow subkeys leak past BOTH the per-job completion AND the bounded obliterate sweep, accumulating until a manual KEYS-pattern DEL (which is exactly what the per-test on_exit purge does — masking the leak in the suite).

WHY IT'S NOT A BLOCKER (the careful classification — the verdict stays BUILD-GRADE):
1. :processed MUST outlive the parent row. emq.3.2's children_values/3 read API consults :processed AFTER the parent runs — deleting it at parent-completion would be a DEFECT, not a fix. The retention is correct-by-design for the deferred read.
2. The leak is OUTSIDE emq.3.1's stated deliverables. The spec scopes the rung to "single-queue, one-level, WRITE-SIDE flow only" (INV9); the read + the lifecycle is emq.3.2. The residue parallels the already-accepted dead-child honest bound (a named limit, not a silent drop).
3. The ONE concrete documentation defect: the @obliterate moduledoc CLAIMS "every reachable job row + its subkeys" (admin.ex:230) — now mildly STALE, since it does NOT reach :dependencies/:processed. That stale claim is a Venus reconcile sync, not a code change.

THE CRAFT FINDING (reusable, for the architect's skill). When a rung adds a new per-job SUBKEY (a `job:<id>:<suffix>` composed via <>), the subkey's LIFECYCLE is part of the rung's surface, not free: (a) the per-job retire (@complete's DEL list) and (b) the bounded queue-wipe (obliterate's del_job enumeration) are FIXED key-lists that do not auto-discover a new suffix. The architect's spec for a subkey-adding rung should NAME the lifecycle disposition explicitly — "the subkey is cleaned at X" or "the subkey is retained for Y (the deferred read), swept only by Z" — so the leak is a NAMED bound (like the dead-child bound) rather than an invisible accumulation a per-test purge masks. GENERALIZES the "a new verdict must be keyed on its OWN discriminator" (L-1) class to lifecycle: "a new subkey must have its OWN named cleanup disposition, never inherit the fixed del-list silently." PROPOSED fold into echo-mq-architect (the subkey-lifecycle craft) — Director-ratified, NOT applied unilaterally (the exact diff is in Apollo's report to the Director). Neither emq.3.md, emq.3.1.md, nor the design canon currently names the flow-subkey cleanup — a genuine undocumented gap the architect's next flow rung (emq.3.2) should close.
