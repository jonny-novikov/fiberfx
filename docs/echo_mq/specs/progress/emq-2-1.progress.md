### D-3 — the counts contract: the four as-built SET names + the metrics counter for "completed" (AS-1, spec D2/INV3, design §6)

### D-4 — the metrics-counter write: land the minimal additive write HERE, in @complete and @retry-to-dead (AS-1/AS-4, spec D4/INV2 no-phantom-counter, design §6)

PROBE RESULT (re-probe T-2): NO metrics counter is written today — @complete DELs the row (jobs.ex:168) with no metrics:* write; @retry writes last_error + state dead/scheduled (jobs.ex:201-211) with no metrics:* write. So a get_metrics over emq:{q}:metrics:completed would read a phantom (INV2 breach) unless the write lands.
RULED: land the MINIMAL additive write HERE (not flagged to emq.2.2), because (a) the read is meaningless without it — shipping get_metrics that always reads {count:0} is a phantom read INV2 forbids; (b) it is small and additive — one HINCRBY per transition, declared key, no wire break, no new key TYPE (metrics:completed/failed are §6-registered suffixes); (c) the completion/dead transition is the ONLY correct write site (the counter counts terminal outcomes). The writes:
- @complete: on the successful retire (the return-1 path), HINCRBY emq:{q}:metrics:completed 'count' 1. Declared as a new KEYS[] entry on @complete (the metrics:completed key) — keeps S-6 declared-keys. This is a WRITE inside a transition script, NOT a new emq.2.1 read-verb-that-mutates, so INV2 (every emq.2.1 VERB is a pure read) holds: the verb get_metrics is pure-read; the counter maintenance rides the EXISTING completion transition (emq.1's state machine), the minimal change D1/D4 explicitly authorizes ("a metrics counter the completion/dead transition maintains — ruled at D1, built minimally here").
- @retry: on the dead-letter path (the att>=max arm, jobs.ex:202-205), HINCRBY emq:{q}:metrics:failed 'count' 1. Declared as a new KEYS[] entry on @retry.
SCOPE GUARD: the counter is a single 'count' field (HINCRBY hash field). The v1 getMetrics-2 also reads prevTS/prevCount + a :data time series (rolling data points) — those are a PRESENTATION concern (the rolling-window meter), deferred (spec Scope "Out": the Prometheus export wrapper beyond the raw metrics read, emq.8). get_metrics reads the 'count' field; the :data series is read as empty-tolerant (read no series that is not written — if absent, answer 0 points). This keeps the write minimal and the read honest (no phantom).
REALIZATION-OVER-LITERAL FLAG: the spec D4 lists "metrics:completed/metrics:failed with :data series where §6 registers them" — §6 registers metrics:completed[:data] as a structure, but emq.2.1 writes only the count field and does NOT write the :data series this rung (deferred per Scope Out). get_metrics reads count (real) + reports the series length honestly (0 when unwritten). Logged for Venus's reconcile.

### D-5 — the rate-gate class word: EMQRATE, registered as an additive minor with its conformance probe (AS-1/AS-6, spec D6/INV6, design §5)

RULED: the at-ceiling refusal leads with the wire class EMQRATE via redis.error_reply (format "EMQRATE <detail>"), per design §5 (the closed wire-class registry — at 2.0 exactly EMQKIND + EMQSTALE; "adding a wire class is an additive protocol minor, registered with its conformance probe in the same change"). EMQRATE joins as the third class. Client-side mapping seam: {:error, {:server, "EMQRATE" <> _}} -> {:error, :rate} (the per-runtime one-line mapping §5 names; matches the EMQKIND/EMQSTALE precedent in jobs.ex:33,292). An unrecognized EMQ* first word still passes through untyped (forward-compatible — INV6). The five-code fence union stands UNEXTENDED (§5: "wire classes are per-call runtime replies", distinct from the boot fence union — no fence-union code added). The gate is a READ-AND-REFUSE: is_maxed reads the limiter/meta and the active count; where a claim WOULD exceed the ceiling it returns the EMQRATE refusal — it performs NO state transition, moves NO set member (INV2). Registered: a conformance scenario `rate` + its probe in the SAME change (the additive-minor law), the 18 prior scenarios byte-unchanged, the count re-pinned 18→19 (or 18→N for all new read scenarios) in both pinning tests.
NOTE the bus-shape adaptation: v1 isMaxed-2 reads meta.concurrency + LLEN active (a LIST).
