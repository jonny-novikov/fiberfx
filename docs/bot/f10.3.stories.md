# F10.3 ¬∑ user stories

> Who wants the minimal valuable game ‚Äî `echo_games` + the Svelte/Vite rewrite + the isolated games web server +
> WebSocket live state + the payments seam ‚Äî what they need, and how we will know it works. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f10.3.md`](f10.3.md).

## F10.3-US1 ‚Äî Play the game launched from a bot

As a **learner**, I want to launch "–≠–º–æ–¥–∂–∏: –ù–∞–π–¥–∏ –ø–∞—Ä—É" from a bot and play it in the browser, so that I have a real,
fun game I can reach straight from chat.

Acceptance criteria
- Given a bot from F10.1/F10.2, when a `/play` command is sent, then the bot answers with a Telegram **Game** whose
  launch URL is served by the games web server (the manual demo).
- Given the launch URL, when it is opened, then the Svelte 5/Vite frontend loads from the games server and shows the
  start screen with the preserved warm cream/coral/sage design and Russian copy.
- Given the game, when a card is tapped, then it flips and the board updates from server-pushed state.

INVEST ‚Äî valuable on its own (a real game launched from chat); testable by a manual launch demo plus a bundle-serve
probe; encodes F10.3-INV5, INV8.
Priority: must ¬∑ Size: 5 ¬∑ Implements deliverables: F10.3-D6, F10.3-D7, F10.3-D9.

## F10.3-US2 ‚Äî Run `echo_games` as a standalone supervised app

As an **operator**, I want `echo_games` to boot as its own supervised umbrella app on its own port, so that I can run it
with the standard Elixir toolchain and it never depends on or disturbs the Portal or the bot engine.

Acceptance criteria
- Given the umbrella, when it starts, then `EchoGames.Application` supervises the games web server's Bandit listener, the
  session `DynamicSupervisor`, and any registry as a new app supervisor (alongside `Portal.Application`,
  `PortalWeb.Application`, and `EchoBot.Application`), `:one_for_one`, named `EchoGames.Supervisor`.
- Given the running umbrella, when `Portal.Application`, `PortalWeb.Application`, and `EchoBot.Application` are inspected,
  then they are unchanged and independent, and `echo_games` has added no child to any of them.
- Given the games server, when its listener is inspected, then it is a separate Bandit listener on its own port (not
  `:4000`, not `portal_web`'s) ‚Äî never a `portal_web` route or child.

INVEST ‚Äî independent of game content; testable by a boot check and a tree/port assertion; encodes F10.3-INV2, INV3, INV1.
Priority: must ¬∑ Size: 3 ¬∑ Implements deliverables: F10.3-D1, F10.3-D5.

## F10.3-US3 ‚Äî Keep the score authoritative on the server

As a **developer**, I want the game's deck, scoring, combo, stars, and timers to live in `echo_games` on the BEAM, so
that a score cannot be forged client-side and a paid game becomes possible.

Acceptance criteria
- Given the game model, when scoring is checked, then it matches `html/game.html` exactly: `BASE_PER_PAIR = 100`,
  `COMBO_STEP = 25` (`bonus = (combo - 1) √ó 25`), `WRONG_PENALTY = 10`, `TIME_BONUS = 5`/s, the three levels `4√ó3 / 4√ó4 /
  6√ó4`, and the per-level star thresholds (`550/800`, `1100/1500`, `1700/2300`).
- Given a card-tap intent over the WebSocket, when the server applies it, then `EchoGames.Game` computes the new
  authoritative state and pushes it down; the client computes no score.
- Given a forged score from the client, when it arrives, then it is never trusted ‚Äî the server's computed score is
  authoritative.

INVEST ‚Äî independent; testable by game-logic unit tests against the constants plus a round-trip authority check; encodes
F10.3-INV4.
Priority: must ¬∑ Size: 5 ¬∑ Implements deliverables: F10.3-D2, F10.3-D3.

## F10.3-US4 ‚Äî See the game update live over a WebSocket

As a **learner**, I want each card tap to update the board live without a page reload, so that the game feels immediate.

Acceptance criteria
- Given an open session, when a `{"type": "tap", "card": <id>}` intent is sent over the WebSocket, then the server
  applies it through `EchoGames.Session` and pushes the authoritative state (the visible deck, the HUD, the timer, the
  level/game result) back.
- Given two revealed cards, when they match, then both stay face-up in the pushed state; when they mismatch, then the
  pushed state flips them back and applies the penalty.
- Given the round trip, when it completes, then the client rendered only pushed state and held no authoritative state of
  its own.

INVEST ‚Äî independent; testable by a WebSocket round-trip test; encodes F10.3-INV5.
Priority: must ¬∑ Size: 3 ¬∑ Implements deliverables: F10.3-D4, F10.3-D3.

## F10.3-US5 ‚Äî Serve the Svelte/Vite bundle from the isolated games server

As an **operator**, I want the Vite static bundle served by the games web server over an explicit `Plug.Static` mount, so
that the frontend ships as built static assets with no separate frontend host and no Portal coupling.

Acceptance criteria
- Given a built Vite bundle in `apps/echo_games/priv/static/`, when the games server starts, then it mounts the bundle
  over an explicit `Plug.Static` clause in `EchoGames.Web.Router`.
- Given the running server, when the bundle path is probed with `curl`, then it returns `200` with the expected content
  type ‚Äî verified by the PROBE, not by reading the config (the F6.5.5 prefix-strip guard).
- Given the games server, when it is compared to `portal_web`, then it shares no listener, port, supervisor, or config
  key with it.

INVEST ‚Äî independent; testable by a static-bundle serve probe and an isolation check; encodes F10.3-INV2, INV7.
Priority: must ¬∑ Size: 3 ¬∑ Implements deliverables: F10.3-D5, F10.3-D6.

## F10.3-US6 ‚Äî Open the payments seam without building the flow

As an **operator**, I want a reserved payment-webhook route and a server-authoritative score now, so that the monetized
game has its integration point placed correctly before the full flow lands.

Acceptance criteria
- Given the games server, when the reserved payment-webhook route is requested, then it exists as a named `Plug.Router`
  clause and returns a not-implemented marker ‚Äî no invoice, no `successful_payment` handling, no validation.
- Given the game, when a level completes, then the score it produces is server-authoritative (the precondition a paid
  outcome requires).
- Given the rung, when the payment surface is inspected, then no payment logic, invoice, or paid outcome is built ‚Äî the
  full flow is deferred to F10.8.

INVEST ‚Äî independent; testable by a route-exists check and a confirmation that no payment logic ships; encodes
F10.3-INV6, INV4.
Priority: should ¬∑ Size: 2 ¬∑ Implements deliverables: F10.3-D7.

## F10.3-US7 ‚Äî Hold the Portal no-touch boundary and the new-server boundary

As a **developer**, I want `echo_games` to never modify, depend on, or supervise the Portal, and the games server to
never be `portal_web`, so that adding the game cannot disturb Portal's or the bot engine's active development.

Acceptance criteria
- Given the games app, when I search `apps/echo_games/lib/`, then no module references `Portal`, `PortalWeb`, a Portal
  context, or a Portal repo (`grep -rE "Portal|PortalWeb" apps/echo_games/lib/` is empty).
- Given the dependency graph, when `apps/echo_games/mix.exs` is inspected, then it lists neither `{:portal, in_umbrella:
  true}` nor `{:portal_web, in_umbrella: true}`, and the game-launch keeps `echo_games` and `echo_bot` decoupled (neither
  lists the other, neither names the other's modules).
- Given the build, when `apps/portal/` and `apps/portal_web/` are diffed, then they are unchanged by F10.3, no child is
  added to `Portal.Application`, and the games server adds no route or child to `portal_web`.

INVEST ‚Äî independent; testable by static searches and a diff; encodes F10.3-INV1, INV2, INV8.
Priority: must ¬∑ Size: 2 ¬∑ Implements deliverables: F10.3-D1, F10.3-D5.

## F10.3-US8 ‚Äî A self-healing game session

As an **operator**, I want a crashed game session to recover on its own without touching other sessions or apps, so that
one player's fault never disturbs another's game or the rest of the umbrella.

Acceptance criteria
- Given the running umbrella, when a `EchoGames.Session` process is killed, then the `DynamicSupervisor` isolates it and
  the crash never reaches another session or another app's tree.
- Given a fresh session, when it starts, then it loads its own authoritative `EchoGames.Game` state under its own
  process.
- Given the session crash, when it happens, then `Portal.Application`, `PortalWeb.Application`, and `EchoBot.Application`
  are undisturbed.

INVEST ‚Äî independent; testable by killing a session process and asserting isolation; encodes F10.3-INV3.
Priority: should ¬∑ Size: 2 ¬∑ Implements deliverables: F10.3-D3.
