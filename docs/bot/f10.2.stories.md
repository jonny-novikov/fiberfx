# F10.2 · user stories

> Who wants the engine that runs N YAML bots under one supervisor, what they need, and how we will know it works.
> Connextra stories with Given/When/Then acceptance criteria; together they cover every deliverable in
> [`f10.2.md`](f10.2.md). The platform-adapter port, the vendored ex_gram wrap, and the YAML v1.0 schema are inherited
> from [`f10.1.md`](f10.1.md) unchanged; these stories cover the **multi-bot** generalization only.

## F10.2-US1 — Run several bots from one config directory

As an **operator**, I want the engine to start one bot per YAML file in a config directory, so that I can run several
bots from one release without editing engine code.

Acceptance criteria
- Given a bot-config directory holding **N** valid v1.0 `*.yaml` files, when the engine boots, then `EchoBot.Config`
  returns N per-bot definitions (one per file) and `EchoBot.Supervisor` runs N supervised bot children.
- Given each definition, when a bot starts, then its supervisor child id is derived from the YAML `name`, and its token
  is resolved from its own `token_env` (the token absent from every YAML).
- Given two real Telegram bots configured with two files and two tokens, when each is sent `/start`, then each answers in
  its own chat (the demo).

INVEST — valuable on its own (several bots from one release); testable by a directory-of-N boot check plus a two-bot
manual demo; encodes F10.2-INV3, INV4.
Priority: must · Size: 5 · Implements deliverables: F10.2-D1, F10.2-D3, F10.2-D4, F10.2-D8.

## F10.2-US2 — Add a bot by dropping a file

As an **operator**, I want adding a bot to be dropping a YAML file plus a handler, so that growing the fleet never
touches engine code.

Acceptance criteria
- Given a running engine with one bot, when a **second** v1.0 `*.yaml` file and its handler module are added to the
  config directory and the engine restarts, then a second supervised bot runs alongside the first with no edit to
  `EchoBot.Application` or `EchoBot.Config`.
- Given the two bots, when each is sent `/start` and `/help`, then each answers with its **own** static text.
- Given the engine code, when it is inspected, then no bot is named in engine code — the running set is exactly the set
  of valid files in the directory.

INVEST — independent of the count; testable by adding a file + handler and asserting a second bot runs; encodes
F10.2-INV3.
Priority: must · Size: 3 · Implements deliverables: F10.2-D5, F10.2-D1, F10.2-D3.

## F10.2-US3 — Isolate every bot from its siblings

As an **operator**, I want one bot's crash to leave the others answering, so that a bug or a revoked token in one bot
never takes down the fleet.

Acceptance criteria
- Given two bots running under `EchoBot.Supervisor`, when one bot's updater process is killed, then `:one_for_one`
  restarts **only** that bot and the sibling bot's process is untouched.
- Given the crashed bot's restart window, when the surviving bot is sent `/start`, then it still answers.
- Given the crash and restart, when the engine and the other umbrella apps are inspected, then `EchoBot.Supervisor`,
  `Portal.Application`, and `PortalWeb.Application` are undisturbed.

INVEST — the load-bearing multi-bot property; testable by killing one bot and feeding the sibling an update; encodes
F10.2-INV2.
Priority: must · Size: 3 · Implements deliverables: F10.2-D3, F10.2-D7.

## F10.2-US4 — Reject a misconfigured fleet

As an **operator**, I want a malformed or colliding bot file to surface loudly, so that a typo never silently runs the
wrong fleet.

Acceptance criteria
- Given a config directory where two files share the same `name`, when the loader reads it, then the duplicate `name` is
  rejected (the supervisor child id must be unique) rather than starting one bot and dropping the other.
- Given a file that is malformed, missing a required key, or carries a non-`"1.0"` `version`, when the loader reads the
  directory, then that file's outcome is reported per file (reconciled to F10.1's loader error convention; recommended
  posture: fail-fast at boot).
- Given a clean directory, when the loader reads it, then every file yields a valid definition and the engine boots the
  full fleet.

INVEST — independent; testable by a colliding-name fixture and a malformed-file fixture; encodes F10.2-INV5, INV4.
Priority: must · Size: 3 · Implements deliverables: F10.2-D2.

## F10.2-US5 — Route each update to the right bot

As a **developer**, I want each bot to dispatch through its own handler with its own token, so that an update to one bot
is never crossed with another.

Acceptance criteria
- Given two bots A and B, when a constructed `/start` is fed to bot A via the fake updater, then bot A's handler answers
  with bot A's welcome text and bot B is untouched.
- Given each bot, when it boots, then it resolves its **own** token from its **own** `token_env` (two bots name two env
  vars) and routes through its **own** `handler` module.
- Given the handlers, when they run, then there is no shared mutable handler state across bots.

INVEST — independent; testable by feeding distinct constructed updates to each bot's fake updater; encodes F10.2-INV7.
Priority: must · Size: 2 · Implements deliverables: F10.2-D4, F10.2-D6.

## F10.2-US6 — Keep the platform-adapter and vendored-wrap boundaries

As a **developer**, I want N bots to run through the same platform-adapter port and the same vendored wrap F10.1 built,
so that multi-bot adds no platform coupling and no new vendored surface.

Acceptance criteria
- Given the engine, when it is searched, then it names only the `EchoBot.Platform` behaviour, no Telegram-only call sits
  in the engine core, and only `EchoBot.Platform.Telegram` names a vendored module.
- Given F10.2's diff, when it is inspected, then it adds no adapter, no platform, and no vendored code — the multi-bot
  fan-out runs N bots through the F10.1 port and wrap unchanged.
- Given every bot, when it polls and replies, then it does so through the inherited `EchoBot.Platform.Telegram` adapter
  (the fake updater in tests).

INVEST — independent; testable by a static search and a diff of the adapter/vendored tree; encodes F10.2-INV6, INV8.
Priority: should · Size: 2 · Implements deliverables: F10.2-D3, F10.2-D6.

## F10.2-US7 — Hold the Portal no-touch boundary across the fleet

As a **developer**, I want the multi-bot engine to never modify, depend on, or supervise the Portal, so that growing the
bot fleet cannot disturb Portal's active development.

Acceptance criteria
- Given the bot app, when `apps/echo_bot/lib/` is searched, then no module references `Portal`, `PortalWeb`,
  `Portal.Engine`, a Portal context, or a Portal repo (`grep -rE "Portal|PortalWeb" apps/echo_bot/lib/` is empty).
- Given the dependency graph, when `apps/echo_bot/mix.exs` is inspected, then it does not list `{:portal, in_umbrella:
  true}` and names neither Portal app.
- Given the build, when `apps/portal/` and `apps/portal_web/` are diffed, then they are unchanged by F10.2, and no child
  is added to `Portal.Application`.

INVEST — independent; testable by a static search and a diff; encodes F10.2-INV1.
Priority: must · Size: 1 · Implements deliverables: F10.2-D8.
