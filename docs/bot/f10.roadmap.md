## What this delivers

A standalone bot engine that loads bots from versioned YAML config and runs them under OTP supervision, plus the first
playable, monetizable game slice on top of it. The near-term arc is three concrete iterations:

- **F10.1** ‚Äî the engine skeleton: standalone `apps/echo_bot` with its own `EchoBot.Application`; a **vendored copy of
  ex_gram** wrapped behind the `EchoBot.Platform` port (Telegram = adapter #1); **one** bot from one YAML v1.0 config;
  polling; `/start` and `/help` answered with static text; a fake-updater test posture. No Portal, no games.
- **F10.2** ‚Äî multi-bot: **N** YAML bots under one engine supervisor ‚Äî where "multibot" actually lands.
- **F10.3** ‚Äî the minimal valuable game: a new **`apps/echo_games`** backend for "–≠–º–æ–¥–∂–∏: –ù–∞–π–¥–∏ –ø–∞—Ä—É", the Svelte 5/Vite
  frontend rewrite of `html/game.html`, the chosen isolated Elixir games web server, WebSocket live state, the bot
  launching the game (Telegram Game), and the **Telegram-payments seam** opened. The first playable, monetizable slice.

F10.4 onward (the richer Telegram surface, webhook delivery, platform-adapter hardening, the optional Portal-consumer
integration, robustness/rate-limits, notifications, and the second-platform seam) stay **abstracts** and are sharpened
into triads only when the shipped engine and the first game earn feedback.

## Architecture decision ‚Äî wrap-then-own, in-BEAM, platform-abstracted, YAML-driven

Four decisions define `echo_bot`, and each is a deliberate product choice.

**Wrap-then-own (a vendored fork, not a hex dependency).** ex_gram is the reference Elixir Telegram Bot API client and
framework: its API is auto-generated to track the current Telegram Bot API, it supports both polling and webhook
updaters, it runs under OTP supervision, and it natively runs multiple bots in one application
([github.com/rockneurotiko/ex_gram](https://github.com/rockneurotiko/ex_gram),
[hexdocs.pm/ex_gram](https://hexdocs.pm/ex_gram)). `echo_bot` does **not** depend on the hex package and does **not**
hand-roll a client from scratch. Instead it carries an **internal, vendored copy** of ex_gram's Telegram-API code inside
the app, and the `EchoBot.Platform.Telegram` adapter **wraps that vendored copy** behind the port. The wrap establishes
the platform-adapter boundary from F10.1; later rungs replace the vendored internals with first-party code behind the
same port, on the engine's own schedule. The vendoring is an **owned fork**: the Operator is the ex_gram team, tasked by
the main maintainer with building next-gen bot modules, so agents may Read, Write, and modify the vendored code directly,
with **no upstream PR to ex_gram main** (see the vendoring contract in F10.1 below). The grammY ecosystem on the Node
side ([grammy.dev](https://grammy.dev)) demonstrates the same posture ‚Äî a maintained, current, plugin-driven framework
owned by its community ‚Äî and informs the design even though `echo_bot` stays on the BEAM.

**In-BEAM under OTP.** The research compared an Elixir-native bot against a Node/Fastify + grammY worker over a queue and
found the in-BEAM choice stronger for an engine that ships every iteration: one runtime, one supervision tree, no extra
infrastructure, crash isolation and restart for free, and a single test stack (`research.report.md` ¬ß3‚Äì¬ß4). That finding
holds, and `echo_bot` keeps it ‚Äî the engine runs as a supervised OTP application. The difference from the research's
framing is the product decision to **own** the engine (via the vendored fork) and to **abstract the platform**, so the
in-BEAM choice buys vendor-independence and a multi-platform future, not only topology simplicity.

**Platform-abstracted (an adapter port).** The engine core speaks to a **platform-adapter behaviour** ‚Äî a port that
defines how the engine fetches updates, sends replies, and describes a platform's command shape. Telegram is adapter #1;
the behaviour is the seam a second platform plugs into later. The core never names Telegram directly; it names the
behaviour, and `EchoBot.Platform.Telegram` (wrapping the vendored copy) implements it. This is the same substitutability
the Portal `EventStore` port relies on, applied to the platform itself.

**YAML-driven (plug-and-play, versioned).** Bots are declared in versioned YAML config files, not in code. A config
loader reads each bot's definition ‚Äî its schema `version`, `name`, `platform`, the env var its token is read from, its
handler module ‚Äî and the engine starts a supervised bot per definition. Dropping a YAML file plus a handler module yields
a running bot; removing the file removes the bot. Secrets stay out of the YAML: the token is referenced by
environment-variable name and read at boot, never written into the config file. The `version` field makes the schema
forward-versionable ‚Äî v1.0 is approved (below); later schema changes bump the version.

## The boundary (the master invariants for the engine + games)

`echo_bot` and `echo_games` are standalone apps. Three boundaries define them:

- **Portal untouched.** Neither `echo_bot` nor `echo_games` joins `Portal.Application`, modifies `portal` or
  `portal_web` code, config, or supervision, or depends on either app. The two Portal apps are under active development
  (F6.x) and stay exactly as they are. Each new app ships its **own** `Application` and its **own** supervision tree.
  Any future Portal touchpoint is `echo_bot` calling Portal's **public facade** (`Portal.courses_of/1`,
  `Portal.deliver_lesson/2`, ‚Ä¶) as a **read-only consumer** ‚Äî never editing Portal ‚Äî and that integration is deferred to
  a later rung and is optional.
- **The games web server is a NEW server, never `portal_web`.** The Svelte/Vite game is served and its WebSocket carried
  by a **new** Elixir web app chosen by the decision matrix below ‚Äî not by `PortalWeb.Endpoint`, not by adding a route or
  a child to `portal_web`. `portal_web` is Phoenix above the Portal boundary and stays scoped to Portal.
- **Platform-adapter port.** The engine core depends only on the platform-adapter behaviour, never on a concrete
  platform. Telegram lives behind the port from F10.1 (wrapping the vendored ex_gram copy). Adding a platform is
  implementing the behaviour, not editing the core.

## "Thin but robust" for the engine

Each rung is a narrow vertical slice ‚Äî usually one capability, end to end ‚Äî built to production quality, not a prototype.
Concretely:

- **Supervised and standalone.** The engine runs under its own `EchoBot.Application`, so a bot crash restarts in
  isolation and never touches any other umbrella app. `echo_bot` boots whether or not Portal is running. `echo_games`
  likewise runs under its own supervisor.
- **Config-driven.** A bot exists because a YAML file declares it; the engine reads the config and starts the supervised
  bot. No bot is hard-coded in the engine core.
- **Platform-abstracted.** The engine speaks the platform-adapter behaviour; the Telegram adapter (wrapping the vendored
  ex_gram copy) is one implementation. The core invents no Telegram-only vocabulary.
- **Harnessed without a live platform.** Tests drive the handler and the updater with a **fake/stub updater** (the
  ex_gram `:noup` analog ‚Äî a no-op updates source) and feed constructed updates, asserting the reply rendered, with no
  live Telegram. This is the engine's "wide pure base plus one process test."
- **Idempotent.** Update handling tolerates a re-delivered update (Telegram can resend), so the same message never
  produces a duplicate effect.

Every rung ships behind the same Definition-of-Done gates the Portal specs use, and the per-iteration table below names
the demo and the harness for each.

## The vendored ex_gram fork (the F10.1 ownership contract)

`echo_bot` carries an **internal, vendored copy** of ex_gram's Telegram-API code ‚Äî the low-level client (the Bot API
method calls and the HTTP transport), the update/message decoders, and the updater shape it wraps ‚Äî under a path inside
the app (recommended: `apps/echo_bot/vendor/ex_gram/`; the F10.1 triad fixes the exact path). Two governance documents
accompany the vendored tree and are **F10.1 deliverables** (the builder authors them; this roadmap specifies their
required content):

- **`apps/echo_bot/vendor/ex_gram/README.md`** ‚Äî states what the directory is and where it came from: a vendored copy of
  ex_gram ([github.com/rockneurotiko/ex_gram](https://github.com/rockneurotiko/ex_gram)), its upstream version at the
  time of vendoring, the original Beer-Ware license preserved, and which parts were taken (the Telegram-API client +
  decoders + updater) versus what `echo_bot` adds around them (the `EchoBot.Platform` port and the adapter wrap). It
  notes that the wrap, not the copy, is the long-lived boundary, and that later rungs replace the copy's internals behind
  the unchanged port.
- **`apps/echo_bot/vendor/ex_gram/CLAUDE.md`** ‚Äî the agent directive for working inside the vendored tree. It states:
  this is a **vendored fork maintained by the ex_gram team**; the Operator is the ex_gram team, tasked by the main
  maintainer to build next-generation bot modules; agents **may Read, Write, and modify this code directly**, treating it
  as owned source; changes are **not** submitted as upstream PRs to ex_gram main ‚Äî there is no fork-back obligation. It
  also states the one constraint that survives ownership: the engine core reaches this code **only** through
  `EchoBot.Platform.Telegram`, never directly, so the platform-adapter boundary stays intact even as the vendored
  internals are rewritten.

The propagation clause every brief carries applies to both files: no gendered pronouns for agents; no perceptual or
interior-state verbs; no first-person narration.

## The bot config YAML ‚Äî schema v1.0 (approved)

The per-bot YAML definition is **version 1.0**, approved with five keys. The `version` field is what makes the schema
forward-versionable: a later schema change (an updater-method key, rate-limit settings) bumps the version and the loader
selects the matching reader.

| Key | Required | v1.0 meaning |
|---|---|---|
| `version` | yes | the schema version string, `"1.0"` at this rung; the loader reads it first and validates the rest against the matching schema. |
| `name` | yes | a stable identifier for the bot (used in the supervisor child id and logs). |
| `platform` | yes | selects the platform adapter; `telegram` selects `EchoBot.Platform.Telegram` at this rung. |
| `token_env` | yes | the **name** of the environment variable the bot token is read from at boot ‚Äî **never** the token itself, and the token never appears in the YAML. |
| `handler` | yes | the module that routes this bot's commands (matches `/start`, `/help`, ‚Ä¶). |

A minimal v1.0 file:

```yaml
version: "1.0"
name: echo_demo_bot
platform: telegram
token_env: ECHO_DEMO_BOT_TOKEN
handler: EchoBot.Handlers.Demo
```

F10.1 reads **one** such file; F10.2 reads **many** (a config directory, one definition per file). Later rungs add
**optional** keys (an updater-method key when webhook lands, rate-limit settings when robustness lands) under a bumped
`version`, without breaking the v1.0 shape.

## The games slice ‚Äî `echo_games`, the Svelte/Vite rewrite, and the web-server choice

F10.3 opens the minimal valuable game. Three pieces land together, all isolated from Portal.

**`apps/echo_games` ‚Äî the game backend.** A new umbrella app (OTP app `:echo_games`, namespace `EchoGames`) holding the
backend for "–≠–º–æ–¥–∂–∏: –ù–∞–π–¥–∏ –ø–∞—Ä—É" (Russian-language targeted ‚Äî Telegram's major audience). It owns the game logic, the
per-session state, the scoring, the WebSocket push path, and the **Telegram-payments seam**. The mechanics it models are
exactly those in the current `html/game.html`:

- **Three levels**, sized `4√ó3 ‚Üí 4√ó4 ‚Üí 6√ó4`, each a deck of emoji **pairs** drawn from a fixed pool; levels 2 and 3 are
  timed (60 s, 80 s); level 3 reshuffles the still-hidden cards on an interval.
- **Scoring**: a base score per matched pair, a combo bonus that grows with consecutive matches, a penalty per mismatch,
  and a time bonus for remaining seconds; **stars** (up to three) awarded per level against score thresholds.
- **Per-session state**: the deck, the flipped/matched cards, the current combo and best combo, the pairs found, the
  per-level score breakdown, and the timers ‚Äî held server-side in `echo_games` so the authoritative score lives on the
  BEAM (the precondition for a monetizable game), with the WebSocket pushing state to the Svelte client.

The current page keeps all of this in browser JavaScript; the rewrite moves the **authoritative** game state to
`echo_games` and makes the frontend a view over server-pushed state, so scoring cannot be forged client-side ‚Äî the seam a
paid/monetized game requires.

**The Svelte 5 (Runes) + Vite frontend.** `html/game.html` is rewritten as a Svelte 5 app using **Runes** for reactive
state and **Vite** for the build, producing a static bundle (`index.html` + hashed JS/CSS assets) the games web server
serves. The visual design ‚Äî the warm cream/coral/sage palette, the card-flip 3D animations, the HUD, the combo pulse, the
timer bar, the result modals, the Russian copy ‚Äî is preserved from the current page; the logic is split so the
authoritative rules run in `echo_games` and the Svelte client renders state and sends intents (card taps) over the
WebSocket.

**The games web server ‚Äî decision matrix.** The game needs an Elixir web server that (a) serves the Vite static bundle,
(b) carries a WebSocket for live state, (c) leaves a clean fit for the Telegram **Game** launch and later **payments**
integration, (d) stays small and framework-light in the engine's ethos, and (e) is **isolated from `portal_web`** (a new
server, never a `portal_web` route). Three candidates were weighed against six criteria. (`portal_web` itself is **not** a
candidate ‚Äî extending it would breach the no-touch boundary; it appears only as the isolation baseline.)

| Criterion (what it must do) | **A. Raw Bandit + WebSock/WebSockAdapter + Plug** | **B. New minimal Plug app (Plug.Router on Bandit)** | **C. New Phoenix app (Channels)** |
|---|---|---|---|
| WebSocket support | ‚úì `WebSock`/`WebSockAdapter` upgrade on a Bandit/Plug route ‚Äî already in the umbrella lock (`websock 0.5.3`, `websock_adapter 0.5.9`) | ‚úì same WebSock upgrade inside a `Plug.Router`; one named handler module | ‚úì Phoenix Channels + socket, the richest model, but the heaviest |
| Static-asset serving (Vite bundle) | ‚ñ≥ wire `Plug.Static` yourself (the F6.5.5 prefix-strip footgun is a known hazard) | ‚úì `Plug.Static` mounted in the router, explicit and small | ‚úì `Plug.Static` via the endpoint, generator-shaped |
| Telegram Game + payments fit | ‚úì direct control of the HTTP surface (game launch URL, payment webhook route) with no framework indirection | ‚úì same direct control; routes are plain `Plug.Router` clauses | ‚úì controllers/routers handle it, but adds Phoenix surface to learn around the seam |
| Footprint (deps + concepts) | ‚úì smallest ‚Äî Bandit + Plug + WebSock, no new framework | ‚úì small ‚Äî a thin `Plug.Router` over the same deps | ‚úó largest ‚Äî a second Phoenix app, endpoint, channels, generators |
| Fits the framework-free engine ethos | ‚úì matches the framework-free posture of the F5 core | ‚úì matches it, with a slightly more ergonomic router | ‚ñ≥ Phoenix is a framework; portal_web already owns that role |
| Isolation from `portal_web` | ‚úì a wholly separate Bandit listener on its own port and supervisor | ‚úì a wholly separate app + listener + supervisor | ‚úì separate, but two Phoenix apps invite accidental coupling/config overlap |

**Recommendation: B ‚Äî a new minimal Plug app (`Plug.Router` on Bandit, with `WebSock`/`WebSockAdapter` for the live
socket), served by its own listener under the `echo_games` (or a sibling `echo_games_web`) supervisor.** It carries the
WebSocket on deps already in the umbrella lock, serves the Vite bundle with an explicit `Plug.Static` mount, gives direct
control of the HTTP surface for the Telegram Game launch URL and the future payments webhook, stays small and
framework-light in the engine's ethos, and is fully isolated from `portal_web` on its own port and supervisor. Option A
is the same stack with a hand-wired router rather than `Plug.Router`; B is preferred for the ergonomics of declared
routes at no real footprint cost. Option C (a second Phoenix app) is rejected: it duplicates the framework `portal_web`
already owns, is the heaviest footprint, and two Phoenix apps invite the config/coupling overlap the no-touch boundary
exists to prevent. **This server choice is an architecture fork ‚Äî recorded here with a recommendation; the Operator
confirms it before the F10.3 triad fixes it.**

**The payments seam (a forward concern carried from F10.3).** The first game ships **WebSocket** live state and enough of
the game to be valuable, and opens the path to **monetized games via Telegram payments**. F10.3 carries the seam ‚Äî the
game-launch flow (the bot sending a Telegram **Game**, the game served by the chosen web server), a server-authoritative
score (the precondition for paying for outcomes), and a reserved route on the games server for a future Telegram payment
webhook ‚Äî without building the full payment flow. The payment integration itself (invoices, the `successful_payment`
update, validation) is a later rung sharpened once the playable slice earns feedback. Telegram's payment and game
surfaces are the Bot API's ([core.telegram.org/bots/api](https://core.telegram.org/bots/api),
[core.telegram.org/bots/payments](https://core.telegram.org/bots/payments)).

## Feature abstracts ‚Äî the F10 ladder

The first three rungs are detailed; the rest are abstracts, sharpened into triads when the shipped slice earns feedback.

**F10.1 ¬∑ Start thin: the engine runs one bot from a vendored-fork wrap.** A standalone `apps/echo_bot` umbrella app with
its own `EchoBot.Application`; the **vendored copy of ex_gram** plus its `README.md` and `CLAUDE.md` governance files; the
`EchoBot.Platform` behaviour with `EchoBot.Platform.Telegram` **wrapping the vendored copy** as adapter #1 (polling
updater, fake-updater for tests); a YAML **v1.0** config loader that reads **one** bot definition; a handler/router
matching `/start` and `/help` and answering static text. No Portal, no multi-bot, no games. *Ships:* a live bot on day
one, proving the engine skeleton, the vendored-wrap path, the YAML v1.0 load path, the adapter port, and the supervision
and update loop. *Who:* Operator, Developer.

**F10.2 ¬∑ Multi-bot: N YAML bots under one engine.** Generalize the loader to read **many** YAML v1.0 files (a config
directory) and start a supervised bot per definition under a `:one_for_one` engine supervisor, each bot with its own
token and handler. A second bot is added by dropping a YAML file, not by editing the engine. This is where "multibot"
actually lands. *Ships:* run several bots from config in one release; one bot's crash never disturbs another. *Who:*
Operator, Developer. *Spec (groomed backlog):* [`f10.2.md`](f10.2.md) ¬∑ [`f10.2.stories.md`](f10.2.stories.md) ¬∑
[`f10.2.llms.md`](f10.2.llms.md) ‚Äî opening `[RECONCILE]` re-grounds the multi-file loader and the `:one_for_one` engine
supervisor against F10.1's as-built `EchoBot.Config` + `EchoBot.Supervisor` when F10.1 ships.

**F10.3 ¬∑ The minimal valuable game: `echo_games` + Svelte/Vite + WebSocket + the payments seam.** Scaffold
**`apps/echo_games`** (the "–≠–º–æ–¥–∂–∏: –ù–∞–π–¥–∏ –ø–∞—Ä–∞" backend ‚Äî server-authoritative game logic, per-session state, scoring,
the WebSocket push path, the payments seam); rewrite `html/game.html` as a **Svelte 5 (Runes) + Vite** frontend; stand up
the chosen **isolated Elixir games web server** (Plug.Router on Bandit + WebSock, recommended ‚Äî Operator-confirmed)
serving the Vite bundle and carrying the live socket; wire a bot (from F10.1/F10.2) to **launch the game** via a Telegram
**Game**; and **open the Telegram-payments seam** (server-authoritative score + a reserved payment-webhook route),
without building the full payment flow. *Ships:* the first playable, monetizable slice ‚Äî a real game launched from a bot,
its state live over WebSocket, its score authoritative on the BEAM. *Who:* Operator, Developer, Learner. *Spec (groomed
backlog):* [`f10.3.md`](f10.3.md) ¬∑ [`f10.3.stories.md`](f10.3.stories.md) ¬∑ [`f10.3.llms.md`](f10.3.llms.md) ‚Äî opening
`[RECONCILE]` re-grounds against F10.1's adapter/supervisor and F10.2's multi-bot surface, and flags the games web
server (Plug.Router on Bandit + WebSock, recommended) as the one **open architecture fork** the Operator confirms
before this triad's build.

**F10.4 ¬∑ Telegram surface: commands, messages, keyboards.** Fill out the wrapped Telegram client and the handler/router
framework ‚Äî command routing, free-text messages, sending replies with formatting and inline keyboards, the message/update
shapes the engine parses ‚Äî owned by `echo_bot` via the vendored copy, tracking the current Bot API. *Ships:* bots with
real conversational surface, not just two static commands. *Who:* Developer.

**F10.5 ¬∑ Webhook delivery for production.** Add a `:webhook` updater behind the platform-adapter port (the engine serves
an inbound webhook endpoint for a bot), keeping polling for dev and tests; the updater is config-selected via a new YAML
key under a bumped schema version, so the choice changes no handler code. *Ships:* production-grade, low-latency delivery.
*Who:* Operator.

**F10.6 ¬∑ Own the wrap: replace the vendored internals behind the port.** Replace parts of the vendored ex_gram internals
with first-party `echo_bot` code behind the **unchanged** `EchoBot.Platform.Telegram` adapter, proving the wrap-then-own
strategy: the boundary held from F10.1, so the swap touches no handler and no engine core. *Ships:* the engine owning more
of its Telegram surface outright, on its own schedule. *Who:* Architect, Developer.

**F10.7 ¬∑ Platform-adapter hardening: prove the port with a second-platform skeleton.** Exercise the platform-adapter
behaviour against a second, minimal platform implementation (even a stub) to prove the core carries no Telegram-only
assumptions, and tighten the behaviour's contract where the second adapter reveals a leak. No production second platform
yet. *Ships:* a port proven to host more than Telegram. *Who:* Architect, Developer.

**F10.8 ¬∑ The full Telegram-payments flow.** Build out the payment seam opened in F10.3: invoices, the pre-checkout and
`successful_payment` updates, server-side validation, and a paid outcome in the game (a level unlock, a cosmetic, a
booster). Monetization end to end. *Ships:* a game a learner can pay inside, validated on the BEAM. *Who:* Operator,
Learner.

**F10.9 ¬∑ Optional Portal-consumer integration.** Let a bot answer from the Portal by calling Portal's **public facade**
as a **read-only consumer** ‚Äî for example `/courses` rendering `Portal.courses_of/1`. `echo_bot` reaches Portal only
through its public functions and renders only the closed `%Portal.Error{}` set; it edits no Portal code and adds no child
to `Portal.Application`. Optional and deferred: the engine and the game are fully useful without it. *Ships:* bots that
surface Portal content, with Portal still untouched. *Who:* Learner, Architect.

**F10.10 ¬∑ Robustness + notifications + the second-platform seam.** The hardening tail: rate limits and retries (`429`
with `retry_after`, a closed error set rendered exhaustively); an event-driven push path (a scheduler or a PubSub topic
drives an outbound message); and promoting a second real platform adapter behind the port hardened in F10.7, with no
engine-core change. *Ships:* an engine that stays up and right under load, messages proactively, and is genuinely
multi-platform. *Who:* Architect, Operator.

Each iteration is one PR-sized increment: a spec triad, the slice built against it, a green harness, a demo, and a short
feedback note that may edit the spec before the next rung starts.

## Seams & open decisions

- **The games web server (an architecture fork ‚Äî F10.3).** The decision matrix above weighs raw Bandit + WebSock (A), a
  new minimal Plug app (B), and a new Phoenix app with Channels (C). *Recommendation:* **B**, a new minimal `Plug.Router`
  app on Bandit with `WebSock`/`WebSockAdapter` for the socket, on its own port and supervisor, fully isolated from
  `portal_web`. **Operator's call** before the F10.3 triad fixes it. (`portal_web` is never a candidate ‚Äî extending it
  breaches the no-touch boundary.)
- **The vendored ex_gram path (a layout choice ‚Äî F10.1).** The recommended path is `apps/echo_bot/vendor/ex_gram/`, with
  the `README.md` + `CLAUDE.md` governance files alongside it. *Recommendation:* keep the vendored copy under
  `apps/echo_bot/vendor/` so it travels with the app and reads plainly as a vendored copy, not a hex dependency. **The F10.1 triad fixes the
  exact path and the subset of ex_gram modules vendored.**
- **The YAML v1.0 schema (approved).** The five keys ‚Äî `version`, `name`, `platform`, `token_env`, `handler`, all
  required ‚Äî are approved for v1.0. Later rungs add optional keys under a bumped `version`. **Settled, not open;** listed
  here as the reference the triads derive from.
- **`echo_games` app shape vs a sibling `echo_games_web` (F10.3).** The game backend (`echo_games`) and the games web
  server can live in one app or split into `echo_games` (logic) + `echo_games_web` (the Plug/WebSock listener), mirroring
  the `portal`/`portal_web` split. *Recommendation:* start with **one** `echo_games` app holding both (the slice is
  small), and split out `echo_games_web` only if the web surface grows. **Operator to confirm at the F10.3 sharpen.**
- **Where the Portal-consumer integration lands.** Placed at **F10.9**, after the engine, multi-bot, the first game, the
  richer surface, webhook, ownership of the wrap, and adapter hardening ‚Äî so the engine and the game prove themselves
  standalone before any Portal touchpoint, and the touchpoint is read-only through the public facade. *Recommendation:*
  keep it at F10.9 and keep it optional. **Operator to confirm the placement.**
- **Polling vs webhook.** Polling for dev and tests (and behind a firewall) from F10.1; webhook for production, served by
  the engine behind the platform-adapter port at F10.5. The updater is config-selected (a YAML key under a bumped
  version), so the choice changes no handler code.
- **Config.** Bot tokens are runtime configuration ‚Äî an environment variable named by the YAML and read at boot ‚Äî never
  compiled in and never written into the YAML.

## Conventions

- `echo_bot` is a standalone umbrella app (`apps/echo_bot`, OTP app `:echo_bot`, namespace `EchoBot`) with its own
  `EchoBot.Application`. `echo_games` is a standalone umbrella app (`apps/echo_games`, OTP app `:echo_games`, namespace
  `EchoGames`) with its own supervisor. Neither depends on, modifies, or supervises `portal` or `portal_web`.
- The vendored ex_gram copy lives inside `apps/echo_bot/` (recommended `apps/echo_bot/vendor/ex_gram/`) with its
  `README.md` + `CLAUDE.md`; it is an owned fork, modifiable directly, no upstream PR. The engine reaches it only through
  `EchoBot.Platform.Telegram`.
- The platform-adapter behaviour is the engine's only platform surface; the core names the behaviour, never a concrete
  platform. Telegram is adapter #1 from F10.1.
- Bots are declared in YAML **v1.0** (`version`, `name`, `platform`, `token_env`, `handler`); the token is referenced by
  env-var name and read at boot, never stored in the file.
- The games web server is a NEW server (recommended Plug.Router on Bandit + WebSock), on its own port and supervisor,
  never `portal_web`. The Svelte 5/Vite bundle is static assets that server serves; the game's authoritative state lives
  in `echo_games`.
- Tests use a fake/stub updater (the ex_gram `:noup` analog) so no live platform is contacted; a handler test feeds a
  constructed update and asserts the rendered reply.
- A+ quality gates and Writerside-friendly markdown throughout: prose over heavy formatting, clean voice, balanced
  fences, resolving links.
