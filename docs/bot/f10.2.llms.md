## Pre-build reconcile (discharge BEFORE the F10.2 build opens)

F10.1 is specced but not built. The two PENDING `[RECONCILE]` rows in [`f10.2.md`](f10.2.md) are pre-build reconciles —
discharge them by probing the **realized** F10.1 code the moment F10.1 ships, before writing any F10.2 code:

- **`EchoBot.Config` return shape.** Read `apps/echo_bot/lib/echo_bot/config.ex`. Fix whether the single-file loader
  returns a **struct** or a **map**, the exact field names, and the resolved-token field. F10.2's directory loader
  returns a **list of that same value**, one per file — match it field-for-field, add no new per-bot field.
- **`EchoBot.Application` per-bot child spec.** Read `apps/echo_bot/lib/echo_bot/application.ex`. Fix the exact child
  spec F10.1 builds for the single bot (the updater process plus any adapter client/registry process). F10.2 **repeats
  that unit per definition** and sets each child's id from the bot's `name`. Reuse F10.1's child spec; do not redesign
  the per-bot process shape.
- **The loader error convention.** Read how F10.1's loader reports a bad single file (a tagged tuple? a raise? a
  `{:error, reason}`?). F10.2's per-file validation and duplicate-name rejection (D2) **reconcile to that convention**;
  the recommended boot posture is fail-fast (a misconfigured bot is an operator error worth surfacing loudly).

If F10.1's realized shape diverges from the names this brief assumes, the body (`f10.2.md`) is authoritative on
**intent** (a directory loader returning a list; a per-bot isolated fan-out; unique `name`-derived child ids); adapt the
**names** to F10.1's reality and keep the intent.

## Requirements

- **F10.2-R1** — `EchoBot.Config` reads **every** `*.yaml` file in a configured bot-config **directory** (the directory
  path resolved from umbrella application config, where F10.1 resolved a single file path) and returns a **list** of
  per-bot definitions, one per file, each the **same** value F10.1's single-file loader returns (the v1.0 five-key
  definition with the token resolved from its `token_env`). The per-file read **reuses** F10.1's v1.0 reader; F10.2 adds
  only the directory walk and the list result. [US: F10.2-US1, F10.2-US2]
- **F10.2-R2** — each `*.yaml` is validated against the v1.0 schema **independently**; a file's outcome (a valid
  definition, or a named error for malformed / wrong-version / missing-key) is reported per file (reconciled to F10.1's
  loader error convention; recommended posture: fail-fast at boot). Two definitions that share a `name` are **rejected**
  (the supervisor child id must be unique) — the loader surfaces the duplicate `name`, never starts one and silently
  drops the other. [US: F10.2-US4]
- **F10.2-R3** — `EchoBot.Application.start/2` starts **one supervised, isolated bot subtree per definition** under the
  existing `:one_for_one` `EchoBot.Supervisor` (strategy + name inherited from F10.1 unchanged). Each bot's child spec
  **reuses** F10.1's single-bot child spec; each child's **id is derived from the bot's YAML `name`**. The shape is a
  **static child list** computed at `init/1` from the loaded definitions (D3 recommendation; `DynamicSupervisor`
  deferred to a later runtime-management rung). It does not name, start, or touch `Portal.Application` or
  `PortalWeb.Application`. [US: F10.2-US1, F10.2-US2, F10.2-US3]
- **F10.2-R4** — each bot resolves its **own** token from its **own** `token_env` env var at boot (two bots name two
  env vars; the engine reads each independently) and routes its commands through its **own** `handler` module. An update
  delivered to bot A's updater is handled by bot A's handler with bot A's identity, never crossed with bot B; no shared
  mutable handler state across bots. [US: F10.2-US1, F10.2-US5]
- **F10.2-R5** — a **second** bot (a second v1.0 `*.yaml` file + a second handler module) is added to the bot-config
  directory to prove the drop-a-file path: the engine runs **two** bots with no engine-code edit beyond placing the
  file and the handler; both answer `/start` and `/help` with their **own** static text. [US: F10.2-US2]
- **F10.2-R6** — every bot runs through the **same** `EchoBot.Platform` behaviour and the **same**
  `EchoBot.Platform.Telegram` adapter F10.1 built; the engine core names only the behaviour, never a concrete platform
  or a vendored module; only `EchoBot.Platform.Telegram` names a vendored module. F10.2 adds no adapter, no platform,
  and no vendored code. [US: F10.2-US6]
- **F10.2-R7** — a per-bot supervised-start test: a directory of **N** v1.0 files yields N running supervised bots (the
  loader returns N definitions; `EchoBot.Supervisor` has N children, each id derived from its `name`); each bot's
  handler is reachable via the fake updater (a constructed `/start` to bot A → bot A's welcome text; to bot B → bot B's).
  No live Telegram. [US: F10.2-US5, F10.2-US6]
- **F10.2-R8** — an isolation test: with two bots running, killing one bot's updater process restarts **only** that bot
  (the sibling's process untouched, the engine + Portal supervisors undisturbed); a `/start` to the surviving bot during
  the crashed bot's restart is still answered. [US: F10.2-US3]
- **F10.2-R9** — no module under `apps/echo_bot/lib/` contains the strings `Portal`, `PortalWeb`, `Portal.Engine`, or a
  Portal context/repo name; `apps/echo_bot/mix.exs` does not list `{:portal, in_umbrella: true}` (the Portal no-touch
  boundary, statically checkable). `apps/portal/` and `apps/portal_web/` are unchanged. [US: F10.2-US7]

## Execution topology

**Runtime topology (what exists and how N bots fan out).**

```text
:portal      app  Portal.Application (Supervisor, :one_for_one, Portal.Supervisor)   # apps/portal — UNTOUCHED by F10.2
├─ Portal.Repo · Portal.Store · Portal.EventStore.adapter() · Portal.Engine · Phoenix.PubSub

:portal_web  app  PortalWeb.Application (Supervisor, :one_for_one, PortalWeb.Supervisor)   # apps/portal_web — UNTOUCHED

:echo_bot    app  EchoBot.Application (Supervisor, :one_for_one, EchoBot.Supervisor)    # apps/echo_bot — F10.2 fans the
│                                                                                         # child list out from 1 → N
│   boot: bot-config DIRECTORY ──▶ EchoBot.Config ──▶ [def_A, def_B, …]   # read every *.yaml, one per file, list result
│            (each file: version read+validated; token_env→token; name→child id; duplicate name REJECTED)
│
├─ bot_A subtree   (child id from name "A")   updater_A ──/start,/help──▶ handler_A ──▶ answer(A's welcome/help)
├─ bot_B subtree   (child id from name "B")   updater_B ──/start,/help──▶ handler_B ──▶ answer(B's welcome/help)
└─ …                                                                       # static text only; no Portal; no game
        (every bot runs through the SAME EchoBot.Platform.Telegram adapter; fake updater under test;
         :one_for_one — killing updater_A restarts ONLY bot_A; bot_B keeps answering throughout)
```

The boundaries: **(1)** the engine core names only `EchoBot.Platform`; every bot polls + replies through the **one**
F10.1 adapter — no new platform surface. **(2)** each bot is its own supervised subtree with a `name`-derived child id,
so `:one_for_one` isolates a crash to that bot. **(3)** `echo_bot` still does not depend on `:portal` and names no
`Portal.*` — the multi-bot engine boots whether or not Portal runs.

**Task topology (build order; each step leaves the umbrella compiling).**

```text
T0 PRE-BUILD reconcile: probe as-built F10.1 (Config return shape · Application child spec · loader error convention)  [pre]
      └─▶ T1 EchoBot.Config: read directory of *.yaml → list of per-bot defs (reuse F10.1 v1.0 reader per file)   [R1]
            └─▶ T2 per-file validation + duplicate-name rejection (reconcile to F10.1 error convention)             [R2]
                  └─▶ T3 EchoBot.Application: fan out child list 1 → N; child id from name; reuse F10.1 child spec  [R3]
                        └─▶ T4 per-bot token + handler dispatch (each bot reads own token_env, routes own handler)  [R4]
                              └─▶ T5 add the SECOND bot: a second *.yaml + a second handler module                  [R5]
                                    └─▶ T6 tests: per-bot supervised-start (N→N) + isolation (kill one, sibling answers) [R7,R8]
                                          └─▶ T7 verify: boot N · drop-a-file · isolation · vendor-wrap grep · no-Portal grep [R6,R9]
```

**Touched files.** Code **inside** `apps/echo_bot/` only, plus the umbrella's shared config (a directory path where
F10.1 named a file path):
- `apps/echo_bot/lib/echo_bot/config.ex` — generalized: read a directory of `*.yaml`, return a list of per-bot
  definitions (reuse F10.1's per-file v1.0 reader).
- `apps/echo_bot/lib/echo_bot/application.ex` — fan the child list out from one bot to N under the existing
  `EchoBot.Supervisor`; each child id from the bot's `name`.
- a **second** handler module (e.g. `apps/echo_bot/lib/echo_bot/bots/<second>.ex`) — the second bot's static handler.
- a **second** YAML definition (e.g. `apps/echo_bot/priv/bots/<second>.yaml`) in the bot-config directory.
- `apps/echo_bot/test/echo_bot/config_test.exs` — extend: a directory of N files → N definitions; a duplicate-`name`
  fixture → rejected; a malformed/wrong-version fixture → reported.
- `apps/echo_bot/test/echo_bot/bot_test.exs` (or a new `multibot_test.exs`) — N supervised bots; per-bot dispatch (A→A,
  B→B); the isolation test (kill one, the sibling answers).
- the `echo_bot` config block in the umbrella `config/config.exs` / `config/test.exs` — the bot-config **directory**
  path (where F10.1 set a file path). Umbrella-level, never `portal`/`portal_web` code.

No file under `apps/portal/` or `apps/portal_web/` is touched. No other app's `mix.exs` and no root `mix.exs` changes.
The vendored ex_gram tree, the platform-adapter behaviour, and the Telegram adapter are inherited from F10.1 **unchanged**
— F10.2 adds no file under `apps/echo_bot/vendor/` and edits neither `platform.ex` nor `platform/telegram.ex`. The
exact module/path names above are this brief's proposal; keep the `EchoBot` namespace, reuse F10.1's `EchoBot.Config` /
`EchoBot.Application` / `EchoBot.Supervisor` / `EchoBot.Platform.Telegram`, and follow the spec's invariants.

## Execution plan — first two stories

A worked trace of how the framework turns a user story into a verified change. Each plan is *story → agent story →
tasks → files → command → gate*; an agent runs the steps top to bottom and stops at the gate.

### Plan A — F10.2-US1 (run several bots from one config directory) · via F10.2-AS0 + AS1 + AS3 + AS4
1. Tasks T0 (pre-build reconcile) → T1 (directory loader) → T3 (fan-out) → T4 (per-bot dispatch).
2. Files: `apps/echo_bot/lib/echo_bot/config.ex`, `apps/echo_bot/lib/echo_bot/application.ex`, the `echo_bot`
   bot-config **directory** path in `config/config.exs` (where F10.1 set a file path).
3. Command: `TMPDIR=/tmp mix compile`, then `iex -S mix` and inspect `EchoBot.Supervisor`'s children (N bot children,
   each id from its `name`); then a manual two-bot live demo (two real tokens in two `token_env` vars).
4. Gate (closes US1): a directory of N v1.0 files boots N supervised bots, each with a `name`-derived child id and a
   token resolved from its own `token_env`; `:one_for_one` / `EchoBot.Supervisor` unchanged from F10.1; the root
   `mix.exs` and both Portal apps unchanged. Requirements satisfied: F10.2-R1, F10.2-R3, F10.2-R4, F10.2-R9.

### Plan B — F10.2-US3 (isolate every bot from its siblings) · via F10.2-AS3 + AS6
1. Tasks T3 (the per-bot supervised subtrees) → T6 (the isolation test).
2. Files: `apps/echo_bot/lib/echo_bot/application.ex`, `apps/echo_bot/test/echo_bot/bot_test.exs` (or
   `multibot_test.exs`), `apps/echo_bot/test/test_helper.exs`.
3. Command: `TMPDIR=/tmp mix test apps/echo_bot/test/`, and the determinism loop
   `for i in $(seq 1 150); do TMPDIR=/tmp mix test || break; done`.
4. Gate (closes US3): with two bots running, killing one bot's updater process restarts **only** that bot under
   `:one_for_one`; the sibling bot's process is untouched and answers a `/start` during the restart window; the engine
   supervisor and the two Portal supervisors are undisturbed. Requirements satisfied: F10.2-R3, F10.2-R8.

Efficiency check: each story is two-to-four files behind one command and one gate, and every branch (the directory walk,
the list result, the duplicate-name rejection, the static fan-out, per-bot dispatch, the isolation kill) is already
specified — so the agent implements rather than decides. The one open design choice (static fan-out vs
`DynamicSupervisor`) is resolved with a recommendation in the body (D3) and is the Operator's to confirm.

### Build notes (umbrella gotchas, carried from the F5/F6/F10.1 arc)

- **`TMPDIR=/tmp mix test`** — the harness tmp overlay can hit ENOSPC and surface as spurious mid-suite ExUnit I/O
  failures; run the suite with `TMPDIR=/tmp`. (Recorded in the F5/F6 retrospectives, carried by F10.1.)
- **The determinism gate is a repeated full-suite loop** — for the process-touching multi-bot suite (boot N bots, kill
  one, assert the sibling answers), run `for i in $(seq 1 150); do TMPDIR=/tmp mix test || break; done` and treat the
  independent loop, not a single green run, as the gate. The isolation test is timing-sensitive (a restart race) — the
  loop is what catches a flaky kill/answer ordering.
- **Reuse F10.1, do not rebuild it** — F10.2 is a generalization: the loader gains a directory walk and a list result;
  the supervisor gains a fan-out; nothing else of F10.1 changes. Do not touch the vendored tree, the platform-adapter
  behaviour, or the Telegram adapter.
- **Static fan-out, not `DynamicSupervisor`** — the bot set is fixed at boot at F10.2 (read from the directory once), so
  a static child list is the simplest shape that satisfies the rung. `DynamicSupervisor` is the natural shape only when
  a later rung adds live add/remove (out of scope). This is recorded as a recommendation (D3) the Operator confirms.
- **No Portal, no games at F10.2** — the engine names no `Portal.*` function and does not depend on `:portal`; the game
  (`echo_games`, Svelte/Vite, WebSocket, payments) is F10.3.
