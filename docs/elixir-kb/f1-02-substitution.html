The substitution model.

In algebra you evaluate f(5) by writing 5², then 25 — you substitute, then simplify. Functional programming makes that the definition of how a program runs: replace a call with its body, replace parameters with arguments, and reduce until a value is left. This lesson makes the model precise, and names the property — referential transparency — that makes it sound.

№ 02The model

Replace equals with equals

The substitution model is a rule for evaluating an expression. To apply a function, take its body, replace each parameter with the corresponding argument, and continue reducing. Repeat until nothing is left to reduce — that final form is the value. For a function $f$ defined by $f(x) = x^2$, evaluating $f(5)$ means writing $5^2$, then $25$. Nothing more mysterious is going on when pure code runs.

Take two definitions, square(x) = x * x and sum_of_squares(a, b) = square(a) + square(b). To evaluate sum_of_squares(3, 4), substitute the body, then keep reducing:

sum_of_squares(3, 4) → square(3) + square(4) → (3·3) + (4·4) → 9 + 16 → 25

Every arrow is one substitution of an equal for an equal. This is the most exact debugging tool you own: you can run the program in your head, on paper, and get precisely what the machine gets — provided the functions are pure. The interactive below does it one step at a time.

№ 03Evaluation order

When are the arguments evaluated?

There is one real choice in the model: when to evaluate an argument. Reducing square(3 + 4) can go two ways.

▸Applicative order — Elixir

Evaluate the argument to a value first, then substitute:

square(3 + 4) → square(7) → 7 * 7 → 49

▹Normal order

Substitute the unevaluated argument first, evaluate later:

square(3 + 4) → (3+4) * (3+4) → 7 * 7 → 49

For pure functions both reach the same value — that is the Church–Rosser property. They differ only in how much work happens (normal order can evaluate 3 + 4 twice) and in how non-terminating arguments are treated. Elixir is eager: arguments are evaluated once, left to right, before the call. Keep that in mind; it is why a crash inside an argument happens before the function ever runs.

№ 04Referential transparency

The property that makes it sound

Substituting equals for equals is only valid when an expression really does equal one fixed value. That condition has a name. An expression is referentially transparent when it can be replaced by its value anywhere in the program — and the value by the expression — without changing what the program means. square.(5) and 25 are interchangeable; swap one for the other as often as you like.

A function is pure when it is deterministic (same input, same output, from F1.01) and free of side effects (it reads and writes nothing outside its arguments and result). A call to a pure function is referentially transparent. That is the whole precondition for the substitution model: pure means substitutable. With a pure function the two sides are equal, so you may replace the call with its value. With tick() — a counter that returns a new number each call — they differ: f + f makes two calls, 2 × f makes one. There is no single value to substitute, so the substitution model does not apply. Reading a clock, generating a random number, or mutating shared state all break transparency the same way.

In Elixir

Binding, and what you can replace

In Elixir = binds a name to a value; pure expressions are interchangeable with their values.

# = binds (it is a match, not assignment); x now stands for 5
x = 5
x * x                       # => 25

# square.(5) and 25 are interchangeable — square is pure
square = fn n -> n * n end
square.(5) + square.(5)     # => 50
2 * square.(5)              # => 50   ← same, because square is pure

# so you can extract a subexpression to a name, or inline it, freely:
a = square.(5)
a + a                       # => 50   (identical to the line above)

# NOT referentially transparent — each call differs:
:rand.uniform(6) + :rand.uniform(6)   # two rolls
2 * :rand.uniform(6)                  # one roll — not the same

One Elixir wrinkle: = is the match operator, not assignment, and a name may be re-bound — x = 1 then x = 2 is allowed. But that never mutates a value; it points the name x at a new, immutable value. Within a single expression a bound name stands for one fixed value, which is exactly what substitution needs. Rebinding is covered in F1.04; for now, treat a bound name as a stand-in you can replace with its value.

Why it matters

What transparency buys you

This is the property the rest of the course quietly spends.

Because a pure expression equals its value, you get a set of powers for free. You can reason locally — understand a piece of code without tracing the whole program. You can refactor without fear — extract a subexpression into a variable, or inline it, and the meaning is guaranteed unchanged. You can memoize — compute once and reuse, since the answer never changes. And you can parallelize — run independent pure calls on different cores with no locks, because none can interfere with another.

↦From F1.01

Determinism — same input, same output — is what makes a call equal one value. Transparency is determinism, cashed out as a rule you can apply.

∘Toward F1.03

Once calls are values you can substitute, you can chain them safely — feed one result into the next. That is composition, and the Elixir pipe |>.

Hold the image from the stepper: a program is an expression, and running it is reduction. Everything functional programming offers — testability, concurrency, the pipe — is built on being allowed to replace equals with equals.

What to carry forward

  • The substitution model: evaluate a call by replacing the function body's parameters with the arguments and reducing, until you reach a value. It is algebra's "plug in and simplify."
  • Referential transparency: a pure expression can be replaced by its value anywhere, and back. That is what makes substitution valid — pure means substitutable.
  • Purity (deterministic + no side effects) is the precondition. Impure calls — random, clock, mutation — break it: f() + f() ≠ 2 × f().
  • Elixir is eager (applicative order): arguments are evaluated once, left to right, before the call.
  • The payoff: reason locally, refactor by extract/inline without fear, memoize, and parallelize.
