# A3 · The roadmap layer — user stories

> The acceptance for chapter **A3**, told as user stories at three levels (chapter / module / dive) for three roles:
> the **learner** (who takes the course), the **developer** (who applies the method to their own project), and the
> **Claude Agent** (the Author who builds the pages). Each story is Connextra ("As a … I want … so that …") with a
> Given/When/Then and acceptance criteria. A module is done when its stories' criteria hold and its pages pass the ten
> gates (see [`a3.llms.md`](a3.llms.md)). The roles: a learner reads and clicks; a developer takes the technique to
> their repo; the Claude Agent builds the module from this file + the brief.

## Chapter A3 — the promise

**As a learner**, I want to understand the roadmap layer, so that I can plan a backlog's delivery as thin, robust
increments rather than a flat to-do list.
- **Given** I have finished A2 (a value ladder of stories), **when** I read A3, **then** I can write a `roadmap.md`
  that orders the work into milestones and points each rung at a spec.
- **Acceptance:** the chapter lands the principle on the real F6 `phoenix.roadmap.md`; every module closes a
  principle→Portal-practice `.bridge`.

**As a developer**, I want to see a real roadmap and the techniques behind it, so that I can roadmap my own chapter.
- **Given** the worked F6 exemplar, **when** I read A3.3 and A3.5–A3.7, **then** I can name what a roadmap line carries
  and order rungs into shippable milestones for my project.
- **Acceptance:** every technique is shown on the real Portal artifact, citable and reproducible.

**As a Claude Agent (Author)**, I want a spec + stories + brief for A3, so that I can build its modules to A+ without
inventing anything.
- **Given** [`a3.llms.md`](a3.llms.md) + this file + a model page, **when** I build a module's hub + dives, **then**
  all ten `jonnify-cms` gates pass and no Portal API is invented.
- **Acceptance:** `STATUS: PASS` on every page; the gate-invisible checks (clamp, route-tag, real Sources, `node
  --check`, no duplicate ids) hold.

## A3.1 · Agile, distilled — `/roadmap/agile-distilled`

**As a learner**, I want the handful of Agile principles that actually drive the workflow, separated from the ceremony,
so that I am not lost in process for its own sake.
- **Given** the Agile values and principles, **when** I use the A3.1 interactive, **then** I see which principles the
  Author/Operator loop actually relies on (working software in short cycles, responding to change, inspect-and-adapt)
  and which ceremony it drops.
- **Acceptance criteria:** dives cover *the values* → *inspect-and-adapt* → *what we keep vs drop*; the hero interactive
  classifies a principle as load-bearing or ceremony over a fixed dataset; the `.bridge` lands the loop on the Portal.

**As a developer**, I want to map each kept principle to a concrete move, so that I can run an Agile loop with one
agent and one reviewer.
- **Given** the distilled set, **when** I read the bridge, **then** each principle names its move in the loop.

## A3.2 · Extreme Programming for small batches — `/roadmap/xp-small-batches`

**As a learner**, I want to see why small batches lower risk and raise learning, so that I default to small releases
and continuous feedback.
- **Given** a batch-size control, **when** I shrink the batch, **then** the readout shows risk and feedback latency
  fall and learning-per-cycle rise, computed over a fixed model.
- **Acceptance criteria:** dives cover *small releases* → *incremental design* → *continuous feedback*, re-cast for an
  Author/Operator pair; the main interactive proves "an agent makes small batches cheap" on the F6 rung cadence.

**As a Claude Agent**, I want the XP practices stated as the cadence I build to, so that each rung I ship is a small,
demoable release.
- **Given** the F6 rung sequence, **when** I plan a build, **then** I ship one rung, demo it, and take feedback before
  the next.

## A3.3 · Anatomy of a roadmap.md — `/roadmap/roadmap-anatomy`

**As a developer**, I want to read a complete, real `roadmap.md` part by part, so that I can write one for my own
chapter.
- **Given** the F6 `phoenix.roadmap.md`, **when** I select a part (what-it-delivers · start/end handoff · architecture
  decision · milestones · the per-iteration table · open decisions), **then** the readout explains that part from the
  real file.
- **Acceptance criteria:** dives cover *what a roadmap carries* → *the per-iteration table* → *the open decisions*; the
  hero interactive decomposes one real `roadmap.md` line into {rung, milestone, the spec it points at, behaviour = none};
  every quote is verbatim from `docs/elixir/specs/phoenix/phoenix.roadmap.md`.

**As a learner**, I want to see that a roadmap orders work without defining behaviour, so that I keep delivery and
definition separate.
- **Given** the anatomy, **when** I re-order the rungs, **then** the "spec edits" counter stays at 0.

## A3.4 · Thin but robust — `/roadmap/thin-but-robust`

**As a learner**, I want to tell a thin-but-robust slice from a flimsy prototype, so that every increment I ship is
production-grade, not a draft to be redone.
- **Given** a set of candidate increments, **when** I classify each, **then** the readout marks a slice
  *thin-but-robust* only when it is a vertical thread built over the facade — contract-guarded, harnessed, gated, and
  leaving the system live — and flags a shallow or untested one as *flimsy*.
- **Acceptance criteria:** dives cover *thin: vertical, not shallow* → *what robust adds* → *always live*; grounded on
  the real F6 "thin but robust for the web" section; the `.bridge` lands robustness on the Portal master invariant
  (the web calls only the facade and renders only the closed error set).

**As a developer**, I want the checklist of what makes a slice robust, so that I can hold my own rungs to production
quality rather than ship a prototype.
- **Given** the F6 robustness properties (over the facade · harnessed · verified and safe · supervised · always live),
  **when** I read the bridge, **then** each property names the concrete move it becomes on a rung.

## A3.5 · Milestones and iterations — `/roadmap/milestones`

**As a learner**, I want to group rungs into shippable milestones and run each through one iteration, so that the most
valuable, least risky thread ships first.
- **Given** the F6 nine-rung ladder, **when** I assign rungs to the three milestones (ship the catalog · make it live ·
  ship to users), **then** the readout shows each milestone is a usable capability boundary and the order respects
  dependency and product priority.
- **Acceptance criteria:** dives cover *shippable milestones* → *the iteration loop (Ships / Demo / Harness / Feedback)*
  → *sequencing the ladder*; every milestone and rung is verbatim from `phoenix.roadmap.md`; re-grouping rungs edits the
  roadmap, never a spec.

**As a developer**, I want the per-iteration columns, so that I can size each rung as a PR-sized increment with a demo
and a harness.
- **Given** a rung, **when** I fill in Ships / Demo / Harness / Feedback, **then** the increment is demoable and
  accepted before the next begins.

## A3.6 · The program roadmap — `/roadmap/program-roadmap`

**As a learner**, I want to see how whole chapters sequence into a program, so that I can plan a system with many
surfaces over one core.
- **Given** the Portal program (an engine core, with web and bot surfaces), **when** I use the interactive, **then** the
  readout shows each surface is a roadmap of its own, all over the one `Portal` facade, shipping on independent cadences.
- **Acceptance criteria:** dives cover *roadmap of roadmaps* → *one core, many surfaces* → *independent cadence*;
  grounded on the program roadmap (`portal.roadmap.md`) and the F6 (web) / F10 (bot) sibling roadmaps; the master
  invariant is named as the seam that decouples the surfaces' cadence.

**As a developer**, I want to know what lets two surfaces ship independently, so that I can add a surface without
disturbing the core.
- **Given** the shared facade, **when** I read the bridge, **then** "call only the facade" is named as the rule that
  keeps each surface's roadmap independent of the others.

## A3.7 · Tracer bullets and walking skeletons — `/roadmap/tracer-bullets`

**As a learner**, I want to ship a thin end-to-end thread before adding depth, so that I de-risk integration early and
keep the roadmap honest about uncertainty.
- **Given** the F6 ladder, **when** I identify the tracer bullet, **then** the readout names the first rung (the engine
  served as a web app: request → facade → render) as the thread through every layer, shipped before persistence or the
  rendered catalog.
- **Acceptance criteria:** dives cover *end to end first* → *tracer vs prototype (and the walking skeleton)* →
  *deferring breadth*; the `.bridge` contrasts a tracer (a real thin feature, kept and grown) with a prototype (built
  to learn, then discarded).

**As a developer**, I want to distinguish a tracer bullet from a walking skeleton from a prototype, so that I choose the
right first increment.
- **Given** the three, **when** I classify each, **then** the readout distinguishes a tracer (a real thin feature,
  kept), a walking skeleton (end-to-end structure, no feature yet), and a prototype (built to learn, then thrown away).

## A3.8 · Workshop — roadmapping Portal — `/roadmap/workshop`

**As a developer**, I want to turn A2's backlog into Portal's chapter roadmap and its program roadmap, so that I leave
with the delivery plan Part VII executes.
- **Given** A2's value ladder, **when** I run the full A3 sequence (milestones, the iteration table, the tracer, the
  deferred seams), **then** I produce a chapter roadmap whose rungs point at specs and define no behaviour, plus a
  program view of web and bot over one facade.
- **Acceptance criteria:** dives cover *ladder to roadmap* → *choose the tracer* → *the program view*; the worked
  output reproduces the real F6 `phoenix.roadmap.md` (milestones, the iteration table, the open decisions) — invented
  nowhere; re-ordering rungs edits the roadmap, never a spec.

**As a learner**, I want to see the whole layer applied once end to end, so that I can roadmap a chapter of my own.
- **Given** the workshop, **when** I follow it, **then** I can name each part of a `roadmap.md` and produce one for my
  own chapter.

## A3.9 · Glossary, references & where the framework implements each idea — `/roadmap/glossary`

**As a learner**, I want a glossary of every key term with a one-line abstract and a link to where the Portal
implements it, so that I can look a concept up and jump to its real code.
- **Given** the expandable glossary list, **when** I expand a term (e.g. "value ladder", "thin but robust", "master
  invariant", "tracer bullet"), **then** I see its abstract, its source, and the `/elixir` chapter where the framework
  implements it.
- **Acceptance criteria:** the hub is a **UX-friendly expandable list** (an accordion; degrades to a visible list
  without JS, no browser storage); dives are *the glossary*, *the annotated sources*, and *the idea→framework
  crosswalk*; every external Source is a real vetted link; every framework link resolves (`/elixir/...` 200).

**As a developer**, I want the bibliography in one place with abstracts, so that I can read deeper on any technique.
- **Given** the sources dive, **when** I scan it, **then** each source has an abstract and the modules that cite it.

**As a Claude Agent**, I want the glossary to be data-driven from a fixed term list, so that the expandable list is
truthful and degradable.
- **Given** a fixed `TERMS` dataset (term → abstract → source → `/elixir` route), **when** I render the accordion,
  **then** each entry is present in static markup and JS only enhances the expand/collapse.
