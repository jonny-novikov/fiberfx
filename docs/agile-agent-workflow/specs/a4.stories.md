# A4 · The spec layer — user stories

> The acceptance for chapter **A4**, told as user stories at three levels (chapter / module / dive) for three roles —
> the **learner**, the **developer**, and the **Claude Agent** — each Connextra ("As a … I want … so that …") with a
> Given/When/Then and acceptance criteria. A module is done when its criteria hold and its pages pass the ten gates
> (see [`a4.llms.md`](a4.llms.md)). **Parent:** [`a4.md`](a4.md) (the chapter spec).

## Chapter A4 — the promise

**As a learner**, I want to write a specification that proves an increment is done rather than asserting it, so that
"done" is verifiable, not an opinion.
- **Given** I have an A3 roadmap (rungs that point at specs), **when** I work through A4, **then** I can write a spec
  (Goal, Rationale, Scope, Deliverables, Invariants, Definition of Done) and the `.stories.md` that accepts it.
- **Acceptance:** every concept lands on a real Portal triad (`docs/elixir/specs/phoenix/f6.N.md` + `.stories.md`); the
  chapter's own spec (`a4.md`/`a4.stories.md`) reads as a worked example of the artifact it teaches.

**As a developer**, I want the anatomy and the traceability rule, so that I can specify a rung of my own project that a
coding agent can build and I can accept by proof.
- **Given** a feature to build, **when** I apply A4.3 + A4.6, **then** my spec constrains the build without dictating
  the solution, and every deliverable traces to a story, an acceptance check, and an invariant.

**As a Claude Agent (Author)**, I want a spec that reads precisely, so that I can implement it without inventing intent
and self-check against its Definition of Done.
- **Given** a rung spec + stories, **when** I build, **then** I satisfy every Given/When/Then and break no invariant.
- **Acceptance:** `STATUS: PASS` on every page; no invented Portal API; the gate-invisible checks hold.

## A4.1 · Specification by Example — `/spec/by-example`

**As a learner**, I want concrete examples to be the specification, so that ambiguity is removed before a line of code
is written and the documentation cannot drift from the system.
- **Given** an ambiguous requirement, **when** I replace it with worked examples, **then** the readout shows the
  readings collapse from many to one and the example becomes the executable check.
- **Acceptance criteria:** dives cover *examples as specification* → *living documentation* → *removing ambiguity*; the
  hero interactive contrasts an abstract requirement (many readings) with an example (one), over a fixed dataset.

## A4.2 · The triad: spec, stories, agent brief — `/spec/the-triad`

**As a developer**, I want to know which artifact answers which question, so that I do not cram everything into one
document.
- **Given** the three artifacts, **when** I select one, **then** the readout names its question — *what & why & done*
  (spec), *who wants what* (stories), *how to build with proof gates* (brief) — from the real F6 triad.
- **Acceptance criteria:** the interactive maps a question to its artifact over the real `f6.N.{md,stories.md,llms.md}`;
  the `.bridge` shows the course's own `a4.{md,stories.md,llms.md}` as the same triad.

## A4.3 · Anatomy of a spec — `/spec/spec-anatomy`

**As a learner**, I want to read a real spec section by section, so that I can write each part so it constrains the
build without over-specifying the solution.
- **Given** the F6 `f6.1.md` spec, **when** I select a section (Goal · Rationale 5W · Scope · Deliverables · Invariants
  · Definition of Done), **then** the readout explains that section from the real file.
- **Acceptance criteria:** every quote is verbatim from `docs/elixir/specs/phoenix/f6.1.md`; the dives cover the
  sections, the five Ws, and the line between constraining and over-specifying.

## A4.4 · From stories to a .stories.md — `/spec/to-stories`

**As a developer**, I want to derive the stories file from the spec, so that every deliverable is realized by a story
and every story names the invariants it exercises.
- **Given** a spec's Deliverables, **when** I write the `.stories.md`, **then** a Coverage line maps each deliverable to
  the stories that realize it and an INVEST line names the invariants each story exercises.
- **Acceptance criteria:** the interactive builds/checks a Coverage map over a real `f6.N.stories.md`; a deliverable
  with no covering story is flagged.

## A4.5 · Invariants: properties that must always hold — `/spec/invariants`

**As a learner**, I want to tell an invariant from an acceptance check, so that I name the rules an increment must
never break.
- **Given** candidate statements, **when** I classify each, **then** the readout distinguishes an invariant (true for
  every value, always) from an acceptance check (true for a specific scenario), using the real F6 master invariant.
- **Acceptance criteria:** the hero interactive classifies statements over a fixed dataset including
  "the web layer calls only the `Portal` facade and renders only `%Portal.Error{}`" (an invariant) vs a single GWT
  scenario (a check).

## A4.6 · Traceability — correct by definition — `/spec/traceability`

**As a Claude Agent**, I want the chain from deliverable → story → acceptance & invariant → requirement → agent story to
be checkable from the text, so that completion closes by a rule, not a claim.
- **Given** a spec + stories, **when** I walk the chain, **then** each link resolves and the completion rule closes; a
  broken link is flagged as not-done.
- **Acceptance criteria:** the interactive steps the traceability chain over a real triad and reports the closure;
  "correct by definition" ties back to A1.05.

## A4.7 · Workshop — specifying Portal's engine — `/spec/workshop`

**As a developer**, I want to write the specs and stories for the engine chapter, so that I leave with the contracts an
agent will implement in Part V.
- **Given** the engine's deliverables, **when** I run the full A4 sequence (by-example → triad → anatomy → stories →
  invariants → traceability), **then** I produce a spec + `.stories.md` whose traceability chain closes.
- **Acceptance criteria:** the workshop runs the chapter's techniques on the real `pragmatic/` (F5 engine) triads;
  the output is a closed, acceptable spec.
