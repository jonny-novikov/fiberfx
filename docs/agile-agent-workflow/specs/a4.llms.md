## What A4 teaches

How to write a specification that is precise, traceable, and accepted only when proven — *correct by definition*. A4
enters with the A3 roadmap (rungs that point at specs) and leaves the reader able to write a rung's spec and its
`.stories.md`, with a traceability chain that makes completion verifiable from the text alone.

## Grounding (no-invent — the real exemplar)

The worked examples are the Portal's real rung triads under `docs/elixir/specs/`: a spec `phoenix/f6.N.md` (Goal,
Rationale 5W, Scope, Deliverables, Invariants, Definition of Done), a stories file `phoenix/f6.N.stories.md` (Connextra
+ Given/When/Then + INVEST + Coverage), and the contract `specs.approach.md`. Quote them **verbatim**; invent no rung,
invariant, or surface. The only Portal API to name freely is `Portal.ID.generate/1`/`decode/1`; the real F6 surfaces
(`Portal` facade, `%Portal.Error{}`, the f6.N rungs) may be cited as they appear in source. This course's OWN specs
(`a3.md`, `a4.md`, `a4.stories.md`) are a second, self-referential exemplar — A4 can show its own spec as a worked spec.
