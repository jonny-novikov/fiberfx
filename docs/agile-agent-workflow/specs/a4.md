## Goal

After A4, the reader can **write a spec for a rung and the stories that accept it**: a Specification-by-Example spec
(Goal, Rationale, Scope, Deliverables, Invariants, Definition of Done), a `.stories.md` of Connextra stories with
Given/When/Then, and the traceability chain that makes "done" verifiable from the text alone — *correct by definition*,
not asserted. The reader leaves able to specify the Portal's engine chapter (the A4.7 workshop).

## Rationale (5W)

- **Why** — a roadmap (A3) orders work but does not define it; an agent cannot build, and an Operator cannot accept,
  what is not specified precisely. Vibe coding (A1.01) skips the spec; the spec layer is what makes an increment
  buildable by an agent and acceptable by proof rather than opinion.
- **What** — a teaching chapter on the spec layer: Specification by Example (A4.1), the spec triad (A4.2), the anatomy
  of a spec (A4.3), deriving a `.stories.md` (A4.4), invariants (A4.5), traceability and "correct by definition"
  (A4.6), and a workshop specifying Portal's engine (A4.7).
- **Who** — the **learner** (writes a spec), the **developer** (specifies their own rung), and the **Claude Author**
  (both: reads a spec to build, and is taught here how a spec must read to be buildable).
- **When** — Part IV, after the roadmap layer (A3) and before the agent brief and build (A5); a spec is what a roadmap
  line points at and what an agent brief wraps.
- **Where** — served at `/course/agile-agent-workflow/spec`; specs under `docs/agile-agent-workflow/specs/`; the worked
  examples are the Portal's real rung triads (`docs/elixir/specs/phoenix/f6.N.md` + `.stories.md`) and `specs.approach.md`.

## Scope

- **In:** how to write a spec and its stories — the anatomy, examples-as-specification, invariants vs acceptance
  checks, and the traceability chain that closes "done"; all landed on the Portal's real triads.
- **Out:** planning delivery (A3), the agent brief / paste-ready prompt and the implementation (A5), reliability
  techniques (A6), the production exemplar (A7). A4 defines and proves; it does not order, brief, or build.

## Deliverables

- The chapter **landing** `/spec` (built first, as the keystone) and **modules A4.1–A4.7**, each a hub + ≥3 deep-dive
  subpages with two interactives per lesson, grounded on the Portal's real rung triads.
- Every page's **md source of record** under `docs/agile-agent-workflow/content/spec/<module>/`.
- This chapter's own specs as the demonstrated artifact: `a4.md` (this file), `a4.stories.md`, `a4.llms.md`.

## Invariants

- Every page lands a **principle → Portal practice**, reuses the design system, and carries the segmented route-tag,
  the canonical footer, and real Sources links.
- Every spec/stories example is **verbatim** from a real Portal triad (`docs/elixir/specs/phoenix/f6.N.md` /
  `f6.N.stories.md`); no rung, invariant, or Portal API is invented (only `Portal.ID` and the real F6 surfaces).
- A spec **constrains the build without over-specifying the solution**; an invariant ("true for every value, always")
  is kept distinct from an acceptance check ("true for a specific scenario").

## Definition of Done

A module is done when **all ten `jonnify-cms` gates pass** on its pages, the gate-invisible checks hold, the relevant
Given/When/Then in [`a4.stories.md`](a4.stories.md) are satisfied, and the four living views agree. The chapter is done
when A4.1–A4.7 are built and accepted — and, fittingly, when this chapter's own triad (`a4.md` / `a4.stories.md` /
`a4.llms.md`) reads as a spec that passes A4.6's traceability rule.
