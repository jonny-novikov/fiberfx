# A5 · The agent brief & implementation — user stories

> The acceptance for chapter **A5**, told as user stories at three levels (chapter / module / dive) for three roles —
> the **learner**, the **developer**, and the **Claude Agent** — each Connextra ("As a … I want … so that …") with a
> Given/When/Then and acceptance criteria. A module is done when its criteria hold and its pages pass the ten gates
> (see [`a5.llms.md`](a5.llms.md)). **Parent:** [`a5.md`](a5.md) (the chapter spec).

## Chapter A5 — the promise

**As a learner**, I want to write an agent brief and run a Claude agent to build an increment, so that I can turn a
spec into working code with a human in the loop.
- **Given** an A4 spec for a rung, **when** I work through A5, **then** I can write its `.llms.md` (references,
  requirements, topology, agent stories, the implementation prompt) and brief, supervise, and review an agent that
  builds it.
- **Acceptance:** every concept lands on a real Portal brief (`docs/elixir/specs/phoenix/f6.N.llms.md` / `.prompt.md`);
  the chapter's own brief (`a5.llms.md`) reads as a worked `.llms.md`.

**As a developer**, I want the brief's anatomy and the practice of running an agent, so that I can implement a rung of
my own project with a coding agent and review it critically.
- **Given** a spec, **when** I apply A5.2–A5.5, **then** I produce a brief that fixes every decision the spec has
  already fixed and leaves the agent no ambiguity; and I know when to let it run and when to intervene (A5.6).

**As a Claude Agent (Author)**, I want a brief that is references-first and unambiguous, so that I assemble a system
rather than a pile of snippets and self-check against the gates.
- **Given** an `.llms.md` brief, **when** I build, **then** I read the references first, follow the task DAG, and
  satisfy every agent story's acceptance gate.
- **Acceptance:** `STATUS: PASS` on every page; no invented Portal API; the agent is never anthropomorphised in prose.

## A5.1 · Writing for an agent: the llms.txt convention — `/brief/llms-txt`

**As a learner**, I want to see why a brief for an agent differs from documentation for a person, so that I write
links-first, prose-second, every reference exact.
- **Given** a person-doc and an agent-brief of the same content, **when** I compare them, **then** the readout shows
  the brief front-loads exact links and removes narrative the agent does not need.
- **Acceptance criteria:** dives cover *links first* → *every reference exact* → *the shape of a machine brief*;
  grounded on a real `f6.N.llms.md`; cite the `llms.txt` convention.

## A5.2 · References and requirements — `/brief/references-requirements`

**As a developer**, I want numbered, testable requirements each traced to a story and an invariant, so that nothing the
agent must do is left implicit.
- **Given** a spec + stories, **when** I write the requirements list, **then** each requirement names the source the
  agent reads, the story it serves, and the check or invariant it satisfies.
- **Acceptance criteria:** the interactive checks a requirement's traceability over a real `f6.N.llms.md`; an untraced
  requirement is flagged.

## A5.3 · Execution topology — `/brief/execution-topology`

**As a Claude Agent**, I want the runtime shape and the build-order task DAG and the exact files I touch, so that I
assemble a system in the right order.
- **Given** the topology section, **when** I read it, **then** I see the processes/supervision/request-flow, the task
  DAG, and the file list — and can build bottom-up without guessing.
- **Acceptance criteria:** the interactive walks a real F6 topology + task DAG; building out of order surfaces a broken
  dependency.

## A5.4 · Agent stories — `/brief/agent-stories`

**As a learner**, I want the executable counterpart of a user story, so that each unit of agent work has a Directive
and an Acceptance gate that closes it.
- **Given** a user story, **when** I write its agent story, **then** it has a Directive (what the agent does) and an
  Acceptance gate (the check), and the first-two-stories plan proves the path is unambiguous before the agent runs.
- **Acceptance criteria:** the interactive pairs a user story with its agent story over a real triad; a story whose
  acceptance gate is missing is flagged.

## A5.5 · The comprehensive implementation prompt — `/brief/implementation-prompt`

**As a developer**, I want the single prompt an agent runs to build an increment in task order, so that the agent
self-checks against the gates and leaves no decision the spec has not fixed.
- **Given** the brief's parts, **when** I assemble the prompt, **then** it carries the references, the task order, the
  agent stories, and the Definition of Done — a real `f6.N.prompt.md` as the model.
- **Acceptance criteria:** the interactive composes/inspects a real implementation prompt; a prompt missing its
  acceptance gates is flagged incomplete.

## A5.6 · Running Claude agents well — `/brief/running-agents`

**As a developer**, I want the practice around the prompt, so that I brief, supervise, and review an agent and know
when to let it run and when to intervene.
- **Given** an agent at work, **when** I review its output, **then** I read it critically — checking against the spec's
  Definition of Done, not the agent's self-report.
- **Acceptance criteria:** dives cover *briefing* → *supervising* → *reviewing*; grounded in Anthropic's building-
  effective-agents guidance and the course's own fan-out practice (adversarial verification, the gate-invisible checks).

## A5.7 · Pragmatic Programming with Claude Agents — `/brief/the-thesis`

**As a learner**, I want the thesis stated plainly, so that I know where the agent's value is real and where it is not.
- **Given** the pairing, **when** I read A5.7, **then** I see the agent as a fast, tireless implementer of *well-
  specified* thin slices, and the human as the source of judgment, taste, and acceptance.
- **Acceptance criteria:** the interactive shows the compounding (speed from the agent × direction from the workflow)
  and names the failure mode (an under-specified slice); ties back to A1's thesis and A1.05's "correct by definition".

## A5.8 · Workshop — briefing the agent for Portal — `/brief/workshop`

**As a developer**, I want to write the engine chapter's agent briefs and run the build, so that I make a first full
pass from spec to running code.
- **Given** the engine spec + stories, **when** I write the briefs and run an agent, **then** the increment is built
  and verified against the Definition of Done.
- **Acceptance criteria:** the workshop runs the full A5 sequence on the real `pragmatic/` (F5 engine) briefs; the
  output is a verified increment.
