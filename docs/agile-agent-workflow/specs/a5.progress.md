## (A) Chapter narrative ‚Äî the whole arc to the end

A5 is one continuous argument carried across eight modules: **a spec is not code; the brief is the layer that
turns it into code, and running an agent well is the practice around it.** The chapter enters with the A4 spec ‚Äî
correct by definition ‚Äî and leaves the reader able to write an `.llms.md` brief and run a Claude Author through a
first full pass from spec to running code.

The orientation dives already established the frame: `/brief/why` (a spec is not runnable; the brief fixes the
four how-to-build decisions ‚Äî sources, runtime shape, build order, proof gates), `/brief/what` (the five parts of
an `.llms.md` and the eight modules ahead), `/brief/how` (write the brief part by part, run the agent, review
against the spec's Definition of Done ‚Äî never the agent's self-report). The eight teaching modules pick up from
there and go deep.

- **A5.1 ¬∑ the convention** (`/brief/llms-txt`). Establishes *how to write for an agent at all*: links first,
  prose second, every reference exact ‚Äî the `llms.txt` form a brief takes. It leaves the reader able to recognise
  a machine brief and to front-load the exact sources an agent reads before it reads any narrative. **Hands to
  A5.2:** the brief's outer shape, ready to be filled with its first two parts.

- **A5.2 ¬∑ references and requirements** (`/brief/references-requirements`). The first two parts of the brief.
  References come first (the sources the agent reads); requirements are numbered, testable, and each one traced to
  the user story it serves and the check or invariant it satisfies. Grounds on `f6.1.llms.md`'s References list
  and `F6.1-R1‚Ä¶R8`. It leaves the reader able to write a requirements list where nothing the agent must do is
  implicit. **Hands to A5.3:** a set of traced requirements that imply a runtime shape and a build order.

- **A5.3 ¬∑ execution topology** (`/brief/execution-topology`). The third part. The runtime shape (the process /
  supervision tree and how a request flows), the build-order task DAG, and the exact files the agent touches.
  Grounds on `f6.1.llms.md`'s "Execution topology" (the two-supervisor runtime tree, the `T1‚ÜíT7` task DAG, the
  touched-file list). It leaves the reader able to read or write a topology an agent can assemble bottom-up
  without guessing. **Hands to A5.4:** an ordered plan of work whose units now need their executable acceptance.

- **A5.4 ¬∑ agent stories** (`/brief/agent-stories`). The fourth part. The executable counterpart of a user story:
  each agent story carries a **Directive** (what the agent does) and an **Acceptance gate** (the check that closes
  it). Grounds on `f6.1.llms.md`'s `F6.1-AS1‚Ä¶AS4` and the first-two-stories execution plan. It leaves the reader
  able to turn a user story into a unit of agent work with an unambiguous closing check. **Hands to A5.5:** the
  references, requirements, topology, and stories ‚Äî the four parts the prompt assembles.

- **A5.5 ¬∑ the implementation prompt** (`/brief/implementation-prompt`). The fifth part. The single prompt an
  agent runs to build the increment in task order, ending on the verification gates, with the Definition of Done
  as the acceptance checklist. Grounds on `f6.1.llms.md`'s "Comprehensive implementation prompt" (the seven-step
  single-agent prompt) and the `f6.6.prompt.md` / `f6.7.prompt.md` orchestration briefs (the multi-agent variant).
  It leaves the reader able to assemble and inspect a runnable implementation prompt. **Hands to A5.6:** a brief
  that is complete; what remains is the human practice of running it.

- **A5.6 ¬∑ running Claude agents well** (`/brief/running-agents`). The practice around the prompt: brief,
  supervise, review. The reviewer reads against the spec's Definition of Done, not the agent's self-report.
  Grounds on Anthropic's *Building effective agents* / *Claude Code best practices* and the course's own fan-out
  practice (adversarial verification, the gate-invisible checks). It leaves the reader able to know when to let an
  agent run and when to intervene. **Hands to A5.7:** the practice, named, so its value can be stated as a thesis.

- **A5.7 ¬∑ the thesis** (`/brief/the-thesis`). *Pragmatic Programming with Claude Agents*, stated plainly: the
  agent is a fast, tireless implementer of well-specified thin slices; the human is the source of decomposition,
  judgement, and acceptance. It names where the value is real (a well-specified slice) and the failure mode (an
  under-specified slice). Ties back to A1's thesis and A1.05 "correct by definition." **Hands to A5.8:** the whole
  argument, ready to be run end to end.

- **A5.8 ¬∑ workshop** (`/brief/workshop`). Runs the full A5 sequence on the Portal's real engine briefs: write
  the briefs, run the agent, verify the increment against the Definition of Done. It is the chapter's proof that
  the method produces a verified increment, not just a description of one. **Hands to A6:** a built increment for
  reliability to harden.

**Continuity at the chapter seams.** A5 continues from **A4** ‚Äî the spec a brief wraps (correct by definition) ‚Äî
and feeds **A6** ‚Äî the increment a brief produces, which reliability hardens. Every module restates this lineage
once and moves on; no module re-teaches A4's spec content or pre-empts A6's reliability content.
