## The arc and the thesis

The course argues one thing, restated on every page: a Claude agent is a fast, tireless implementer of *well-specified*
thin slices, and the human is the source of decomposition, judgement, and acceptance. Neither no-plan (vibe coding) nor
all-plan (the big-bang spec) ships reliable software; the unit that does is a thin slice of value, specified only
enough, proven before the next begins. A4 lands that thesis on the *spec* layer — where each slice the roadmap orders
becomes a precise, provable specification.

The chapter sits at a known place in the course. A0 framed the method in three questions (why, what, who). A1 made the
case for thin, provable slices, named the Author/Operator loop, drew the roadmap/spec line (A1.04), and fixed what
"done" means — a closure over traced, executed checks (A1.05, "correct by definition"). A2 turned a vision into a
dependency-ordered value ladder of INVEST stories with Given/When/Then — a backlog. A3 planned that backlog's delivery
as a roadmap: thin-but-robust rungs grouped into milestones, each rung a *line that points at a spec and defines no
behaviour*. **A0–A3 are built and accepted; the A4 chapter landing `/spec` is built.** A roadmap, though, orders work
but does not define it. A4 is the **spec layer** — where each rung the roadmap points at is defined precisely enough
that a Claude Author can build it and an Operator can accept it by proof. A5 then briefs the agent and builds it; A6
makes the increment production-grade; A7 runs the whole loop on the Portal end to end. A4 depends only downward on
A0–A3 and feeds forward into A5.

A4 lands the thesis twice on every page, the same way A1–A3 did: the **principle** (Specification by Example, the
spec triad, invariants, the traceability closure) and the **Portal practice** it becomes (a real `f6.N.md` /
`f6.N.stories.md` triad read verbatim). The two-layer separation A1.04 introduced and A3 expanded is the chapter's
spine: the roadmap is *how we deliver*, the spec is *what we build and prove*, and feedback edits the spec — never the
shipped code, never a derived artifact. Several A4 interactives prove a consequence of that separation by holding a
fact at a fixed value (a "spec edits" counter, a deliverable→story coverage count) while a control moves.

**The master invariant (quote it verbatim — the exact form in `phoenix/phoenix.md` and `phoenix/phoenix.roadmap.md`):**

> The web layer calls only the `Portal` facade and renders only the closed `%Portal.Error{}` set. No controller,

**The F6.1 spec, section by section (the A4.3 anatomy — every quote verbatim from `phoenix/f6.1.md`):**

- **Goal** — "After F6.1, the Portal runs as a Phoenix app. A browser request travels through the endpoint's plug

- **Rationale (5W)** — the five bold bullets, e.g. **Why** "F5 produced a correct engine but no way for a human to

**The F6.1 stories (the A4.4 + A4.6 grounding — verbatim from `phoenix/f6.1.stories.md`):** five Connextra stories

deliverable→story map A4.4 builds, and a deliverable absent from it is a deliverable with no covering story.

**The traceability chain and the completion rule (the A4.6 grounding — verbatim from `specs.approach.md`):**

```text
Deliverable (fN.M.md  · FN.M-D#)
   └─ realized by → User story (fN.M.stories.md · FN.M-US#)
        ├─ accepted by → Acceptance criteria (Given/When/Then on the story)
        └─ encodes     → Invariant (fN.M.md · FN.M-INV#)        (named on the story's INVEST line)
   └─ built by   → Agent story (fN.M.llms.md · FN.M-AS#)        [implements FN.M-US#]
        └─ governed by → Requirement (fN.M.llms.md · FN.M-R#)   [US: FN.M-US#]
             └─ proven by → Invariant (FN.M-INV#)  or  an acceptance test
```

The completion rule, verbatim: "A rung is done only when (a) every Deliverable maps to at least one User story, (b)
every User story's acceptance criteria pass, (c) every Requirement is satisfied, and (d) every Invariant holds under
test. 'Correct by definition' means exactly this closure: there is no behavior in the increment that is not pinned by
an acceptance check or an invariant, and no gate that is merely asserted rather than run." A4.6 ties this back to
A1.05.

**The triad (the A4.2 grounding — verbatim from `specs.approach.md`'s artifact table):** each rung is three files,
each answering one question — `fN.M.md` answers *what & why & done* (Goal · Rationale 5W · Scope · Deliverables ·

contract concrete from the user's side. The `.llms.md` is the agent-facing brief." The real triad to map a question
to is `f6.1.{md,stories.md,llms.md}`; the **second, self-referential triad** is the course's own
`a4.{md,stories.md,llms.md}` — A4 is a worked example of the artifact it teaches.

**The closed error set (a fixed dataset for A4.5's invariant-vs-check classifier — from `f6.1.llms.md` /
`specs.approach.md`):** `%Portal.Error{code, message, field}` with `code` one of

### THE CRITICAL GATE RULE — bare-route href + `data-sr-hash` (or the `links` gate FAILS)

The `links` gate (`apollo.go: gateLinks`) **exact-matches the raw `href` and does NOT strip a `#fragment`**, so
`href=".../specimens#f6-1"` is read as a dangling route and **FAILS**. The chip's `.sr-link` therefore **MUST**
carry a **bare-route href** `href="/course/agile-agent-workflow/spec/specimens"` plus a **`data-sr-hash="f6-1"`**
attribute (the rung hash). The `bindSpecrefs()` JS appends `#f6-1` on click. This **degrades** (JS off → the link
lands on the viewer's first stop, which is valid) and **passes the gate**. Each chip on a page needs a **unique
id** of the form `sr-<section>-<rung>` (e.g. `sr-chain-f61`, `sr-portal-f65`).

**(b) JS — the `bindSpecrefs()` function and its `bindSpecrefs();` call, placed INSIDE the page's existing IIFE**
(the `(function () { "use strict"; … })();` block that already runs the interactives and the Snowflake decoder).
Add it once; it binds every `.specref` on the page. If your page already carries it (the by-example pages do),
leave it.

```js
  /* ---------- specref chips: build-stamp mechanics, bound by class (multiple per page) ----------
     The chip text is static markup; JS only adds the toggle. The .sr-link href is the bare viewer
     route (so it resolves with no JS, landing on the ladder's first stop); JS appends the rung hash
     (data-sr-hash) at click time so an enhanced click deep-links to the matching rung. */
  function bindSpecrefs() {
    var chips = document.querySelectorAll('.specref');
    chips.forEach(function (chip) {
      var toggle = function () {
        var open = chip.classList.toggle('open');
        chip.setAttribute('aria-expanded', open ? 'true' : 'false');
      };
      chip.addEventListener('click', function (ev) {
        var link = ev.target.closest('.sr-link');
        if (link) {
          var hash = link.getAttribute('data-sr-hash');
          if (hash && link.getAttribute('href').indexOf('#') < 0) {
            link.setAttribute('href', link.getAttribute('href') + '#' + hash);
          }
          return;
        }
        toggle();
      });
      chip.addEventListener('keydown', function (ev) {
        if (ev.key === 'Enter' || ev.key === ' ' || ev.key === 'Spacebar') {
          if (ev.target.closest('.sr-link')) return;
          ev.preventDefault(); toggle();
        }
      });
    });
  }
  bindSpecrefs();
```
