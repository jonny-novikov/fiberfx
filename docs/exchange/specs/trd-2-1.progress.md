## {trd-2-1-decisions} Decisions

### D-1 — slice boundary: TRD.2.1 = Exchange.OrderBook + Exchange.Decider (pure) only

TRD.2.1 ships exactly two pure modules — `Exchange.OrderBook` (the per-side `gb_trees` price ladder, level-FIFO by branded mint order; `new/0`, `best/2`) and `Exchange.Decider` (`decide/2` + `evolve/2`, the matching rule over events). EXPLICITLY DEFERRED to TRD.2.2: the `Exchange.Book` GenServer (single writer, INV-1, G3, AS-3); the `EchoCache.Ring` drain + admission-reconcile (INV-2/INV-7, G4, AS-4); cancel-against-the-book matching; the per-account `EchoData.BrandedTree` index (cancel/queries). The slice is exhaustively property-testable with zero processes started — that is the reductive cut (see T-1 option (c)). Grounding: trd.2.1.prompt.md F-1 + the contract block; trd.2.specs.md Decomposition steps one/two (this rung) vs steps three/four (TRD.2.2).

### D-2 — self-trade = reject the aggressor in full, book unchanged (all-or-nothing)

An incoming `:place` order that would cross a resting order of the SAME account is rejected in full: `decide/2` returns `[{:rejected, %{order: <taker id>, reason: :self_trade}}]` and emits NO `:fill` and NO `:rested`; the book is left byte-unchanged (the fold of `evolve` over the emitted events leaves state == the input book, since the single `:rejected` event is a no-op in `evolve`). The detection requires the resting entry to carry `account` — so an `OrderBook` resting entry MUST hold at least `{id, account, side, price, quantity}` (account is REQUIRED, not optional). The check is all-or-nothing: even if non-self makers sit AHEAD of the self-crossing maker in price-time order, the aggressor is rejected without filling against them. Grounding: trd.2.1.prompt.md F-2 + the contract block; INV-6 (self-trade is a closed-reason rejection). The richer "fill non-self makers, reject only the self-crossing remainder" reading is recorded as V-1 (deferred).

### D-3 — closed `:rejected` reason set: `:self_trade` and `:no_liquidity`, no other atom

`Exchange.Decider` emits `{:rejected, %{order, reason}}` with `reason` drawn from a CLOSED two-atom set for this rung: `:self_trade` (D-2 — the aggressor would cross a same-account resting order) and `:no_liquidity` (D-4 — a market order with no remaining maker liquidity to fill its remainder). No other reason atom is emitted by this rung's `decide/2`. The set is closed: a property/gate asserts every `:rejected` event's reason is a member, and the chapter documents it as the exhaustive set. (A limit order with no cross does NOT reject — it rests; reject is reserved for self-trade and the market-no-liquidity remainder.) Grounding: trd.2.1.prompt.md F-2/F-4 + the contract block ("closed reason set, ≥ `:self_trade`, + the market reason `:no_liquidity`").

### D-4 — market-remainder rule: a market order matches at maker prices in price-time order and NEVER rests; the unfilled remainder is `{:rejected, :no_liquidity}`. A limit remainder rests.

A `:place` order with `type: :market` (carrying `price: :market`, the atom) matches whatever liquidity exists on the opposite side at the MAKERS' prices, walking price-time order (best price first; within a price, earliest mint order first), emitting one `:fill` per maker consumed. It NEVER rests: if quantity remains after all crossable liquidity is exhausted, the remainder yields `{:rejected, %{order: <taker id>, reason: :no_liquidity}}` (D-3) — NOT a `:rested`. (A market order is unpriced, so a `:rested` market entry would have no ladder key; resting it is incoherent.) Contrast a `:place` `type: :limit`: it crosses all opposite-side liquidity at or through its limit price, and any UNFILLED remainder RESTS as `{:rested, %{order, instrument, side, price, quantity}}` at its limit price (becoming a maker for future orders). Grounding: trd.2.1.prompt.md F-4 + the contract block; trd.2.md Design (the remainder of a limit order rests).

### D-5 — `Exchange.Decider` is pure modulo the FIL mint; the forbidden-effect grep set is empty

`decide/2` and `evolve/2` are side-effect-free EXCEPT for one sanctioned effect: the `FIL` fill id is minted INSIDE `decide` via `EchoData.Snowflake.next_branded("FIL")` (the canon, snowflake.ex:104) at the moment a `:fill` event is constructed. This is the SAME id-effect the Gateway is granted at TRD.1.1 (mint at acceptance). `evolve/2` mints nothing. The module `Exchange.Decider` contains NONE of: `GenServer`, `:ets`, `System.monotonic_time`, `System.os_time`, `Process.` — a grep for that set over `decider.ex` is empty (AS-2). NOTE: `next_branded/1` transitively reads `System.system_time` inside `EchoData.Snowflake`, but that impurity is encapsulated upstream — the Decider names only `next_branded`, reads no clock itself, and assigns no other ordinal. Properties NEVER assert id-equality across two `decide` calls (two mints are distinct by construction). Grounding: trd.2.1.prompt.md F-5 + AS-2; trd.2.specs.md INV-3 + Mars notes.

### D-6 — fill price = the maker's; one `:fill` per maker matched, each with its own branded FIL id

When an aggressor crosses resting liquidity, each maker consumed produces exactly one `{:fill, %{taker, maker, instrument, price, quantity, id}}` event, where `price` is the MAKER's resting price (NOT the taker's limit/market price) and `quantity` is the quantity matched against THAT maker (≤ that maker's resting quantity and ≤ the taker's remaining quantity). An aggressor that sweeps N makers emits N `:fill` events, in price-time order of the makers consumed (best maker price first; within a price, earliest mint order first). Each `:fill` carries its OWN branded `FIL` id minted at construction (D-5) — N fills ⇒ N distinct FIL ids. `taker` = the aggressor's order id; `maker` = the resting order's id; both carried so the downstream log/Go-workers attribute both sides (the Go-seam freeze). Grounding: trd.2.1.prompt.md the contract block "fill price = the maker's (G1)" + "one `:fill` per maker matched, each with its own `FIL` id"; trd.2.specs.md event vocabulary + G1; INV-5 (price-time from the id).

### D-7 — :rested event carries `account` (realization over the @type literal)

The spec's `@type event` literal (trd.2.1.specs.md:101-103) defines `:rested` as `%{order, instrument, side, price, quantity}` — NO account. But trd.2.1.specs.md:157-158 (rule D-2) states self-trade "detection needs the maker's `account`, which the resting entry carries" and the surface (§124) pins the resting tuple as `{id, account, side, price, quantity}` (account REQUIRED). The book's state is the EXTERNAL fold of `evolve` over emitted events (INV-4, §132-133; G6 §215 "the fold of evolve over the emitted events"). For a rest-then-cross G6 scenario (order A rests from account X via a :rested event folded into the book; a later crossing order B from account X is decided against that book), `decide` reads the resting maker's account from the book tuple — which `evolve(:rested)` must populate. The account is NOT recoverable from the :rested event's `order` id alone, and a pure external `Enum.reduce(events, book, &evolve/2)` has no command in scope. So the literal `@type` (account-less :rested) CANNOT satisfy line 158 under the spec's own external-fold model — an internal contradiction.
RESOLUTION (realization over literal, mars.md:34-37): the public `:rested` event map carries `account: binary()` in addition to `{order, instrument, side, price, quantity}`. `evolve(:rested)` reads it to build the resting tuple `{order, account, side, price, quantity}` the surface requires; `decide` stamps it from the placing order. AS-7 freezes ONLY the `:fill` key set (§220-222) — no gate freezes `:rested`'s key set — so this widening breaks no gate. `:fill` and `:rejected` shapes are UNCHANGED (the frozen Go-seam contract). Deviation flagged against trd.2.1.specs.md:101. Feedback to the spec (sync the `@type event` :rested arm to add `account`) routes to the Director.

### D-8 — the `:rested` event carries `account`: the spec syncs to the as-built (accepted widening)

## {trd-2-1-alternatives} Alternatives

### V-1 — self-trade variant: fill non-self makers first, reject only the self-crossing remainder (DEFERRED; D-2 chosen)

The alternative reading of self-trade prevention (STP): instead of rejecting the aggressor in full (D-2), the Decider fills the aggressor against all NON-self makers it crosses in price-time order, and only when it reaches a maker of the SAME account does it stop and reject the UNFILLED remainder `{:rejected, %{order, reason: :self_trade}}` (the self maker and everything behind it untouched). This is the richer real-venue "cancel-newest" / "decrement-and-cancel" family of STP modes.

Why NOT chosen for the MVP (D-2 all-or-nothing is chosen):
1. Clean event stream — all-or-nothing emits a SINGLE `:rejected` event with the book provably unchanged (state == fold leaves the input book); the partial variant emits a mix of `:fill`s + a terminal `:rejected`, and the "book unchanged on self-trade" property (G6 as the prompt states it) no longer holds verbatim — it would need re-specifying as "no SELF fill", a weaker, harder-to-state invariant.
2. Conservative STP — rejecting the whole aggressor is the strictly safer default (no surprise partial executions); a desk can always re-submit a smaller order. The permissive variant is an OPTIMIZATION, not a correctness requirement.
3. Satisfies G6 stated pairwise — the prompt's G6 ("two same-account crossing orders do not self-fill; the aggressor is rejected `:self_trade`, the book unchanged") is satisfied EXACTLY by D-2 and only approximately by the variant.
4. Smaller surface — all-or-nothing is one early-exit branch in `decide`; the variant interleaves matching with a per-maker account check and a remainder-reject, widening the property space.

The variant pairs naturally with configurable STP modes (reject-taker / cancel-maker / cancel-both) — a TRD.2.2-or-later concern once the Book owns order lifecycle and cancel. Recorded so the choice is a decision, not an omission. Grounding: trd.2.1.prompt.md F-2 ("recorded as a `tool_x_alternative` (V-n), deferred") + the Settled forks table.

## {trd-2-1-learnings} Learnings

### L-1 — the AS-2 pure-grep catches the MODULEDOC, not just `#` comments

The AS-2 forbidden-effect grep strips `# .*` line comments (the trd_1_1_check.exs:250 idiom) but a `@moduledoc """..."""` is a STRING literal, not a `#` comment — so prose inside it survives stripping. decider.ex's moduledoc literally wrote "no GenServer, no ETS, ... no `Process`" and "the Exchange.Book GenServer that drives this Decider", and the AS-2 test (decider_test.exs) correctly FAILED on the surviving "GenServer" token. gateway.ex dodged this by phrasing its moduledoc as "no process, no ETS" (no bare `GenServer`/`Process.` token). FIX: reword the moduledoc to describe the forbidden set WITHOUT the bare token names ("no server behaviour, no in-memory table, no monotonic or wall clock, no process-dictionary state"; "the single-writer server that will drive this Decider is TRD.2.2"). The gate stays honest — it proves the SOURCE holds no forbidden token, and the doc must not smuggle one past it. Carry for Stage-4: the rung gate trd_2_1_check.exs must apply the SAME comment-strip + token-refute, and it will pass only because the doc is now clean. A future code-author adding the forbidden token in prose is re-caught by this gate.

### L-2 — `evolve/2` assumes in-mint-order fold; the TRD.2.2 Book must preserve it

`ladder_consume` consumes the FIFO head of a price level regardless of whether that head's id equals `f.maker`, and `maker_side/2` infers the maker's side by "is the sell-head at `f.price` the maker? else buy". Both are CORRECT under the invariant that events are folded in the exact order `decide` emitted them (INV-4, "in mint order"): each `:fill` then consumes the very head the walk consumed, and the maker is unique by branded id so the side inference cannot false-match. The assumption is sound for this rung — `decide` emits fills head-first in price-time order, and any single caller folding the returned list in order reproduces the walk. The note for TRD.2.2: the `Exchange.Book` (the single writer) MUST fold each `decide` result in emitted order before processing the next command (it will, being the sole sequencer draining the Ring in order) — an out-of-order or interleaved fold across commands is undefined against `evolve` as built. Not a defect this rung; a contract the Book inherits. A defensive `evolve` could match `f.maker` against the head and the side explicitly, but that widens the pure core past the reductive cut; deferred. Grounding: Stage-3 reconcile of `evolve/2` + `maker_side/2` + `ladder_consume/4`.
