Defining functions

A named function is declared with def. It can have several clauses, and Elixir picks the first whose pattern and guard match the arguments — so a function definition often reads as a small table of cases rather than a chain of ifs.

The portal is full of small functions like this. A learner's progress maps to a status; a helper that only Portal.Progress needs is kept private with defp; and where a one-off function is handy — counting, filtering — an anonymous function or a capture does the job.

Clauses & dispatch

This status/1 turns a completion percentage into a label. It has three clauses: an exact match for 0, a guarded clause for 100 and above, and a catch-all. Choose a percentage to see which clause Elixir selects.

Clauses are tried top to bottom, so order matters: the exact and guarded cases come before the catch-all. One function, several shapes — no branching keyword in sight.

Public, private, anonymous

def makes a function callable from other modules; defp keeps it private to its own. For a small, local operation you do not want to name at all, an anonymous function works inline — and the capture operator & is a shorthand for one, turning fn r -> r.completed end into & &1.completed.

defmodule Portal.Progress do
  # public: other modules may call it
  def percent_complete(user) do
    total = Catalog.lesson_count()
    done  = user |> progress_for() |> count_done()
    percent(done, total)
  end

  # private: a helper only this module uses
  defp percent(_done, 0), do: 0
  defp percent(done, total), do: round(done / total * 100)
end

# anonymous function, then the capture shorthand
done = Enum.count(records, fn r -> r.completed end)
done = Enum.count(records, & &1.completed)
