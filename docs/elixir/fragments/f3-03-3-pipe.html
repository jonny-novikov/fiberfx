The pipe operator

The pipe |> takes the value on its left and passes it as the first argument to the call on its right. Chained, it turns a stack of nested calls into a sequence you read top to bottom — the value flowing through one function after another.

It is how the portal composes its module functions. To find how many lessons a learner has finished, you start with the learner and pipe them through Portal.Progress and a few Enum steps — each line one clear transformation.

Build the pipeline

This pipeline starts with a learner and asks how many lessons they have completed. The first step always runs — it loads their progress records. Toggle the remaining steps and watch the value change shape at each stage.

Each step is one call with one job, and the subject — the learner's data — flows straight down the page. Read it as a sentence: take the records, keep the completed, take their ids, count them.

First-argument threading

The pipe's whole rule is that x |> f(y) means f(x, y) — the left value becomes the first argument. That is why Elixir's functions are written with their main subject first: Enum.filter(list, fun), Portal.Progress.complete(user, lesson). The same code without the pipe nests inside out and reads backwards.

# nested: read inside-out, right to left
Enum.count(Enum.filter(Progress.for_user(user), & &1.completed))

# piped: read top-to-bottom, the value first
user
|> Progress.for_user()
|> Enum.filter(& &1.completed)
|> Enum.count()
