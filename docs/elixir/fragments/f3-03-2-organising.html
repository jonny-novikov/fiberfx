Organising with modules

A module is declared with defmodule and groups related functions under one name. To call across modules you write the full name — or bring it closer with a directive: alias to shorten it, import to drop the prefix entirely, require to use its macros.

The portal's modules call one another constantly: Portal.Auth looks up learners through Portal.Accounts, and reports through Portal.Progress. A directive at the top of the module decides how those calls read.

Reaching other modules

Take one call — Portal.Accounts.get_user(id) — and see how each directive changes the way it is written at the call site.

All four call the same function. Directives change names and scope, never behaviour — reach for alias most often, and keep import for the few functions you use constantly.

Inside a module

A module also holds attributes — names beginning with @. Some are documentation (@moduledoc, @doc); others are compile-time constants you set once and reuse. Documentation written this way is not a comment: it is compiled in, and h Portal.Auth.verify prints it back in IEx.

defmodule Portal.Auth do
  @moduledoc "Magic-link sign-in for the learning portal."
  alias Portal.Accounts

  @token_ttl 900  # seconds a link stays valid \u2014 a compile-time constant

  @doc "Verify a magic-link token and return its claims."

end

import
call a module's functions with no prefix at all.
require
load a module so its macros are available.
use
invoke a module's setup macro, injecting code into yours.
