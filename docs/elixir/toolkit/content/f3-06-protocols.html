Polymorphism: protocols & behaviours

Two kinds of polymorphism Three deep dives

A struct carries its module in the __struct__ tag, and F3.05 matched on that tag by hand. Elixir turns the same idea into a language feature two ways. A protocol dispatches a function to a different implementation depending on a value's type, at runtime. A behaviour is a contract a module promises to fulfil, checked at compile time.

The portal needs both: one summarize/1 that reads any entity, and a Notifier contract that email and SMS modules each satisfy. This module separates the two mechanisms, then takes each apart.

Two kinds of polymorphism

The question that separates them: does the behaviour vary by the value passed in, or by the module doing the work. A protocol answers the first — pick the implementation from the value's type. A behaviour answers the second — a module declares it implements a contract. Choose what varies and see which mechanism fits.

Both are polymorphism: one function name, many behaviours. The split is when and on what the choice is made — a protocol on a value at runtime, a behaviour on a module at compile time.

Three deep dives

Three pages in order: how a protocol call resolves to an implementation, writing those implementations per struct, then the behaviour as the compile-time counterpart.

F3.06.1
Defining a protocol →
defprotocol declares the contract; a call resolves to an implementation by the value's type, or raises Protocol.UndefinedError.
F3.06.2
Implementing for a struct →
One defimpl per type builds a dispatch table; new types are added without touching the protocol or the other implementations.
F3.06.3
Behaviours & callbacks →
@callback declares the contract, @behaviour and @impl fulfil it — the compile-time counterpart that OTP is built on.
