F5 · the engine · module 6

Where engine state lives

A home for the fold Three deep dives

The fold from F5.05 is pure: it takes the event log and returns the current state, then forgets it. That is fine for a calculation, but the Portal must keep the state alive between thousands of HTTP requests — the next request needs to see the enrollment the last one recorded. On the BEAM, live in-memory state lives in a process. F5.06 picks the process that owns the folded state — a GenServer, an Agent, or ETS — builds it around decide and evolve, and puts a supervisor around that so a crash restarts cleanly and replays the log.

One process holds the state; one supervisor keeps it alive. Select a piece to see what it does and which dive builds it.

A home for the fold

Reading the diagram: the event log is the source of truth from F5.05. A process folds it once on start and holds the result, answering commands and queries from that state. A supervisor sits above the process and restarts it if it dies. The three selectable pieces are the three questions this module answers — which process, how it is built, and what keeps it alive. Select a piece.

A pure fold computes the state and lets it go; a process is what holds onto it. Choosing that process and the boundary around it is the whole of this module.

Three deep dives

In order: which kind of process should hold the state and why; building that process as a GenServer around decide and evolve; and the supervisor that restarts it and lets the log rebuild the state.
