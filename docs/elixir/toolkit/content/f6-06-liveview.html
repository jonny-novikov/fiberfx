Phoenix LiveView fundamentals

Three parts of the loop From request to process Three deep dives

Everything so far has been one stateless round trip: a request arrives, a controller builds a response, HEEx renders it, the connection closes. LiveView changes the shape. A LiveView is a stateful server process connected to the browser over a WebSocket — an OTP process, the same kind of long-lived, message-driven actor the F5 engine was built from. It holds socket assigns as its state, renders the same HEEx from F6.05 as a function of those assigns, and when state changes it re-renders and pushes only the diff down the wire. The lifecycle is small and regular: mount/3 sets the initial state, render/1 produces markup, handle_event/3 turns a browser event into new state, and the loop continues. You write no JavaScript for the common interactions — a search box, a button, a live-updating list — because the server holds the state and the client only applies diffs. The same contexts from F6.04 still supply the data; LiveView adds the statefulness and the live connection on top of the view you already have.

Three dives: mounting a LiveView and its socket assigns, handling events to update state, and how render diffs reach the browser — the interactive layer over the same HEEx and contexts.

Three parts of the loop

A LiveView is a tight lifecycle, and this module takes each part in turn. mount/3 sets the socket’s initial state. handle_event/3 updates that state when the browser sends an event. render/1 turns state into HEEx, and LiveView ships only what changed. Select a part.

It is the same shape as a GenServer from F5: initial state, then messages that return new state. A LiveView is that pattern with a browser on the other end and HEEx as the projection of its state.

From request to process

The shift is from a thing that happens once to a thing that stays alive. A controller request (F6.01) runs, responds, and is gone — stateless by design. A LiveView mounts a process that lives for the duration of the page, holds state, and exchanges small messages with the client. The first paint is still plain HTML over HTTP (so it is fast and indexable); then the socket connects and the process takes over.

Because the state lives in a server process, an event does not re-run a whole request — it sends a message, the process updates its assigns, and the client receives a small diff. That is the entire reason LiveView feels instant without a single line of front-end state management.
