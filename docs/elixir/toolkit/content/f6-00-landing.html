F6 · chapter overview

Phoenix framework

F5 left a supervised engine behind a clean facade — correct, tested, and framework-free, but headless. F6 gives it a real product around it: a learning platform people actually use. We put Phoenix on top of the F5 engine without reaching into it — the endpoint joins the same supervision tree, the router sends requests to controllers and LiveViews, and every handler calls only the Portal facade. Then we take it to the next level: server-rendered HEEx, interactive LiveView with no hand-written JavaScript, real-time updates over PubSub, authentication and sessions, and a release deployed to production — finishing with a live dashboard that streams engine state to many clients at once. The engine never learns that Phoenix exists; that discipline is what makes the platform fast to build and safe to change.

Nine modules, one platform. Each adds a Phoenix capability on top of the same F5 engine, so by the end you have a real, real-time learning product — not nine framework demos.

The chapter arc

Read it in order: each module adds one layer to the platform without disturbing the engine beneath it. The first pair stands Phoenix up and routes a request; the second gives it data and domain boundaries with Ecto and contexts; the third renders and makes the UI interactive with HEEx and LiveView; the last three make it real-time, secure, and deployed, ending in a live dashboard. F6.01 is built; the rest are planned, listed so the path is visible from here.

Phoenix is not the product; the platform is. This chapter spends every module dressing the same engine, so the framework lands as a tool you aim at a domain, not a thing you build for its own sake.
