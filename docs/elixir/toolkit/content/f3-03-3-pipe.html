The pipe operator

The pipe operator |> takes the value on its left and passes it as the first argument to the function call on its right. That single rule turns a nest of function calls, read inside-out, into a top-to-bottom pipeline read in the order it runs — the same composition you saw with folds in F2, written as syntax.

The same grade computation, two ways: nested calls versus a pipeline. Both produce :b; one is far easier to read.

Inside-out vs left-to-right The value goes first

Inside-out vs left-to-right

Averaging then grading a score list is two function calls. Nested, the inner call must be read first; piped, the steps read in run order. Select a form — the data flow is identical, only the reading changes.

Both forms call average then grade on the same data. The pipe does not change what runs — it changes the order you read it in, from inside-out to the order of execution.

The value goes first

The whole rule is: x |> f(a, b) is rewritten to f(x, a, b) — the piped value becomes the first argument, and any arguments you write are appended after it. That is why Elixir’s core functions take their main subject first (Enum.filter(list, fun), String.split(str, on)): it makes them pipe cleanly. Design your own functions the same way and they slot into a pipeline without wrappers.

# x |> f(a, b)  is  f(x, a, b) — the value is inserted as the first argument
[80, 92, 74]
|> Enum.filter(&(&1 >= 80))   # Enum.filter([80, 92, 74], fun) => [80, 92]
|> Enum.sum()                 # Enum.sum([80, 92])            => 172

# the nested equivalent, read inside-out:
Enum.sum(Enum.filter([80, 92, 74], &(&1 >= 80)))
