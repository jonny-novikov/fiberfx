# F0 · Roadmap — migrating the design system through the specs

> A delivery plan for the design system as a portable artifact: the path from the static `/elixir` HTML — body
> fragments assembled by `build_page.py`, gated by nine Apollo gates, stamped — through a SQLite content store that
> holds and reproduces every page byte-for-byte, to the Phoenix Portal rendering the same tokens, envelope, and
> anatomy in HEEx. The look and the byte contract survive the framework change. This file is the delivery view; the
> system itself — tokens, head, envelope, anatomy, bootstrap, stamp, gates — is specified in [`f0.md`](f0.md).

## What we are delivering

The design system as a migratable, tooled artifact rather than CSS copied across pages. F0 fixes one visual language
(the dark-editorial `:root` tokens, four font families, the page envelope and its anatomy vocabulary) and one quality
bar (the nine Apollo gates); this roadmap delivers the toolchain that carries that system across a framework boundary
without changing its look or its bytes.

The vehicle is `jonnify-cms`: a CGO-free Go pipeline that ports the Python builder, gates each page to A+, and — the
load-bearing step — decomposes every published page into a SQLite **content store** and recomposes it byte-identically.
The store is what makes the system portable: it is the seam between the static HTML the course ships today and the
Phoenix Portal (F6) that renders the same system in HEEx tomorrow. The near-term, already-reached goal is byte parity
between the published corpus and what the store reproduces; the Portal rendering F0 in HEEx, parity-checked against the
static baseline, follows.

## Where this starts and ends

- **Start (the static system + `build_page.py`).** An author writes only a body fragment; `build_page.py`'s
  `_assemble` supplies the shared head (tokens + base CSS), the exact document envelope, the bootstrap, and a fresh
  branded Snowflake stamp, then validates the result through the nine Apollo gates. ~200 hand-authored pages render in
  this system; A+ across all nine gates is the ship bar. The full system is specified in [`f0.md`](f0.md) (D1–D8); the
  Python builder lives at `docs/elixir/toolkit/build_page.py`.
- **Today (the content store reproduces it byte-for-byte).** `jonnify-cms` decomposes each published `/elixir` page
  into a deduplicated head template, a body-fragment template, and per-page data in a filesystem-mirrored SQLite store,
  and recomposes the exact published bytes. A permanent, offline Go parity test
  (`apps/jonnify-cms/internal/builder/parity_test.go`) proves it: every one of the 204 published pages round-trips
  byte-identical, every page carries one internally consistent stamp, and the corpus dedups into 66 distinct head
  templates — no page skipped. The content store and its byte-parity builder are specified in
  `apps/jonnify-cms/docs/specs/07-content-store.md`; the gate suite and assembly semantics in
  `apps/jonnify-cms/docs/specs/05-build-validate.md`.
- **End (the Portal renders F0 in HEEx, parity-checked against the static baseline).** F6's web layer (the HEEx view
  rung, [`../phoenix/phoenix.md`](../phoenix/phoenix.md)) reproduces the F0 tokens, envelope, and anatomy as HEEx
  components, so the rendered look matches the static pages. Parity is checked the same way the static system is
  validated: the e2e baseline `apps/e2e/tests/elixir-hero.spec.ts` is written against computed styles and geometry, not
  pixel snapshots, so a single spec validates both — the static server today and the Portal under `E2E_BASE_URL`
  tomorrow. Passing against both is the proof that the migration preserved the design contract. This rung is planned,
  not built.

  > **[RECONCILE]** The F6 rung that executes this END step — milestones 4 (HEEx components reproducing the tokens and
  > anatomy) and 5 (the parity gate) — is **F6.5.5 · Apply the design system** ([`../phoenix/f6.5.5.md`](../phoenix/f6.5.5.md)),
  > authored after the F6.5 HEEx views and F6.6 LiveView shipped on the still-unstyled `portal_web`. Its delivery is
  > **bundler-free and aligned with this file's no-build model**: F0's system carried as an inline `<style>` / single
  > shared head becomes a Portal **root layout + a committed `priv/static/assets/app.css`** served over the EXISTING
  > `Plug.Static` mount (`portal_web.ex:14` `static_paths` includes `assets`; `endpoint.ex:28` mounts it) — **no
  > esbuild, no tailwind**. The as-built gap F6.5.5 closes: `portal_web.ex:30` sets `layouts: []` (and the `:html` /
  > `:live_view` macros declare no root layout either), so there is no `<head>` today and therefore nowhere for the
  > F0 tokens to be declared once (F0-INV2); `priv/static/assets/` holds only `.keep`. F6.5.5 is the prerequisite that
  > makes "the Portal renders the same system" buildable, and milestone 5's parity gate verifies it. *(Why: this END
  > step was specified before the rung that executes it existed; F6.5.5 is now that rung, and its committed-CSS
  > delivery is aligned with — not a departure from — F0's no-build model.)*

## Architecture decision — a filesystem-mirrored, pure-Go SQLite content store

The store is SQLite opened through `database/sql` with the pure-Go driver `modernc.org/sqlite`, keeping `jonnify-cms`
CGO-free (stdlib + cobra + the driver). The reasoning matches the standard-stack choice the Phoenix roadmap makes
([`../phoenix/phoenix.roadmap.md`](../phoenix/phoenix.roadmap.md)): a single static binary with no C toolchain builds
and runs the same on any target, so the migration tool itself carries no platform-specific build step.

The model is deliberate, and it diverges from the older single-shared-head assumption. An empirical probe of the live
pages disproves that one `_head.html` reproduces every published head — chapters carry a per-chapter accent that
changes the head CSS, and published heads have drifted since the partial was extracted. The store resolves this by
holding **each page's own head template**, deduplicated by content hash (204 pages → 66 distinct heads), rather than
one shared head; and it holds each page's **rendered body fragment** verbatim (only the build stamp re-placeholded),
because the repository commits already-built pages, not the pre-substitution fragments `build_page.py` consumes. The
two index pages are round-tripped from their stored bytes rather than regenerated from the manifest — from-manifest
generation is recorded as a separate, later capability, not a parity requirement.

Decomposition is lossless and assembly is a pure function of (head template, fragment, title, desc, stamp): the same
inputs yield byte-identical output, and the stamp is the only per-build-varying region. That determinism is what makes
the store a sound seam — the bytes the static server ships and the bytes a future Portal would emit are pinned to the
same contract, so "the Portal renders the same system" is a checkable claim rather than a resemblance. The trade stays
reversible: storing per-page heads costs more rows than one shared head, but it is the model that actually reproduces
this corpus, and from-manifest regeneration can be added later without a schema change.

## Milestones / build order

Five rungs, each a step in carrying the system across the boundary. The first three are built; the last two are
planned and sharpen as F6's web layer lands.

| Milestone | Ships | Status | Basis |
| --- | --- | --- | --- |
| 1 · Specify the design system | tokens, head, envelope, anatomy, bootstrap, stamp, the nine gates, and the toolchain contract (F0-D1–D8) written down and held by invariants | **done** | [`f0.md`](f0.md) — the F0 contract, with INV1–INV7 |
| 2 · Port the builder to Go | `cms check` runs the nine Apollo gates against the built pages with Python-gate parity; `cms build` assembles the envelope | **done** | `05-build-validate.md` — `cms check` runnable now; gate parity with `build_page.py` specified gate-for-gate |
| 3 · Content store + byte-parity test | filesystem-mirrored SQLite store decomposes each published page and recomposes it byte-identically; the parity test gates it | **done** | `07-content-store.md` §7 — 204/204 pages round-trip byte-for-byte, 204 consistent stamps, 66 distinct heads, zero skips |
| 4 · Portal HEEx components reproducing tokens/anatomy | the F0 `:root` tokens, document envelope, and page anatomy rendered as HEEx components in the Portal | planned — **executed by F6.5.5** `[RECONCILE]` | [`../phoenix/f6.5.5.md`](../phoenix/f6.5.5.md) — the F6 rung that applies the design system (root layout + committed `app.css` over `Plug.Static`, no bundler); F6.5/F6.6 shipped unstyled, F6.5.5 lands the tokens/anatomy |
| 5 · Parity gate: static vs Portal | one computed-style + geometry e2e spec validating the static baseline and the Portal under `E2E_BASE_URL`, asserting the design contract held across the migration | planned — **gate of F6.5.5** `[RECONCILE]` | `apps/e2e/tests/elixir-hero.spec.ts` — baseline written; the Portal target lands with [`../phoenix/f6.5.5.md`](../phoenix/f6.5.5.md) (milestone 4) |

The rungs depend only downward: milestone 3 reproduces what milestone 2 assembles; milestone 4 renders the tokens and
anatomy that milestone 1 fixed; milestone 5 checks milestone 4 against the milestone 1–3 baseline. The content store
(milestone 3) is the pivot — every page the static system ships is provably reproducible from it, so the Portal rung
inherits a pinned byte contract rather than a CSS resemblance to match.
