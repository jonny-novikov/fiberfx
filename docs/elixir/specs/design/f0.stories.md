# F0 · user stories

> Who needs the design system, what they need, and how it is known to work. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f0.md`](f0.md). The roles are design-system roles:
> author (writes a fragment), reader/learner (consistent, accessible, fast pages), agent (builds a page to the bar),
> maintainer/Portal (reproduces the system).

## F0-US1 — Author against a fixed token palette

As an **author**, I want one authoritative `:root` palette of surfaces, text, accents, four font stacks, and a prose
measure, so that every page reads as one visual language and a figure's SVG uses the correct hex by role.

Acceptance criteria
- Given a new or edited page, when its colours and fonts are chosen, then they come from the `:root` tokens — surfaces
  `--ink/--ink-2/--ink-3`, text `--cream/--cream-soft/--cream-dim`, accents
  `--gold/--blue/--sage/--burgundy/--elixir` (each with a `-bright` variant except `--burgundy`), `--line`, the four
  stacks `--serif-display/--serif/--sans/--mono`, and `--measure: 68ch` — at the fixed values.
- Given an `<svg>` figure, when a presentation attribute (`fill`, `stroke`) is set, then it uses the literal hex mapped
  by role (result → `#f0cd7f`/`#d4a85a`, node fill → `#161d38`, idle stroke → `#2a3252`, counterexample → `#c4504c`,
  code accent → `#cdb8f0`, axis text → `#a39c89`), because SVG attributes cannot read CSS custom properties.
- Given any two pages, when their palettes are compared, then the token values are identical.

INVEST — independent of any single lesson; small; testable by diffing the `:root` block and inspecting SVG hex by role.
Priority: must · Size: 2 · Implements deliverables: F0-D1.

## F0-US2 — Receive the head and envelope from the builder, not by hand

As an **author**, I want the builder to supply the same `<head>` (tokens + base CSS + the Google Fonts link, with
`{{TITLE}}`/`{{DESC}}` placeholders) and to wrap the fragment in the exact document envelope, so that I write only a
body fragment and never re-declare the head or the document scaffolding.

Acceptance criteria
- Given a page key with a title and description, when the builder assembles the page, then it emits exactly
  `<!doctype html>\n<html lang="en">\n` + the filled head + `\n<body>\n` + the fragment + `\n` + the bootstrap +
  `\n</body>\n</html>\n`.
- Given the shared head, when a page is built, then `{{TITLE}}` and `{{DESC}}` are HTML-escaped per page and the
  `:root` tokens and base CSS are declared once in that one inline `<style>`.
- Given a body fragment, when it is authored, then it contains neither a `<head>` nor the bootstrap script — only the
  markup between `\n<body>\n` and the appended bootstrap.

INVEST — valuable (the author/tooling boundary); small; testable by byte-comparing the emitted envelope and grepping a
fragment for a forbidden `<head>`.
Priority: must · Size: 3 · Implements deliverables: F0-D2, F0-D3.

## F0-US3 — Build a page from the fixed anatomy vocabulary

As an **author**, I want a named, top-to-bottom page anatomy — skip link, sticky `.site` header with the page's
`.route-tag`, `main#main.wrap`, a `.hero`, content `<section>`s, a synthesis section, a `.pager`, and the
`.site-foot` stamp — so that I assemble a page from known containers instead of inventing structure per lesson.

Acceptance criteria
- Given a page, when its body fragment is read top to bottom, then it carries a skip link, a `.site` header whose
  `.route-tag` mirrors the page's clean route, a single `main#main.wrap` column, a `.hero` (`.crumbs`, `.eyebrow`, one
  `<h1>`, `.lede`, `.kicker`, `.toc-mini`), content sections, a synthesis section with a `.note`, a `.pager`, and the
  `.site-foot`.
- Given a content `<section>`, when it presents a concept, then it uses the vocabulary as specified — `.prose` capped
  at `--measure`, a `.deflist`, a `.fig` figure carrying its `<svg>` / `pre.code` / `.geo-readout`, a `.bridge`
  pairing the idea with its Elixir form, and a closing `.take`.
- Given the page, when its block-level containers are stacked, then every `.fig`, `.bridge`, and block `<div>` is
  balanced — no unclosed open and no mismatched close.

INVEST — independent of content; small; testable by checking the container inventory is present and balanced.
Priority: must · Size: 3 · Implements deliverables: F0-D4.

## F0-US4 — Append the progressive-enhancement bootstrap once

As an **author**, I want the builder to append one progressive-enhancement bootstrap that adds `html.js` and reveals
`.reveal` elements via `IntersectionObserver`, so that the reveal animation is wired identically on every page and the
author writes no per-page reveal script.

Acceptance criteria
- Given a built page, when the document loads, then the appended bootstrap adds the `js` class to
  `document.documentElement`, then on `DOMContentLoaded` reveals `.reveal` elements via an `IntersectionObserver`.
- Given `prefers-reduced-motion: reduce`, or a browser where `IntersectionObserver` is absent, when the page loads,
  then the bootstrap short-circuits to revealing all elements immediately.
- Given a body fragment, when it is authored, then it carries no reveal script of its own — the bootstrap is the only
  one, supplied by the builder.

INVEST — independent of any page's content; small; testable by reading the appended bootstrap and toggling reduced
motion.
Priority: must · Size: 2 · Implements deliverables: F0-D5.

## F0-US5 — Certify every page A+ across the nine gates

As an **agent**, I want every page held to the nine Apollo gates with A+ meaning all nine pass and any FAIL a hard
stop, so that I build a page to a known bar rather than re-deciding what "good enough" means.

Acceptance criteria
- Given a built page, when `cms check` (or the Python `check`) runs, then it reports `containers`, `svg`, `no-future`,
  `voice`, `storage`, `motion`, `degrade`, `links`, and `pager`, in that order, and grades `A+` only when all nine
  pass.
- Given a page that fails any single gate, when it is checked, then the grade is not `A+` and the status is `FAIL`,
  blocking the page.
- Given the same page, when both the Python `check` and `cms check` run, then they agree gate-for-gate.

INVEST — independent of any one gate; small; testable by running both checkers and by introducing then reverting a
single-gate violation.
Priority: must · Size: 3 · Implements deliverables: F0-D7.

## F0-US6 — Read every page without JavaScript and without motion or storage

As a **reader/learner**, I want the content visible without JavaScript, reduced motion honoured, and no web storage
used, so that the page is fast, accessible, and works in any browser and any privacy setting.

Acceptance criteria
- Given JavaScript is disabled, when the page renders, then all content is visible, because `.reveal` is JS-gated
  (`html.js .reveal`) rather than hidden unconditionally.
- Given `prefers-reduced-motion: reduce`, when the page renders, then the reveal animation does not run and content is
  shown immediately.
- Given any page, when its document is scanned, then neither `localStorage` nor `sessionStorage` appears anywhere.

INVEST — valuable (accessibility and resilience); small; testable by the `degrade`/`motion`/`storage` gates and by a
no-JS render.
Priority: must · Size: 2 · Implements deliverables: F0-D7, F0-D5.

## F0-US7 — Carry one branded stamp that decodes on the page to its build time

As a **reader/learner**, I want each page to carry exactly one branded Snowflake build stamp that the page itself
decodes client-side to a UTC timestamp, so that the page is self-describing about when it was built without any tool.

Acceptance criteria
- Given a built page, when its footer is read, then it carries exactly one 14-character stamp: a three-letter
  namespace (`TSK`) plus the 11-char base62 encoding of a 64-bit Snowflake
  (`timestamp(41) << 22 | node(10) << 12 | seq(12)`, epoch `1704067200000`).
- Given the footer stamp widget, when it is activated, then an on-page decoder expands it to its namespace, snowflake,
  node, seq, and UTC timestamp using those same constants.
- Given the canonical vector `TSK0KHTOWnGLuC`, when the page decoder and the tool each decode it, then both yield
  namespace `TSK`, snowflake `274557032793636864`, node `0`, seq `0`, timestamp `2026-01-27 15:11:37 UTC`.

INVEST — independent of page content; small; testable by counting stamps and decoding the canonical vector two ways.
Priority: must · Size: 2 · Implements deliverables: F0-D6.

## F0-US8 — Reproduce the published corpus byte-for-byte

As a **maintainer/Portal**, I want `jonnify-cms` to decompose every published page into a head template, a body-fragment
template, and per-page data and recompose it byte-identically, so that the look and contract are reproducible and the
F6 migration renders the same system unchanged.

Acceptance criteria
- Given the published `/elixir` tree, when `cms build --load` runs, then each conforming page decomposes into a
  deduplicated head template, a stamp-re-placeholded body-fragment template, and per-page title/description/stamp
  data.
- Given the loaded store, when `cms build --verify` runs, then every stored page reassembles equal to its published
  file byte-for-byte, with the same inputs yielding byte-identical output.
- Given the parity test, when it runs, then it asserts the loaded page count equals the on-disk `*.html` count, the
  skip list is empty, and every page round-trips byte-identically.

INVEST — valuable (reproducibility and migration confidence); estimable; testable by the offline parity test over the
whole corpus.
Priority: must · Size: 5 · Implements deliverables: F0-D8.

## F0-US9 — Hold the voice across prose and comments

As a **reader/learner**, I want prose and code comments free of hype/dismissive words, first-person narration, and
gendered or perceptual verbs applied to tools, so that the writing stays precise and the reading experience is
consistent across every page.

Acceptance criteria
- Given a page's visible text, when the `voice` gate runs, then none of `revolutionary`, `blazing-fast`, `magical`,
  `simply`, `just`, `obviously`, `effortless` appears (word-boundary, case-insensitive), and the gate passes.
- Given prose or a code comment, when it is reviewed, then it carries no first-person narration and no gendered or
  perceptual verb applied to a tool or an agent.
- Given a page that introduces a forbidden word in visible text, when it is checked, then the `voice` gate fails and
  the page is blocked.

INVEST — independent of content topic; small; testable by the `voice` gate and a prose review.
Priority: must · Size: 1 · Implements deliverables: F0-D7.
