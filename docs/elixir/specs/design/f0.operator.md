# F0 · Operator's guide — supervising Claude's design-system delivery to the Portal

> A field manual for the **Human L0 Operator**: the person in the conversation who owns the design intent — what a
> token means, what counts as the page anatomy, where the component-library seam sits — reviews each styling
> increment, and gives the go/no-go. Claude runs the rest — Author, Director, and the specialized lead-team — turning
> the design-system rung into specs, applying it over the unchanged `Portal` facade, verifying it, and shipping it.
> The workload is the F0 design system (tokens · the four font families · the page anatomy) carried from the static
> `/elixir` pages onto the Portal, and it holds **three** standing promises every styling rung:
> the [master invariant](../phoenix/phoenix.md) (the layout and CSS name only facade-rendered assigns — no engine, no
> repo, no `GenServer.call`), the [liveness criterion](../phoenix/phoenix.md) (the Portal stays live and hot in dev),
> and the **design contract** (the Portal renders in the same tokens and anatomy as the static `/elixir` and agile
> baseline — proven by **computed style**, not by resemblance).

This guide is the *process* view for the styling rung. The *what-to-build* lives in the F0 triad
([`f0.md`](f0.md) · [`f0.stories.md`](f0.stories.md) · [`f0.llms.md`](f0.llms.md)) and the rung's brief; the
*delivery plan* lives in [`f0.roadmap.md`](f0.roadmap.md) (milestones 4–5); the *build conventions and footguns* live
in [`echo/CLAUDE.md`](../../../../echo/CLAUDE.md). When this guide and the F0 spec disagree about what to build, the
spec wins.

**The rung this guide governs: F6.5.5 · Apply the design system.** The styling rung that executes the F0 foundation
([`f0.roadmap.md`](f0.roadmap.md) milestones 4 + 5) on the currently-**unstyled** `portal_web`.

> **As-built mechanism (reframed 2026-06-05, shipped `7c7f317`).** An earlier draft of this rung planned a **root layout +
> a committed `app.css` + restyled `CatalogComponents`** (the three-piece build this guide originally described). The
> Operator **reframed** it to publish **TWO STATIC PARITY PAGES** instead — the contract below is what shipped, and what
> this guide now governs.

The build, over the facade and with **no bundler**, is:

- **two full-document HEEx templates, NO root layout** — `page_html/home.html.heex` reproduces `html/courses.html` (the
  two-course index, served at `:4000/`) and a new `page_html/elixir.html.heex` reproduces `elixir/index.html` (the Elixir
  index, served at `:4000/elixir` via a new `PageController.elixir/2` + a `get("/elixir", …)` route). Each template carries
  its OWN `<head>` (the two masters' heads differ irreconcilably), rendered bare — `layouts: []` at
  [`portal_web.ex`](../../../../echo/apps/portal_web/lib/portal_web.ex):30 is UNCHANGED, NOT replaced;
- **four per-page committed assets** — `courses.css`/`courses.js` (from `courses.html`) + `elixir-index.css`/`elixir-index.js`
  (from `elixir/index.html`), each the master `<style>`/`<script>` body relocated VERBATIM, served over the `Plug.Static`
  mount **corrected this rung from `at: "/assets"` to `at: "/"`** (the F6.1 latent fix —
  [`endpoint.ex`](../../../../echo/apps/portal_web/lib/portal_web/endpoint.ex):35). There is **NO `app.css`** (the shared
  token stylesheet for the dynamic UI lands at F6.8);
- **the deep-link remap, with a CONFIGURABLE base** — every nav deep-link to a route the Portal does NOT serve
  (`/elixir/<…>`, `/course/<…>`) renders against `config :portal_web, :deep_link_base_url` (default
  `https://jonnify.fly.dev`, read via `PortalWeb.deep_link_base/0`); the `/`+`/elixir` self-links stay relative, and the

No CoreComponents (deferred to F6.8; see [`echo/CLAUDE.md`](../../../../echo/CLAUDE.md) §8). The hard problem this guide
exists to solve is **visual acceptance with no screenshots** — validation is headless and text-only (§4–§6); those proxies

parity pages at two origins (`apps/e2e/tests/index-parity.spec.ts`).

## 1. Two roles, and what each decides

The loop has exactly two parties, mirroring the F6 lead-team pattern, narrowed here to the design decision split.

- **L0 — the Operator (the human).** Owns the *why* and the *what-next* of the look: priorities, the acceptance of a
  shipped styling rung, and every decision that changes the **design contract** or the **component-library seam**.
  Reviews the F0 spec **body** (it is authoritative) and the styled increment, then returns feedback. Feedback edits
  the spec, because the spec is the single source of truth and the build follows it.
- **L1+L2 — Claude (the Author + Director + lead-team).** Turns the styling rung into specs at the F0 quality bar,
  then applies the system as a Director coordinating three specialized peers (Venus, Mars, Apollo). Claude decides
  implementation details with one obvious answer and proceeds; Claude **stops and asks** the Operator on anything that
  changes the design contract or the no-CoreComponents stance.

The decision split is the load-bearing rule — and for a design rung it turns on *contract vs. realization*:

| Decision | Owner | Mechanism |
| --- | --- | --- |
| A **token value** (a new hex, a font-stack change, a `--measure` change) | **Operator** | changes F0-D1 — Claude STOPs and ASKs |
| A **new anatomy element** (a container the F0-D4 vocabulary does not name) | **Operator** | changes the design contract — STOP and ASK |
| **Dropping the no-CoreComponents stance** (pulling in the generated component library early) | **Operator** | changes the component-library seam ([`echo/CLAUDE.md`](../../../../echo/CLAUDE.md) §8) — STOP and ASK |
| Which of two reasonable CSS realizations of the *same* anatomy/token | Claude (Director/Mars) | logged as a decision, surfaced for veto |
| A styling detail with one answer (the literal hex the token already fixes) | Claude (Mars) | proceeds; cited to the F0-D1 line |

The boundary in one line: **a token, an anatomy element, or the component-library seam is the Operator's; which of two
reasonable CSS realizations of an already-fixed contract is Claude's, logged for veto.** The per-rung loop the
Operator drives is **sharpen → apply → ship → demo → review → feedback → adapt**.

## 3. The per-rung pipeline — the stages the Director runs

The styling rung gets a Director brief that names the stages and carries only what is *new about this rung* (the
discipline is in the agent definitions). [`../phoenix/f6.6.prompt.md`](../phoenix/f6.6.prompt.md) is the worked
template for the shape; this rung's stages, and the Operator's touch-point at each:

1. **Venus · reconcile + brief.** Runs the lag-1 reconcile *pre-build* — does the F0-claimed surface (the token set,
   the anatomy vocabulary, the `Plug.Static` mount, the empty `layouts: []` seam) exist in the `portal_web` the rung
   styles? — corrects any drift in the F0 body, pins the design contract this rung renders, and refreshes the brief.
   Reports a delta table + a BUILD-GRADE/BLOCKED verdict. *Operator: ratifies any change to a token or the anatomy
   that Venus surfaces (a contract change — surfaced for veto, or escalated per §1).*
2. **Mars-1 · apply.** Builds the two full-document templates (`home.html.heex` reproducing `courses.html`, a new
   `elixir.html.heex` reproducing `elixir/index.html`) + their four per-page assets (`courses.css`/`.js`,
   `elixir-index.css`/`.js`, each the master `<style>`/`<script>` relocated VERBATIM), adds the `elixir/2` action +
   `/elixir` route, applies the deep-link remap against the configurable `:deep_link_base_url` (assets stay
   `~p"/assets/…"`, Portal-local), and corrects the `Plug.Static` mount to `at: "/"` — citing the master line for every
   token and container, inventing no new anatomy, keeping the diff inside the facade. NO root layout, NO `app.css`, NO
   `CatalogComponents` restyle. Compiles clean. *Operator: none.*
3. **Mars-2 · harden.** Adds the `ConnTest` coverage that each rendered page carries its per-page stylesheet `<link>`
   + the anatomy + the styled envelope, and the two-route/two-origin `index-parity.spec.ts`; NO determinism loop (no
   id/process surface this rung). *Operator: none.*
4. **Apollo · verify.** Runs the post-build reconcile (does the as-built layout/CSS/component set satisfy F0's
   promises?), re-runs the build gate independently, **and runs the design-contract gate** (§4): the master-invariant
   grep, the clamp-spacing parse, the token-fidelity diff, and the computed-style **parity** run against the live
   Portal. Syncs the F0 body to what shipped, and renders **BUILD-GRADE** or **BLOCKED (n deltas)**. *Operator: reads
   the verdict — the proxy-gate results are the visual evidence in lieu of a screenshot.*
5. **Director · ratify + commit (LAW-4).** On BUILD-GRADE, one scoped commit (explicit pathspec, never `git add -A`,
   excluding the Operator's out-of-band work; the pathspec is §7). *Operator: the demo + the accept/iterate call.*
6. **Director · feedback loop.** Folds the rung's findings forward into the downstream specs as `[RECONCILE]` markers.
   *Operator: none; visible in the downstream spec diffs.*

If Apollo returns **BLOCKED**, the needed change routes back through the Director to a fresh Mars spawn — Apollo never
edits production code, never commits. That separation is what keeps the verifier adversarial; a one-line CSS fix is
still a BLOCK that routes to Mars, not a grade Apollo softens.

## 4. The standing gates — what "shipped" means

Three promises hold at *every* styling rung, plus the build gate, plus the acceptance rule. The first two are the F6
standing invariants; the third — **the design-contract gate** — is what makes "rendered correctly" checkable without a
screenshot.

- **Master invariant.** The layout and the CSS name only facade-rendered assigns — no layout, component, or template
  names `Portal.Engine`, a repo, or `GenServer.call`. Apollo proves it by grep over the new web modules
  (`grep -rE "Portal\.Engine|Repo|GenServer\.call"` empty); a boundary leak is a defect even when every test is green.
- **Liveness criterion.** Every rung leaves the Portal **running and serving the styled envelope** in dev: the
  umbrella boots, the endpoint binds `:4000`, `GET /health` answers `200`, and the rung's two published routes (`/` and
  `/elixir`) each render — **with their per-page stylesheet `<link>` and a real `<head>`, not a bare `<h1>Portal</h1>`,
  and with the assets served LOCALLY (no `fly.dev/assets`).** The runbook is §5.
- **The build gate** (Mars runs it; Apollo reproduces it; the Operator can too):
  `TMPDIR=/tmp mix compile --warnings-as-errors` clean → `TMPDIR=/tmp mix test` green → for any id- or
  process-touching change, the **≥100-iteration determinism loop** (a single green run is not proof; see
  [`echo/CLAUDE.md`](../../../../echo/CLAUDE.md) §4).

### 4.1 The visual-acceptance problem, stated plainly

A styling rung's real question is *"does the Portal render in the F0 design system?"* — and there is **no screenshot**
in this loop to answer it. Validation is headless and text-only. So acceptance is not "a human looked at it"; it is
**A+ across a stack of text-checkable proxies for the look** plus a **computed-style parity** run. Each proxy is a
narrow, runnable claim; together they stand in for the eye. The proxies, cited to the static system that already
runs them ([`apps/jonnify-cms/docs/specs/05-build-validate.md`](../../../../apps/jonnify-cms/docs/specs/05-build-validate.md)
§6, the nine Apollo gates):

- **containers** (`05-build-validate.md` §6.1) — the rendered HEEx envelope's block containers are balanced; a missing
  close in a layout or component is a structural defect a passing controller test can hide.
- **voice** (§6.4) — the layout chrome and any rendered copy carry none of the hype/dismissive set
  (`revolutionary | blazing-fast | magical | simply | just | obviously | effortless`), word-boundary,
  case-insensitive.
- **links** (§6.8) — internal hrefs the layout emits (the nav, the pager) resolve to real routes, not dangling paths.
- **clamp-spacing** — **the single highest-leverage check.** The bug that shipped invisibly across 204 static pages:
  `clamp()` written with no space around the inner `+` — `clamp(1.9rem+4.2vw, …)` — is **invalid CSS**, so the browser
  **silently** discards the rule and falls back to the UA default (an `h1` collapses to `2em` = 32px). A green test
  suite never sees it; a screenshot would, but there is none. The proxy is a **parse check** on the two per-page CSS files
  (`courses.css` + `elixir-index.css`): every `clamp()` has whitespace around its inner operators, and
  `CSS.supports('font-size', '<the clamp expr>')` is true. (Here it holds by construction — each file is the master
  `<style>` relocated VERBATIM.) The index-parity spec
  ([`apps/e2e/tests/index-parity.spec.ts`](../../../../apps/e2e/tests/index-parity.spec.ts)) carries the computed-style
  half of this guard at two origins: on `/elixir` an `h1` must render **> 70px**, `.prose p` **> 18px** — exactly the
  collapse the no-space `clamp()` caused.
- **degrade** (§6.7) + **storage** (§6.5) + **motion** (§6.6) — the rendered page is visible without JS (`.reveal` is
  JS-gated), uses no `localStorage`/`sessionStorage`, and honours `prefers-reduced-motion` — F0-INV4, rendered.
- **token-fidelity** — the `elixir-index.css` `:root` block is **diffed byte-for-byte** against the F0-D1 authoritative
  values (this is the F0-INV2 page; `courses.css`'s `:root` is a SEPARATE, page-specific token set — `--surface`/`--body`/
  `--shadow*`/`--rad` — copied byte-for-byte from `courses.html` but NOT an F0-D1 fidelity check).
  The source of truth is the static head partial [`docs/elixir/toolkit/_head.html`](../../toolkit/_head.html) (the
  canonical `HEAD_CSS` `:root` block — surfaces `--ink/--ink-2/--ink-3`; text `--cream/--cream-soft/--cream-dim`;
  accents `--gold/--blue/--sage/--burgundy/--elixir` each with a `-bright` variant except `--burgundy`; `--line`; the
  four stacks `--serif-display/--serif/--sans/--mono`; `--measure: 68ch`) — the same block any built `elixir/*.html`
  carries. A drifted hex is a contract violation even though every gate and test is green.
- **computed-style parity** — [`apps/e2e/tests/elixir-hero.spec.ts`](../../../../apps/e2e/tests/elixir-hero.spec.ts)
  run **twice**: once against the static baseline (`npm test -- elixir-hero`) and once against the Portal
  (`E2E_BASE_URL=http://localhost:4000 npm test -- elixir-hero`). It asserts the type scale, the lesson-hero layout,
  the subheader sizing, and the lede style **by computed style and geometry, not pixels** — so one spec validates
  both, and passing against both is the proof the Portal reproduced the design contract
  ([`f0.roadmap.md`](f0.roadmap.md) milestone 5).

**State it as the acceptance rule:** *"A+ across these proxies + parity green against the Portal" replaces "a human
looked at a screenshot."* The proxies are not a weaker substitute for the eye on this rung; the clamp-spacing case is
proof that the **proxy catches what the eye missed** — 204 pages shipped the silent fallback unnoticed until the
computed-style check named it.

- **Acceptance.** A styling rung is done iff the master invariant holds, the Portal is live and serves the styled
  envelope, the build gate is green across the loop, **every design-contract proxy is A+, the token-fidelity diff is
  empty, and the parity spec passes against the Portal** — and the post-build reconcile is build-grade (every F0 claim
  is `MATCH` or an explicit `[RECONCILE]`-DEFERRED). Nothing softer.

curl or the parity run fails, the rung is not shippable, regardless of a green suite.

## 6. How to read a verdict

What the Operator sees at a styling rung's close, and what each part means — the proxy-gate results are the **visual
evidence read in lieu of a screenshot**:

- **Apollo's BUILD-GRADE / BLOCKED verdict + delta table** — each F0 / F6.5.5 deliverable mapped to an as-built
  `file:line` and a verdict (`MATCH` / `STALE` / `INVENTED` / `MISSING` / `DEFERRED`): the two full-document templates
  (`page_html/home.html.heex` + `elixir.html.heex`) → do they reproduce the masters' heads/bodies with `layouts: []`
  UNCHANGED; the four per-page assets (`courses.css`/`.js` + `elixir-index.css`/`.js`) → present under
  `priv/static/assets/`, served LOCALLY at `/assets/*` (the `Plug.Static` mount corrected to `at: "/"`); the configurable
  deep-link base (`:deep_link_base_url` + `PortalWeb.deep_link_base/0`) → no host literal outside `config/`, assets stay
  Portal-local; the new `elixir/2` + `/elixir` route → facade-free. BUILD-GRADE iff all are `MATCH` or
  `[RECONCILE]`-DEFERRED. A BLOCKED verdict names the blocking deltas; the Director routes the fix to Mars
  and re-verifies — it does not ship.
- **The proxy-gate results (the screenshot stand-in)** — the **clamp-spacing parse** result on the two per-page CSS files
  (`courses.css` + `elixir-index.css`), the **token-fidelity diff** (empty = `elixir-index.css`'s `:root` matches F0-D1
  byte-for-byte; any line = a drifted token named), and the **computed-style + geometry parity** result (the `index-parity`
  spec green against `PORTAL_BASE_URL=…:4000`, both routes, with the `/elixir` `h1` > 70px and `.prose p` > 18px assertions
  that the no-space-`clamp()` collapse would fail, PLUS the assets-Portal-local assertion). These are what the Operator reads
  where a screenshot would otherwise sit.
- **The gate result Apollo reproduced** — compile clean, the test pass-count, and the determinism-loop result
  (e.g. `100/100`). Apollo re-runs the gate; the build's own report is evidence, not proof.
- **The `[RECONCILE]` markers planted forward** — the findings this rung folds into the downstream specs, each with a
  why. The Operator's preview of what the next rungs must reconcile.
- **When the Operator is asked** — Claude STOPs and ASKs only for a token-value change, a new anatomy element, or
  dropping the no-CoreComponents stance (§1). A pinned contract the reconcile already ratified is surfaced for veto,
  not asked as a blocking fork.

## 7. The LAWS, for the Operator

Five rules govern how Claude runs the styling rung; they explain behaviours the Operator will observe:

- **LAW-1 — real multi-agent spawns.** Each named peer (Venus/Mars/Apollo) is a *real* spawned subagent with its own
  context, not role-play. The Operator sees genuine `Agent` spawns and `SendMessage` reports.
- **LAW-1a — the Director does not edit code.** Once the team is spawned, the Director coordinates and ratifies but
  must not `Edit`/`Write` implementation files (the two templates, the four per-page assets, the controller/router/endpoint,
  the resolver) — peers write them. The Director reads and reviews by eye but hands every code change to Mars.
- **LAW-2 — Opus.** All peers run on Opus.
- **LAW-3 — framing.** Emitted prose (prompts, comments, specs, this guide) carries no gendered pronouns for agents,
  no perceptual or interior-state verbs applied to tools or agents, and no first-person narration — and propagates
  that rule downstream.
- **LAW-4 — one scoped commit at the rung's close.** Exactly one Director commit per rung, with a contextualized
  message, using an **explicit pathspec** — never `git add -A`. For this rung the pathspec covers the two templates
