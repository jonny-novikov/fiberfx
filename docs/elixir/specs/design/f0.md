# F0 · The Design System — the dark-editorial system the course renders in

> The foundation beneath every chapter: one coherent visual language (the jonnify dark-editorial tokens, four
> font families, and a fixed page anatomy), assembled by a deterministic pipeline (body fragment → shared head →
> bootstrap → branded build stamp) and held to an A+ bar by nine Apollo gates. Every page F1–F6 renders in it; the
> Phoenix Portal (F6) reproduces it. Built story-first and proven by invariants, following
> [the specs approach](../specs.approach.md).

This is the design-system contract. Where F5/F6 specify *what the Portal does*, F0 specifies *what every page is made
of and how it is built* — so the look, the accessibility, and the byte-for-byte reproducibility are written down and
checked, not merely embodied in CSS. It is the spec that replaces the older notion of a per-chapter build guide: one
system, documented once, that every chapter inherits.

## Goal

After F0, the design system is a specified, tooled, gated artifact rather than an implicit convention. An author writes
only a body fragment; the builder supplies the same head (tokens + base CSS), the same envelope, the same bootstrap,
and a fresh branded Snowflake stamp; nine gates certify the result A+; and `jonnify-cms` can decompose any published
page into a content store and recompose it byte-for-byte. The same system that renders the static `/elixir` pages
today is the one the Portal renders in HEEx tomorrow, unchanged in look and contract.

## Rationale (5W)

- **Why** — ~200 hand-authored pages need one visual language and one quality bar, and the F6 migration must
  reproduce those pages exactly. A design system carried only in copied CSS cannot be reasoned about, gated, or
  migrated with confidence; specifying it makes consistency, accessibility, and reproducibility verifiable claims.
- **What** — the `:root` token palette, the four font families, the page envelope and its anatomy vocabulary, the
  progressive-enhancement bootstrap, the branded build stamp, the nine Apollo gates, and the `jonnify-cms` toolchain
  that assembles, gates, and stores them.
- **Who** — authors (write fragments to the system), readers and learners (consistent, accessible, fast pages),
  coding agents (build pages to the bar without re-deciding the look), and the Portal (renders the same system).
- **When** — F0, the foundation under the whole ladder; F1–F6 content and the F6 web layer all render in it. It
  depends on nothing above it.
- **Where** — the shared head (`_head.html` / `HEAD_CSS`: tokens + base CSS), the body-fragment anatomy, and the
  assembly pipeline in `docs/elixir/toolkit/build_page.py` and its Go successor `apps/jonnify-cms`. The boundary:
  tooling owns the head, the envelope, the stamp, and the gates; an author owns only the fragment body; neither
  redefines the other's surface.

## Scope

- **In** — the token palette and the four font stacks (D1); the shared head (D2); the exact document envelope (D3);
  the body-fragment anatomy and container vocabulary (D4); the bootstrap (D5); the branded Snowflake build stamp and
  its on-page decoder (D6); the nine Apollo gates (D7); the `jonnify-cms` build/gate/store pipeline and its
  byte-parity guarantee (D8).
- **Out** — the per-lesson *content* (each page's prose, figures, and code — owned by F1–F6); the interactive-widget
  craft of a single figure (the visualization conventions, referenced not restated); the Portal's framework specifics
  (F6); the choice of KaTeX and Google Fonts as CDN dependencies (noted as the two external assets, not specified
  here).

## Deliverables

- **F0-D1 — Tokens.** The authoritative `:root` palette: surfaces `--ink/--ink-2/--ink-3`, text
  `--cream/--cream-soft/--cream-dim`, accents `--gold/--blue/--sage/--burgundy/--elixir` (each with a `-bright`
  variant except `--burgundy`), `--line`, the four font stacks (`--serif-display`, `--serif`, `--sans`, `--mono`),
  and the prose measure `--measure: 68ch`. Values are fixed and copied from the built pages.
- **F0-D2 — The shared head.** `_head.html` / `HEAD_CSS`: `<meta>` charset and viewport, the `{{TITLE}}`/`{{DESC}}`
  placeholders, the Google Fonts stylesheet link (Cormorant Garamond, PT Serif, Manrope, JetBrains Mono), and one
  inline `<style>` carrying the tokens and base CSS. It is the canonical head an author never re-declares; individual published pages may
  bake a per-chapter accent (and historical CSS revisions) into their own copy, so the as-built corpus carries several
  head variants, which the content store (F0-D8) preserves per page rather than collapsing to one.
- **F0-D3 — The document envelope.** Exactly `<!doctype html>\n<html lang="en">\n` + head + `\n<body>\n` + fragment +
  `\n` + bootstrap + `\n</body>\n</html>\n`. This is the byte contract the builder emits and the content store
  reproduces.
- **F0-D4 — Page anatomy.** The body fragment, top to bottom: skip link + sticky `.site` header with the page's
  `.route-tag`; `main#main.wrap` (the single 1080px column); a `.hero` (`.crumbs`, `.eyebrow`, one `<h1>`, `.lede`,
  `.kicker`, `.toc-mini`); content `<section>`s (`.prose` capped at `--measure`, `.deflist`, a `.fig` figure with its
  `<svg>` / `pre.code` / `.geo-readout`, a `.bridge` pairing idea↔Elixir, a `.take`); a synthesis section with a
  `.note`; a `.pager`; and the `.site-foot` with the build stamp.
- **F0-D5 — The bootstrap.** The progressive-enhancement `<script>` the builder appends: it adds `html.js`, then on
  `DOMContentLoaded` reveals `.reveal` elements via `IntersectionObserver`, short-circuiting to "reveal all" under
  `prefers-reduced-motion: reduce` or when `IntersectionObserver` is absent.
- **F0-D6 — The branded Snowflake stamp.** A 14-character id: a three-letter namespace (`TSK`) + the 11-char base62
  encoding of a 64-bit Snowflake (`timestamp(41) << 22 | node(10) << 12 | seq(12)`, epoch `1704067200000`). The page
  footer renders `{{BUILD_ID}}` and a client-side decoder expands it to its UTC timestamp using the same constants.
  It is the course's cross-system pivot-key convention.
- **F0-D7 — The nine Apollo gates.** `containers` (balanced block containers), `svg` (≥1 well-formed `<svg>`),
  `no-future` (no `/future` links), `voice` (no hype/dismissive words), `storage` (no web storage), `motion`
  (`prefers-reduced-motion` honoured), `degrade` (`.reveal` is JS-gated), `links` (internal hrefs resolve to
  live/built routes), `pager` (a pager links a real route). A+ is all nine passing; a FAIL is a hard stop.
- **F0-D8 — `jonnify-cms`.** The Go toolchain (CGO-free, stdlib + cobra + `modernc.org/sqlite`) that ports the Python
  builder: `build` assembles a fragment into the envelope; `apollo` runs the nine gates; the SQLite **content store**
  decomposes each published page into a head template + body-fragment template + per-page data and recomposes it
  byte-identically (the 204 published pages dedup to 66 distinct head templates, all reproduced byte-for-byte); `stamp` mints/decodes the branded id. A parity test proves recompose == published across the
  whole corpus.

## Invariants

- **F0-INV1 (envelope)** — every page is exactly the D3 envelope; the head and the bootstrap are builder-supplied and
  never hand-authored; an author edits only the fragment body between `\n<body>\n` and the bootstrap.
- **F0-INV2 (tokens declared once)** — the palette is declared once in the shared head and never redeclared per page;
  inside `<svg>`, presentation attributes use the literal hex mapped by role, because SVG attributes cannot read CSS
  custom properties.
- **F0-INV3 (A+ or stop)** — every page passes all nine Apollo gates; a gate FAIL blocks the page. A page that passes
  the Python `check` passes `cms check` and vice versa (gate parity).
- **F0-INV4 (degrade)** — content is visible without JavaScript: `.reveal` is JS-gated (`html.js .reveal`), reduced
  motion is honoured, and no `localStorage`/`sessionStorage` is used.
- **F0-INV5 (one decodable stamp)** — every page carries exactly one branded Snowflake stamp; the on-page decoder and
  the tool decode it to the same timestamp from the same constants; the stamp is the only per-build-varying region of
  the bytes.
- **F0-INV6 (deterministic, reproducible)** — assembly is a pure function of (fragment, head, title, desc, stamp): the
  same inputs yield byte-identical output. `jonnify-cms` reproduces every published page byte-for-byte (the parity
  test is the proof of this invariant).
- **F0-INV7 (voice)** — prose and code comments carry no hype/dismissive words (`revolutionary`, `blazing-fast`,
  `magical`, `simply`, `just`, `obviously`, `effortless`), no first-person narration, and no gendered or perceptual
  verbs applied to tools or agents.

## Definition of Done

- [ ] The token palette, the four font stacks, and `--measure` match the authoritative values (F0-D1).
- [ ] A page is the exact envelope, with the head and bootstrap builder-supplied (F0-D2, D3, D5; INV1).
- [ ] The anatomy vocabulary is used as specified, and SVG uses literal hex by role (F0-D4; INV2).
- [ ] Every page is A+ across the nine gates, and the Python and Go gate runners agree (F0-D7; INV3).
- [ ] Content degrades without JS and honours reduced motion; no web storage (F0-D5; INV4).
- [ ] Each page carries one branded stamp that decodes client-side to its build timestamp (F0-D6; INV5).
- [ ] `jonnify-cms` recomposes the published corpus byte-for-byte (F0-D8; INV6) and the parity test is green.
- [ ] Voice discipline holds across prose and comments (INV7).
