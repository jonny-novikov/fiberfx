# F5.9 · Lab: assemble the Portal engine, LiveView-ready

> Wire the F5 parts into one running system. The `EventStore` port and its in-memory and Postgres adapters, the
> `Portal` facade over the engine, the closed `%Portal.Error{}` contract, a supervision tree whose engine replays its
> log on boot, and a LiveView mount sketch that touches only the facade — then state the F6 handoff. The build target:
> `mix run` boots the app, an enroll persists an event, and a restart of the engine replays to the same state.

## Goal

After F5.9, the Portal engine runs as an assembled, supervised system with a UI-ready boundary. `Portal.Application`
starts the configured `EventStore` adapter first, then `{Portal.Engine, []}`, `:one_for_one`. The engine's `init/1`
reads its stream through the port and replays it into state; a command appends events before it evolves, so a restart
replays to the same state. The `Portal` facade sits over the engine's `command/1`/`query/1` wrappers; failures cross as
`%Portal.Error{}` from a no-catch-all `from/1`. A `PortalWeb.EnrollmentLive` sketch shows `mount/3` loading state with a
query, `handle_event/3` issuing a command and branching on the error contract, and `render/1` drawing from `assigns` —
calling only the facade. F6 then adds `PortalWeb.Endpoint` to the same tree and changes nothing below the boundary.

## Rationale (5W)

- **Why** — A system proves itself only assembled and running: starting the store before the engine, replaying the log
  on boot, and appending before evolving are what make "recover by replay" true rather than aspirational. A LiveView
  sketch over the facade proves the boundary is genuinely UI-ready, and a stated handoff means F6 is an addition, not a
  rework.
- **What** — the supervision tree, the append-before-evolve command path, the `Portal` facade, the `Portal.Error`
  contract, a LiveView mount sketch, and the F6 handoff statement.
- **Who** — Operators (a booting, supervised, recoverable system) and developers (a UI-ready facade with a finite
  outcome set).
- **When** — the ninth and final F5 rung; it assembles F5.1–F5.8 and hands off to F6, which adds only the HTTP
  endpoint.
- **Where** — `lib/portal/application.ex` (the tree), `lib/portal/engine.ex` (init + command path), `lib/portal.ex`
  (facade), `lib/portal/error.ex`, `lib/portal/event_store.ex` and its adapters, and `lib/portal_web/live/
  enrollment_live.ex` (the sketch).

## Scope

- **In** — assembling the supervision tree (store then engine, `:one_for_one`); the engine's boot replay and the
  append-before-evolve command path; the `Portal` facade over `command/1`/`query/1`; the closed `%Portal.Error{}`
  contract end-to-end; a `PortalWeb.EnrollmentLive` mount/`handle_event`/`render` sketch over the facade; and the F6
  handoff statement. The build target is the integration check.
- **Out** — the HTTP endpoint, router, and a real rendered template (→ F6); the concrete Ecto schema/migration behind
  the Postgres adapter (→ F6.3); authentication; styling; deployment. The LiveView here is a mount sketch, not a
  shipped page.

## Deliverables

- **F5.9-D1** — the assembled tree: `Portal.Application` starts `Portal.EventStore.adapter()` first, then
  `{Portal.Engine, []}`, with strategy `:one_for_one`; the engine's `init/1` reads its stream via `read_stream/1` and
  folds with `replay/1`.
- **F5.9-D2** — the append-before-evolve command path: `handle_call({:command, cmd}, …)` runs the F5.4 contract guard,
  then `decide` → `append/2` (through the port) → `evolve`, so a restart replays to the same state.
- **F5.9-D3** — the `Portal` facade over the engine wrappers: `enroll/2`, `deliver_lesson/2`, `progress_of/1` calling
  `command/1`/`query/1` (the only `GenServer.call` callers), returning `:ok | {:ok, data} | {:error, %Portal.Error{}}`.
- **F5.9-D4** — the closed `Portal.Error` contract assembled end-to-end: a `%Portal.Error{code, message, field}` with
  the closed `code` union, produced by a `from/1` with no catch-all.
- **F5.9-D5** — a `PortalWeb.EnrollmentLive` mount sketch: `mount/3` loads progress via `Portal.progress_of/1` (sets
  `:progress`, `:error`); `handle_event/3` issues `Portal.deliver_lesson/2`, re-queries on `:ok`, sets `:error` on
  `{:error, %Portal.Error{}}`; `render/1` draws from `@progress`/`@error` — touching only the facade.
- **F5.9-D6** — the F6 handoff stated: F6 adds `PortalWeb.Endpoint` to the same tree and calls the same facade;
  everything below the boundary (engine, facade, ports, error contract) is unchanged. The one structural change is the
  `Endpoint` child.
- **F5.9-D7** — verification (the build target): `mix run` boots the app; an enroll persists an event through the port;
  restarting the engine replays the log to the same state; the LiveView sketch compiles and calls only `Portal`.

## Invariants

- **F5.9-INV1 (one assembled tree)** — `Portal.Application` supervises the store then the engine, `:one_for_one`; a
  crashed engine restarts on its own and recovers by replay.
- **F5.9-INV2 (append before evolve)** — a command appends events through the port before folding them in, so
  durability and in-memory state never diverge and a restart replays to the same state.
- **F5.9-INV3 (the master invariant, complete)** — the LiveView (and any F6 web) calls only the `Portal` facade; only
  the engine wrappers call `GenServer.call`; nothing above the boundary names the engine, the store, or a repo.
- **F5.9-INV4 (closed outcomes, exhaustive)** — the LiveView consumes `{:ok, data}` and each `%Portal.Error{}` code;
  `from/1` has no catch-all.
- **F5.9-INV5 (the handoff invariant)** — F6's only structural change is adding `PortalWeb.Endpoint` to the tree; the
  engine, facade, ports, and error contract are unchanged. This is the F5→F6 contract.

## Definition of Done

- [ ] `Portal.Application` starts the store then the engine, `:one_for_one` (F5.9-INV1).
- [ ] `init/1` reads the stream and replays; a command appends before it evolves (F5.9-INV2).
- [ ] `mix run` boots, an enroll persists an event, and an engine restart replays to the same state (F5.9-D7).
- [ ] `Portal` is the only facade and the only `GenServer.call` path; the LiveView names only `Portal` (F5.9-INV3).
- [ ] The LiveView sketch handles `{:ok, data}` and every `%Portal.Error{}` code (F5.9-INV4).
- [ ] The F6 handoff is stated: F6 adds only the `Endpoint` (F5.9-INV5).
- [ ] Every deliverable maps to a user story and every invariant is checked (see `./f5.9.stories.md`).
