# F5.9 ┬╖ Lab: assemble the Portal engine, LiveView-ready

> Wire the F5 parts into one running system. The `EventStore` port and its in-memory and Postgres adapters, the
> `Portal` facade over the engine, the closed `%Portal.Error{}` contract, a supervision tree whose engine replays its
> log on boot, and a compile-only LiveView mount sketch that touches only the facade тАФ then state the F6 handoff. The
> build target: `mix run` boots the app, an enroll persists an event, a restart of the **engine process** replays to
> the same state, and the sketch **compiles** (it cannot run тАФ there is no endpoint until F6.1).

## Goal

After F5.9, the Portal engine runs as an assembled, supervised system with a UI-ready boundary. `Portal.Application`
supervises the as-built four children with **only one swap** тАФ `Portal.EventLog` is replaced by the configured
`Portal.EventStore.adapter()` тАФ giving `[Portal.Store, Portal.EventStore.adapter(), Portal.Engine, {Bandit, plug:
Portal.Web.Router, port: port()}]`, `:one_for_one`. `Portal.Store` stays (the dual-write `%Enrollment{}` read model
still backs `courses_of/1`); Bandit stays as the F5 HTTP front door, now repointed (at F5.8) to the facade and
replaced by `PortalWeb.Endpoint` only at F6.1. The engine's `init/1` reads its stream through the port and replays it
into state; a command appends events **before** it evolves тАФ the F5.8 inversion, because the port's `append/2` can fail
тАФ so an engine-process restart replays to the same state. The `Portal` facade (populated at F5.8, **confirmed** here)
sits over the engine's `command/1`/`query/1` wrappers; failures cross as `%Portal.Error{}` from a no-catch-all
`from/1`. A `PortalWeb.EnrollmentLive` sketch shows `mount/3` loading state with a query, `handle_event/3` issuing a
command and branching on the error contract, and `render/1` drawing from `assigns` тАФ calling only the facade, and
**compiling without running** (no endpoint or socket exists until F6.1). F6.1 then adds only `PortalWeb.Endpoint` to
the same tree and its `CourseController.index/2` calls `Portal.courses_of/1`, changing nothing below the boundary.

## Rationale (5W)

- **Why** тАФ A system proves itself only assembled and running: starting the store and the adapter before the engine,
  replaying the log on boot, and appending before evolving are what make "recover by replay" true rather than
  aspirational. Appending before evolving is the F5.8 inversion of the as-built order, and it earns its keep precisely
  because the `EventStore.append/2` port is fallible (a failed append must abort and leave state unchanged). A LiveView
  sketch over the facade proves the boundary is genuinely UI-ready even though it only **compiles** (no endpoint until
  F6.1, so it cannot mount or serve), and a stated handoff means F6 is an addition, not a rework.
- **What** тАФ the assembled supervision tree (the as-built four children with `EventLog тЖТ EventStore.adapter()` swapped),
  the append-before-evolve command path, the confirmed `Portal` facade, the `Portal.Error` contract, a compile-only
  LiveView mount sketch, and the F6 handoff statement.
- **Who** тАФ Operators (a booting, supervised, recoverable system) and developers (a UI-ready facade with a finite
  outcome set).
- **When** тАФ the ninth and final F5 rung; it assembles F5.1тАУF5.8 and hands off to F6, which adds only the HTTP
  endpoint.
- **Where** тАФ `lib/portal/application.ex` (the tree), `lib/portal/engine.ex` (init + command path), `lib/portal.ex`
  (facade), `lib/portal/error.ex`, `lib/portal/event_store.ex` and its adapters, and `lib/portal_web/live/
  enrollment_live.ex` (the sketch).

## Scope

- **In** тАФ assembling the supervision tree (the as-built four children тАФ `Portal.Store`, the swapped-in
  `Portal.EventStore.adapter()`, `Portal.Engine`, `{Bandit, тАж}` тАФ `:one_for_one`); the engine's boot replay and the
  append-before-evolve command path (the F5.8 inversion); the `Portal` facade over `command/1`/`query/1`; the closed
  `%Portal.Error{}` contract end-to-end; a compile-only `PortalWeb.EnrollmentLive` mount/`handle_event`/`render` sketch
  over the facade; and the F6 handoff statement. The build target is the integration check.
- **Out** тАФ the HTTP endpoint, router, and a real rendered template (тЖТ F6.1); the concrete Ecto schema/migration behind
  the Postgres adapter (тЖТ F6.3); removing the retained dual-write `Portal.Store` projection (тЖТ F6.3/F6.4); a
  progress-advancing command (тЖТ F6); authentication; styling; deployment. The LiveView here is a compile-only mount
  sketch тАФ it does not run (no endpoint or socket until F6.1), not a shipped page.

## Deliverables

- **F5.9-D1** тАФ the assembled tree: `Portal.Application` supervises the as-built four children with **only**
  `Portal.EventLog` swapped for the configured adapter тАФ `[Portal.Store, Portal.EventStore.adapter(), Portal.Engine,
  {Bandit, plug: Portal.Web.Router, port: port()}]` тАФ strategy `:one_for_one`, the adapter started before the engine.
  `Portal.Store` stays (it backs the dual-write `courses_of/1` read model); Bandit stays as the F5 HTTP front door
  (repointed to the facade at F5.8) and is replaced by `PortalWeb.Endpoint` only at F6.1. The engine's `init/1` reads
  its stream via `read_stream/1` and folds with `replay/1`.
- **F5.9-D2** тАФ the append-before-evolve command path: `handle_call({:command, cmd}, тАж)` runs the F5.4 contract guard,
  then `decide` тЖТ `append/2` (through the port) тЖТ `evolve` тАФ **inverting** the as-built F5.6 order (engine.ex:75тАУ77,
  which evolves then appends). The inversion exists because `EventStore.append/2` is fallible: a failed append aborts
  the command and leaves state unchanged, so the fold never leads durable storage and an engine-process restart
  replays to the same state.
- **F5.9-D3** тАФ the `Portal` facade (populated at F5.8, **confirmed** here) over the engine wrappers: `enroll/2`,
  `deliver_lesson/2`, `progress_of/1`, `courses_of/1` calling `command/1`/`query/1` (the only `GenServer.call`
  callers), returning `:ok | {:ok, data} | {:error, %Portal.Error{}}`. `progress_of/1` reads `enrollment.progress`,
  which is **structurally 0 by two independent mechanisms** тАФ no progress-advancing event exists yet (`LessonDelivered`
  folds a `delivered` set, not `progress`, core.ex:64тАУ68) **and** the `Portal.Store` projection mints every
  `%Enrollment{}` with `progress: 0` (engine.ex:130, :171) тАФ so it is an honest read of 0, not a calculation.
- **F5.9-D4** тАФ the closed `Portal.Error` contract assembled end-to-end: a `%Portal.Error{code, message, field}` with
  the closed `code` union, produced by a `from/1` with no catch-all.
- **F5.9-D5** тАФ a **compile-only** `PortalWeb.EnrollmentLive` mount sketch: `mount/3` loads progress via
  `Portal.progress_of/1` (sets `:progress`, `:error`; the loaded progress is structurally 0 тАФ see D3);
  `handle_event/3` issues `Portal.deliver_lesson/2`, re-queries on `:ok`, sets `:error` on `{:error, %Portal.Error{}}`;
  `render/1` draws from `@progress`/`@error` тАФ touching only the facade. It **compiles and names only `Portal`; it does
  not run** тАФ there is no endpoint or socket until F6.1, so the sketch cannot mount or serve. The build target is that
  it compiles, not that it renders.
- **F5.9-D6** тАФ the F6 handoff stated and made checkable against F6.1: F6.1 adds **only** `PortalWeb.Endpoint` to the
  same tree (f6.1.md F6.1-D2/INV2 тАФ the store and engine entries are unmoved, the strategy stays `:one_for_one`), and
  its `PortalWeb.CourseController.index/2` calls `Portal.courses_of/1` (f6.1.md F6.1-D4) тАФ the first concrete consumer
  of this facade. Everything below the boundary (engine, facade, ports, error contract) is unchanged; the one
  structural change is the `Endpoint` child replacing Bandit.
- **F5.9-D7** тАФ verification (the build target): `mix run` boots the app; an enroll persists an event through the port;
  restarting the **engine process** replays the log to the same state (the supervised adapter outlives the engine тАФ
  full-VM durability is F6.3/Postgres); the LiveView sketch **compiles** and calls only `Portal` (it does not run тАФ no
  endpoint until F6.1).

## Invariants

- **F5.9-INV1 (one assembled tree)** тАФ `Portal.Application` supervises the as-built four children with only
  `EventLog тЖТ EventStore.adapter()` swapped: `[Portal.Store, Portal.EventStore.adapter(), Portal.Engine, {Bandit, тАж}]`,
  `:one_for_one`, the adapter before the engine. `Portal.Store` and Bandit are retained (the dual-write read model and
  the F5 HTTP front door); a crashed engine restarts on its own and recovers by replay (the un-killed adapter process
  survives).
- **F5.9-INV2 (append before evolve тАФ the fallible-port inversion)** тАФ a command appends events through the port
  **before** folding them in, inverting the as-built evolve-then-append order (engine.ex:75тАУ77). The inversion is
  forced by the port being fallible (`EventStore.append/2` can return `{:error, term}`): a failed append aborts the
  command, so durability and in-memory state never diverge and an engine-process restart replays to the same state.
- **F5.9-INV3 (the master invariant, complete)** тАФ the LiveView (and any F6 web) calls only the `Portal` facade; only
  the engine wrappers call `GenServer.call`; nothing above the boundary names the engine, the store, or a repo.
- **F5.9-INV4 (closed outcomes, exhaustive)** тАФ the LiveView consumes `{:ok, data}` and each of the four
  `%Portal.Error{}` codes; `from/1` has no catch-all. The union is final at four even though only two
  (`:course_not_found`/`:already_enrolled`) have producers today тАФ the sketch still branches on all four.
- **F5.9-INV5 (the handoff invariant, checkable against F6.1)** тАФ F6.1's only structural change is adding
  `PortalWeb.Endpoint` to the tree (replacing Bandit), and its `CourseController.index/2` calls `Portal.courses_of/1`
  (f6.1.md F6.1-D2/D4/INV2); the engine, facade, ports, and error contract are unchanged. This is the F5тЖТF6 contract,
  and `f6.1` is its first concrete consumer.
