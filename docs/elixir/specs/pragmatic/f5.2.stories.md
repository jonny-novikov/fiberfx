# F5.2 · user stories

> Who needs the modeled domain, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f5.2.md`](f5.2.md).

## F5.2-US1 — Entities with an enforced shape

As a **developer**, I want each entity to be a struct that cannot be built incomplete, so that a partial or malformed
record can never flow through the system.

Acceptance criteria
- Given all enforced keys, when I build an entity struct, then it constructs successfully.
- Given a missing enforced key, when I try to build the struct, then it raises at build time.
- Given an `Enrollment`, when I omit `progress`, then it defaults to `0` and stays within `0..100` by type.

INVEST — independent; small; testable by constructing structs with and without required keys.
Priority: must · Size: 2 · Implements deliverables: F5.2-D1.

## F5.2-US2 — Bounded contexts with clear ownership

As an **architect**, I want the domain split into Accounts, Catalog, and Learning contexts that reference each other
only by id, so that a change in one area cannot reach into another.

Acceptance criteria
- Given the contexts, when they compile, then each owns its entities and is the only module that builds or persists
  them.
- Given a cross-context reference, when one context needs another's entity, then it uses a branded id, not the struct.
- Given the modules, when I inspect them, then no context aliases or constructs another context's struct.

INVEST — independent; testable by inspecting module references; encodes F5.2-INV2.
Priority: must · Size: 2 · Implements deliverables: F5.2-D2.

## F5.2-US3 — A small public API per context

As a **developer**, I want each context to expose a narrow, documented public surface, so that callers depend on a
stable API and the internals stay free to change.

Acceptance criteria
- Given a context, when I list its public functions, then they are exactly the documented set and every other function
  is private.
- Given a public function, when I read it, then it carries an `@spec`.

INVEST — independent; small; testable by inspecting exports and specs; encodes F5.2-INV3.
Priority: must · Size: 2 · Implements deliverables: F5.2-D3.

## F5.2-US4 — Enroll a learner

As a **learner** (served by the developer), I want enrolling in a course to create a durable enrollment, so that my
place in a course is recorded.

Acceptance criteria
- Given valid ids, when `Learning.enroll("USR1", "CRS1")` runs, then it returns `{:ok, %Enrollment{progress: 0}}`.
- Given that enrollment, when I read it back, then it is retrievable from the store and `Learning.courses_of/1` lists
  it.
- Given the enrollment, when it is built, then its id is a freshly minted `ENR` branded id.

INVEST — valuable (the first domain write); small; testable from `iex` and a store read.
Priority: must · Size: 3 · Implements deliverables: F5.2-D4.

## F5.2-US5 — A typed model

As a **developer**, I want every entity and public function typed, so that the domain is legible and Dialyzer can check
callers.

Acceptance criteria
- Given an entity, when I inspect it, then it has a `@type t` covering its fields.
- Given a public function, when I inspect it, then it has an `@spec` and `id`/`*_id` fields are typed as branded-id
  strings.
- Given the types, when compared to the structs, then they match.

INVEST — independent; small; testable by inspection and Dialyzer; encodes F5.2-INV5.
Priority: should · Size: 2 · Implements deliverables: F5.2-D5.
