# F5.9 · user stories

> Who the assembled system serves, what they need, and how it will be known to work. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f5.9.md`](f5.9.md).

## F5.9-US1 — A booting, supervised system

As an **operator**, I want `mix run` to boot a supervised Portal engine, so that the system starts cleanly and a
crashed engine restarts on its own.

Acceptance criteria
- Given the app, when `mix run` is invoked, then it boots without error.
- Given the tree, when it starts, then it is the as-built four children — `Portal.Store`,
  `Portal.EventStore.adapter()`, `Portal.Engine`, and Bandit — with only `EventLog` swapped for the configured
  `EventStore` adapter, started before the engine, `:one_for_one`.
- Given an engine crash, when it happens, then `:one_for_one` restarts the engine and it recovers by replay.

INVEST — independent; testable by booting and by a forced crash; encodes F5.9-INV1.
Priority: must · Size: 2 · Implements deliverables: F5.9-D1.

## F5.9-US2 — A durable enroll that survives an engine restart

As an **operator**, I want an enroll to persist and survive an engine restart, so that no learner progress is lost when
the engine process crashes.

Acceptance criteria
- Given a successful enroll, when it runs, then its event is appended through the port **before** state evolves, and a
  failed append aborts the command leaving state unchanged — the same fallible-port inversion stated at F5.8.
- Given an **engine-process** restart (the supervised adapter survives the engine's crash; `init/1` re-reads it), when
  the engine boots, then `read_stream/1` + `replay/1` reproduce the pre-restart state.
- Given a query after an engine restart, when it runs, then it reflects the enroll made before the restart. (A
  **full-VM** restart empties the in-memory adapter; durability across that is the Postgres adapter at F6.3.)

INVEST — independent; testable by persist-then-engine-restart; encodes F5.9-INV2.
Priority: must · Size: 3 · Implements deliverables: F5.9-D2.

## F5.9-US3 — A UI-ready facade

As a **developer**, I want a LiveView sketch that drives the engine through the facade, so that the boundary is
confirmed genuinely UI-ready before F6.

Acceptance criteria
- Given `mount/3`, when it runs, then it loads progress via `Portal.progress_of/1` into `assigns` — which reads
  `enrollment.progress`, structurally `0` today for two independent reasons (no progress-advancing event yet —
  `LessonDelivered` folds into a `delivered` set, never `enrollment.progress`; and the Store projection is pinned at
  `progress: 0`), so the sketch mounts on `0` and a progress calculation is a later/F6 concern.
- Given `handle_event/3`, when a lesson is delivered, then it calls `Portal.deliver_lesson/2`, re-queries on `:ok`, and
  sets `:error` on `{:error, %Portal.Error{}}`.
- Given the sketch, when inspected, then it names only `Portal` — no engine, store, or repo — and **compiles only**: it
  does not run, because no endpoint or socket exists until F6.1.

INVEST — independent; testable by the sketch's call graph and a compile; encodes F5.9-INV3.
Priority: must · Size: 3 · Implements deliverables: F5.9-D3, F5.9-D5.

## F5.9-US4 — Exhaustive outcome handling

As a **developer**, I want the UI to handle a finite outcome set, so that every result is handled and a new code forces
a new branch.

Acceptance criteria
- Given the facade's result, when consumed, then the sketch handles `{:ok, data}` and one branch per `Portal.Error`
  code (including the reserved `:lesson_locked` and `:invalid_progress`, which have no producer yet).
- Given `from/1`, when inspected, then it has no catch-all.

INVEST — independent; testable by the sketch's exhaustiveness; encodes F5.9-INV4.
Priority: should · Size: 1 · Implements deliverables: F5.9-D4.

## F5.9-US5 — A clean F6 handoff

As an **architect**, I want F6 to be a single addition, so that adding the web changes nothing below the boundary.

Acceptance criteria
- Given F6, when it starts, then its one structural change is adding `PortalWeb.Endpoint` to the tree (replacing
  Bandit); store and engine are unmoved, `:one_for_one`.
- Given the engine, facade, ports, and error contract, when F6 lands, then they are unchanged — the first concrete
  consumer, `PortalWeb.CourseController.index/2`, calls **only** `Portal.courses_of/1`.
- Given the handoff, when read, then that one structural change and that consumer are documented as a forward reference
  checkable against [`../phoenix/f6.1.md`](../phoenix/f6.1.md).

INVEST — independent; testable by diffing the tree against F6.1; encodes F5.9-INV5.
Priority: must · Size: 1 · Implements deliverables: F5.9-D6.
