# F5.9 · user stories

> Who the assembled system serves, what they need, and how we will know it works. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f5.9.md`](f5.9.md).

## F5.9-US1 — A booting, supervised system

As an **operator**, I want `mix run` to boot a supervised Portal engine, so that the system starts cleanly and a
crashed engine restarts on its own.

Acceptance criteria
- Given the app, when I run `mix run`, then it boots without error.
- Given the tree, when it starts, then the `EventStore` adapter starts before the engine.
- Given an engine crash, when it happens, then `:one_for_one` restarts the engine.

INVEST — independent; testable by booting and by a forced crash; encodes F5.9-INV1.
Priority: must · Size: 2 · Implements deliverables: F5.9-D1.

## F5.9-US2 — A durable enroll that survives a restart

As an **operator**, I want an enroll to persist and survive a restart, so that no learner progress is lost to a crash.

Acceptance criteria
- Given a successful enroll, when it runs, then its event is appended through the port before state evolves.
- Given the engine restarting, when it boots, then `read_stream/1` + `replay/1` reproduce the pre-restart state.
- Given a query after restart, when it runs, then it reflects the enroll made before the restart.

INVEST — independent; testable by persist-then-restart; encodes F5.9-INV2.
Priority: must · Size: 3 · Implements deliverables: F5.9-D2.

## F5.9-US3 — A UI-ready facade

As a **developer**, I want a LiveView sketch that drives the engine through the facade, so that I can confirm the
boundary is genuinely UI-ready before F6.

Acceptance criteria
- Given `mount/3`, when it runs, then it loads progress via `Portal.progress_of/1` into `assigns`.
- Given `handle_event/3`, when a lesson is delivered, then it calls `Portal.deliver_lesson/2`, re-queries on `:ok`, and
  sets `:error` on `{:error, %Portal.Error{}}`.
- Given the sketch, when inspected, then it names only `Portal` — no engine, store, or repo.

INVEST — independent; testable by the sketch's call graph; encodes F5.9-INV3.
Priority: must · Size: 3 · Implements deliverables: F5.9-D3, F5.9-D5.

## F5.9-US4 — Exhaustive outcome handling

As a **developer**, I want the UI to handle a finite outcome set, so that every result is handled and a new code forces
a new branch.

Acceptance criteria
- Given the facade's result, when consumed, then the sketch handles `{:ok, data}` and one branch per `Portal.Error`
  code.
- Given `from/1`, when inspected, then it has no catch-all.

INVEST — independent; testable by the sketch's exhaustiveness; encodes F5.9-INV4.
Priority: should · Size: 1 · Implements deliverables: F5.9-D4.

## F5.9-US5 — A clean F6 handoff

As an **architect**, I want F6 to be a single addition, so that adding the web changes nothing below the boundary.

Acceptance criteria
- Given F6, when it starts, then it adds only `PortalWeb.Endpoint` to the tree.
- Given the engine, facade, ports, and error contract, when F6 lands, then they are unchanged.
- Given the handoff, when read, then the one structural change is documented.

INVEST — independent; testable by diffing the tree; encodes F5.9-INV5.
Priority: must · Size: 1 · Implements deliverables: F5.9-D6.
