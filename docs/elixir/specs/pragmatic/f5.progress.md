> increment — what was **exceptional**, what was **good**, what **went not well**, **opportunities** to improve the
> process, and **opened gaps / stubs** carried forward — plus the **spec refinements** each increment feeds back into
> the source of truth. This is the "review → feedback → adapt" half of the roadmap's loop made durable; the specs
> ([`pragmatic.md`](pragmatic.md) + the `f5.N` triads) stay the single source of truth, and this log records why they
> changed. Newest entry appended at the end.

The engine is built as a real Mix umbrella at `/Users/jonny/dev/jonnify/echo` (apps `echo_data` + `portal`; asdf
Elixir 1.18.4 / OTP 28). The umbrella is not the Go `jonnify` server documented in the repo-root `CLAUDE.md` — it is a

## Iteration 1 — F5.1–F5.3 · the walking skeleton (shipped 2026-06-02)

**Delivered.** F5.1 (a supervised app answering HTTP on `:4000` through a thin Bandit+Plug seam, minting branded
Snowflake ids, stubbed `Portal.Engine` boundary), F5.2 (seven entity structs in the Accounts / Catalog / Learning
contexts over a key/value store), F5.3 (the `enroll` use case wired end-to-end `POST /enroll` → engine → context →
store → `201`, plus a `GET /lessons/:id` read slice).

**Verified.** `mix test` green (13 `portal` + 2 `echo_data` example tests, no mocks); a live node returns
`201 / 200 / 404 / 200 / 404` for the five smoke curls on `:4000`; killing `Portal.Engine` yields a new pid and
continued service (INV3 as a running gate); the boundary-purity grep over `lib/portal_web/` is empty.

### Exceptional

- **The master invariant is compiler-enforced, not just documented.** Modelling the core as a separate umbrella app
  (`apps/portal`) whose `mix.exs` lists no Phoenix/Ecto means illegal coupling does not compile — the spec asked for a
  "discipline," the umbrella turns it into a build error. F6 can swap Bandit→Phoenix above the boundary without the
  core able to depend downward.
- **`echo_data` as a first-class library app.** The reference `EchoData.Base62` + `EchoData.Snowflake` primitives
  became a pure, tested umbrella app, and `Portal.ID` is a thin wrapper over them. Identity has one home, reused
  verbatim, with its own tests.
- **A genuine no-mocks walking skeleton.** `enroll_slice_test.exs` drives the real router → engine → contexts → store
  with real minted ids end to end — the skeleton actually walks, which is the whole point of a tracer bullet.

### Good

- **The F4 `Portal.Store` gap was closed cleanly.** No F4 CHAMP code existed and F5.2 was its first caller, so the
  iteration stood up a minimal in-memory GenServer (a namespace-partitioned map-of-maps) behind the exact
  `get/2 · all/2 · put/1` surface. The real branded CHAMP swaps in later with zero caller change.
- **One result→status mapper.** The router routes every boundary result through a single `status_for`-style clause set,
  so F5.4's reasons and F5.8's `%Portal.Error{}` add clauses, not rewrites.
- **`@derive {Jason.Encoder}` on the entities** keeps the web layer "dumb" — no struct→map mapping leaks domain shape
  into the router.
- **Supervision is tested as a running gate**, not asserted in prose: monitor → `Process.exit(Portal.Engine, :kill)` →
  await `:DOWN` → poll for a fresh pid → assert a request still succeeds.

### Went not well

- **`"USR1"` / `"CRS1"` placeholder ids are not valid branded ids.** The spec prose used them as stand-ins, but a
  branded id is 14 chars (3-letter namespace + 11-char Base62). This produced a real correctness question — *"why is
  `USR1` considered valid?"* — and the honest answer is it is not. Fix: a new `Portal.ID.valid?/1` (full-format check),
  every placeholder in seeds/tests replaced with `Portal.ID.new/1`-minted ids, and linked references
  (`lesson.course_id == course.id`). The specs still carry the placeholders and must be swept (see refinements below).
- **Test assertions coupled to stub internals had to evolve across rungs.** `supervision_test` asserted
  `{:error, :not_implemented}` (the F5.1 stub) but the F5.3 real engine returns `{:error, :unknown_command}` for an
  incomplete command — the assertion was loosened to `{:error, _}`. A test pinned to a stub's exact reason is brittle
  by construction; pin behaviour, not the placeholder.
- **A destructive shell step crashed the runtime.** An `rm -rf` issued from inside the directory being removed left the
  shell with no cwd and crashed the BEAM. Process-hygiene lesson: never delete the shell's own working directory.
- **A small spec inaccuracy surfaced.** F5.1-D2 states a 41-bit Snowflake timestamp; the as-built `EchoData.Snowflake`
  is 42 bits (`<<22 | <<12 |` layout otherwise identical, epoch `1_704_067_200_000`). Round-trip is unaffected, but the
  spec should match the primitive.

### Opportunities to improve (process)

- **Sharpen specs from build-learnings immediately.** This log is that step; the refinement table below is its output.
- **Give implementation agents a "ground-facts" brief.** Fan-out authoring on this Portal API has previously invented
  or redefined arity / arg-order / return types (gates pass the drift). The F5.4–F5.7 implementation should hand each
  agent the *real* surface — `Portal.ID.{new,valid?,namespace,snowflake,at}/1`, the `dispatch/1` + `query/2` boundary,
  `Portal.Store.{get,all,put}`, the seven entity shapes — with an explicit "do not invent; cite the spec" rule and an
  adversarial verify pass.
- **`valid?/1` is the boundary guard, not `namespace/1`.** `namespace/1` slices the first three bytes only — `"USR1"`
  passes a `namespace == "USR"` check. The F5.4 precondition must use `valid?/1`.
- **Soften or author the `build-guide/` companions.** Every `f5.N.llms.md` tells an agent to "read
  `build-guide/f5-0N-*.md` first," but `../../build-guide/` does not exist. Implementation agents should be pointed at
  the real triad (`f5.N.md` + `.stories.md`) until the companions are authored.
- **Document the Portal build conventions where a fresh session can find them.** The repo-root `CLAUDE.md` scopes
  itself to the Go server and declares other modules out of scope, so a dedicated `echo/CLAUDE.md` (umbrella layout,
  the master invariant, the run/test commands, the F5→F6 arc) is the right home — proposed, pending confirmation of the
  intended target.

### Opened gaps & stubs (carried forward)

- **`Portal.Store` is an in-memory stand-in**, not the F4 branded CHAMP — empties on restart; swap behind the same
  surface at F4 / F5.8.
- **`Portal.Engine` is a `GenServer.call` boundary but state is not event-sourced yet** — `decide`/`evolve` is F5.5,
  the supervised state home + replay is F5.6.
- **No durable `EventStore`** (F5.8); **no `%Portal.Error{}` struct yet** (F5.4 introduces it — see refinements).
- **Seeds are per-node and in-memory** — `mix run priv/seeds.exs` runs in a separate node that exits; live `curl` needs
  one `iex -S mix` node (`Code.eval_file("apps/portal/priv/seeds.exs")`).
- **`query/2` returns lists** for `:courses_of` (a learner's enrollments) — the boundary type must admit `[term]`, not
  only `{:ok, term} | :error`.
- **`build-guide/f5-0N-*.md` companions are absent** (referenced by every `.llms.md` and `pragmatic.md`).

## Iteration 2 — F5.4 · the enroll contract (shipped 2026-06-03)

**Delivered.** A closed error vocabulary as data — `Portal.Error` (new `apps/portal/lib/portal/error.ex`): a struct
`%Portal.Error{code, message}` with `@enforce_keys [:code, :message]`, a closed `@type code :: :course_not_found |
:already_enrolled`, and `new/1` that builds it from a code via a private `message/1` (no `@derive Jason.Encoder`, no
`from/1` yet — its moduledoc records F5.8 as the extension point). `Portal.Learning.enroll/2` rewritten as a no-`else`
`with` chain — `check_ref(user_id, "USR")` → `check_ref(course_id, "CRS")` → `course_exists(course_id)` →
`not_already_enrolled(user_id, course_id)`, then a success branch that mints an `ENR` id, calls `Portal.Store.put/1`,
and returns `{:ok, enrollment}`; each precondition folds a rejection to `Error.new(:course_not_found)` /
`Error.new(:already_enrolled)`, and `check_ref/2` gates on `Portal.ID.valid?/1` (full 14-char format) **and**
`Portal.ID.namespace/1` so a bare `"USR1"` is now rejected. The router's `POST /enroll` extends the F5.3 mapper —
`{:ok, enrollment}` → `201 %{data: %{id: …}}`, `{:error, %Portal.Error{code, message}}` → `422 %{error: %{code,
message}}` — with **no** catch-all clause, so a non-`%Portal.Error{}` value crashes (INV4). `Portal.Store.reset/0`
(new) synchronously clears every namespace (`GenServer.call(:reset)` → `{:reply, :ok, %{}}`) as the determinism fix.
A property suite `apps/portal/test/portal/enroll_contract_test.exs` (new, `async: false`, `ExUnitProperties`, a `setup`
calling `Portal.Store.reset/0`) covers the postcondition (INV5) and the `0..100` state invariant (INV6) as StreamData
properties, one example per error code (each asserting `Store.all("ENR")` is unchanged on rejection), a duplicate
enroll yielding exactly one stored enrollment, a per-code message assertion, and an impossible-state test
(`progress: 101` tripping a `when p in 0..100` guard → `FunctionClauseError`). `apps/portal/mix.exs` gained
`{:stream_data, "~> 1.0", only: :test}` (mix.lock pins 1.3.0); `enroll_slice_test.exs` gained a `422` example;
`learning_test.exs` now seeds the `Course` before calling `enroll/2`.

**Verified.** `mix test` green — 2 properties + 21 `portal` tests + 2 `echo_data` example tests, 0 failures. The
determinism gate passes across six seeds (103518, 312540, 0, 42, 999999, 777, all 0 failures); seed 312540 **failed**
before the `reset/0` fix. The INV1 boundary-purity grep over `apps/portal/lib/portal_web/` is empty; `mix compile
--warnings-as-errors` is clean; `mix deps.get` resolved stream_data 1.3.0.

### Exceptional

- **The error vocabulary is data, not strings.** `%Portal.Error{}` carries a closed `@type code` union, and the web
  layer maps a code to `422` by matching the struct — so adding a code is a compile-checked change, not a hunt for the
  matching string literal. The closed type is the contract; the compiler is the enforcer.
- **The no-`else` `with` chain reads as the contract itself.** Each precondition is one line that folds to a
  `%Portal.Error{}`, and the success branch is the only site that mints an id or writes to the store — a rejected
  enroll mints nothing and writes nothing. Fail-fast ordering and effect-free rejection are the structure, not a
  comment about the structure.
- **The invariants are property-tested, not example-only.** StreamData drives the INV5 postcondition (a successful
  enroll yields `{:ok, %Enrollment{progress: 0}}` and stores it) and the INV6 `0..100` invariant across many generated
  inputs, so the guarantee holds over a space of cases rather than a handful of hand-picked ones.

### Good

- **The single result→status mapper was extended, not rewritten.** The `%Portal.Error{}` → `422` clause slotted in
  beside the F5.3 `201` / `404` paths with no churn to either — the Iteration-1 "one mapper" decision paid exactly the
  dividend it was predicted to.
- **"Impossible states crash, never return" is a running gate.** INV4 is modelled as a guard clause
  (`assert_progress_in_range!/1`, `when p in 0..100`) plus an `assert_raise FunctionClauseError` test — a behaviour the
  suite checks, not a sentence in a spec.
- **`Portal.ID.valid?/1` is now the boundary guard.** The full 14-char format check replaces the partial
  `namespace/1`-only gate at the enroll precondition, closing the `"USR1"`-passes hole the Iteration-1 retrospective
  flagged it must.

### Went not well

- **The contract suite was flaky — seed-dependent — and a single green run hid it.** One single-seed run reported 0
  failures; an independent multi-seed verification caught seed 312540 failing (`:course_not_found` expected,
  `:already_enrolled` observed). Root cause: a cross-process branded-id collision. `EchoData.Snowflake` keeps its
  12-bit sequence in the **process dictionary** (reset per process) and `Portal.ID.new` hardcodes `worker_id: 1`, so
  two test processes minting in the same millisecond can produce the **same** branded id — a freshly-minted
  "nonexistent" course id can equal another test's stored course id. Fixed with `Portal.Store.reset/0` + a per-test
  `setup` reset, **not** by changing `id.ex`: production is single-writer through the serialized `Portal.Engine`, so
  the collision cannot occur there. The lesson: a single green run is not evidence of determinism.
- **A harness ENOSPC condition produced spurious mid-suite I/O failures.** The tmp overlay at
  `/private/tmp/claude-501/<proj>/tasks` reaching 0 MB free surfaced as ExUnit I/O errors orthogonal to the logic
  flake. Mitigation: run `mix test` with `TMPDIR=/tmp`.

### Opportunities to improve (process)

- **Make adversarial multi-seed verification the standard gate for any property- or concurrency-touching suite.** A
  single seed fixes test order and hides order-dependent collisions; the independent re-run, not the implementer's
  single report, is the ratification. The seed-312540 catch is the proof of why one report is insufficient.
- **The id-collision is a latent design footgun to schedule out.** It is production-safe today (single-writer), but any
  future multi-process test or multi-writer runtime hits it. Options for a later rung: a per-BEAM-process or
  scheduler-derived `worker_id`, or moving the sequence out of the process dictionary. The constraint belongs where a
  fresh session finds it — the proposed `echo/CLAUDE.md`.
- **Hand implementation agents the real Portal surface as ground-facts under a no-invent rule with an adversarial
  verify pass.** Carried from Iteration 1, applied here, and it held — no Portal API drift surfaced in F5.4.

### Opened gaps & stubs (carried forward)

- **Latent id-uniqueness defect.** `Portal.ID.new` hardcodes `worker_id: 1` and the Snowflake sequence is per-process,
  so same-millisecond cross-process mints collide. Masked in production by the single-writer engine; a hazard for any
  multi-process test, mitigated per-suite via `Store.reset/0`.
- **`%Portal.Error{}` is not yet `Jason.Encoder`-derivable and has no `from/1`** — F5.8 extends it (adds `:field`, the
  full union, `from/1`).
- **The engine is still not event-sourced** — `decide`/`evolve` is F5.5; the supervised state home + replay is F5.6.
- **`%Portal.Error{}` is wired only into `enroll`** — other use cases adopt it as they gain contracts.
- **(carried) `Portal.Store` is an in-memory stand-in**, and the `build-guide/` companions are still absent.

## Iteration 3 — F5.5 · commands, queries & events (shipped 2026-06-03)

**Delivered.** The pure functional core as the canonical Decider — `Portal.Engine.Core` (new
`apps/portal/lib/portal/engine/core.ex`): `initial_state/0` (`%{enrollments: %{}, delivered: %{}}`), `evolve/2` (one
clause per event type, `(event, state)` arg order, `update_in` folds), `replay/1` (`Enum.reduce(log, initial_state(),
&evolve/2)`), `decide/2` (pure, events-only, reading the occurrence time `at` from the command 4-tuple), `authorize/2`
(the F5.4 contract at the boundary — reads the folded state, returns bare-atom reasons), and `query/2` (a pure read of
folded state for `{:enrollments, user_id}`). Two past-tense event structs (new `apps/portal/lib/portal/learning/events/`),
owned by the Learning context: `LearnerEnrolled` (`user_id, course_id, at`) and `LessonDelivered`
(`user_id, lesson_id, at`) — each with `@enforce_keys`, `@type t`, and `@derive Jason.Encoder`. A new
`apps/portal/test/portal/engine/core_test.exs` with example tests for `decide`/`evolve`/`replay`/`authorize`/`query`
and a StreamData property (`replay(log) == incremental fold`, `max_runs: 200`) drawn over a fixed 2-element id pool.

**Verified.** The full suite is green — 3 properties + 37 `portal` tests + 2 `echo_data`, 0 failures — across six seeds
(0, 1, 42, 312540, 999999, 7). Purity (INV2) holds: the only call-out in `core.ex` is `Portal.Catalog.course/1` inside
`authorize/2`; `decide`/`evolve`/`replay`/`query` are I/O-free. Scope held — only new files were added; the F5.4 path,
the `Portal.Engine` GenServer, the web boundary, and `mix.exs` are all untouched.

### Exceptional

- **The pure core needs no store, and the structure proves it.** Functional-core / imperative-shell is enforced by
  module boundary, not convention — `core.ex` carries no GenServer, no `Portal.Store`, and the purity grep shows
  `decide`/`evolve`/`replay`/`query` calling nothing impure (the lone `Portal.Catalog.course/1` is confined to
  `authorize/2`, the boundary). The split is a checked property of the file, not a comment about intent.
- **`at` is the command tuple's 4th element, so `decide` has no hidden clock.** Threading the occurrence time in as
  data keeps `decide` a deterministic function of `(state, command)` — there is no `now()` reachable from the core, so
  identical inputs always produce identical events. Determinism is a consequence of the signature, not of discipline at
  the call site.
- **The INV5 test is the canonical-Decider proof.** A duplicate enroll is rejected by `authorize/2`, and yet `decide`
  applied to the already-folded state still returns `[%LearnerEnrolled{}]` — `decide` proposes facts and carries no
  error channel; the boundary decides admissibility. The test states the pattern's defining property as an executable
  assertion rather than prose.
- **The `replay == fold` property forces the collisions it must survive.** Drawing commands from a fixed 2-element id
  pool makes repeats and overlaps frequent, so the StreamData property actually exercises the idempotent set-fold
  instead of skimming a sparse space of all-distinct ids.

### Good

- **The F5.4 `Portal.Error.new/1` seam is exactly the F5.6 hook it was built to be.** Designed in F5.4 to take a bare
  code, it is precisely the atom → `%Portal.Error{}` mapper the F5.6 GenServer will place over `authorize/2`'s
  bare-atom reasons — the F5.4 signature was the planned join, and F5.5 confirmed the fit without changing it.
- **The design's explicit "what stays untouched" scope-line kept the build to new files only.** Naming the F5.4 path,
  the GenServer, the web boundary, and `mix.exs` as out-of-bounds up front meant the increment added `core.ex`, the two
  event structs, and one test file — and nothing else moved.
- **`authorize/2` mirrors `Learning.enroll/2`'s with-chain but reads the fold, not the Store.** The F5.4 contract shape
  carried over intact; only its state source changed (folded state in place of `Portal.Store`), which is what makes the
  two paths reconcilable at F5.6 rather than divergent designs.

### Went not well

- **Two contract implementations now coexist.** The F5.4 `Learning.enroll/2` reads the `Portal.Store`; the F5.5
  `Core.authorize/2` reads the folded state. This is intentional transitional duplication — F5.6 reconciles it when the
  GenServer becomes the single home for folded state — but until then the `check_ref` / `course_exists` logic is
  written twice and must be retired, not left to drift.
- **The spec surfaces lagged the resolved body.** `f5.5.stories.md` (US3/US5), `f5.5.llms.md` (R3 plus three more
  spots), and `f5.5.md` D6 each still framed `decide` with an error channel — the contract *inside* `decide` — and
  omitted `at`, while the spec body (the resolved Design-decision, INV5, D3, Scope) and `decider-pattern.md` §2.1
  already encoded the canonical form. The agent brief (`llms.md`) was the **most** stale and, taken literally, would
  have directed construction of the non-canonical Decider. The build proceeded from the ratified design and its binding
  decisions instead of the brief — ground-facts overrode the stale surface — but a brief that contradicts the body it
  briefs is a latent mis-build waiting for a less careful pass.

### Opportunities to improve (process)

- **Make "the spec BODY is authoritative; the brief and stories may lag it" an explicit rule.** The `f5.5.md` body and
  `decider-pattern.md` §2.1 were correct while the `.llms.md` brief and `.stories.md` were stale — so any implementation
  pass should verify the `decide` / contract shapes against the body **and** `decider-pattern.md` §2.1 before trusting
  the brief. The rule belongs where a fresh session finds it (the proposed `echo/CLAUDE.md`).
- **Keep build → adversarial-multi-seed-verify as the standing ratification step.** The gate held again — the
  independent re-run across six seeds, not the implementer's single report, is what ratifies a property- or
  concurrency-touching increment. Carried from Iterations 1–2, applied here, still earning its place.

### Opened gaps & stubs (carried forward)

- **The pure core has no runtime home yet.** F5.6 stands up the supervised `Portal.Engine` GenServer: it holds the
  folded state, recovers by `replay` at `init`, routes `dispatch`/`query` through the fold, and maps `authorize/2`'s
  bare atom → `%Portal.Error{}` via `Portal.Error.new/1`.
- **The two contract paths coexist until F5.6 reconciles them** — `Learning.enroll/2` (reads the Store) and
  `Core.authorize/2` (reads the fold); the duplicated `check_ref` / `course_exists` logic retires there.
- **The live `Portal.Engine.query(:courses_of)` still reads the Store** — F5.6 decides `:enrollments` (the new folded

  surface absent from the inventory does not exist and is not asserted.
- **An order inversion was named with its cause, not smoothed over.** As-built F5.6 runs `decide → evolve → append`,
  which is safe only because `Portal.EventLog.append/1` is in-process and total. F5.8 introduces a **fallible**
  `EventStore.append/2` (`{:error, term}`, especially from Postgres), so the order **must invert** to
  `decide → append → evolve`: a failed append aborts the command and leaves state unchanged, so the in-memory fold
  never leads durable storage. The refinement states F5.8-INV5 / F5.9-INV2 as that inversion **with its reason**
  ("F5.8 inverts F5.6's order because the port can fail"), rather than as a property that was always true.
- **A dropped supervision child was caught by diffing the spec tree against the running tree.** The F5.9 body's
  `[adapter, engine]` tree silently dropped both `Portal.Store` (which the F5.6 dual-write read model still requires)
  and the Bandit front door. Diffing against the as-built four-child tree surfaced the gap; the corrected tree is the
  as-built list with **only** `EventLog` swapped for `EventStore.adapter()`, which is also exactly what makes the F6.1
  handoff a single `Endpoint` addition rather than a re-shuffle.

### Good

- **The transitional dual-write was named as retained, closing a removal hazard.** The specs were silent on the F5.6
  dual home (held fold = truth; `Portal.Store` `%Enrollment{}` rows = the byte-for-byte web read model). Silence invites
  an implementer to "tidy" it away and break the F5.3/F5.4 web. The refined bodies mark it **retained-and-transitional**
  — kept through F5.9, superseded at F6 (F6.3 durable `EventStore` + F6.4 contexts/read-models) — so the projection is
  protected by the spec, not by luck.
- **The closed error union is honest about which codes have producers.** The union is final at four codes, but only
  `:course_not_found` and `:already_enrolled` are produced today (by the enroll `authorize` path); `:lesson_locked`
  and `:invalid_progress` are **reserved** — no deliver-lesson gate or progress validation is built. `from/1` still
  maps all four with **no catch-all**, so adding a producer later needs no `from/1` change. Stating this prevents an
  agent from inventing producers to "complete" the union.
- **The F6 handoff became checkable by naming its first consumer.** F5.9-INV5 ("F6 adds only `PortalWeb.Endpoint`") now
  cites the concrete consumer from `../phoenix/f6.1.md` — `PortalWeb.CourseController.index/2` calling **only**
  `Portal.courses_of/1`, with the tree change being `Endpoint` replacing Bandit — so the handoff is a forward reference
  a reader can verify against F6.1, not a bare assertion.
- **The stories' Coverage lines absorbed the rename and the honesty notes without losing the one-Deliverable-per-story
  map.** The boundary-name pair, the structurally-0 `progress_of/1` read (D-J), and the engine-process restart
  sharpening (D-N) folded into the acceptance criteria while every refined Deliverable kept exactly one home.

### Went not well

- **Six files drifted from the core in lockstep because they were authored ahead of the build.** F5.8 and F5.9 were
  written when the engine boundary was assumed to be `command/1` + `query/1` and `Portal` was assumed creatable fresh;
  the F5.4–F5.7 build then settled `dispatch/1` + `query/2`, the `Portal` stub, the separate `EventLog`, and the
  dual-write — and the unbuilt triads were not re-synced as each rung landed. The drift was uniform (every file named
  the wrong boundary verb or the wrong creation verb), which is the signature of specs that ran ahead of the code
  rather than tracking it.
- **The "extend vs create" verb was wrong per file in more than one place.** Four modules exist and are
  changed/populated/extended (`portal.ex`, `engine.ex`, `error.ex`, `application.ex`); four are genuinely new and
  created (`event_store.ex`, the two adapters, `enrollment_live.ex`). The briefs and bodies mixed these, so a literal
  reading would have an agent recreate existing modules — the same class of "create what already exists" error the F5.7
  refinement corrected for the test files, recurring here for the lib modules.

### Opportunities to improve (process)

- **Re-sync an unbuilt rung's triad at the close of every preceding rung, not in a batch at build time.** The drift
  corrected here accumulated across F5.5, F5.6, and F5.7 — each of which changed a surface F5.8/F5.9 name (the boundary
  verbs, the `EventLog`, the dual-write) without updating the downstream specs. Folding each rung's surface changes
  forward into the next unbuilt triad as part of that rung's retrospective would keep the lag at one rung instead of
  letting it compound into a fourteen-delta reconciliation.
- **Keep the ground-facts surface — a read-from-code inventory plus a named-delta list — as the standing input to any
  spec-alignment pass.** Carried in spirit from the Iteration 1–7 "hand agents the real surface under a no-invent rule"
  through-line, here applied to spec refinement rather than implementation: the inventory is what made "invent nothing"
  enforceable, and the numbered deltas are what made the cross-file verification a checklist (every delta has an owner
  and a correction) rather than a re-read.
- **An invariant should name the granularity it claims.** The durability invariant held for an **engine-process**
  restart but read as an unqualified "survives a restart," which a reader could take as full-VM durability the
  in-memory adapter does not provide. Sharpening it to "engine-process restart" (full-VM durability deferred to
  F6.3/Postgres) is the same lesson as the F5.6 "name WHAT the invariant protects" — state the scope, not just the
  guarantee.

### Opened gaps & stubs (carried forward)

- **Two `Portal.Error` codes are reserved with no producer.** `:lesson_locked` and `:invalid_progress` are in the final
  union and mapped by `from/1`, but nothing produces them: there is no deliver-lesson lock gate and no progress
  validation. Their producers arrive with a later lesson-delivery contract (F6); until then the no-catch-all `from/1`
  maps them defensively and no path reaches them.
- **`progress_of/1` reads a structurally-0 value — for two independent reasons.** No progress-advancing event exists
  (`LessonDelivered` folds into a `delivered` set and never touches `enrollment.progress`), **and** the Store
  projection is pinned at `progress: 0` at every mint and reproject — so `enrollment.progress` stays `0` and the F5.9
  LiveView sketch mounts on it reading `0`. A progress calculation is a later/F6 concern; the spec carries an explicit
  honesty note naming both mechanisms so the absence is documented, not hunted for, and so a reader does not assume
  adding the event alone would lift the value. (This is the build-side of F5.7's deferred numeric-progress monotonic
  property.)
- **The LiveView sketch is compile-only — it cannot run at F5.9.** No endpoint or socket exists until F6.1, so
  `enrollment_live.ex` compiles and names only `Portal` but does not mount or serve. The F5.9 build target is therefore
  `mix run` boots **and** the sketch **compiles**, not "renders."
- **ENR-id instability across a restart is unchanged by F5.8/F5.9.** The F5.6-recorded gap (`reproject_enrollments`
  re-mints fresh `ENR` ids, so a `data.id` cached from `POST /enroll` dangles after a restart) is neither closed nor
  worsened here; it is removed by the F6 fold-only collapse. Carried, not addressed, because this pass changed no code.
- **Downstream reconciliation — `f6.1.md`'s supervision tree omits the `Portal.Store` this refinement retains.** This
  F5.9 pass keeps the dual-write `Portal.Store` `%Enrollment{}` projection (D-E) and `courses_of/1` reads it; the F6
  handoff (D-K) cites `f6.1.md` F6.1-D4, whose `CourseController.index/2` calls `Portal.courses_of/1` — which depends on
  `Portal.Store` being started. But `f6.1.md`'s F6.1-D2 tree is the three-child `[Portal.EventStore.adapter(),
  {Portal.Engine, []}, PortalWeb.Endpoint]`, which **never starts `Portal.Store`**: that tree predates this refinement
  (it was written before F5.9 pinned the Store as retained), so as specced F6.1 would boot a controller whose read model
  is unstarted. The reconciliation belongs to F6 work, not this pass (`f6.1.md` is out of this task's edit scope):
  F6.1's tree should become `[Portal.Store, Portal.EventStore.adapter(), Portal.Engine, PortalWeb.Endpoint]` — a
  Bandit → `Endpoint` swap with `Portal.Store` retained until the F6.3/F6.4 read-model work collapses the dual-write.
  Recorded here as a forward reference so the F5.9 handoff invariant (INV5) and F6.1's tree are reconciled when F6
  resumes.
- **(carried)** the F5.4–F5.7 gaps that F5.8/F5.9 do **not** close: the latent same-millisecond id-collision footgun
  (production-safe via the single-writer engine; mitigated per-suite via `Store.reset/0`), the proposed
  `Portal.StoreCase` / `Portal.EngineCase` template, the two coexisting enroll paths, and the `:courses_of` vs
  `:enrollments` two-layer naming (unified at F6.4).

## F5.8 — build-verification retrospective (Apollo, 2026-06-03)

The F5.8 seams (the `Portal.EventStore` driven port + two adapters, the populated `Portal` facade, the
append-before-evolve inversion, the closed four-code `%Portal.Error{}` + no-catch-all `from/1`, the 4-child tree
swap pulled forward) were built by Mars and adversarially verified against the `f5.8.md` body + `f5.8.stories.md`
acceptance criteria + Venus's pinned contracts. **Verdict: PASS** — all eight gates green on the final reverted,
post-`D-CONTRACT-3-ADDENDUM` tree. Suite: echo_data 2; portal 6 doctests / 4 properties / 50 tests / 0 failures
(the 45+4 floor retained; +1 doctest = `Portal.Error.from/1`, +5 = `event_store_test.exs`). Determinism: 150/150
green on `PORT=4123`.

**Exceptional**
- The abort-branch guard is **non-vacuous, proven by mutation**. A scratch injection of the wrong order
  (evolve+project before the append result is checked, returning `new_state` on the `{:error}` branch — the as-built
  F5.6 order) turned `event_store_test.exs`'s abort test RED at exactly `assert :sys.get_state(Portal.Engine) ==
  fold_before` (fold gained a second course under the failing-stub adapter); the engine restored byte-identical, the
  test green again. A test that fails against the wrong implementation and passes against the right one is the only
  kind that ratifies F5.8-INV5 — the dual-write torn-write (a failed append leaving a phantom `%Enrollment{}` Store
  row) is structurally impossible because `project/2` is reachable only inside the `:ok ->` arm.
- The two-channel error contract (`D-CONTRACT-3-ADDENDUM`) is principled: the facade advertises only the **reachable**
  domain outcome (`{:error, %Portal.Error{}}`), the raw infra `{:error, term}` from the fallible port being
  unreachable today (`InMemory.append/2` is total) and first firing at F6.3; the engine's `command/1` correctly keeps
  `term()` for its reachable bare-atom outcomes.

**Good**
- Every public signature matched the spec body verbatim — `EventStore` callbacks, the `command/1`/`query/1`
  single-tagged-tuple rename, the `%Portal.Error{code, message, field}` struct and the closed four-code union — with
  **no invented arity/arg-order/return/struct** (the fan-out drift hazard did not recur).
- The master-invariant greps were empty (`EventStore.(InMemory|Postgres)` in `engine.ex`; `Portal.Engine|
  GenServer.call|Repo` under `portal_web/`); the router repointed onto the facade names nothing below the boundary.
- The config seam is genuinely config-only: per-env resolution verified (`prod → Postgres`, `test`/`dev →
  InMemory`), `MIX_ENV=prod mix compile` green with the dependency-free Postgres stub (no Ecto/postgrex), swapping
  the value changes no caller. The dual-write `Portal.Store` projection is retained byte-for-byte (`courses_of/1`
  rows; `enroll_slice_test` green).

**Not-well**
- The `:4000`/`mix run` environment hazard cost real time: `mix test` boots Bandit on `:4000`, and back-to-back runs
  collide on a socket in **TIME_WAIT** that `lsof -sTCP:LISTEN` cannot see, surfacing as a false `:eaddrinuse`
  whole-suite abort orthogonal to any logic. Two determinism-loop starts were lost to it (and to concurrent `mix`
  build-lock contention across agents) before moving to a dedicated `PORT=4123`.
- A stale-tree risk materialized: the first loop ran against a checkout that changed underneath it (the
  facade-spec addendum + the mutation revert), so it had to be discarded and re-run against the final tree — a loop on
  a stale checkout ratifies nothing.

**Opportunities**
- Adopt a **dedicated test port** (or a per-agent port) as the standing convention for this umbrella, so the
  determinism loop measures determinism, not port contention. Pairs with serializing `mix` across concurrent agents
  (a shared `_build` build-lock is a coordination point).
- The carried `Portal.StoreCase` / `Portal.EngineCase` case template (CLAUDE.md §4) would turn the per-suite reset
  convention into "cannot forget" — `event_store_test.exs` hand-rolls the same Store+adapter+Engine reset every
  other fold-touching suite does.

**Opened-gaps** (recorded, NOT F5.8 blockers)
- `event_log.ex` is now **dead code**: no longer started in the supervision tree (replaced by
  `EventStore.adapter()`), no longer called by the engine, reached by nothing in `lib/` (only comments reference it).
  It compiles cleanly. Leaving it is the conservative do-no-harm choice this rung; removal is a separate cleanup
  candidate (the F5.6 crash-recovery test comments still reference it as history).
- The facade spec (`enroll/2`, `deliver_lesson/2`) must **widen** to include the raw infra `{:error, term}` channel
  at F6.3, when the Postgres adapter makes that channel reachable — else dialyzer/callers under-handle it (Mars's
  learning L-5, paired with the existing F6.3 Postgres body/schema/migration carries).
- Pre-existing, unchanged here: `f6.1.md`'s supervision tree still omits the `Portal.Store` that `courses_of/1`
  needs (the F6 reconciliation already recorded above).

**Spec-refinements** (candidates for the next body re-sync)
- R-1 (`Portal.lesson/1`, the fifth facade fn) and R-2 (the `application.ex`/config swap pulled into F5.8) — both
  already tabled above; the build confirmed both land exactly as the Director ratified.
- New: the **two-channel error contract** (`D-CONTRACT-3-ADDENDUM`) — the body should state that the facade
  advertises only the reachable domain `%Portal.Error{}` outcome at F5.8 and widens to the raw infra `{:error,
  term}` at F6.3; today's narrowing is correct because `InMemory.append/2` is total.
