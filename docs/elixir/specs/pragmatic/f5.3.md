# F5.3 ¬∑ Tracer bullets: a walking skeleton

> Make one use case real, end to end. Wire **enroll** through every layer ‚Äî `POST /enroll` ‚Üí the `Portal.Engine`
> boundary ‚Üí `Portal.Learning.enroll/2` ‚Üí `Portal.Store` ‚Üí `201` ‚Äî so the thin server from F5.1 does real work and
> persists a real `%Enrollment{}`. Then grow the skeleton with a second thin slice, **deliver a lesson**. Every layer
> is touched; none is finished.

## Goal

After F5.3, a real `curl` enrolls a learner: `POST /enroll` round-trips through the web layer, the engine boundary, the
`Learning` context, and the store, and returns `201` with the new branded id ‚Äî and the enrollment is afterwards
retrievable. A second slice, `GET /lessons/:id`, reads a lesson through the same path and returns `200` (or `404`). The
`Portal.Engine` boundary stubbed in F5.1 is now real for these slices, delegating to the F5.2 contexts; the web layer
still holds no domain logic.

## Rationale (5W)

- **Why** ‚Äî A walking skeleton proves the architecture by running it: one thin vertical slice through every layer
  surfaces integration problems immediately, while leaving each layer free to deepen later. It is a tracer bullet ‚Äî fired
  end to end, then refined.
- **What** ‚Äî the enroll slice wired through web ‚Üí engine boundary ‚Üí context ‚Üí store, a second deliver-lesson slice, the
  result‚Üístatus mapping, and an end-to-end test with no mocks.
- **Who** ‚Äî Developers (the architecture proven runnable, a slice to extend), Operators (a real request that persists a
  real entity).
- **When** ‚Äî the third rung; depends on F5.1 (the web layer and engine boundary) and F5.2 (the domain and its public
  APIs). It is the slice that F5.4 then hardens with a contract.
- **Where** ‚Äî `lib/portal/engine.ex` (the boundary becomes real), `lib/portal_web/router.ex` (slice routes and status
  mapping). The contexts and store are unchanged.

## Scope

- **In** ‚Äî implementing `Portal.Engine.dispatch/1` for enroll and `query/2` for the lesson and courses reads, by
  delegating to the F5.2 contexts; the enroll web slice (`POST /enroll` ‚Üí 201/422) persisting a real `%Enrollment{}`;
  the deliver-lesson web slice (`GET /lessons/:id` ‚Üí 200/404); JSON encoding of the responses; minimal seed data so the
  slice round-trips locally; an end-to-end test with no mocks; the tracer-bullet discipline (thin but complete, grown
  one vertical slice at a time).
- **Out** ‚Äî input validation and the enroll contract, plus the closed `%Portal.Error{}` it introduces (‚Üí F5.4);
  recasting the slice as commands/queries/events (‚Üí F5.5); the supervised state home for a folded read model (‚Üí F5.6);
  the `EventStore` port and the `Portal` facade (‚Üí F5.8); any deep work inside a single layer (the point is breadth, not
  depth).

## Deliverables

- **F5.3-D1** ‚Äî `Portal.Engine.dispatch/1` handles `%{type: :enroll, user_id: u, course_id: c}` by delegating to
  `Portal.Learning.enroll(u, c)`, returning `{:ok, %Enrollment{}} | {:error, reason}` (no longer `:not_implemented`).
- **F5.3-D2** ‚Äî `Portal.Engine.query/2` handles `(:lesson, id)` ‚Üí `Portal.Catalog.lesson(id)` and `(:courses_of,
  user_id)` ‚Üí `Portal.Learning.courses_of(user_id)`, returning `{:ok, term} | :error | [..]` as the context defines.
- **F5.3-D3** ‚Äî the enroll slice: `POST /enroll` parses `user`/`course` params, calls the engine boundary, and maps
  `{:ok, e}` ‚Üí `201` (with `e.id`) and `{:error, _}` ‚Üí `422`; the enrollment is persisted by the context via
  `Portal.Store.put/1`.
- **F5.3-D4** ‚Äî the deliver-lesson slice: `GET /lessons/:id` calls the engine query and maps `{:ok, lesson}` ‚Üí `200`
  and `:error` ‚Üí `404`; the existing `GET /courses/:user_id` returns the learner's enrollments as `200`.
- **F5.3-D5** ‚Äî JSON encoding (via `Jason`) of the slice responses ‚Äî the created enrollment's id, a lesson, and a
  courses list ‚Äî in a consistent envelope.
- **F5.3-D6** ‚Äî minimal seed data (a `USR` user, a `CRS` course, an `LSN` lesson), minted with `Portal.ID.new/1`, loaded
  into the store so the slices round-trip in local verification, plus a documented way to seed. (Gotcha: the in-memory
  store is per-node, and `mix run priv/seeds.exs` runs in a separate node that exits ‚Äî for live `curl`, seed inside one
  running node: `iex -S mix` then `Code.eval_file("apps/portal/priv/seeds.exs")`.)
- **F5.3-D7** ‚Äî an end-to-end test, no mocks, exercising the real stack: `POST /enroll` ‚Üí `201` and the enrollment is
  retrievable; `GET /lessons/:id` ‚Üí `200`; an unknown lesson ‚Üí `404`.
- **F5.3-D8** ‚Äî the tracer-bullet discipline documented: every layer touched and none finished; the skeleton is the
  architecture, grown one thin vertical slice at a time.

## Invariants

- **F5.3-INV1 (master, held)** ‚Äî the web layer still calls only the `Portal.Engine` boundary (`dispatch`/`query`); the
  contexts and store are reached only through it, and the router holds no domain logic.
- **F5.3-INV2 (end-to-end realness)** ‚Äî the enroll slice touches every layer with no mocks or stubs; a request persists
  a real `%Enrollment{}` that is retrievable afterwards ‚Äî the skeleton genuinely walks.
- **F5.3-INV3 (railway result mapping)** ‚Äî every boundary call returns a tagged result and the web maps it
  deterministically: `{:ok, _}` ‚Üí `2xx`, an expected `{:error, _}`/`:error` ‚Üí `4xx`; no expected failure becomes a
  `500`.
- **F5.3-INV4 (thin slices, grown vertically)** ‚Äî the skeleton grows one vertical slice at a time (enroll, then
  deliver-lesson); each slice is complete top to bottom yet leaves every layer deliberately unfinished.
- **F5.3-INV5 (identity)** ‚Äî a created enrollment carries a freshly minted `ENR` branded id and is stored under it.

## Definition of Done

- [ ] `POST /enroll` against the running app returns `201` with the new id, and the enrollment is retrievable from the
      store (F5.3-INV2).
- [ ] `GET /lessons/:id` returns `200` with the lesson; an unknown id returns `404`.
- [ ] `GET /courses/:user_id` returns the learner's enrollments, including a newly created one.
- [ ] The web layer calls only the engine boundary; the engine delegates to the F5.2 contexts (F5.3-INV1).
- [ ] Expected failures map to `4xx`, successes to `2xx`; no expected failure is a `500` (F5.3-INV3).
- [ ] An end-to-end test exercises the real stack with no mocks (F5.3-D7).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.3.stories.md`).
