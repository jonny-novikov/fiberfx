# F5.3 · user stories

> Who needs the walking skeleton, what they need, and how we will know it works. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f5.3.md`](f5.3.md).

## F5.3-US1 — Enroll round-trips end to end

As a **developer**, I want a single enroll request to travel through every layer and persist a real entity, so that the
architecture is proven by running, not by inspection.

Acceptance criteria
- Given the running app, when I `POST /enroll` with a valid `user` and `course`, then I receive `201` with the new
  enrollment id.
- Given that response, when I read the enrollment back, then it exists in the store with `progress: 0`.
- Given the path, when the request is handled, then it passes web → engine boundary → context → store with no mocks.

INVEST — valuable (the architecture proven); testable via one request plus a store read.
Priority: must · Size: 3 · Implements deliverables: F5.3-D1, F5.3-D3.

## F5.3-US2 — The architecture is proven runnable

As an **operator**, I want a real `curl` to work against the supervised app, so that there is a demonstrable,
deployable slice from the first iteration.

Acceptance criteria
- Given the F5.1 supervised server, when I `curl -X POST .../enroll`, then it returns `201` and the system stays up.
- Given the engine boundary, when it handles the request, then it delegates to the `Learning` context (no longer
  `:not_implemented`).

INVEST — independent; testable with a live `curl`; demonstrates the boundary is real.
Priority: must · Size: 2 · Implements deliverables: F5.3-D1, F5.3-D6.

## F5.3-US3 — Deliver a lesson (a second slice)

As a **developer**, I want a second thin slice that reads a lesson, so that the skeleton grows by another vertical cut
rather than by deepening one layer.

Acceptance criteria
- Given a known lesson, when I `GET /lessons/:id`, then I receive `200` with the lesson.
- Given an unknown lesson id, when I `GET /lessons/:id`, then I receive `404`.
- Given the slice, when it is handled, then it reuses the engine query → context → store path.

INVEST — independent of slice 1; testable via two requests; encodes vertical growth.
Priority: must · Size: 2 · Implements deliverables: F5.3-D2, F5.3-D4.

## F5.3-US4 — Result-to-status mapping

As a **developer**, I want each boundary result mapped to a deterministic HTTP status, so that callers get the right
status and bugs never hide behind a `200` or surface as a `500`.

Acceptance criteria
- Given `{:ok, _}`, when the web maps it, then the response is a `2xx` (`201` for enroll, `200` for reads).
- Given an expected `{:error, _}`/`:error`, when the web maps it, then the response is a `4xx` (`422` / `404`).
- Given an expected failure, when it is handled, then it never becomes a `500`.

INVEST — independent; small; testable by request/response pairs; encodes F5.3-INV3.
Priority: must · Size: 2 · Implements deliverables: F5.3-D3, F5.3-D4, F5.3-D5.

## F5.3-US5 — Grow the skeleton vertically

As a **developer**, I want to add capability one thin vertical slice at a time, so that each step is shippable and the
architecture stays whole.

Acceptance criteria
- Given a new capability, when I add it, then it is a vertical cut through every layer, not a finished single layer.
- Given slice 2, when it is added, then no layer had to be completed first.
- Given the approach, when I read it, then the tracer-bullet discipline is documented.

INVEST — small; testable by inspecting the slices and the note; frames the method.
Priority: should · Size: 1 · Implements deliverables: F5.3-D7, F5.3-D8.
