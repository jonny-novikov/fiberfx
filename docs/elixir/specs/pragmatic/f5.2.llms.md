# F5.2 — Model the Portal domain · agent brief (llms)

- **Upstream.** `Portal.ID.new/1` (F5.1, mints branded ids) and the F4 store `Portal.Store` (`get/2`, `all/2`,

- Spec system, the master invariant, and the control-flow matrix: [`../specs.approach.md`](../specs.approach.md).

## Requirements

- **F5.2-R1** — seven entity structs (`User`, `Session`, `Course`, `Lesson`, `Page`, `Enrollment`, `Progress`), each
  with `@enforce_keys`, `defstruct`, and a `@type t`; `id`/`*_id` fields are branded-id strings; `@enforce_keys` lists
  every field except `progress` (default `0`, type `0..100`). [US: F5.2-US1]
- **F5.2-R2** — three context modules (`Portal.Accounts`, `Portal.Catalog`, `Portal.Learning`), each owning its
  entities and referencing other contexts only by branded id; no context aliases or constructs another's struct.
  [US: F5.2-US2]
- **F5.2-R3** — the documented public API per context, each function with an `@spec`, all other functions private:
  `Accounts.user/1 :: {:ok, %User{}} | :error`; `Catalog.course/1`, `Catalog.lesson/1` likewise; `Learning.enroll/2 ::
  {:ok, %Enrollment{}} | {:error, atom}`; `Learning.courses_of/1 :: [%Enrollment{}]`. [US: F5.2-US3]
- **F5.2-R4** — reads delegate to `Portal.Store` (`get/2`, `all/2`); `Learning.enroll/2` mints an `ENR` id via
  `Portal.ID.new("ENR")`, builds `%Enrollment{progress: 0}`, stores it with `Portal.Store.put/1`, and returns `{:ok,
  enrollment}`. [US: F5.2-US4]
- **F5.2-R5** — every entity has a `@type t` and every public function an `@spec` matching the structs; `id`/`*_id`
  are typed as branded-id strings. [US: F5.2-US5]
- **F5.2-R6** — verification: building a struct with all enforced keys succeeds and omitting one raises;
  `Learning.enroll("USR1", "CRS1")` returns `{:ok, %Enrollment{progress: 0}}` and the enrollment is retrievable; no
  context references another context's struct. [US: F5.2-US1, F5.2-US2, F5.2-US4]

## Execution topology

**Runtime topology (no new processes; modules over the F4 store and F5.1 id generator).**

```text
Portal.Application (unchanged from F5.1: Portal.Engine stub + Bandit)

Domain (new in F5.2 — plain modules, no processes):
  Portal.Accounts ─owns→ User, Session
  Portal.Catalog  ─owns→ Course, Lesson, Page
  Portal.Learning ─owns→ Enrollment, Progress

READ:  Context.<entity>(id) ─▶ Portal.Store.get(kind, id)        ─▶ {:ok, struct} | :error
WRITE: Learning.enroll(user_id, course_id)
         ─▶ Portal.ID.new("ENR") ─▶ %Enrollment{progress: 0} ─▶ Portal.Store.put/1 ─▶ {:ok, enrollment}

Cross-context references travel by branded id only — never by embedding another context's struct.
```

**Task topology (build order; each step leaves the app compiling).**

```text
T1 entity structs (@enforce_keys, defstruct, @type t) under each context dir   [R1]
     └─▶ T2 context modules owning their entities (alias own only)             [R2]
           └─▶ T3 public APIs with @spec; helpers private                       [R3]
                 └─▶ T4 reads via Portal.Store; enroll mints ENR + put          [R4]
                       └─▶ T5 typespecs reviewed against structs                 [R5]
                             └─▶ T6 verify (enforce-keys raise, enroll round-trip, no cross-struct)  [R6]
```

**Touched files.** `lib/portal/accounts/{user,session}.ex`, `lib/portal/catalog/{course,lesson,page}.ex`,
`lib/portal/learning/{enrollment,progress}.ex`, `lib/portal/accounts.ex`, `lib/portal/catalog.ex`,
`lib/portal/learning.ex`.

## Agent stories

Each agent story implements one user story and closes on a gate. Build in task-topology order.

### F5.2-AS1 — Entities that cannot be built incomplete [implements F5.2-US1, F5.2-US5]
Directive: create the seven entity structs with `@enforce_keys`, `defstruct`, and a `@type t`; `id`/`*_id` typed as
branded-id strings; `@enforce_keys` excludes only `progress` (T1, T5).
Acceptance gate: all enforced keys → builds; omit one → raises; `Enrollment` defaults `progress` to `0`; each entity
has a matching `@type t`.

### F5.2-AS2 — Bounded contexts [implements F5.2-US2]
Directive: create `Portal.Accounts`, `Portal.Catalog`, `Portal.Learning`, each owning and aliasing only its own
entities, referencing others by branded id (T2).
Acceptance gate: the modules compile; no context aliases or constructs another's struct; cross-context references use
ids.

### F5.2-AS3 — Public APIs over the store [implements F5.2-US3, F5.2-US4]
Directive: add the documented public functions with `@spec` (helpers private); reads delegate to `Portal.Store`;
`Learning.enroll/2` mints `ENR`, builds `%Enrollment{progress: 0}`, and stores it (T3, T4).
Acceptance gate: each context exposes exactly its API; `Learning.enroll("USR1", "CRS1")` → `{:ok, %Enrollment{progress:
0}}`, retrievable, and listed by `courses_of/1`.

## Execution plan — first two stories

*story → agent story → tasks → files → command → gate.*

### Plan A — F5.2-US1 (entities with an enforced shape) · via F5.2-AS1
1. Tasks T1 → T5.
2. Files: `lib/portal/accounts/{user,session}.ex`, `lib/portal/catalog/{course,lesson,page}.ex`,
   `lib/portal/learning/{enrollment,progress}.ex`.
3. Command: `mix test test/portal/entities_test.exs` (construct each struct; assert a missing key raises).
4. Gate (closes US1): all enforced keys build; omitting one raises; `progress` defaults to `0`; each entity has a
   `@type t`. Requirements: F5.2-R1, F5.2-R5.

### Plan B — F5.2-US4 (enroll a learner) · via F5.2-AS3
1. Tasks T2 → T3 → T4 (after AS1).
2. Files: `lib/portal/learning.ex` (and `accounts.ex`, `catalog.ex` for the read APIs).
3. Command: `iex -S mix` then `Portal.Learning.enroll("USR1", "CRS1")`, then `Portal.Learning.courses_of("USR1")`.
4. Gate (closes US4): returns `{:ok, %Enrollment{progress: 0}}` with a minted `ENR` id; retrievable from the store;
   listed by `courses_of/1`. Requirements: F5.2-R3, F5.2-R4.

Efficiency check: the entities are seven small structs behind one test; the first write is one context function over
`Portal.Store` + `Portal.ID` — the field lists, the id namespaces, and the return shapes are all fixed by the spec.

## Comprehensive implementation prompt

> Paste into a coding agent (e.g. Claude Code) at the root of the `portal` app, after F5.1. It executes F5.2-AS1..AS3
> in task-topology order and ends on the verification gates. Treat the Requirements and the Definition of Done in
> [`f5.2.md`](f5.2.md) as the acceptance checklist.

```text
You are implementing spec F5.2 (Model the Portal domain) in the `portal` app, on top of F5.1. Read
specs/pragmatic/f5.2.md, specs/pragmatic/f5.2.stories.md, and build-guide/f5-02-domain.md first. Model shape only — no
input validation yet (that is F5.4). Make illegal states unrepresentable via @enforce_keys; keep bounded contexts as
real ownership boundaries; reference other contexts only by branded id. Control flow: tagged tuples / :error returns;
no exceptions for expected outcomes.

Build in this order, keeping the app compiling after each step:

1) Entity structs (T1 → R1, R5). Under lib/portal/<context>/<entity>.ex create modules with @enforce_keys, defstruct,
   and @type t: Accounts.User (USR: id, email, name); Accounts.Session (SES: id, user_id, token); Catalog.Course
   (CRS: id, title, slug); Catalog.Lesson (LSN: id, course_id, title); Catalog.Page (PGE: id, lesson_id, body);
   Learning.Enrollment (ENR: id, user_id, course_id, progress default 0, type 0..100); Learning.Progress (PRG: id,
   enrollment_id, lesson_id, percent). All id/*_id fields are branded-id strings (String.t()); @enforce_keys lists
   every field except progress.

2) Context modules (T2 → R2). Create Portal.Accounts, Portal.Catalog, Portal.Learning at lib/portal/<context>.ex. Each
   context owns and aliases only its own entities and is the only module that builds or persists them. Reference other
   contexts' entities by branded id only — never embed or alias another context's struct.

3) Public APIs (T3 → R3). Add, each with @spec, everything else defp: Accounts.user(user_id) :: {:ok, %User{}} |
   :error; Catalog.course(course_id) and Catalog.lesson(lesson_id) likewise; Learning.enroll(user_id, course_id) ::
   {:ok, %Enrollment{}} | {:error, atom}; Learning.courses_of(user_id) :: [%Enrollment{}].

4) Store delegation + enroll (T4 → R4). Reads call Portal.Store.get/2 and Portal.Store.all/2. Learning.enroll/2 mints
   an ENR id with Portal.ID.new("ENR"), builds %Enrollment{progress: 0}, stores it with Portal.Store.put/1, and
   returns {:ok, enrollment}. Do not add validation.

5) Typespecs (T5 → R5). Confirm every entity has a @type t and every public function an @spec matching the structs;
   id/*_id are branded-id strings.

6) Verify (T6 → R6 + DoD). Confirm: building a struct with all enforced keys succeeds and omitting one raises; from
   iex, Learning.enroll("USR1", "CRS1") returns {:ok, %Enrollment{progress: 0}}, the enrollment is retrievable, and
   courses_of/1 lists it; no context references another context's struct. Report each result against the F5.2
   Definition of Done.

Deliverables: the touched files listed in specs/pragmatic/f5.2.llms.md, all compiling, with the domain modeled and the
first write working, and every F5.2 invariant exercised. Stop when the Definition of Done in specs/pragmatic/f5.2.md
is fully checked.
```
