# F5.7 · user stories

> Who needs the test suite, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f5.7.md`](f5.7.md).

## F5.7-US1 — Fast feedback on every change

As a **developer**, I want the suite to run in milliseconds, so that I run it constantly while I work.

Acceptance criteria
- Given the suite, when I run `mix test`, then the pure layers complete without starting a process.
- Given the tests, when they run, then they are `async: true` wherever there is no shared process state.
- Given a change to the core, when I save and run, then feedback is near-instant.

INVEST — independent; testable by the suite's shape and runtime; encodes F5.7-INV1, INV2.
Priority: must · Size: 2 · Implements deliverables: F5.7-D6.

## F5.7-US2 — The pure core is pinned by examples

As a **developer**, I want `decide`, `evolve`, and `replay` covered by plain example tests, so that the core's
behaviour is nailed down without processes or mocks.

Acceptance criteria
- Given a valid command, when `decide` runs in a test, then it emits the expected events.
- Given an event, when `evolve` runs, then it folds it into the expected state.
- Given a log, when `replay` runs, then it derives the expected state — all with no process and no mocks.

INVEST — independent; small; testable directly; encodes F5.7-INV1.
Priority: must · Size: 2 · Implements deliverables: F5.7-D1.

## F5.7-US3 — Invariants checked over many inputs

As a **developer**, I want invariants checked across generated inputs, so that they hold beyond the examples I thought
of.

Acceptance criteria
- Given generated logs, when replayed, then `replay(log)` equals incremental folding.
- Given any produced enrollment, when checked, then `0 ≤ progress ≤ 100`.
- Given a forward event, when applied, then a monotonic value (streak/progress) never decreases.

INVEST — independent; testable as a property suite; encodes F5.7-INV3.
Priority: must · Size: 3 · Implements deliverables: F5.7-D2.

## F5.7-US4 — The contract is tested as a contract

As a **developer**, I want the F5.4 preconditions and postcondition turned into assertions, so that the engine's rules
are verified, not assumed.

Acceptance criteria
- Given a malformed or nonexistent course, when enroll is attempted, then a test asserts `:course_not_found`.
- Given a duplicate, when enroll is attempted, then a test asserts `:already_enrolled`.
- Given a valid enroll, when it runs, then a test asserts the postcondition (a stored `%Enrollment{progress: 0}` /
  `%LearnerEnrolled{}`).

INVEST — independent; testable per reason; encodes F5.7-INV4.
Priority: must · Size: 2 · Implements deliverables: F5.7-D3.

## F5.7-US5 — Documentation that cannot lie

As a **developer**, I want public examples run as doctests, so that the documentation stays true to the code.

Acceptance criteria
- Given a public function with an `@doc` example, when the suite runs, then the example is executed.
- Given a wrong example, when the suite runs, then it fails.

INVEST — independent; small; testable by the doctest run; encodes F5.7-INV5.
Priority: should · Size: 1 · Implements deliverables: F5.7-D4.

## F5.7-US6 — One end-to-end smoke test

As an **operator**, I want a single command-then-query through the running engine, so that the wiring is proven once
without a slow, process-heavy suite.

Acceptance criteria
- Given the live `Portal.Engine`, when a command runs then a query runs, then the query reflects the command.
- Given the suite, when I count process tests, then there is exactly one.

INVEST — independent; testable through the GenServer; encodes F5.7-INV1 (the tip).
Priority: must · Size: 2 · Implements deliverables: F5.7-D5.
