# F5.2 · Model the Portal domain

> Give the Portal a shape before any behaviour. Model the domain as plain structs grouped into the **Accounts**,
> **Catalog**, and **Learning** bounded contexts, each with a small public API and keyed by branded Snowflake ids —
> entities whose enforced keys make incomplete states impossible to build, contexts that reference each other only by
> id, and a public surface the engine will call.

## Goal

After F5.2, the Portal's domain is modeled and callable. Seven entity structs (`User`, `Session`, `Course`, `Lesson`,
`Page`, `Enrollment`, `Progress`) carry `@enforce_keys` and a `@type t`; three context modules own them and expose a
small public API over the F4 `Portal.Store`; `Learning.enroll/2` mints an `ENR` id and stores a fresh enrollment.
Contexts reference one another only by branded id, and the web layer is unchanged.

## Rationale (5W)

- **Why** — Before behaviour, the domain needs a clear, explicit shape. Plain structs make the model legible, let the
  compiler enforce required fields, and bounded contexts keep coupling at the boundary instead of spreading through
  the system.
- **What** — seven entity structs, three context modules owning them, and a small typed public API over the F4 store.
- **Who** — Developers (a legible model and APIs to call), Architects (bounded contexts and clear ownership), Learners
  (the courses and enrollments that will hold their learning, modeled here).
- **When** — the second rung; depends on F5.1 (`Portal.ID`) and F4 (`Portal.Store`). It is the model that F5.3 wires
  end to end and F5.4 then hardens with a contract.
- **Where** — `lib/portal/<context>/<entity>.ex` (structs) and `lib/portal/<context>.ex` (contexts). No web changes;
  no validation yet — that is F5.4.

## Scope

- **In** — the entity structs (`@enforce_keys`, `defstruct`, `@type t`); the three context modules owning their
  entities; the public APIs (`Accounts.user/1`, `Catalog.course/1`, `Catalog.lesson/1`, `Learning.enroll/2`,
  `Learning.courses_of/1`); reads delegating to `Portal.Store`; `Learning.enroll/2` minting an `ENR` id, building
  `%Enrollment{progress: 0}`, and storing it; `@spec` on every public function.
- **Out** — input validation and the design-by-contract guard, plus the closed `%Portal.Error{}` contract it introduces
  (→ F5.4); commands/queries/events (→ F5.5); the supervised state home (→ F5.6); the `EventStore` port and the `Portal`
  facade (→ F5.8); wiring `enroll` through the web layer end to end (→ F5.3).

## Deliverables

- **F5.2-D1** — seven entity struct modules, each with `@enforce_keys`, `defstruct`, and a `@type t`:
  `Portal.Accounts.User` (USR: `id`, `email`, `name`), `Portal.Accounts.Session` (SES: `id`, `user_id`, `token`),
  `Portal.Catalog.Course` (CRS: `id`, `title`, `slug`), `Portal.Catalog.Lesson` (LSN: `id`, `course_id`, `title`),
  `Portal.Catalog.Page` (PGE: `id`, `lesson_id`, `body`), `Portal.Learning.Enrollment` (ENR: `id`, `user_id`,
  `course_id`, `progress` default `0`, type `0..100`), `Portal.Learning.Progress` (PRG: `id`, `enrollment_id`,
  `lesson_id`, `percent`). All `id`/`*_id` fields are branded-id strings; `@enforce_keys` lists every field except
  `progress`; each struct also carries `@derive {Jason.Encoder, only: [...]}` so the F5.3 web layer can encode it
  directly (a serialization concern, not domain logic).
- **F5.2-D2** — three context modules (`Portal.Accounts`, `Portal.Catalog`, `Portal.Learning`), each the only module
  that builds or persists its entities, referencing other contexts' entities **by branded id only** (never embedding
  or aliasing another context's struct).
- **F5.2-D3** — the public APIs, each with an `@spec`, everything else private (`defp`): `Accounts.user(user_id) ::
  {:ok, %User{}} | :error`; `Catalog.course/1` and `Catalog.lesson/1` likewise; `Learning.enroll(user_id, course_id)
  :: {:ok, %Enrollment{}} | {:error, atom}`; `Learning.courses_of(user_id) :: [%Enrollment{}]`.
- **F5.2-D4** — reads delegate to `Portal.Store` (`Portal.Store.get/2`, `Portal.Store.all/2`); `Learning.enroll/2` mints
  an `ENR` id with `Portal.ID.new("ENR")`, builds `%Enrollment{progress: 0}`, stores it with `Portal.Store.put/1`, and
  returns `{:ok, enrollment}`. **Build note (F5.2 stands this up):** the F4 branded CHAMP has no code yet and F5.2 is its
  first caller, so F5.2 stands up `Portal.Store` as a minimal in-memory GenServer — a namespace-partitioned map-of-maps
  behind the exact `get/2 · all/2 · put/1` surface, supervised as the first child of `Portal.Application` — and the real
  F4 CHAMP swaps in later behind the same surface with no caller change.
- **F5.2-D5** — typespecs: a `@type t` on each entity and an `@spec` on each public function; `id`/`*_id` fields typed
  as branded-id strings; the types match the struct definitions.
- **F5.2-D6** — verification: building a struct with all enforced keys succeeds and omitting one raises; from `iex`,
  `Learning.enroll(Portal.ID.new("USR"), Portal.ID.new("CRS"))` returns `{:ok, %Enrollment{progress: 0}}` and the
  enrollment is retrievable; no context references another context's struct. (F5.2 mints real 14-char branded ids; it
  does **not** yet reject a malformed one — a bare `"USR1"` is not a valid id, and the boundary check that rejects it is
  F5.4's `Portal.ID.valid?/1` precondition.)

## Invariants

- **F5.2-INV1 (make illegal states unrepresentable · seed)** — every entity is a struct with `@enforce_keys`, so a
  struct missing a required field cannot be constructed — it raises at build time rather than flowing on as a partial
  value.
- **F5.2-INV2 (bounded-context ownership)** — a context references another context's entities only by branded id; no
  context embeds, aliases, or constructs another context's struct.
- **F5.2-INV3 (narrow public surface)** — each context exposes exactly its documented public API; all other functions
  are private.
- **F5.2-INV4 (identity)** — ids are branded-id strings at the struct boundary (the transport form), the integer
  snowflake is canonical, and `enroll` mints via `Portal.ID.new("ENR")`.
- **F5.2-INV5 (typed model)** — every entity has a `@type t` and every public function an `@spec` that matches the
  structs, so the model is legible and Dialyzer-checkable. (Input validation is deferred to F5.4 — F5.2 models shape,
  not rules.)

## Definition of Done

- [ ] Every entity has a struct with `@enforce_keys`, `defstruct`, and a `@type t`.
- [ ] Building a struct with all enforced keys succeeds; omitting one raises (F5.2-INV1).
- [ ] Three context modules exist; none references another context's struct directly — only branded ids (F5.2-INV2).
- [ ] Each context exposes exactly its documented public API; helpers are private (F5.2-INV3).
- [ ] Reads delegate to `Portal.Store`; `Learning.enroll(Portal.ID.new("USR"), Portal.ID.new("CRS"))` returns `{:ok,
      %Enrollment{progress: 0}}` from `iex` and the enrollment is retrievable.
- [ ] Every public function has an `@spec` and every entity a `@type t` matching the struct (F5.2-INV5).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.2.stories.md`).
