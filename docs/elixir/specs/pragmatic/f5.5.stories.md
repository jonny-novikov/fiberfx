# F5.5 · user stories

> Who needs the command/query/event core, what they need, and how we will know it works. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f5.5.md`](f5.5.md).

## F5.5-US1 — Writes and reads are separate

As a **developer**, I want the write path and the read path kept apart, so that commands never leak data and queries
never cause a change.

Acceptance criteria
- Given a command, when it runs, then it returns `:ok` or `{:error, reason}` and never returns domain data.
- Given a query, when it runs, then it returns data and changes no state.
- Given the two paths, when I inspect them, then they are distinct functions with distinct return shapes.

INVEST — independent; testable by the return shapes of each path; encodes CQS.
Priority: must · Size: 2 · Implements deliverables: F5.5-D1, F5.5-D6.

## F5.5-US2 — Every change is recorded as a fact

As an **architect**, I want each change captured as a past-tense event, so that the system has an auditable history of
what happened.

Acceptance criteria
- Given a successful enroll, when it is decided, then a `%LearnerEnrolled{user_id, course_id, at}` is produced.
- Given a delivered lesson, when it is decided, then a `%LessonDelivered{user_id, lesson_id, at}` is produced.
- Given an event, when I inspect it, then it is an immutable, past-tense struct with enforced keys.

INVEST — independent; testable by decoding what `decide` emits; encodes F5.5-INV3.
Priority: must · Size: 2 · Implements deliverables: F5.5-D2, F5.5-D3.

## F5.5-US3 — The engine is two pure functions

As a **developer**, I want decisions and state changes expressed as pure functions, so that the core is deterministic
and exhaustively testable without processes or I/O.

Acceptance criteria
- Given `(state, command)`, when `decide/2` runs, then it returns `[event]` and performs no I/O; the closed `{:error,
  reason}` is returned earlier by the boundary's `authorize/2`, not by `decide`.
- Given `(event, state)`, when `evolve/2` runs, then it returns the next state and folds exactly one event.
- Given the same inputs, when either function runs again, then it returns the same output.

INVEST — independent; small; testable as plain functions; encodes F5.5-INV2.
Priority: must · Size: 3 · Implements deliverables: F5.5-D3, F5.5-D4.

## F5.5-US4 — State is the fold of the log

As an **architect**, I want the current state to be reconstructible from the event log, so that state is never the
only copy of the truth.

Acceptance criteria
- Given a log of events, when `replay/1` runs, then it returns the same state as folding those events one at a time.
- Given `replay/1`, when I inspect it, then it is `reduce` over `initial_state/0` with `evolve/2`.
- Given any sequence of events, when replayed, then a `StreamData` property confirms the result matches incremental
  folding.

INVEST — independent; testable as a property; encodes F5.5-INV4.
Priority: must · Size: 2 · Implements deliverables: F5.5-D5, F5.5-D7.

## F5.5-US5 — The contract still holds in the core

As a **learner** (served by the developer), I want the enroll rules enforced before any event is recorded, so that the
history never contains an invalid change.

Acceptance criteria
- Given a duplicate enroll, when the boundary's `authorize/2` runs, then it returns `{:error, :already_enrolled}`,
  `decide/2` is not reached, and no event is emitted.
- Given an enroll the contract rejects, when `authorize/2` runs, then it returns a closed tagged reason before `decide`;
  `decide`, if reached, emits events only and never a reason.
- Given a valid enroll, when decided, then it emits exactly the expected event(s).

INVEST — valuable (history stays valid); testable via `decide` on valid and invalid commands; encodes F5.5-INV5.
Priority: must · Size: 2 · Implements deliverables: F5.5-D3, F5.5-D8.
