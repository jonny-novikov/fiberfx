# F5.4 · user stories

> Who needs the enroll contract, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f5.4.md`](f5.4.md).

## F5.4-US1 — A bad enroll is rejected at the door

As a **learner**, I want an enroll with a bad or nonexistent course to be refused cleanly, so that I get a clear error
and no broken record is created.

Acceptance criteria
- Given a malformed `course_id` (wrong namespace), when I enroll, then I receive `422 :course_not_found`.
- Given a well-formed but nonexistent course, when I enroll, then I receive `422 :course_not_found`.
- Given either rejection, when I inspect the store, then nothing was written.

INVEST — valuable (data stays clean); testable via a request plus a store check.
Priority: must · Size: 3 · Implements deliverables: F5.4-D1, F5.4-D6.

## F5.4-US2 — No double enrollment

As a **learner**, I want a second enroll in the same course to be refused, so that I am never enrolled twice.

Acceptance criteria
- Given I am already enrolled, when I enroll again, then I receive `422 :already_enrolled`.
- Given that rejection, when I inspect the store, then exactly one enrollment exists for the pair.

INVEST — independent; testable by enrolling twice and counting; encodes a precondition.
Priority: must · Size: 2 · Implements deliverables: F5.4-D1, F5.4-D5.

## F5.4-US3 — Fail fast, with no partial writes

As a **developer**, I want every precondition checked before any effect, so that a rejected enroll cannot leave half-
written state.

Acceptance criteria
- Given a request that will fail a precondition, when it is handled, then no `ENR` id is minted and nothing is stored.
- Given the checks, when they run, then they complete before any build or store step.

INVEST — independent; testable by asserting no store mutation on rejection; encodes F5.4-INV2.
Priority: must · Size: 2 · Implements deliverables: F5.4-D6.

## F5.4-US4 — Expected failures and bugs are handled differently

As a **developer**, I want expected failures returned as tagged tuples and impossible states left to crash, so that the
value channel stays clean and bugs are loud.

Acceptance criteria
- Given an expected failure, when it occurs, then it is `{:error, reason}` mapped to `422`, never a raise.
- Given an impossible state (`progress` outside `0..100`), when it would occur, then it crashes (a bug), never returns a
  tuple.
- Given the reasons, when I list them, then they form a small closed set.

INVEST — independent; testable by exercising both kinds; encodes F5.4-INV3, INV4.
Priority: must · Size: 2 · Implements deliverables: F5.4-D3, F5.4-D4, F5.4-D5.

## F5.4-US5 — The contract is proven

As a **developer**, I want the postcondition and invariant checked by property tests, so that the contract holds across
inputs, not only on the examples I thought of.

Acceptance criteria
- Given generated valid inputs, when enroll succeeds, then it always returns `{:ok, %Enrollment{progress: 0}}` and the
  enrollment is stored.
- Given any enrollment the system produces, when I check `progress`, then `0 ≤ progress ≤ 100`.
- Given each expected error, when exercised by an example test, then it returns the documented reason.

INVEST — independent; testable as a property suite; encodes F5.4-INV5, INV6.
Priority: should · Size: 3 · Implements deliverables: F5.4-D2, F5.4-D7.
