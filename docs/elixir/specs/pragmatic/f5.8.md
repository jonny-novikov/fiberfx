# F5.8 ┬Ę Boundaries & integration seams

> Draw the engine's edges as explicit ports. A driven `Portal.EventStore` behaviour the core calls to persist and load
> events, with the adapter chosen by config; interchangeable `InMemory` and `Postgres` adapters behind it; the driving
> `Portal` facade as the only surface the web calls; and a closed `Portal.Error` vocabulary, produced by a no-catch-all
> `from/1`, that the UI consumes exhaustively.

## Goal

After F5.8, the core depends only on ports, callers depend only on the facade, and failures cross the boundary as typed
errors. `Portal.EventStore` is a new behaviour with `append/2` and `read_stream/1`; `Portal.EventStore.InMemory` (an
Agent) and `Portal.EventStore.Postgres` (Ecto) satisfy it interchangeably, selected by `config :portal, :event_store`,
and swap in **behind the existing `Portal.EventLog` surface** (`all/0`/`append/1`) the engine folds today. The `Portal`
facade (`enroll/2`, `deliver_lesson/2`, `progress_of/1`, `courses_of/1`) is **populated** here ŌĆö `lib/portal.ex` exists
as a reserved moduledoc stub, and F5.8 fills it with these four functions ŌĆö and becomes the only module the web calls.
It delegates to the engine's `command/1` and `query/1` wrappers, the **only** callers of `GenServer.call`; these wrap
and rename the as-built `Portal.Engine.dispatch/1` and `Portal.Engine.query/2` to a single tagged-tuple argument each
(D-B), and the router is repointed from those internals to the facade. `Portal.Error` (the `%Portal.Error{code,
message}` struct introduced in F5.4, today a two-code union with `new/1` only) is **extended** here to its closed
four-code union and an optional `field`, and a new `from/1` maps internal reasons to it with no catch-all clause.

## Rationale (5W)

- **Why** ŌĆö Boundaries are where a system stays changeable: a driven port lets storage swap (in-memory in tests,
  Postgres in production) without touching the core, a driving facade keeps the web ignorant of the engine, and a
  closed error set with no catch-all means a new failure mode cannot leak to the UI untyped ŌĆö the compiler forces it to
  be named. The port also makes appends **fallible** where the in-memory log never was, which forces the command path
  to invert its order (append before evolve) so the fold never leads durable storage ŌĆö see F5.8-INV5. Without these
  seams, the web reaches into the engine and storage choices harden into the domain.
- **What** ŌĆö the new `EventStore` behaviour (swapped in behind the `Portal.EventLog` surface), its two adapters and
  config switch, the engine's use of the port (and the inverted command order), the populated `Portal` facade over the
  renamed `command/1`/`query/1` wrappers, the `Portal.Error` contract extended to its four-code union with `from/1`,
  and an exhaustive consumer sketch. The dual-write `Portal.Store` `%Enrollment{}` read model (F5.6) is **retained
  unchanged** so the F5.3/F5.4 web stays byte-for-byte.
- **Who** ŌĆö Architects (substitutable storage and a stable contract) and developers (one surface to call, one error
  set to handle).
- **When** ŌĆö the eighth rung; it formalizes the seams the F5.7 tests assert against and assembles into the running
  system in F5.9. It is the F5ŌåÆF6 contract: F6 calls this facade and these errors unchanged.
- **Where** ŌĆö `lib/portal/event_store.ex` (the new behaviour), `lib/portal/event_store/in_memory.ex`,
  `lib/portal/event_store/postgres.ex` (all new), `lib/portal.ex` (the reserved stub, populated here),
  `lib/portal/engine.ex` (the renamed wrappers + inverted command order), and `lib/portal/error.ex` (extended).

## Scope

- **In** ŌĆö the driven `Portal.EventStore` behaviour (`append/2`, `read_stream/1`) generalizing the `Portal.EventLog`
  surface (`append/1`/`all/0`) it replaces; the `InMemory` and `Postgres` adapters and the config switch; the engine
  reading and appending through the port, inverting its command order to append-before-evolve; renaming the boundary's
  `dispatch/1`/`query/2` to the facade's `command/1`/`query/1` wrappers and repointing the router; the driving `Portal`
  facade (populated into the stub) as the sole web surface; the closed `Portal.Error` struct (extended to four codes +
  `field`) and the no-catch-all `from/1`; an exhaustive consumer sketch of the finite outcome set.
- **Out** ŌĆö assembling the supervision tree and the LiveView itself (ŌåÆ F5.9); the concrete Ecto schema and migration
  behind the Postgres adapter (ŌåÆ F6.3); removing or unifying the retained dual-write `Portal.Store` projection (ŌåÆ F6.3/
  F6.4); a producer for the reserved `:lesson_locked`/`:invalid_progress` codes and any progress-advancing command (ŌåÆ
  F6); the HTTP endpoint (ŌåÆ F6); authentication and authorization; multi-stream or cross-aggregate concerns.

## Deliverables

- **F5.8-D1** ŌĆö `Portal.EventStore`, a behaviour (driven port): `@callback append(stream :: String.t(), events ::
  [struct]) :: :ok | {:error, term}` and `@callback read_stream(stream :: String.t()) :: {:ok, [struct]} | {:error,
  term}`.
- **F5.8-D2** ŌĆö two adapters behind the port: `Portal.EventStore.InMemory` (an Agent, for dev and tests) and
  `Portal.EventStore.Postgres` (Ecto, for production), each implementing the behaviour; `config :portal, :event_store`
  selects the module (`Postgres` in prod, `InMemory` in test), and `Portal.EventStore.adapter/0` resolves it.
- **F5.8-D3** ŌĆö the engine uses the port: `init/1` calls `read_stream/1` (generalizing the as-built `EventLog.all/0`)
  and folds with `replay/1`; the command handler runs `decide` ŌåÆ `append/2` (through the port) ŌåÆ `evolve`, appending
  the events **before** folding them in. This **inverts** the as-built F5.6 order (`decide` ŌåÆ `evolve` ŌåÆ
  `EventLog.append`, engine.ex:75ŌĆō77) ŌĆö see F5.8-INV5 for why the fallible port forces it. The renamed `command/1`
  wrapper (D4) is the GenServer client; the dual-write `Portal.Store` projection (`project/2`, `reproject_enrollments/1`)
  is retained unchanged.
- **F5.8-D4** ŌĆö `Portal`, the facade (driving port), **populated into the reserved `lib/portal.ex` stub**: `enroll/2`,
  `deliver_lesson/2`, `progress_of/1`, `courses_of/1` returning `:ok | {:ok, data} | {:error, %Portal.Error{}}`; it
  runs the F5.4 contract guard, then delegates to the engine's `command/1` and `query/1` wrappers ŌĆö the renamed,
  single-tagged-tuple successors of `dispatch/1`/`query/2` and the only callers of `GenServer.call`. `courses_of/1`
  reads the `Portal.Store` `%Enrollment{}` rows (the web read shape) while the core's `query({:enrollments, user_id})`
  returns bare `[course_id]` (the fold read) ŌĆö a deliberate two-layer naming, unification deferred to F6.4.
  `progress_of/1` reads `enrollment.progress`, which is **structurally 0 by two independent mechanisms**: no
  progress-advancing event exists yet (the core folds set membership only ŌĆö `LessonDelivered` updates a `delivered`
  set, not `progress`, core.ex:64ŌĆō68), **and** the `Portal.Store` projection mints every `%Enrollment{}` with
  `progress: 0` (enroll-time, engine.ex:130; reproject, engine.ex:171). So this is an honest read of 0, not a
  calculation ŌĆö a progress-advancing command (and the second mechanism's removal) is an F6 concern.
- **F5.8-D5** ŌĆö `Portal.Error` **extended to its final shape**: the `%Portal.Error{code, message}` introduced in F5.4
  (today a two-code union, error.ex:11) gains an optional `field`, and its `code` union is completed to `@type code ::
  :already_enrolled | :course_not_found | :lesson_locked | :invalid_progress`. The union is **final at four codes, but
  only two have producers today** ŌĆö `:course_not_found` and `:already_enrolled` are produced by `Core.authorize/2`
  (core.ex:114); `:lesson_locked` and `:invalid_progress` are **reserved** (no deliver-lesson gate, no progress
  validation is built ŌĆö do not invent producers for them). `from/1` maps **each** of the four internal reasons to a
  `%Portal.Error{}` with **no catch-all clause**, so adding a producer later needs no `from/1` change, and an unmapped
  reason fails to match and surfaces as a bug rather than leaking.
- **F5.8-D6** ŌĆö the master invariant realized at the seams: the web calls only `Portal`; the core (`decide`/`evolve`)
  names only the `EventStore` behaviour; failures cross the boundary as `%Portal.Error{}`.
- **F5.8-D7** ŌĆö an exhaustive consumer sketch: the finite set of outcomes a caller handles (`{:ok, data}` and one
  branch per `Portal.Error` code), showing the closed set is consumed exhaustively (the F6 LiveView will adopt it).
- **F5.8-D8** ŌĆö verification: swapping the `:event_store` config between adapters changes no caller; `Portal` is the
  only web surface and the only path to `GenServer.call`; `from/1` has no catch-all; an enroll persists through the
  port and survives an **engine-process** restart by replay (the supervised adapter process outlives the engine and its
  `init/1` re-reads it ŌĆö full-VM durability is the F6.3 Postgres concern).

## Invariants

- **F5.8-INV1 (driven port)** ŌĆö the core reaches the event store only through the `Portal.EventStore` behaviour; no core
  module names a concrete adapter, and the adapter is config-selected.
- **F5.8-INV2 (driving port ŌĆö the master invariant)** ŌĆö the web calls only the `Portal` facade; only the engine's
  `command/1` and `query/1` wrappers call `GenServer.call`; nothing above the boundary names the engine, a repo, or the
  store.
- **F5.8-INV3 (closed error set, exhaustive)** ŌĆö failures cross as `%Portal.Error{}` whose `code` is the closed
  four-code union; `from/1` maps all four with no catch-all, so a new internal reason must be mapped explicitly and
  cannot silently leak. The union is final even though only `:course_not_found`/`:already_enrolled` have producers
  today (`:lesson_locked`/`:invalid_progress` are reserved), so adding a producer later changes no signature.
- **F5.8-INV4 (adapter substitutability)** ŌĆö `InMemory` and `Postgres` satisfy the same behaviour; selecting one is
  config-only and changes no caller (parse-don't-validate at the storage seam).
- **F5.8-INV5 (append before evolve ŌĆö an inversion forced by the fallible port)** ŌĆö the command handler appends events
  through the port **before** folding them into state. This **inverts** the as-built F5.6 order, which evolves then
  appends (engine.ex:75ŌĆō77): that order was safe only because `Portal.EventLog.append/1` is total and in-process
  (event_log.ex:35 returns `:ok` unconditionally), so it could never fail after the fold. `Portal.EventStore.append/2`
  is **fallible** (`{:error, term}`, especially for Postgres), so a failed append must abort the command ŌĆö return the
  error and leave state unchanged (railway style) ŌĆö and the fold must never lead durable storage. Hence the order is
  inverted here, and durability and in-memory state agree only because the append is the gate the evolve waits on.
- **F5.8-INV6 (typed results at the seam)** ŌĆö the facade returns typed results and the UI consumes a finite outcome
  set; no raw reason or untyped error escapes the boundary.

## Definition of Done

- [ ] `Portal.EventStore` defines `append/2` and `read_stream/1` as `@callback`s (F5.8-D1).
- [ ] `InMemory` and `Postgres` implement the behaviour and are interchangeable by config (F5.8-INV4).
- [ ] The engine reads through `read_stream/1` and appends through `append/2` before evolving ŌĆö inverting the as-built
      evolve-then-append order, so a failed append aborts and leaves state unchanged (F5.8-INV5).
- [ ] `Portal` is the only web surface and the only path to `GenServer.call` (F5.8-INV2).
- [ ] `Portal.Error` has a closed `code` union and `from/1` has no catch-all (F5.8-INV3).
- [ ] A consumer sketch handles `{:ok, data}` and every error code exhaustively (F5.8-D7).
- [ ] Swapping the `:event_store` config changes no caller (F5.8-D8).
- [ ] Every deliverable maps to a user story and every invariant is checked (see `./f5.8.stories.md`).
