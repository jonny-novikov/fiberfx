# F5.8 ┬╖ Boundaries & integration seams

> Draw the engine's edges as explicit ports. A driven `Portal.EventStore` behaviour the core calls to persist and load
> events, with the adapter chosen by config; interchangeable `InMemory` and `Postgres` adapters behind it; the driving
> `Portal` facade as the only surface the web calls; and a closed `Portal.Error` vocabulary, produced by a no-catch-all
> `from/1`, that the UI consumes exhaustively.

## Goal

After F5.8, the core depends only on ports, callers depend only on the facade, and failures cross the boundary as typed
errors. `Portal.EventStore` is a behaviour with `append/2` and `read_stream/1`; `Portal.EventStore.InMemory` (an Agent)
and `Portal.EventStore.Postgres` (Ecto) satisfy it interchangeably, selected by `config :portal, :event_store`. The
`Portal` facade (`enroll/2`, `deliver_lesson/2`, `progress_of/1`, `courses_of/1`) is the only module the web calls,
delegating to the engine's `command/1` and `query/1` wrappers тАФ the only callers of `GenServer.call`. `Portal.Error` is
a struct with a closed `code` union, and `from/1` maps internal reasons to it with no catch-all clause.

## Rationale (5W)

- **Why** тАФ Boundaries are where a system stays changeable: a driven port lets storage swap (in-memory in tests,
  Postgres in production) without touching the core, a driving facade keeps the web ignorant of the engine, and a
  closed error set with no catch-all means a new failure mode cannot leak to the UI untyped тАФ the compiler forces it to
  be named. Without these seams, the web reaches into the engine and storage choices harden into the domain.
- **What** тАФ the `EventStore` behaviour, its two adapters and config switch, the engine's use of the port, the `Portal`
  facade, the `Portal.Error` contract with `from/1`, and an exhaustive consumer sketch.
- **Who** тАФ Architects (substitutable storage and a stable contract) and developers (one surface to call, one error
  set to handle).
- **When** тАФ the eighth rung; it formalizes the seams the F5.7 tests assert against and assembles into the running
  system in F5.9. It is the F5тЖТF6 contract: F6 calls this facade and these errors unchanged.
- **Where** тАФ `lib/portal/event_store.ex` (the behaviour), `lib/portal/event_store/in_memory.ex`,
  `lib/portal/event_store/postgres.ex`, `lib/portal.ex` (the facade), and `lib/portal/error.ex`.

## Scope

- **In** тАФ the driven `Portal.EventStore` behaviour (`append/2`, `read_stream/1`); the `InMemory` and `Postgres`
  adapters and the config switch; the engine reading and appending through the port; the driving `Portal` facade as the
  sole web surface over `command/1`/`query/1`; the closed `Portal.Error` struct and the no-catch-all `from/1`; an
  exhaustive consumer sketch of the finite outcome set.
- **Out** тАФ assembling the supervision tree and the LiveView itself (тЖТ F5.9); the concrete Ecto schema and migration
  behind the Postgres adapter (тЖТ F6.3); the HTTP endpoint (тЖТ F6); authentication and authorization; multi-stream or
  cross-aggregate concerns.

## Deliverables

- **F5.8-D1** тАФ `Portal.EventStore`, a behaviour (driven port): `@callback append(stream :: String.t(), events ::
  [struct]) :: :ok | {:error, term}` and `@callback read_stream(stream :: String.t()) :: {:ok, [struct]} | {:error,
  term}`.
- **F5.8-D2** тАФ two adapters behind the port: `Portal.EventStore.InMemory` (an Agent, for dev and tests) and
  `Portal.EventStore.Postgres` (Ecto, for production), each implementing the behaviour; `config :portal, :event_store`
  selects the module (`Postgres` in prod, `InMemory` in test), and `Portal.EventStore.adapter/0` resolves it.
- **F5.8-D3** тАФ the engine uses the port: `init/1` calls `read_stream/1` and folds with `replay/1`; the command handler
  runs `decide` тЖТ `append/2` (through the port) тЖТ `evolve`, appending the events before folding them in.
- **F5.8-D4** тАФ `Portal`, the facade (driving port): `enroll/2`, `deliver_lesson/2`, `progress_of/1`, `courses_of/1`
  returning `:ok | {:ok, data} | {:error, %Portal.Error{}}`; it runs the F5.4 contract guard, then delegates to the
  engine's `command/1` and `query/1` wrappers тАФ the only callers of `GenServer.call`.
- **F5.8-D5** тАФ `Portal.Error`, a struct `%Portal.Error{code, message, field}` with `@type code :: :already_enrolled |
  :course_not_found | :lesson_locked | :invalid_progress`; `from/1` maps each internal reason to a `%Portal.Error{}`
  with **no catch-all clause**, so an unmapped reason fails to match and surfaces as a bug rather than leaking.
- **F5.8-D6** тАФ the master invariant realized at the seams: the web calls only `Portal`; the core (`decide`/`evolve`)
  names only the `EventStore` behaviour; failures cross the boundary as `%Portal.Error{}`.
- **F5.8-D7** тАФ an exhaustive consumer sketch: the finite set of outcomes a caller handles (`{:ok, data}` and one
  branch per `Portal.Error` code), showing the closed set is consumed exhaustively (the F6 LiveView will adopt it).
- **F5.8-D8** тАФ verification: swapping the `:event_store` config between adapters changes no caller; `Portal` is the
  only web surface and the only path to `GenServer.call`; `from/1` has no catch-all; an enroll persists through the
  port and survives an engine restart by replay.

## Invariants

- **F5.8-INV1 (driven port)** тАФ the core reaches the event store only through the `Portal.EventStore` behaviour; no core
  module names a concrete adapter, and the adapter is config-selected.
- **F5.8-INV2 (driving port тАФ the master invariant)** тАФ the web calls only the `Portal` facade; only the engine's
  `command/1` and `query/1` wrappers call `GenServer.call`; nothing above the boundary names the engine, a repo, or the
  store.
- **F5.8-INV3 (closed error set, exhaustive)** тАФ failures cross as `%Portal.Error{}` whose `code` is the closed union;
  `from/1` has no catch-all, so a new internal reason must be mapped explicitly and cannot silently leak.
- **F5.8-INV4 (adapter substitutability)** тАФ `InMemory` and `Postgres` satisfy the same behaviour; selecting one is
  config-only and changes no caller (parse-don't-validate at the storage seam).
- **F5.8-INV5 (append before evolve)** тАФ the command handler appends events through the port before folding them into
  state, so durability and in-memory state agree and a restart replays to the same state.
- **F5.8-INV6 (typed results at the seam)** тАФ the facade returns typed results and the UI consumes a finite outcome
  set; no raw reason or untyped error escapes the boundary.

## Definition of Done

- [ ] `Portal.EventStore` defines `append/2` and `read_stream/1` as `@callback`s (F5.8-D1).
- [ ] `InMemory` and `Postgres` implement the behaviour and are interchangeable by config (F5.8-INV4).
- [ ] The engine reads through `read_stream/1` and appends through `append/2` before evolving (F5.8-INV5).
- [ ] `Portal` is the only web surface and the only path to `GenServer.call` (F5.8-INV2).
- [ ] `Portal.Error` has a closed `code` union and `from/1` has no catch-all (F5.8-INV3).
- [ ] A consumer sketch handles `{:ok, data}` and every error code exhaustively (F5.8-D7).
- [ ] Swapping the `:event_store` config changes no caller (F5.8-D8).
- [ ] Every deliverable maps to a user story and every invariant is checked (see `./f5.8.stories.md`).
