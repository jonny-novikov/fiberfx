## References

- ExUnit — <https://hexdocs.pm/ex_unit/ExUnit.html>. The test framework; `async: true` for process-free tests.
- StreamData — <https://hexdocs.pm/stream_data/StreamData.html>. Property-based testing; `check all` over generators.
- ExUnit.DocTest — <https://hexdocs.pm/ex_unit/ExUnit.DocTest.html>. Runs `@doc` examples as tests.
- Gary Bernhardt, *Boundaries* — <https://www.destroyallsoftware.com/talks/boundaries>. Why the pure core gets a wide,
  fast, mock-free base and the process its single test.
- Upstream specs (the units under test): the contract [`f5.4.md`](f5.4.md), the pure core [`f5.5.md`](f5.5.md), the
  engine [`f5.6.md`](f5.6.md), and the pattern [`decider-pattern.md`](decider-pattern.md).

## Requirements

- **F5.7-R1** — the pure layers (`decide`/`evolve`/`replay`, contract) are tested with no process and no mocks; the
  process test is one smoke test at the tip. [US: F5.7-US1, F5.7-US2, F5.7-US6]
- **F5.7-R2** — example tests assert `decide` events, `evolve` folds, and `replay` derivation. [US: F5.7-US2]
- **F5.7-R3** — `StreamData` properties assert `replay == fold`, `0..100`, and set-monotonicity (a forward event never
  shrinks a user's enrolled/delivered MapSet). The numeric-progress monotonic is DEFERRED — no progress-advancing event
  exists yet (F5.8 / F6). [US: F5.7-US3]
- **F5.7-R4** — contract tests assert each F5.4 closed reason and the postcondition. [US: F5.7-US4]
- **F5.7-R5** — doctests run the public `@doc` examples; a wrong example fails the suite. [US: F5.7-US5]
- **F5.7-R6** — one command → query SMOKE test (designated as such) runs through `Portal.Engine` as the process tip;
  the F5.6 crash-recovery + behavioural process tests are retained (the engine's DoD), not deleted to hit a literal
  count. [US: F5.7-US6]
- **F5.7-R7** — the suite is deterministic (no real-clock sleeps, seeded generators) and fast. [US: F5.7-US1]

## Execution topology

```text
The pyramid (most tests at the base):

                 ┌───────────────┐
                 │   process     │  smoke tip: command → query (+ retained F5.6 crash/behavioural gates)
                 ├───────────────┤
                 │   doctest     │  public @doc examples (deterministic pure fns only)
                 ├───────────────┤
                 │   contract    │  F5.4 preconditions (each reason) + postcondition
                 ├───────────────┤
                 │   property    │  replay == fold · 0..100 · set-monotonicity
                 ├───────────────┤
                 │   example     │  decide / evolve / replay  (no process, no mocks)
                 └───────────────┘

What to test where:
  decide emits correct events ............. example
  evolve / replay derive state ............ example
  replay == fold, 0..100, set-monotonic ... property
  precondition → typed reason ............. contract
  postcondition → stored enrollment ....... contract
  public function example ................. doctest
  command visible to a later query ........ process (the smoke tip; F5.6 gates retained)
```

```text
Task DAG:
  T1 example(core) ─▶ T2 property(invariants) ─▶ T3 contract(F5.4) ─▶ T4 doctest ─▶ T5 process(smoke) ─▶ T6 verify
                                                                                     (T1..T4 async: true)
```

## Execution plan — first two stories

1. **F5.7-AS1 — example tests of the core.** The store-free core cases already exist; split them into a new
   `test/portal/engine/core_pure_test.exs`, `use ExUnit.Case, async: true`, holding the pure cases: `decide(state,
   {:enroll, u, c, at})` returns `[%LearnerEnrolled{}]`; `evolve(%LearnerEnrolled{}, state)` returns the enrolled state;
   `replay([e1, e2])` returns the folded state. No process, no mocks. Run `mix test
   test/portal/engine/core_pure_test.exs` — green and fast.
2. **F5.7-AS2 — properties of the invariants.** In the same pure file, `use ExUnitProperties`. Keep the `event_gen`
   generator. Add `property "replay equals fold"` (`max_runs: 200`) and `property "a forward event never shrinks the
   fold's sets"` (set-monotonicity); the `0..100` invariant rides with the contract suite. Run with a fixed seed;
   confirm all pass.

## Comprehensive implementation prompt

```text
You are extending the Portal umbrella with its test suite (rung F5.7). Read specs/pragmatic/f5.7.md and the units
under test: the pure core (Portal.Engine.Core: decide/2 -> [event], evolve/2, replay/1, initial_state/0), the F5.4
enroll contract (closed reasons :course_not_found, :already_enrolled), and the F5.6 engine (Portal.Engine GenServer).
The engine command path is decide -> append -> evolve; decide returns events only, and the F5.4 contract guards
rejection at the boundary with tagged tuples.

The suite ALREADY EXISTS (it was built incrementally across F5.4–F5.6). This rung AUDITS the as-built and ADDS the
missing pyramid layers — do not recreate existing files; move/add only. Build the pyramid weighted to the pure base:

1) Example tests (T1 -> R2). Split the store-free core cases into a NEW test/portal/engine/core_pure_test.exs, `use
   ExUnit.Case, async: true` (leaving core_test.exs for the Store-touching authorize describe, `async: false`,
   Store.reset). Assert: a valid command makes decide/2 emit the expected events; evolve/2 folds one event into the
   expected state; replay/1 derives the expected state from a log. No process is started and no mocks are used.

2) Property tests (T2 -> R3). In core_pure_test.exs add `use ExUnitProperties`. Write an event-log generator and
   assert: replay(log) == Enum.reduce(log, initial_state(), &evolve/2) (max_runs: 200); and set-monotonicity — a
   forward event never shrinks a user's enrolled/delivered MapSet. The numeric-progress monotonic is DEFERRED (no
   progress-advancing event yet); the 0..100 invariant rides with the contract suite. Use a fixed seed; no real-clock
   sleeps.

3) Contract tests (T3 -> R4). test/portal/enroll_contract_test.exs already exists; audit it — each F5.4 closed reason
   has a test (a nonexistent/malformed course -> :course_not_found; a duplicate -> :already_enrolled) and the
   postcondition is asserted (a valid enroll yields a stored %Enrollment{progress: 0} / a %LearnerEnrolled{}).

4) Doctests (T4 -> R5). Add runnable @doc examples to the public modules and `doctest` lines so the suite runs them;
   doctest ONLY deterministic pure functions (Portal.ID.valid?/1 + namespace/1; Portal.Error.new/1 — add its missing
   @doc; Portal.Engine.Core.initial_state/0 + query/2 on fixed ids), NEVER the clock/mint/Store-reading ones. A wrong
   example must fail.

5) Process smoke test (T5 -> R6). test/portal/engine_test.exs already drives Portal.Engine; DESIGNATE its
   command-then-query round-trip as the F5.7 smoke test (the process tip) and keep the F5.6 crash-recovery + behavioural
   process tests in place — do NOT delete F5.6 coverage to reach a one-test count.

6) Verify (T6 -> R1, R7). Run `mix test`. Confirm it is green and fast, the base holds the most tests, the process tier
   is a thin tip (the smoke test + the retained F5.6 gates), properties pass, and doctests pass. Keep the pure tiers
   `async: true`.

Constraints: no mocking frameworks (the pure core needs none); deterministic and fast (no Process.sleep, seeded
generators); the EventStore adapters and LiveView tests are out of scope (F5.8 / F6). Match Portal conventions:
branded Snowflake IDs, pure functions, tagged tuples for expected outcomes.
```
