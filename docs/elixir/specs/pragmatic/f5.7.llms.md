# F5.7 · agent brief (llms)

> Implementation brief for an agent building the test suite. References, traced requirements, the execution topology,
> and a paste-ready prompt. Pairs with the spec [`f5.7.md`](f5.7.md) and the stories [`f5.7.stories.md`](f5.7.stories.md).

## References

- ExUnit — <https://hexdocs.pm/ex_unit/ExUnit.html>. The test framework; `async: true` for process-free tests.
- StreamData — <https://hexdocs.pm/stream_data/StreamData.html>. Property-based testing; `check all` over generators.
- ExUnit.DocTest — <https://hexdocs.pm/ex_unit/ExUnit.DocTest.html>. Runs `@doc` examples as tests.
- Gary Bernhardt, *Boundaries* — <https://www.destroyallsoftware.com/talks/boundaries>. Why the pure core gets a wide,
  fast, mock-free base and the process its single test.
- Upstream specs (the units under test): the contract [`f5.4.md`](f5.4.md), the pure core [`f5.5.md`](f5.5.md), the
  engine [`f5.6.md`](f5.6.md), and the pattern [`decider-pattern.md`](decider-pattern.md).

## Requirements

- **F5.7-R1** — the pure layers (`decide`/`evolve`/`replay`, contract) are tested with no process and no mocks; the
  process test is one smoke test at the tip. [US: F5.7-US1, F5.7-US2, F5.7-US6]
- **F5.7-R2** — example tests assert `decide` events, `evolve` folds, and `replay` derivation. [US: F5.7-US2]
- **F5.7-R3** — `StreamData` properties assert `replay == fold`, `0..100`, and monotonic progress. [US: F5.7-US3]
- **F5.7-R4** — contract tests assert each F5.4 closed reason and the postcondition. [US: F5.7-US4]
- **F5.7-R5** — doctests run the public `@doc` examples; a wrong example fails the suite. [US: F5.7-US5]
- **F5.7-R6** — exactly one process test runs a command then a query through `Portal.Engine`. [US: F5.7-US6]
- **F5.7-R7** — the suite is deterministic (no real-clock sleeps, seeded generators) and fast. [US: F5.7-US1]

## Execution topology

```text
The pyramid (most tests at the base):

                 ┌───────────────┐
                 │   process     │  1 test: command → query through Portal.Engine
                 ├───────────────┤
                 │   doctest     │  public @doc examples
                 ├───────────────┤
                 │   contract    │  F5.4 preconditions (each reason) + postcondition
                 ├───────────────┤
                 │   property    │  replay == fold · 0..100 · monotonic
                 ├───────────────┤
                 │   example     │  decide / evolve / replay  (no process, no mocks)
                 └───────────────┘

What to test where:
  decide emits correct events ............. example
  evolve / replay derive state ............ example
  replay == fold, 0..100, monotonic ....... property
  precondition → typed reason ............. contract
  postcondition → stored enrollment ....... contract
  public function example ................. doctest
  command visible to a later query ........ process (the only process test)
```

```text
Task DAG:
  T1 example(core) ─▶ T2 property(invariants) ─▶ T3 contract(F5.4) ─▶ T4 doctest ─▶ T5 process(smoke) ─▶ T6 verify
                                                                                     (T1..T4 async: true)
```

Touched files: `test/portal/engine/core_test.exs` (example + property), `test/portal/enroll_contract_test.exs`
(contract), `test/portal/engine_test.exs` (the one process test), `test/test_helper.exs` (`ExUnit.start`), and the
public `@doc` strings in `lib/` (doctests, no new file).

## Agent stories

- **F5.7-AS1** [implements F5.7-US2, F5.7-US1] — Directive: write `core_test.exs` with `async: true`, no process and no
  mocks: cases for `decide` (valid command → expected events), `evolve` (one event → expected state), `replay` (log →
  expected state). Acceptance gate: all green, no process started, runs in milliseconds.
- **F5.7-AS2** [implements F5.7-US3] — Directive: add `StreamData` properties to `core_test.exs`: a generator for
  command sequences/event logs; assert `replay(log) == Enum.reduce(log, initial_state(), &evolve/2)`, `progress in
  0..100`, and that a forward event never decreases a monotonic value. Acceptance gate: properties pass with the
  default run count and a fixed seed.
- **F5.7-AS3** [implements F5.7-US4, F5.7-US5] — Directive: write `enroll_contract_test.exs` asserting `:course_not_found`,
  `:already_enrolled`, and the valid-enroll postcondition; add `doctest` lines for the public modules. Acceptance gate:
  each reason has a test, the postcondition is asserted, doctests run.
- **F5.7-AS4** [implements F5.7-US6] — Directive: write `engine_test.exs` with one test that starts `Portal.Engine`,
  issues a command, then a query, and asserts the query reflects the command. Acceptance gate: exactly one process test
  in the suite.

## Execution plan — first two stories

1. **F5.7-AS1 — example tests of the core.** Create `test/portal/engine/core_test.exs`, `use ExUnit.Case, async:
   true`. Add cases: `decide(state, {:enroll, u, c})` returns `[%LearnerEnrolled{}]`; `evolve(%LearnerEnrolled{},
   state)` returns the enrolled state; `replay([e1, e2])` returns the folded state. No process, no mocks. Run
   `mix test test/portal/engine/core_test.exs` — green and fast.
2. **F5.7-AS2 — properties of the invariants.** In the same file, `use ExUnitProperties`. Write a generator for a list
   of valid command/event sequences. Add `property "replay equals fold"`, `property "progress within 0..100"`, and
   `property "forward events are monotonic"`. Run with a fixed seed; confirm all pass.

## Comprehensive implementation prompt

```text
You are extending the Portal umbrella with its test suite (rung F5.7). Read specs/pragmatic/f5.7.md and the units
under test: the pure core (Portal.Engine.Core: decide/2 -> [event], evolve/2, replay/1, initial_state/0), the F5.4
enroll contract (closed reasons :course_not_found, :already_enrolled), and the F5.6 engine (Portal.Engine GenServer).
The engine command path is decide -> append -> evolve; decide returns events only, and the F5.4 contract guards
rejection at the boundary with tagged tuples.

Build the suite as a pyramid weighted to the pure base:

1) Example tests (T1 -> R2). Create test/portal/engine/core_test.exs, `use ExUnit.Case, async: true`. Assert: a valid
   command makes decide/2 emit the expected events; evolve/2 folds one event into the expected state; replay/1 derives
   the expected state from a log. No process is started and no mocks are used.

2) Property tests (T2 -> R3). In core_test.exs add `use ExUnitProperties`. Write a generator for valid command/event
   sequences and assert: replay(log) == Enum.reduce(log, initial_state(), &evolve/2); every produced progress is in
   0..100; a forward event never decreases a monotonic value. Use a fixed seed; no real-clock sleeps.

3) Contract tests (T3 -> R4). Create test/portal/enroll_contract_test.exs. Assert each F5.4 closed reason has a test
   (a nonexistent/malformed course -> :course_not_found; a duplicate -> :already_enrolled) and assert the
   postcondition (a valid enroll yields a stored %Enrollment{progress: 0} / a %LearnerEnrolled{}).

4) Doctests (T4 -> R5). Add runnable @doc examples to the public modules (Portal.ID, the contract, the core) and a
   `doctest` line in the relevant test modules so the suite runs them; a wrong example must fail.

5) One process test (T5 -> R6). Create test/portal/engine_test.exs with a single test: start Portal.Engine, issue a
   command, then a query, and assert the query reflects the command. This is the only process test in the suite.

6) Verify (T6 -> R1, R7). Run `mix test`. Confirm it is green and fast, the base holds the most tests, exactly one
   process test exists, properties pass, and doctests pass. Keep T1..T4 `async: true`.

Constraints: no mocking frameworks (the pure core needs none); deterministic and fast (no Process.sleep, seeded
generators); the EventStore adapters and LiveView tests are out of scope (F5.8 / F6). Match Portal conventions:
branded Snowflake IDs, pure functions, tagged tuples for expected outcomes.
```
