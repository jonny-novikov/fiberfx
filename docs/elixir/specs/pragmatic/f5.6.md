# F5.6 · Where engine state lives

> Give the folded state a home. Keep it alive between requests in a single supervised process: `Portal.Engine`, a
> GenServer whose `init/1` replays the log once into the starting state, with a command `handle_call` that runs
> `decide` then `evolve` and a query `handle_call` that only reads — supervised so a crash restarts the engine and
> re-folds the log. Production-bound: the recovery path is the harness's headline test.

## Goal

After F5.6, the F5.5 pure core has a runtime home. `Portal.Engine` is a GenServer started under `Portal.Application`;
`init/1` folds the current log (read live from the separate `Portal.EventLog` process) once into the held state. A
command call gates on `Core.authorize`: on `:ok` it runs `Core.decide` for the events, folds them with `Core.evolve`,
records them, and keeps the new state; on `{:error, reason}` it replies the mapped error with state unchanged. A query
call reads the held state and changes nothing. If the process crashes, the supervisor restarts it and `init/1` re-folds
the log to the same state. The web still calls only the `Portal.Engine` boundary.

## Rationale (5W)

- **Why** — A pure fold needs somewhere to live between requests. A single GenServer serialises commands and queries
  over one consistent, folded state, and OTP supervision turns "rebuild the state from the log" into automatic crash
  recovery — the property that makes the engine robust enough to ship.
- **What** — the state-home decision (GenServer vs Agent vs ETS), the `Portal.Engine` GenServer hosting the F5.5 core,
  and supervision that recovers by replay.
- **Who** — Operators (the engine self-heals and rebuilds its state on restart), Developers (one place where state
  lives, with serialised, predictable call semantics).
- **When** — the sixth rung; hosts the F5.5 core. It precedes F5.7 (the test pyramid) and F5.8 (the durable
  `EventStore` port and the `Portal` facade).
- **Where** — `lib/portal/engine.ex` (now the GenServer host, using `Portal.Engine.Core`) and
  `lib/portal/application.ex` (the supervised child). The pure core and the web boundary are unchanged.

## Scope

- **In** — the documented choice of a GenServer (and why, versus an Agent or ETS); `Portal.Engine` as a GenServer with
  `init/1` folding the live log (`Core.replay(Portal.EventLog.all())`), a command `handle_call` (authorize → decide →
  evolve → append → project → reply, keep new state) and a query `handle_call` (read, reply, state unchanged); the
  public boundary (`dispatch/1`, `query/2`) re-expressed as `GenServer.call`; supervision of a separate `Portal.EventLog`
  plus the engine (started plain) so a crash re-folds the current log; a crash-recovery test.
- **Out** — durable, swappable persistence of the log via the `EventStore` port and config-selected adapters (→ F5.8;
  here `load_events()` reads a log durable enough to survive a process crash); the `Portal` facade (→ F5.8; the closed
  `%Portal.Error{}` struct is introduced earlier, in F5.4); horizontal scale beyond the single process (the throughput
  ceiling is the accepted trade-off, revisited only if measured); read-model projection stores (the read model is the
  held fold).

## Deliverables

- **F5.6-D1** — the state-home decision documented: a GenServer is chosen (serialised command/query over one folded
  state); an Agent (simpler, less call control) and ETS (concurrent reads, manual consistency) are the alternatives;
  the trade-off — one process is a throughput ceiling — is acknowledged and accepted for now.
- **F5.6-D2** — `Portal.Engine` as a GenServer: `init(events) -> {:ok, Portal.Engine.Core.replay(events)}`, folding the
  log once at startup.
- **F5.6-D3** — the command callback `handle_call({:command, cmd}, _from, state)` cases on `Core.authorize(state,
  cmd)` (the F5.5-canonical Decider: `decide` is events-only and carries no error channel, so the shell cannot case on
  `decide`'s return). On `:ok` — `Core.decide(state, cmd)` for the events, `Core.evolve` each into the state,
  `Portal.EventLog.append` them, and reply, keeping the new state. On `{:error, reason}` — map the bare atom to a
  `%Portal.Error{}` via `Portal.Error.new/1` and reply it with state unchanged. `dispatch/1` keeps the F5.3 web map and
  stamps the occurrence time `at` at the shell (the web map → a 4-tuple core command); the success reply is the
  projected web shape (see D5).
- **F5.6-D4** — the query callback `handle_call({:query, q}, _from, state)`: read derived data from the held state,
  reply with the data, change nothing.
- **F5.6-D5** — the public boundary preserved: `Portal.Engine.dispatch/1` → `GenServer.call({:command, cmd})` and
  `query/2` → `GenServer.call({:query, q})`, so the F5.3 web layer calls the same boundary and no domain logic moves to
  the web. To keep the web byte-for-byte unchanged, F5.6 retains the Store `%Enrollment{}` projection as a **dual-write**
  alongside the held fold (the fold is the source of truth; the Store rows are its read-model projection): the command
  callback mints the `ENR` id and `Store.put`s the enrollment after `authorize` gates, so `dispatch/1` still returns
  `{:ok, %Enrollment{}}` and `:courses_of` still returns `%Enrollment{}` rows. The fold-only collapse — dropping the
  Store projection and the surrogate `ENR` id (so a cached `data.id` no longer resolves) — is the F6 cleanup, when the
  Plug router retires.
- **F5.6-D6** — supervision under `:one_for_one`; a process crash restarts the engine and `init/1` re-folds the log to
  the same state. **Build note:** `{Portal.Engine, load_events()}` as a *static child arg* is INSUFFICIENT — a
  supervisor evaluates a child's args exactly once, so a restart would re-fold a boot-time snapshot, not the current
  log. The log must live in a **separate, longer-lived process** (`Portal.EventLog`, started BEFORE the engine) that
  `init/1` reads live; the engine is started **plain** (`Portal.Application` starts `Portal.Engine` with no event-list
  arg) and folds `Portal.EventLog.all()` in `init/1`.
- **F5.6-D7** — the log is loaded by `Portal.EventLog.all()` at init and appended on each successful command, durable
  enough to survive a process crash; because the log is its own process (see D6), a restarted engine folds the events
  recorded right up to the crash, not a stale boot snapshot. The durable, swappable `EventStore` port is deferred to
  F5.8.
- **F5.6-D8** — tests: a command then a query reflects the change through the live GenServer; killing the engine
  restarts it and a subsequent query returns the same (re-folded) state; a query never mutates.

## Invariants

- **F5.6-INV1 (state is the held fold)** — the GenServer's state equals `Core.replay(log)`; the only mutation is
  folding an event via `Core.evolve/2` inside a command call.
- **F5.6-INV2 (command/query separation at the process)** — a `{:command, _}` call may change state and replies `:ok |
  {:error, reason}`; a `{:query, _}` call never changes state and replies with data.
- **F5.6-INV3 (crash recovery by replay)** — a process crash is recovered by the supervisor restarting the engine and
  `init/1` re-folding the log to the same state; no state lives only in volatile memory the log cannot reconstruct.
- **F5.6-INV4 (functional core, imperative shell)** — the GenServer is the shell; `decide`/`evolve`/`replay` stay pure
  and are only invoked by `handle_call`; no business logic is added to the callbacks.
- **F5.6-INV5 (master, held)** — the web calls only the `Portal.Engine` boundary (`dispatch`/`query`); the callbacks
  and state shape are private; no domain logic in the web.
- **F5.6-INV6 (serialised consistency)** — commands and queries are serialised through the single GenServer, so a query
  never observes a half-applied command; the throughput ceiling is the accepted trade-off.

## Definition of Done

- [ ] `Portal.Engine.init/1` folds the live `Portal.EventLog.all()` once into the starting state (F5.6-INV1).
- [ ] A command through the live engine changes state and replies `:ok`/`{:error, _}`; a query reads and replies data
      without mutating (F5.6-INV2).
- [ ] `dispatch/1` and `query/2` wrap `GenServer.call`; the F5.3 web layer is unchanged and calls only the boundary
      (F5.6-INV5).
- [ ] `Portal.Application` supervises a separate `Portal.EventLog` and the engine (started plain); killing the engine
      restarts it and a query returns the same re-folded state from the live log (F5.6-INV3).
- [ ] `decide`/`evolve`/`replay` remain pure; the callbacks add no business logic (F5.6-INV4).
- [ ] A crash-recovery test and a command→query test pass through the live process (F5.6-D8).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.6.stories.md`).
