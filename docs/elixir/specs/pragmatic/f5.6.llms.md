## Requirements

- **F5.6-R1** — a decision record selects a GenServer (serialised command/query over one folded state) and records the
  Agent/ETS trade-offs and the throughput ceiling. [US: F5.6-US1]
- **F5.6-R2** — `Portal.Engine` is a GenServer with `init(events) -> {:ok, Portal.Engine.Core.replay(events)}`. [US:
  F5.6-US2]
- **F5.6-R3** — `handle_call({:command, cmd}, _from, state)` cases on `Core.authorize(state, cmd)` (`decide` is
  events-only, so the shell gates on `authorize`, not on a `decide` error return): on `:ok`, `Core.decide` for the
  events, `Core.evolve` each into state, `Portal.EventLog.append` them, reply the projected web shape, keep the new
  state; on `{:error, reason}`, map the atom via `Portal.Error.new/1` and reply it with state unchanged. `dispatch/1`
  keeps the web map and stamps `at` (a 4-tuple at the shell). [US: F5.6-US2]
- **F5.6-R4** — `handle_call({:query, q}, _from, state)` reads derived data from state, replies the data, changes
  nothing. [US: F5.6-US3]
- **F5.6-R5** — `Portal.Engine.dispatch/1` → `GenServer.call({:command, cmd})` and `query/2` → `GenServer.call({:query,
  q})`; the F5.3 web layer is unchanged. [US: F5.6-US5]
- **F5.6-R6** — `Portal.Application` supervises the engine under `:one_for_one`; a crash restarts it and `init/1`
  re-folds the log. The log lives in a **separate, longer-lived process** (`Portal.EventLog`, started BEFORE the
  engine); the engine is started **plain** and reads the CURRENT log in `init/1`. A static `{Portal.Engine,
  load_events()}` child arg is INSUFFICIENT — a supervisor evaluates child args once, so it would re-fold a boot-time
  snapshot on restart. [US: F5.6-US4]
- **F5.6-R7** — `Portal.EventLog.all()` reads the log at init and the command path appends new events, durable enough
  to survive a process crash. [US: F5.6-US4]
- **F5.6-R8** — verification: command→query reflects the change; killing the engine restarts it and a query returns the
  same re-folded state; a query never mutates; the callbacks add no business logic. [US: all]

## Execution topology

**Runtime topology (the shell that hosts the F5.5 core).**

```text
Portal.Application (Supervisor, :one_for_one)
├─ Portal.EventLog                                        # the log: a SEPARATE, longer-lived process
└─ Portal.Engine  (started plain, no event-list arg)      # crash → restart → init re-folds the CURRENT log
        init(:ok) ─▶ {:ok, Portal.Engine.Core.replay(Portal.EventLog.all())}   # fold the live log once

   dispatch/1 ─▶ stamp `at`, web map → 4-tuple ─▶ GenServer.call({:command, cmd})
                  handle_call({:command, cmd}, _, state):
                    case Portal.Engine.Core.authorize(state, cmd)   # the contract; decide is events-only
                      ├─ :ok ─▶ events = Core.decide(state, cmd)
                      │         ─▶ Enum.reduce(events, state, &Core.evolve/2) = state'
                      │         ─▶ Portal.EventLog.append(events) ─▶ {:reply, project(cmd, events), state'}
                      └─ {:error, reason} ─▶ {:reply, {:error, Portal.Error.new(reason)}, state}   # unchanged
   query/2   ─▶ GenServer.call({:query, q})
                  handle_call({:query, q}, _, state): read derived data ─▶ {:reply, data, state}  # no mutation

(The web from F5.3 calls only dispatch/1 and query/2; handle_call and the state shape are private. The pure core does
 the deciding and folding; the GenServer adds no business logic — it authorizes, folds, records, projects.)
```

**Task topology (build order; each step keeps the suite green).**

```text
T1 decision record: GenServer vs Agent vs ETS, trade-off accepted        [R1]
     └─▶ T2 Portal.Engine GenServer: init/1 folds EventLog.all() via replay [R2]
           └─▶ T3 command handle_call (authorize → decide → evolve → append → project) [R3]
                 └─▶ T4 query handle_call (read, reply, no mutation)        [R4]
                       └─▶ T5 dispatch/1, query/2 wrap GenServer.call       [R5]
                             └─▶ T6 supervise EventLog + Engine (plain, live log) [R6, R7]
                                   └─▶ T7 crash-recovery + command→query tests [R8]
```

**Touched files.** `lib/portal/engine.ex` (now the GenServer host), `lib/portal/application.ex`,
`test/portal/engine_test.exs`, plus a short decision note (a `@moduledoc` on `Portal.Engine` or `docs/`).

## Agent stories

### F5.6-AS1 — Host the fold in a supervised GenServer [implements F5.6-US1, F5.6-US2]
Directive: record the GenServer choice (T1); implement `Portal.Engine` with `init/1` folding the live
`Portal.EventLog.all()` via `Core.replay` and the command `handle_call` (authorize → decide → evolve → append → project
→ reply; map the reject atom via `Portal.Error.new/1`) (T2, T3); supervise the engine **plain** under `:one_for_one`,
with `Portal.EventLog` started before it (T6).
Acceptance gate: a command through the live engine applies once and replies `:ok`/`{:error, _}` (never data); the
engine starts under the supervisor folding the log.

### F5.6-AS2 — Read and preserve the boundary [implements F5.6-US3, F5.6-US5]
Directive: implement the query `handle_call` (read, no mutation) (T4); wrap `dispatch/1` and `query/2` over
`GenServer.call` so the F5.3 web layer is unchanged (T5).
Acceptance gate: a query returns data and leaves state unchanged; the web tests pass untouched; the callbacks add no
business logic.

### F5.6-AS3 — Recover by replay [implements F5.6-US4]
Directive: ensure new events are recorded so `load_events()` returns them after a crash; write the crash-recovery test
(T7).
Acceptance gate: killing `Portal.Engine` restarts it and a subsequent query returns the same re-folded state.

## Execution plan — first two stories

*story → agent story → tasks → files → command → gate.*

### Plan A — F5.6-US2 (a command changes the live state) · via F5.6-AS1
1. Tasks T2 → T3 → T6.
2. Files: `lib/portal/engine.ex`, `lib/portal/application.ex`.
3. Command: `iex -S mix` then `u = Portal.ID.new("USR"); c = Portal.ID.new("CRS")` (seed a `%Course{}` so the contract
   admits the enroll), then `Portal.Engine.dispatch(%{type: :enroll, user_id: u, course_id: c})` (the web map — the
   shell stamps `at` and authorizes), then `Portal.Engine.query(:courses_of, u)`.
4. Gate (closes US2): the command replies `{:ok, %Enrollment{}}` and the later query reflects the change; the engine
   started by folding the log. Requirements: F5.6-R2, F5.6-R3, F5.6-R6.

### Plan B — F5.6-US4 (recover from a crash) · via F5.6-AS3
1. Task T7 (after AS1).
2. Files: `test/portal/engine_test.exs` (and the `load_events()`/record path in `lib/portal/engine.ex`).
3. Command: `mix test test/portal/engine_test.exs` — enroll, capture state, `Process.exit(pid, :kill)`, await
   restart, query again.
4. Gate (closes US4): after the kill, the supervisor restarts the engine and `init/1` re-folds the log to the same
   state; the post-crash query matches. Requirements: F5.6-R6, F5.6-R7.

Efficiency check: hosting the core is one GenServer plus a supervised child; recovery is `init/1` re-running `replay` —
no extra recovery code — the supervisor re-runs the fold on restart.

> in task-topology order and ends on the verification gates. Treat the Requirements and the Definition of Done in

You are implementing spec F5.6 (Where engine state lives) in the `portal` app, on top of F5.5. Read
specs/pragmatic/f5.6.md, specs/pragmatic/f5.6.stories.md, and specs/design/f0.md first. Host the F5.5 pure core
in a single supervised GenServer. The GenServer is the imperative shell: it must add NO business logic — it only calls

Build in this order, keeping the suite green after each step:

1) Decision record (T1 → R1). Document the choice of a GenServer (serialised command/query over one folded state) vs

   ceiling.

2) Engine GenServer (T2, T3, T4 → R2, R3, R4). Rework lib/portal/engine.ex into a GenServer (started plain — no
   event-list arg; see step 4):

   - handle_call({:command, cmd}, _from, state): case Core.authorize(state, cmd) — decide is events-only, so gate on

6) Verify (R8 + DoD). Confirm: command→query round-trips through the live process; the held state equals replay(log);
