# F5.5 ¬∑ Commands, queries & events

> Reframe the engine around what changed and what is asked. Split the write path (`:ok | {:error, reason}`, never
> data) from the read path (data, never a mutation), record every change as a past-tense **event**, and collapse the
> engine to two pure functions: `decide/2` turns a command into events, `evolve/2` folds one event into state, and
> `replay/1` derives the current state from the log. Production-bound and harnessed ‚Äî the pure core ships behind a
> full test suite.

## Goal

After F5.5, the Portal has a pure functional core. A command ‚Äî `{:enroll, user_id, course_id}` or `{:deliver_lesson,
user_id, lesson_id}` ‚Äî runs the F5.4 contract and returns `[event]` on success or `{:error, reason}` on an expected
failure, mutating nothing. `evolve/2` folds one event into state; `replay/1` is `reduce` over `initial_state/0`, so the
current state is exactly the fold of the event log. Queries read derived data and never change state. This core is
unit-pure and property-tested; F5.6 gives it a runtime home.

## Rationale (5W)

- **Why** ‚Äî Separating commands from queries and recording changes as events makes the engine's behaviour explicit,
  auditable, and reconstructible: state stops being a thing you mutate and becomes the fold of a history. Two small
  pure functions are trivial to test exhaustively, which is what lets a thin increment ship to production with
  confidence.
- **What** ‚Äî command/query separation, past-tense domain events, and the pure `decide` / `evolve` / `replay` trio.
- **Who** ‚Äî Developers (a pure core that is simple to test and reason about), Architects (an event-sourced, auditable,
  rebuildable engine shape).
- **When** ‚Äî the fifth rung; reframes the F5.4 enroll contract. Its pure core is wired into a supervised GenServer in
  F5.6 and given a durable event-store port in F5.8.
- **Where** ‚Äî `Portal.Learning.Events.*` (the event structs, owned by the context that emits them) and
  `Portal.Engine.Core` (the pure `decide`/`evolve`/`replay`). The F5.6 GenServer adopts this core; the web boundary is
  unchanged.

## Scope

- **In** ‚Äî command/query separation (CQS); the past-tense event structs `%LearnerEnrolled{}` and `%LessonDelivered{}`;
  `decide/2 :: (state, command) -> [event] | {:error, reason}` reusing the F5.4 contract; `evolve/2 :: (event, state)
  -> state` with `initial_state/0`; `replay/1 :: ([event]) -> state` as `reduce`; the command/query tuples; the
  example + property test harness for the core.
- **Out** ‚Äî the runtime home for the folded state and the GenServer callbacks (‚Üí F5.6); durable persistence of the log
  via the `EventStore` port and the `Portal` facade (‚Üí F5.8); HTTP rewiring (the F5.3 web boundary stays; F5.6 routes
  it through the held state); read-model projection tables (not needed ‚Äî the read model is the folded state).

A noted design decision (for review): `decide/2` is **pure** over `(state, command)`. Its state-dependent and
invariant checks ‚Äî already-enrolled, `progress` bounds ‚Äî run inside `decide`; the F5.4 **reference-existence** check
(`:course_not_found`, a catalog lookup) is resolved at the boundary before `decide` is called, so the core performs no
I/O. Both still surface the same closed reasons. The occurrence time `at` is supplied by the boundary and passed in, so
`decide` stays deterministic.

## Deliverables

- **F5.5-D1** ‚Äî command/query separation: a command path that returns `:ok | {:error, reason}` and never returns
  domain data, and a query path that returns data and never changes state.
- **F5.5-D2** ‚Äî past-tense event structs (`@enforce_keys`, `@type t`), owned by their context:
  `Portal.Learning.Events.LearnerEnrolled` (`user_id`, `course_id`, `at`) and
  `Portal.Learning.Events.LessonDelivered` (`user_id`, `lesson_id`, `at`).
- **F5.5-D3** ‚Äî `Portal.Engine.Core.decide/2 :: (state, command) -> [event] | {:error, reason}`: pure; for
  `{:enroll, u, c}` it runs the contract's state/invariant checks and emits `[%LearnerEnrolled{}]` or returns
  `{:error, :already_enrolled}`; for `{:deliver_lesson, u, l}` it emits `[%LessonDelivered{}]`; it mutates nothing.
- **F5.5-D4** ‚Äî `Portal.Engine.Core.evolve/2 :: (event, state) -> state`: pure; one clause per event type folds exactly
  one event into the state; plus `initial_state/0`.
- **F5.5-D5** ‚Äî `Portal.Engine.Core.replay/1 :: ([event]) -> state`: `Enum.reduce(events, initial_state(), &evolve/2)`
  ‚Äî the current state derived from the log alone.
- **F5.5-D6** ‚Äî the command and query tuples: commands `{:enroll, user_id, course_id}` and `{:deliver_lesson, user_id,
  lesson_id}`; query `{:enrollments, user_id}` reading the derived state.
- **F5.5-D7** ‚Äî the test harness: example tests for `decide` (success emits the right events; a duplicate returns
  `{:error, :already_enrolled}`); `evolve` folds one event; `replay` rebuilds state from a log; a `StreamData`
  property that `replay(log)` equals folding the same events incrementally.
- **F5.5-D8** ‚Äî verification: a command returns `:ok`/`{:error, _}` and never data; a query returns data and never
  mutates; `replay(log)` reconstructs the state the incremental folds produced; events are past-tense and immutable.

## Invariants

- **F5.5-INV1 (command/query separation)** ‚Äî the command path returns `:ok | {:error, reason}` and never domain data;
  the query path returns data and never changes state.
- **F5.5-INV2 (pure core)** ‚Äî `decide/2`, `evolve/2`, and `replay/1` are pure functions of their arguments: no I/O, no
  process, no hidden clock; identical inputs always produce identical outputs (functional core, imperative shell).
- **F5.5-INV3 (events are facts)** ‚Äî events are past-tense, immutable structs recording what happened; the command path
  produces only events, and state changes only by folding an event through `evolve/2`.
- **F5.5-INV4 (state is the fold of the log)** ‚Äî the current state equals `replay(log)`; `replay` is `reduce` over
  `initial_state/0`, so state is reconstructible from events alone.
- **F5.5-INV5 (contract preserved)** ‚Äî `decide/2` enforces the F5.4 contract before emitting events; expected failures
  return the same closed reasons (`:course_not_found`, `:already_enrolled`, ‚Ä¶) as tagged tuples, never as events.

## Definition of Done

- [ ] A command returns `:ok`/`{:error, reason}` and never domain data; a query returns data and never mutates
      (F5.5-INV1).
- [ ] `decide/2`, `evolve/2`, `replay/1` are pure ‚Äî no I/O, deterministic on their inputs (F5.5-INV2).
- [ ] `%LearnerEnrolled{}` and `%LessonDelivered{}` exist as past-tense structs with enforced keys and `@type t`.
- [ ] `replay(log)` reconstructs the same state as incremental folding; a `StreamData` property confirms it
      (F5.5-INV4).
- [ ] An expected failure returns a closed tagged reason, never an event (F5.5-INV5).
- [ ] The pure core ships behind its example + property test harness (F5.5-D7).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.5.stories.md`).
