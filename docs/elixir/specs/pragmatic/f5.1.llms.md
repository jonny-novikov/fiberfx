## Requirements

- **F5.1-R1** — a supervised Mix project `portal` (`--sup`) with `bandit ~> 1.5`, `plug ~> 1.16`, `jason ~> 1.4`; no
  Phoenix/Ecto; `mix deps.get` and `mix compile` clean. [US: F5.1-US1]
- **F5.1-R2** — `Portal.ID` implements `new/1` (`"ENR" <> base62`), `snowflake/1`, `namespace/1`, `at/1`; the snowflake
  layout is `timestamp(41b, epoch 1704067200000) <<22 | node(10b) <<12 | seq(12b)`; pure except clock/sequence;
  `snowflake(new(ns))` round-trips. [US: F5.1-US2]
- **F5.1-R3** — `Portal.Engine` is a GenServer named `__MODULE__` with `dispatch/1` and `query/2` returning `{:ok,
  term} | {:error, atom}`, both stubbed to `{:error, :not_implemented}`. [US: F5.1-US1, F5.1-US3]
- **F5.1-R4** — `Portal.Web.Router` (Plug.Router) wires `:match`, `Plug.Parsers` (`:urlencoded`, `:json` via `Jason`),
  `:dispatch`, the `GET /courses/:user_id` and `POST /enroll` routes mapping to the engine boundary with the documented
  statuses, and `match _` → 404. [US: F5.1-US1, F5.1-US3]
- **F5.1-R5** — `Portal.Application` supervises `[Portal.Engine, {Bandit, plug: Portal.Web.Router, port: 4000}]` with
  `:one_for_one`. [US: F5.1-US1, F5.1-US4]
- **F5.1-R6** — the router contains no domain logic (only parse → boundary → format); the domain depends on nothing in
  the web layer. [US: F5.1-US3]
- **F5.1-R7** — a roadmap states the current state, the F6 target (replace the web layer with Phoenix), and what is
  preserved (domain, ids, boundary). [US: F5.1-US5]
- **F5.1-R8** — boot succeeds; `POST /enroll` → `422 :not_implemented`; an unknown path → `404`; killing
  `Portal.Engine` restarts it and a later request succeeds. [US: F5.1-US1, F5.1-US4]

## Execution topology

**Runtime topology (what exists and how a request flows).**

```text
Portal.Application (Supervisor, :one_for_one)
├─ Portal.Engine                         # GenServer, stubbed boundary (dispatch/1, query/2)
└─ {Bandit, plug: Portal.Web.Router, port: 4000}
        │  (request)
        ▼
   Plug.Parsers → Plug.Router (:match → :dispatch)
        ├─ GET  /courses/:user_id → Portal.Engine.query(:courses_of, user_id) → 200 | 422
        ├─ POST /enroll          → Portal.Engine.dispatch(%{type: :enroll, …}) → 201 | 422
        └─ match _               → 404
```

Both domain routes return `{:error, :not_implemented}` from the stub until F5.2+ gives the engine real behaviour. The
web layer never holds logic; it only maps HTTP to the boundary.

**Task topology (build order; each step leaves the app compiling).**

```text
T1 scaffold `portal --sup` + deps (bandit/plug/jason)            [R1]
     └─▶ T2 Portal.ID (mint + decode branded Snowflakes)         [R2]
           └─▶ T3 Portal.Engine GenServer (stubbed dispatch/query) [R3]
                 └─▶ T4 Portal.Web.Router (parsers + routes + 404)  [R4, R6]
                       └─▶ T5 Portal.Application supervises Engine + Bandit  [R5]
                             └─▶ T6 development roadmap (now / F6 / preserved)  [R7]
                                   └─▶ T7 verify (boot, 422, 404, restart)      [R8]
```

**Touched files.** `mix.exs`, `lib/portal/id.ex`, `lib/portal/engine.ex`, `lib/portal_web/router.ex`,
`lib/portal/application.ex`, plus a short `ROADMAP.md` (or a `@moduledoc` on `Portal.Application`).

## Agent stories

Each agent story implements one user story and closes on a gate. Build in task-topology order.

### F5.1-AS1 — Boot a supervised HTTP app [implements F5.1-US1, F5.1-US4]
Directive: scaffold `portal --sup` with the three deps (T1); add the stubbed `Portal.Engine` (T3); build
`Portal.Web.Router` (T4); supervise `[Portal.Engine, {Bandit, …port: 4000}]` `:one_for_one` (T5).
Acceptance gate: `mix compile` clean; app boots; `POST /enroll` → `422 :not_implemented`; unknown path → `404`;
killing `Portal.Engine` restarts it and a later request succeeds.

### F5.1-AS2 — Branded Snowflake ids [implements F5.1-US2]
Directive: implement `Portal.ID` with `new/1`, `snowflake/1`, `namespace/1`, `at/1` and the documented layout (T2).
Acceptance gate: `new("ENR")` returns `"ENR" <> base62`; `snowflake(new("ENR"))` round-trips; `at/1` returns a UTC
`DateTime`; the functions are pure except the clock/sequence.

### F5.1-AS3 — Hold the replaceable seam [implements F5.1-US3, F5.1-US5]
Directive: keep the router free of domain logic (parse → boundary → format only); confirm the domain has no dependency
on the web layer; write the roadmap (T6).
Acceptance gate: the router calls only the `Portal.Engine` boundary; no domain logic in `lib/portal_web/`; the roadmap
states current/target/preserved.

## Execution plan — first two stories

*story → agent story → tasks → files → command → gate.*

### Plan A — F5.1-US1 (serve over HTTP from day one) · via F5.1-AS1
1. Tasks T1 → T3 → T4 → T5.
2. Files: `mix.exs`, `lib/portal/engine.ex`, `lib/portal_web/router.ex`, `lib/portal/application.ex`.
3. Command: `mix deps.get && mix compile && mix run --no-halt`, then `curl -i -X POST localhost:4000/enroll` and
   `curl -i localhost:4000/nope`.
4. Gate (closes US1): `422 :not_implemented` on enroll, `404` on the unknown path; killing `Portal.Engine` restarts
   it. Requirements: F5.1-R1, F5.1-R3, F5.1-R4, F5.1-R5, F5.1-R8.

### Plan B — F5.1-US2 (branded Snowflake ids) · via F5.1-AS2
1. Task T2.
2. Files: `lib/portal/id.ex`.
3. Command: `mix test test/portal/id_test.exs` (pure, no process).
4. Gate (closes US2): `new("ENR")` prefixed + Base62; `snowflake(new("ENR"))` round-trips; `at/1` UTC.
   Requirements: F5.1-R2.

Efficiency check: the running app is four files behind one command and one gate; the id generator is one pure module
behind one test — the layout, statuses, and supervision strategy are all fixed by the spec.

## Comprehensive implementation prompt

> Paste into a coding agent (e.g. Claude Code) in an empty directory. It executes F5.1-AS1..AS3 in task-topology order
> and ends on the verification gates. Treat the Requirements and the Definition of Done in [`f5.1.md`](f5.1.md) as the
> acceptance checklist.

```text
You are implementing spec F5.1 (Start thin: a running Portal). Read specs/pragmatic/f5.1.md,
specs/pragmatic/f5.1.stories.md, and build-guide/f5-01-start-thin.md first. Build a thin, supervised, framework-free
foundation. Master-invariant seed: the web layer holds no domain logic and the domain depends on nothing in the web
layer, so F6 can later replace Bandit with Phoenix untouched. Control flow: tagged tuples / `with`; no exceptions for
expected outcomes.

Build in this order, keeping the app compiling after each step:

1) Scaffold (T1 → R1). `mix new portal --sup`. Add bandit ~> 1.5, plug ~> 1.16, jason ~> 1.4 to mix.exs; no Phoenix,
   no Ecto. Run mix deps.get.

2) Portal.ID (T2 → R2). Create lib/portal/id.ex. A snowflake is a 64-bit integer: timestamp(41 bits, ms since epoch
   1704067200000) <<22 | node(10 bits, default 1) <<12 | seq(12 bits, monotonic per ms). A branded id is a 3-letter
   uppercase namespace followed by Base62(0-9A-Za-z) of the snowflake. Implement new/1, snowflake/1, namespace/1,
   at/1. Pure except the clock and sequence.

3) Portal.Engine stub (T3 → R3). Create lib/portal/engine.ex: a GenServer named __MODULE__ with dispatch/1 and
   query/2, each returning {:ok, term} | {:error, atom}, both stubbed to {:error, :not_implemented}.

4) Router (T4 → R4, R6). Create lib/portal_web/router.ex using Plug.Router: plug :match; plug Plug.Parsers, parsers:
   [:urlencoded, :json], json_decoder: Jason; plug :dispatch. Routes: get "/courses/:user_id" ->
   Portal.Engine.query(:courses_of, user_id) returning 200 on {:ok,_} and 422 on {:error,_}; post "/enroll" ->
   Portal.Engine.dispatch(%{type: :enroll, user_id: params["user"], course_id: params["course"]}) returning 201/422;
   match _ -> 404 "not found". No domain logic in the router.

5) Supervise (T5 → R5). In lib/portal/application.ex, start children [Portal.Engine, {Bandit, plug:
   Portal.Web.Router, port: 4000}] with strategy: :one_for_one.

6) Roadmap (T6 → R7). Document what exists now (a thin supervised server over a stubbed boundary), what F6 replaces
   (the web layer, with Phoenix), and what stays (the domain, ids, the engine boundary).

7) Verify (T7 → R8 + DoD). Confirm: mix compile clean; the app boots; POST /enroll -> 422 :not_implemented; GET /nope
   -> 404; killing Portal.Engine restarts it and a later request succeeds; Portal.ID.new("ENR") mints and
   Portal.ID.snowflake/1 round-trips. Report each result against the F5.1 Definition of Done.

Deliverables: the touched files listed in specs/pragmatic/f5.1.llms.md, all compiling, with the app runnable and every
F5.1 invariant exercised. Stop when the Definition of Done in specs/pragmatic/f5.1.md is fully checked.
```
