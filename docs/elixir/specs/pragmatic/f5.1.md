# F5.1 ¬∑ Start thin: a running Portal

> Ship a running, supervised Portal from day one. Stand up a `portal` Mix app that answers real HTTP on port 4000
> through a thin `Bandit` + `Plug` web layer, mint identity with `Portal.ID` (branded Snowflake ids), and put a stubbed
> `Portal.Engine` boundary between the web and the domain-to-come ‚Äî a web layer thin enough that Phoenix replaces it
> untouched in F6.

## Goal

After F5.1, `mix run` boots a supervised application that serves HTTP on port 4000. A request travels through a thin
`Plug.Router` to a stubbed `Portal.Engine` boundary (`dispatch/1`, `query/2`) and back. `Portal.ID` mints and decodes
branded Snowflake ids. The web layer holds no domain logic, so F6 can swap `Bandit` for Phoenix without touching
anything below it.

## Rationale (5W)

- **Why** ‚Äî A pragmatic build ships a running, supervised system from day one rather than a big upfront framework, and
  the web layer must be a thin, replaceable adapter so F6 can swap `Bandit` ‚Üí Phoenix without disturbing the domain.
- **What** ‚Äî a supervised `portal` app, the `Portal.ID` generator, a stubbed engine boundary, and a thin router that
  answers real HTTP.
- **Who** ‚Äî Operators (run and serve it), Developers (a runnable foundation and the id convention), Architects (a
  replaceable web seam established on day one).
- **When** ‚Äî the first rung of the pragmatic ladder; depends only on F4 (the branded CHAMP store) and the language/OTP.
- **Where** ‚Äî a new `portal` app: `Portal.ID`, `Portal.Engine` (stub), `Portal.Web.Router`, `Portal.Application`. The
  domain itself is modeled in F5.2.

## Scope

- **In** ‚Äî the supervised Mix project and its deps (`bandit`, `plug`, `jason`; no Phoenix/Ecto); `Portal.ID` (mint and
  decode branded Snowflakes); the stubbed `Portal.Engine` boundary; the thin router (`:match`, parsers, `:dispatch`, a
  read route, a write route, a 404 fallback); supervision of the engine and `Bandit`; the replaceable-web-layer
  discipline; a development roadmap to Phoenix; verification of boot, the stub responses, and engine restart.
- **Out** ‚Äî the domain model (‚Üí F5.2); real engine logic (‚Üí F5.3+); the contract (‚Üí F5.4); commands/queries/events
  (‚Üí F5.5); the supervised state home and event-log persistence (‚Üí F5.6 / F5.8); Phoenix itself (‚Üí F6).

## Deliverables

- **F5.1-D1** ‚Äî a supervised Mix project `portal` (generated with `--sup`) with `bandit ~> 1.5`, `plug ~> 1.16`,
  `jason ~> 1.4` in `mix.exs`; no Phoenix, no Ecto; `mix deps.get` and `mix compile` clean.
- **F5.1-D2** ‚Äî `Portal.ID` (lib/portal/id.ex): a snowflake is a 64-bit integer `timestamp(41 bits, ms since epoch
  1704067200000) <<22 | node(10 bits) <<12 | seq(12 bits)`; a branded id is a 3-letter uppercase namespace + the
  Base62 (0-9A-Za-z) encoding of the snowflake; public `new/1`, `snowflake/1`, `namespace/1`, `at/1`; pure except the
  clock and a monotonic per-ms sequence.
- **F5.1-D3** ‚Äî `Portal.Engine` (lib/portal/engine.ex): a GenServer named `__MODULE__` exposing `dispatch/1` and
  `query/2`, each returning `{:ok, term} | {:error, atom}`, both stubbed to `{:error, :not_implemented}`.
- **F5.1-D4** ‚Äî `Portal.Web.Router` (Plug.Router): `plug :match`; `Plug.Parsers` (`:urlencoded`, `:json` via `Jason`);
  `plug :dispatch`; `GET /courses/:user_id` ‚Üí `Portal.Engine.query(:courses_of, user_id)` (200 on `{:ok, _}`, 422 on
  `{:error, _}`); `POST /enroll` ‚Üí `Portal.Engine.dispatch(%{type: :enroll, user_id: params["user"], course_id:
  params["course"]})` (201 / 422); `match _` ‚Üí 404.
- **F5.1-D5** ‚Äî `Portal.Application` supervises `[Portal.Engine, {Bandit, plug: Portal.Web.Router, port: 4000}]` with
  `strategy: :one_for_one`.
- **F5.1-D6** ‚Äî the web layer is a thin, replaceable adapter: the router only parses a request, calls the engine
  boundary, and formats a response ‚Äî no domain logic ‚Äî plus a documented roadmap stating what exists now, what F6
  replaces (the web layer), and what stays (the domain, ids, engine boundary).
- **F5.1-D7** ‚Äî verification: the app boots under its supervisor; `POST /enroll` returns `422 :not_implemented`; an
  unknown path returns `404`; killing `Portal.Engine` triggers a supervised restart and requests succeed again.

## Invariants

- **F5.1-INV1 (replaceable seam ¬∑ seed of the master invariant)** ‚Äî the web layer holds no domain logic; it only
  translates HTTP ‚Üî the engine boundary, and the domain depends on nothing in the web layer, so F6 can replace `Bandit`
  with Phoenix untouched.
- **F5.1-INV2 (identity)** ‚Äî every entity is identified by a Snowflake (a 64-bit integer, canonical) whose transport
  form is a branded id (3-letter namespace + Base62); `Portal.ID` mints and decodes; the integer is canonical, the
  string is transport.
- **F5.1-INV3 (supervision ¬∑ let it crash)** ‚Äî the application is supervised `:one_for_one`; a crashed child restarts
  on its own without taking the system down.
- **F5.1-INV4 (pure id functions)** ‚Äî `Portal.ID` is pure except the clock and sequence; `snowflake(new(ns))`
  round-trips and `at/1` returns a UTC `DateTime`.
- **F5.1-INV5 (functional core, imperative shell ¬∑ seed)** ‚Äî side effects (HTTP, the clock) live at the edges; the id
  math and request‚Üíboundary mapping are pure.

## Definition of Done

- [ ] `mix deps.get` and `mix compile` are clean; the app boots under `Portal.Application`.
- [ ] `Bandit` listens on port 4000.
- [ ] `POST /enroll` returns `422 :not_implemented`; an unknown path returns `404`.
- [ ] `Portal.ID.new("ENR")` mints a branded id; `Portal.ID.snowflake/1` round-trips it; `at/1` returns a UTC
      `DateTime`.
- [ ] Killing `Portal.Engine` restarts it and a subsequent request succeeds (F5.1-INV3).
- [ ] The router calls only the `Portal.Engine` boundary; no domain logic lives in the web layer (F5.1-INV1).
- [ ] A roadmap documents the current state, the F6 target, and what is preserved.
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.1.stories.md`).
