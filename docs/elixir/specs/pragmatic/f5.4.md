# F5.4 · The enroll contract

> Harden the enroll slice so invalid input is rejected at the door and nothing downstream is ever corrupted. Express
> the contract the functional way: parse the request into a well-formed command or a typed `{:error, reason}` before
> any id is minted or stored (precondition), guarantee a stored `%Enrollment{progress: 0}` on success (postcondition),
> and keep `progress` within `0..100` for every enrollment the system produces (invariant). Expected failures are
> tagged tuples mapped to `422`; impossible states crash.

## Goal

After F5.4, `Portal.Learning.enroll/2` checks its preconditions before it acts: a malformed reference or a course that
does not exist returns `{:error, :course_not_found}`, a repeat returns `{:error, :already_enrolled}`, and only a valid
request mints an `ENR` id and stores `%Enrollment{progress: 0}`. The router maps those expected reasons to `422`. An
out-of-range `progress` is treated as a bug, not an expected error. The expected reasons are the seed of the closed
`%Portal.Error{}` set finalized in F5.8.

## Rationale (5W)

- **Why** — A slice that round-trips is not yet trustworthy: it must reject bad input at the boundary and protect its
  invariants, or corrupt state leaks downstream. Stating the contract — what the caller must meet, what the function
  guarantees, what is always true — turns "it usually works" into "it is correct or it fails cleanly."
- **What** — a precondition guard (parse-don't-validate), a guaranteed postcondition, a state invariant, a clean split
  between expected and impossible failures, the router mapping, and property tests.
- **Who** — Learners (a bad or duplicate enroll is rejected, their data stays consistent), Developers (a contract that
  fails fast and a closed error vocabulary to map).
- **When** — the fourth rung; hardens the F5.3 enroll slice. Its expected reasons are completed into `%Portal.Error{}`
  in F5.8.
- **Where** — `lib/portal/learning.ex` (the contract) and `lib/portal_web/router.ex` (status mapping). Lineage: this is
  design by contract, re-expressed functionally as boundary parsing, types, and properties.

## Scope

- **In** — the precondition `with` chain on `enroll/2` (well-formed `USR`/`CRS` ids, the course exists, not already
  enrolled) returning typed errors; the guaranteed postcondition (`{:ok, %Enrollment{progress: 0}}`, stored); the
  `0..100` `progress` invariant; the expected-vs-impossible failure split; the router mapping of expected reasons to
  `422`; fail-fast ordering (check before mint/build/store); property tests for the postcondition and invariant plus
  example tests for each expected error.
- **Out** — the full closed `%Portal.Error{}` struct and codes (→ F5.8 wraps these reasons); commands/queries/events
  (→ F5.5); progress-update commands and the `:invalid_progress`/`:lesson_locked` paths (introduced as the model grows,
  finalized in F5.8); persistence of the rejection as an event (→ F5.5/F5.6).

## Deliverables

- **F5.4-D1** — a precondition guard on `Portal.Learning.enroll/2`: a `with` chain that (a) confirms `user_id` is a
  well-formed `USR…` and `course_id` a well-formed `CRS…` branded id (via `Portal.ID.namespace/1`), (b) confirms the
  course exists (`Portal.Catalog.course/1`), and (c) confirms the learner is not already enrolled — short-circuiting to
  `{:error, :course_not_found}` or `{:error, :already_enrolled}` before any effect.
- **F5.4-D2** — the postcondition: on success `enroll/2` returns `{:ok, %Enrollment{progress: 0}}` and the enrollment
  is stored; the guarantee holds by construction (the success branch is the only path that mints and stores).
- **F5.4-D3** — the invariant: `progress` is always within `0..100` for every enrollment the system produces — encoded
  in the struct type and treated as impossible to violate through the public API.
- **F5.4-D4** — the failure split: expected failures are tagged tuples `{:error, reason}` with `reason` in a small
  closed set (`:course_not_found`, `:already_enrolled`); an impossible state (`progress` out of `0..100`) is a
  `raise`/failed match (a crash), never a returned tuple.
- **F5.4-D5** — the router maps each expected reason to `422` and `{:ok, _}` to `201`; the expected reasons are the
  only enroll errors the web handles.
- **F5.4-D6** — fail-fast ordering: every precondition is checked before an id is minted, a struct built, or the store
  written; a rejected enroll leaves the store unchanged.
- **F5.4-D7** — property tests (`StreamData`): the postcondition (a successful enroll ⇒ `progress: 0`, stored) and the
  invariant (`0 ≤ progress ≤ 100`) hold across generated inputs; example tests cover each expected error.
- **F5.4-D8** — verification: a malformed or nonexistent course → `422 :course_not_found` with nothing stored; a
  duplicate enroll → `422 :already_enrolled` with exactly one enrollment; a valid enroll → `201` with a stored
  `%Enrollment{progress: 0}`; the property tests pass.

## Invariants

- **F5.4-INV1 (parse, don't validate)** — untrusted enroll input is parsed at the door into a well-formed command or a
  typed `{:error, reason}`; no unvalidated reference reaches the mint/build/store step.
- **F5.4-INV2 (fail fast, no partial effect)** — all preconditions are checked before any id mint, struct build, or
  store write; a rejected enroll leaves the store byte-for-byte unchanged.
- **F5.4-INV3 (closed expected-error vocabulary · seed)** — expected failures use a small, closed set of reasons
  (`:course_not_found`, `:already_enrolled`, …) — the seed of the `%Portal.Error{}` codes finalized in F5.8 — and the
  web maps each to `422`.
- **F5.4-INV4 (expected vs impossible split)** — expected domain failures are tagged tuples and are never raised;
  impossible states (`progress` ∉ `0..100`) are bugs and crash under supervision, never returned as a tuple.
- **F5.4-INV5 (postcondition by construction)** — a successful enroll always yields `{:ok, %Enrollment{progress: 0}}`
  that is stored; this is guaranteed by the single success path and pinned by a property test, not a runtime assertion.
- **F5.4-INV6 (state invariant)** — `0 ≤ progress ≤ 100` holds for every enrollment the system produces; encoded in the
  type and checked by a property.

## Definition of Done

- [ ] A malformed or nonexistent course → `422 :course_not_found`, and nothing is written (F5.4-INV1, INV2).
- [ ] A duplicate enroll → `422 :already_enrolled`, and exactly one enrollment exists (F5.4-INV3).
- [ ] A valid enroll → `201` with a stored `%Enrollment{progress: 0}` (F5.4-INV5).
- [ ] Preconditions run before any mint/build/store; a rejected enroll mints no id and writes nothing (F5.4-INV2).
- [ ] Expected failures are tagged tuples mapped to `422`; an out-of-range `progress` crashes rather than returning a
      tuple (F5.4-INV4).
- [ ] Property tests for the postcondition and the `0..100` invariant pass; example tests cover each expected error
      (F5.4-D7).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.4.stories.md`).
