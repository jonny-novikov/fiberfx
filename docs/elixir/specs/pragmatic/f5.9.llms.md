## Requirements

- **F5.9-R1** — `Portal.Application` starts `EventStore.adapter()` then `{Portal.Engine, []}`, `:one_for_one`; `init/1`
  reads the stream and replays. [US: F5.9-US1]
- **F5.9-R2** — the command path runs the contract, then `decide` → `append/2` → `evolve`, appending before folding.
  [US: F5.9-US2]
- **F5.9-R3** — `Portal` (`enroll/2`, `deliver_lesson/2`, `progress_of/1`) delegates to `command/1`/`query/1`, the only
  `GenServer.call` callers. [US: F5.9-US3]
- **F5.9-R4** — `Portal.Error` has a closed `code` union; `from/1` has no catch-all. [US: F5.9-US4]
- **F5.9-R5** — `PortalWeb.EnrollmentLive` `mount/3` queries, `handle_event/3` commands and branches on
  `%Portal.Error{}`, `render/1` draws from `assigns`; it names only `Portal`. [US: F5.9-US3, F5.9-US4]
- **F5.9-R6** — the F6 handoff adds only `PortalWeb.Endpoint`; everything below the boundary is unchanged. [US:
  F5.9-US5]
- **F5.9-R7** — `mix run` boots, an enroll persists, and an engine restart replays to the same state. [US: F5.9-US1,
  F5.9-US2]

## Execution topology

```text
The assembled tree and the UI path:

   Portal.Application (Supervisor, :one_for_one)
     ├─ Portal.EventStore.adapter()        # started first; the engine reads from it on boot
     └─ {Portal.Engine, []}                # init/1: read_stream -> replay -> {:ok, state}
            │
            ▼
   PortalWeb.EnrollmentLive (sketch)        # F6 mounts this under PortalWeb.Endpoint
     ├─ mount/3        ─▶ Portal.progress_of/1 ─▶ assign :progress | :error
     ├─ handle_event/3 ─▶ Portal.deliver_lesson/2 ─▶ :ok (re-query) | {:error, %Portal.Error{}} (assign :error)
     └─ render/1       ─▶ draws from @progress, @error          # touches only Portal

   command path:  contract guard ─▶ decide ─▶ append/2 (port) ─▶ evolve     # append BEFORE evolve
```

```text
Task DAG:
  T1 tree(store→engine) ─▶ T2 command append-before-evolve ─▶ T3 facade ─▶ T4 error+from/1 ─▶ T5 LiveView sketch ─▶ T6 handoff note ─▶ T7 verify(mix run, persist, restart-replay)
```

Touched files: `lib/portal/application.ex` (the tree), `lib/portal/engine.ex` (init + command path), `lib/portal.ex`
(facade), `lib/portal/error.ex`, `lib/portal/event_store.ex` and `lib/portal/event_store/in_memory.ex` (+ `postgres.ex`
from F5.8/F6.3), and `lib/portal_web/live/enrollment_live.ex` (the sketch).

## Agent stories

- **F5.9-AS1** [implements F5.9-US1, F5.9-US2] — Directive: assemble `Portal.Application` (store then engine,
  `:one_for_one`); make `init/1` read the stream and replay; make the command path append before it evolves.
  Acceptance gate: `mix run` boots; an enroll appends before folding; an engine restart replays to the same state.
- **F5.9-AS2** [implements F5.9-US3] — Directive: confirm the `Portal` facade over `command/1`/`query/1` and write the
  `PortalWeb.EnrollmentLive` sketch (`mount/3` query, `handle_event/3` command, `render/1` from `assigns`). Acceptance
  gate: the sketch names only `Portal` and compiles.
- **F5.9-AS3** [implements F5.9-US4] — Directive: branch the sketch on `{:ok, data}` and each `%Portal.Error{}` code;
  confirm `from/1` has no catch-all. Acceptance gate: every outcome is handled; adding a code forces a new branch.
- **F5.9-AS4** [implements F5.9-US5] — Directive: write the F6 handoff note in the module docs — F6 adds only
  `PortalWeb.Endpoint`; the engine, facade, ports, and error contract are unchanged. Acceptance gate: the one
  structural change is documented.

## Execution plan — first two stories

1. **F5.9-AS1 — assemble and recover.** In `lib/portal/application.ex`, `children = [Portal.EventStore.adapter(),
   {Portal.Engine, []}]`, `Supervisor.start_link(children, strategy: :one_for_one)`. In `lib/portal/engine.ex`,
   `init/1` calls `adapter().read_stream(stream)` then `Core.replay/1`; the command handler runs the contract, then
   `decide`, `append/2`, `evolve`. Run `mix run`, enroll, restart the engine process, and confirm the state replays.
2. **F5.9-AS2 — the facade and the LiveView sketch.** Confirm `lib/portal.ex` delegates to `command/1`/`query/1`.
   Create `lib/portal_web/live/enrollment_live.ex` with `mount/3` (query `Portal.progress_of/1` into `assigns`),
   `handle_event/3` (call `Portal.deliver_lesson/2`, re-query on `:ok`, assign `:error` on `{:error, _}`), and
   `render/1` drawing from `@progress`/`@error`. Keep it calling only `Portal`.

## Comprehensive implementation prompt

```text
You are assembling the Portal engine into a running, supervised, UI-ready system (rung F5.9, the F5 lab). Read
specs/pragmatic/f5.9.md and the seams from F5.8: the EventStore port + adapters, the Portal facade, and the closed
Portal.Error contract. The command path is decide -> append -> evolve; decide returns events only and the F5.4
contract guards rejection at the boundary.

Wire it together:

1) Supervision tree (T1 -> R1). In lib/portal/application.ex, start children in order: Portal.EventStore.adapter()
   first, then {Portal.Engine, []}, with Supervisor strategy :one_for_one. The engine's init/1 reads its stream via
   read_stream/1 and folds it with Core.replay/1 into {:ok, state}.

2) Command path (T2 -> R2). In lib/portal/engine.ex, handle_call({:command, cmd}, _, state) runs the F5.4 contract
   guard; on a valid command it calls decide, then append/2 through the adapter, then evolve — appending events
   BEFORE folding them in, so a restart replays to the same state. handle_call({:query, q}, ...) reads without
   mutating.

3) Facade (T3 -> R3). Confirm lib/portal.ex exposes enroll/2, deliver_lesson/2, progress_of/1 over command/1 and
   query/1 (the only GenServer.call callers), returning :ok | {:ok, data} | {:error, %Portal.Error{}}.

4) Error contract (T4 -> R4). Confirm lib/portal/error.ex defines %Portal.Error{code, message, field} with the closed
   code union and a from/1 with no catch-all.

5) LiveView sketch (T5 -> R5). Create lib/portal_web/live/enrollment_live.ex:
   - mount/3 loads progress via Portal.progress_of/1 and assigns :progress and :error;
   - handle_event/3 issues Portal.deliver_lesson/2, re-queries on :ok, and assigns :error on {:error, %Portal.Error{}};
   - render/1 draws from @progress and @error.
   It must call only Portal — never the engine, store, or a repo. Branch on {:ok, data} and each Portal.Error code.

6) Handoff note (T6 -> R6). In the module docs, state the F6 handoff: F6 adds only PortalWeb.Endpoint to the same
   tree and calls the same facade; the engine, facade, ports, and error contract are unchanged.

7) Verify (T7 -> R7). Confirm `mix run` boots the app, an enroll persists an event through the port, and restarting
   the engine replays the log to the same state. Confirm the LiveView sketch compiles and names only Portal.

Constraints: the web names only Portal; only command/1 and query/1 call GenServer.call; append happens before evolve;
from/1 has no catch-all; branded Snowflake IDs and pure core throughout. Out of scope: the HTTP endpoint/router and a
real template (F6), and the Postgres adapter body / schema / migration (F6.3).
```
