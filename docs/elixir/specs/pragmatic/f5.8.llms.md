## Requirements

- **F5.8-R1** — `Portal.EventStore` is a behaviour with `append/2 :: (String.t(), [struct]) :: :ok | {:error, term}`
  and `read_stream/1 :: (String.t()) :: {:ok, [struct]} | {:error, term}`. [US: F5.8-US1]
- **F5.8-R2** — `InMemory` (Agent) and `Postgres` (Ecto) implement the behaviour; `config :portal, :event_store`
  selects one; `adapter/0` resolves it. [US: F5.8-US1]
- **F5.8-R3** — the engine reads via `read_stream/1`+`replay/1` and, per command, runs `decide` → `append/2` →
  `evolve`. [US: F5.8-US4]
- **F5.8-R4** — `Portal` is the only web surface (`enroll/2`, `deliver_lesson/2`, `progress_of/1`, `courses_of/1`),
  delegating to `command/1`/`query/1`, the only `GenServer.call` callers. [US: F5.8-US2]
- **F5.8-R5** — `Portal.Error{code, message, field}` has a closed `code` union; `from/1` maps each reason with no
  catch-all. [US: F5.8-US3]
- **F5.8-R6** — a consumer sketch handles `{:ok, data}` and each `Portal.Error` code exhaustively. [US: F5.8-US5]
- **F5.8-R7** — swapping `:event_store` changes no caller; the facade is the sole boundary. [US: F5.8-US1, F5.8-US2]

## Agent stories

- **F5.8-AS1** [implements F5.8-US1] — Directive: define `Portal.EventStore` with the two `@callback`s, write the
  `InMemory` Agent adapter, stub the `Postgres` adapter signature, and add the config switch + `adapter/0`. Acceptance
  gate: `InMemory` satisfies the behaviour; `adapter/0` returns the configured module; swapping config changes no
  caller.
- **F5.8-AS2** [implements F5.8-US4] — Directive: make the engine read through `read_stream/1` + `replay/1` and run
  `decide` → `append/2` → `evolve` per command. Acceptance gate: an enroll appends before it folds; a restart replays
  to the same state through the port.
- **F5.8-AS3** [implements F5.8-US2] — Directive: write the `Portal` facade over `command/1`/`query/1`, running the F5.4
  contract guard first. Acceptance gate: the facade is the only `GenServer.call` path; no caller names the engine or
  store.
- **F5.8-AS4** [implements F5.8-US3, F5.8-US5] — Directive: define `Portal.Error` with the closed `code` union and
  `from/1` (no catch-all); write the exhaustive consumer sketch. Acceptance gate: every reason maps; no catch-all; the
  sketch handles `{:ok, data}` and each code.

## Execution plan — first two stories

1. **F5.8-AS1 — the driven port and adapters.** Create `lib/portal/event_store.ex` with `@callback append(stream,
   events)` and `@callback read_stream(stream)`. Create `lib/portal/event_store/in_memory.ex` as an `Agent` keyed by
   stream. Add `def adapter, do: Application.fetch_env!(:portal, :event_store)`. Set `config/test.exs` to `InMemory`
   and `config/config.exs` to `Postgres`. Stub `Postgres` to satisfy the behaviour (F6.3 fills the body).
2. **F5.8-AS2 — the engine through the port.** In `lib/portal/engine.ex`, `init/1` calls `EventStore.adapter().
   read_stream(stream)` then `Core.replay/1`. The command handler runs the contract, `decide`, then `append/2` through
   the adapter, then `evolve`. Confirm append-before-evolve and a clean replay on restart.

## Comprehensive implementation prompt

```text
You are drawing the boundaries of the Portal engine (rung F5.8). Read specs/pragmatic/f5.8.md, the engine spec
(f5.6.md), and the pure core (Portal.Engine.Core: decide/2 -> [event], evolve/2, replay/1). The command path is
decide -> append -> evolve; decide returns events only and the F5.4 contract guards rejection at the boundary.

Build the seams:

1) Driven port (T1 -> R1). Create lib/portal/event_store.ex defining a behaviour:
   @callback append(stream :: String.t(), events :: [struct]) :: :ok | {:error, term}
   @callback read_stream(stream :: String.t()) :: {:ok, [struct]} | {:error, term}

2) Adapters + config (T2 -> R2). Create lib/portal/event_store/in_memory.ex as an Agent keyed by stream (for dev and
   tests) and lib/portal/event_store/postgres.ex as an Ecto-backed adapter satisfying the behaviour (leave the body
   for F6.3; keep the signatures). Add `def adapter, do: Application.fetch_env!(:portal, :event_store)`. Set
   config/test.exs to Portal.EventStore.InMemory and config/config.exs to Portal.EventStore.Postgres.

3) Engine uses the port (T3 -> R3). In lib/portal/engine.ex, init/1 calls adapter().read_stream(stream) and folds with
   Core.replay/1. The command handler runs the F5.4 contract guard; on a valid command it calls decide, then
   append/2 through the adapter, then evolve — appending events before folding them in.

4) Driving port (T4 -> R4). Write lib/portal.ex as the facade: enroll/2, deliver_lesson/2, progress_of/1,
   courses_of/1, returning :ok | {:ok, data} | {:error, %Portal.Error{}}. It delegates to Portal.Engine.command/1 and
   query/1 — the only callers of GenServer.call. No controller or LiveView may name the engine or the store.

5) Error contract (T5 -> R5). Create lib/portal/error.ex:
   defstruct [:code, :message, :field]
   @type code :: :already_enrolled | :course_not_found | :lesson_locked | :invalid_progress
   from/1 maps each internal reason to a %Portal.Error{} with NO catch-all clause, so an unmapped reason fails to
   match (surfacing a bug rather than leaking an untyped error).

6) Consumer sketch (T6 -> R6). Write a small exhaustive sketch (a function or doc example) that handles {:ok, data}
   and one branch per Portal.Error code, showing the closed set is consumed exhaustively (F6's LiveView adopts it).

7) Verify (T7 -> R7). Confirm: swapping :event_store between adapters changes no caller; Portal is the only web surface
   and the only GenServer.call path; from/1 has no catch-all; an enroll persists through the port and a restart
   replays to the same state.

Constraints: the core names only the behaviour, never a concrete adapter; the web names only Portal; expected failures
are tagged tuples mapped to %Portal.Error{}; branded Snowflake IDs and pure functions throughout. Out of scope: the
supervision tree and LiveView (F5.9), the Ecto schema/migration (F6.3), and the HTTP endpoint (F6).
```
