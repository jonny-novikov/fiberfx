# F5.8 · agent brief (llms)

> Implementation brief for the boundary-drawing rung: the driven `EventStore` port and adapters, the `Portal` facade
> (the rename of the engine's `dispatch/1`/`query/2`), and the extended closed `Portal.Error`. References, traced
> requirements, the execution topology, and a paste-ready prompt. Pairs with the spec [`f5.8.md`](f5.8.md) and the
> stories [`f5.8.stories.md`](f5.8.stories.md).

## References

- Elixir behaviours / `@callback` — <https://hexdocs.pm/elixir/typespecs.html#behaviours>. Defines the driven port as
  a contract adapters implement.
- Ports & Adapters (hexagonal architecture), Alistair Cockburn — <https://alistair.cockburn.us/hexagonal-architecture/>.
  The driven/driving distinction this rung realizes.
- `Agent` — <https://hexdocs.pm/elixir/Agent.html>. The in-memory adapter's process.
- `Ecto.Adapters.SQL.Sandbox` — <https://hexdocs.pm/ecto_sql/Ecto.Adapters.SQL.Sandbox.html>. Isolates the Postgres
  adapter's tests (F6.3 supplies the schema and migration; this rung leaves the body a stub).
- As-built surface this rung changes — the brief renames and extends, it does not create from nothing:
  - `lib/portal/engine.ex` — the F5.6 boundary GenServer. As-built it exposes `dispatch/1` (a map command, `is_map`)
    and `query/2` (`query(name, arg)`); the command callback runs `Core.authorize → decide → Enum.reduce(evolve) →
    Portal.EventLog.append → project`. F5.8 renames the pair to `command/1`/`query/1` and reroutes the read+append
    through the port (see Requirements R3/R4).
  - `lib/portal/event_log.ex` — the in-memory log F5.8 supersedes. Its `all/0`/`append/1` (total, in-process,
    returns `:ok`) are generalized to `read_stream/1`/`append/2` behind the `Portal.EventStore` port (its own
    moduledoc states the swap).
  - `lib/portal/error.ex` — the F5.4 struct. As-built: `@enforce_keys [:code, :message]`, `defstruct [:code,
    :message]`, a two-code union (`:course_not_found | :already_enrolled`), and `new/1` only. F5.8 extends it (adds
    `:field`, completes the union, adds `from/1`).
  - `lib/portal.ex` — the reserved facade module. As-built: a moduledoc-only stub with no functions; its doc reads
    "By F5.8 this module becomes the public facade." F5.8 populates it.
  - `lib/portal_web/router.ex` — the as-built Plug router that today names `Portal.Engine.dispatch/1` and
    `Portal.Engine.query/2` directly; F5.8 repoints it onto the facade so the web no longer names the engine.
- Upstream: the pure core [`f5.5.md`](f5.5.md), the engine [`f5.6.md`](f5.6.md), the F6.3 persistence chapter
  [`../phoenix/phoenix.md`](../phoenix/phoenix.md), and the pattern [`decider-pattern.md`](decider-pattern.md).

## Requirements

Every Requirement names whether the file is **created** (genuinely new) or **changed/extended/populated** (the
as-built file already exists), so the implementer renames and extends rather than re-deriving an extant surface.

- **F5.8-R1** (create `lib/portal/event_store.ex`) — `Portal.EventStore` is a behaviour with two `@callback`s,
  `append(stream :: String.t(), events :: [struct]) :: :ok | {:error, term}` and `read_stream(stream :: String.t())
  :: {:ok, [struct]} | {:error, term}`. These **generalize** the as-built `Portal.EventLog.append/1` (arity 1, total,
  `:ok`) and `all/0` (arity 0): a stream key is added, and `append/2` gains a **fallible** `{:error, term}` return the
  in-process `EventLog` did not have. [US: F5.8-US1]
- **F5.8-R2** (create `event_store/in_memory.ex` and `event_store/postgres.ex`) — `InMemory` (an `Agent` keyed by
  stream) and `Postgres` (Ecto) implement the behaviour; `config :portal, :event_store` selects one and
  `Portal.EventStore.adapter/0` resolves it via `Application.fetch_env!(:portal, :event_store)`. `Postgres` is a
  signature-only stub at this rung — F6.3 fills the body, schema, and migration. [US: F5.8-US1]
- **F5.8-R3** (change `lib/portal/engine.ex`) — `init/1` reads through `read_stream/1` then `Core.replay/1`; per
  command the handler runs the F5.4 contract guard, then `decide → append/2 (port) → evolve`. This **inverts** the
  as-built order (`decide → evolve → EventLog.append`, engine.ex:75–77): the as-built fold preceded the in-process,
  total append (`EventLog.append/1` returns `:ok` unconditionally, event_log.ex:35, so it cannot fail after the fold);
  F5.8 appends first because `EventStore.append/2` is **fallible**, so a failed append aborts the command and leaves
  state unchanged (the fold never leads durable storage). [US: F5.8-US4]
- **F5.8-R4** (populate `lib/portal.ex`; change `lib/portal/engine.ex`; repoint `lib/portal_web/router.ex`) —
  `Portal` is the only web surface (`enroll/2`, `deliver_lesson/2`, `progress_of/1`, `courses_of/1`), running the F5.4
  contract guard then delegating to the engine wrappers `command/1`/`query/1` — the **rename** of the as-built
  `dispatch/1`/`query/2`, and the only `GenServer.call` callers. The as-built router (which names
  `Portal.Engine.dispatch/1`/`query/2` directly) is repointed onto the facade so nothing above the boundary names the
  engine. The facade returns `:ok | {:ok, data} | {:error, %Portal.Error{}}`. [US: F5.8-US2]
- **F5.8-R4a** (retained, no change beyond the rename) — the dual-write `Portal.Store` `%Enrollment{}` projection
  (F5.6-INV5) is **kept** and transitional: the held fold stays the source of truth, and the command path still mints
  an `ENR` id + `Portal.Store.put` on enroll and `reproject_enrollments/1` at `init`, so the F5.3/F5.4 web read stays
  byte-for-byte. `courses_of/1` reads the Store `%Enrollment{}` rows (web shape); the core
  `query({:enrollments, user_id})` returns `[course_id]` (the fold read) — a deliberate two-layer naming, unified at
  F6.4 (contexts). Do not remove the projection; F6.3/F6.4 supersede it. `progress_of/1` reads
  `enrollment.progress`, which is structurally `0` for **two independent reasons** — no progress-advancing event
  exists (the core has no `progress` field; `LessonDelivered` folds into a `delivered` MapSet, core.ex:64–68) **and**
  the projection hardcodes `progress: 0` at every mint (engine.ex:130) and reproject (engine.ex:171) — so it is an
  honest read of `0`, not a calculation (a progress-advancing command is an F6 concern). [US: F5.8-US2]
- **F5.8-R5** (extend `lib/portal/error.ex`) — extend the F5.4 struct: add an optional `:field` to the as-built
  `defstruct [:code, :message]` (error.ex:10; `@enforce_keys [:code, :message]` at :9), complete the two-code union
  (error.ex:11) to `@type code :: :already_enrolled | :course_not_found | :lesson_locked | :invalid_progress`, and add
  `from/1` mapping each reason with **no catch-all**. The four codes are the final closed set, but only
  `:course_not_found` and `:already_enrolled` have producers today (from `Core.authorize/2`, core.ex:114–122,
  enroll-only); `:lesson_locked` and `:invalid_progress` are **reserved** (no deliver-lesson gate or progress
  validation is built — `Core.authorize` returns `:ok` for `:deliver_lesson`, core.ex:124) — `from/1` still maps all
  four so a later producer needs no `from/1` change. [US: F5.8-US3]
- **F5.8-R6** (create, in the facade or a doc example) — a consumer sketch handles `{:ok, data}` and each
  `Portal.Error` code exhaustively. [US: F5.8-US5]
- **F5.8-R7** — swapping `:event_store` between adapters changes no caller; the facade is the sole boundary. [US:
  F5.8-US1, F5.8-US2]

## Agent stories

- **F5.8-AS1** [implements F5.8-US1] — Directive: define `Portal.EventStore` with the two `@callback`s (generalizing
  the as-built `EventLog.append/1`/`all/0` with a stream key and a fallible `append/2` return), write the `InMemory`
  Agent adapter, stub the `Postgres` adapter signature, and add the config switch + `adapter/0`. Acceptance gate:
  `InMemory` satisfies the behaviour; `adapter/0` returns the configured module; swapping config changes no caller.
- **F5.8-AS2** [implements F5.8-US4] — Directive: change the engine to read through `read_stream/1` + `replay/1` and
  run `decide → append/2 → evolve` per command — inverting the as-built `decide → evolve → EventLog.append` order so a
  fallible append aborts before any fold. Acceptance gate: an enroll appends before it folds; a failed append leaves
  state unchanged; an engine-process restart replays to the same state through the port (the supervised adapter
  outlives the engine — full-VM durability is the F6.3 Postgres concern).
- **F5.8-AS3** [implements F5.8-US2] — Directive: populate the reserved `lib/portal.ex` stub into the `Portal` facade
  over `command/1`/`query/1` (the rename of the as-built `dispatch/1`/`query/2`), running the F5.4 contract guard
  first, and repoint `lib/portal_web/router.ex` onto it; retain the dual-write `Portal.Store` projection unchanged.
  Acceptance gate: the facade is the only `GenServer.call` path; no caller names the engine or store; `courses_of/1`
  still returns the byte-for-byte `%Enrollment{}` rows.
- **F5.8-AS4** [implements F5.8-US3, F5.8-US5] — Directive: extend `Portal.Error` (add `:field`, complete the closed
  `code` union, add `from/1` with no catch-all) and write the exhaustive consumer sketch. Acceptance gate: every
  reason maps; no catch-all; the sketch handles `{:ok, data}` and each of the four codes (two with producers today,
  two reserved).

## Execution plan — first two stories

1. **F5.8-AS1 — the driven port and adapters (new files).** Create `lib/portal/event_store.ex` with `@callback
   append(stream, events)` and `@callback read_stream(stream)` — the generalization of the as-built
   `EventLog.append/1`/`all/0`. Create `lib/portal/event_store/in_memory.ex` as an `Agent` keyed by stream. Add `def
   adapter, do: Application.fetch_env!(:portal, :event_store)`. Set `config/test.exs` to `InMemory` and
   `config/config.exs` to `Postgres`. Stub `Postgres` to satisfy the behaviour (F6.3 fills the body).
2. **F5.8-AS2 — the engine through the port (change `engine.ex`).** In `lib/portal/engine.ex`, `init/1` calls
   `EventStore.adapter().read_stream(stream)` then `Core.replay/1`. The command handler runs the contract, `decide`,
   then `append/2` through the adapter, then `evolve` — the **inverted** order (the as-built path folds before the
   in-process `EventLog.append`; here the fallible port append precedes the fold). Confirm append-before-evolve, that
   a failed append leaves state unchanged, and a clean replay on restart.

## Comprehensive implementation prompt

```text
Task: draw the boundaries of the Portal engine (rung F5.8). This rung mostly RENAMES and EXTENDS an as-built engine —
create only the three new event-store files; the rest already exist. Read specs/pragmatic/f5.8.md, the engine spec
(f5.6.md), and the pure core (Portal.Engine.Core: decide/2 -> [event], evolve/2, replay/1).

As-built surface to cite (do not re-derive; verify before changing):
- lib/portal/engine.ex — the F5.6 boundary GenServer. Today it exposes dispatch/1 (a map command, is_map) and
  query/2 (name, arg); the command callback runs Core.authorize -> decide -> Enum.reduce(evolve) ->
  Portal.EventLog.append -> project (evolve BEFORE append). It also dual-writes a Portal.Store %Enrollment{} row on
  enroll and reprojects it at init (reproject_enrollments/1) so the F5.3/F5.4 web read is byte-for-byte.
- lib/portal/event_log.ex — the in-memory log. all/0 (arity 0) and append/1 (arity 1, in-process, total, returns :ok).
- lib/portal/error.ex — @enforce_keys [:code, :message], defstruct [:code, :message], @type code ::
  :course_not_found | :already_enrolled, and new/1 only (no :field, no from/1).
- lib/portal.ex — a moduledoc-only stub, no functions (its doc reserves it as the F5.8 facade).
- lib/portal_web/router.ex — names Portal.Engine.dispatch/1 and Portal.Engine.query/2 directly.

Build the seams:

1) Driven port — CREATE lib/portal/event_store.ex (T1 -> R1), a behaviour:
   @callback append(stream :: String.t(), events :: [struct]) :: :ok | {:error, term}
   @callback read_stream(stream :: String.t()) :: {:ok, [struct]} | {:error, term}
   These generalize EventLog.append/1 and all/0: a stream key is added, and append/2 gains a fallible {:error, term}
   return the in-process log did not have. The port supersedes EventLog's surface (its moduledoc states the swap).

2) Adapters + config — CREATE lib/portal/event_store/in_memory.ex (T2 -> R2) as an Agent keyed by stream (dev/tests)
   and CREATE lib/portal/event_store/postgres.ex as an Ecto-backed adapter satisfying the behaviour (leave the body
   for F6.3; keep the signatures). Add `def adapter, do: Application.fetch_env!(:portal, :event_store)`. Set
   config/test.exs to Portal.EventStore.InMemory and config/config.exs to Portal.EventStore.Postgres.

3) Engine uses the port — CHANGE lib/portal/engine.ex (T3 -> R3). init/1 calls adapter().read_stream(stream) and folds
   with Core.replay/1. The command handler runs the F5.4 contract guard; on a valid command it calls decide, then
   append/2 through the adapter, then evolve. This INVERTS the as-built order (decide -> evolve -> EventLog.append):
   append now precedes the fold because EventStore.append/2 is fallible, so a failed append must abort the command and
   leave state unchanged (the fold never leads durable storage). RETAIN the dual-write Portal.Store projection and
   reproject_enrollments/1 unchanged — the F5.3/F5.4 web read stays byte-for-byte (superseded only at F6.3/F6.4).

4) Driving port — POPULATE lib/portal.ex (T4 -> R4) into the facade: enroll/2, deliver_lesson/2, progress_of/1,
   courses_of/1, returning :ok | {:ok, data} | {:error, %Portal.Error{}}. It runs the F5.4 contract guard, then
   delegates to the engine wrappers command/1 and query/1 — the RENAME of the as-built dispatch/1 and query/2, and the
   only callers of GenServer.call. RENAME those two engine functions and REPOINT lib/portal_web/router.ex onto the
   facade, so no controller or web module names the engine or the store. Two-layer naming is deliberate: courses_of/1
   reads the Store %Enrollment{} rows (web shape) while Core.query({:enrollments, user_id}) returns [course_id] (the
   fold read); unification is deferred to F6.4.

5) Error contract — EXTEND lib/portal/error.ex (T5 -> R5). Add an optional :field to the struct, complete the union to
   @type code :: :already_enrolled | :course_not_found | :lesson_locked | :invalid_progress, and add from/1 mapping

   The four codes are the final closed set, but only :course_not_found and :already_enrolled have producers today
   (from authorize, enroll-only); :lesson_locked and :invalid_progress are RESERVED — no deliver-lesson gate or
   progress validation is built. Do not invent producers for the reserved codes; from/1 maps all four anyway so a
   later producer needs no from/1 change.

6) Consumer sketch (T6 -> R6). Write a small exhaustive sketch (a function or doc example) that handles {:ok, data}

7) Verify (T7 -> R7). Confirm: swapping :event_store between adapters changes no caller; Portal is the only web surface
   and the only GenServer.call path; from/1 has no catch-all; an enroll persists through the port, a failed append
   leaves state unchanged, and an engine restart replays to the same state.

Constraints: the core names only the behaviour, never a concrete adapter; the web names only Portal; expected failures

supervision tree and LiveView (F5.9), the Ecto schema/migration (F6.3), and the HTTP endpoint (F6).

Prose discipline (carry into any code comment, doc, or sub-prompt this work emits): impersonal and structural — no
```
