## Requirements

- **F5.4-R1** — `Portal.Learning.enroll/2` runs a `with` chain that checks (a) `user_id` and `course_id` are well-formed
  branded ids of the right namespace — `Portal.ID.valid?/1` (full 14-char format) **and** `Portal.ID.namespace/1` equal
  to `"USR"` / `"CRS"` (so a bare `"USR1"`, which passes `namespace/1` but fails `valid?/1`, is rejected), (b) the
  course exists (`Portal.Catalog.course/1`), and (c) the learner is not already enrolled — short-circuiting to
  `{:error, %Portal.Error{code: :course_not_found}}` or `{:error, %Portal.Error{code: :already_enrolled}}`. [US:
  F5.4-US1, F5.4-US2]
- **F5.4-R2** — on success `enroll/2` returns `{:ok, %Enrollment{progress: 0}}` and stores it; the success branch is
  the only path that mints and stores. [US: F5.4-US5]
- **F5.4-R3** — `progress` is typed `0..100` and treated as impossible to violate through the public API. [US:
  F5.4-US4]
- **F5.4-R4** — expected failures are `{:error, %Portal.Error{code: code}}` with `code` in a closed set
  (`:course_not_found`, `:already_enrolled`); an out-of-range `progress` is a `raise`/failed match, never a tuple. [US:
  F5.4-US4]
- **F5.4-R5** — the router maps each expected `%Portal.Error{}` code to `422` and `{:ok, _}` to `201`. [US: F5.4-US2,
  F5.4-US4]
- **F5.4-R6** — preconditions run before any id mint, struct build, or store write; a rejected enroll writes nothing.
  [US: F5.4-US1, F5.4-US3]
- **F5.4-R7** — `StreamData` property tests cover the postcondition (success ⇒ `progress: 0`, stored) and the invariant
  (`0 ≤ progress ≤ 100`); example tests cover each expected error. [US: F5.4-US5]
- **F5.4-R8** — verification: malformed/nonexistent course → `422 :course_not_found`, nothing stored; duplicate →
  `422 :already_enrolled`, one enrollment; valid → `201`, stored `%Enrollment{progress: 0}`; properties pass. [US: all]

## Execution topology

**Runtime topology (the contract is a parse step before the F5.3 effect).**

```text
POST /enroll ─▶ Portal.Engine.dispatch(%{type: :enroll, user_id, course_id})
                 ─▶ Portal.Learning.enroll/2
                      with :ok            <- check_ref(user_id, "USR"),      # parse the references
                           :ok            <- check_ref(course_id, "CRS"),
                           {:ok, _course} <- Portal.Catalog.course(course_id),  # course exists?
                           :not_enrolled  <- not_already(user_id, course_id) do # not a repeat?
                        # success path — the ONLY path that mints + stores:
                        Portal.ID.new("ENR") ▶ %Enrollment{progress: 0} ▶ Portal.Store.put/1 ▶ {:ok, enrollment}
                      else
                        {:error, %Portal.Error{code: :course_not_found}} = e -> e   # malformed/unknown ref
                        {:error, %Portal.Error{code: :already_enrolled}} = e -> e   # a repeat
                      end
                 ◀─ {:ok, e} ─▶ 201   |   {:error, %Portal.Error{}} ─▶ 422

Impossible state (progress ∉ 0..100): failed match / raise ─▶ crash (a bug), under supervision — never a tuple.
```

**Task topology (build order; each step keeps the suite green).**

```text
T1 precondition with-chain: parse USR/CRS refs, course exists, not-already   [R1, R6]
     └─▶ T2 typed expected errors (:course_not_found, :already_enrolled)      [R4]
           └─▶ T3 success path mints + stores ONLY after checks pass          [R2]
                 └─▶ T4 progress 0..100 invariant + impossible-state crash    [R3, R4]
                       └─▶ T5 router maps reasons → 422, ok → 201             [R5]
                             └─▶ T6 StreamData properties + example tests     [R7]
                                   └─▶ T7 verify (reject paths, dup, valid)   [R8]
```

**Touched files.** `lib/portal/learning.ex` (the contract), `lib/portal_web/router.ex` (status mapping),
`test/portal/enroll_contract_test.exs` (properties + examples).

## Agent stories

### F5.4-AS1 — Parse the preconditions [implements F5.4-US1, F5.4-US3]
Directive: add the `with` chain to `enroll/2` that parses the `USR`/`CRS` references, confirms the course exists, and
confirms not-already-enrolled, before any effect (T1, T2); ensure rejection writes nothing (T1 ordering).
Acceptance gate: a malformed or nonexistent course → `{:error, :course_not_found}` with no store write; the checks run
before any mint/build/store.

### F5.4-AS2 — Guarantee and map the result [implements F5.4-US2, F5.4-US4]
Directive: make the success branch the only mint+store path returning `{:ok, %Enrollment{progress: 0}}` (T3); type
`progress` `0..100` and crash on the impossible (T4); map reasons → `422`, ok → `201` (T5).
Acceptance gate: duplicate → `422 :already_enrolled` with one enrollment; valid → `201`, stored; an out-of-range
`progress` crashes rather than returning a tuple.

### F5.4-AS3 — Prove the contract [implements F5.4-US5]
Directive: write `StreamData` properties for the postcondition and the `0..100` invariant, plus example tests for each
expected error (T6).
Acceptance gate: properties pass across generated inputs; each expected error example returns its documented reason.

## Execution plan — first two stories

*story → agent story → tasks → files → command → gate.*

### Plan A — F5.4-US1 (reject a bad enroll at the door) · via F5.4-AS1
1. Tasks T1 → T2.
2. Files: `lib/portal/learning.ex`.
3. Command: `mix test test/portal/enroll_contract_test.exs:1` (the rejection examples), plus a manual
   `curl -i -X POST "localhost:4000/enroll?user=USR1&course=CRSbogus"` (both ids malformed → rejected by `valid?/1`).
4. Gate (closes US1): a malformed id or nonexistent course → `422` `%Portal.Error{code: :course_not_found}`; the store
   is unchanged. Requirements: F5.4-R1, F5.4-R6.

### Plan B — F5.4-US2 (no double enrollment) · via F5.4-AS2
1. Tasks T3 → T5 (after AS1).
2. Files: `lib/portal/learning.ex`, `lib/portal_web/router.ex`.
3. Command: enroll the same pair twice via `curl`, then count enrollments in the store.
4. Gate (closes US2): second enroll → `422 :already_enrolled`; exactly one enrollment exists. Requirements: F5.4-R1,
   F5.4-R5.

Efficiency check: the contract is one `with` chain plus a status map in two files; the reasons are a fixed closed set,
the success path is the single mint+store branch, and the impossible case is a crash — no decision is left open.
