# F5.8 · user stories

> Who the boundaries serve, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f5.8.md`](f5.8.md).

## F5.8-US1 — Storage I can swap without touching the core

As an **architect**, I want the core to depend only on a storage port, so that I can run in-memory in tests and
Postgres in production without changing domain code.

Acceptance criteria
- Given the core, when it persists or loads events, then it calls only the `Portal.EventStore` behaviour.
- Given the config, when `:event_store` is set, then `Portal.EventStore.adapter/0` resolves the chosen adapter.
- Given a swap from `InMemory` to `Postgres`, when callers run, then none of them change.

INVEST — independent; testable by substituting adapters; encodes F5.8-INV1, INV4.
Priority: must · Size: 3 · Implements deliverables: F5.8-D1, F5.8-D2.

## F5.8-US2 — One façade for every caller

As a **developer**, I want a single `Portal` facade as the only surface the web calls, so that controllers and
LiveViews never reach into the engine or storage.

Acceptance criteria
- Given the web, when it acts, then it calls only `Portal` (`enroll/2`, `deliver_lesson/2`, `progress_of/1`,
  `courses_of/1`).
- Given the engine, when a call is made, then only `command/1` and `query/1` call `GenServer.call`.
- Given any module above the boundary, when inspected, then it names no engine, repo, or store.

INVEST — independent; testable by the call graph; encodes F5.8-INV2.
Priority: must · Size: 2 · Implements deliverables: F5.8-D4, F5.8-D6.

## F5.8-US3 — Failures that cross as typed errors

As a **developer**, I want failures to cross the boundary as a closed `Portal.Error`, so that the UI handles a finite,
known set and a new failure cannot leak untyped.

Acceptance criteria
- Given an internal reason, when it crosses the boundary, then it is a `%Portal.Error{}` whose `code` is in the closed
  union.
- Given `from/1`, when inspected, then it has no catch-all clause.
- Given a new internal reason, when added, then it must be mapped explicitly or it fails to match.

INVEST — independent; testable per reason and by the absence of a catch-all; encodes F5.8-INV3, INV6.
Priority: must · Size: 2 · Implements deliverables: F5.8-D5.

## F5.8-US4 — Durable, swappable storage that survives a restart

As an **operator**, I want events appended before state evolves, so that a crash and restart replay to exactly the same
state regardless of adapter.

Acceptance criteria
- Given a command, when it succeeds, then its events are appended through the port before they are folded in.
- Given an engine restart, when it boots, then `read_stream/1` + `replay/1` reproduce the pre-crash state.
- Given an adapter swap, when the system runs, then durability behaves the same through the port.

INVEST — independent; testable by restart and by adapter swap; encodes F5.8-INV5.
Priority: must · Size: 3 · Implements deliverables: F5.8-D3.

## F5.8-US5 — A UI that handles every outcome

As a **developer** (building the F6 UI), I want a finite outcome set to consume, so that handling is exhaustive and a
new code forces a new branch.

Acceptance criteria
- Given the facade's result, when consumed, then the sketch handles `{:ok, data}` and one branch per `Portal.Error`
  code.
- Given a new error code, when added to the union, then the consumer gains a branch (no silent fall-through).

INVEST — independent; testable by the sketch's exhaustiveness; encodes F5.8-INV6.
Priority: should · Size: 1 · Implements deliverables: F5.8-D7.
