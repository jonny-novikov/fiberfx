# F5.7 · Pragmatic testing

> Build a test suite shaped like the engine: a wide base of example tests on the pure core (`decide`, `evolve`,
> `replay`) with no processes and no mocks, `StreamData` properties for the invariants, the F5.4 contract turned into
> assertions, doctests that keep the documentation true, and exactly one thin process test — a single command-then-query
> through the GenServer — at the tip.

## Goal

After F5.7, `mix test` is fast, deterministic, and weighted to the pure base. Example tests pin `decide`/`evolve`/
`replay`; `StreamData` properties pin the invariants (`replay == fold`, `0..100`); contract tests turn the F5.4
preconditions and postcondition into assertions; doctests keep public examples honest; and one process test exercises a
command then a query through `Portal.Engine`. The pure layers need no process and no mocks, so the suite runs in
milliseconds and gives constant feedback.

## Rationale (5W)

- **Why** — The engine's value is concentrated in pure functions, so the tests should be too: a fast, deterministic
  base that runs on every save catches regressions cheaply, while a single process test is enough to prove the wiring.
  Tests weighted to the base stay fast; tests weighted to processes get slow and flaky.
- **What** — example tests of the core, property tests of the invariants, contract tests of F5.4, doctests of the
  public examples, and one command→query process smoke test.
- **Who** — Developers (constant, trustworthy feedback and a suite that documents the engine's behaviour).
- **When** — the seventh rung; it pins the F5.5 core, the F5.4 contract, and the F5.6 engine. It precedes F5.8, where
  the boundaries it tests against are formalized.
- **Where** — `test/portal/engine/core_test.exs` (example + property), `test/portal/enroll_contract_test.exs`
  (contract), doctests in the public `@doc` strings, and `test/portal/engine_test.exs` (the one process test).

## Scope

- **In** — example tests of `decide`/`evolve`/`replay` with no process; `StreamData` property tests of the invariants;
  contract tests of the F5.4 preconditions (each closed reason) and the postcondition; doctests on the public
  functions; one command→query process test; the pyramid shape (most tests at the base) and the fast, deterministic,
  `async`-where-possible discipline.
- **Out** — the durable `EventStore` adapters and their integration tests (→ F5.8 / F6.3, with the Ecto sandbox); the
  LiveView test (→ F6); load/performance testing; mutation testing; mocking frameworks (the pure core needs none).

## Deliverables

- **F5.7-D1** — example tests of the pure core, no process and no mocks: `decide` emits the right events for valid
  commands, `evolve` folds one event correctly, and `replay` derives the expected state from a log.
- **F5.7-D2** — `StreamData` property tests: `replay(log)` equals incremental folding; `progress` stays within
  `0..100`; any monotonic invariant (a streak or progress never decreases on a forward event) holds over generated
  inputs.
- **F5.7-D3** — contract tests turning the F5.4 contract into assertions: each closed reason has a test (a malformed or
  nonexistent course → `:course_not_found`; a duplicate → `:already_enrolled`), and the postcondition (a valid enroll →
  a stored `%Enrollment{progress: 0}` / a `%LearnerEnrolled{}`) is asserted.
- **F5.7-D4** — doctests: runnable examples in the public `@doc` strings (`Portal.ID`, the contract, the core) that the
  suite runs, so a wrong example fails the build.
- **F5.7-D5** — exactly one process test: a single command-then-query through the live `Portal.Engine` GenServer — the
  only place in the suite a process is started.
- **F5.7-D6** — the pyramid shape: most tests at the pure base, the process test at the tip; the suite is fast,
  deterministic (no wall-clock sleeps, seeded generators), and `async: true` wherever there is no shared process state.
- **F5.7-D7** — verification: `mix test` is green and fast; the base holds the most tests; exactly one process test
  exists; properties pass; doctests pass.

## Invariants

- **F5.7-INV1 (weighted to the pure base)** — most tests exercise pure functions (`decide`/`evolve`/`replay`, the
  contract) with no process and no mocks; the process test is a single smoke test at the tip.
- **F5.7-INV2 (deterministic and fast)** — tests are deterministic (no real-clock sleeps, no network, seeded
  generators) and `async: true` wherever there is no shared process state.
- **F5.7-INV3 (invariants are properties)** — state invariants (`replay == fold`, `0..100`, monotonic progress) are
  `StreamData` properties, not example-only checks.
- **F5.7-INV4 (the contract is tested as a contract)** — every closed reason in the F5.4 set has a test, and the
  postcondition is asserted; an untested reason is a gap, not an option.
- **F5.7-INV5 (docs stay true)** — public-function examples are doctests run by the suite, so documentation cannot
  drift from behaviour.

## Definition of Done

- [ ] Example tests pin `decide`/`evolve`/`replay` with no process and no mocks (F5.7-INV1).
- [ ] `StreamData` properties pin `replay == fold` and the `0..100` invariant (F5.7-INV3).
- [ ] Each F5.4 closed reason has a contract test and the postcondition is asserted (F5.7-INV4).
- [ ] Doctests on the public functions run and pass (F5.7-INV5).
- [ ] Exactly one process test (command→query through `Portal.Engine`) exists (F5.7-D5).
- [ ] `mix test` is green, fast, and `async` where there is no shared process state (F5.7-INV2).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.7.stories.md`).
