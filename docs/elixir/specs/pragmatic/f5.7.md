# F5.7 ¬∑ Pragmatic testing

> Build a test suite shaped like the engine: a wide base of example tests on the pure core (`decide`, `evolve`,
> `replay`) with no processes and no mocks, `StreamData` properties for the invariants, the F5.4 contract turned into
> assertions, doctests that keep the documentation true, and exactly one thin process test ‚Äî a single command-then-query
> through the GenServer ‚Äî at the tip.

## Goal

After F5.7, `mix test` is fast, deterministic, and weighted to the pure base. Example tests pin `decide`/`evolve`/
`replay`; `StreamData` properties pin the invariants (`replay == fold`, `0..100`); contract tests turn the F5.4
preconditions and postcondition into assertions; doctests keep public examples honest; and one process test exercises a
command then a query through `Portal.Engine`. The pure layers need no process and no mocks, so the suite runs in
milliseconds and gives constant feedback.

## Rationale (5W)

- **Why** ‚Äî The engine's value is concentrated in pure functions, so the tests should be too: a fast, deterministic
  base that runs on every save catches regressions cheaply, while a single process test is enough to prove the wiring.
  Tests weighted to the base stay fast; tests weighted to processes get slow and flaky.
- **What** ‚Äî example tests of the core, property tests of the invariants, contract tests of F5.4, doctests of the
  public examples, and one command‚Üíquery process smoke test.
- **Who** ‚Äî Developers (constant, trustworthy feedback and a suite that documents the engine's behaviour).
- **When** ‚Äî the seventh rung; it pins the F5.5 core, the F5.4 contract, and the F5.6 engine. It precedes F5.8, where
  the boundaries it tests against are formalized.
- **Where** ‚Äî `test/portal/engine/core_test.exs` (example + property), `test/portal/enroll_contract_test.exs`
  (contract), doctests in the public `@doc` strings, and `test/portal/engine_test.exs` (the one process test).

## Scope

- **In** ‚Äî example tests of `decide`/`evolve`/`replay` with no process; `StreamData` property tests of the invariants;
  contract tests of the F5.4 preconditions (each closed reason) and the postcondition; doctests on the public
  functions; one command‚Üíquery process test; the pyramid shape (most tests at the base) and the fast, deterministic,
  `async`-where-possible discipline.
- **Out** ‚Äî the durable `EventStore` adapters and their integration tests (‚Üí F5.8 / F6.3, with the Ecto sandbox); the
  LiveView test (‚Üí F6); load/performance testing; mutation testing; mocking frameworks (the pure core needs none).

## Deliverables

- **F5.7-D1** ‚Äî example tests of the pure core, no process and no mocks: `decide` emits the right events for valid
  commands, `evolve` folds one event correctly, and `replay` derives the expected state from a log.
- **F5.7-D2** ‚Äî `StreamData` property tests: `replay(log)` equals incremental folding; `progress` stays within
  `0..100`; and a set-monotonicity property ‚Äî a forward event never shrinks a user's enrolled/delivered MapSet ‚Äî holds
  over generated inputs. (The numeric-progress monotonic ‚Äî progress never decreasing ‚Äî is DEFERRED: no
  progress-advancing event exists yet to drive it; F5.8 / F6 adds it. The set-monotonicity property is the monotonic
  the current model actually exhibits.)
- **F5.7-D3** ‚Äî contract tests turning the F5.4 contract into assertions: each closed reason has a test (a malformed or
  nonexistent course ‚Üí `:course_not_found`; a duplicate ‚Üí `:already_enrolled`), and the postcondition (a valid enroll ‚Üí
  a stored `%Enrollment{progress: 0}` / a `%LearnerEnrolled{}`) is asserted.
- **F5.7-D4** ‚Äî doctests: runnable examples in the public `@doc` strings (`Portal.ID`, the contract, the core) that the
  suite runs, so a wrong example fails the build.
- **F5.7-D5** ‚Äî one command-then-query SMOKE test at the process tip, designated as such, through the live
  `Portal.Engine` GenServer: the thin tip of the pyramid. The F5.6 crash-recovery and behavioural process tests are
  RETAINED as the engine's Definition of Done (not deleted to hit a literal one-test count) ‚Äî "one smoke at the tip" is
  the pyramid's intent, not a process-test census.
- **F5.7-D6** ‚Äî the pyramid shape: most tests at the pure base, the process test at the tip; the suite is fast,
  deterministic (no wall-clock sleeps, seeded generators), and `async: true` wherever there is no shared process state.
- **F5.7-D7** ‚Äî verification: `mix test` is green and fast; the base holds the most tests; the process tier is a thin
  tip (one command ‚Üí query smoke test plus the retained F5.6 gates); properties pass; doctests pass.

## Invariants

- **F5.7-INV1 (weighted to the pure base)** ‚Äî most tests exercise pure functions (`decide`/`evolve`/`replay`, the
  contract) with no process and no mocks; the process tier is a thin tip ‚Äî one command ‚Üí query smoke test plus the
  retained F5.6 crash-recovery/behavioural gates ‚Äî never the bulk of the suite.
- **F5.7-INV2 (deterministic and fast)** ‚Äî tests are deterministic (no real-clock sleeps, no network, seeded
  generators) and `async: true` wherever there is no shared process state.
- **F5.7-INV3 (invariants are properties)** ‚Äî state invariants (`replay == fold`, `0..100`, and set-monotonicity ‚Äî a
  forward event never shrinks the fold's MapSets) are `StreamData` properties, not example-only checks. (The numeric
  monotonic ‚Äî progress never decreasing ‚Äî is deferred until a progress-advancing event exists; the set-monotonicity
  property meets INV3's monotonic intent meanwhile.)
- **F5.7-INV4 (the contract is tested as a contract)** ‚Äî every closed reason in the F5.4 set has a test, and the
  postcondition is asserted; an untested reason is a gap, not an option.
- **F5.7-INV5 (docs stay true)** ‚Äî public-function examples are doctests run by the suite, so documentation cannot
  drift from behaviour.

## Definition of Done

- [ ] Example tests pin `decide`/`evolve`/`replay` with no process and no mocks (F5.7-INV1).
- [ ] `StreamData` properties pin `replay == fold` and the `0..100` invariant (F5.7-INV3).
- [ ] Each F5.4 closed reason has a contract test and the postcondition is asserted (F5.7-INV4).
- [ ] Doctests on the public functions run and pass (F5.7-INV5).
- [ ] One command‚Üíquery SMOKE test (through `Portal.Engine`) marks the process tip; the F5.6 crash-recovery and
      behavioural process tests are retained (F5.7-D5).
- [ ] `mix test` is green, fast, and `async` where there is no shared process state (F5.7-INV2).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f5.7.stories.md`).
