# F5.1 · user stories

> Who needs the thin running Portal, what they need, and how we will know it works. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f5.1.md`](f5.1.md).

## F5.1-US1 — Serve the Portal over HTTP from day one

As an **operator**, I want a supervised app that answers real HTTP immediately, so that there is a running system to
deploy and extend from the first iteration.

Acceptance criteria
- Given the application, when it starts, then `Bandit` listens on port 4000 and the supervision tree is running.
- Given the running app, when I `POST /enroll`, then I receive `422` with `:not_implemented` (the stubbed boundary).
- Given the running app, when I request an unknown path, then I receive `404`.

INVEST — independent of any domain logic; small (wiring only); testable via boot and two requests.
Priority: must · Size: 3 · Implements deliverables: F5.1-D3, F5.1-D4, F5.1-D5.

## F5.1-US2 — A branded Snowflake id generator

As a **developer**, I want one id convention for every entity, so that records are time-ordered, self-describing, and
compact across the whole system.

Acceptance criteria
- Given a namespace, when I call `Portal.ID.new("ENR")`, then I get `"ENR"` followed by the Base62 of a fresh
  snowflake.
- Given a branded id, when I call `Portal.ID.snowflake/1`, then it round-trips to the original integer; `namespace/1`
  returns the prefix; `at/1` returns a UTC `DateTime`.
- Given the id, when I inspect it, then the integer snowflake is the canonical value and the branded string is its
  transport form.

INVEST — independent; small; testable as pure functions with no process.
Priority: must · Size: 2 · Implements deliverables: F5.1-D2.

## F5.1-US3 — A web layer built for replacement

As an **architect**, I want the web layer to be a thin, replaceable adapter, so that F6 can swap `Bandit` for Phoenix
without touching the domain.

Acceptance criteria
- Given a request, when the router handles it, then it only parses the request, calls the engine boundary, and formats
  a response — no business logic.
- Given the domain, when I inspect it, then it has no dependency on the web layer.
- Given a future swap to Phoenix, when the web layer is replaced, then the engine boundary and domain are unchanged.

INVEST — independent; testable by inspecting the router and dependencies; this is the seed of the master invariant.
Priority: must · Size: 2 · Implements deliverables: F5.1-D4, F5.1-D6.

## F5.1-US4 — A self-healing app

As an **operator**, I want the app to recover from a crash on its own, so that a transient fault does not take the
service down.

Acceptance criteria
- Given the running app, when the `Portal.Engine` process is killed, then the supervisor restarts it.
- Given the restart, when I issue a request afterwards, then it succeeds.

INVEST — small; testable by killing the process and re-requesting; relies on `:one_for_one`.
Priority: should · Size: 2 · Implements deliverables: F5.1-D5, F5.1-D7.

## F5.1-US5 — A development roadmap

As a **developer**, I want the path from this thin server to Phoenix written down, so that "start thin" is a deliberate
first step rather than throwaway work.

Acceptance criteria
- Given the roadmap, when I read it, then it states what exists now (a thin supervised server), what F6 replaces (the
  web layer), and what stays (the domain, ids, engine boundary).
- Given the roadmap, when planning F6, then it identifies the web layer as the replaceable part and the domain as
  stable.

INVEST — small; testable by inspecting the document; frames the value ladder.
Priority: could · Size: 1 · Implements deliverables: F5.1-D6.
