- Property-based testing with `StreamData` — pin `replay == incremental fold` and the invariants:

## Requirements

- **F5.5-R1** — a command path returns `:ok | {:error, reason}` (never data); a query path returns data (never a
  mutation); commands and queries are distinct. [US: F5.5-US1]
- **F5.5-R2** — `Portal.Learning.Events.LearnerEnrolled` (`user_id`, `course_id`, `at`) and `…LessonDelivered`
  (`user_id`, `lesson_id`, `at`) are past-tense structs with `@enforce_keys` and `@type t`. [US: F5.5-US2]
- **F5.5-R3** — `Portal.Engine.Core.decide/2 :: (state, command) -> [event]` is pure and returns events only; `{:enroll,
  u, c}` emits `[%LearnerEnrolled{}]` and `{:deliver_lesson, u, l}` emits `[%LessonDelivered{}]`. A separate
  `authorize/2` at the boundary runs the F5.4 contract's state/invariant checks and returns the closed `{:error,
  :already_enrolled | :course_not_found}` *before* `decide`. [US: F5.5-US2, F5.5-US3, F5.5-US5]
- **F5.5-R4** — `Portal.Engine.Core.evolve/2 :: (event, state) -> state` is pure, one clause per event type, folding
  exactly one event; `initial_state/0` returns the empty state. [US: F5.5-US3]
- **F5.5-R5** — `Portal.Engine.Core.replay/1 :: ([event]) -> state` is `Enum.reduce(events, initial_state(),
  &evolve/2)`. [US: F5.5-US4]
- **F5.5-R6** — the tuples are wired: commands `{:enroll, user_id, course_id}`, `{:deliver_lesson, user_id,
  lesson_id}`; query `{:enrollments, user_id}`. [US: F5.5-US1]
- **F5.5-R7** — example tests cover `decide`/`evolve`; a `StreamData` property confirms `replay(log)` equals
  incremental folding; example tests cover each expected error. [US: F5.5-US4, F5.5-US5]
- **F5.5-R8** — verification: a command never returns data; a query never mutates; `replay` reconstructs the folded
  state; events are immutable past-tense structs. [US: all]

## Execution topology

**Runtime topology (the pure core; no process yet — F5.6 hosts it).**

```text
COMMAND (write):  command tuple ─▶ Portal.Engine.Core.decide(state, command)
                                     ├─ success ─▶ [%LearnerEnrolled{} | %LessonDelivered{}]
                                     └─ expected failure ─▶ {:error, :already_enrolled | :course_not_found}
                  events ─▶ Enum.reduce(events, state, &evolve/2) ─▶ state'        # fold the new events in
                  result ─▶ :ok | {:error, reason}                                 # CQS: no data on the write path

QUERY (read):     {:enrollments, user_id} ─▶ read derived value from state ─▶ data # no mutation

REBUILD:          replay(log) = Enum.reduce(log, initial_state(), &evolve/2)       # state is the fold of the log

(Reference existence — :course_not_found — is resolved at the boundary via Portal.Catalog.course/1 BEFORE decide,
 keeping decide pure. The wall-clock `at` is supplied to decide by the boundary.)
```

**Task topology (build order; each step keeps the suite green).**

```text
T1 event structs (LearnerEnrolled, LessonDelivered) — past-tense, enforced keys   [R2]
     └─▶ T2 Portal.Engine.Core.evolve/2 + initial_state/0 (fold one event)        [R4]
           └─▶ T3 Portal.Engine.Core.replay/1 (reduce over the log)               [R5]
                 └─▶ T4 decide/2 (→ events only) + authorize/2 (contract → reason) [R3]
                       └─▶ T5 command/query tuples + CQS shapes                    [R1, R6]
                             └─▶ T6 example tests + replay property                [R7]
                                   └─▶ T7 verify (CQS, purity, rebuild)            [R8]
```

## Agent stories

### F5.5-AS1 — Events and the fold [implements F5.5-US2, F5.5-US3]
Directive: create the two past-tense event structs (T1); implement `evolve/2` (one clause per event) and
`initial_state/0` (T2); implement `replay/1` as `reduce` (T3).
Acceptance gate: each event is an immutable past-tense struct; `evolve` folds one event; `replay(log)` equals
incremental folding.

### F5.5-AS2 — Decide, purely [implements F5.5-US1, F5.5-US5]
Directive: implement `decide/2 :: (state, command) -> [event]` emitting events only, and a separate `authorize/2` at the
boundary running the contract's state/invariant checks and returning the closed `{:error, reason}` before `decide` (T4);
wire the command/query tuples and the CQS shapes (T5).
Acceptance gate: a valid enroll emits `[%LearnerEnrolled{}]`; a duplicate returns `{:error, :already_enrolled}` with no
event; commands return `:ok`/`{:error, _}` (never data); queries return data (never mutate); `decide` does no I/O.

### F5.5-AS3 — Prove the core [implements F5.5-US4]
Directive: write example tests for `decide`/`evolve` and a `StreamData` property for `replay == incremental fold`, plus
example tests per expected error (T6).
Acceptance gate: the property holds across generated logs; the examples pass.

## Execution plan — first two stories

*story → agent story → tasks → files → command → gate.*

### Plan A — F5.5-US3 (the engine is two pure functions) · via F5.5-AS1
1. Tasks T1 → T2 → T3.
2. Files: `lib/portal/learning/events/{learner_enrolled,lesson_delivered}.ex`, `lib/portal/engine/core.ex`.

4. Gate (closes US3): `evolve` folds one event; `replay(log)` reconstructs state; both functions are pure.
   Requirements: F5.5-R2, F5.5-R4, F5.5-R5.

### Plan B — F5.5-US1 (writes and reads are separate) · via F5.5-AS2
1. Tasks T4 → T5.
2. Files: `lib/portal/engine/core.ex`.
3. Command: `mix test test/portal/engine/core_test.exs` (decide + CQS cases).
4. Gate (closes US1): a command returns `:ok`/`{:error, _}` and never data; a query returns data and never mutates.
   Requirements: F5.5-R1, F5.5-R3, F5.5-R6.

Efficiency check: the core is one module plus two event structs behind one test file; the CQS shapes and the closed
reasons are fixed by the spec, and reference existence is resolved at the boundary so `decide` stays pure.

## Comprehensive implementation prompt

> Paste into a coding agent (e.g. Claude Code) at the root of the `portal` app, after F5.4. It executes F5.5-AS1..AS3
> in task-topology order and ends on the verification gates. Treat the Requirements and the Definition of Done in
> [`f5.5.md`](f5.5.md) as the acceptance checklist.

```text
You are implementing spec F5.5 (Commands, queries & events) in the `portal` app, on top of F5.4. Read
specs/pragmatic/f5.5.md, specs/pragmatic/f5.5.stories.md, and specs/design/f0.md first. Build a PURE functional
core — no processes, no I/O (F5.6 will host it). Keep decide/evolve/replay deterministic. CQS: commands return :ok |
{:error, reason} and never data; queries return data and never mutate. Reference existence (:course_not_found) is
resolved at the boundary before decide; the wall-clock `at` is supplied to decide.

Build in this order, keeping the suite green after each step:

1) Events (T1 → R2). Create past-tense structs with @enforce_keys and @type t:
   Portal.Learning.Events.LearnerEnrolled (user_id, course_id, at) and
   Portal.Learning.Events.LessonDelivered (user_id, lesson_id, at).

2) Fold (T2, T3 → R4, R5). In lib/portal/engine/core.ex add initial_state/0, evolve/2 :: (event, state) -> state with
   one clause per event type (fold exactly one event), and replay/1 :: ([event]) -> state as
   Enum.reduce(events, initial_state(), &evolve/2).

3) Decide (T4 → R3). Add decide/2 :: (state, command) -> [event], pure and events-only. For {:enroll, u, c}: emit
   [%LearnerEnrolled{user_id: u, course_id: c, at: at}]; for {:deliver_lesson, u, l}: emit [%LessonDelivered{...}]. Add a

   against state) and returns the closed {:error, :already_enrolled | :course_not_found} BEFORE decide is reached, so
   decide carries no error channel.

4) Tuples + CQS (T5 → R1, R6). Define the command tuples {:enroll, user_id, course_id} and {:deliver_lesson, user_id,
   lesson_id} and the query {:enrollments, user_id}. The boundary supplies the occurrence time `at` as the command's 4th
   element when calling decide ({:enroll, user_id, course_id, at}, {:deliver_lesson, user_id, lesson_id, at}), so the
   pure core holds no clock (mirrors f5.5.md D6). The command path turns a non-empty event list into :ok after the
   caller folds + records the events; the query path reads derived data from state.

5) Tests (T6 → R7). In test/portal/engine/core_test.exs add example tests for decide and evolve, a StreamData property
   that replay(log) equals folding the same events incrementally, and example tests for each expected error.

6) Verify (T7 → R8 + DoD). Confirm: a command never returns data; a query never mutates; replay reconstructs the
   folded state; events are immutable past-tense structs; decide does no I/O. Report each result against the F5.5
   Definition of Done.

Deliverables: the touched files listed in specs/pragmatic/f5.5.llms.md, with a pure, property-tested core, and every
F5.5 invariant exercised. Stop when the Definition of Done in specs/pragmatic/f5.5.md is fully checked.
```
