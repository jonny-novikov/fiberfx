# F5.6 · user stories

> Who needs the folded state to live somewhere, what they need, and how we will know it works. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f5.6.md`](f5.6.md).

## F5.6-US1 — State has a deliberate home

As a **developer**, I want a reasoned choice of where state lives, so that the engine's concurrency and consistency
behaviour is intentional, not accidental.

Acceptance criteria
- Given the options, when the choice is made, then a GenServer is selected with the reason (serialised command/query
  over one folded state).
- Given the alternatives, when documented, then the Agent and ETS trade-offs are recorded.
- Given the choice, when reviewed, then the throughput-ceiling trade-off is stated and accepted.

INVEST — independent; small; testable by inspecting the decision record; frames F5.6-INV6.
Priority: should · Size: 1 · Implements deliverables: F5.6-D1.

## F5.6-US2 — A command changes the live state

As a **developer**, I want a command to run through the held state, so that the change is applied once, consistently,
and kept between requests.

Acceptance criteria
- Given the running engine, when I issue a command, then it runs `decide`, folds the events with `evolve`, and keeps
  the new state.
- Given a successful command, when it returns, then it replies `:ok` (or `{:error, reason}`), never domain data.
- Given a later request, when it runs, then it sees the applied change.

INVEST — independent; testable through the live GenServer; encodes F5.6-INV1, INV2.
Priority: must · Size: 3 · Implements deliverables: F5.6-D2, F5.6-D3.

## F5.6-US3 — A query reads without changing anything

As a **developer**, I want queries to read the held state only, so that reading can never cause a change.

Acceptance criteria
- Given the running engine, when I issue a query, then it reads from the held state and returns data.
- Given a query, when it completes, then the state is unchanged.

INVEST — independent; small; testable by a query plus a state check; encodes F5.6-INV2.
Priority: must · Size: 2 · Implements deliverables: F5.6-D4.

## F5.6-US4 — The engine recovers from a crash

As an **operator**, I want the engine to rebuild its state on restart, so that a process crash never loses the truth.

Acceptance criteria
- Given a running engine with state, when the process is killed, then the supervisor restarts it.
- Given the restart, when `init/1` runs, then it re-folds the log to the same state.
- Given a query after recovery, when it runs, then it returns the same data as before the crash.

INVEST — valuable (robustness); testable by killing the process and re-querying; encodes F5.6-INV3.
Priority: must · Size: 3 · Implements deliverables: F5.6-D6, F5.6-D7.

## F5.6-US5 — The boundary stays the same

As a **developer**, I want the web to keep calling the same engine boundary, so that hosting the state in a process
changes nothing above it.

Acceptance criteria
- Given the boundary, when I inspect it, then `dispatch/1` and `query/2` wrap `GenServer.call`.
- Given the F5.3 web layer, when it runs, then it is unchanged and calls only the boundary.
- Given the callbacks, when inspected, then `handle_call` and the state shape are private and add no business logic.

INVEST — independent; testable by inspection and the unchanged web tests; encodes F5.6-INV4, INV5.
Priority: must · Size: 2 · Implements deliverables: F5.6-D5.
