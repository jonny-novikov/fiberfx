## Requirements

- **F5.3-R1** — `Portal.Engine.dispatch/1` handles `%{type: :enroll, user_id, course_id}` by delegating to
  `Portal.Learning.enroll/2`, returning `{:ok, %Enrollment{}} | {:error, reason}`. [US: F5.3-US1, F5.3-US2]
- **F5.3-R2** — `Portal.Engine.query/2` handles `(:lesson, id)` → `Portal.Catalog.lesson/1` and `(:courses_of,
  user_id)` → `Portal.Learning.courses_of/1`. [US: F5.3-US3]
- **F5.3-R3** — `POST /enroll` parses `user`/`course`, calls the boundary, and maps `{:ok, e}` → `201` (with `e.id`),
  `{:error, _}` → `422`; the context persists the enrollment via `Portal.Store.put/1`. [US: F5.3-US1, F5.3-US4]
- **F5.3-R4** — `GET /lessons/:id` maps `{:ok, lesson}` → `200`, `:error` → `404`; `GET /courses/:user_id` returns the
  enrollments as `200`. [US: F5.3-US3, F5.3-US4]
- **F5.3-R5** — responses are `Jason`-encoded in a consistent envelope (the created id, a lesson, a courses list).
  [US: F5.3-US4]
- **F5.3-R6** — minimal seed data (a `USR`, a `CRS`, an `LSN`) is loadable into the store so the slices round-trip
  locally. [US: F5.3-US2]
- **F5.3-R7** — an end-to-end test, no mocks, covers: enroll → `201` and retrievable; lesson → `200`; unknown lesson →
  `404`. [US: F5.3-US1, F5.3-US3]
- **F5.3-R8** — the tracer-bullet discipline is documented: every layer touched, none finished; grown one vertical
  slice at a time. [US: F5.3-US5]

## Execution topology

**Runtime topology (the F5.1 tree unchanged; the boundary now does real work).**

```text
Portal.Application (unchanged: Portal.Engine + Bandit)

ENROLL slice:
  POST /enroll ─▶ Portal.Engine.dispatch(%{type: :enroll, user_id, course_id})
                   ─▶ Portal.Learning.enroll/2 ─▶ Portal.ID.new("ENR") + %Enrollment{progress: 0} + Portal.Store.put/1
                   ◀─ {:ok, %Enrollment{}}  ─▶ 201 (id) | {:error, _} ─▶ 422

DELIVER-LESSON slice:
  GET /lessons/:id ─▶ Portal.Engine.query(:lesson, id) ─▶ Portal.Catalog.lesson/1 ─▶ {:ok, lesson} | :error
                   ◀─ 200 (lesson) | 404

READ courses:
  GET /courses/:user_id ─▶ Portal.Engine.query(:courses_of, user_id) ─▶ Portal.Learning.courses_of/1 ─▶ 200 (list)
```

The web maps results to statuses; the contexts and store do the work; no layer is mocked.

**Task topology (build order; each step leaves the app compiling and runnable).**

```text
T1 Portal.Engine.dispatch/1 → Learning.enroll/2 (enroll boundary real)   [R1]
     └─▶ T2 Portal.Engine.query/2 → Catalog.lesson/1, Learning.courses_of/1  [R2]
           └─▶ T3 web: POST /enroll mapping (201/422) + JSON               [R3, R5]
                 └─▶ T4 web: GET /lessons/:id (200/404) + courses 200      [R4, R5]
                       └─▶ T5 seed data (USR/CRS/LSN)                       [R6]
                             └─▶ T6 end-to-end test (no mocks)             [R7]
                                   └─▶ T7 tracer-bullet note               [R8]
```

**Touched files.** `lib/portal/engine.ex`, `lib/portal_web/router.ex`, `priv/seeds.exs` (or a `Portal.Seeds` module),
`test/portal_web/enroll_slice_test.exs`.

## Agent stories

### F5.3-AS1 — Make the boundary real [implements F5.3-US1, F5.3-US2]
Directive: implement `Portal.Engine.dispatch/1` (enroll → `Learning.enroll/2`) and `query/2` (lesson, courses_of →
the contexts) (T1, T2); add seed data (T5).
Acceptance gate: `POST /enroll` → `201` with a new id and the enrollment is retrievable; the boundary delegates to the
contexts (no `:not_implemented`).

### F5.3-AS2 — Wire and map the slices [implements F5.3-US3, F5.3-US4]
Directive: map `POST /enroll` to `201/422`, add `GET /lessons/:id` (`200/404`) and `GET /courses/:user_id` (`200`),
and `Jason`-encode the responses (T3, T4).
Acceptance gate: each slice returns the documented status; expected failures are `4xx`, never `500`; responses are
valid JSON.

### F5.3-AS3 — Prove and document the skeleton [implements F5.3-US1, F5.3-US5]
Directive: write the end-to-end test with no mocks and the tracer-bullet note (T6, T7).
Acceptance gate: the test exercises the real stack (enroll persisted, lesson read, unknown → `404`); the note records
the discipline.

## Execution plan — first two stories

*story → agent story → tasks → files → command → gate.*

### Plan A — F5.3-US1 (enroll round-trips end to end) · via F5.3-AS1 + AS2
1. Tasks T1 → T3 (+ T5 seed).
2. Files: `lib/portal/engine.ex`, `lib/portal_web/router.ex`, `priv/seeds.exs`.
3. Command: `mix run priv/seeds.exs` then `mix run --no-halt`, then `curl -i -X POST
   "localhost:4000/enroll?user=USR1&course=CRS1"`.
4. Gate (closes US1): `201` with the new `ENR` id; the enrollment is retrievable from the store with `progress: 0`.
   Requirements: F5.3-R1, F5.3-R3, F5.3-R6.

### Plan B — F5.3-US3 (deliver a lesson) · via F5.3-AS2
1. Tasks T2 → T4.
2. Files: `lib/portal/engine.ex`, `lib/portal_web/router.ex`.
3. Command: `curl -i localhost:4000/lessons/LSN1` and `curl -i localhost:4000/lessons/LSNbogus`.
4. Gate (closes US3): known lesson → `200` with the lesson; unknown → `404`. Requirements: F5.3-R2, F5.3-R4.

Efficiency check: making the boundary real and mapping the slice is two files behind one `curl`; the second slice
reuses the same query → context path — no layer is reworked.

## Comprehensive implementation prompt

> Paste into a coding agent (e.g. Claude Code) at the root of the `portal` app, after F5.1–F5.2. It executes
> F5.3-AS1..AS3 in task-topology order and ends on the verification gates. Treat the Requirements and the Definition of
> Done in [`f5.3.md`](f5.3.md) as the acceptance checklist.

```text
You are implementing spec F5.3 (Tracer bullets: a walking skeleton) in the `portal` app, on top of F5.1–F5.2. Read
specs/pragmatic/f5.3.md, specs/pragmatic/f5.3.stories.md, and specs/design/f0.md first. Wire ONE use
case end to end, then a second thin slice — every layer touched, none finished, no mocks. Master invariant: the web
layer calls only the Portal.Engine boundary; the engine delegates to the F5.2 contexts. Control flow: tagged tuples;
expected failures map to 4xx, never 500.

Build in this order, keeping the app runnable after each step:

1) Make the boundary real (T1, T2 → R1, R2). In lib/portal/engine.ex, implement dispatch/1 for
   %{type: :enroll, user_id: u, course_id: c} by calling Portal.Learning.enroll(u, c); implement query/2 for
   (:lesson, id) -> Portal.Catalog.lesson(id) and (:courses_of, user_id) -> Portal.Learning.courses_of(user_id).
   Return the contexts' tagged results.

2) Enroll slice (T3 → R3, R5). In lib/portal_web/router.ex, POST /enroll parses user/course params, calls
   Portal.Engine.dispatch(...), and maps {:ok, e} -> 201 with Jason-encoded %{id: e.id}, {:error, _} -> 422. The
   context persists via Portal.Store.put/1.

3) Lesson + courses slices (T4 → R4, R5). GET /lessons/:id calls Portal.Engine.query(:lesson, id) and maps
   {:ok, lesson} -> 200 (JSON), :error -> 404. GET /courses/:user_id calls query(:courses_of, user_id) -> 200 with the
   list.

4) Seed (T5 → R6). Add priv/seeds.exs loading one USR user, one CRS course, one LSN lesson into Portal.Store so the
   slices round-trip locally.

5) End-to-end test (T6 → R7). In test/portal_web/enroll_slice_test.exs, drive the real stack with no mocks: POST
   /enroll -> 201 and the enrollment is retrievable; GET /lessons/:id -> 200; an unknown lesson -> 404.

6) Tracer-bullet note (T7 → R8). Document that every layer is touched and none finished, and that the skeleton grows
   one vertical slice at a time.

7) Verify (R-checks + DoD). Confirm: a real curl enroll returns 201 and persists a retrievable enrollment; lessons
   return 200/404; courses returns 200; the web calls only the engine boundary; expected failures are 4xx, never 500.
   Report each result against the F5.3 Definition of Done.

Deliverables: the touched files listed in specs/pragmatic/f5.3.llms.md, with the enroll slice walking end to end and
every F5.3 invariant exercised. Stop when the Definition of Done in specs/pragmatic/f5.3.md is fully checked.
```
