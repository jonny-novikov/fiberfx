# FP-Native Foundations & "Ship Every Iteration" Delivery for an Elixir/Phoenix Real-Time Learning Platform ("Portal")

## TL;DR
- **Replace the Meyer / design-by-contract framing with an FP-native correctness stack**: railway-oriented programming (`with` + tagged tuples) for control flow, "parse, don't validate" + "make illegal states unrepresentable" at boundaries, a functional core / imperative shell architecture, property-based testing (StreamData) for invariants, and the emerging Elixir set-theoretic type system (plus Dialyzer today) for static checks. These are the functional analogues of preconditions/postconditions/invariants.
- **Anchor rapid delivery in tracer-bullet/walking-skeleton thinking** (Pragmatic Programmer), Lean Startup MVP / Build-Measure-Learn, XP small releases, and Continuous Delivery, executed concretely with Phoenix generators (`mix phx.gen.auth`, `phx.gen.live`, `phx.gen.context`) and spec-/story-driven workflows (INVEST user stories, Given-When-Then, Specification by Example, llms.txt, Anthropic agent guidance).
- **Build the real-time Portal on BEAM-native primitives** — Phoenix PubSub, Presence, Channels, LiveView, and Oban for jobs — with a Node interop layer bridged by BullMQ (Redis), a Telegram bot front-end (validating Login Widget / WebApp `initData`), and passwordless/magic-link auth. For error handling, prefer `with` + tagged tuples for business pipelines and `Ecto.Multi` for transactional ones; reserve exceptions for truly exceptional faults.

## Key Findings

1. **Railway-oriented programming is the canonical FP replacement for design-by-contract control flow**, and it maps almost one-to-one onto Elixir's `with` expression and `{:ok, _} | {:error, _}` tagged tuples. Scott Wlaschin (its originator) explicitly notes it is the `Either`/`Result` monad with a railway analogy, and warns against overusing it.
2. **"Parse, don't validate" (Alexis King) and "make illegal states unrepresentable" (Yaron Minsky) are the boundary and data-modeling disciplines** that functionally express invariants — instead of runtime precondition checks, you encode constraints in types/structs and parse untrusted input once at the edge. King presents the slogan as the encapsulation of type-driven design: strengthen argument types so partial functions become total (e.g., a `NonEmpty` type lets `head` "unconditionally return the first element without forcing us to handle the case we know is impossible").
3. **Elixir's type story is in transition**: Dialyzer provides success typing today; the new set-theoretic type system (Castagna/Duboc/Valim) is being incrementally shipped into the compiler and currently surfaces as warnings inferred from patterns/guards, with user-provided signatures and the phase-out of Erlang typespecs planned for later milestones.
4. **The functional-core/imperative-shell pattern (Gary Bernhardt) maps cleanly to Elixir/OTP**: pure decide/evolve logic tested with fast, doubles-free unit and property tests; side effects (DB, PubSub, HTTP, Telegram) pushed to the OTP edge.
5. **Phoenix gives you a true "ship every iteration" toolchain out of the box** — generators scaffold a full vertical slice (context + schema + LiveView + tests), which is the literal embodiment of a tracer bullet / walking skeleton.
6. **The BEAM real-time stack (PubSub/Presence/Channels/LiveView) plus Oban covers the Portal's core needs natively**; BullMQ is the right bridge when Node workers must share a queue, and Telegram provides a low-friction passwordless front-end if `initData`/Login Widget signatures are validated server-side.

## Details

### 1. FP-native correctness & specification techniques

**Railway-oriented programming (ROP).** Canonical source: Scott Wlaschin, "Railway Oriented Programming," F# for Fun and Profit — landing page https://fsharpforfunandprofit.com/rop/ (slides, NDC London/Oslo 2014 and Functional Programming eXchange talk videos). Wlaschin states plainly that "Any Haskeller… will immediately recognize this approach as the Either type, specialized to use a list of a custom error type for the Left case," and that he invented the analogy, not the technique. Crucially, he also published a counterweight, "Against Railway-Oriented Programming" (https://fsharpforfunandprofit.com/posts/against-railway-oriented-programming/), warning that `Result` should not be used everywhere — a contested-use caveat worth citing in your methodology doc. Slides: https://www.slideshare.net/ScottWlaschin/railway-oriented-programming and https://speakerdeck.com/swlaschin/railway-oriented-programming-a-functional-approach-to-error-handling.

**Elixir `with` + tagged tuples.** The idiomatic mapping of ROP onto Elixir. The official `with` special form and the `{:ok, value} | {:error, reason}` convention are documented in the Elixir docs (lists-and-tuples: https://hexdocs.pm/elixir/lists-and-tuples.html). Two excellent community write-ups: Drew Olson, "Elixir's Secret Weapon" (https://blog.drewolson.org/elixirs-secret-weapon/) shows how `with` replaces nested `case` for dependent ok/error operations; and Erlang Solutions, "Can't Live `with` It, Can't Live `with`out It" (https://www.erlang-solutions.com/blog/cant-live-with-it-cant-live-without-it/) covers the `else`-clause ambiguity problem and the tuple-wrapping / normalize-return-values solutions (also citing Chris Keathley's "Good and Bad Elixir," which argues to avoid `else` in `with` blocks). Tyler Pachal's "Using tuple-wrapping to improve `with` expressions" (https://tylerpachal.medium.com/using-tuple-wrapping-to-improve-with-expressions-in-elixir-f491b06265fa) is a good practical reference. Dave Thomas's "Programming Elixir" is the standard book reference for the tagged-tuple idiom. Libraries that add ROP plumbing (use sparingly): `OK` (https://github.com/CrowdHailer/OK) provides result-monad `~>>`/`~>` pipes and `OK.for/1`.

**"Parse, don't validate."** Canonical source: Alexis King, "Parse, don't validate" (2019), https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/. King frames it as a "single, snappy slogan that encapsulates what type-driven design means to me": a parser turns less-structured input into more-structured output and preserves the proof in the type, whereas validation throws that information away ("shotgun parsing"). The payoff is totality — strengthening argument types (e.g., `NonEmpty`) so `head` "should unconditionally return the first element without forcing us to handle the case we know is impossible." It applies directly to Elixir boundaries — parse external input (Telegram `initData`, HTTP params, API responses) once into a constrained struct, then let downstream functions trust it. One Medium engineering essay explicitly extends the principle "from C all the way to Elixir."

**Typespecs, Dialyzer, and the set-theoretic type system.** Today, Elixir uses `@spec`/`@type` typespecs with Dialyzer's *success typing* (it only reports provable failures, not all type errors). `Hammox` (https://hexdocs.pm/hammox/readme.html) extends Mox to enforce typespecs on mocks at runtime — a useful complement since Dialyzer can't catch Mox mocks diverging from a behaviour's typespec. The new direction is the **gradual set-theoretic type system**: announced at ElixirConf EU 2022 (José Valim, "My Future with Elixir: set-theoretic types," https://elixir-lang.org/blog/2022/10/05/my-future-with-elixir-set-theoretic-types/), a CNRS/Remote partnership with PhD student Guillaume Duboc led by Giuseppe Castagna. The academic foundation is "The Design Principles of the Elixir Type System" (Castagna, Duboc, Valim), The Art, Science, and Engineering of Programming, vol. 8, no. 2 (2024), article 4, DOI 10.22152/programming-journal.org/2024/8/4, preprint https://arxiv.org/pdf/2306.06391. Current status (per https://hexdocs.pm/elixir/gradual-set-theoretic-types.html): the compiler infers types from patterns and guards and surfaces findings *as warnings*, without requiring code changes; types are built from unions/intersections/negations; a later milestone introduces set-theoretic function signatures, at which point Erlang typespecs "will be phased out of the language and have their postprocessing moved into a separate library." The system has been shipping incrementally since Elixir v1.17, with implementation details (e.g., lazy BDDs) added in v1.19 (https://elixir-lang.org/blog/2025/12/02/lazier-bdds-for-set-theoretic-types/).

**Property-based testing.** Lineage: Koen Claessen & John Hughes, "QuickCheck: A Lightweight Tool for Random Testing of Haskell Programs," ICFP 2000, DOI 10.1145/351240.351266 (PDF: http://www.cs.tufts.edu/~nr/cs257/archive/john-hughes/quick.pdf). In Elixir: **StreamData** (https://github.com/whatyouhide/stream_data, docs https://hexdocs.pm/stream_data/), announced on the official blog (https://elixir-lang.org/blog/2017/10/31/stream-data-property-based-testing-and-data-generation-for-elixir/). Core macro is `check all` with generators and filters; failures *shrink* to a minimal counterexample. The README explicitly credits the original QuickCheck paper. For **stateful** property testing or stored counterexamples, StreamData does not yet support them — use **PropCheck** (a wrapper around Erlang's PropEr) instead. Book: Fred Hébert, "Property-Based Testing with PropEr, Erlang, and Elixir" (propertesting.com). State invariants (e.g., "a streak never decreases on a same-day correct answer") as properties rather than DbC postconditions.

**Functional core, imperative shell.** Gary Bernhardt, "Boundaries" talk (SCNA 2012, https://www.destroyallsoftware.com/talks/boundaries) and the "Functional Core, Imperative Shell" screencast (https://www.destroyallsoftware.com/screencasts/catalog/functional-core-imperative-shell). The pattern — pure values and logic in the core (many fast unit tests, no test doubles), side-effects/dependencies in a thin shell (few integration tests) — maps onto Elixir as pure `decide/evolve` functions wrapped by GenServers/LiveViews/contexts that perform I/O. A worked Elixir example exists (Brian Sung, Medium) and a Haskell-flavored treatment (Javier Casas).

**Totality / make illegal states unrepresentable.** Yaron Minsky coined the slogan in his 2010 "Effective ML" Harvard guest lecture (Vimeo: https://vimeo.com/14313378; notes summarizing the nine pieces of advice: https://gist.github.com/jonschoning/8394412). Scott Wlaschin's "Designing with Types" series and his book "Domain Modeling Made Functional" carry the same idea into F#. Richard Feldman's "Making Impossible States Impossible" (elm-conf 2016) popularized it for front-end. In Elixir: model state with tagged unions of distinct structs rather than one "bag of fields" struct, so impossible combinations can't be constructed (cf. the typestate-pattern-in-Erlang treatment by Radek Szymczyszyn).

**How invariants are expressed functionally (vs DbC).** Synthesis for the methodology doc:
- *DbC preconditions* → parse-at-boundary (smart constructors returning `{:ok, struct} | {:error, _}`) + guards/pattern matching.
- *DbC postconditions* → property-based tests asserting invariants over generated inputs; return-type discipline via typespecs/Dialyzer/the new type system.
- *DbC class invariants* → "make illegal states unrepresentable" data modeling so the invariant holds by construction; assertions for the residual cases the type system can't express.

### 2. Testing developer experience (DX) in Elixir

- **ExUnit + doctests + async.** ExUnit is the built-in framework; `use ExUnit.Case, async: true` runs test modules concurrently. Doctests execute examples embedded in `@doc`.
- **Fixtures vs setup.** Prefer `setup`/`setup_all` callbacks and `start_supervised!/1` (which guarantees a process finishes before the test/owner exits) over ad-hoc fixtures.
- **Mox for behaviour-based mocking.** Mox (https://hexdocs.pm/mox/Mox.html, repo https://github.com/dashbitco/mox) follows José Valim's "Mocks and explicit contracts" (https://dashbit.co/blog/mocks-and-explicit-contracts): mocks must be defined from a behaviour, defined in `test_helper.exs`/`setup_all` (not per-test), support `async: true`, and rely on pattern matching for input assertions. Use `verify_on_exit!`, `expect`, `stub`, `stub_with`. "Mock" is a noun, not a verb — don't globally monkey-patch. Hammox adds typespec enforcement.
- **Ecto.Sandbox for DB tests.** `Ecto.Adapters.SQL.Sandbox` (https://hexdocs.pm/ecto_sql/Ecto.Adapters.SQL.Sandbox.html, testing guide https://hexdocs.pm/ecto/testing-with-ecto.html). Idiomatic setup uses `start_owner!/2` with `shared: not tags[:async]` and `on_exit` calling `stop_owner/1` — this fixed the dangling-process problem where LiveView processes outlived the test and took the connection with them (German Velasco, https://www.germanvelasco.com/blog/how-live-view-got-rid-of-dangling-processes-in-tests). Allowances (`allow/3`) keep async tests possible across collaborating processes; shared mode is the fallback. Caveat: tasks spawned inside LiveView under `Phoenix.Ecto.SQL.Sandbox` (browser/E2E mode) don't automatically inherit the sandbox (phoenix_ecto issue #157).
- **Phoenix.LiveViewTest.** https://hexdocs.pm/phoenix_live_view/Phoenix.LiveViewTest.html — `live/2`, `render_click`, `element/3`, `has_element?`, `render_async` (awaits `assign_async`/`start_async`), `assert_patch`, `assert_redirect`, `assert_push_event`. Default async assertion timeout is 100ms.
- **Phoenix ConnTest** for controller/endpoint tests; **Wallaby / PhoenixTest** for browser/feature tests. PhoenixTest (https://hexdocs.pm/phoenix_test/) unifies LiveView and static-page testing and shows a `FeatureCase` template using the sandbox owner pattern.
- **Test pyramid guidance.** Functional core → many fast, deterministic, doubles-free unit + property tests (`async: true`). Imperative shell/contexts → fewer integration tests with the Ecto sandbox. Feature/E2E → a thin top layer (Wallaby/PhoenixTest). Tag and isolate slow/external tests per "Mocks and explicit contracts."

### 3. Pragmatic / rapid iterative delivery foundations

- **Tracer bullets & walking skeleton.** David Thomas & Andrew Hunt, "The Pragmatic Programmer" (20th Anniversary Edition, Addison-Wesley, 2019, ISBN 9780135957059; publisher page https://pragprog.com/titles/tpp20/). Tip: "Use Tracer Bullets to Find the Target." Tracer code is *lean but complete* and becomes part of the final system's skeleton (unlike a throwaway prototype) — build an end-to-end vertical slice early, then flesh it out. The "walking skeleton" is the equivalent term from Cockburn / "Growing Object-Oriented Software, Guided by Tests."
- **Lean Startup / MVP / Build-Measure-Learn.** Eric Ries, "The Lean Startup" (Crown Business, 2011, ISBN 9780307887894; official site https://theleanstartup.com/book). Introduces the Minimum Viable Product and the Build-Measure-Learn loop with validated learning.
- **XP small releases.** Kent Beck (with Cynthia Andres), "Extreme Programming Explained: Embrace Change," 2nd ed. (Addison-Wesley, 2004, ISBN 9780321278654). "Small Releases" is a primary practice: put a simple system into production quickly and release on a very short cycle.
- **Continuous Delivery.** Jez Humble & David Farley, "Continuous Delivery: Reliable Software Releases through Build, Test, and Deployment Automation" (Addison-Wesley, 2010, ISBN 9780321601919; companion site https://continuousdelivery.com/). Core concept: the automated deployment pipeline.
- **Phoenix generators / RAD.** Official docs: `mix phx.gen.auth` (https://hexdocs.pm/phoenix/mix_phx_gen_auth.html) — generates an `Accounts` context with `User`/`UserToken`, LiveView-based by default, and (Phoenix 1.8) magic-link/passwordless auth with a `Scope` struct for per-user scoping; `mix phx.gen.live` (https://hexdocs.pm/phoenix/Mix.Tasks.Phx.Gen.Live.html) — generates LiveViews + context + schema + tests for a resource; the Contexts guide (https://hexdocs.pm/phoenix/contexts.html) frames contexts as API boundaries. Book: Bruce Tate & Sophie DeBenedetto, "Programming Phoenix LiveView: Interactive Elixir Web Programming Without Writing Any JavaScript" (Pragmatic Bookshelf, 1st ed., 2024, ISBN 9781680508215; https://pragprog.com/titles/liveview/programming-phoenix-liveview/). These generators *are* tracer bullets: one command scaffolds a full vertical slice that compiles, runs, and ships.
- **Spec-/story-driven development with coding agents.**
    - User stories: Mike Cohn, "User Stories Applied: For Agile Software Development" (Addison-Wesley, 2004, ISBN 9780321205681; https://www.mountaingoatsoftware.com/books/user-stories-applied).
    - INVEST: Bill Wake, "INVEST in Good Stories, and SMART Tasks" (2003), https://xp123.com/invest-in-good-stories-and-smart-tasks/ (Independent, Negotiable, Valuable, Estimable, Small, Testable).
    - Gherkin / Given-When-Then: official Cucumber docs (https://cucumber.io/docs/gherkin/ and reference https://cucumber.io/docs/gherkin/reference/); origin in Dan North's BDD work, "Introducing BDD" (2006), https://dannorth.net/blog/introducing-bdd/, with the Given-When-Then template (Dan North & Chris Matts, ~2004).
    - Specification by Example: Gojko Adzic, "Specification by Example" (Manning, 2011, ISBN 9781617290084; https://gojko.net/books/specification-by-example/) — collaborative specs + living documentation.
    - llms.txt: Jeremy Howard / Answer.AI (proposed Sept 2024), https://llmstxt.org/ — a root `/llms.txt` Markdown file giving LLMs a curated, low-noise map of a site/codebase. Caveat: it is a community convention with no standards-body backing and inconsistent adoption by major AI providers; treat as a documentation aid, not a guarantee.
    - Anthropic: "Building Effective Agents" (https://www.anthropic.com/research/building-effective-agents) — distinguishes workflows (predefined paths) from agents (model-directed), recommends the simplest sufficient design; "Best practices for Claude Code" (https://code.claude.com/docs/en/best-practices) — verifiable checks (tests/build/lint exit codes) as stop conditions, subagents for isolated tasks, Writer/Reviewer patterns. These pair naturally with the verifiable-test-loop ethos of the functional core.

### 4. Real-time production architecture on the BEAM + Node interop

- **Phoenix PubSub.** https://hexdocs.pm/phoenix_pubsub/Phoenix.PubSub.html — realtime publish/subscribe; default `PG2`/Distributed-Elixir adapter exchanges messages across clustered nodes; supports custom dispatchers and "fastlaning" (encode once, write to many sockets). Subscribe in LiveView `mount`, broadcast on events, handle in `handle_info`.
- **Phoenix Presence.** https://hexdocs.pm/phoenix/Phoenix.Presence.html and guide https://hexdocs.pm/phoenix/presence.html — distributed, conflict-free presence tracking built on PubSub + `Phoenix.Tracker`; `mix phx.gen.presence` scaffolds a module. Keep metadata minimal ("online"/"away"). For LiveView, track in the Presence process and stream diffs to views to avoid holding the full user list in each LiveView's memory.
- **Phoenix Channels & LiveView** provide the transport and server-rendered real-time UI respectively; both ride on PubSub.
- **Oban (BEAM-native jobs).** https://github.com/oban-bg/oban — PostgreSQL-backed (also SQLite/MySQL) job processing with transactional enqueue (outbox pattern), retries with backoff, unique jobs, cron, pruning (`Pruner`), rescue (`Lifeline`), and Telemetry. It's the standard Elixir choice; Redis-backed alternatives (Exq, Verk, Flume) are largely unmaintained. Per Oban's own benchmark "One Million Jobs a Minute with Oban" (oban.pro), a single queue on a single node "peaks at around 17,699 jobs/sec and finishes one million jobs in 57s," and throughput "tops out around 32,000 jobs/sec, over two billion jobs/day" at concurrency 2,000 (Erlang/OTP 25, Elixir 1.14.0-dev, PostgreSQL 14.2, Apple M1 Pro). Use Oban for streak rollovers, scheduled quiz delivery, and notification dispatch.
- **BullMQ (Node/Redis bridge).** https://bullmq.io/ — Redis-backed queue for Node.js (also Python/Elixir/PHP/Bun), with delayed jobs, cron/repeatable jobs, and rate limiting. A shared Redis queue is the clean way to bridge Node workers and the BEAM: enqueue from one runtime, consume in another. There is even a BullMQ Elixir client. A vendor-published benchmark (https://bullmq.io/articles/benchmarks/bullmq-elixir-vs-oban/) reports BullMQ outperforming Oban: "BullMQ scales from 911 to 8,300 jobs/sec (9.1x) when going from 10 to 100 concurrent processors; Oban scales from 523 to 4,400 (8.4x)"; on CPU-bound work BullMQ hit ~24.5K vs Oban's ~4.0K jobs/sec, and on single-job inserts ~5,800 vs ~2,900/sec. This is BullMQ's own benchmark (Redis vs Postgres) and should be read with that bias; note it directly contradicts Oban's own ~17.7K–32K jobs/sec figures above, underscoring that throughput claims are configuration- and vendor-dependent. For most Portal workloads Oban's transactional guarantees and zero extra infrastructure are the better default, with BullMQ reserved specifically for the Node interop seam.
- **Telegram Bot API + Elixir/Node libraries.** Official API: https://core.telegram.org/bots/api. Elixir: **Telegex** (https://github.com/telegex/telegex) — framework + client, fully typed, code-generated from the official docs, with polling and webhook (`Telegex.Hook.GenHandler`) modes; **ExGram** (https://github.com/rockneurotiko/ex_gram) — low-level API + opinionated framework with a context DSL, polling/webhooks, middleware, multi-bot. A bot fits the functional-core/imperative-shell model as a thin imperative front-end/worker that hands parsed updates to the pure core. Node equivalents include telegraf/grammY.
- **One-click / passwordless auth.**
    - Telegram Login Widget: https://core.telegram.org/widgets/login and "Log In With Telegram" https://core.telegram.org/bots/telegram-login (now supports OpenID Connect with PKCE, plus the legacy widget). Validate by recomputing the HMAC-SHA256 hash with your bot token server-side before trusting any user data.
    - Telegram WebApp `initData` validation: https://docs.telegram-mini-apps.com/platform/init-data — sort key=value pairs (excluding `hash`), join with `\n`, derive the key as `HMAC-SHA256(bot_token, "WebAppData")`, then `HMAC-SHA256(data_check_string, that_key)`, and compare to `hash`; a newer Ed25519 third-party validation path exists with a published public key. An Elixir helper exists: `telegram_miniapp_validation` on Hex.
    - Magic-link / passwordless generally: Phoenix 1.8 `mix phx.gen.auth` now generates magic-link login and removes password registration by default (https://hexdocs.pm/phoenix/mix_phx_gen_auth.html), using signed, single-use `UserToken`s delivered by email — the canonical Elixir pattern.
- **Gamification mechanics.** Reputable first-party engineering references: Duolingo's blog on streaks — "Improving the streak: Forming habits one lesson at a time" (Anton Yu, Nov 19, 2020, https://blog.duolingo.com/improving-the-streak/), which reports that "learners who reach a streak of 7 are 2.4 times more likely to continue using Duolingo the next day than learners without a streak," and that separating the daily-goal and streak mechanics produced "a 3.3% increase in Day 14 retention… a 1% increase in our overall daily active learners; and, in just 20 days, a 10.5% increase in the percentage of daily learners on a streak," increasing learners on a 7+ day streak by "over 40%." Also "How Streaks keep Duolingo learners committed…" (https://blog.duolingo.com/how-streaks-keep-duolingo-learners-committed-to-their-language-goals/) and "How we protect learner streaks from site issues" (https://blog.duolingo.com/protecting-streaks-from-site-issues/) for streak-repair infrastructure. For scheduling/delivering notifications, the official platform docs are the canonical engineering references: Android `AlarmManager` (https://developer.android.com/develop/background-work/services/alarms) for exact-time reminders and WorkManager (https://developer.android.com/develop/background-work/background-tasks/persistent) for deferrable/periodic work; on the BEAM, implement these with Oban cron + scheduled jobs.

### 5. Error-handling / control-flow decision matrix inputs (Elixir)

| Approach | Best for | Readability | Composability | Testability | Error typing | Authoritative source |
|---|---|---|---|---|---|---|
| `with` + tagged tuples (ROP) | Sequential dependent ops that all return `{:ok,_}/{:error,_}` (business pipelines) | High when returns are uniform; `else` ambiguity hurts otherwise | High (linear, short-circuits) | High (pure, pattern-matchable) | Structural via tagged tuples; no enforced types yet | Elixir `with` docs; Wlaschin ROP; Erlang Solutions `with` post |
| Exceptions (`raise`/`try`/`rescue`, bang funcs) | Truly exceptional / programmer-error cases; "let it crash" | High for happy path; control flow hidden | Low (non-local) | Lower (must assert raises) | Exception structs; not in the value channel | Elixir "try, catch, rescue" docs; tagged-tuple basics posts |
| `Ecto.Multi` | Multi-step DB transactions needing atomicity + introspection | High (named steps, pipeline) | High (data structure you can append/merge/prepend) | High (`to_list/1` lets you assert without running) | Per-step `{:error, name, value, changes_so_far}` | https://hexdocs.pm/ecto/Ecto.Multi.html; SmartLogic & Elixir School Multi guides |
| `Repo.transact/2` (fn) | Simpler transactions with normal control flow | High | Medium | Medium | `{:ok,_}/{:error,_}` | https://hexdocs.pm/ecto/Ecto.Repo.html |
| Result libraries (`OK`, etc.) | Heavy monadic piping when `with` feels verbose | Medium (extra operators/macros) | High | High | Structural | https://github.com/CrowdHailer/OK |

Decision guidance for the Portal: default to `with` + tagged tuples for context/business logic; switch to `Ecto.Multi` whenever ≥2 DB writes must be atomic (e.g., record quiz answer + bump streak + enqueue notification); reserve exceptions for invariant violations and unrecoverable faults; avoid result libraries unless `with` ergonomics genuinely break down. This is the FP-native replacement for DbC's pre/post/invariant contract checks.

## Recommendations

**Stage 1 — Re-found the methodology doc (week 1).** Replace the Meyer/DbC reference section with the FP stack: ROP (`with` + tagged tuples), parse-don't-validate at boundaries, make-illegal-states-unrepresentable for domain types, functional core/imperative shell for architecture, property-based testing for invariants, and the Elixir type story (Dialyzer now; set-theoretic types emerging). Use the citations above verbatim in a references section. **Threshold to revisit:** when Elixir ships user-provided set-theoretic function signatures (a later milestone), upgrade the "typing" section from "warnings only" to "checked signatures" and plan typespec phase-out.

**Stage 2 — Encode delivery cadence (week 1–2).** Adopt tracer-bullet delivery: each iteration ships a thin end-to-end vertical slice via `mix phx.gen.live`/`phx.gen.auth`. Write stories as INVEST + Given-When-Then; keep an `llms.txt` (and a CLAUDE.md / agent guide) so coding agents have a curated map; gate agent/CI work on verifiable checks (test/build/lint exit codes) per Anthropic's guidance. **Threshold:** if a slice can't be demoed end-to-end in one iteration, it's too big — split the story.

**Stage 3 — Stand up the real-time core (week 2–4).** Build on PubSub + Presence + LiveView for live quiz/leaderboard/presence, Oban for streak rollovers and scheduled notification dispatch. Keep decide/evolve logic pure; push DB/PubSub/HTTP/Telegram to the shell. Use `Ecto.Multi` for the answer→streak→notification transaction.

**Stage 4 — Add Telegram + Node interop (week 3–5).** Use Telegex or ExGram for the bot; validate Login Widget / WebApp `initData` HMAC server-side before trusting any user; wire passwordless/magic-link auth via Phoenix 1.8 `phx.gen.auth`. Introduce BullMQ only at the Node seam (shared Redis queue); keep Oban as the BEAM-side default. **Threshold:** adopt BullMQ only if a genuine Node worker must share the queue; otherwise stay Oban-only to avoid Redis as an extra dependency.

**Stage 5 — Lock in test DX (continuous).** `async: true` everywhere the sandbox allows; Mox for external-service behaviours (+ Hammox for typespec enforcement); `start_supervised!` and the `start_owner!`/`stop_owner` sandbox pattern to avoid dangling processes; property tests for gamification invariants (streak monotonicity, scoring totality). Keep the pyramid bottom-heavy: many pure unit/property tests, fewer sandbox integration tests, a thin Wallaby/PhoenixTest E2E layer.

## Caveats
- **ROP is contested even by its author** — Wlaschin's "Against Railway-Oriented Programming" warns against using `Result` everywhere; treat `with`/tagged tuples as the default for uniform ok/error pipelines, not a universal hammer. The `with`/`else` ambiguity is a known footgun; prefer normalizing return values over a catch-all `else`.
- **Elixir's set-theoretic type system is not finished.** As of the latest releases it only emits warnings inferred from patterns/guards; user-provided signatures and typespec phase-out are future milestones. Don't promise compile-time contract enforcement it doesn't yet deliver; Dialyzer's success typing remains the pragmatic static check today, with Hammox for runtime mock/behaviour conformance.
- **The BullMQ-vs-Oban benchmark is vendor-published** (BullMQ's site, Redis vs Postgres) and directly conflicts with Oban's own published figures. Higher raw throughput is plausible but orthogonal to Oban's transactional guarantees and operational simplicity; choose per workload, not per benchmark.
- **llms.txt has no standards-body backing** and inconsistent adoption by major AI providers; it's a useful curated map for agents, not a guarantee that any given model reads it.
- **Telegram auth security hinges on server-side signature validation.** Never trust `initData`/widget payloads client-side; always recompute the HMAC (or verify the Ed25519 signature / OIDC JWT) with your bot token before creating a session.
- **Sandbox + async edge cases exist** (e.g., tasks spawned inside LiveView don't auto-inherit the sandbox in browser/E2E mode); use allowances or `start_supervised!` and test the async paths explicitly.
- **Some cited write-ups are community blogs rather than primary docs** (used only to illustrate idioms); the canonical docs/papers/books are cited alongside them and should be preferred in the references section. The Duolingo retention figures are first-party product results from A/B tests — useful as directional evidence, not as guaranteed outcomes for a different product.
