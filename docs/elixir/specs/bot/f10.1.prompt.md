# F10.1 ship prompt — Director's orchestration brief

> Ship **F10.1 (the echo_bot engine skeleton — one bot from a vendored-fork wrap)** by fanning out the specialized
> lead-team — `venus` (reconcile + brief) → `mars` ×2 (build, then harden) → `apollo` (verify) — with the Director in
> the loop at each gate, one LAW-4 commit at close, and the **F10.2–F10.3 reconcile** as the closing feedback loop. This
> file is the single artifact a Director reads to run the rung; the *discipline* lives in the agent definitions
> (`.claude/agents/{venus,mars,apollo}.md`), so the per-stage prompts below carry only the rung delta.

## The rung in one paragraph

F10.1 stands up a **new, standalone** umbrella app `apps/echo_bot` (OTP `:echo_bot`, namespace `EchoBot`, `mod:
{EchoBot.Application, []}`) that loads **one** bot from a versioned YAML config (`version: "1.0"`) and runs it under its
own `:one_for_one` `EchoBot.Supervisor`. The Telegram surface is a **vendored, owned copy of ex_gram** under
`apps/echo_bot/vendor/ex_gram/` (the minimal `getUpdates` long-poll + `sendMessage` subset, with its `README.md`
provenance + `CLAUDE.md` ownership docs), **wrapped** behind the `EchoBot.Platform` behaviour by
`EchoBot.Platform.Telegram` (adapter #1); a two-command handler answers `/start` and `/help` with static text; a fake
updater drives the handler in tests with no live Telegram. Portal is **untouched** — `echo_bot` does not depend on,
modify, or supervise `portal`/`portal_web`, and names no `Portal.*`. No games, no Portal touchpoint at this rung. Spec
triad: [`f10.1.md`](f10.1.md) (body, authoritative) · [`f10.1.stories.md`](f10.1.stories.md) (Connextra +
Given/When/Then, covers every deliverable) · [`f10.1.llms.md`](f10.1.llms.md) (the agent brief Venus refreshes).

## Mode — Flat-L2 lead-team, review gates (not Workflow)

Real `Agent` spawns + SendMessage coordination, Director-in-loop between stages — **not** a deterministic `Workflow`
fan-out. The rung is sequential (Mars builds from Venus's brief; Apollo verifies Mars's tree) with two human-grade
decision points (the reconcile result, the final commit), so the value is in the *gates*, not in parallelism. Process
weight scales to **risk**: F10.1 adds a brand-new umbrella app over an unchanged engine and an unchanged Portal — a new
greenfield app, moderate risk — so this is a lead-team, not the full audit ceremony. All peers Opus (LAW-2). The
Director coordinates and ratifies; peers write the code (LAW-1a); one commit at close (LAW-4).

## What the roadmap already settled — and the additions it ratifies

The F10 roadmap ([`f10.roadmap.md`](f10.roadmap.md)) **decided** the questions that used to be the F10.1 forks. The
triad already encodes the decisions. So Venus does **not** surface these as open forks — they are closed:

| Settled by the roadmap | Decision |
|---|---|
| Library posture | **wrap-then-own a vendored copy** of ex_gram (not a hex dep, not a from-scratch hand-roll); the copy is owned source, modifiable directly, no upstream PR. |
| Vendored path | `apps/echo_bot/vendor/ex_gram/`, carrying its `README.md` (provenance + Beer-Ware license) + `CLAUDE.md` (ownership directive). |
| YAML schema | **v1.0, approved** — five required keys (`version`, `name`, `platform`, `token_env`, `handler`); the loader reads `version` first. |
| Scope | **one** bot, polling, `/start` + `/help` static text; **no** games (F10.3), **no** Portal touchpoint (F10.9, optional). |

These additions are **greenfield** — `apps/echo_bot` does not yet exist — so there is no STALE/INVENTED/MISSING claim
against as-built `echo_bot` code. Venus's pre-build reconcile is therefore against the **umbrella conventions** the new
app must match, ratified by the roadmap:

- The standard umbrella `mix.exs` block (`build_path`/`config_path`/`deps_path`/`lockfile` → `../../`, `elixir: "~>
  1.18"`) — copy from `apps/portal_web/mix.exs:8-12`.
- `apps_path: "apps"` in the root `mix.exs` auto-discovers `apps/echo_bot` — **no root `mix.exs` edit** (`mix.exs:6`).
- `:one_for_one` + named supervisor — the `Portal.Supervisor` / `PortalWeb.Supervisor` pattern
  (`apps/portal/lib/portal/application.ex:40`, `apps/portal_web/lib/portal_web/application.ex:21`).
- The umbrella `config/config.exs` composes per-app blocks with `config :app, ...` and ends `import_config
  "#{config_env()}.exs"` (`config/config.exs:52`) — the `echo_bot` block is added there (umbrella-level, never
  `portal`/`portal_web` code).

Venus confirms each convention cite resolves, pins the YAML v1.0 + vendored-path additions in the brief (ratified by the
roadmap, not re-opened), and refreshes [`f10.1.llms.md`](f10.1.llms.md).

## Why the prompts are short (the evolved process)

An early-arc Mars prompt ran ~180 lines — the full contract plus every discipline (cite-don't-invent, the master
invariant, the determinism gate, framing, the report format). F10.1's Mars prompt is ~20, because all of that now lives
in `mars.md`. The spawn carries only what is *new about this rung* — the new app, the vendored-wrap boundary, the YAML
v1.0 schema. That is DRY applied one level up: the "how the agent works" fact has one authority (the agent definition),
and the prompt carries only the delta. Cheap acceptance — the Given/When/Then stories Mars + Apollo verify against — is
the multiplier.
