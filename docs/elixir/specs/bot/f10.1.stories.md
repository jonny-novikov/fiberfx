## F10.1-US2 â€” Run the engine as a standalone supervised app

As an **operator**, I want `echo_bot` to boot as its own supervised umbrella app, so that I can run it with the

  the adapter reaches the vendored copy directly.

## F10.1-US4 â€” Define a bot in versioned YAML

As an **operator**, I want to declare a bot in a versioned YAML file (`version: "1.0"`), so that adding or removing a
bot is plug-and-play, the schema can evolve, and no token is ever written into config.

Acceptance criteria
- Given a YAML v1.0 file with `version`, `name`, `platform`, `token_env`, and `handler`, when the loader reads it, then
  it reads the `version` first, validates against the v1.0 schema, and returns one bot definition with the five fields.
- Given the definition's `token_env`, when the engine boots, then the token is read from that named environment variable
  and does not appear in the YAML.
- Given `platform: telegram`, when the engine starts the bot, then `EchoBot.Platform.Telegram` is selected and the named
  `handler` module routes its commands.

INVEST â€” independent; testable by a loader unit test plus a boot check; encodes F10.1-INV5.

## F10.1-US5 â€” Keep the engine platform-abstracted

As a **developer**, I want the engine core to speak a platform-adapter behaviour rather than Telegram directly, so that

Acceptance criteria

  reply-send) by wrapping the vendored copy, and the YAML's `platform` field selects it.
- Given the behaviour, when a second platform is imagined, then adding it is implementing the behaviour, not editing the
  core.

## F10.1-US7 â€” A self-healing bot

As an **operator**, I want the bot to recover on its own, so that a transient crash does not leave Telegram unanswered.

- Given the bot crash, when it restarts, then `Portal.Application` and `PortalWeb.Application` are undisturbed.
