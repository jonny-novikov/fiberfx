# F10.1 · user stories

> Who wants the bot that answers, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f10.1.md`](f10.1.md).

## F10.1-US1 — Answer from Telegram

As a **learner**, I want the bot to reply when I send `/start` or `/help`, so that I know it is live and learn what it
can do.

Acceptance criteria
- Given the bot is supervised, when a `/start` update is fed via the `:noup` updater, then the handler answers with the
  welcome text.
- Given the bot is supervised, when a `/help` update is fed via the `:noup` updater, then the handler answers with the
  help text.
- Given a live polling dev bot, when a real Telegram message sends `/start`, then a reply arrives in the chat (the
  demo).

INVEST — valuable on its own (a live bot day one); testable by constructed `:noup` updates and one manual demo; encodes
F10.1-INV3.
Priority: must · Size: 3 · Implements deliverables: F10.1-D3, F10.1-D5, F10.1-D6.

## F10.1-US2 — Run the bot as a supervised umbrella app

As an **operator**, I want the bot to boot as a supervised umbrella app, so that I can run it with the standard
Elixir toolchain and a crash never takes the engine down.

Acceptance criteria
- Given the umbrella, when it starts, then `Bot.Application` supervises the ex_gram children (`ExGram` + the
  `Portal.Bot` updater) as a third app supervisor, `:one_for_one`.
- Given the boot sequence, when apps start, then the `:bot` → `:portal` dependency boots the domain tree before the bot
  tree.
- Given the running umbrella, when `Portal.Application` and `PortalWeb.Application` are inspected, then they are
  unchanged by F10.1.

INVEST — independent of reply content; testable by a boot check and a tree assertion; encodes F10.1-INV2.
Priority: must · Size: 3 · Implements deliverables: F10.1-D1, F10.1-D2.

## F10.1-US3 — Establish the boundary seam

As a **developer**, I want the bot to be a driving adapter that names only the `Portal` facade, so that adding the chat
front door does not risk the domain core and later command rungs have a clean seam.

Acceptance criteria
- Given the bot app, when I search `apps/bot/lib/`, then no module references `Portal.Engine`, a context module, a repo,
  or `GenServer.call`.
- Given the F10.1 handler, when it answers, then it composes static text through the context DSL and makes no facade
  call (the seam is established, not yet exercised).
- Given the dependency graph, when `mix.exs` files are inspected, then ex_gram appears only in `apps/bot/mix.exs` and
  the domain apps stay bot-free.

INVEST — independent; testable by a static search and a dep-graph check; encodes F10.1-INV1, INV2.
Priority: must · Size: 2 · Implements deliverables: F10.1-D1, F10.1-D3.

## F10.1-US4 — A self-healing bot

As an **operator**, I want the bot to recover on its own, so that a transient crash does not leave Telegram unanswered.

Acceptance criteria
- Given the running umbrella, when the `Portal.Bot` updater process is killed, then `Bot.Application` restarts it.
- Given the restart, when a `/start` update is fed afterwards, then it is answered again.
- Given the bot crash, when it restarts, then `Portal.Application` and `PortalWeb.Application` are undisturbed.

INVEST — independent; testable by killing the process and re-feeding an update; encodes F10.1-INV2.
Priority: should · Size: 2 · Implements deliverables: F10.1-D2, F10.1-D6.

## F10.1-US5 — The same answer on a resend

As a **learner**, I want a redelivered message to behave the same, so that a Telegram resend never doubles a reply or
an effect.

Acceptance criteria
- Given a `/start` update with a known `update_id`, when the same update is delivered twice, then the handler answers
  the same welcome text and produces no duplicate effect.
- Given the F10.1 handler, when the same update is handled, then the reply is a pure function of the update (no
  accumulating state to double).

INVEST — valuable for trust under Telegram resends; testable today by feeding the same constructed update twice;
encodes F10.1-INV4.
Priority: should · Size: 1 · Implements deliverables: F10.1-D4.
