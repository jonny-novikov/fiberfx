# F10 · Roadmap — the Portal Telegram bot (Elixir, ex_gram)

> A near-term delivery plan for a Telegram bot that lets a learner use the Portal from chat. Built in Elixir with
> **ex_gram**, running **in-BEAM** as a driving adapter over the `Portal` facade — a sibling of the F5.9 LiveView. The
> bot calls only `Portal` and renders only the closed `%Portal.Error{}` set; everything below the boundary is the
> engine you already specced in F5. This file carries the **feature abstracts** for the F10 ladder; each abstract
> graduates into a full spec triad (`f10.N.md` / `.stories.md` / `.llms.md`) when its iteration is sharpened.

## What we are delivering

A supervised Telegram bot that turns chat messages into Portal commands and queries: a learner sends `/enroll`,
`/courses`, `/lesson`, `/progress`, and the bot answers by calling the `Portal` facade and rendering the result. The
bot is a thin **driving adapter** — the same role the F5.9 LiveView plays — so the domain, the event log, and the
supervision tree are unchanged. The one concern the bot owns that the web did not is **identity**: mapping a Telegram
chat to a Portal learner.

The near-term goal is the full learner loop shipped thin but robust: **answer → identity → browse → enroll → learn &
progress** (rungs F10.1–F10.5). Notifications, hardening, webhook delivery, and the off-BEAM scale-out path
(F10.6–F10.9) are specified as abstracts and deferred until the shipped loop earns feedback.

## Architecture decision — ex_gram, in-BEAM

ex_gram is the Telegram Bot API client and framework for Elixir: its API tracks the current Telegram Bot API, it
supports both polling and webhook updaters, it runs under OTP supervision, and it offers a context DSL and inline
keyboards for composing replies. It fits the Portal because the bot is a small adapter over an engine that already
runs in the same VM.

In-BEAM is the right default for the MVP. The bot is a driving port like the LiveView: it issues `Portal.enroll/2` and
`Portal.progress_of/1` as in-process function calls, with no network hop, no serialization, and no second runtime to
operate. It joins `Portal.Application` as one more supervised child, so OTP gives it crash isolation and restart for
free, and it reads the same `%Portal.Error{}` contract the web reads. A separate Node service would buy independent
scaling at the cost of a queue, a deploy target, and a second copy of the domain vocabulary — cost the MVP does not
need.

That trade is kept reversible. The off-BEAM path — a Node/Fastify worker using grammY, talking to Portal over EchoMQ
(on BullMQ) instead of in-process calls — is documented as the F10.9 seam, so moving the bot out later is a planned
migration rather than a rewrite.

## The boundary (the master invariant, restated for the bot)

The bot calls **only** the `Portal` facade — `enroll/2`, `deliver_lesson/2`, `progress_of/1`, `courses_of/1` — and
never names the engine, the event store, or a repo. Failures cross to the bot as `%Portal.Error{}` whose `code` is the
closed union (`:already_enrolled | :course_not_found | :lesson_locked | :invalid_progress`), and the bot's job is to
render each code as a friendly message. This is exactly the contract the F5.9 LiveView consumes, so the bot adds a
second consumer of the same closed set without changing anything below the facade — the F5→F6 handoff invariant holds
for F10 too: the only structural change to the tree is adding the bot's children.

Identity is the bot's own boundary. A Telegram update carries a `chat_id`, not a Portal `USR`; the bot resolves one to
the other through a small identity seam (see *Seams & open decisions*) before it issues any command, so the facade
always receives a well-formed branded user id.

## How this roadmap runs

Two roles, the same loop the F5 roadmap uses (see [`../pragmatic/pragmatic.roadmap.md`](../pragmatic/pragmatic.roadmap.md)):

- **Author (Claude)** turns each feature abstract into a spec triad and a build plan, at the same quality bar as the F5
  specs — Goal, Rationale (5W), Scope, Deliverables, Invariants, Definition of Done; user stories; an agent brief with
  a paste-ready prompt.
- **Operator (the person in this conversation)** reviews the delivered specs and the shipped increment, then returns
  feedback asking for the next feature's specs or a change to a shipped one.

The loop per rung is **sharpen → build → ship → demo → review → feedback → adapt**. Feedback edits the spec, because
the spec is the single source of truth; the build follows the spec, never the other way around. "Adopt / learn /
evolve" here means this feature-development process, not end-user telemetry.

## "Thin but robust" for the bot

Each rung is a narrow vertical slice — usually one command, end to end — built to production quality, not a prototype.
Concretely:

- **Supervised.** The bot starts under `Portal.Application` as ex_gram children (`ExGram` plus the bot module), so a
  crash restarts in isolation and never takes the engine down.
- **Contract-guarded.** Every learner action goes through the `Portal` facade and its F5.4 contract; the bot adds no
  domain logic of its own and invents no new error vocabulary.
- **Harnessed without Telegram.** Tests use ex_gram's `:noup` updater (the no-op updates worker) and feed constructed
  updates to the handler, asserting the `Portal` calls made and the message rendered — the real in-memory engine
  behind the facade, no live Telegram and no mocks of the domain. This is the bot's analog of "a wide pure base plus
  one process test."
- **Idempotent.** Update handling tolerates a re-delivered update (Telegram can resend), so the same message never
  produces a duplicate effect.
- **Typed at the edge.** The bot renders the closed `%Portal.Error{}` set exhaustively; an unhandled code is a gap, not
  a silent fall-through.

Every rung ships behind the same Definition-of-Done gates the Portal specs use, and the per-iteration table below names
the demo and the harness for each.

## Feature abstracts — the F10 ladder

**F10.1 · Start thin: a bot that answers.** A supervised ex_gram bot in the Portal tree (polling in dev), replying to
`/start` and `/help` with static text. Establishes the driving-adapter seam — a `Portal.Bot` handler that will call
only the facade — and the `:noup` test posture. *Ships:* a live bot on day one, proving the supervision and update
loop. *Who:* Operator, Developer.

**F10.2 · Identity: link a Telegram chat to a learner.** Resolve a Telegram `chat_id` to a Portal `USR`; `/start`
links an existing learner or provisions one and remembers the binding. Introduces the bot's one genuinely new concern
behind a small identity boundary, idempotently (a second `/start` is a no-op). *Ships:* every later command knows who
is asking. *Who:* Developer, Learner.

**F10.3 · Browse the catalog.** `/courses` lists available courses via a Portal query (`courses_of/1` or a catalog
read), rendered as a message, optionally with an inline keyboard. Read-only — the query path through the facade. *Ships:*
a learner can see what to enroll in. *Who:* Learner, Developer.

**F10.4 · Enroll from chat.** `/enroll <course>` issues `Portal.enroll/2`; the closed reasons `:already_enrolled` and
`:course_not_found` map to friendly replies and success confirms. The first command path through the F5.4 contract,
surfaced in chat. *Ships:* the core action works from Telegram. *Who:* Learner, Developer.

**F10.5 · Learn a lesson and check progress.** `/lesson <id>` issues `Portal.deliver_lesson/2`; `/progress` issues
`Portal.progress_of/1` and renders `0..100`. Completes the learner loop — browse → enroll → learn → progress — end to
end over the facade. *Ships:* a usable bot for the whole core journey. *Who:* Learner, Operator.

**F10.6 · Notifications: events become messages.** Subscribe to engine events (`LearnerEnrolled`, `LessonDelivered`)
over PubSub and push a Telegram message to the bound chat. The first event-driven (push) path, distinct from the
request/response commands — the events-to-notifications seam from the Decider article
([`../pragmatic/decider-pattern.md`](../pragmatic/decider-pattern.md)). *Ships:* the bot tells learners things
proactively. *Who:* Learner, Architect.

**F10.7 · Robustness: error contract, rate limits, retries.** Handle Telegram API failures and rate limiting (`429`
with `retry_after`), keep update handling idempotent under resends, and render the full closed `%Portal.Error{}` set.
Supervised and correct under load. *Ships:* the bot stays up and right when Telegram misbehaves. *Who:* Operator,
Developer.

**F10.8 · Webhook delivery for production.** Switch the updater from `:polling` to `:webhook` (ex_gram serves
`/telegram/<bot_token_hash>`), fronted by the F6 Phoenix endpoint and TLS; keep polling for dev and tests. A
deployment concern, not a domain change. *Ships:* production-grade, low-latency delivery. *Who:* Operator.

**F10.9 · Scale-out seam: grammY + Fastify + EchoMQ.** Document the exit path to move the bot off-BEAM — a Node/Fastify
worker using grammY, reaching Portal over EchoMQ (on BullMQ) rather than in-process calls — for when traffic or team
boundaries warrant independent scaling. Specified, not built, so the in-BEAM choice stays reversible. *Ships:* a known,
low-risk migration if the bot must scale on its own. *Who:* Architect.

## Near-term delivery (the iterations that ship first)

F10.1–F10.5 are the production-bound iterations — the learner loop, thin but robust. F10.6–F10.9 are deferred pending
feedback from the shipped loop.

| Rung | Ships (the slice) | Demo | Harness | Feedback asked |
| --- | --- | --- | --- | --- |
| F10.1 | supervised bot answering `/start`, `/help` | message the bot, get a reply | `:noup` updater + handler tests (fake update → expected answer) | is the command surface and tone right? |
| F10.2 | chat→learner identity; `/start` provisions/links | `/start` twice resolves to one learner | identity-map tests; provisioning idempotent | is one chat = one learner the right model? |
| F10.3 | `/courses` catalog listing (query path) | `/courses` lists seeded courses | query-path test through the real in-memory engine | listing format — plain text or inline keyboard? |
| F10.4 | `/enroll <course>` (command + error mapping) | enroll succeeds; duplicate → "already enrolled" | a test per `%Portal.Error{}` code → message | error wording and confirmation UX? |
| F10.5 | `/lesson <id>` + `/progress` (full loop) | deliver a lesson; `/progress` shows movement | end-to-end handler test through the facade | is the loop complete and usable; notifications next? |

Each iteration is one PR-sized increment: a spec triad, the slice built against it, a green harness, a demo, and a
short feedback note that may edit the spec before the next rung starts.

## Seams & open decisions

- **Where identity lives.** The chat→learner map is reference/identity data, not event-sourced domain state. The
  cleanest home is a small `Portal.Identity` context exposed through the facade (e.g. `Portal.link_telegram/2`,
  `Portal.learner_for_chat/1`), keeping the bot off the store. The link can carry a branded Snowflake id (namespace
  `TGL`) over an integer column. Decide in F10.2.
- **Polling vs webhook.** Polling for dev and tests (and behind a firewall); webhook for production, served by ex_gram
  at `/telegram/<bot_token_hash>` and fronted by the F6 endpoint ([`../phoenix/phoenix.md`](../phoenix/phoenix.md)).
  The updater is config-selected, so the choice changes no handler code — the same substitutability the F5.8 event
  store has. Built in F10.8.
- **Notifications transport.** Events reach the bot via PubSub (Phoenix.PubSub), the engine publishing on append and
  the bot subscribing per bound chat. This is the push counterpart to the command path and the first place the event
  log earns its keep outside replay. Built in F10.6.
- **The off-BEAM seam.** If the bot must scale or ship independently, the F10.9 path replaces in-process facade calls
  with EchoMQ messages to a Node/Fastify + grammY worker. Documented now so the migration is bounded.
- **Config.** The bot token is runtime configuration (an environment variable read at boot), never compiled in;
  multiple bots (e.g. a staging bot) are supported by ex_gram if needed.

## Conventions

- Branded Snowflake ids for any new identifier (integer column, branded transport form with a namespace prefix and
  base62 encoding); the bot passes well-formed branded ids to the facade.
- The master invariant holds: the bot calls only `Portal` and renders only the closed `%Portal.Error{}` set; it names
  no engine, store, or repo.
- ex_gram idioms: the bot is a supervised child of `Portal.Application`; replies use the context DSL
  (`context |> answer(...)`); tests use the `:noup` updater so no live Telegram is contacted.
- A+ quality gates and Writerside-friendly markdown throughout: prose over heavy formatting, clean voice, balanced
  fences, resolving links.
