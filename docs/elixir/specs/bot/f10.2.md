# F10.2 · Multi-bot: N YAML bots under one engine

> Generalize the engine from one bot to **N**. The config loader reads **every** `*.yaml` in a config directory, one
> `EchoBot.Config` definition per file, and `EchoBot.Application` starts a supervised bot per definition under one
> `:one_for_one` engine supervisor — each bot with its own token (from `token_env`), its own handler, and its own
> isolated process subtree. A second bot is added by dropping a YAML file, not by editing the engine. One bot's crash
> never disturbs a sibling or the engine. Still polling; still the `/start` + `/help` class of static handlers; **no**
> games, **no** Portal touchpoint, and **no** change to `portal` or `portal_web`. This is where "multibot" actually
> lands.

## Goal

After F10.2, the engine runs **N** bots from a directory of YAML v1.0 files, each under its own supervised, isolated
process subtree, all under one `:one_for_one` `EchoBot.Supervisor`. At boot, `EchoBot.Config` reads **every** `*.yaml`
file in the configured bot-config directory, producing one bot definition per file (the same per-bot shape F10.1's
single-file loader returns). `EchoBot.Application` starts a supervised bot per definition — each with its own token
resolved from its `token_env`, its own `handler` module, and its own updater process — and assigns each a supervisor
child id derived from its YAML `name`. A second bot is added by **dropping a YAML file into the directory** and
restarting the engine, never by editing engine code. Per-bot isolation is the load-bearing property: one bot's updater
crashing restarts **only that bot** under `:one_for_one`, leaving every sibling bot and the engine itself answering. The
engine still polls, still answers the `/start` + `/help` class of static commands, runs **no** game, names **no**
`Portal.*` function, and touches **no** file under `apps/portal/` or `apps/portal_web/`. The umbrella keeps its four
apps (`:echo_data`, `:portal`, `:portal_web`, `:echo_bot`); F10.2 changes only code **inside** `apps/echo_bot/`.

## Rationale (5W)

- **Why** — "multibot" is the engine's defining capability, and it lands here, not at F10.1. F10.1 proved the skeleton
  end to end with one bot; F10.2 proves the skeleton **generalizes** — that the loader, the supervisor, and the
  handler dispatch are per-bot, not global, so adding the second, fifth, or tenth bot is a config drop, not a code
  change. The hard property F10.2 buys is **isolation**: N independent bots in one BEAM release, each crash-contained,
  so a bug or a revoked token in one bot cannot take down the others. That isolation is what makes "run several bots
  from one release" a production posture rather than a demo, and it is the precondition for everything later in the
  ladder (the richer surface, the game, the second platform) running across multiple bots at once.
- **What** — a directory-reading config loader (read every `*.yaml`, one `EchoBot.Config` definition per file, the same
  per-bot shape F10.1 returns); a fan-out in `EchoBot.Application` that starts one supervised, isolated bot subtree per
  definition under the existing `:one_for_one` `EchoBot.Supervisor`; per-bot supervisor child ids derived from each
  bot's YAML `name` (with duplicate-`name` rejection); per-bot token resolution (each bot reads its own `token_env`);
  per-bot handler dispatch (each bot routes through its own `handler` module); and verification that N bots boot, that
  dropping a YAML file adds a bot, and that killing one bot leaves the siblings and the engine answering.
- **Who** — Operators (run several bots from one config directory in one release, add a bot by dropping a file) and
  Developers (generalize the loader and the supervisor to per-bot, keep Portal untouched, and prove sibling isolation).
- **When** — the second rung of the F10 value ladder; it depends on F10.1's shipped skeleton (the loader, the
  supervisor, the platform-adapter port + vendored wrap, the handler/router seam) and on nothing in Portal. The
  minimal valuable game (F10.3), the richer Telegram surface (F10.4), webhook (F10.5), owning the wrap (F10.6), and the
  later rungs build on the multi-bot engine this rung lands.
- **Where** — code **inside** `apps/echo_bot/` only: the config loader (`apps/echo_bot/lib/echo_bot/config.ex` —
  generalized to read a directory), `apps/echo_bot/lib/echo_bot/application.ex` (the per-bot fan-out under the existing
  supervisor), a second handler module and a second YAML definition under the bot-config directory
  (`apps/echo_bot/priv/bots/` or the config-resolved path), and the test files. The bot-config **directory** path is
  resolved from the umbrella `config/*` (a directory, where F10.1 named a file) — an umbrella-level config edit, never
  `portal`/`portal_web` code. No file under `apps/portal/` or `apps/portal_web/` is touched; `apps/echo_bot/mix.exs`
  still does **not** list `{:portal, in_umbrella: true}`; the root `mix.exs` is unchanged.

## Scope

- **In** — generalizing `EchoBot.Config` to read **all** `*.yaml` files in a configured bot-config **directory**,
  returning a **list** of per-bot definitions (each the same shape F10.1's single-file loader returns: `version`,
  `name`, `platform`, `token_env`, `handler`, the token resolved from its `token_env`); validating each file against
  the v1.0 schema independently (a malformed or wrong-version file is reported per file, not silently dropping the
  whole set — see D2); rejecting two definitions that share a `name` (the supervisor child id must be unique); the
  fan-out in `EchoBot.Application` that starts **one supervised, isolated bot subtree per definition** under the
  existing `:one_for_one` `EchoBot.Supervisor`, each child id derived from the bot's `name`; per-bot token resolution
  (each bot reads its own `token_env` at boot) and per-bot handler dispatch (each bot routes through its own `handler`
  module); a **second** bot (a second YAML definition + a second handler module) added to prove the drop-a-file path; a
  per-bot supervised-start test (N definitions → N running bots) and an **isolation test** (kill one bot, a sibling
  still answers, the engine supervisor is undisturbed); the static-fan-out-vs-`DynamicSupervisor` shape decision
  (D3, with a recommendation).
- **Out** — runtime hot-add/remove of a bot **without** an engine restart (F10.2 adds a bot by dropping a file and
  restarting; a live drop-in/drop-out file watcher is a later concern, not this rung); the minimal valuable game —
  `apps/echo_games`, the Svelte 5/Vite rewrite, the games web server, WebSocket live state, the payments seam (→
  F10.3); the richer Telegram surface — free-text, inline keyboards, formatting, real command routing beyond
  `/start`+`/help` (→ F10.4); webhook delivery (→ F10.5); replacing the vendored internals (→ F10.6); proving the port
  with a second platform (→ F10.7); the full payments flow (→ F10.8); any Portal touchpoint — the engine names no
  `Portal.*` function and adds no child to `Portal.Application` (→ F10.9, optional); robustness, rate limits, and
  notifications (→ F10.10); any new YAML key or a schema version bump (the v1.0 shape is unchanged); any change to the
  platform-adapter behaviour, the Telegram adapter, or the vendored ex_gram copy (inherited from F10.1 unchanged).

## Deliverables

- **F10.2-D1 — the directory config loader.** Generalize `EchoBot.Config` so the engine loads **every** `*.yaml` file
  in a configured bot-config **directory** (the directory path resolved from umbrella application config, where F10.1
  resolved a single file path). It returns a **list** of per-bot definitions — one per file — each the **same** value
  F10.1's single-file loader returns (the v1.0 five-key definition with the token resolved from its `token_env` at
  boot; the token never written in any YAML). The per-file read reuses F10.1's v1.0 reader (read `version` first,
  validate against the v1.0 schema, then the four other keys); F10.2 adds only the directory walk and the list result.
- **F10.2-D2 — per-file validation + duplicate-name rejection.** Each `*.yaml` file is validated against the v1.0
  schema **independently**: a file's outcome (a valid definition, or a named error for a malformed / wrong-version /
  missing-key file) is reported per file. Two definitions that share a `name` are a **rejected** collision (the YAML
  `name` is the supervisor child id, which must be unique) — the loader surfaces the duplicate `name` rather than
  silently starting one and dropping the other. The exact failure posture for a single bad file among good ones (boot
  the good bots and log the bad file, **or** refuse to boot until the directory is clean) is fixed at build against
  F10.1's loader error convention — recommend **fail-fast at boot** (a misconfigured bot is an operator error worth
  surfacing loudly), reconciled to whatever error shape F10.1's loader already returns.
- **F10.2-D3 — the per-bot supervised fan-out.** `EchoBot.Application.start/2` starts **one supervised, isolated bot
  subtree per definition** under the existing `:one_for_one` `EchoBot.Supervisor` (the strategy and name are inherited
  from F10.1 unchanged). Each bot's child spec reuses F10.1's single-bot child spec (the updater process plus any
  adapter client/registry process the bot needs), and each child's **id is derived from the bot's YAML `name`** so the
  supervisor can name, restart, and isolate each bot independently. **Shape decision (static fan-out vs
  `DynamicSupervisor`):** at F10.2 the bot set is **fixed at boot** (read from the directory once), so a **static child
  list** — `EchoBot.Supervisor` started with the N bot child specs computed at `init/1` from the loaded definitions —
  is the **recommended** shape: it is the simplest thing that satisfies the rung (N supervised children, each
  isolated, each restarted independently under `:one_for_one`), and it keeps the supervisor declarative. A
  `DynamicSupervisor` is **not** required here because nothing adds or removes a bot at runtime at F10.2; it is the
  natural shape only when a **later** rung introduces live add/remove (out of scope here). *Recommendation:* static
  fan-out at F10.2, `DynamicSupervisor` deferred to the rung that adds runtime bot management. The Operator confirms
  this shape (it is a design choice, recorded with a recommendation, not silently decided).
- **F10.2-D4 — per-bot token + handler dispatch.** Each bot resolves its **own** token from its **own** `token_env`
  env var at boot (two bots name two different env vars; the engine reads each independently), and each bot routes its
  commands through its **own** `handler` module (the YAML `handler` key per definition). The dispatch is per-bot: an
  update arriving at bot A's updater is handled by bot A's handler with bot A's identity, never crossed with bot B.
- **F10.2-D5 — the second bot (prove drop-a-file).** A **second** bot definition (a second `*.yaml` v1.0 file in the
  bot-config directory) plus a **second** handler module are added, so the engine runs **two** bots from the same
  directory with no engine-code edit beyond placing the file and the handler. This is the concrete demonstration that
  "add a bot = drop a YAML file (and its handler)" holds. Both bots answer `/start` and `/help` with their own static
  text.
- **F10.2-D6 — the per-bot supervised-start test.** A test asserts that a bot-config directory with **N** v1.0 files
  yields **N** running, supervised bots: the loader returns N definitions; `EchoBot.Supervisor` has N bot children,
  each with a child id derived from its `name`; each bot's handler is reachable via the fake updater (a constructed
  `/start` to bot A returns bot A's welcome text; the same to bot B returns bot B's). No live Telegram is contacted.
- **F10.2-D7 — the isolation test.** A test asserts sibling isolation under `:one_for_one`: with two bots running,
  killing **one** bot's updater process restarts **only that bot** (the sibling bot's process is untouched and keeps
  answering, the engine supervisor and the other apps' supervisors are undisturbed). A `/start` fed to the surviving
  bot during the crashed bot's restart is still answered.
- **F10.2-D8 — verification.** `mix compile` is clean and the umbrella boots with N bots under `EchoBot.Application`,
  `Portal.Application` and `PortalWeb.Application` unchanged and independent; the directory loader reads every `*.yaml`
  and yields one definition per file; two `name`-colliding files are rejected; a `/start` to each bot via the fake
  updater answers that bot's welcome text; dropping a second YAML file + handler adds a second running bot with no
  engine edit; killing one bot's updater restarts only that bot and a sibling still answers; the engine core still
  reaches the vendored copy only through `EchoBot.Platform.Telegram`; `grep -rE "Portal|PortalWeb"
  apps/echo_bot/lib/` returns nothing and `apps/echo_bot/mix.exs` does not depend on Portal; `apps/portal/` and
  `apps/portal_web/` are unchanged; a live manual demo (two real Telegram bots, each with its own token, each answering
  `/start`).

## Invariants

- **F10.2-INV1 (Portal untouched — the no-touch boundary, inherited from F10.1-INV1)** — no file under `apps/portal/`
  or `apps/portal_web/` is modified; `apps/echo_bot/mix.exs` does **not** list `{:portal, in_umbrella: true}` and names
  neither Portal app; no module under `apps/echo_bot/lib/` references `Portal`, `PortalWeb`, `Portal.Application`,
  `Portal.Engine`, a Portal context, or a Portal repo (`grep -rE "Portal|PortalWeb" apps/echo_bot/lib/` returns
  nothing); `echo_bot` adds **no** child to `Portal.Application`. The multi-bot engine boots whether or not Portal is
  running. Any future Portal touchpoint (F10.9, optional) is out of scope here.
- **F10.2-INV2 (per-bot isolation — the load-bearing multi-bot property)** — each bot runs as its own supervised
  process subtree under the existing `:one_for_one` `EchoBot.Supervisor`, with a child id derived from its YAML
  `name`. One bot's crash restarts **only that bot** and never disturbs a sibling bot, the engine supervisor, or any
  other umbrella app's tree. With N bots running, killing one leaves N−1 answering throughout that bot's restart.
- **F10.2-INV3 (config-directory-driven — N bots from N files, plug-and-play)** — the set of running bots is exactly
  the set of valid `*.yaml` v1.0 files in the configured bot-config directory; the engine starts one supervised bot per
  file and hard-codes no bot. Adding a bot is dropping a YAML file (and its handler) and restarting; removing a bot is
  removing its file and restarting. No bot is named in engine code.
- **F10.2-INV4 (YAML v1.0 schema unchanged — read per file)** — each file is the **same** approved v1.0 schema
  (`version`, `name`, `platform`, `token_env`, `handler`, all required, the `version` read first and validated). F10.2
  bumps no version and adds no key; the only loader change is reading a directory of files and returning a list. Each
  bot's token is referenced by its `token_env` env-var name and read at boot — never written in any YAML, never
  compiled in.
- **F10.2-INV5 (unique bot identity)** — two bot definitions may not share a `name` (the YAML `name` is the supervisor
  child id and the log identifier, which must be unique). The loader rejects a duplicate `name` rather than starting one
  bot and silently dropping the other; the supervisor never holds two children with the same id.
- **F10.2-INV6 (platform-abstracted + vendored-wrap boundary preserved — inherited from F10.1-INV3, INV4)** — every bot
  runs through the **same** `EchoBot.Platform` behaviour and the **same** `EchoBot.Platform.Telegram` adapter F10.1
  built; the engine core names only the behaviour, never a concrete platform or a vendored module. F10.2 adds no
  adapter, no platform, and no vendored code; the engine core still reaches the vendored ex_gram copy **only** through
  `EchoBot.Platform.Telegram`, and no module outside the adapter names a vendored module.
- **F10.2-INV7 (per-bot dispatch — no crossing)** — an update delivered to bot A's updater is handled by bot A's
  `handler` with bot A's token and identity, and never by bot B's handler. Each bot's token resolution (`token_env`),
  handler dispatch (`handler`), and updater process are scoped to that bot; there is no shared mutable handler state
  across bots.
- **F10.2-INV8 (harnessed without a live platform — inherited from F10.1-INV6)** — the updater is config/adapter-
  selected (polling for dev, the fake updater for tests) for **every** bot, so N bots are exercised in tests with no
  live Telegram. A handler test feeds a constructed update to a chosen bot and asserts that bot's rendered reply.
