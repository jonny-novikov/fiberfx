## Requirements

- **F10.3-R1** — a new standalone umbrella app `apps/echo_games` exists: `apps/echo_games/mix.exs` declares `app:
  :echo_games`, `mod: {EchoGames.Application, []}`, the umbrella's standard
  `build_path`/`config_path`/`deps_path`/`lockfile`/`elixir` block, and deps `{:bandit, "~> 1.5"}`, `{:plug, "~> 1.16"}`,
  `{:websock, "~> 0.5"}`, `{:websock_adapter, "~> 0.5"}`, `{:jason, "~> 1.4"}` (all already in `mix.lock`). It does
  **NOT** list `{:portal, in_umbrella: true}` or `{:portal_web, in_umbrella: true}` and names neither Portal app.
  `apps/portal/mix.exs`, `apps/portal_web/mix.exs`, and the root `mix.exs` are unchanged. [US: F10.3-US2, F10.3-US7]
- **F10.3-R2** — `EchoGames.Game` (pure): the deck builder (a level's `rows × cols` board of emoji pairs from the
  16-emoji pool, shuffled), the flip/match resolution, and the scoring ported **exactly** from `html/game.html` —
  `BASE_PER_PAIR = 100`, `COMBO_STEP = 25` (`bonus = (combo - 1) × 25`), `WRONG_PENALTY = 10`, `TIME_BONUS = 5`/s, the
  three levels and their star thresholds, the level-3 12 s reshuffle. The module is a **pure function of state and
  intent** — no process, no I/O, no clock in the core (elapsed seconds arrive as an argument). [US: F10.3-US3]
- **F10.3-R3** — `EchoGames.Session` (a `GenServer`, one per session, under a `DynamicSupervisor`): holds the
  authoritative `EchoGames.Game` state, applies a card-tap intent, advances levels, runs the timers and the level-3
  reshuffle, and produces the state to push. A session crash isolates under the `DynamicSupervisor`. [US: F10.3-US4,
  F10.3-US8]
- **F10.3-R4** — the WebSocket protocol (JSON): the client sends a card-tap intent (`{"type": "tap", "card": <id>}`) and
  a new-game/level intent; the server pushes the authoritative state (the visible deck `{id, emoji-if-revealed,
  matched}`, the HUD, the timer, the level/game result with stars and the score breakdown). The client renders pushed
  state and computes **no** score. Define the message vocabulary as the rung design (small, versioned, JSON). [US:
  F10.3-US4, F10.3-US3]
- **F10.3-R5** — the new isolated games web server: `EchoGames.Web.Router` (`Plug.Router`) run by a **Bandit** listener
  under `EchoGames.Application` on its own port (config-resolved; not `:4000`), serving the Vite bundle over an
  **explicit** `Plug.Static` mount (verified by a `curl` PROBE, the F6.5.5 guard), the WebSocket upgrade route via
  `WebSock`/`WebSockAdapter`, the game-launch landing, and the reserved payment-webhook route. It is **not**
  `PortalWeb.Endpoint` and adds no route/child to `portal_web`. [US: F10.3-US2, F10.3-US5]
- **F10.3-R6** — the Svelte 5 (Runes) + Vite frontend under `apps/echo_games/assets/`, building to
  `apps/echo_games/priv/static/`: a rewrite of `html/game.html` that **preserves** the visual design (palette, card-flip
  animation, HUD, combo pulse, timer bar, result modals, Russian copy) and **moves the logic server-side** — it renders
  pushed state and sends card-tap intents over the WebSocket, computing no score. Vite is a build-time Node toolchain
  outside the BEAM; the server serves only the built output. [US: F10.3-US1, F10.3-US5]
- **F10.3-R7** — the bot game-launch: a bot command (specced `/play`, re-grounded against F10.1/F10.2) sends a Telegram
  **Game** whose launch URL the games server serves. The launch is an `echo_bot` surface; `echo_games` and `echo_bot`
  stay **decoupled** (no cross-app `in_umbrella` dep, neither names the other). [US: F10.3-US1, F10.3-US7]
- **F10.3-R8** — the payments seam: a **reserved** payment-webhook route (a named `Plug.Router` clause returning a
  not-implemented marker, no payment logic) plus the server-authoritative score (R2/R3/R4) as the precondition for a paid
  game. No invoice, no `successful_payment`, no validation, no paid outcome — deferred to F10.8. [US: F10.3-US6]
- **F10.3-R9** — tests: game-logic unit tests (the deck builder, the flip/match resolution, the scoring + star
  thresholds against the `html/game.html` constants); a WebSocket round-trip test (a tap intent → an
  authoritative-state push, asserting the server computed the score); and a static-bundle serve check (a `curl` PROBE →
  `200` + expected content type). No Portal contacted; the server runs on its own test port. [US: F10.3-US3, F10.3-US4,
  F10.3-US5]
- **F10.3-R10** — no module under `apps/echo_games/lib/` contains the strings `Portal` or `PortalWeb`;
  `apps/echo_games/mix.exs` lists neither Portal app; the games server adds no route or child to `portal_web` (the
  no-touch boundary + the new-server boundary, statically checkable). [US: F10.3-US7]
- **F10.3-R11** — killing an `EchoGames.Session` process isolates the crash under the `DynamicSupervisor`; another
  session and the other app supervisors (`Portal`, `PortalWeb`, `EchoBot`) are undisturbed. [US: F10.3-US8]

## Execution topology

**Runtime topology (what exists and how a tap flows).**

```text
:portal      app  Portal.Application (Supervisor, :one_for_one, Portal.Supervisor)        # apps/portal — UNTOUCHED
├─ Portal.Repo · Portal.Store · Portal.EventStore.adapter() · Portal.Engine · {Phoenix.PubSub, Portal.PubSub}

:portal_web  app  PortalWeb.Application (Supervisor, :one_for_one, max_restarts: 300)      # apps/portal_web — UNTOUCHED
├─ PortalWeb.Telemetry · PortalWeb.Endpoint (:4000) · PortalWeb.Presence

:echo_bot    app  EchoBot.Application (Supervisor, :one_for_one, EchoBot.Supervisor)       # apps/echo_bot — F10.1, NO dep
├─ the YAML-loaded bot(s) updater(s)                                                       # on :portal; launches the game

:echo_games  app  EchoGames.Application (Supervisor, :one_for_one, EchoGames.Supervisor)   # apps/echo_games — NEW,
│                                                                                          # NO dep on :portal / :echo_bot
├─ {Bandit, plug: EchoGames.Web.Router, port: <own, config-resolved, NOT :4000>}          # the NEW games web server
├─ {DynamicSupervisor, EchoGames.SessionSupervisor}                                        # one child per active session
└─ (a Registry, if the WebSocket handler routes by session key)
        │
        │  GET /            ──▶ Plug.Static ──▶ apps/echo_games/priv/static/ (the Vite bundle)   [probe with curl → 200]
        │  GET /ws  (upgrade)──▶ WebSock/WebSockAdapter ──▶ EchoGames.Web.Socket
        │  POST /payments/webhook ──▶ reserved route ──▶ 501 not-implemented marker (seam only; full flow → F10.8)
        ▼
   client tap {"type":"tap","card":id} ──▶ EchoGames.Web.Socket ──▶ EchoGames.Session ──apply──▶ EchoGames.Game (pure)
                                                                              │  (authoritative: deck, combo, score, timers)
                                                          push authoritative state ◀──────────────┘
        (the client renders pushed state and computes NO score; the score lives on the BEAM — the payments precondition)
```

The four boundaries: **(1)** `echo_games` does not depend on `:portal`/`:portal_web` — no Portal in its `mix.exs`, names
no `Portal.*` — so the game runs whether or not Portal runs. **(2)** the games server is a **new** Bandit listener on its
own port and supervisor — never `portal_web`. **(3)** the score is **server-authoritative** in `EchoGames.Game`/
`Session`; the client renders pushed state — the precondition a paid game requires. **(4)** the bot launches the game by
sending a Telegram Game URL the games server serves; `echo_games` and `echo_bot` carry no compile-time dependency on each
other.

### Build notes (umbrella gotchas, carried from the F5/F6 arc)

- **`TMPDIR=/tmp mix test`** — the harness tmp overlay can hit ENOSPC and surface as spurious mid-suite ExUnit I/O
  failures; run the suite with `TMPDIR=/tmp`. (Recorded in the F5/F6 retrospectives.)
- **The determinism gate is a repeated full-suite loop, not a single green run** — for any process-touching suite (the
  session `GenServer`, the WebSocket round-trip), run `for i in $(seq 1 150); do TMPDIR=/tmp mix test || break; done`
  and treat the independent loop, not the implementer's single run, as the gate.
- **The `Plug.Static` mount is verified by a PROBE, never a config-read** — `curl` the served bundle path and assert
  `200` + the expected content type. F6.5.5's brief asserted a mount served `/assets/*` from the config, but the
  as-built `curl` returned `404` (the `at:` prefix-strip latent). Probe the claim.
- **The games server is a NEW server on its own port, never `portal_web`** — `portal_web`'s endpoint runs on `:4000`;
  the games server uses a different config-resolved port and a separate Bandit listener under `EchoGames.Application`. It
  adds no route or child to `portal_web`.
- **No Portal at F10.3** — `echo_games` names no `Portal.*` function and does not depend on `:portal`; the Portal
  consumer integration is a later optional rung (F10.9). The full Telegram-payments flow is F10.8; F10.3 opens only the
