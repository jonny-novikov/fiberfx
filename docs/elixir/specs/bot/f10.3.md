# F10.3 · The minimal valuable game: `echo_games` + Svelte/Vite + WebSocket + the payments seam

> Scaffold a new standalone umbrella app `apps/echo_games` as the **server-authoritative** backend for "Эмоджи: Найди
> пару" — the rewrite of `html/game.html` — holding the game logic, the per-session state, the scoring, the WebSocket

> supervises `portal` or `portal_web`.

## Goal

After F10.3, the first playable, monetizable slice runs end to end. A bot launches the game; a browser opens the
**Svelte 5 (Runes) + Vite** frontend served by a **new, isolated Elixir games web server** (`Plug.Router` on Bandit,
recommended option B); the player taps cards; each tap is an intent sent over a **WebSocket** to `echo_games`, which holds
the **authoritative** game state (the deck, the flipped/matched cards, the combo, the per-level score, the timers) and
**pushes** the updated state back; the score the player sees is computed on the BEAM, not in the browser, so it cannot be
forged client-side. The game lands as a **new umbrella app** `apps/echo_games` (OTP app `:echo_games`, namespace
`EchoGames`) with its **own** supervision tree and the games web server's listener under it. The Telegram-**payments
seam** is opened: the server-authoritative score is the precondition for a paid outcome, and a **reserved**
payment-webhook route exists on the games server — the full payment flow is deferred to F10.8. The umbrella gains a
fifth app (`:echo_games`, alongside `:echo_data`, `:portal`, `:portal_web`, `:echo_bot`) and a fourth application that
boots a supervision tree. The two Portal apps are left exactly as F6 left them: `echo_games` does not depend on them,
does not join `Portal.Application`, and touches no file under `apps/portal/` or `apps/portal_web/`. The games server is a
separate Bandit listener on its **own** port — never a `portal_web` route or child.

## Rationale (5W)

- **Why** — A bot engine earns its keep when it ships something a learner plays and can pay for. "Эмоджи: Найди пару"
  already exists as a single-file browser game (`html/game.html`) where all scoring is client-side — which means a score
  can be forged, so the game cannot be monetized. Moving the **authoritative** state and scoring onto the BEAM
  (`echo_games`) and making the frontend a view over **server-pushed** state is the precondition a paid game requires.
  The Svelte 5/Vite rewrite preserves the look and feel while splitting authority from presentation. The games web server
  is a **new, isolated** server (not `portal_web`) so the games surface and the Portal surface never couple. Opening the
  payments **seam** now — a server-authoritative score and a reserved webhook route — places the integration point
  correctly before the full flow lands (F10.8).
- **What** — a standalone `apps/echo_games` umbrella app with its own supervision tree; the **server-authoritative**
  game model (deck, per-session state, the scoring rules ported exactly from `html/game.html`); a **WebSocket protocol**
  carrying card-tap intents up and authoritative state down; the **Svelte 5 (Runes) + Vite** frontend rewrite producing a
  static bundle; a **new isolated games web server** (recommended `Plug.Router` on Bandit + `WebSock`/`WebSockAdapter`)
  serving the bundle over an explicit `Plug.Static` mount and carrying the socket; a bot **launching** the game via a
  Telegram **Game**; and the **payments seam** opened (server-authoritative score + a reserved payment-webhook route),
  the full flow deferred.
- **Who** — Learners (play a real game launched from a bot, with the score live over the socket), Operators (run the
  game server alongside the bot engine, isolated from Portal), and Developers (the server-authoritative split and the
  WebSocket protocol that the payments flow later rides).
- **When** — the third rung of the F10 value ladder; it builds on the engine skeleton (F10.1) and multi-bot (F10.2) for
  the launch, and opens the path to the full Telegram-payments flow (F10.8), a paid in-game outcome, and the optional
  Portal-consumer integration (F10.9).
- **Where** — a **new umbrella app `apps/echo_games`** (OTP app `:echo_games`, namespace `EchoGames`, `mod:
  {EchoGames.Application, []}`): `apps/echo_games/mix.exs`, `apps/echo_games/lib/echo_games/application.ex`, the game
  model + scoring, the per-session server, the WebSocket handler, the games web server (`Plug.Router` + the `Plug.Static`
  bundle mount + the reserved payment-webhook route), the Vite/Svelte frontend under
  `apps/echo_games/assets/` building to `apps/echo_games/priv/static/`, and the `echo_games` config block in the umbrella
  `config/config.exs` (+ `config/runtime.exs` for the port, `config/test.exs`). The game-launch handler is an `echo_bot`
  surface (a new bot command), pinned but re-grounded against F10.2 (RECONCILE). No file under `apps/portal/` or
  `apps/portal_web/` is touched; `apps/echo_games/mix.exs` does **not** list `{:portal, in_umbrella: true}` or `{:portal_web,
  in_umbrella: true}`.

## Scope

- **In** — the new standalone app `apps/echo_games` and its `EchoGames.Application`; the **server-authoritative** game
  model and scoring (the deck of emoji pairs, the three levels sized `4×3 → 4×4 → 6×4`, the combo/penalty/time-bonus
  rules and the per-level star thresholds, all ported from `html/game.html`); a **per-session** game server (one process
  per active session) holding the deck, the flipped/matched cards, the combo and best combo, the pairs found, the
  per-level score breakdown, and the timers; the **WebSocket protocol** (a card-tap intent up; the authoritative state
  down); the **Svelte 5 (Runes) + Vite** frontend rewrite of `html/game.html` building to a static bundle; the **new
  isolated games web server** (recommended `Plug.Router` on Bandit + `WebSock`/`WebSockAdapter`) serving the bundle over
  an explicit `Plug.Static` mount and upgrading the WebSocket route; a bot **launching** the game via a Telegram **Game**
  (the launch handler an `echo_bot` surface); the **payments seam** — a server-authoritative score plus a **reserved**
  payment-webhook route returning a not-implemented marker; game-logic unit tests, a WebSocket round-trip test, and a
  static-bundle serve check.
- **Out** — the full Telegram-payments flow — invoices, the pre-checkout and `successful_payment` updates, server-side
  validation, a paid in-game outcome (→ F10.8); the richer Telegram conversational surface (→ F10.4); webhook delivery
  for the bot updater (→ F10.5); replacing the vendored ex_gram internals (→ F10.6); the second-platform adapter (→
  F10.7); any Portal touchpoint — `echo_games` names no `Portal.*` function and adds no child to `Portal.Application` (→
  F10.9, optional); leaderboards, accounts, persistence of scores beyond the session (a later rung); any edit to `portal`
  or `portal_web`, including adding a route or a child to `portal_web` (forbidden — the games server is a NEW server).

## Deliverables

- **F10.3-D1** — the new standalone umbrella app `apps/echo_games`: `apps/echo_games/mix.exs` (app `:echo_games`,
  namespace `EchoGames`, `mod: {EchoGames.Application, []}`, with the umbrella's standard
  `build_path`/`config_path`/`deps_path`/`lockfile` block pointing at `../../` and `elixir: "~> 1.18"`, matching
  `apps/portal_web/mix.exs:8-12`), deps `{:bandit, "~> 1.5"}`, `{:plug, "~> 1.16"}`, `{:websock, "~> 0.5"}`,
  `{:websock_adapter, "~> 0.5"}`, `{:jason, "~> 1.4"}` — all already in `mix.lock` (verified: `bandit 1.11.1`,
  `plug 1.19.2`, `websock 0.5.3`, `websock_adapter 0.5.9`, `jason 1.4.5`). The app does **not** list `{:portal,
  in_umbrella: true}` or `{:portal_web, in_umbrella: true}`; it does not depend on `portal` or `portal_web`. Adding
  `:echo_games` is the umbrella's normal app-discovery (`apps_path: "apps"` in the root `mix.exs:6` discovers any
  `apps/*` directory) plus the per-app config block below; no edit to any other app's `mix.exs` and no edit to the root
  `mix.exs`.
- **F10.3-D2** — the **server-authoritative** game model + scoring (`EchoGames.Game`, a pure module): the deck builder
  (a level's `rows × cols` board of emoji **pairs** drawn from the 16-emoji pool, shuffled), the flip/match resolution
  (a tap reveals a card; two revealed cards either match — both stay face-up — or mismatch and flip back), and the exact
  scoring ported from `html/game.html`: **`BASE_PER_PAIR = 100`** per matched pair, a combo bonus **`COMBO_STEP = 25`**
  per consecutive match (`bonus = (combo - 1) × 25`), a mismatch penalty **`WRONG_PENALTY = 10`**, a **`TIME_BONUS = 5`**
  per remaining second on a timed level, and **stars** (1–3) per level by score threshold. The three levels are
  `{rows: 3, cols: 4, time: 0, reshuffle: 0, star2: 550, star3: 800}`, `{rows: 4, cols: 4, time: 60, reshuffle: 0,
  star2: 1100, star3: 1500}`, `{rows: 4, cols: 6, time: 80, reshuffle: 12, star2: 1700, star3: 2300}` (displayed as
  `4×3 → 4×4 → 6×4` cols×rows; level 2/3 timed; level 3 reshuffles the still-hidden cards every 12 s). The module is a
  **pure function of the state and the intent** — no process, no I/O, no clock-in-core (the timer's elapsed-seconds
  arrives as an argument, the F5.5 boundary-supplied-`at` pattern) — so the scoring is unit-testable and the
  authoritative state cannot be forged client-side.
- **F10.3-D3** — the **per-session game server** (`EchoGames.Session`, a `GenServer`, one process per active session,
  under a `DynamicSupervisor`): holds the authoritative `EchoGames.Game` state for one session, applies a card-tap
  intent through `EchoGames.Game`, advances levels, runs the level timers (and the level-3 reshuffle), and produces the
  state to push to the client. A session crash isolates to that session under the `DynamicSupervisor` and never reaches
  another session or another app's tree. The session is keyed so the WebSocket handler routes a socket's intents to its
  session.
- **F10.3-D4** — the **WebSocket protocol** (`EchoGames` JSON over the socket): the client sends a **card-tap intent**
  (`{"type": "tap", "card": <id>}`) and a **new-game / level intent**; the server pushes the **authoritative state** —
  the visible deck (each card `{id, emoji-if-revealed, matched}`), the HUD (level, score, combo, pairs found / total),
  the timer, and the level/game result (stars, the score breakdown). The contract is **server-authoritative**: the
  client never computes a score, only renders pushed state and sends taps. The protocol's message set is defined here as
  the rung design (a small, versioned, JSON message vocabulary).
- **F10.3-D5** — the **new isolated games web server** (recommended option B — Operator-confirmed): a
  `EchoGames.Web.Router` (`Plug.Router`) run by a **Bandit** listener under `EchoGames.Application`, on its **own** port
  (`config`-resolved, default not `:4000` — `portal_web`'s port), serving (a) the Vite bundle over an **explicit**
  `Plug.Static` mount (carrying the F6.5.5 prefix-strip lesson: the static mount is verified by a PROBE — a `curl`
  against the served asset path returning `200` with the bundle's content type — never by reading the config), (b) the
  **WebSocket upgrade** route via `WebSock`/`WebSockAdapter` to the D4 handler, (c) the game-launch landing the Telegram
  Game opens, and (d) the **reserved** payment-webhook route (D7). It is **not** `PortalWeb.Endpoint`; it adds no route
  or child to `portal_web`.
- **F10.3-D6** — the **Svelte 5 (Runes) + Vite** frontend rewrite of `html/game.html`: a Svelte 5 app under
  `apps/echo_games/assets/` using **Runes** for reactive state and **Vite** for the build, producing a static bundle
  (`index.html` + hashed JS/CSS) into `apps/echo_games/priv/static/` for the games server to serve. The visual design —
  the warm cream/coral/sage palette, the card-flip 3D animation, the HUD, the combo pulse, the timer bar, the result
  modals, the Russian copy — is **preserved** from `html/game.html`; the **logic is moved server-side** so the frontend
  renders pushed state and sends card-tap intents over the WebSocket, computing no score. The Vite build is a
  **build-time** Node toolchain (outside the BEAM); the games server serves only the built static output.
- **F10.3-D7** — the **bot game-launch** + the **payments seam**: a bot (from F10.1/F10.2) gains a command (specced as
  `/play`, re-grounded against F10.2 — RECONCILE) that sends a Telegram **Game**, whose launch URL is served by the games
  server; and the games server carries a **reserved** payment-webhook route (a named `Plug.Router` clause returning a
  not-implemented marker, no payment logic) so the integration point exists and is isolated from the start. The
  precondition for a paid game — a **server-authoritative score** — is established by D2/D3/D4. The full payment flow
  (invoices, `successful_payment`, validation, the paid outcome) is **out of scope** (→ F10.8).
- **F10.3-D8** — tests: game-logic unit tests (the deck builder, the flip/match resolution, the exact scoring and star
  thresholds against the `html/game.html` constants, including the combo bonus, the mismatch penalty, and the time
  bonus); a **WebSocket round-trip** test (a tap intent in → an authoritative-state push out, asserting the server, not
  the client, computed the score); and a **static-bundle serve check** (a PROBE — `curl` the served bundle path → `200`
  with the expected content type, the F6.5.5 prefix-strip guard). No Portal is contacted; the games server runs on its
  own test port.
- **F10.3-D9** — verification: the umbrella boots with `echo_games` under `EchoGames.Application` (a new app supervisor),
  the games server listening on its own port; the Vite bundle is served (probed by `curl`); a WebSocket tap intent yields
  a server-authoritative state push; the scoring matches the `html/game.html` constants exactly; a bot launches the game
  via a Telegram Game (manual demo); the reserved payment-webhook route exists and returns the not-implemented marker; a
  session crash restarts in isolation under the `DynamicSupervisor`; `grep -rE "Portal|PortalWeb" apps/echo_games/lib/`
  returns nothing; `apps/echo_games/mix.exs` lists neither Portal app; `apps/portal/` and `apps/portal_web/` are
  unchanged and the games server adds no route or child to `portal_web`.
