## References

- ex_gram — the Telegram Bot API client and framework: <https://github.com/rockneurotiko/ex_gram> and
  <https://hexdocs.pm/ex_gram>. Read the bot DSL (`command/2`, `context |> answer(...)`), the updater methods
  (`:polling`, `:webhook`, `:noup`), and how the bot and its `ExGram` client are supervised under OTP.
- Telegram Bot API — the update and message shapes the bot answers: <https://core.telegram.org/bots/api>.
- The F10 roadmap — the ladder, the master invariant restated for the bot, and the "thin but robust" gates this rung
  must satisfy: [`./f10.roadmap.md`](f10.roadmap.md). The bot stack decision (ex_gram, in-BEAM) and the as-built error
  set: [`./research.report.md`](research.report.md).
- The closest analog — standing a supervised adapter up as a new umbrella app that joins the tree and proves it is
  alive: [`../phoenix/f6.1.md`](../phoenix/f6.1.md) (F6.1 bootstrapped `apps/portal_web` the same way; F10.1 bootstraps
  `apps/bot`). The sibling driving adapter and its boundary framing: [`../pragmatic/f5.9.md`](../pragmatic/f5.9.md).
- **Upstream contract (do not modify).** The `Portal` facade — the single web-facing surface the bot will call **from
  F10.2 onward** (F10.1 makes no facade call). As built (`apps/portal/lib/portal.ex`) it is a thin `defdelegate`-only
  module exposing, among others, the Enrollment slice the bot needs later: `enroll/2`
  (`{:ok, %Portal.Enrollment.Enrolled{}} | {:error, %Portal.Error{} | term()}`, `enrollment.ex:60-67`),
  `deliver_lesson/2` (`:ok | {:error, %Portal.Error{}}`, `enrollment.ex:74-76`), `courses_of/1`
  (`{:ok, [%Enrolled{}]}`, `enrollment.ex:83-87`), `progress_of/1` (`{:ok, 0}` — structurally zero this rung,
  `enrollment.ex:96-97`). The closed error set is `%Portal.Error{code, message, field}` with `code` in
  `:already_enrolled | :course_not_found | :user_not_found | :lesson_locked | :invalid_progress | :invalid`
  (`apps/portal/lib/portal/error.ex:22-28`) — **six** codes as built (the roadmap text predates the F6.4
  `:user_not_found` and the changeset-bridge `:invalid`; cite the code, not the roadmap prose). The supervision tree as
  built is two app supervisors: `Portal.Application` over `[Portal.Repo, Portal.Store, Portal.EventStore.adapter(),
  Portal.Engine]` (`apps/portal/lib/portal/application.ex:33-38`) and `PortalWeb.Application` over `[PortalWeb.Telemetry,
  PortalWeb.Endpoint]`, `max_restarts: 300` (`apps/portal_web/lib/portal_web/application.ex:13-22`). F10.1 adds a
  **third** app supervisor, `Bot.Application`, in a new `apps/bot`; the two existing app supervisors are unchanged.
- Identity & ids (a forward seam, NOT F10.1). The `TGL` chat→learner binding and `Portal.ID.new("TGL")` mints arrive in
  F10.2 (`apps/portal/lib/portal/id.ex` mints `namespace <> Base62(snowflake)`); F10.1 mints no id and resolves no
  identity.
- Spec system and the master invariant: [`../specs.approach.md`](../specs.approach.md).

## Requirements

- **F10.1-R1** — a new umbrella app `apps/bot` exists: `apps/bot/mix.exs` declares `app: :bot`, `mod:
  {Bot.Application, []}`, deps `{:portal, in_umbrella: true}` + `{:ex_gram, ...}` (and an HTTP client + `jason` if
  ex_gram needs them). ex_gram is listed ONLY here; `apps/portal/mix.exs` and `apps/portal_web/mix.exs` are unchanged.
  [US: F10.1-US2, F10.1-US3]
- **F10.1-R2** — `Bot.Application.start/2` supervises the ex_gram children (`ExGram` + the `Portal.Bot` updater),
  `strategy: :one_for_one`. The updater method is read from config (`:polling` in dev/`config/runtime.exs`, `:noup` in
  `config/test.exs`); the `:bot` → `:portal` app dependency orders the boots. `Portal.Application` and
  `PortalWeb.Application` are not touched. [US: F10.1-US2, F10.1-US4]
- **F10.1-R3** — `Portal.Bot` is an ex_gram handler matching `command "start"` and `command "help"`, answering static
  text with `context |> answer(text)` — a welcome string for `/start`, a help string for `/help`. The handler names no
  engine, context, repo, or facade function at F10.1. [US: F10.1-US1, F10.1-US3]
- **F10.1-R4** — handling the same update twice yields the same single reply and no duplicate effect; the handler is a
  pure function of the update (no accumulating state, no facade write). [US: F10.1-US5]
- **F10.1-R5** — the bot token is runtime configuration read at boot (an environment variable in
  `config/runtime.exs`), never compiled in; tests need no token because `:noup` contacts no Telegram. [US: F10.1-US2]
- **F10.1-R6** — handler tests run the updater as `:noup` and feed a constructed `/start` (and `/help`) update to the
  handler, asserting the rendered reply text; no live Telegram is contacted. [US: F10.1-US1, F10.1-US5]
- **F10.1-R7** — no module under `apps/bot/lib/` contains the strings `Portal.Engine`, `Repo`, `GenServer.call`, or a
  `Portal.Enrollment`/`Portal.Catalog`/`Portal.Accounts` context name (the master invariant, statically checkable; at
  F10.1 the handler also names no `Portal.` facade function). [US: F10.1-US3]
- **F10.1-R8** — killing the `Portal.Bot` updater process restarts it under `Bot.Application`; a later update is
  answered, and the two existing app supervisors are undisturbed. [US: F10.1-US4]

## Execution topology

**Runtime topology (what exists and how an update flows).**

```text
:portal  app  Portal.Application (Supervisor, :one_for_one)   # apps/portal — UNCHANGED by F10.1
├─ Portal.Repo · Portal.Store · Portal.EventStore.adapter() · Portal.Engine

:portal_web app  PortalWeb.Application (Supervisor, :one_for_one, max_restarts: 300)   # apps/portal_web — UNCHANGED
├─ PortalWeb.Telemetry · PortalWeb.Endpoint

:bot app  Bot.Application (Supervisor, :one_for_one)   # apps/bot — NEW, dep {:portal, in_umbrella: true}
├─ ExGram                              # F10.1 — the Telegram client process
└─ Portal.Bot (the updater)            # F10.1 — :polling (dev) | :noup (test); boots after :portal via the app dep
        │  (update: /start, /help)
        ▼
   ExGram updater → Portal.Bot.handle/2  ──command "start"──▶  context |> answer(welcome_text)
                                         └─command "help" ──▶  context |> answer(help_text)
        (no facade call at F10.1 — static text only; the seam for F10.2+)
```

The boundary is an **app-level dependency arrow** (`:bot` → `:portal`): the handler will reach the domain only through
the `Portal` facade in later rungs, never `Portal.Engine`. At F10.1 the handler makes no facade call, so the only path
is update → static reply.

**Task topology (build order; each step leaves the umbrella compiling).**

```text
T1 create apps/bot (mix.exs: :bot, dep {:portal,…} + {:ex_gram,…}, mod Bot.Application) + fetch deps
      └─▶ T2 ex_gram config: token from env (runtime.exs), updater :polling (dev) / :noup (test)   [R1, R5]
            └─▶ T3 Bot.Application (:one_for_one over [ExGram, {Portal.Bot, …}])                   [R2]
                  └─▶ T4 Portal.Bot handler: command "start" / command "help" → answer static text [R3, R4]
                        └─▶ T5 :noup handler test (constructed /start, /help → asserted reply)      [R6]
                              └─▶ T6 verify: boot, reply, idempotent resend, updater restart, grep  [R7, R8]
```

**Touched files.** New app `apps/bot/`: `apps/bot/mix.exs`, `apps/bot/lib/bot/application.ex`,
`apps/bot/lib/portal/bot.ex` (the `Portal.Bot` handler), `apps/bot/test/portal/bot_test.exs`,
`apps/bot/test/test_helper.exs`, plus the ex_gram config in the umbrella `config/config.exs`, `config/runtime.exs`, and
`config/test.exs` (token + updater keys under `:ex_gram` / `:bot`). No file under `apps/portal/` or `apps/portal_web/`
is touched; `apps/portal/mix.exs` and `apps/portal_web/mix.exs` stay bot-free.

## Agent stories

Each agent story implements one or more user stories and closes on a gate. Build them in task-topology order.

### F10.1-AS1 — Stand the bot up as a supervised app [implements F10.1-US2, F10.1-US3]
Directive: create `apps/bot` with `mix.exs` (`:bot`, `mod: {Bot.Application, []}`, deps `{:portal, in_umbrella: true}` +
`{:ex_gram, ...}`) and fetch deps (T1); configure the token from env and the updater (`:polling` dev / `:noup` test) in
the umbrella config (T2); build `Bot.Application` supervising `[ExGram, {Portal.Bot, ...}]`, `:one_for_one` (T3).
Acceptance gate: `mix compile` clean; the umbrella boots with a third app supervisor `Bot.Application`;
`Portal.Application` and `PortalWeb.Application` are unchanged; ex_gram appears only in `apps/bot/mix.exs`.

### F10.1-AS2 — Answer /start and /help [implements F10.1-US1]
Directive: build `Portal.Bot` matching `command "start"` and `command "help"`, each answering static text via
`context |> answer(text)` (T4).
Acceptance gate: a `/start` update fed via `:noup` answers the welcome text; a `/help` update answers the help text; the
handler makes no facade call.

### F10.1-AS3 — Harness without Telegram and stay idempotent [implements F10.1-US1, F10.1-US5]
Directive: write a `:noup` handler test that feeds constructed `/start` and `/help` updates and asserts the rendered
reply (T5); assert that feeding the same update twice yields the same single reply (R4).
Acceptance gate: the `:noup` test passes without contacting Telegram; a redelivered update produces the same reply and
no duplicate effect.

### F10.1-AS4 — Self-heal and hold the boundary [implements F10.1-US3, F10.1-US4]
Directive: confirm `:one_for_one` restarts the `Portal.Bot` updater; confirm the master-invariant search is empty (T6).
Acceptance gate: killing the updater restarts it and a later update is answered; `grep -rE
"Portal\.Engine|Repo|GenServer\.call|Portal\.(Enrollment|Catalog|Accounts)" apps/bot/lib/` returns nothing; the two
existing app supervisors are undisturbed.

## Execution plan — first two stories

A worked trace of how the framework turns a user story into a verified change, to confirm the path is short and leaves
no decision to the agent that the spec has not already fixed. Each plan is *story → agent story → tasks → files →
command → gate*; an agent runs the steps top to bottom and stops at the gate.

### Plan A — F10.1-US2 (run the bot as a supervised umbrella app) · via F10.1-AS1
1. Tasks T1 → T2 → T3 (new app → ex_gram config → supervisor).
2. Files: `apps/bot/mix.exs` (+ ex_gram), `config/config.exs`, `config/runtime.exs`, `config/test.exs`,
   `apps/bot/lib/bot/application.ex`.
3. Command: `mix deps.get && mix compile`, then `iex -S mix` and inspect the supervision tree for `Bot.Application`.
4. Gate (closes US2): the umbrella boots with `Bot.Application` (third app supervisor) over `[ExGram, {Portal.Bot, …}]`,
   `:one_for_one`; the `:bot` tree boots after `:portal`; `Portal.Application`/`PortalWeb.Application` unchanged.
   Requirements satisfied: F10.1-R1, F10.1-R2, F10.1-R5.

### Plan B — F10.1-US1 (answer from Telegram) · via F10.1-AS2 + F10.1-AS3
1. Tasks T4 → T5 (handler → `:noup` test).
2. Files: `apps/bot/lib/portal/bot.ex`, `apps/bot/test/portal/bot_test.exs`, `apps/bot/test/test_helper.exs`.
3. Command: `TMPDIR=/tmp mix test apps/bot/test/portal/bot_test.exs`, plus a manual live demo against the polling dev
   bot (a real `/start` returns a reply).
4. Gate (closes US1): `/start` via `:noup` → welcome text; `/help` via `:noup` → help text; a redelivered update → the
   same reply (F10.1-R4); `grep -rE "Portal\.Engine|Repo|GenServer\.call|Portal\.(Enrollment|Catalog|Accounts)"
   apps/bot/lib/` is empty (F10.1-R7). Requirements satisfied: F10.1-R3, F10.1-R4, F10.1-R6.

Efficiency check: each story is two-to-five files behind one command and one gate, and every branch (the two commands,
the resend) is already specified — so the agent implements rather than decides.

### Build notes (umbrella gotchas, carried from the F5/F6 arc)

- **`TMPDIR=/tmp mix test`** — the harness tmp overlay can hit ENOSPC and surface as spurious mid-suite ExUnit I/O
  failures; run the suite with `TMPDIR=/tmp`. (Recorded in the F5/F6 retrospectives.)
- **The determinism gate is a repeated full-suite loop, not a single green run** — for any process-touching suite, run
  `for i in $(seq 1 150); do TMPDIR=/tmp mix test || break; done` and treat the independent loop, not the
  implementer's single run, as the gate.
- **No facade call at F10.1** — the handler is static text only; the `Portal` facade surface (`enroll/2`,
  `deliver_lesson/2`, `courses_of/1`, `progress_of/1`) and the closed `%Portal.Error{}` set are the contract F10.2+
  exercises, cited in References so the build does not invent them early.
