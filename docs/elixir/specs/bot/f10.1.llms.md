## Requirements

- **F10.1-R1** — a new standalone umbrella app `apps/echo_bot` exists: `apps/echo_bot/mix.exs` declares `app: :echo_bot`,
  `mod: {EchoBot.Application, []}`, the umbrella's standard `build_path`/`config_path`/`deps_path`/`lockfile`/`elixir`
  block, and deps a YAML parser (e.g. `{:yaml_elixir, "~> 2.9"}`) plus whatever HTTP/JSON deps the vendored ex_gram client
  needs. It does **NOT** list `{:portal, in_umbrella: true}` and names neither Portal app. `apps/portal/mix.exs`,
  `apps/portal_web/mix.exs`, and the root `mix.exs` are unchanged. [US: F10.1-US2, F10.1-US6]
- **F10.1-R2** — a **vendored copy of ex_gram** lives under `apps/echo_bot/vendor/ex_gram/`: the minimal subset (the
  `getUpdates` long-poll + `sendMessage` client, the update/message decoders, the updater shape) copied as owned source,
  plus `apps/echo_bot/vendor/ex_gram/README.md` (provenance, upstream version at vendoring, preserved Beer-Ware license,
  what was taken vs what the wrap adds) and `apps/echo_bot/vendor/ex_gram/CLAUDE.md` (the owned-fork directive:
  modifiable directly, no upstream PR, reachable only through `EchoBot.Platform.Telegram`; carries the prose-discipline
  propagation clause). [US: F10.1-US3]
- **F10.1-R3** — `EchoBot.Application.start/2` supervises the engine's children (the loaded single bot's updater process
  plus any client/registry process the adapter needs), `strategy: :one_for_one`, name `EchoBot.Supervisor`. It does not
  name, start, or touch `Portal.Application` or `PortalWeb.Application`. [US: F10.1-US2, F10.1-US7]
- **F10.1-R4** — a YAML **v1.0** config loader reads **one** bot definition from a config file (its path resolved from
  application config). It reads the `version` field **first**, validates the rest against the v1.0 schema, and returns
  the five-key definition `version`, `name`, `platform`, `token_env`, `handler`. It resolves `token_env` to the named
  env var's value at boot (the token is NOT in the YAML), selects the adapter from `platform` (`telegram` →
  `EchoBot.Platform.Telegram`), and names the `handler` module. [US: F10.1-US4]
- **F10.1-R5** — a platform-adapter behaviour `EchoBot.Platform` defines the engine's only platform surface — start an
  updater, send a reply, the command/update shape routed on. `EchoBot.Platform.Telegram` implements it by **wrapping the
  vendored ex_gram copy**, with a polling updater selectable as a fake updater for tests. The engine core names the
  behaviour, never Telegram or a vendored module directly; no module outside the adapter names a vendored module. [US:
  F10.1-US5, F10.1-US3]
- **F10.1-R6** — the handler/router answers `/start` with a welcome string and `/help` with a help string, as static
  text. The handler is a pure function of the update; it names no `Portal.*` function and no vendored module (it reaches
  the platform only through the behaviour). [US: F10.1-US1, F10.1-US6]
- **F10.1-R7** — handling the same update twice yields the same single reply and no duplicate effect; the handler is a
  pure function of the update (no accumulating state, no external write). [US: F10.1-US8]
- **F10.1-R8** — the bot token is runtime configuration read at boot from the env var the YAML's `token_env` names; it is
  never compiled in and never written in the YAML. Tests need no token because the fake updater contacts no Telegram.
  [US: F10.1-US4]
- **F10.1-R9** — handler tests run the fake updater and feed a constructed `/start` (and `/help`) update, asserting the
  rendered reply text; a loader test asserts a YAML v1.0 file yields one definition with the five fields (the `version`
  read and validated) and a resolved token. No live Telegram is contacted. [US: F10.1-US1, F10.1-US4, F10.1-US8]
- **F10.1-R10** — no module under `apps/echo_bot/lib/` contains the strings `Portal`, `PortalWeb`, `Portal.Engine`, or a
  Portal context/repo name; `apps/echo_bot/mix.exs` does not list `{:portal, in_umbrella: true}` (the Portal no-touch
  boundary, statically checkable). [US: F10.1-US6]
- **F10.1-R11** — killing the bot's updater process restarts it under `EchoBot.Supervisor`; a later update is answered,
  and the two Portal app supervisors are undisturbed. [US: F10.1-US7]

## Execution topology

**Runtime topology (what exists and how an update flows).**

```text
:portal      app  Portal.Application (Supervisor, :one_for_one, Portal.Supervisor)   # apps/portal — UNTOUCHED by F10.1
├─ Portal.Repo · Portal.Store · Portal.EventStore.adapter() · Portal.Engine

:portal_web  app  PortalWeb.Application (Supervisor, :one_for_one, max_restarts: 300)   # apps/portal_web — UNTOUCHED
├─ PortalWeb.Telemetry · PortalWeb.Endpoint

:echo_bot    app  EchoBot.Application (Supervisor, :one_for_one, EchoBot.Supervisor)    # apps/echo_bot — NEW,
│                                                                                         # NO dep on :portal
├─ (adapter client/registry process, if the adapter needs one)
└─ the one bot's updater       # F10.1 — polling (dev) | fake (test); started from the YAML v1.0 definition
        │
        │  boot: YAML v1.0 file ──▶ EchoBot config loader ──▶ %{version, name, platform, token_env→token, handler}
        │           (version read+validated first)             (platform selects EchoBot.Platform.Telegram)
        ▼
   Telegram polling updater ──update (/start,/help)──▶ handler/router ──command "start"──▶ answer(welcome_text)
   (wraps vendor/ex_gram; fake updater under test)                    └─command "help" ──▶ answer(help_text)
        (static text only at F10.1; no Portal call; the engine core names EchoBot.Platform, never the vendored copy)
```

The three boundaries: **(1)** `echo_bot` does not depend on `:portal` — no Portal in its `mix.exs`, names no `Portal.*`
— so the engine boots whether or not Portal runs. **(2)** the engine core names the platform-adapter behaviour; the
Telegram module implements it, selected by the YAML `platform` field — so a second platform plugs in without editing the
core. **(3)** the vendored ex_gram copy is reached **only** through `EchoBot.Platform.Telegram` — no module outside the
adapter names a vendored module, so the wrap is the long-lived boundary even as the copy's internals are later replaced.

**Task topology (build order; each step leaves the umbrella compiling).**

```text
T1 create apps/echo_bot (mix.exs: :echo_bot, mod EchoBot.Application, YAML+client deps, NO :portal) + deps.get   [R1]
      └─▶ T2 vendor ex_gram subset under vendor/ex_gram/ + README.md (provenance/license) + CLAUDE.md (ownership)  [R2]
            └─▶ T3 platform-adapter behaviour EchoBot.Platform + EchoBot.Platform.Telegram wrapping the vendor   [R5]
                  └─▶ T4 YAML v1.0 loader: read version first, then {name,platform,token_env,handler}, resolve token [R4,R8]
                        └─▶ T5 EchoBot.Application (:one_for_one over the loaded bot's updater + adapter process)   [R3]
                              └─▶ T6 handler/router: command "start"/"help" → answer static text                   [R6,R7]
                                    └─▶ T7 tests: loader unit test + fake-updater handler test (/start,/help,resend) [R9]
                                          └─▶ T8 verify: boot, reply, resend, restart, vendor-wrap grep, no-Portal grep [R10,R11]
```

**Touched files.** New app `apps/echo_bot/` only, plus the umbrella's shared config:
- `apps/echo_bot/mix.exs`
- `apps/echo_bot/vendor/ex_gram/` — the vendored subset, plus `apps/echo_bot/vendor/ex_gram/README.md` and
  `apps/echo_bot/vendor/ex_gram/CLAUDE.md` (the two governance docs)
- `apps/echo_bot/lib/echo_bot/application.ex`
- `apps/echo_bot/lib/echo_bot/config.ex` (the YAML v1.0 loader)
- `apps/echo_bot/lib/echo_bot/platform.ex` (the platform-adapter behaviour)
- `apps/echo_bot/lib/echo_bot/platform/telegram.ex` (the Telegram adapter: wraps the vendored copy; polling + fake updater)
- `apps/echo_bot/lib/echo_bot/bot.ex` (the handler/router seam) and the one bot's handler module (e.g.
  `apps/echo_bot/lib/echo_bot/bots/hello_bot.ex`)
- `apps/echo_bot/priv/bots/hello_bot.yaml` (or `config/`-resolved path) — the one bot definition (YAML v1.0)
- `apps/echo_bot/test/test_helper.exs`, `apps/echo_bot/test/echo_bot/config_test.exs`,
  `apps/echo_bot/test/echo_bot/bot_test.exs`
- the `echo_bot` config block in the umbrella `config/config.exs`, `config/runtime.exs`, `config/test.exs` (the config
  file path; the updater method per env: polling in dev, fake in test).

No file under `apps/portal/` or `apps/portal_web/` is touched. No other app's `mix.exs` and no root `mix.exs` changes
(the umbrella's `apps_path: "apps"` discovers `apps/echo_bot` automatically). The exact module names above are this
brief's proposal; keep the `EchoBot` namespace, the vendored copy under `apps/echo_bot/vendor/ex_gram/`, the
platform-adapter / config-loader split, and follow the spec's invariants.

### Build notes (umbrella gotchas, carried from the F5/F6 arc)

- **`TMPDIR=/tmp mix test`** — the harness tmp overlay can hit ENOSPC and surface as spurious mid-suite ExUnit I/O
  failures; run the suite with `TMPDIR=/tmp`. (Recorded in the F5/F6 retrospectives.)
- **The determinism gate is a repeated full-suite loop, not a single green run** — for any process-touching suite, run
  `for i in $(seq 1 150); do TMPDIR=/tmp mix test || break; done` and treat the independent loop, not the implementer's
  single run, as the gate.
- **No Portal, no games at F10.1** — the engine names no `Portal.*` function and does not depend on `:portal`; the game
  (`echo_games`, Svelte/Vite, WebSocket, payments) is F10.3. The Portal facade and the closed `%Portal.Error{}` set are a
  later optional consumer's contract (F10.9), cited in References so the build does not pull them in early.
- **Wrap the vendored copy; keep F10.1 small** — F10.1 vendors only the minimal ex_gram subset (long-poll `getUpdates`,
  route two commands, `sendMessage`) and wraps it behind the adapter port. The platform-adapter behaviour and the wrap
  boundary are `echo_bot`'s; later rungs replace the vendored internals behind the unchanged port (F10.6).
