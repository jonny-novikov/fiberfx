# F10.1 · Start thin: a bot that answers

> Stand a supervised ex_gram Telegram bot up inside the Portal umbrella as a driving adapter: a `Portal.Bot` handler
> that replies to `/start` and `/help` with static text, polling Telegram in dev and harnessed without Telegram by the
> `:noup` updater in tests. The handler establishes the driving-adapter seam — the place a later rung calls only the
> `Portal` facade — and proves the supervision and update loop on day one, with nothing below the facade changed.

## Goal

After F10.1, the Portal answers from Telegram. A `/start` or `/help` update travels through ex_gram's updater, into a
supervised `Portal.Bot` handler, and back out as a static reply composed with the context DSL (`context |> answer(...)`).
The bot lands as a **new umbrella app** `apps/bot` (OTP app `:bot`, namespace `Portal.Bot`) that depends on
`{:portal, in_umbrella: true}`, so the supervision tree gains a **third app supervisor**: `Bot.Application` owns the
ex_gram children (`ExGram` plus the `Portal.Bot` updater), while `Portal.Application` (the three F5 domain children) and
`PortalWeb.Application` (telemetry + endpoint) are untouched. The bot is a sibling of the F6 LiveView and the F5.9
`EnrollmentLive` sketch — a driving adapter over the same boundary. At F10.1 the handler answers with static text and
makes **no** facade call; the seam it establishes is that every later command path will reach the domain only through
`Portal` and render only the closed `%Portal.Error{}` set. The boundary stays an app-level, compiler-enforced
dependency arrow: ex_gram lives only in `apps/bot/mix.exs`, so the domain apps stay bot-free and the master invariant
cannot be broken by accident.

## Rationale (5W)

- **Why** — The Portal serves the web (F6) but cannot yet be reached from Telegram, where the first learners are. The
  platform needs a chat front door, and it must be added without disturbing the engine or the boundary that protects
  it. Starting thin — a bot that answers static text — proves the supervision and the update loop end to end before any
  domain behaviour rides on it.
- **What** — a supervised ex_gram bot wired into the umbrella as a new app, replying to `/start` and `/help` with static
  text, polling in dev and harnessed by the `:noup` updater in tests, with the update lifecycle visible from update to
  rendered reply.
- **Who** — Operators (run a live bot and watch it answer), Developers (stand the adapter up and keep the domain core
  untouched), and the future Learner (reachable from chat once the command paths land in F10.2+).
- **When** — the first rung of the F10 value ladder; it depends only on the shipped F6 Portal (the `Portal` facade and
  the supervised engine), nothing else. Identity (F10.2), browse (F10.3), enroll (F10.4), and learn/progress (F10.5)
  build on this seam.
- **Where** — a **new umbrella app `apps/bot`** (OTP app `:bot`, namespace `Portal.Bot`, dep
  `{:portal, in_umbrella: true}`): `apps/bot/mix.exs`, `apps/bot/lib/bot/application.ex`, the bot module
  `apps/bot/lib/portal/bot.ex`, and config for the ex_gram token + updater under the `:bot` (and `:ex_gram`) keys in the
  umbrella `config/config.exs` + `config/runtime.exs`. No file under `apps/portal/` or `apps/portal_web/` is touched.

## Scope

- **In** — the ex_gram dependency in the new app; the supervised `Bot.Application` (the `ExGram` child plus the
  `Portal.Bot` updater started in `:polling` for dev and selectable as `:noup` for tests); the `Portal.Bot` handler
  matching `/start` and `/help` and answering static text via the context DSL; idempotent handling of a re-delivered
  update; the `:noup` test posture (constructed updates fed to the handler, the rendered reply asserted); verification
  of boot, a live reply, idempotency, and bot restart.
- **Out** — chat→learner identity and the `TGL`-namespaced binding (→ F10.2); any facade call — `enroll/2`,
  `deliver_lesson/2`, `courses_of/1`, `progress_of/1` (→ F10.3–F10.5); rendering the `%Portal.Error{}` set (→ F10.4);
  inline keyboards beyond plain-text replies (→ F10.3); event-driven notifications over PubSub (→ F10.6); rate limits
  and retries (→ F10.7); webhook delivery (→ F10.8); the off-BEAM scale-out seam (→ F10.9).

## Deliverables

- **F10.1-D1** — the new umbrella app `apps/bot`: `apps/bot/mix.exs` (app `:bot`, namespace `Portal.Bot`, deps
  `{:portal, in_umbrella: true}` + `{:ex_gram, ...}`, `mod: {Bot.Application, []}`), depending on `:portal` so the
  umbrella boots the domain tree before the bot tree. ex_gram is listed **only** here, so `apps/portal/mix.exs` and
  `apps/portal_web/mix.exs` stay bot-free.
- **F10.1-D2** — `Bot.Application.start/2`: a `:one_for_one` supervisor over the ex_gram children — the `ExGram` client
  process and the `Portal.Bot` updater. The updater runs the configured method (`:polling` in dev, `:noup` in test), so
  the choice of updater changes no handler code. A bot-tree crash restarts in isolation and never disturbs
  `Portal.Application` or `PortalWeb.Application`.
- **F10.1-D3** — `Portal.Bot`: an ex_gram handler (`use ExGram.Dsl` / the bot macro) that matches `command "start"` and
  `command "help"` and answers static text with `context |> answer(text)` — the welcome text for `/start`, the help
  text for `/help`. The handler names no engine, store, repo, or facade function at F10.1; it composes replies only
  through the context DSL.
- **F10.1-D4** — idempotent update handling: a re-delivered update (Telegram may resend the same `update_id`) produces
  the same single reply and no duplicate effect. At F10.1 the only effect is an outbound message, so idempotency is
  satisfied by the handler being a pure function of the update — there is no accumulating state to double.
- **F10.1-D5** — the `:noup` test posture: handler tests run the updater as `:noup` (the no-op updates worker, so no
  live Telegram is contacted) and feed a constructed `/start` (and `/help`) update to the handler, asserting the
  rendered reply text. This is the bot's analog of "a wide pure base plus one process test."
- **F10.1-D6** — verification: the umbrella boots with the bot under `Bot.Application`; a `/start` and a `/help` update
  fed via `:noup` produce the welcome and help text; a re-delivered update produces the same reply (idempotent);
  killing the `Portal.Bot` updater process triggers a supervised restart and a later update is answered again; a live
  manual demo (a real Telegram message to the polling dev bot returns a reply).

## Invariants

- **F10.1-INV1 (master, seam stated)** — `apps/bot/lib/` calls **only** the `Portal` facade and renders **only** the
  closed `%Portal.Error{}` set; no module under `apps/bot/lib/` names `Portal.Engine`, a context module, a repo, or
  `GenServer.call` against the domain. At F10.1 the handler makes **zero** facade calls (static replies only), so the
  invariant holds vacuously and the seam is established for F10.2+ — the bot is a second consumer of the same boundary
  the F6 web reads, adding nothing below the facade.
- **F10.1-INV2 (supervised, isolated)** — the umbrella gains a third app supervisor, `Bot.Application`, `:one_for_one`,
  owning the ex_gram children (`ExGram` + the `Portal.Bot` updater). `Portal.Application` and `PortalWeb.Application`
  are unchanged. A bot crash restarts in isolation and never takes the engine or the endpoint down; the `:bot` →
  `:portal` app dependency orders the boots so the domain tree is ready first. The boundary is this app-level dependency
  arrow, compiler-enforced because ex_gram lives only in `apps/bot/mix.exs`.
- **F10.1-INV3 (harnessed without Telegram)** — the updater is config-selected (`:polling` for dev, `:noup` for tests),
  so the same handler code is exercised with no live Telegram. A handler test feeds a constructed update and asserts the
  rendered reply; the updater choice changes no handler code, the same substitutability the F5.8 event store and the
  F10.8 webhook switch rely on.
- **F10.1-INV4 (idempotent update handling)** — handling the same update twice yields the same single reply and no
  duplicate effect. At F10.1 the handler is a pure function of the update (no state, no facade write), so a re-delivered
  update is harmless by construction; the invariant is named here so later command rungs preserve it.
- **F10.1-INV5 (typed at the edge — reserved)** — the bot will render the closed `%Portal.Error{}` set exhaustively, an
  unhandled code being a gap rather than a silent fall-through. At F10.1 no facade call is made, so no error is rendered;
  the invariant is stated as the contract F10.4 first exercises, naming the as-built closed set
  (`:already_enrolled | :course_not_found | :user_not_found | :lesson_locked | :invalid_progress | :invalid`).

## Definition of Done

- [ ] `mix compile` is clean and the umbrella boots with the bot under `Bot.Application` (a third app supervisor),
      `Portal.Application` and `PortalWeb.Application` unchanged (F10.1-INV2).
- [ ] A `/start` update fed via the `:noup` updater is answered with the welcome text; a `/help` update is answered with
      the help text (F10.1-D3, F10.1-D5).
- [ ] A re-delivered update produces the same single reply and no duplicate effect (F10.1-INV4).
- [ ] Killing the `Portal.Bot` updater process restarts it under `Bot.Application` and a subsequent update is answered
      (F10.1-INV2).
- [ ] No module under `apps/bot/lib/` references `Portal.Engine`, a context module, a repo, or `GenServer.call`; the
      handler makes no facade call at F10.1 (F10.1-INV1).
- [ ] ex_gram appears only in `apps/bot/mix.exs`; `apps/portal/mix.exs` and `apps/portal_web/mix.exs` are unchanged
      (F10.1-INV2).
- [ ] A live demo: a real Telegram message to the polling dev bot returns a reply (F10.1-D6).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f10.1.stories.md`).
