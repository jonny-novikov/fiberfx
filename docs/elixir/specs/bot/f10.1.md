# F10.1 · Start thin: the engine runs one bot from a vendored-fork wrap

> Stand the `echo_bot` engine up as a standalone umbrella app that loads **one** bot from a versioned YAML config file
> (`version: "1.0"`) and runs it under its own supervision tree. The bot polls Telegram through a platform-adapter port
> (Telegram is adapter #1, **wrapping a vendored, owned copy of ex_gram**) and answers `/start` and `/help` with static
> text, harnessed without a live platform by a fake updater in tests. The rung establishes the engine skeleton — the
> vendored-fork wrap, the YAML v1.0 config loader, the platform-adapter behaviour, the handler/router seam, and the
> standalone `EchoBot.Application` — with **no** Portal touchpoint, **no** games, and **no** change to `portal` or
> `portal_web`.

## Goal

After F10.1, a YAML file plus a handler module is a running bot. A `/start` or `/help` update travels from Telegram
through the platform-adapter polling updater, into a supervised handler the YAML named, and back out as a static reply.
The engine lands as a **new umbrella app** `apps/echo_bot` (OTP app `:echo_bot`, namespace `EchoBot`) with its **own**
`EchoBot.Application` and its **own** supervision tree. The Telegram surface is a **vendored, owned copy of ex_gram** —
its low-level client, decoders, and updater shape, carried under `apps/echo_bot/vendor/ex_gram/` with its provenance
`README.md` and ownership `CLAUDE.md` — **wrapped** behind `EchoBot.Platform.Telegram`; the engine core reaches the
vendored code only through that adapter. The umbrella gains a fourth app (`:echo_bot`, alongside `:echo_data`,
`:portal`, `:portal_web`) and a third application that boots a supervision tree (alongside `Portal.Application` and
`PortalWeb.Application`; `:echo_data` is a library app with no `Application`). The two Portal apps are left exactly as F6
left them: `echo_bot` does not depend on them, does not join `Portal.Application`, and touches no file under
`apps/portal/` or `apps/portal_web/`. At F10.1 the engine loads a single bot definition, the handler answers static
text, and the platform-adapter boundary is in place so a second platform plugs in later without editing the core. The
engine boots whether or not Portal is running.

## Rationale (5W)

- **Why** — The platform needs a first-party, vendor-lock-free bot engine that owns its Telegram surface. The decision is
  **wrap-then-own**: rather than hand-roll a client from scratch or depend on the hex package, `echo_bot` carries a
  **vendored, owned copy** of ex_gram's Telegram-API code and wraps it behind the platform-adapter port, so the boundary
  is established from day one and later rungs replace the vendored internals behind the unchanged port on the engine's own
  schedule. The Operator is the ex_gram team, tasked to build next-generation bot modules, so the copy is owned source
  (modifiable directly, no upstream PR). Starting thin — one YAML bot answering static text — proves the engine skeleton
  (the vendored wrap, the config load, the adapter port, supervision, the update loop) end to end before any richer
  behaviour, any game, or any Portal touchpoint rides on it.
- **What** — a standalone `apps/echo_bot` umbrella app with its own `EchoBot.Application`; the **vendored copy of
  ex_gram** plus its `README.md` (provenance + license) and `CLAUDE.md` (ownership directive) governance files; a YAML
  **v1.0** config loader (the `version` field read first) that reads one bot definition; a platform-adapter behaviour
  with a Telegram polling updater (adapter #1) that **wraps the vendored copy**; a handler/router that matches `/start`
  and `/help` and answers static text; the fake-updater test posture, with the update lifecycle visible from update to
  rendered reply.
- **Who** — Operators (run a live bot from a YAML file and watch it answer), Developers (stand the engine skeleton up,
  wrap the vendored copy, and keep Portal untouched), and the future second-platform author (the adapter port is in place
  to extend).
- **When** — the first rung of the F10 value ladder; it depends on nothing but the umbrella itself — not on Portal.
  Multi-bot (F10.2), the minimal valuable game (F10.3, `echo_games` + Svelte/Vite + WebSocket + the payments seam), the
  richer Telegram surface (F10.4), webhook (F10.5), owning the wrap (F10.6), adapter hardening (F10.7), the full payments
  flow (F10.8), and the optional Portal-consumer integration (F10.9) build on this skeleton.
- **Where** — a **new umbrella app `apps/echo_bot`** (OTP app `:echo_bot`, namespace `EchoBot`, `mod:
  {EchoBot.Application, []}`): `apps/echo_bot/mix.exs`, `apps/echo_bot/lib/echo_bot/application.ex`, the config loader,
  the platform-adapter behaviour, the Telegram adapter wrapping the vendored copy, the vendored tree under
  `apps/echo_bot/vendor/ex_gram/` (with its `README.md` + `CLAUDE.md`), the handler/router, the one bot's handler module,
  the YAML config file, and the engine config block in the umbrella `config/config.exs` (+ `config/runtime.exs`,
  `config/test.exs`). The umbrella `config/*` edits are umbrella-level, never `portal`/`portal_web` code. No file under
  `apps/portal/` or `apps/portal_web/` is touched, and `apps/echo_bot/mix.exs` does **not** list `{:portal, in_umbrella:
  true}`.

## Scope

- **In** — the new standalone app `apps/echo_bot` and its `EchoBot.Application`; the **vendored ex_gram copy** under
  `apps/echo_bot/vendor/ex_gram/` (the minimal subset needed for `getUpdates` long-poll + `sendMessage` — the low-level
  client, the update/message decoders, the updater shape) plus its `README.md` and `CLAUDE.md`; the engine's
  dependencies (a YAML parser plus whatever HTTP/JSON deps the vendored client needs); a YAML **v1.0** config loader that
  reads the `version` first and then **one** bot definition (`name`, `platform`, `token_env`, `handler`); a
  platform-adapter behaviour and a Telegram implementation that **wraps the vendored copy** with a polling updater
  (selectable as a fake updater for tests); a handler/router matching `/start` and `/help` and answering static text;
  idempotent handling of a re-delivered update; the fake-updater test posture (constructed updates fed to the handler,
  the rendered reply asserted); verification of boot, a live reply, idempotency, and bot restart.
- **Out** — loading more than one bot / a config directory (→ F10.2); the minimal valuable game — `apps/echo_games`, the
  Svelte 5/Vite rewrite of `html/game.html`, the isolated games web server, WebSocket live state, and the payments seam
  (→ F10.3); the richer Telegram surface — free-text messages, inline keyboards, formatting (→ F10.4); webhook delivery
  (→ F10.5); replacing the vendored internals with first-party code behind the unchanged port (→ F10.6); proving the port
  with a second platform (→ F10.7); the full Telegram-payments flow (→ F10.8); any Portal touchpoint — the engine names
  no `Portal.*` function and adds no child to `Portal.Application` (→ F10.9, optional); robustness, rate limits, and
  notifications (→ F10.10).

## Deliverables

- **F10.1-D1** — the new standalone umbrella app `apps/echo_bot`: `apps/echo_bot/mix.exs` (app `:echo_bot`, namespace
  `EchoBot`, `mod: {EchoBot.Application, []}`, with the umbrella's standard
  `build_path`/`config_path`/`deps_path`/`lockfile` block pointing at `../../` and `elixir: "~> 1.18"`), deps a YAML
  parser plus whatever HTTP/JSON deps the vendored ex_gram client needs. The app does **not** list `{:portal,
  in_umbrella: true}`; it does not depend on `portal` or `portal_web`. Adding `:echo_bot` to the umbrella is the
  umbrella's normal app-discovery (`apps_path: "apps"` already discovers any `apps/*` directory) plus the per-app config
  block below; no edit to any other app's `mix.exs` and no edit to the root `mix.exs` is required.
- **F10.1-D2** — the **vendored ex_gram copy** under `apps/echo_bot/vendor/ex_gram/`: a copy of the minimal subset of
  ex_gram's Telegram-API code needed at this rung — the low-level client (`getUpdates` long-poll + `sendMessage`), the
  update/message decoders, and the updater shape — vendored as owned source. Two governance documents accompany it
  (the builder authors them):
  - `apps/echo_bot/vendor/ex_gram/README.md` — states what the directory is and where it came from: a vendored copy of
    ex_gram ([github.com/rockneurotiko/ex_gram](https://github.com/rockneurotiko/ex_gram)), its upstream version at the
    time of vendoring, the original Beer-Ware license preserved, which parts were taken (the client + decoders + updater)
    versus what `echo_bot` adds around them (the `EchoBot.Platform` port and the adapter wrap), and that the wrap — not
    the copy — is the long-lived boundary that later rungs replace internals behind.
  - `apps/echo_bot/vendor/ex_gram/CLAUDE.md` — the agent ownership directive: this is a **vendored fork maintained by
    the ex_gram team**; the Operator is the ex_gram team, tasked by the main maintainer to build next-generation bot
    modules; agents **may Read, Write, and modify this code directly** as owned source; changes are **not** submitted as
    upstream PRs to ex_gram main (no fork-back obligation); and the one constraint that survives ownership — the engine
    core reaches this code **only** through `EchoBot.Platform.Telegram`, never directly. It carries the propagation
    clause (no gendered pronouns for agents; no perceptual or interior-state verbs; no first-person narration).
- **F10.1-D3** — `EchoBot.Application.start/2`: a `:one_for_one` supervisor (`EchoBot.Supervisor`) over the engine's
  children — at F10.1, the loaded single bot (its updater process) plus whatever client/registry process the adapter
  needs. The engine reads the bot definition from YAML at boot and starts the supervised bot. A bot crash restarts in
  isolation. `Portal.Application` and `PortalWeb.Application` are not named, not started, and not touched by this tree.
- **F10.1-D4** — the YAML **v1.0** config loader: reads **one** bot definition from a config file (path resolved from
  application config). It reads the `version` field **first** and validates the rest against the matching schema; v1.0's
  five required keys are `version` (`"1.0"`), `name`, `platform`, `token_env`, and `handler`. The loader resolves
  `token_env` to the named environment variable's value at boot (the token is **not** in the YAML), selects the platform
  adapter from `platform` (`telegram` → `EchoBot.Platform.Telegram`), and names the `handler` module that routes the
  bot's commands.
- **F10.1-D5** — the platform-adapter behaviour `EchoBot.Platform`: a behaviour defining the engine's platform surface —
  how to start an updater that delivers updates, how to send a reply, and the command/update shape the engine routes on.
  `EchoBot.Platform.Telegram` implements it by **wrapping the vendored ex_gram copy**, with a **polling** updater
  (long-poll `getUpdates`, `sendMessage` to reply) selectable as a **fake** updater for tests. The engine core names the
  behaviour, never Telegram or the vendored modules directly; no module outside the adapter names the vendored copy.
- **F10.1-D6** — the handler/router seam: a `/start` and `/help` router that the bot's handler module uses to answer
  static text — a welcome string for `/start`, a help string for `/help`. The handler is a pure function of the update;
  it names no Portal function (none exists in this app's deps) and no vendored module (it reaches the platform only
  through the behaviour).
- **F10.1-D7** — idempotent update handling: a re-delivered update (Telegram may resend the same `update_id`) produces
  the same single reply and no duplicate effect. At F10.1 the only effect is an outbound message, so idempotency holds by
  the handler being a pure function of the update — there is no accumulating state to double.
- **F10.1-D8** — the fake-updater test posture: handler tests run the Telegram adapter's **fake** updater (a no-op
  updates source, the ex_gram `:noup` analog, so no live Telegram is contacted) and feed a constructed `/start` (and
  `/help`) update to the handler, asserting the rendered reply text. A loader test asserts a YAML v1.0 file yields a bot
  definition with the five fields (the `version` read and validated) and a resolved token. This is the engine's "wide
  pure base plus one process test."
- **F10.1-D9** — verification: the umbrella boots with `echo_bot` under `EchoBot.Application`; the YAML v1.0 loader
  produces one bot definition (the `version` validated); a `/start` and a `/help` update fed via the fake updater produce
  the welcome and help text; a re-delivered update produces the same reply (idempotent); killing the bot's updater
  process triggers a supervised restart and a later update is answered again; the engine core reaches the vendored copy
  only through `EchoBot.Platform.Telegram` (no module outside the adapter names a vendored module); the vendored tree
  carries its `README.md` and `CLAUDE.md`; `portal` and `portal_web` are unchanged and `apps/echo_bot/mix.exs` does not
  depend on them; a live manual demo (a real Telegram message to the polling dev bot returns a reply).

## Invariants

- **F10.1-INV1 (Portal untouched — the no-touch boundary)** — no file under `apps/portal/` or `apps/portal_web/` is
  modified; `apps/echo_bot/mix.exs` does **not** list `{:portal, in_umbrella: true}` and names neither app; no module
  under `apps/echo_bot/lib/` references `Portal`, `PortalWeb`, `Portal.Application`, `Portal.Engine`, a Portal context,
  or a Portal repo (`grep -rE "Portal|PortalWeb" apps/echo_bot/lib/` returns nothing); `echo_bot` adds **no** child to
  `Portal.Application`. The engine boots whether or not Portal is running. Any future Portal touchpoint (F10.9, optional)
  is `echo_bot` calling Portal's **public facade** as a read-only consumer — never editing Portal — and is out of scope
  here.
- **F10.1-INV2 (standalone, supervised, isolated)** — the umbrella gains a fourth app, `:echo_bot` (the third with a
  supervision tree, after `Portal.Application` and `PortalWeb.Application`), with its own `EchoBot.Application`
  (`:one_for_one`, `EchoBot.Supervisor`) owning the engine's children. `Portal.Application` and `PortalWeb.Application`
  are unchanged and independent. A bot crash restarts in isolation under `EchoBot.Supervisor` and never reaches another
  app's tree.
- **F10.1-INV3 (platform-abstracted — the adapter port)** — the engine core depends only on the platform-adapter
  behaviour, never on a concrete platform. `EchoBot.Platform.Telegram` is one implementation, selected from the YAML's
  `platform` field; the core names the behaviour, so adding a platform later is implementing the behaviour, not editing
  the core. No Telegram-only type or call appears in the engine core.
- **F10.1-INV4 (vendoring / ownership — the wrap boundary)** — the engine core reaches the vendored ex_gram code **only**
  through the `EchoBot.Platform.Telegram` adapter; no module outside the adapter names a vendored module. The vendored
  tree lives under `apps/echo_bot/vendor/ex_gram/` and carries its `README.md` (provenance + preserved Beer-Ware license)
  and `CLAUDE.md` (the owned-fork directive: modifiable directly, no upstream PR, reachable only through the adapter).
  The copy is owned source; the wrap, not the copy, is the long-lived boundary.
- **F10.1-INV5 (YAML-driven — plug-and-play, versioned)** — a bot exists because a YAML **v1.0** file declares it. The
  loader reads the `version` first, validates the rest against the matching schema, reads the definition (`name`,
  `platform`, `token_env`, `handler`), and the engine starts the supervised bot from it; no bot is hard-coded in the
  engine. The token is referenced by env-var name (`token_env`) and read at boot — it is **never** written in the YAML
  and **never** compiled in.
- **F10.1-INV6 (harnessed without a live platform)** — the updater is config/adapter-selected (polling for dev, the fake
  updater for tests), so the same handler code is exercised with no live Telegram. A handler test feeds a constructed
  update and asserts the rendered reply; the updater choice changes no handler code — the same substitutability the F10.5
  webhook switch will rely on.
- **F10.1-INV7 (idempotent update handling)** — handling the same update twice yields the same single reply and no
  duplicate effect. At F10.1 the handler is a pure function of the update (no state, no external write), so a
  re-delivered update is harmless by construction; the invariant is named here so later rungs preserve it.

- [ ] The vendored ex_gram copy lives under `apps/echo_bot/vendor/ex_gram/` with its `README.md` and `CLAUDE.md`; no
      module outside `EchoBot.Platform.Telegram` names a vendored module (F10.1-D2, F10.1-INV4).
- [ ] The YAML v1.0 loader reads the `version` first, then one bot definition (`name`, `platform`, `token_env`,
      `handler`), and resolves the token from the named env var; the token does not appear in the YAML (F10.1-D4,
      F10.1-INV5).
- [ ] A `/start` update fed via the fake updater is answered with the welcome text; a `/help` update is answered with
      the help text (F10.1-D6, F10.1-D8).
- [ ] A re-delivered update produces the same single reply and no duplicate effect (F10.1-INV7).
- [ ] Killing the bot's updater process restarts it under `EchoBot.Supervisor` and a subsequent update is answered
      (F10.1-INV2).
- [ ] The engine core names only the platform-adapter behaviour; `EchoBot.Platform.Telegram` implements it by wrapping
      the vendored copy; no Telegram-only call appears in the core (F10.1-INV3, INV4).
- [ ] No module under `apps/echo_bot/lib/` references `Portal` or `PortalWeb`; `apps/echo_bot/mix.exs` does not list
      `{:portal, in_umbrella: true}`; `apps/portal/` and `apps/portal_web/` are unchanged (F10.1-INV1).
- [ ] A live demo: a real Telegram message to the polling dev bot returns a reply (F10.1-D9).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f10.1.stories.md`).
