## The honest-door contract (the load-bearing behavior — INV3)

`Portal.Auth.sign_in/2` MUST return the IDENTICAL `{:error, :invalid_credentials}` for two distinct failures:

1. the identifier resolves to NO user, and
2. the identifier resolves to a user but the password is wrong.

No branch, message, status, or early-return distinguishes them at the facade — a caller gets ONE bit on success
(`{:ok, %Session{}}`) and ZERO on failure (the same error either way). `SessionController.create/2` maps both
to the SAME HTTP failure (401 or 403 — pick one and use it for both; the page reads `r.status===401 ||
r.status===403`). `Portal.Auth.request_reset/1` ALWAYS returns `:ok`; `SessionController.request_reset/2` always
answers the same 200. Neither endpoint can be used as an account-existence oracle. *(The login page's copy makes
this contract explicit — the hero `.auth-card` hint `login.html:467` "failures never say which half was wrong",
and the reset lede `:457` "the portal answers the same way either way, so an address can not be fished for." The
build realizes that copy. NOTE: the 685L snapshot's standalone `#how` "Two doors, one rule" prose — which once
spelled "one bit away on success and zero on failure" — was folded out by the Operator in the 733L file; the
contract now lives in the hero hint + reset lede.)*

## CSRF for the JS `fetch` POSTs (resolve at build — do not invent a bypass)

The `:browser` pipeline runs `plug(:protect_from_forgery)` (`router.ex:33`), so a `POST /auth/session` from
`login.js`'s `fetch` (`login.html:670`/`:710`, `Content-Type: application/x-www-form-urlencoded`) needs a valid CSRF
token or Phoenix rejects it. Two sanctioned resolutions (pick one, state it; do NOT disable
`protect_from_forgery` globally):
- **(a)** Render the CSRF token into the login page (a `<meta name="csrf-token" content={Plug.CSRFProtection.get_csrf_token()}>`
  in the `<head>`, or a hidden `_csrf_token` input the JS reads) and have `login.js` send it as the
  `x-csrf-token` header / a body field. This keeps `login.js` a relocated-verbatim file PLUS a minimal
  token-read addition — note the addition explicitly (it is the one allowed deviation from "verbatim" for the
  JS, and it is small).
- **(b)** Scope the two `/auth/*` POSTs into a pipeline WITHOUT `:protect_from_forgery` but WITH `:fetch_session`
  (a deliberate carve-out for the form-POST endpoints). This keeps `login.js` byte-verbatim but widens the
  surface — less preferred.
Resolution (a) is recommended (it keeps CSRF on); whichever is chosen, the sign-in cookie write needs
`fetch_session` in the POST's pipeline. **This is a build-time mechanism choice, not a spec fork** — state which
in the build notes.
