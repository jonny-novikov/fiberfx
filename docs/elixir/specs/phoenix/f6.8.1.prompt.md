## The rung in one paragraph

F6.8.1 gives the Portal a real front door. The **static** `elixir/login.html` (733L — operator-edited since spec authoring: the `#how` section folded out, footer → canonical `.site-foot` + clickable build-stamp popover) is ported the F6.5.5
way — a `PageController.login/2` (`render(conn, :login)`, no assigns) rendering a complete-document
`page_html/login.html.heex`, its five `<style>` blocks extracted **verbatim** to `priv/static/assets/login.css`
and its three authored `<script>` blocks to `login.js`, served at `GET /login` with **no layout, no
CoreComponents**. The forms POST to a **NET-NEW `Portal.Auth` facade**: `sign_in/2` →
`{:ok, %Session{}} | {:error, :invalid_credentials}` — the **honest door**, the SAME error for a wrong name
and a wrong password — and `request_reset/1` → always `:ok` (no account enumeration), over the **existing**
`%User{}`/`%Session{}` (F6.4 entities, ns USR/SES) plus the net-new credential/hash + session-mint internals.
A `SessionController` (`POST /auth/session`, `POST /auth/reset`) writes `:user_id` into the F6.1 signed session
cookie; the F6.2 `RequireUser` plug **evolves** into `PortalWeb.UserAuth` (`fetch_current_user/2` +
`require_authenticated_user/2` redirecting to `~p"/login"` + `on_mount/4 :ensure_authenticated` — the named
surface F6.9's `live_session` forward-ref expects). Registration is **deferred**. Triad:
[`f6.8.1.md`](f6.8.1.md) · [`f6.8.1.stories.md`](f6.8.1.stories.md) · [`f6.8.1.llms.md`](f6.8.1.llms.md) ·

## Mode — Flat-L2 lead-team, review gates

Real `Agent` spawns + SendMessage, Director-in-loop — not a `Workflow` fan-out. Risk is **moderate**. The
defining feature is that F6.8.1 is a **HYBRID** rung, and its gate splits accordingly:

- **A static-parity page** (the login document) — judged the **F6.5.5 way**: TEXT-PROXY visual acceptance
  (envelope + clamp-spacing + token-fidelity), **no screenshots**, and the ≥100 determinism loop **does not
  apply to the page** (a static document mints no id, spawns no process — looping it is theatre).
- **A real auth backend below the facade** (`Portal.Auth`, password hashing, the session mint
  `Portal.ID.new("SES")`) — judged the **F6.7 way**: the master invariant + the **≥100 determinism loop
  DO apply**, because the SES mint shares the same-millisecond branded-id collision hazard
  (`echo/CLAUDE.md §4`; `id.ex:13` `@node 1`).

So the gate is: **text-proxy parity for the page + the ≥100 loop for the auth backend + the liveness check for
both.** All peers Opus (LAW-2); the Director coordinates + ratifies (LAW-1a); one commit at close (LAW-4) — and
the **`bcrypt_elixir` dependency** is the one new dep, confirmed at the Stage-5 gate (it is added to
`apps/portal/mix.exs` + `mix.lock` at build, not before).

`f6.8.1.md`'s `## [RECONCILE]` block is **closed** (ratified by the Director this session); Venus does **not**

| **Interim auth** | `RequireUser` (bare `:current_user_id`, redirect `~p"/"`) **evolves** into `PortalWeb.UserAuth` (`current_user` = `%User{}`, redirect `~p"/login"`, the `on_mount` hook); `/my/courses` reconciles to `current_user` (RECONCILE 3, 4). |
| **`live "/courses"`** | stays **PUBLIC**; gating the inline create is deferred with the dynamic-UI pass (RECONCILE 5). F6.8.1 ships the gate, not its application to the catalog. |
| **Persistence** | **RATIFIED = Option A** — Store-backed user + a private `password_hash` + `{:bcrypt_elixir, "~> 3.1"}`, NO schema, NO migration. Ecto `users`/`sessions` (Option B) is a documented alternative, not built. |

Two **build-time mechanism picks** (state which, do not silently choose — neither is a spec fork):

- **CSRF for the JS `fetch` POSTs** (`f6.8.1.llms.md` "CSRF for the JS `fetch` POSTs"). The `:browser` pipeline
  runs `protect_from_forgery` (`router.ex:33`), so the `fetch` POST needs a token. **Recommended (a):** render
  `<meta name="csrf-token" content={Plug.CSRFProtection.get_csrf_token()}>` into the page `<head>` and have
  `login.js` send it as the `x-csrf-token` header — keeps CSRF **on**, the one allowed (small, noted) deviation
  from verbatim JS. (Do NOT disable `protect_from_forgery` globally.)
- **Session persistence detail** — the minted `%Session{}` is held for later lookup (in `Portal.Store` under
  `SES`, or engine-held); the request carries `:user_id` in the signed cookie regardless (so it is cluster-safe
  without server-side session sharing — F6.8.2's libcluster needs no session store).

## Why the prompts are short (the evolved process)

The discipline lives in the agent definitions; the spawn carries only the rung delta. F6.8.1's one process
delta is the **hybrid gate**: it is the FIRST rung that is part static-parity page (F6.5.5's text-proxy
acceptance, the ≥100 loop dropped *for the page*) and part id-minting auth backend (F6.7's master invariant +
the ≥100 loop, applied *to the backend*). Do not collapse the two — looping the static page is theatre, and
skipping the loop on the minting backend misses the same-ms flake. Cheap acceptance — the Given/When/Then
stories Mars + Apollo verify against — is the multiplier.
