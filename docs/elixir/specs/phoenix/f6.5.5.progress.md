## Mentoring Q&A (the Director's questions — answered from this rung's evidence)

**Q1 — Transient vs standing liveness. What failed: the build or the gate?**
The **gate**, not the build. The build is correct: re-booted durably from a non-ephemeral context (the
Director's main session), `:4000/` serves the styled courses index and `:4000/elixir` the styled Elixir index
at parity with Fiber — the Operator independently confirmed it. What failed is the *form* of the liveness
proof: Mars-3 and this verifier each booted `:4000`, curled it green, then KILLED it before the turn ended,
and agent processes are reaped at turn-end — so the Operator's on-demand probe hit a dead port (*"claimed DONE
by Implementors is falsy"*). **A transient gate (boot→curl→kill) proves CAPABILITY** — the node *can* serve;
**a STANDING property** is that the node *is still serving when the acceptor looks*. An Operator-facing "the
Portal lives at :4000" asserts the latter, and capability-proof does not discharge it. The durable fix: leave
the node running from a context that outlives the turn (the Director's session or the deploy), or hand the
Director a one-line boot command. The guardrail it earns sharpens the existing liveness line in `mars.md` +
`apollo.md` (§5, item 1) — this is the F6.6 "green suite, dead route" hazard (there `server: false` never
bound the port) lifted one level to "verified-live gate, dead-at-turn-end node."

**Q2 — Realization over literal. Why was the `<script>` literal wrong and the `<meta>` realization right?**
The brief's default injection was `<script>window.__deepLinkBase = "{PortalWeb.deep_link_base()}";</script>`,
but **HEEx 1.1.31 interprets a bare `{…}` inside a `<script>` BODY as an expression delimiter at compile time**
— so that literal would mis-parse/mis-render rather than emit the resolver value. Mars-3 shipped the
behavior-identical `<meta name="deep-link-base" content={PortalWeb.deep_link_base()}>` (`elixir.html.heex:15`)
— an `{…}` in an ATTRIBUTE, which HEEx evaluates cleanly — and the JS reads it (`elixir-index.js:8-9`,
`(META && META.content) || window.__deepLinkBase`). The brief itself had *sanctioned* the `<meta>` form as the
equal alternative, so this was building to the contract's intent, not around it. What keeps "realization over
literal" from sliding into "build whatever you want" is the **two-part discipline `mars.md:34-37` states**:
(a) the deviation is taken ONLY when the literal would not compile or would breach an invariant — a *forced*
divergence, not a preference; and (b) it is FLAGGED with its citing `file:line` for the verifier. Both held:
the divergence was forced (HEEx parse) and flagged, and the e2e arc-selector test (test 12, green on the
`example.test` override) proves the realization is behavior-identical — the injected value drives the static
JS exactly as the literal would have.

**Q3 — Who is stale, the spec or the code, when a sanctioned realization drifts the spec?**
INV9(a) said the host literal "lives ONLY in `config.exs`," but the as-built (correctly) also carries it at
the resolver default `portal_web.ex:27` — `Application.get_env(:portal_web, :deep_link_base_url, "https://jonnify.fly.dev")`
— the form **D9 / §How.5 / AS8 explicitly PRESCRIBE**. The other verifier read this as a code defect and
proposed deleting the 3rd arg to match the spec's words; the ratified resolution KEPT the prescribed code and
synced the spec wording. **The general principle: the stale party is the one NOT pinned by an authority the
other already cites.** Here the resolver's 3-arg form is pinned by D9/§How.5/AS8 (the spec's own deliverable),
while the "ONLY in config.exs" sentence is an incidental description that predates the prescribed form — so the
*words* are stale, not the code. Deleting a prescribed, idiomatic default to satisfy an incidental sentence
would amend the code to match prose while *breaking* the deliverable that prose was supposed to serve (and add
a `nil`-risk for any env omitting the key). Rule of thumb: **sync prose to a deliverable-pinned realization;
sync code only when the realization itself violates an invariant or a higher authority.** (Echoes `echo/CLAUDE.md`
§7: "the body is authoritative" — but the *deliverable* clause of the body outranks an incidental sentence of
it.)

**Q4 — Config crosses the boundary EXACTLY once. Why one injection, not templated JS or a duplicated base?**
Two surfaces need the host: the server-rendered `<a href>` markup (HEEx, reads config at render) and the
static `elixir-index.js` arc link-builder (a `Plug.Static` file served byte-for-byte, which CANNOT read Elixir
config). Templating the JS would re-introduce a build step (the umbrella has none, by design — F6.5.5 §How.2)
and make the asset non-static; duplicating the base literal into both surfaces re-creates the drift surface the
whole pass exists to remove (change the host in one place, miss the other). The right shape: **read it once on
the server, inject it once into the `<head>` (`elixir.html.heex:15`), and have the static JS read that single
injection** — the injection IS the lone server→static boundary crossing. This is the **runtime analog of
F0-INV2 ("tokens declared once per page, never per component")**: F0-INV2 declares each value once at compile
time in one `:root` block; here the deep-link base is declared once at *render* time in one `<meta>`. Same
single-source-of-truth shape, lifted from static-CSS to a runtime config value. **What made the grep the PROOF
rather than a claim:** INV9(a) is checkable — `grep -rn jonnify.fly.dev echo/apps/portal_web/lib echo/apps/portal_web/priv`
returns ONLY `portal_web.ex:27`+`:21`+`config.exs:46`, ZERO in any asset or `.heex` body. A value that crossed
the boundary twice (a baked literal) would light up the asset or a second template line; "exactly once" is
literally the count the grep returns. The config-swap (`:4001`, `example.test` → both surfaces re-render,
`jonnify.fly.dev`→0) turns "single source" from a structural claim into an exercised one.

**Q5 — The inward false-green. What's the guard, and why is it the "a check counts only if it RUNS" class turned inward?**
The INV1 boundary grep (`Portal.Engine|Repo|GenServer.call|CoreComponents|PortalWeb.Layouts`) fired two false
positives on as-built CONTENT: `CoreComponents` matched a pre-existing F6.5 COMMENT (`portal_web.ex:76`,
naming the `CatalogComponents` import slot), and `GenServer|Snowflake|Supervisor` matched CURRICULUM TITLES in
the rendered `.heex` chapter cards (course lesson names like "OTP: GenServer & supervisors"). The guard: **do
not take the coarse grep's CLEAN/leak verdict — read the matched line, then re-grep comment-excluded and
`import`/`alias`-only.** Both resolved to non-leaks. Why this is the **"a check counts only if it RUNS" class
turned inward on the verifier**: that rule says a check whose result is not actually produced by execution is
worthless (an inert doctest, a `server:false` liveness curl). A grep that emits a verdict on a *coarse pattern
match* — or that would emit CLEAN on a tool error/typo'd pattern — is the same defect from the verifier's side:
the verdict is an artifact of the check's *shape*, not of the property it claims to test. A green grep over the
wrong pattern is exactly as unauditable as a green suite that never bound the port. The fix is symmetric:
make the check actually discriminate (read the hit; tighten the pattern) before vouching its verdict.

**Q6 — Grade-first firewall + "no guardrail owed." Why is "re-fires correctly → no edit" right, and how does the firewall stop mentoring from softening the grade?**
"A guardrail that re-fires correctly earns its place — no edit" is right because **the durable-line test is a
recurring or predictably-repeated FAILURE, not a successful application.** This pass, `mars.md`'s
"Realization over literal" line produced the correct `<meta>` choice, the clamp-lint held by construction, and
`venus.md`'s probe-not-config-read line drove Venus-2 to probe the swap — each fired and HELD. Adding a new
line for a guardrail that is *working* would accrete the stale-advice bloat the next stateless spawn must read
past; the lean-defs discipline says a line earns its place by catching a real, repeatable failure, never by
restating a win. (Contrast the ONE line genuinely owed — the liveness-persistence guardrail — which earned it
by *recurring DESPITE* the F6.6 guardrail, so it SHARPENS that line rather than stacking.) **How the firewall
stops the softening:** mentoring is downstream of the verdict and runs on the verdict's evidence — so the grade
is rendered and reported FIRST (the §4 BUILD-GRADE verdict was written before any §5 finding was drafted;
firewall note above), and once written it does not move for a teachable fix. The structural protection is that
a finding which WOULD change the grade is not a "mentoring note" at all — it is a BLOCKING delta that routes
to Mars through the Director (the grade stays BLOCKED until corrected). So the only things that reach the
mentoring stage are non-blocking by construction; there is nothing grade-moving left to soften. A softer grade
bought by the mentoring relation would forfeit the one thing a verdict is for — being trustworthy independent
of who built it.
