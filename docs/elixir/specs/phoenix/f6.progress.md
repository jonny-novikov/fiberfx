## F6.6 — LiveView (the interactive catalog) · shipped `3cf2480`

**Slice.** `PortalWeb.CatalogLive` makes F6.5's static `/courses` interactive without a reload — live search
(`phx-change` → `Portal.search_courses/1`) and live create (`phx-submit` → `Portal.create_course/1`) over a
`stream`, reusing the F6.5 `course_card`; `live "/courses"` supersedes the static index. One ratified facade
addition: `Portal.search_courses/1` → `Catalog.search_courses/1 :: [Course.t()]` (read-only).

**Team & gate.** Venus (reconcile + brief) → Mars-1 (build) → Mars-2 (harden) → Apollo (verify:
BUILD-GRADE, every promise MATCH). Gate: compile `--warnings-as-errors` clean; echo_data 2 · portal 10
doctests / 4 properties / 86 · portal_web 37; **determinism 100/100** (Apollo) + 30/30 (Director re-run) —
this rung is id-touching (`CRS` mint + `Repo.insert`), so the ≥100 loop is the real gate.

### Went exceptionally well
- **Shift-left caught the `@enforce_keys` trap pre-build.** Venus's lag-1 reconcile corrected the brief's
  `change_course/1` → arity-0 `change_course/0` (a bare `%Course{}` literal under `@enforce_keys` does not
  compile) BEFORE Mars touched code — the F6.5 mid-build correction, retired one rung earlier.
- **The ≥100 determinism loop caught a real §4 PK-collision** the multi-seed run missed: `create_course/1`
  mints outside the single-writer engine and `courses` has no `unique_constraint(:id)`, so two same-ms seq-0
  mints RAISE `Ecto.ConstraintError`. The 100-loop failed at iteration 4 while all five seeds passed — the
  canonical §4 signature. Held for the suite with a per-test `delete_all(Course)`.
- **Independent convergence.** Apollo's spawn and the Director's ratification surfaced the SAME findings (the
  inert doctest, the `@courses`-as-assign drift) from separate contexts — agreement that is evidence, not an
  echo chamber.

### Went well
- **Facade-only held (INV1).** A grep of the CatalogLive code region is clean — no `Portal.Engine` / `Repo`

  the `Portal` facade.

### Post-ship finding — the liveness gate must be RUN, not just documented
The standing liveness criterion (Portal live on `:4000`) was added in `6e4eb79`, AFTER F6.6 shipped
(`3cf2480`), so F6.6's own gate never ran it — and `mix test` runs the endpoint with `server: false`
(`config/test.exs`), so a 100%-green suite never bound `:4000`. With no dev daemon running, `:4000` read as
inaccessible. **Diagnosis: not a code defect** — the node boots clean (Postgres up, `portal_dev` migrated,
`PortalWeb.Endpoint` binds `127.0.0.1:4000`), `/health` → 200, `/courses` renders `CatalogLive`. Root cause:
the check was documented but never executed — the same **"a check counts only if it RUNS"** class as the
inert doctest. Mentoring folded forward (Operator-ratified): `mars.md` runs the liveness check before
reporting done; `apollo.md` re-runs the liveness boot + curl as part of the gate (not just `mix test`);
`f6.7.prompt.md` bakes the liveness gate into Stage 4/5. *(Deploy note for F6.8: the endpoint binds
`127.0.0.1`, not `0.0.0.0` — fine for local dev, but a Fly/clustered deploy needs the broader bind.)*
