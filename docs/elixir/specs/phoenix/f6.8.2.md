## Invariants

- **F6.8.2-INV1 (release-native migration).** Migrations run through `Portal.Release` via `bin/portal eval`, not
  `mix` (the release has none); the schema is migrated before the app serves. `Portal.Release` names `Portal.Repo`
  (the F6.3 repo), an ops task outside the web boundary.

- **F6.8.2-INV2 (config by phase ‚Äî no secret at compile time).** Secrets and host (`DATABASE_URL`,
  `SECRET_KEY_BASE`, `PHX_HOST`) are read at boot from `runtime.exs`; compile-time `config.exs`/`prod.exs` hold no
  secret; a missing required prod env var fails the boot loudly (the existing `runtime.exs` raises).

- **F6.8.2-INV3 (the deploy bind is reachable; dev stays loopback).** Under prod/release the endpoint binds
  `{0, 0, 0, 0}` (or `:any`) so the Fly edge proxy reaches it; under dev it stays `127.0.0.1`. The `/health`
  check in `fly.toml` targets the F6.1 liveness route (as-built `router.ex:113`), so a healthy machine is one the
  proxy can reach AND the app answers.

- **F6.8.2-INV4 (cluster-correct real-time).** With the libcluster topology, `Phoenix.PubSub` broadcasts and
  `PortalWeb.Presence` counts are correct across all nodes; F6.7's single-node assumption is not baked into the
  deploy. The web still reaches PubSub/Presence only through the `Portal` facade (`Portal.subscribe/1`,
  `Portal.broadcast/2`) ‚Äî clustering is a supervision-tree + transport concern, not a new web‚Üíengine path.

- **F6.8.2-INV5 (the id mint is multi-node-safe).** `Portal.ID`'s `worker_id` is derived per machine via
  `:erlang.phash2(System.get_env("FLY_MACHINE_ID"), 1024)` (OPERATOR-PINNED), so two nodes minting in the same
  millisecond do not collide; single-node behavior is unchanged. The branded-id format, namespaces, and the
  `Portal.ID` surface are otherwise unchanged.

- **F6.8.2-INV6 (the release is Portal-scoped; `echo_bot` excluded; the templates are inputs).** The `releases/0`
  names only the Portal apps (`portal_web`/`portal`/`echo_data`); `apps/echo_bot` is excluded and untouched
  (the out-of-band F10 bot is a separate release). The Operator's `docs/deploy/Dockerfile.template` +
  `docs/deploy/fly.toml` are reference INPUTS ‚Äî F6.8.2 builds FROM them and does not edit them in place.

- **F6.8.2-INV7 (the engine + domain are unchanged; deploy is a web/ops concern).** The F5 engine, the
  `EventStore` port, the catalog/learning/accounts contexts, the store, and the branded-id format are
  byte-unchanged; F6.8.2 adds a release, a config split, a migrate task, clustering, and a per-machine
  `worker_id` (a runtime-derived value) as web/ops concerns, not domain changes. The web still calls only the
  `Portal` facade.
