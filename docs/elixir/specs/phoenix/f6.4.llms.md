## Requirements

- **F6.4-R1** ‚Äî `Catalog` exposes `list_courses/0`, `get_course!/1`, `fetch_course/1`, `create_course/1`; the `Course`
  alias, `Ecto.Query` import, and all `Repo.*` are private. [US: F6.4-US2]
- **F6.4-R2** ‚Äî `Enrollment` reads and appends through the `Portal.EventStore` port (not `Repo`) and exposes a public
  `%Enrolled{}` struct; it runs unchanged on both adapters. [US: F6.4-US3]
- **F6.4-R3** ‚Äî `Accounts` exposes the small surface the web and orchestration need, internals private. [US: F6.4-US5]
- **F6.4-R4** ‚Äî `Portal` `defdelegate`s each public function to its context and owns no logic; the web calls only it.
  [US: F6.4-US1]
- **F6.4-R5** ‚Äî `Enrollment.enroll/2` calls `Catalog.fetch_course/1`, branches on the public struct, and folds to
  `%Portal.Error{}`. [US: F6.4-US4]
- **F6.4-R6** ‚Äî `enroll_and_welcome/2` chains `Catalog`/`Enrollment`/`Accounts` with `with`, short-circuiting to one
  closed error. [US: F6.4-US5]
- **F6.4-R7** ‚Äî the dependency graph is one-way with no cycles; no context touches another's schema or `Repo`; `Multi`
  within one context, `with` across. [US: F6.4-US4, F6.4-US6]

## Agent stories

- **F6.4-AS1** [implements F6.4-US2] ‚Äî Directive: write `Portal.Catalog` with the four public functions over the F6.3
  `Course` schema; keep the schema alias, `Ecto.Query` import, and all `Repo.*` private. Acceptance gate: each function
  returns its documented shape; nothing outside the module names `Repo` or `Course`.
- **F6.4-AS2** [implements F6.4-US3] ‚Äî Directive: write `Portal.Enrollment` reading and appending through the
  `EventStore` port, exposing `%Enrolled{}`. Acceptance gate: it uses the port (not `Repo`) and runs unchanged on
  in-memory and Postgres adapters.
- **F6.4-AS3** [implements F6.4-US1, F6.4-US4] ‚Äî Directive: write the `Portal` facade as `defdelegate`s; wire
  `Enrollment.enroll/2` to call `Catalog.fetch_course/1` and fold to `%Portal.Error{}`. Acceptance gate: the web calls
  only `Portal`; the edge is one-way; outcomes are closed errors.
- **F6.4-AS4** [implements F6.4-US5, F6.4-US6] ‚Äî Directive: write `enroll_and_welcome/2` as a `with` chain across
  contexts; confirm `Multi` is used only within one context. Acceptance gate: any step's failure short-circuits to one
  `%Portal.Error{}`; no transaction spans contexts; no cycles.

## Execution plan ‚Äî first two stories

1. **F6.4-AS1 ‚Äî the Catalog context.** Create `lib/portal/catalog.ex`: `alias Portal.Catalog.Course`, `import
   Ecto.Query`, `alias Portal.Repo` ‚Äî all private. Add `list_courses/0` (`Repo.all(Course)`), `get_course!/1`
   (`Repo.get!`), `fetch_course/1` (`Repo.get` ‚Üí `{:ok, _} | {:error, :not_found}`), and `create_course/1` (changeset
   ‚Üí `Repo.insert`). Confirm no other module names `Repo` or `Course`.
2. **F6.4-AS2 ‚Äî the Enrollment context over the port.** Create `lib/portal/enrollment.ex` calling
   `Portal.EventStore.adapter()` for `append/2` and `read_stream/1`; define `%Portal.Enrollment.Enrolled{}` as the
   published struct. Implement `enroll/2` over the port. Run the suite on the in-memory adapter, then confirm the same
   code compiles against Postgres.

## Comprehensive implementation prompt

```text
You are building the Portal's contexts and web-facing facade (rung F6.4). Read specs/phoenix/f6.4.md, the F6.3
Course schema/changeset, and the F5.9 boundary (Portal.EventStore port, the Portal facade, the closed Portal.Error
contract). The catalog is Repo-backed (CRUD); enrollment is event-sourced through the port.

Build the domain as a graph of public APIs:

1) Catalog (T1 -> R1). Create lib/portal/catalog.ex with `alias Portal.Catalog.Course`, `import Ecto.Query`, and
   `alias Portal.Repo` kept PRIVATE to the module. Public functions:
     list_courses/0 :: [%Course{}]
     get_course!/1  :: %Course{} (raises on miss; for controllers)
     fetch_course/1 :: {:ok, %Course{}} | {:error, :not_found} (for composing contexts)
     create_course/1 :: {:ok, %Course{}} | {:error, %Ecto.Changeset{}}
   No module outside Catalog may name Repo or the Course schema.

2) Enrollment (T2 -> R2). Create lib/portal/enrollment.ex that reads and appends through the Portal.EventStore port
   (Portal.EventStore.adapter()), NOT the Repo. Define %Portal.Enrollment.Enrolled{} as the published struct returned
   to callers (never a schema). Implement enroll/2 over the port so the in-memory adapter serves tests and Postgres
   serves production unchanged.

3) Accounts (T3 -> R3). Create lib/portal/accounts.ex exposing the small surface the web and the orchestration need,
   with internals private.

4) Facade (T4 -> R4). Write lib/portal.ex as a thin module that `defdelegate`s each public function to its context
   (Catalog, Enrollment, Accounts). It owns no logic and is the only module the web calls.

5) Composition (T5 -> R5). In Enrollment.enroll/2, call Catalog.fetch_course/1, branch on the public struct (never a
   schema), and fold every outcome into the closed %Portal.Error{} set via Portal.Error.from/1 (no catch-all).

6) Orchestration (T6 -> R6). Add enroll_and_welcome/2 that chains public calls across Catalog, Enrollment, and
   Accounts with a `with` expression, short-circuiting to one closed %Portal.Error{} on any failure.

7) Discipline (T7 -> R7). Keep the dependency graph one-way (Enrollment -> Catalog), with no cycles and no context
   touching another's schema or Repo. Use Ecto.Multi only within a single context; use a `with` chain for consistency
   across contexts ‚Äî never one transaction spanning contexts.

8) Verify (T8). Confirm the web calls only Portal; no controller names a context, schema, or Repo; Repo and the
   Course schema are private to Catalog; Enrollment runs against both adapters unchanged; the orchestration returns
   one closed error on any failure.

Constraints: branded Snowflake IDs (integer columns; branded transport form); pure functions where practical;
expected failures are tagged tuples folded to %Portal.Error{}. Out of scope: HEEx templates and forms (F6.5),
LiveView (F6.6), PubSub (F6.7).
```
