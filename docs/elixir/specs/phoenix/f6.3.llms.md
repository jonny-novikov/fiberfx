# F6.3 — Persistence with Ecto · agent brief (llms)

> The agent-facing brief for F6.3. Read the references first, satisfy the requirements, build in task-topology order,

## Requirements

- **F6.3-R1** — `Portal.Repo` (`use Ecto.Repo, otp_app: :portal, adapter: Ecto.Adapters.Postgres`) is configured and
  added to `Portal.Application` as the **first** child, before the store and the engine. [US: F6.3-US1, F6.3-US6]
- **F6.3-R2** — migrations create `:courses` and `:events` with `primary_key: false` and an explicit `add :id,
  :bigint, primary_key: true`; `:events` has `:stream`, `:seq`, `:type`, `:data`, an inserted-at, and an index on
  `[:stream, :seq]`. [US: F6.3-US1, F6.3-US3]
- **F6.3-R3** — `Portal.Catalog.Course` uses `@primary_key {:id, :id, autogenerate: false}` with fields `:title`,
  `:published` (default `false`), and `timestamps(type: :utc_datetime)`. [US: F6.3-US2]
- **F6.3-R4** — `Course.changeset/2` runs `cast(attrs, [:title, :published])`, `validate_required([:title])`,
  `validate_length(:title, min: 3)`, `unique_constraint(:title)`, returning an `%Ecto.Changeset{}`. [US: F6.3-US2]
- **F6.3-R5** — `Portal.Error.from_changeset/1` turns a failed changeset into `%Portal.Error{}` with a closed `code`
  (extend the `@type code` union with `:invalid`) and `field:` from the first error; callers never receive a raw
  changeset. [US: F6.3-US2]
- **F6.3-R6** — composable query functions return an `Ecto.Queryable` (`published/1`), run with `Repo.all`; `Repo.get
  (Course, id)` fetches by Snowflake id. [US: F6.3-US1]
- **F6.3-R7** — `Portal.EventStore.Postgres` implements the F5 behaviour: `append/2` via `Repo.insert_all(Event,
  rows)`, `read_stream/1` via `Repo.all(from e in Event, where: e.stream == ^stream, order_by: e.seq)`; config selects
  Postgres in prod, `InMemory` in test; no caller of the port changes. [US: F6.3-US3, F6.3-US5, F6.3-US6]
- **F6.3-R8** — any write touching more than one row uses `Ecto.Multi` / `Repo.transaction`, returning `{:ok, _} |
  {:error, step, reason, _}`; `append/2` commits its batch atomically. [US: F6.3-US4]
- **F6.3-R9** — verification holds: migrations run; a course round-trips; an invalid changeset yields a typed error;
  the Postgres adapter round-trips in `:seq` order; switching to `InMemory` changes no caller; DB tests run in the
  sandbox; pure builders are tested without a DB; `grep -rE "Repo|Ecto\.(Schema|Query|Changeset)" lib/portal_web/` is
  empty. [US: all]

## Execution topology

**Runtime topology (supervision order, write path, engine-boot read path).**

```text
Portal.Application (Supervisor, :one_for_one)
├─ Portal.Repo                       # F6.3 — the one new child, starts FIRST
├─ Portal.EventStore.adapter()       # Postgres (prod, uses Repo) | InMemory (test)
├─ {Portal.Engine, []}               # init/1 reads the log THROUGH the port
├─ {Phoenix.PubSub, …} / Telemetry   # (as F6.1 wired)
└─ PortalWeb.Endpoint                # F6.1

WRITE (a context write, in F6.4 and beyond):
  untrusted attrs ─▶ Course.changeset/2 (PARSE) ─▶ valid? ─┬─ true  ─▶ Repo.insert | Ecto.Multi ─▶ Postgres
                                                           └─ false ─▶ Portal.Error.from_changeset/1 ─▶ {:error, %Portal.Error{}}

ENGINE BOOT (read):
  Portal.Engine.init/1 ─▶ Portal.EventStore.read_stream("portal") ─▶ Portal.EventStore.Postgres
                                                                   └─▶ Repo.all(order_by: :seq) ─▶ replay/1
```

Only `Portal.EventStore.Postgres` (and the F6.4 contexts, later) touch `Repo`; the web layer and the `Portal` facade
never do.

**Task topology (build order; each step leaves the app compiling and the suite green).**

```text
T1 Portal.Repo + config (dev/test/runtime) + supervise as FIRST child           [R1]
     └─▶ T2 migrations: :courses + :events, :bigint Snowflake ids, [:stream,:seq] index  [R2]
           └─▶ T3 Portal.Catalog.Course schema (autogenerate: false)            [R3]
                 └─▶ T4 Course.changeset/2 + Portal.Error.from_changeset/1       [R4, R5]
                       ├─▶ T5 composable queries (published/1, Repo.get)         [R6]
                       └─▶ T6 Event schema + Portal.EventStore.Postgres + config [R7]
                             └─▶ T7 Ecto.Multi / Repo.transaction atomic pattern [R8]
                                   └─▶ T8 verify (migrate, round-trips, swap, sandbox, web grep)  [R9 + DoD]
```

**Touched files.** `lib/portal/repo.ex`, `config/config.exs`, `config/dev.exs`, `config/test.exs`,
`config/runtime.exs`, `lib/portal/application.ex`, `priv/repo/migrations/*_create_courses.exs`,
`priv/repo/migrations/*_create_events.exs`, `lib/portal/catalog/course.ex`, `lib/portal/error.ex` (add
`from_changeset/1`, extend `@type code`), `lib/portal/event_store/event.ex`, `lib/portal/event_store/postgres.ex`.

## Agent stories

Each agent story implements one user story and closes on a gate. Build in task-topology order.

### F6.3-AS1 — Durable storage, wired first [implements F6.3-US1, F6.3-US6]
Directive: create `Portal.Repo`, configure it per env, and add it as the first supervision child (T1); add the
`:courses` and `:events` migrations with `:bigint` Snowflake ids (T2).
Acceptance gate: `mix ecto.create && mix ecto.migrate` succeeds; the tree starts Repo → store → engine; no
`lib/portal_web/` module names `Repo`.

### F6.3-AS2 — Parse at the boundary [implements F6.3-US2]
Directive: build `Portal.Catalog.Course` (T3); write `Course.changeset/2` and `Portal.Error.from_changeset/1`,
extending the closed `@type code` with `:invalid` (T4).
Acceptance gate: valid attrs → valid changeset → `%Course{}` on insert; missing/short title and duplicate title →
`%Portal.Error{}` with the right `field`; no raw changeset escapes the boundary.

### F6.3-AS3 — Queries by Snowflake [implements F6.3-US1]
Directive: add `published/1` (an `Ecto.Queryable`) run with `Repo.all`, and `Repo.get(Course, id)` (T5).
Acceptance gate: inserting two courses (one published) and running `published/1` returns only the published one;
`Repo.get` fetches by the minted Snowflake id.

### F6.3-AS4 — The Postgres event-store adapter [implements F6.3-US3, F6.3-US5]
Directive: add the `Event` schema and `Portal.EventStore.Postgres` (`append/2` via `Repo.insert_all`, `read_stream/1`
ordered by `:seq`); select it in prod config and `InMemory` in test (T6).
Acceptance gate: `append/2` then `read_stream/1` returns the events in `:seq` order; the suite is green under
`InMemory` with no caller change; both adapters satisfy the same behaviour.

### F6.3-AS5 — Atomic multi-row writes [implements F6.3-US4]
Directive: establish the `Ecto.Multi` / `Repo.transaction` pattern; ensure `append/2` commits its batch atomically
(T7).
Acceptance gate: a multi-step `Ecto.Multi` with a forced failing step rolls back fully (DB unchanged); success returns
`{:ok, _}`, failure `{:error, step, reason, _}`.

## Execution plan — first two stories

*story → agent story → tasks → files → command → gate.*

### Plan A — F6.3-US1 (data survives restarts) · via F6.3-AS1
1. Tasks T1 → T2.
2. Files: `lib/portal/repo.ex`, `config/{config,dev,test,runtime}.exs`, `lib/portal/application.ex`,
   `priv/repo/migrations/*_create_courses.exs`, `priv/repo/migrations/*_create_events.exs`.
3. Command: `mix ecto.create && mix ecto.migrate && mix compile`.
4. Gate (closes US1): migrations create both tables with `:bigint` ids; boot order is Repo → store → engine;
   `grep -rE "Repo" lib/portal_web/` is empty. Requirements: F6.3-R1, F6.3-R2.

### Plan B — F6.3-US2 (parse at the boundary) · via F6.3-AS2
1. Tasks T3 → T4.
2. Files: `lib/portal/catalog/course.ex`, `lib/portal/error.ex` (add `from_changeset/1`, extend `@type code`).
3. Command: `mix test test/portal/catalog/course_test.exs` (validation cases need no DB; the insert case uses the
   sandbox).
4. Gate (closes US2): valid → `%Course{}`; missing/short/duplicate title → `%Portal.Error{}` with `field`; the closed
   `@type code` now includes `:invalid`. Requirements: F6.3-R4, F6.3-R5.

Efficiency check: durability is a Repo plus two migrations behind one command; the parse boundary is one changeset
plus one bridge function behind one test — and the error shape, the id type, and the supervision order are all fixed
by the spec, so the agent implements rather than decides.
