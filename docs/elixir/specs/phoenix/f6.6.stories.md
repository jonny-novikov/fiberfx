# F6.6 · user stories

> Who the interactive catalog serves, what they need, and how we will know it works. Connextra stories with
> Given/When/Then acceptance criteria; together they cover every deliverable in [`f6.6.md`](f6.6.md). Stories honor the
> [`f6.6.md`](f6.6.md) `## [RECONCILE]` direction (the F6.5 routing + component decisions folded forward): the live
> catalog shares the `/courses` namespace, the create form reuses the local `PortalWeb.CatalogComponents` `<.input>`,
> and search runs through the `Portal` facade.

## F6.6-US0 — One live URL for the catalog

As an **architect**, I want the live catalog to take over the existing `/courses` URL, so that one catalog resource
keeps one address and the static index does not split from its interactive successor.

Acceptance criteria
- Given the router, when inspected, then `live "/courses", CatalogLive` is declared in the public `:browser` scope and
  **supersedes** F6.5's static `GET /courses` index — there is no second `/catalog` surface (per [`f6.6.md`](f6.6.md)
  `## [RECONCILE]`, route).
- Given the catalog resource, when routed, then the live view addresses the same resource F6.5 named `resources
  "/courses"` (F6.5-INV7), so `/courses` is the one catalog URL.
- Given `~p"/courses"`, when navigated, then `CatalogLive` mounts and renders the catalog over the unchanged F6.4 facade
  and F6.3 changeset.

INVEST — independent; testable through the router pipeline; encodes F6.6-INV1 and the [RECONCILE] route direction.
Priority: must · Size: 2 · Implements deliverables: F6.6-D6.

## F6.6-US1 — Search as I type

As a **learner**, I want the course list to filter as I type, so that I find a course without submitting or reloading.

Acceptance criteria
- Given the search box, when I type, then `phx-change="search"` fires `handle_event("search", params, socket)`.
- Given the event, when it runs, then it filters through the `Portal.search_courses/1` facade function (not the
  `Catalog` context directly, per [`f6.6.md`](f6.6.md) `## [RECONCILE]`, facade-only) and re-assigns `@courses`.
- Given each keystroke, when handled, then the rendered list narrows without a reload, and the view names only `Portal`.

INVEST — independent; testable with `render_change`; encodes F6.6-INV1, INV3 and the [RECONCILE] facade-only direction.
Priority: must · Size: 3 · Implements deliverables: F6.6-D3.

## F6.6-US2 — Create without a reload

As a **learner** (an author), I want to add a course inline, so that I see it appear without leaving the page.

Acceptance criteria
- Given the form, when rendered, then it posts BOTH required fields through the local `<.input field={@form[:title]} />`
  and `<.input field={@form[:slug]} />` from `PortalWeb.CatalogComponents` — no `CoreComponents` is referenced (per
  [`f6.6.md`](f6.6.md) `## [RECONCILE]`, components; F6.5-INV8).
- Given the form, when I submit, then `phx-submit="create"` calls `Portal.create_course/1`.
- Given `{:ok, _}`, when it returns, then the view `put_flash`es and resets the form.
- Given `{:error, %Ecto.Changeset{}}` (for example a title-only submission `validate_required` rejects), when it
  returns, then the view assigns `to_form(changeset)` and the F6.3 errors show inline per field.

INVEST — independent; testable with `render_submit`; encodes F6.6-INV5 and the [RECONCILE] component direction.
Priority: must · Size: 3 · Implements deliverables: F6.6-D4.

## F6.6-US3 — Fast first paint

As a **visitor**, I want the page to render before the socket connects, so that first paint is fast and indexable.

Acceptance criteria
- Given a first request, when it renders, then a disconnected HTTP `mount/3` paints the page.
- Given the socket, when it connects, then a connected `mount/3` takes over.
- Given side effects, when present, then `connected?/1` guards them so they run once, on the live socket.

INVEST — independent; testable by the disconnected and connected mounts; encodes F6.6-INV2.
Priority: must · Size: 2 · Implements deliverables: F6.6-D2.

## F6.6-US4 — Large lists stay light

As a **developer**, I want the list rendered as a stream, so that a large, append-only catalog does not sit in socket
memory.

Acceptance criteria
- Given the list, when initialized, then it uses `stream/3` and `phx-update="stream"`.
- Given a new course, when added on `{:ok, _}`, then `stream_insert/3` inserts the row.
- Given the socket, when inspected, then the full list is not held as an assign.

INVEST — independent; testable by the stream usage; encodes F6.6-INV4.
Priority: should · Size: 2 · Implements deliverables: F6.6-D5.

## F6.6-US5 — One interactive surface over the facade

As a **developer**, I want the LiveView to call only the facade and reuse the F6.5 component, so that interactivity adds
no boundary leaks and no duplicated markup.

Acceptance criteria
- Given `CatalogLive` (`use PortalWeb, :live_view`), when inspected, then it calls only `Portal` (and the `Catalog`
  context as the facade fronts it) — no engine, `Repo`, or `GenServer.call`.
- Given `mount/3`, when it runs, then it assigns `@courses` from the facade plus the search and form assigns, and
  `handle_event/3` returns `{:noreply, socket}`.
- Given `render/1`, when it draws a row, then it reuses the F6.5 `course_card` component rather than duplicating row
  markup, and it changes state only through assigns and events.

INVEST — independent; testable by inspection and rendering; encodes F6.6-INV1, INV6.
Priority: must · Size: 1 · Implements deliverables: F6.6-D1.
