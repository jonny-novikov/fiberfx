# F6.8 · user stories

> Who auth and deployment serve, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.8.md`](f6.8.md).

## F6.8-US1 — Register and sign in

As a **learner**, I want to create an account and sign in, so that I have a real identity on the platform.

Acceptance criteria
- Given the registration page, when I submit valid details, then an account is created and I am signed in.
- Given the login page, when I submit correct credentials, then I am authenticated; when wrong, then I see an error.
- Given a session, when I log out, then I am signed out.

INVEST — independent; testable through the generated auth flow; encodes F6.8-INV1.
Priority: must · Size: 3 · Implements deliverables: F6.8-D1, F6.8-D3.

## F6.8-US2 — Protected areas

As an **operator**, I want protected routes to require a signed-in user, so that anonymous visitors cannot reach them.

Acceptance criteria
- Given an anonymous request to a protected route, when it arrives, then it is redirected to login.
- Given an authenticated user, when they request it, then they are admitted.
- Given a protected LiveView, when it mounts, then `on_mount` enforces auth before it connects.

INVEST — independent; testable by anonymous vs authenticated requests; encodes F6.8-INV2.
Priority: must · Size: 2 · Implements deliverables: F6.8-D2.

## F6.8-US3 — Deploy from the environment

As an **operator**, I want secrets and host read from the environment at boot, so that I deploy without compiling them
in.

Acceptance criteria
- Given a boot, when `runtime.exs` runs, then it reads `DATABASE_URL`, `SECRET_KEY_BASE`, and `PHX_HOST` from env.
- Given a missing required variable, when the app boots, then it fails loudly rather than starting misconfigured.
- Given compile-time config, when inspected, then it holds no secret.

INVEST — independent; testable by booting with and without the env; encodes F6.8-INV3.
Priority: must · Size: 2 · Implements deliverables: F6.8-D4.

## F6.8-US4 — Migrate a release

As an **operator**, I want to migrate the database from the release, so that I do not need `mix` on the server.

Acceptance criteria
- Given a built release, when I run `bin/portal eval "Portal.Release.migrate()"`, then pending migrations run.
- Given the deploy, when it proceeds, then the schema is migrated before the app serves.
- Given no `mix` on the host, when migrating, then the release task still works.

INVEST — independent; testable by running the eval against a fresh database; encodes F6.8-INV4.
Priority: must · Size: 2 · Implements deliverables: F6.8-D5, F6.8-D7.

## F6.8-US5 — Real-time across nodes

As an **instructor**, I want real-time to be correct across the cluster, so that updates and viewer counts hold when
the app runs on more than one node.

Acceptance criteria
- Given a libcluster topology, when nodes start, then they form a cluster.
- Given two nodes, when one broadcasts, then subscribers on the other receive it.
- Given Presence, when viewers connect across nodes, then the count is correct cluster-wide.

INVEST — independent; testable with two connected nodes; encodes F6.8-INV5.
Priority: should · Size: 3 · Implements deliverables: F6.8-D6.

## F6.8-US6 — The engine stays untouched

As a **developer**, I want auth and deployment to add no domain changes, so that the F5 engine and the boundary are
unaffected.

Acceptance criteria
- Given the web layer, when inspected, then it calls only `Portal`; the `Accounts` context owns `User`/token schemas
  and `Repo` privately.
- Given the engine, store, and facade, when F6.8 lands, then they are unchanged.
- Given any web module, when inspected, then none names `Portal.Engine`, a repo, or `GenServer.call`.

INVEST — independent; testable by inspection and a diff; encodes F6.8-INV1, INV6.
Priority: must · Size: 1 · Implements deliverables: F6.8-D3.
