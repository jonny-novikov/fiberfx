# F6.8 · user stories

> Who auth and deployment serve, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.8.md`](f6.8.md), honoring the
> [`f6.8.md` `[RECONCILE]`](f6.8.md) direction folded forward from F6.5 (`5a440fd`).

## F6.8-US0 — Retire the interim auth for the generated one

As an **architect**, I want the generated `UserAuth`, `Accounts`, and component home to supersede their F6.2–F6.4
interim stand-ins, so that one real identity model — not two — backs every protected surface.

Acceptance criteria
- Given F6.2's hand-rolled `RequireUser`, the `:require_auth` pipeline, and the bare `:current_user_id` assign, when
  `phx.gen.auth` lands, then they are replaced by `fetch_current_user` → `conn.assigns.current_user` (a `%User{}`) and
  `require_authenticated_user`, and `RequireUser` no longer exists.
- Given F6.5's protected `get "/my/courses"` reading `current_user_id`, when reconciled, then it reads
  `current_user.id`, pipes through `require_authenticated_user`, and the unauthenticated redirect targets the generated
  login path (not `~p"/"`).
- Given the component reckoning the auth UI forces, when decided, then EITHER the generated `CoreComponents` is adopted
  (superseding F6.5-INV8) OR the generated login/registration/reset templates are ported to the local
  `PortalWeb.CatalogComponents` set — one ratified home, with no reference to an unported `<.simple_form>`/`<.input>`/
  `<.header>`/`<.flash>`.
- Given F6.4's Store-backed `Accounts` (`user/1`/`welcome/1`), when `phx.gen.auth Accounts User users` lands, then the
  context is Repo-backed over `users`/tokens and the two surfaces are reconciled into one `Accounts`.

INVEST — independent; testable through the router pipeline, a `grep` for `RequireUser`/`:require_auth`, and the auth
template imports; encodes F6.8-INV1, INV2. Reconciles the [RECONCILE] direction (auth + components + context).
Priority: must · Size: 3 · Implements deliverables: F6.8-D1, F6.8-D2, F6.8-D3.

## F6.8-US1 — Register and sign in

As a **learner**, I want to create an account and sign in, so that I have a real identity on the platform.

Acceptance criteria
- Given the generated registration page, when I submit valid details, then `Accounts` creates a `%User{}` row, I am
  signed in, and a session token is issued.
- Given the login page, when I submit correct credentials, then I am authenticated and `conn.assigns.current_user` is a
  `%User{}`; when I submit wrong credentials, then I see an error and remain anonymous.
- Given a session, when I log out, then the token is dropped and `conn.assigns.current_user` is `nil`.
- Given the `Accounts` context, when inspected, then it owns the `User`/token schemas and its `Repo` calls privately —
  no caller outside the context names them.

INVEST — independent; testable through the generated auth flow end to end; encodes F6.8-INV1.
Priority: must · Size: 3 · Implements deliverables: F6.8-D1, F6.8-D3.

## F6.8-US2 — Protected areas turn anonymous visitors away

As a **visitor**, I want protected routes to send me to login instead of serving me, so that no anonymous request
reaches a signed-in-only surface (such as `/my/courses`).

Acceptance criteria
- Given an anonymous request to a protected controller route, when it arrives, then `require_authenticated_user`
  redirects it to the login path with a `302` — it is never served a `200`.
- Given a learner with a valid session, when they request the same route, then they are admitted and the page renders.
- Given a protected LiveView in its `live_session`, when it mounts, then `on_mount {PortalWeb.UserAuth,
  :ensure_authenticated}` enforces auth before the socket connects — an anonymous mount is redirected, not rendered.

INVEST — independent; testable by anonymous vs authenticated requests and a LiveView mount; encodes F6.8-INV2.
Priority: must · Size: 2 · Implements deliverables: F6.8-D2.

## F6.8-US3 — Deploy from the environment

As an **operator**, I want secrets and host read from the environment at boot, so that I deploy without compiling them
in.

Acceptance criteria
- Given a boot, when `config/runtime.exs` runs, then it reads `DATABASE_URL`, `SECRET_KEY_BASE`, and `PHX_HOST` from
  the environment.
- Given a missing required variable, when the app boots, then it raises and fails the boot loudly rather than starting
  misconfigured.
- Given compile-time `config.exs`/`prod.exs`, when inspected, then they hold only fixed, committable settings and no
  secret.

INVEST — independent; testable by booting with and without the env; encodes F6.8-INV3.
Priority: must · Size: 2 · Implements deliverables: F6.8-D4.

## F6.8-US4 — Migrate a release

As an **operator**, I want to migrate the database from the release, so that I do not need `mix` on the server.

Acceptance criteria
- Given a built release, when I run `bin/portal eval "Portal.Release.migrate()"`, then `Portal.Release.migrate/0` runs
  the pending migrations and `rollback/2` is available.
- Given the deploy sequence `MIX_ENV=prod mix release` → `bin/portal eval "Portal.Release.migrate()"` →
  `bin/portal start`, when it proceeds, then the schema is migrated before the app serves and the F5 supervision tree
  comes up serving.
- Given no `mix` on the host, when migrating, then the release task still works (it loads the app and runs Ecto without
  `mix`).

INVEST — independent; testable by running the eval against a fresh database on a `mix`-less host; encodes F6.8-INV4.
Priority: must · Size: 2 · Implements deliverables: F6.8-D5, F6.8-D7.

## F6.8-US5 — Real-time across nodes

As an **instructor**, I want real-time to be correct across the cluster, so that updates and viewer counts hold when
the app runs on more than one node.

Acceptance criteria
- Given the libcluster topology in the supervision tree, when nodes start, then they form a cluster (`Node.list/0`
  shows the peers).
- Given two connected nodes, when one broadcasts on a topic, then a subscriber on the other node receives the message.
- Given `PortalWeb.Presence`, when viewers connect across both nodes, then the count is correct cluster-wide, not
  per-node.

INVEST — independent; testable with two connected nodes; encodes F6.8-INV5.
Priority: should · Size: 3 · Implements deliverables: F6.8-D6.

## F6.8-US6 — The engine stays untouched

As a **developer**, I want auth and deployment to add no domain changes, so that the F5 engine and the boundary are
unaffected.

Acceptance criteria
- Given the web layer, when inspected, then `PortalWeb.UserAuth`, the auth LiveViews, and the protected scopes call
  only `Portal` (which `defdelegate`s the auth queries the web needs, e.g. `get_user_by_session_token/1`,
  `register_user/1`).
- Given any web module, when `grep -rE "Portal\.Engine|Repo|GenServer\.call|Portal\.Accounts" lib/portal_web/` runs,
  then it matches nothing — no web module names the engine, a repo, `GenServer.call`, or a context directly.
- Given the F5 engine, store, and the facade's domain functions, when F6.8 lands, then a diff shows them unchanged.

INVEST — independent; testable by inspection and a diff; encodes F6.8-INV1, INV6.
Priority: must · Size: 1 · Implements deliverables: F6.8-D3.

## F6.8-US7 — Prove the deployed product end to end

As an **operator**, I want the whole auth-and-deploy path verified in one pass, so that a green checklist is the
release gate — not a hope that it works.

Acceptance criteria
- Given the running app, when registration and login are exercised, then both succeed; when a protected route is hit,
  then an anonymous user is redirected and an authenticated one is admitted; when a protected LiveView mounts anonymous,
  then `on_mount` enforces before connect.
- Given a built release, when the deploy sequence runs, then it builds, migrates from env config via `bin/portal eval`,
  and boots — failing loudly if a required env var is absent.
- Given two clustered nodes, when one broadcasts and viewers connect across both, then PubSub delivery and the Presence
  count are correct cluster-wide.
- Given the web after all of the above, when inspected, then it still calls only the `Portal` facade.

INVEST — independent; testable as the end-to-end acceptance pass; encodes F6.8-INV1, INV2, INV3, INV4, INV5, INV6.
Priority: must · Size: 2 · Implements deliverables: F6.8-D8.
