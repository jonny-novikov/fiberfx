# F6.8.1 · user stories

> Who the auth slice serves, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.8.1.md`](f6.8.1.md). Two acceptance faces, one per
> half of the seam: the login PAGE is judged the F6.5.5 way — TEXT PROXIES (full-document envelope, clamp-spacing,
> token-fidelity, no-facade-call) — and the auth GATE is judged by BEHAVIOR (the honest door: one error for both
> wrong-name and wrong-password; `:ok` for every reset request; redirect-when-anonymous to `/login`; admit-with-a-loaded
> `current_user`). Stories honor the [`f6.8.1.md`](f6.8.1.md) reconcile: the static `login.html` ported the F6.5.5 way
> (`PageController.login/2` + complete-document `login.html.heex` + extracted `login.css`/`login.js`, NO layout, NO
> `CoreComponents`), a NET-NEW `Portal.Auth` facade (`sign_in/2`, `request_reset/1`) over the EXISTING `%User{}`/`%Session{}`
> entities, the F6.2 `RequireUser` gate evolved into `PortalWeb.UserAuth` (`current_user` + `on_mount`), registration
> deferred. Framing: no gendered pronouns for agents; no perceptual/interior-state verbs for tools or agents; no
> first-person narration.

## F6.8.1-US1 — The login page, rendered in the design system

As a **Learner**, I want the Portal to render the sign-in page in the same dark-editorial design system as the rest of the
course, at the route I expect (`/login`), so that the door to my progress, labs, and notes looks like it belongs to the
portal — not an unstyled form.

Acceptance criteria
- Given a request to `~p"/login"`, when it renders, then the response is the `elixir/login.html` document reproduced as a
  full-document HEEx template — `<!doctype html>` → `<html lang="en">` → a `<head>` carrying
  `<title>Sign in · Functional Programming in Elixir · jonnify</title>`, the Google-Fonts `<link>`, and
  `<link rel="stylesheet" href={~p"/assets/login.css"}>` → `<body>` with the `.topbar`, the `.hero` (the `.auth-card`
  with the sign-in + reset forms and the SVG sign-in-flow), and the canonical `.site-foot` `<footer>` (per
  [`f6.8.1.md`](f6.8.1.md) D1, INV2; the CURRENT 733L master body `login.html:424-526`). *(The 685L snapshot's
  standalone `#how` "Two doors, one rule" section was folded out by the Operator — the body is `.topbar` + `.hero`
  + `.site-foot`.)*
- Given the page renders, when inspected, then it is a COMPLETE DOCUMENT with NO root-layout wrap and NO doubled
  header/footer — the template carries its OWN `<head>` and its OWN single body topbar/footer (the master's own); no
  `PortalWeb.Layouts` module exists (per [`f6.8.1.md`](f6.8.1.md) INV2; `f6.5.5.md` §How.1).
- Given `:4000`, when curled, then `/login` returns a 200 whose body contains `<head`, `login.css`, "Welcome" (the
  hero `<h1>Welcome back</h1>`, `login.html:433`) and "Learning Portal" (the `.site-foot` build stamp,
  `login.html:515`); a 200 with a bare unstyled form is a REGRESSION; a 404 on `/login` is a REGRESSION (D1).

INVEST — independent; testable by the liveness curl + an envelope/body grep on `/login`; encodes F6.8.1-D1 and the
full-document envelope.
Priority: must · Size: 2 · Implements deliverables: F6.8.1-D1, D2.

## F6.8.1-US2 — Sign in with name-or-email and password; ask for a reset

As a **Learner**, I want to sign in with either my account name or my email plus my password, and — if the password has
gone missing — to ask for a reset link in one step, so that one door takes me to everything behind the portal and a lost
password is recoverable.

Acceptance criteria
- Given the rendered `/login` page, when the sign-in form is read, then it posts to `/auth/session`
  (`action="/auth/session"`, `login.html:436`) carrying `identifier` + `password`, with `data-redirect="/elixir"` on
  success; and the reset form posts to `/auth/reset` (`action="/auth/reset"`, `login.html:455`) carrying `email` (per
  [`f6.8.1.md`](f6.8.1.md) D1, D6, D7).
- Given valid credentials, when the sign-in form is submitted, then `POST /auth/session` calls
  `Portal.Auth.sign_in(identifier, password)`, the response is a success the page reads as `r.ok` (`login.html:672`), and
  the page redirects to `/elixir` (the `data-redirect` target) (per [`f6.8.1.md`](f6.8.1.md) D3, D6).
- Given an email, when the reset form is submitted, then `POST /auth/reset` calls `Portal.Auth.request_reset(email)` and
  the response is a 200 the page reads as "if an account matches, a reset link is on its way" (`login.html:712-714`),
  regardless of whether the email matches any account (per [`f6.8.1.md`](f6.8.1.md) D4, D7, INV3).

INVEST — independent; testable by reading the rendered form `action`s + driving each POST and asserting the response
shape; encodes the two facade endpoints.
Priority: must · Size: 2 · Implements deliverables: F6.8.1-D3, D4, D6, D7.

## F6.8.1-US3 — The honest door: one failure tells nothing about which half was wrong

As a **Learner**, I want a failed sign-in to look exactly the same whether my name was wrong or my password was wrong, so
that the door does not leak whether an account exists — and as an **Architect**, I want that property to live in the
facade, not the controller, so it cannot be undone by a UI change.

Acceptance criteria
- Given a non-existent identifier and any password, when `Portal.Auth.sign_in/2` is called, then it returns
  `{:error, :invalid_credentials}` (per [`f6.8.1.md`](f6.8.1.md) D3, INV3).
- Given an existing identifier and a WRONG password, when `Portal.Auth.sign_in/2` is called, then it returns the
  IDENTICAL `{:error, :invalid_credentials}` — the same tuple, by the same path, with no message, status, or timing
  that distinguishes the wrong-name case from the wrong-password case (per [`f6.8.1.md`](f6.8.1.md) INV3). *(the honest
  door — one bit on success, zero on failure.)*
- Given valid credentials, when `Portal.Auth.sign_in/2` is called, then it returns `{:ok, %Session{}}` whose `id` is a
  branded `SES…` id minted by `Portal.ID.new("SES")` and whose `user_id` is the resolved user's id (per
  [`f6.8.1.md`](f6.8.1.md) D3, INV4).
- Given `SessionController.create/2`, when sign-in fails for EITHER reason, then it answers the SAME HTTP failure (401 or
  403 — the page reads `r.status===401 || r.status===403`, `login.html:677`); the controller does not branch its response
  on which half failed (per [`f6.8.1.md`](f6.8.1.md) D6, INV3).

INVEST — independent; testable by three `Portal.Auth.sign_in/2` unit cases (wrong name, wrong password, valid) asserting
identical error tuples + a `%Session{}` with a `SES` id, plus a controller test asserting the same status for both
failures; encodes F6.8.1-INV3, INV4.
Priority: must · Size: 2 · Implements deliverables: F6.8.1-D3, D6.

## F6.8.1-US4 — A reset request never confirms whether an account exists

As a **Learner**, I want asking for a reset to answer the same way whether or not my email is on file, so that the reset
endpoint can not be used to fish for which addresses have accounts.

Acceptance criteria
- Given an email that MATCHES an account, when `Portal.Auth.request_reset/1` is called, then it returns `:ok` (per
  [`f6.8.1.md`](f6.8.1.md) D4, INV3).
- Given an email that matches NO account, when `Portal.Auth.request_reset/1` is called, then it returns the IDENTICAL
  `:ok` — no error, no different value, no signal that the address is unknown (per [`f6.8.1.md`](f6.8.1.md) INV3).
- Given `POST /auth/reset`, when called with either a matching or a non-matching email, then it answers the SAME 200 (the
  page treats any `r.ok` as "a link is on its way", `login.html:712-714`); the difference, if any, plays out in an
  inbox, which is out of scope this rung (per [`f6.8.1.md`](f6.8.1.md) D7, INV3; Scope `Out` — email delivery).

INVEST — independent; testable by two `request_reset/1` cases (matching, non-matching) asserting identical `:ok` + a
controller test asserting the same 200 for both; encodes the no-enumeration half of F6.8.1-INV3.
Priority: must · Size: 1 · Implements deliverables: F6.8.1-D4, D7.

## F6.8.1-US5 — A real protected area: signed-in admits, anonymous goes to the login page

As an **Operator**, I want the learner surface gated on a real session — a signed-in learner admitted with a loaded
identity, an anonymous request sent to the login page — so that "behind one door" is enforced, not assumed.

Acceptance criteria
- Given valid credentials, when `POST /auth/session` succeeds, then the controller writes the learner's id into the
  signed session cookie (`put_session(conn, :user_id, session.user_id)` — the cookie F6.1 configured, `endpoint.ex:52`)
  and redirects to the `data-redirect` target (per [`f6.8.1.md`](f6.8.1.md) D6, INV4).
- Given a signed-in session, when `GET /my/courses` is requested, then `PortalWeb.UserAuth.fetch_current_user/2` loads
  `current_user` (a `%User{}`, via `Portal.Auth`) into `conn.assigns`, `require_authenticated_user/2` admits the request,
  and `EnrollmentController.index/2` reads the loaded `current_user` (reconciling F6.5's `:current_user_id`) (per
  [`f6.8.1.md`](f6.8.1.md) D9, INV5).
- Given NO session, when `GET /my/courses` is requested, then `require_authenticated_user/2` REDIRECTS the request to
  `~p"/login"` (the target moved from `~p"/"`, discharging the `RequireUser` moduledoc's owed change) — the protected
  action is never reached (per [`f6.8.1.md`](f6.8.1.md) D9, INV5).
- Given `PortalWeb.SessionController` and `PortalWeb.UserAuth`, when inspected, then each names ONLY `Portal.Auth` (the
  user load + the sign-in/reset calls) — no `Portal.Engine`, `Repo`, or `GenServer.call` (per [`f6.8.1.md`](f6.8.1.md)
  INV1).

INVEST — independent; testable by a controller/plug test (sign-in then admit; anonymous then redirect-to-`/login`) + a
boundary grep on `session_controller.ex` + `user_auth.ex`; encodes F6.8.1-INV5, INV1.
Priority: must · Size: 3 · Implements deliverables: F6.8.1-D6, D8, D9.

## F6.8.1-US6 — The routes: one static page, two POST endpoints, the rest unchanged

As a **Developer**, I want the login page served by a single new route + action in the existing public scope, and the two
auth POST endpoints declared beside it, so that the auth surface is reachable without touching the pipelines, the macros,
or anything below the facade.

Acceptance criteria
- Given `router.ex`, when inspected, then a NEW `get("/login", PageController, :login)` sits in the existing public
  `scope "/", PortalWeb` block (`pipe_through :browser`, `:59-70`), and a NEW `post("/auth/session",
  SessionController, :create)` + `post("/auth/reset", SessionController, :request_reset)` are declared in a public
  `:browser` scope (the sign-in cookie write needs `fetch_session`) (per [`f6.8.1.md`](f6.8.1.md) D8).
- Given `page_controller.ex`, when inspected, then it has a NEW `login/2` that does `render(conn, :login)` from no assigns
  (no facade call), beside the existing `home/2`/`elixir/2`/`agile/2` (per [`f6.8.1.md`](f6.8.1.md) D1, INV2).
- Given `portal_web.ex` and `page_html.ex`, when compared to before, then they are UNCHANGED — the controller macro still
  sets `layouts: []` (no root-layout binding is added), and `PageHTML` still `embed_templates "page_html/*"` (the new
  `login.html.heex` is picked up with no change) (per [`f6.8.1.md`](f6.8.1.md) INV2; `f6.5.5.md` §How.1).
- Given the footer of the rendered `/login`, when read, then the *"No account yet — registration opens with the portal
  build. Soon."* copy (`login.html:469`) is preserved as static text — no register action, route, or form is added (per
  [`f6.8.1.md`](f6.8.1.md) INV6).

INVEST — independent; testable by inspecting the new route/action + confirming `page_html.ex`/`portal_web.ex` unchanged +
a grep for the preserved footer copy + the absence of a register route; encodes F6.8.1-D8, INV6.
Priority: must · Size: 2 · Implements deliverables: F6.8.1-D8, INV6.

## F6.8.1-US7 — The login page's CSS/JS carried once, verbatim from its master

As a **Developer**, I want the master's `<style>` and `<script>` blocks to live exactly once in committed static files
served by the mount that already exists, so that the design system and the progressive-enhancement behavior are carried
faithfully with no bundler and no drift — and every `clamp()` survives the parser.

Acceptance criteria
- Given `priv/static/assets/login.css`, when compared to the FIVE `<style>` bodies of the CURRENT (733L)
  `elixir/login.html` (`:17-260`, `:263-275`, `:278-321`, `:324-355`, `:362-421`), then it is those bodies concatenated
  in source order VERBATIM — not reformatted, not re-indented — so its `:root` block (`:17-32`) matches BYTE-FOR-BYTE: the
  F0 surfaces, cream text, the gold/blue/burgundy/sage accents, `--line`, the four font stacks, the radius/shadow tokens
  (per [`f6.8.1.md`](f6.8.1.md) D2, INV2). *(token-fidelity proxy; the fifth/auth block carries the new `.site-foot` /
  build-stamp-popover rules `:395-419` verbatim.)*
- Given `priv/static/assets/login.js`, when compared to the THREE authored `<script>` bodies (`:529-530` the reveal
  observer, `:534-559` the stamp decoder + popover toggle, `:563-731` the sign-in/reset IIFE), then it is those bodies
  concatenated VERBATIM (the two KaTeX CDN `<script defer src>` tags stay in the head, not relocated) (per
  [`f6.8.1.md`](f6.8.1.md) D2).
- Given `login.css`, when scanned, then every `clamp()` has spaces around its `+`/`-` operators
  (`clamp(2.4rem, 5.5vw, 4.2rem)`, NEVER `5.5vw+`) — holding by construction because the file is the master `<style>`
  body relocated UNCHANGED (per [`f6.8.1.md`](f6.8.1.md) D2; the elixir clamp-spacing regression guard).
- Given `:4000`, when `login.css` is fetched, then `curl :4000/assets/login.css` returns 200 `text/css` from the
  `Plug.Static` `at: "/"` mount F6.5.5 corrected — no further endpoint change (per [`f6.8.1.md`](f6.8.1.md) D2).

INVEST — independent; testable by a verbatim diff of `login.css`/`login.js` against the master blocks + a clamp-spacing
parse over `login.css` + a `curl :4000/assets/login.css` check; encodes F6.8.1-D2 and the no-bundler delivery.
Priority: must · Size: 2 · Implements deliverables: F6.8.1-D2.

## F6.8.1-US8 — The credential lands as a thin slice, the hash never crosses the boundary

As an **Architect**, I want the password credential added as the thinnest coherent slice over the existing Accounts
entities, and the hash kept private to the context — never serialized to the web — so that auth becomes real without a
schema reckoning and without leaking a secret through the JSON surface.

Acceptance criteria
- Given the RATIFIED persistence choice (Option A), when `Portal.Accounts` is read, then the seeded user carries a
  `password_hash` verified by `bcrypt_elixir`, and a session-mint helper mints `%Session{}` with `Portal.ID.new("SES")`
  — with NO new schema and NO new migration, the Accounts entities staying Store-backed (per [`f6.8.1.md`](f6.8.1.md)
  D5, §The persistence fork).
- Given `apps/portal/mix.exs`, when its `deps/0` is read, then the ONE net-new dependency — `{:bcrypt_elixir, "~> 3.1"}`
  — is declared THERE (the consuming app, where `Portal.Auth` lives), not assumed transitively, and is present in
  `mix.lock` (per [`f6.8.1.md`](f6.8.1.md) D5; the per-app DEP-GRAPH-VISIBILITY rule).
- Given `Portal.Accounts.User`'s `@derive {Jason.Encoder, …}` (`user.ex:3`), when inspected after the change, then it
  does NOT serialize the password hash — the `@derive only: […]` set excludes the credential, so the hash never reaches
  the web (per [`f6.8.1.md`](f6.8.1.md) D5, INV7).
- Given the engine, the catalog/learning contexts, the store mechanism, and the ids, when compared to before, then they
  are byte-unchanged — the only domain change is the additive credential/session-mint surface under `Portal.Accounts` +
  the new `Portal.Auth` facade (per [`f6.8.1.md`](f6.8.1.md) INV7).

INVEST — independent; testable by reading `Portal.Accounts`/`mix.exs`/`mix.lock` + an `@derive` inspection + a diff-scope
review below the facade; encodes F6.8.1-D5, INV7.
Priority: must · Size: 2 · Implements deliverables: F6.8.1-D5.

## F6.8.1-US9 — The LiveView auth hook F6.9 rides

As a **Developer**, I want the auth gate to expose a LiveView `on_mount` hook and a loaded `current_user`, so that F6.9's
protected `/dashboard` `live_session` can ride this rung's auth exactly as its forward-ref names it — without the auth
contract having to be rebuilt at the capstone.

Acceptance criteria
- Given `PortalWeb.UserAuth`, when inspected, then it exposes `on_mount/4` with an `:ensure_authenticated` clause — the
  hook F6.9's `live_session` names as `{PortalWeb.UserAuth, :ensure_authenticated}` (`f6.9.stories.md:84`) — which
  assigns `current_user` (a `%User{}`, via `Portal.Auth`) on a connected mount and halts/redirects an anonymous mount to
  `~p"/login"` (per [`f6.8.1.md`](f6.8.1.md) D9, INV5; the F6.9 forward-ref).
- Given `PortalWeb.UserAuth`, when inspected, then it ALSO exposes the two plugs `fetch_current_user/2` (assigns
  `current_user` or `nil`) and `require_authenticated_user/2` (admit-or-redirect-to-`/login`), evolving the F6.2
  `PortalWeb.RequireUser` into the named `UserAuth` surface (the `:current_user_id`-only assign superseded by a loaded
  `current_user`) (per [`f6.8.1.md`](f6.8.1.md) D9, RECONCILE 3, RECONCILE 4).
- Given the auth hook and plugs, when their dependencies are traced, then the user load goes through `Portal.Auth` only —
  `PortalWeb.UserAuth` names no `Portal.Engine`, `Repo`, or `GenServer.call` (per [`f6.8.1.md`](f6.8.1.md) INV1).

INVEST — independent; testable by inspecting `user_auth.ex` for the three surfaces (`on_mount/4`,
`fetch_current_user/2`, `require_authenticated_user/2`) + a boundary grep; encodes the F6.9 forward-ref (RECONCILE 4) and
F6.8.1-D9.
Priority: must · Size: 2 · Implements deliverables: F6.8.1-D9.
