# F6.5.5 · Apply the design system

> Make the Portal render in the jonnify dark-editorial design system. A **root layout** (`PortalWeb.Layouts` +
> `root.html.heex`) replaces `layouts: []` (`portal_web.ex:30`) with the F0 document envelope — `<!doctype><html><head>…</head>`
> — so there is finally a `<head>` to carry CSS; a committed **`priv/static/assets/app.css`** holds the F0 `:root` tokens, the
> base CSS, and the four Google-Fonts families, served by the EXISTING `Plug.Static` `/assets` mount (`endpoint.ex:28`) with
> NO bundler and NO endpoint change; and the LOCAL `PortalWeb.CatalogComponents` (`course_card/1`·`panel/1`·`input/1`) are
> restyled to the F0 anatomy. The work executes the F0 design-system FOUNDATION's Portal-rendering milestones (`f0.roadmap.md`
> M4–M5) over the facade, with NO `CoreComponents` — the local set is proven to carry F0.

## Where this rung sits (inserted between F6.5 and F6.6 — both shipped)

A NEW rung in the F6 ladder, between **F6.5 · Views with HEEx** (`f6.5.md`, server-rendered catalog — shipped) and **F6.6 ·
LiveView** (`f6.6.md`, interactive catalog — shipped, `3cf2480`). F6.5 and F6.6 built the catalog's MARKUP and INTERACTIVITY;
neither built its LOOK. The Portal renders UNSTYLED today: a request returns a bare body fragment with no `<html>`/`<head>`,
because `portal_web.ex:30` sets `layouts: []` and no layout module exists (`grep` confirms no `Layouts`, no `root.html.heex`;
`priv/static/assets/` holds only `.keep`). F6.5.5 is the F6 rung that APPLIES the F0 design system to that markup — the
styling foundation every later rung (F6.7 real-time, F6.8 auth-UI, F6.9 dashboard) inherits.

This rung CLOSES an explicit predecessor deferral. F6.6's stories already note (`f6.6.stories.md:51-53`) that the live
create's success flash is set on the socket but "NOT yet PAINTED" because "this hand-built umbrella has NO root-layout
`<.flash_group>` rendering `@flash`," routing the rendered-flash assertion to "the later layout rung." **F6.5.5 is that
layout rung** — the root layout it introduces is where `@flash` and the envelope live. (Painting the flash itself is left to
the layout's content; this rung delivers the root layout that makes a `<.flash_group>` possible — the rendered-flash
assertion stays F6.6's, now unblocked.)

## Goal

After F6.5.5, the Portal renders in the F0 dark-editorial design system. A request to `:4000` returns the F0 document
envelope (`f0.md`-D3: `<!doctype html>` → `<html>` → `<head>` with the tokens stylesheet and the four font families →
`<body>`), not a bare `<h1>`. The catalog index, the `course_card` rows, the `panel` frame, and the `input` field carry the
F0 anatomy vocabulary (`.mod` cards, the `.pill.live` published badge, `.btn`/`.btn.ghost` actions, the `.fig` panel) and the
F0 token palette declared ONCE in the shared head (`f0.md`-INV2). The look is reached three ways — a root layout (the
envelope), a committed `app.css` (the tokens + base CSS + fonts), and the restyled LOCAL components — all OVER the facade
(the master invariant holds: nothing names `Portal.Engine`, a `Repo`, or `GenServer.call`), and with NO `CoreComponents`
(deferred to F6.8; this rung PROVES the local set carries F0). The engine, the contexts, the store, and the boundary below
the facade are byte-unchanged.

## Rationale

The Operator named **Why · When · What · How**; **How** is the load-bearing subsection, with **Who · Where** completing the 5W.

### Why

An unstyled platform is not shippable. The catalog renders correct markup but no look — a bare `<h1>Courses</h1>` and
class-less rows (`course_html/index.html.heex:8-12`) — and F0 already specified this rendering as planned work, not a new
idea: `f0.roadmap.md` M4 ("Portal HEEx components reproducing tokens/anatomy") and M5 ("Parity gate: static vs Portal") are
the two milestones this rung executes. The design system is a SPECIFIED, gated artifact (`f0.md`-D1–D8); a Portal that does
not render it leaves the F0 contract half-delivered. Styling the Portal is therefore closing a foundation already drawn, not
opening scope.

### When

The styling rung between F6.5 (views shipped) and F6.6 (LiveView shipped). It comes AFTER the markup and interactivity exist
(there is something to style) and BEFORE the later rungs that inherit the look: F6.7 (real-time) renders in it, F6.8 (auth
UI + the `CoreComponents` fold) styles atop it, F6.9 (dashboard) extends it. Inserting it here means every subsequent rung
ships styled rather than back-filling the look across an accreting surface.

### What

The slice (three surfaces, all over the facade, no bundler, no `CoreComponents`):
1. a **root layout** — `PortalWeb.Layouts` + `root.html.heex` — replacing `layouts: []`, carrying the F0 envelope;
2. a committed **`priv/static/assets/app.css`** — the F0 `:root` tokens + base CSS + the four Google-Fonts families — served
   by the existing `Plug.Static` `/assets` mount;
3. the LOCAL **`PortalWeb.CatalogComponents`** restyled to the F0 anatomy.

### How (load-bearing)

The mechanism, surface by surface — each cites the real code it changes or relies on.

**1 · The root layout supplies the `<head>` the CSS needs.** Today the controller macro sets `layouts: []`
(`portal_web.ex:30`) and the LiveView macro sets no `:layout` (`portal_web.ex:52-58`), so every render — controller and
LiveView alike — emits a bare body fragment with no `<html>`/`<head>` (confirmed: zero `doctype`/`<head>`/`<html>` in any
`.heex`). The fix introduces a `PortalWeb.Layouts` module (`use PortalWeb, :html` + `embed_templates "components/layouts/*"`)
and a `root.html.heex` template that IS the F0-D3 envelope: `<!doctype html>` → `<html lang="en">` → a `<head>` carrying the
F0 head (`f0.md`-D2: charset, viewport, the `<title>`, the four Google-Fonts `<link>`s, and a `<link rel="stylesheet"
href={~p"/assets/app.css"}>`) → `<body>` wrapping `{@inner_content}`. The layout is then BOUND two places, because Phoenix
has two render paths through it:
- the controller path — `layouts: []` becomes `layouts: [html: PortalWeb.Layouts]` (`portal_web.ex:30`), so the F6.5
  controller renders (`course_html.ex`, `enrollment_html.ex`, `page_html.ex`, `lesson_html.ex`) wrap in the root layout;
- the LiveView path — the `:live_view` macro (`portal_web.ex:52`) gains `layout: {PortalWeb.Layouts, :root}` (or the per-view
  equivalent), so `CatalogLive` and `EnrollmentLive` render the SAME envelope on the dead (HTTP first-paint) render.

The root layout is rendered once around the whole document; the `:root` tokens are therefore declared once (`f0.md`-INV2),
in the head, never per page. This rung delivers ONLY the root layout (the `<head>` envelope) — Phoenix's optional second
(app/inner) layout is NOT introduced; the F0 page anatomy lives in the templates and components themselves, the same way the
static `/elixir` pages carry their anatomy in the body fragment, not a wrapper.

**2 · `app.css` is a committed static file, served by the mount that already exists — no bundler, faithful to F0's
no-build model.** F0's system carries the tokens + base CSS in an INLINE `<style>` assembled by a Python builder
(`f0.md`-D2/D5; the static `/elixir` pages have no asset pipeline). The umbrella has no esbuild/tailwind/dart_sass
(`mix.exs` deps list — `phoenix`, `phoenix_ecto`, `phoenix_live_view`, `phoenix_html`, telemetry, `jason`, `bandit`,
`lazy_html` test-only — NO asset compiler), and `priv/static/assets/` holds only `.keep`. Rather than ADD a bundler (scope
F0 explicitly never took), the F0 head CSS is committed as a single static file `priv/static/assets/app.css` and served by
the `Plug.Static` mount ALREADY in the endpoint: `plug Plug.Static, at: "/assets", from: :portal_web, only:
PortalWeb.static_paths()` (`endpoint.ex:28-32`), and `static_paths/0` already includes `"assets"` (`portal_web.ex:14`,
`~w(assets fonts images favicon.ico robots.txt)`). So `~p"/assets/app.css"` resolves and serves with **NO endpoint change,
NO config change, NO bundler** — the umbrella's no-asset-pipeline reality ALIGNS with F0's no-build, inline-style model
(a committed file is the umbrella's equivalent of F0's inline `<style>`; both ship verbatim with no compile step). The file's
`:root` block is the F0-D1 palette copied BYTE-FOR-BYTE (`f0.md`-D1; `f0.md`-INV2 — never re-invented), and the base CSS is
the F0 base; the four Google-Fonts families are `<link>`ed from the head (the head holds the font links; `app.css` holds the
tokens + base + the `font-family` token stacks that reference them).

**3 · The LOCAL `CatalogComponents` are restyled to the F0 anatomy — proving the local set carries F0, no `CoreComponents`.**
The three components exist and keep their arities and data hooks; only their markup/classes change to the F0 vocabulary:
- `course_card/1` (`catalog_components.ex:32`) → the F0 `.mod` card. It KEEPS `attr :course` / `attr :class`, the
  `data-course-card` hook, the `~p"/courses/#{@course.id}"` link (`data-course-link`), and the `:if={@course.published}`
  badge — the badge `<span>` becomes the F0 `.pill.live` (KEEPING the `data-published-badge` hook and the literal "Published"
  text, `:36-37`).
- `panel/1` (`:48`) → the F0 `.fig` frame. It KEEPS the `slot :inner_block` + `render_slot/1` contract and the `data-panel`
  hook; only the wrapper class changes.
- `input/1` (`:68`) → the F0 field styling. It KEEPS the `attr :field, Phoenix.HTML.FormField` contract, the `data-field` /
  `data-field-errors` hooks, and the inline error list; the buttons the create form submits through (rendered in the
  template/LiveView, not `input/1` itself) become `.btn` / `.btn.ghost`.

No `CoreComponents` module is created or referenced (`f6.5.md`-INV8 preserved; the F6.8 fold of `input/1` into a styled set
is where `CoreComponents` is reconsidered). F6.5.5 strengthens that seam: it PROVES the three local components carry the
full F0 anatomy, so the deferred `CoreComponents` decision inherits a worked example rather than a guess.

**Two MANDATORY agile-shell layout rules** (the `/course/agile-agent-workflow` pages carry these; the Portal envelope
adopts them so the Portal reads as the same system):
- the clickable segmented **`.route-tag`** breadcrumb in the header (the model page renders it as `<a>` segments joined by
  `.rsep` separators with a final `.rcur` current-segment span — `html/agile-agent-workflow/what/four-artifacts.html:319`);
- the canonical 3-column **`.site-foot`** footer (model page `:504`).

**The delivery is build-free.** No node/asset build, no esbuild/tailwind step, no `mix assets.*`. The CSS ships as a
committed file; the envelope ships as a HEEx template; the components ship as restyled functions — every one a source file
served or rendered verbatim, exactly as F0 ships its static pages.

### Who

- **Visitor / Learner** — the LOOK: the catalog, cards, badge, and form render in the dark-editorial system the rest of
  jonnify uses, so the Portal is visually coherent with the courses it hosts.
- **Developer / Operator** — the FOUNDATION: the root layout + `app.css` + restyled components are the styling base every
  later F6 rung renders in, accepted via text proxies (below) rather than screenshots.

### Where

`lib/portal_web/components/layouts.ex` (new — `PortalWeb.Layouts`); `lib/portal_web/components/layouts/root.html.heex`
(new — the F0 envelope); `priv/static/assets/app.css` (new — tokens + base + fonts); `lib/portal_web.ex` (`layouts: []` →
`layouts: [html: PortalWeb.Layouts]` at `:30`; the `:live_view` macro gains `layout: {PortalWeb.Layouts, :root}` at `:52`);
`lib/portal_web/components/catalog_components.ex` (the three components restyled). Nothing in `apps/portal`, `apps/echo_data`,
the endpoint, or the config changes.

## Scope

- **In** — (1) `PortalWeb.Layouts` + `root.html.heex`, the F0-D3 envelope replacing `layouts: []`, bound on BOTH the
  controller render path (`layouts: [html: PortalWeb.Layouts]`) and the LiveView render path
  (`layout: {PortalWeb.Layouts, :root}`); (2) the committed `priv/static/assets/app.css` carrying the F0-D1 tokens (byte-for-byte),
  the F0 base CSS, and the four Google-Fonts `font-family` stacks, served by the existing `Plug.Static` `/assets` mount, with
  no endpoint/config change and no bundler; (3) the LOCAL `CatalogComponents` restyled to the F0 anatomy
  (`course_card`→`.mod`, `panel`→`.fig`, published-badge→`.pill.live`, the form buttons→`.btn`/`.btn.ghost`), each KEEPING
  its arity, attrs, and `data-*` hooks; (4) the two mandatory agile-shell rules in the envelope — the clickable `.route-tag`

- **Out** — `CoreComponents` (the kitchen-sink set; deferred to F6.8 where `input/1` folds into a styled component module);
  a node/asset/bundler build of any kind (esbuild, tailwind, dart_sass, `mix assets.*`); styling the authentication UI
  (→ F6.8 fold); the dashboard (→ F6.9); pixel-diff / screenshot testing (acceptance is the text proxies below);
  the inline KaTeX/Google-Fonts CDN-dependency choices as a SPEC concern (F0 noted them as external assets, not specified —
  the four font links are adopted, not re-litigated); any change to the markup STRUCTURE F6.5/F6.6 shipped beyond classes and
  the envelope; and any change below the facade (engine, contexts, store, ids — byte-unchanged).

## Invariants

- **F6.5.5-INV1 (facade-only — the master invariant)** — the layout, `app.css`, and restyled components name ONLY
  facade-rendered assigns and the `~p"/assets/app.css"` static route; nothing names `Portal.Engine`, a `Repo`,
  `GenServer.call`, or any module below the facade. Styling adds zero boundary crossings (preserves `f6.6.md`-INV1 /
  `f6.5.md`-INV7).
- **F6.5.5-INV2 (tokens declared once, byte-for-byte)** — the `:root` token palette is declared EXACTLY ONCE, in `app.css`,
  linked from the single shared root `<head>`; it is never redeclared per page or per component, and its values are copied
  byte-for-byte from `f0.md`-D1 (cite `f0.md`-INV2). Inside any `<svg>`, presentation attributes use the literal hex mapped
  by role (custom properties do not cascade into SVG attributes) — `f0.md`-INV2.
- **F6.5.5-INV3 (every `clamp()` spaced)** — every `clamp()` in `app.css` (and any inline style) has spaces around `+`/`-`
  (`clamp(2.7rem, 1.9rem + 4.2vw, 5.1rem)`, NEVER `1.9rem+4.2vw`); the unspaced form is invalid CSS, silently dropped to the
  UA default (h1 → 32px). This is the highest-leverage proxy — the unspaced bug shipped invisibly across 204 static pages
  before it was caught.
- **F6.5.5-INV4 (no `CoreComponents` — the LOCAL set carries F0)** — the build creates and references NO `CoreComponents`
  module; the three LOCAL components (`course_card`/`panel`/`input`) carry the full F0 anatomy. The F6.8 `CoreComponents`
  fold is deferred (`f6.5.md`-INV8 preserved); this rung is its worked example.
- **F6.5.5-INV5 (verified route, HEEx-escaped)** — the stylesheet `<link>` uses the verified `~p"/assets/app.css"` route (a
  typo fails to compile); every link in the envelope is a `~p` route or a literal external font URL; user data rendered in
  the layout/components stays HEEx-escaped (`f6.5.md`-INV2/INV3 preserved).
- **F6.5.5-INV6 (below the facade unchanged)** — the engine, the three contexts, `Portal.Store`, the ids, and the
  `EventStore` port are byte-unchanged; no file under `apps/portal` or `apps/echo_data`, no endpoint plug, and no config key
  is touched. Styling is purely an `apps/portal_web` view-layer concern.
- **F6.5.5-INV7 (renders-in-the-system == computed-style parity)** — "the Portal renders in the design system" is the
  CHECKABLE claim that the computed-style + geometry e2e (`apps/e2e/tests/elixir-hero.spec.ts`, written against computed
  styles and geometry — NOT pixels, `f0.roadmap.md` M5) passes against the styled Portal under `E2E_BASE_URL`, just as it
  passes against the static baseline.
- **F6.5.5-INV8 (liveness — the styled envelope on :4000)** — after this rung, `:4000` serves the STYLED envelope: a 200
  response carries the F0 `<head>` and the `app.css` stylesheet `<link>`, not a bare `<h1>`. A 200 with bare body markup
  (no `<head>`, no stylesheet link) is a REGRESSION. *(The determinism ≥100-iteration loop is NOT required for this rung —
  styling touches no id-minting and no process state, so the same-millisecond mint hazard the loop guards cannot fire here;
  the liveness criterion is the styled-envelope curl, not a repeated-suite loop.)*
