TWO-page contract below** ‚Äî both course-index pages, each at the route a Portal visitor would expect, each at parity with the
live static original:

What the two-page contract changes vs. the prior (single-page) framing (recorded here so the supersession is auditable):

- **Proof-pages ‚Äî TWO, not one.** Was: a single page at `/` reproducing `elixir/index.html`. Now: `/` reproduces
  `html/courses.html` (the two-course index) and `/elixir` reproduces `elixir/index.html` (the Elixir course index). The
  `/elixir` action + route is the ONE new web surface this rung adds.
- **Mechanism ‚Äî full-document templates, NO root layout.** Was: a `PortalWeb.Layouts` root layout (`root.html.heex`) replacing
  `layouts: []`, carrying a shared `<head>` for every render. Now: **each page is a complete-document HEEx template**

  (`layouts: []` already renders the current `:home` template bare, so `portal_web.ex` needs NO binding change). This
  SUPERSEDES the prior rung's root-layout mechanism and is the load-bearing correction.
- **Asset list ‚Äî four files, not two.** Was: a single `app.css` + `portal-index.js`. Now: each page's CSS + JS extract

- **Acceptance ‚Äî two routes, each vs its baseline.** A NEW Playwright spec (or specs) visits BOTH index pages and compares
  each Phoenix route to its Fiber baseline (`:4000/` vs `:8765/courses`; `:4000/elixir` vs `:8765/elixir`) on computed style +
  geometry, AND asserts the deep links resolve correctly (remapped where Portal does not serve them, relative where it does).

The mechanics that survive both reframes unchanged: the strangler-fig reframe (publish real pages, deep tree stays on Fiber),
computed-style + geometry parity (not pixels), the `https://jonnify.fly.dev` link-remap rule, no-bundler / no-`CoreComponents`,

## Rationale

The Operator named the reframe: **publish, real production, the routes a visitor expects**. A strangler-fig first slice ships
more value than a synthetic catalog skinned in the tokens ‚Äî the Portal serves the *actual* course-index pages at the *actual*
routes, byte-faithful to the originals, while the deeper tree stays on production Fiber. The acceptance is correspondingly
concrete: not "renders in the system" as a claim, but two two-origin computed-style comparisons against the live static pages.

### Why

An unstyled platform is not shippable, and a *synthetic* styled catalog proves less than *real* styled pages. The Portal's
root renders a bare `<h1>Portal</h1>` (`home.html.heex:6`); F0 already specified the Portal reproducing the static system as
planned work (`f0.roadmap.md` M4 "Portal HEEx components reproducing tokens/anatomy" + M5 "Parity gate: static vs Portal").
Reframing the proof onto two real course-index pages executes both milestones against pages that ALREADY EXIST in the system
and are independently servable for byte-for-byte comparison ‚Äî closing a foundation already drawn, against live witnesses.

### How (load-bearing)

The mechanism, surface by surface ‚Äî each cites the real code it changes or relies on.

**1 ¬∑ Each parity page is a COMPLETE-DOCUMENT HEEx template ‚Äî NO root layout.** Today the controller macro sets `layouts: []`
(`portal_web.ex:30`), so the current `page_html/home.html.heex` (`<h1>Portal</h1>`) already renders BARE ‚Äî no `<html>`/`<head>`
wrapper is injected. The two-page contract LEANS ON this: each template renders the entire document
(`<!doctype html>` ‚Üí `<html lang="en">` ‚Üí `<head>‚Ä¶</head>` ‚Üí `<body>‚Ä¶</body>` ‚Üí `</html>`), reproducing its OWN golden
master's `<head>`. A single shared root layout CANNOT serve both at byte-parity because the two heads are irreconcilable:
`html/courses.html`'s head (`:3-10`) has `<title>Courses ¬∑ jonnify</title>`, the `Cormorant Garamond:‚Ä¶1,400;1,500;1,600`
font subset, and two `<style>` blocks declaring `--surface`/`--surface-2`/`--body`/`--shadow-*`/`--rad`; `elixir/index.html`'s
head (`:3-10`) has `<title>Functional Programming in Elixir ‚Äî a jonnify course</title>`, a different font subset, and a
`<style>` declaring `--ink-2`/`--ink-3`/`--elixir`/`--measure:68ch` and the `.chap` system. They share token NAMES
(`--ink`/`--cream`/`--gold`/`--line`) but diverge in values, in the surface/measure tokens, and in every component rule.
Therefore: **two full-document templates, NO `PortalWeb.Layouts`, NO layout/macro change to `portal_web.ex`.** This SUPERSEDES
the prior rung's "root layout replacing `layouts:[]`" mechanism ‚Äî that mechanism would have forced ONE head, which cannot hold
both pages at byte-parity. *(The configurable-base pass adds exactly ONE function to `portal_web.ex` ‚Äî the
`deep_link_base/0` resolver beside `static_paths/0`, ¬ßHow.5 ‚Äî and NO layout binding, NO `:30` macro edit; the `layouts: []`
contract is untouched.)*

> **The new web surface for ratification.** The `/` route already exists (`router.ex:62`, `get("/", PageController, :home)`).

delimiter, so inlining a CSS `<style>` full of `*{‚Ä¶}`/`a{‚Ä¶}` rules in HEEx is unviable (HEEx would mis-parse the braces); and
inlining the JS would hit the same `{` problem plus HEEx mis-parsing the `<` comparisons. So each page's CSS and JS extract to

- `courses.css` ‚Äî the TWO `<style>` blocks of `html/courses.html` (`:12-110` the shared chrome + `:111-133` the courses-page
  chrome), concatenated, the CSS between the tags. Opens with the `:root` token block (`:13-28`).
- `courses.js` ‚Äî the `<script>` body of `html/courses.html` (`:174-189`: the reveal-on-scroll `IntersectionObserver` + the
  `.filter-bar` tag-filter).
- `elixir-index.css` ‚Äî the `<style>` body of `elixir/index.html` (`:12-308`, the CSS between `<style>` and `</style>`). Opens

- `elixir-index.js` ‚Äî the two `<script>` bodies of `elixir/index.html` (`:550-711` the arc chapter-selector + pipe builder

> | 2 | **Portal static assets** | `~p"/assets/courses.css"`, `‚Ä¶/courses.js`, `‚Ä¶/elixir-index.css`, `‚Ä¶/elixir-index.js`, `favicon.ico`, any local vendored asset | **LOCAL** ‚Äî server-rendered `<link>`/`<script>` tags via the `~p"/assets/‚Ä¶"` sigil, served at `:4000/assets/*` by Portal's OWN `Plug.Static` mount (this is exactly what FIX A `at: "/"` enabled). **NEVER remapped to the base.** *(Load-bearing: a Portal page pulling its own CSS/JS from `fly.dev` is a hollow shell, not a self-contained styled page ‚Äî and fails liveness even if it visually matches.)* |
> | 3 | **External CDN** | jsDelivr KaTeX, Google Fonts (`fonts.googleapis.com`, `fonts.gstatic.com`) | **UNTOUCHED** ‚Äî already absolute external. |
> | 4 | **Navigation deep-links to NON-served routes** | `/elixir/algebra`, `/elixir/<chapter>/<module>`, `/elixir/course‚Ä¶`, `/course/agile-agent-workflow` (and any `/edu/‚Ä¶`, `/school/‚Ä¶` were a page to carry them) | **THE BASE APPLIES HERE ‚Äî and only here.** `PortalWeb.deep_link_base() <> "<relative-path>"`. |
> | 5 | **Same-page anchors** | `#main`, any `#‚Ä¶` | **UNTOUCHED** ‚Äî in-page. |
>
> The two surfaces that need the base (the server-rendered `<a href>` markup and the static JS's arc link-builder) both carry
> ONLY category-4 links. The JS injection (below) hands the static JS the base VALUE for its ~6 `CH` nav routes (category 4)
> ONLY ‚Äî the JS must NEVER apply it to an asset fetch (the assets are category-2 `<link>`/`<script>` tags the server already
> rendered with `~p"/assets/‚Ä¶"`, untouched by the JS).

So the host is read from application config, ONCE, and reaches both **category-4** surfaces from that single read:

- **Config key + resolver.** `config :portal_web, :deep_link_base_url, "https://jonnify.fly.dev"` (in `config/config.exs`,
  overridable per env ‚Äî `config/runtime.exs` for a deployed override). A resolver `PortalWeb.deep_link_base/0` (a one-liner
  beside `static_paths/0` in `portal_web.ex`) returns `Application.get_env(:portal_web, :deep_link_base_url, "https://jonnify.fly.dev")`.
  The default keeps the rendered output byte-identical to the shipped pages; configurability changes only WHERE the host comes
  from, not what renders, so computed-style + geometry parity is untouched.
- **Surface (i) ‚Äî the HEEx markup (server-rendered).** Every remapped `<a href={‚Ä¶}>` in both templates interpolates the
  resolver with a RELATIVE path literal: `href={PortalWeb.deep_link_base() <> "/elixir/algebra"}` (the path is the literal; the
  base is the config read). The base is read at render time ‚Äî no `jonnify.fly.dev` literal survives in either `.heex`.
- **Surface (ii) ‚Äî the static JS (served byte-for-byte).** `elixir-index.js` is a `Plug.Static` file that CANNOT read Elixir
  config at serve time. So the `elixir.html.heex` `<head>` INJECTS the base ONCE ‚Äî **as-built, a
  `<meta name="deep-link-base" content={PortalWeb.deep_link_base()}>` tag** (`elixir.html.heex:15`), placed before
  `<script src={~p"/assets/elixir-index.js"}>`. The static JS reads it
  (`var META = document.querySelector('meta[name="deep-link-base"]'); var BASE = (META && META.content) || window.__deepLinkBase || ""`,
  `elixir-index.js:8-9`), reverts its six arc `CH` `"route"` fields to RELATIVE `/elixir/<chapter>` paths, and builds each "Open
  Fn" link as `BASE + route` (`elixir-index.js:37`) ‚Äî so the rendered link is still `https://jonnify.fly.dev/elixir/<chapter>`
  by default, but the host now comes from the injected value, not a baked literal. *(The `<meta>` carries an `{‚Ä¶}` HEEx
  expression in an ATTRIBUTE, which HEEx evaluates cleanly; the equivalent `<script>window.__deepLinkBase = "<base>";</script>`
  form is also HEEx-safe and the JS still reads it via the `|| window.__deepLinkBase` fallback ‚Äî the brief sanctioned EITHER,
  and the `<meta>` was chosen because HEEx 1.1.31 interprets a bare `{‚Ä¶}` inside a `<script>` BODY. Either way this single
  injected tag is the lone boundary crossing.)*

This is the load-bearing decision: **config crosses the server/static boundary exactly once** (the injected `<meta name="deep-link-base">`
the static JS reads ‚Äî or the equivalent `window.__deepLinkBase`), read by the markup at render time and by the static JS at run
time. The invariant it creates is **TWO-SIDED** (INV9), both
grep/probe-checkable:
- **(a) no hardcoded host** ‚Äî NO `jonnify.fly.dev` literal survives anywhere except the single `config.exs` default ‚Äî not in
  `courses.css`/`.js`, not in `elixir-index.css`/`.js`, not in either `.heex` body (only the one injected `window.__deepLinkBase`
  line, which carries the resolver VALUE).
- **(b) assets stay Portal-local** ‚Äî EVERY `/assets/*` reference in either page resolves to Portal's OWN `~p"/assets/‚Ä¶"` mount,
  proven by a probe: `:4000/assets/courses.css` ‚Üí **200 from Portal**, `content-type: text/css`, NOT a redirect/proxy to
  `fly.dev` (same for the `.js`). The base injection must NOT regress this ‚Äî the asset `<link>`/`<script>` tags (category 2)
  stay `~p"/assets/‚Ä¶"`, untouched. *(The as-built ALREADY satisfies (b): the rendered tags are root-relative `/assets/‚Ä¶` and
  the four assets serve 200 locally ‚Äî confirmed by probe. The polish PRESERVES it; it must not sweep an asset ref into the base.)*

The remap was always a DELIBERATE divergence from the golden master (whose links are relative); making the host configurable
changes neither the rule nor the rendered output, only its source ‚Äî and it touches ONLY category-4 nav links, never the
category-2 assets the Portal serves itself.

## Styling craft ‚Äî durable discipline (carry forward to every styling rung)

These are the load-bearing rules this presentation rung establishes; they generalize to F6.8's dynamic-UI styling and any later
visual work. They are encoded above as invariants, restated here as reusable craft.

1. **Clamp-spacing is the single highest-leverage check (INV3).** An unspaced `+`/`-` inside `clamp()` ‚Äî
   `clamp(2.7rem, 1.9rem+4.2vw, 5.1rem)` ‚Äî is **invalid CSS**: the browser silently drops the whole declaration and falls back
   to the UA default (`h1` ‚Üí 32px). It is **invisible** without a computed-style probe ‚Äî it shipped across 204 static pages
   undetected. Every styling rung gates on a clamp-spacing parse (`CSS.supports` / a grep for `[0-9a-z%)][+-][0-9a-z(]` inside
   `clamp(‚Ä¶)`) over every CSS file it touches.
2. **Per-page extracted CSS over `Plug.Static`, no bundler (¬ßHow.2).** HEEx treats `{` as an expression delimiter, so a CSS
   `<style>` body full of `a{‚Ä¶}` rules cannot be inlined in HEEx ‚Äî extract each page's CSS (and its JS, which also carries `{`
   and `<` HEEx would mis-parse) to a committed `priv/static/assets/*.css` / `*.js` file, `<link>`ed / `<script src>`'d via the
   verified `~p"/assets/‚Ä¶"` route. The umbrella has no asset pipeline by design ‚Äî this IS the no-build posture.
3. **VERBATIM relocation keeps parity byte-exact.** Relocate each master `<style>`/`<script>` body UNREFORMATTED ‚Äî do NOT
   re-indent, re-order, or re-wrap. Because the file is the master content moved unchanged, its `:root`/tokens and every
   `clamp()` are already correct by construction (INV2, INV3), and a byte-diff against the master `<style>` body is CLEAN. The
   moment a build "tidies" the CSS, parity drift and the clamp footgun both re-enter.
4. **Visual acceptance is computed-style + geometry, NEVER screenshots (INV7).** A headless build cannot take a screenshot, and
   pixels are brittle across font-rendering/sub-pixel differences between two servers. The proxy is `getComputedStyle`
   (type scale, font family/style) + `getBoundingClientRect` (geometry: ordering, multi-column, same-row) compared between the
   two origins ‚Äî robust, text-checkable, and the thing that actually catches a dropped clamp.
5. **A config value that crosses the server/static boundary crosses it EXACTLY ONCE (INV9, ¬ßHow.5).** When a server-rendered
   surface and a byte-served static asset both need the same runtime value (here, the deep-link base), read it from config once
   on the server and INJECT it once into the page (a single `<meta>` or `window.__x = "‚Ä¶"`) for the static asset to read ‚Äî never
   bake the value into the static file (which cannot read config) and never read it N times. One read, one injection, both
   surfaces.
