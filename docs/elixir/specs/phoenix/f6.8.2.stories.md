# F6.8.2 · user stories

> Who the deploy slice serves, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.8.2.md`](f6.8.2.md). The acceptance is BUILD-LOCAL —
> the live `fly deploy` is DEFERRED to the Operator (*"the user will create the fly app and machines"*), so every gate is
> a local check: the prod release builds, `Portal.Release.migrate/0` runs, the distilled `fly.toml` is schema-valid, a
> local prod boot answers `/health` 200. Stories honor the [`f6.8.2.md`](f6.8.2.md) reconcile: the Operator's two
> templates (`docs/deploy/Dockerfile.template`, `docs/deploy/fly.toml`) are reference INPUTS built FROM, not edited; the
> image version is corrected to Elixir 1.18.4 / OTP 28.1; the `apps/echo` build becomes a scoped umbrella release
> EXCLUDING `echo_bot`; the loopback bind, the `@node` mint, `Portal.Release`, libcluster, and the strangler-fig boundary
> are the carried deploy deliverables. Roles: Operator (the deployable artifacts), Developer (the release + config split +
> clustering to maintain). Framing: no gendered pronouns for agents; no perceptual/interior-state verbs for tools or
> agents; no first-person narration.

## F6.8.2-US1 — A scoped release that builds, excluding the out-of-band bot

As an **Operator**, I want a single umbrella release that packages only the Portal apps — not the F10 bot — so that
`mix release` produces one self-contained artifact I can deploy without dragging in an unrelated app.

Acceptance criteria
- Given the root `mix.exs` (today `:18-20` is `deps: []` with NO `releases/0`), when a `releases/0` is added, then it
  defines the Portal release (a pinned name) listing only `portal_web`, `portal`, `echo_data` as `:permanent`, and
  EXCLUDING `echo_bot` (per [`f6.8.2.md`](f6.8.2.md) D1, INV6).
- Given `MIX_ENV=prod mix release <name>`, when run, then it builds a self-contained release clean (no `mix` needed at
  runtime); the assembled release's applications include `portal_web`/`portal`/`echo_data` and NOT `echo_bot` (per
  [`f6.8.2.md`](f6.8.2.md) D1, D10).
- Given `apps/echo_bot`, when the release is built, then it is neither listed nor touched — the out-of-band F10 bot is a
  separate release (per [`f6.8.2.md`](f6.8.2.md) INV6; the echo-bot do-not-touch boundary).

INVEST — independent; testable by reading `releases/0` + `MIX_ENV=prod mix release <name>` + inspecting the release's app
list; encodes F6.8.2-D1, INV6.
Priority: must · Size: 2 · Implements deliverables: F6.8.2-D1.

## F6.8.2-US2 — A complete image, built from the template, on the right toolchain

As a **Developer**, I want a multi-stage Dockerfile completed from the Operator's template — version-corrected and
de-Codemoji'd — so that the image builds the real Portal apps on this umbrella's toolchain and ends in a runnable
release, not a half-built layer.

Acceptance criteria
- Given `docs/deploy/Dockerfile.template` (which pins `OTP_VERSION` 27 via `:3` and builds `apps/echo` via `:32-45`, and
  ENDS at `mix release` `:56` with no runtime stage), when the Dockerfile is authored FROM it, then the build args set
  `ELIXIR_VERSION=1.18.4` + `OTP_VERSION≈28.1` (the `.tool-versions` toolchain), `NODE_VERSION` is dropped, the
  `apps/echo` copy/build is replaced by copying the root `mix.exs`/`mix.lock`/`config` + `apps/portal`/`apps/portal_web`/`apps/echo_data`
  and building from the umbrella ROOT, and a slim runtime stage is added that `COPY`s the release, `EXPOSE 4000`, and
  `ENTRYPOINT ["bin/<name>", "start"]` (per [`f6.8.2.md`](f6.8.2.md) D2; RECONCILE 1, 2).
- Given the build pipeline, when it runs, then it is `mix deps.get --only prod` → `mix deps.compile` → `mix compile` →
  `mix phx.digest` → `mix release <name>` from the umbrella root (not `cd apps/echo`) (per [`f6.8.2.md`](f6.8.2.md) D2).
- Given `docs/deploy/Dockerfile.template`, when the rung is complete, then it is UNEDITED — the authored Dockerfile is a
  new file at the umbrella root; the template is a reference INPUT (per [`f6.8.2.md`](f6.8.2.md) INV6).

INVEST — independent; testable by diffing the authored Dockerfile against the template (version args, the umbrella-root
build, the runtime stage) + confirming the template is unedited; encodes F6.8.2-D2, RECONCILE 1/2.
Priority: must · Size: 3 · Implements deliverables: F6.8.2-D2.

## F6.8.2-US3 — The config split finished: secrets and host at boot, reachable bind in prod

As an **Operator**, I want the release to read its secrets, database URL, and host from the environment at boot — and to
bind an address the edge proxy can reach in production while staying loopback in dev — so that one image runs in any
environment and a misconfigured boot fails loudly instead of binding the wrong interface.

Acceptance criteria
- Given `config/runtime.exs:34` (today `http: [ip: {127, 0, 0, 1}, port: port]` in every non-test env), when it is
  reconciled, then under prod/release the bind is `{0, 0, 0, 0}` (or `:any`) and under dev it stays `127.0.0.1` (per
  [`f6.8.2.md`](f6.8.2.md) D3, INV3; the F6.6 loopback finding).
- Given a production boot, when env is read, then `PHX_HOST` is read into the endpoint `url` host (joining the existing
  `SECRET_KEY_BASE` raise `runtime.exs:15-29` and `DATABASE_URL` raise `:43-58`), and a missing required prod env var
  fails the boot loudly (per [`f6.8.2.md`](f6.8.2.md) D3, INV2).
- Given `config/prod.exs` (today only the event-store adapter pick), when it is extended, then it sets the endpoint
  `url: [host: …, port: 443, scheme: "https"]` (the host resolved at runtime), `cache_static_manifest:
  "priv/static/cache_manifest.json"` (the `mix phx.digest` output), and `server: true` gated on `PHX_SERVER` — and holds
  NO secret at compile time (per [`f6.8.2.md`](f6.8.2.md) D4, INV2).
- Given the deployed `:deep_link_base_url`, when set, then it is set in `runtime.exs` (the override mechanism already
  exists at `:65-67`) (per [`f6.8.2.md`](f6.8.2.md) D3, D9).

INVEST — independent; testable by reading `runtime.exs`/`prod.exs` + a boot with/without the required env vars (loud
raise on missing prod secret) + a grep that no secret lives in compile-time config; encodes F6.8.2-D3, D4, INV2, INV3.
Priority: must · Size: 3 · Implements deliverables: F6.8.2-D3, D4.

## F6.8.2-US4 — Migrations run from a release that has no mix

As an **Operator**, I want to migrate the database from the release binary — not `mix` — so that the schema is brought up
to date before the app serves, on a machine that has no build toolchain.

Acceptance criteria
- Given a NEW `apps/portal/lib/portal/release.ex`, when read, then `Portal.Release` exposes `migrate/0` (loads the app
  and runs `Ecto.Migrator` for `Portal.Repo` to the latest version) and `rollback/2` (`Repo`, `version`) (per
  [`f6.8.2.md`](f6.8.2.md) D5, INV1).
- Given `bin/<name> eval "Portal.Release.migrate()"`, when run against a prod-config DB, then the migration applies (the
  schema is brought current) without invoking `mix` (per [`f6.8.2.md`](f6.8.2.md) D5, D10, INV1).
- Given the deploy sequence, when ordered, then migrate runs BEFORE the app serves (the schema is current before the
  endpoint accepts traffic) (per [`f6.8.2.md`](f6.8.2.md) INV1).

INVEST — independent; testable by reading `release.ex` + running `bin/<name> eval` against a local prod-config DB;
encodes F6.8.2-D5, INV1.
Priority: must · Size: 2 · Implements deliverables: F6.8.2-D5.

## F6.8.2-US5 — Real-time correct across the cluster

As a **Developer**, I want a libcluster topology in the supervision tree so that the F6.7 PubSub broadcasts and Presence
counts — single-node today — are correct across every node once the app runs on more than one machine.

Acceptance criteria
- Given `apps/portal_web/mix.exs` (where `libcluster`/`dns_cluster` are NOT locked today), when its `deps/0` is read,
  then `{:libcluster, "~> 3.3"}` is declared THERE (the per-app DEP-GRAPH-VISIBILITY edge — Presence + the endpoint live
  in `portal_web`) and present in `mix.lock` (per [`f6.8.2.md`](f6.8.2.md) D6).
- Given `PortalWeb.Application`, when read, then a `Cluster.Supervisor` child is added with a Fly-suited topology (e.g.
  DNSPoll/Gossip over the `.internal` network — the `fly.toml:70-78` IPv6/`ERL_AFLAGS` trio its substrate) (per
  [`f6.8.2.md`](f6.8.2.md) D6, INV4).
- Given the libcluster topology, when nodes are connected, then `Phoenix.PubSub` broadcasts and `PortalWeb.Presence`
  counts span all nodes; the web still reaches PubSub/Presence ONLY through the `Portal` facade (`Portal.subscribe/1`,
  `Portal.broadcast/2`) — clustering is a supervision-tree + transport concern, not a new web→engine path (per
  [`f6.8.2.md`](f6.8.2.md) INV4).

INVEST — independent; testable by reading `apps/portal_web/mix.exs`/`mix.lock` + `PortalWeb.Application` (the
`Cluster.Supervisor` child) + a boundary grep (the web names only the facade for PubSub); encodes F6.8.2-D6, INV4.
Priority: must · Size: 3 · Implements deliverables: F6.8.2-D6.

## F6.8.2-US6 — The branded-id mint is safe on more than one machine

As a **Developer**, I want the branded-id `worker_id` derived per machine so that two nodes minting in the same
millisecond do not produce the same id once the app is clustered — the latent that single-node never exercised.

Acceptance criteria
- Given `apps/portal/lib/portal/id.ex:13` (today `@node 1`, its own comment "F6.8 derives it per machine"), when
  reconciled, then the 10-bit node is derived from the machine identity (e.g. `:erlang.phash2(System.get_env("FLY_MACHINE_ID"),
  1024)`), so two machines minting in the same millisecond do not collide (per [`f6.8.2.md`](f6.8.2.md) D7, INV5;
  `echo/CLAUDE.md §4`).
- Given a single node, when ids are minted, then the behavior is unchanged (one node → one `worker_id`); the branded-id
  format, namespaces, and the `Portal.ID` surface are otherwise unchanged (per [`f6.8.2.md`](f6.8.2.md) D7, INV5).
- Given the catalog write path (`create_course/1`, F6.6), when the mint hardening is considered, then a
  `unique_constraint(:id)` is noted as a complementary guard (per [`f6.8.2.md`](f6.8.2.md) D7). *(the exact `worker_id`
  source — `FLY_MACHINE_ID` hash vs. a region+ordinal scheme — is the open item the Director pins.)*

INVEST — independent; testable by reading the reconciled `id.ex` (the per-machine derivation) + confirming single-node
behavior is unchanged; encodes F6.8.2-D7, INV5.
Priority: must · Size: 2 · Implements deliverables: F6.8.2-D7.

## F6.8.2-US7 — A fly.toml distilled to the Portal essentials

As an **Operator**, I want a `fly.toml` that carries only what the Portal needs — distilled from the full Codemoji
production config — so that I deploy a Phoenix release with a health check aligned to the app's real liveness route and
no dead Fastify/Redis/Datadog config to confuse the next reader.

Acceptance criteria
- Given `docs/deploy/fly.toml` (the full Codemoji config), when the Portal `fly.toml` is authored, then it KEEPS `app`
  (a placeholder the Operator names), `primary_region = "fra"`, `kill_signal`/`kill_timeout`, `[build] dockerfile` +
  `[build.args]` (`ELIXIR_VERSION=1.18.4`, `OTP_VERSION≈28.1`, `DEBIAN_DATE` — drop `NODE_VERSION`),
  `[deploy] strategy = "rolling"`, `[experimental] auto_rollback`, the `[env]` Phoenix vars (`PHX_HOST`/`PHX_SERVER`/`MIX_ENV`/`PORT`/`POOL_SIZE`)
  + the Erlang IPv6/cluster trio (`ERL_INETRC`/`ECTO_IPV6`/`ERL_AFLAGS`, `fly.toml:70-78`) + `TZ`/`LANG`, `[compute]`,
  and `[[services]]` `internal_port = 4000` + the 80→443 `force_https` ports (per [`f6.8.2.md`](f6.8.2.md) D8).
- Given the health check, when read, then `[[services.http_checks]]` is `GET /health` — which MATCHES the F6.1 liveness
  route (`router.ex:93`, `PortalWeb.CourseController, :health`) — so a healthy machine is one the proxy reaches AND the
  app answers (per [`f6.8.2.md`](f6.8.2.md) D8, INV3; the alignment that makes the distilled config correct).
- Given the Codemoji-only config, when distilled, then it is DROPPED entirely: all Fastify/`WORKER_*`/`FASTIFY_PORT`/`QUEUE_*`/`NODE_*`,
  Redis/BullMQ, SQLitestream/`DISTR_WATCH_*`/`PACKAGES_DB`/`VOLUME_WATCH_*`, all `DD_*` (Datadog), `GAME_DB_NAME`,
  `DEV_LIVE`, and the Codemoji header comments (per [`f6.8.2.md`](f6.8.2.md) D8).
- Given the distilled `fly.toml`, when validated, then it passes `fly config validate` (if the CLI is available, else a
  schema review); `docs/deploy/fly.toml` itself is UNEDITED (per [`f6.8.2.md`](f6.8.2.md) D8, D10, INV6).

INVEST — independent; testable by a keep/drop diff of the distilled `fly.toml` against the template + `fly config
validate` + confirming the `/health` path matches `router.ex:93` + the template is unedited; encodes F6.8.2-D8.
Priority: must · Size: 3 · Implements deliverables: F6.8.2-D8.

## F6.8.2-US8 — The two-origin boundary, stated and set, with BUILD-LOCAL acceptance

As an **Operator**, I want the production boundary named — which routes the Portal serves vs. which deep links fall back
to the Fiber app — and the deployed deep-link base set, so that the strangler-fig slice is explicit and I can deploy the
artifacts and run the live `fly deploy` myself, against a rung whose gate is a local prod-release boot.

Acceptance criteria
- Given the deploy spec, when read, then it names the routes the Portal OWNS in production (`/`, `/elixir`,
  `/course/agile-agent-workflow`, `/login`, `/auth/session`, `/auth/reset`, `live /courses`, `/my/courses`, `/health`)
  vs. the Fiber fallback (every deeper `/elixir/…` / `/course/…` subpage via `:deep_link_base_url`) (per
  [`f6.8.2.md`](f6.8.2.md) D9, INV7).
- Given the deployed environment, when `:deep_link_base_url` is set, then it is set in `runtime.exs` (the F6.5.5
  mechanism), pointing the Portal's deep links at the Fiber origin (per [`f6.8.2.md`](f6.8.2.md) D9).
- Given BUILD-LOCAL verification, when run, then `MIX_ENV=prod mix release <name>` builds, `Portal.Release.migrate/0`
  runs against a prod-config DB, the distilled `fly.toml` passes `fly config validate`, and a local prod-config boot
  answers `GET /health` with 200 — and the live `fly deploy` (+ `fly apps create`, the machines) is [DEFERRED to the
  Operator] (per [`f6.8.2.md`](f6.8.2.md) D10; the Director instruction). *(the "deployed URL renders" gate became "the
  local prod-release boot + `/health` 200 + the artifacts present and reconciled".)*
- Given the whole change set, when reviewed, then the F5 engine, the `EventStore` port, the catalog/learning/accounts
  contexts, and the store are byte-unchanged — F6.8.2 adds a release, a config split, a migrate task, clustering, and a
  per-machine `worker_id` (a runtime-derived value) as web/ops concerns; the web still calls only the `Portal` facade
  (per [`f6.8.2.md`](f6.8.2.md) INV7).

INVEST — independent; testable by reading the stated boundary + the deployed `:deep_link_base_url` + the BUILD-LOCAL
recipe (release builds, migrate runs, `fly config validate`, `/health` 200) + a diff-scope review below the facade;
encodes F6.8.2-D9, D10, INV7.
Priority: must · Size: 2 · Implements deliverables: F6.8.2-D9, D10.
