# F6.3 · Persistence with Ecto

> Make the Portal durable without making the core depend on a database. Add `Ecto` as one more adapter *behind the F5
> `Portal.EventStore` port*: a Repo, a migration with Snowflake `:bigint` ids, a `Course` schema, a changeset that
> parses untrusted input at the boundary, an error bridge into the closed `%Portal.Error{}` set, composable queries,
> and the Postgres event-store adapter — selectable by config, with tests staying in-memory. Nothing above the
> `Portal` facade learns that Postgres exists.

## Goal

After F6.3, the Portal persists in PostgreSQL in production and keeps running on the in-memory adapter in tests, with
the choice made by config alone. Untrusted attributes enter the domain only through an `Ecto.Changeset` that yields a
typed struct or a field-level `%Portal.Error{}`. Snowflake ids are `:bigint` columns the Portal mints itself. The
engine's event log is read and appended through `Portal.EventStore.Postgres`, which satisfies the exact F5 behaviour —
so the engine, contexts, and web layer are unchanged.

## Rationale (5W)

- **Why** — F5/F6.1–F6.2 hold state in memory; a learner's enrollment, progress, and streak must survive a restart or
  deploy. Durability has to arrive as an *adapter*, so the engine and facade stay database-free and the F5 boundary
  holds.
- **What** — a Repo and migration (Snowflake `:bigint` ids), a `Course` schema, a parse-at-the-boundary changeset with
  an error bridge, composable queries, and the Postgres adapter for the F5 `Portal.EventStore` port, chosen by config.
- **Who** — Operators (data survives restarts/deploys), Learners (durable progress underpins streaks), Developers (a
  single, safe persistence pattern; the web stays free of Ecto).
- **When** — the third rung; depends on F6.1's running app and the F5.08/F5.09 `EventStore` port. It precedes the
  Catalog *context* (F6.4), which orchestrates these primitives behind the facade.
- **Where** — a new persistence layer (`Portal.Repo`, `Portal.Catalog.Course`, `Portal.EventStore.Postgres`,
  `Portal.EventStore.Event`) plus `Portal.Error.from_changeset/1` and one new supervision child, `Portal.Repo`. The
  web layer and the `Portal` facade are untouched.

## Scope

- **In** — the Repo + config + supervision wiring (Repo starts first); a migration creating `:courses` and `:events`
  with explicit `:bigint` Snowflake ids; the `Course` schema; a changeset (`cast`/`validate`/`unique_constraint`) as
  the parse boundary; `Portal.Error.from_changeset/1`; composable queries (`published/1`, `Repo.get` by id); the
  `Portal.EventStore.Postgres` adapter (`append/2`, `read_stream/1`) selected by config; the `Ecto.Multi`/transaction
  pattern for atomic multi-row writes; sandbox-based DB tests plus DB-free tests of the pure changeset/query builders.
- **Out** (deliberately deferred, to avoid overengineering) — the **Catalog context** public API and cross-context
  composition (→ F6.4); separate read-model / projection tables or CQRS read stores; a second database or read
  replicas; soft-deletes, multi-tenancy, and caching; any `Ecto`/`Repo`/schema reference in `lib/portal_web/`.

## Deliverables

- **F6.3-D1** — `Portal.Repo` (`use Ecto.Repo, otp_app: :portal, adapter: Ecto.Adapters.Postgres`), configured in
  `config/*.exs` and `config/runtime.exs`, added to `Portal.Application` as the **first** child (before the store and
  engine, since the Postgres adapter and the engine's `init` read through it).
- **F6.3-D2** — migrations creating `:courses` (`primary_key: false`, explicit `add :id, :bigint, primary_key: true` —
  the canonical Snowflake integer the branded-string surface decodes to — plus `:title`, `:slug`, `:published`,
  `timestamps(type: :utc_datetime)`) and `:events` (`:bigint` id, `:stream`, `:seq`, `:type`, `:data` map/jsonb,
  inserted_at; index on `[:stream, :seq]`), because the Portal mints its own Snowflakes.
- **F6.3-D3** — extend the existing `Portal.Catalog.Course` struct
  (`echo/apps/portal/lib/portal/catalog/course.ex`) into an Ecto schema. The `:id` **surfaces as the branded string**
  (the frozen `Portal.ID` convention) via a custom `Ecto.Type` over the `:bigint` column — not `@primary_key {:id, :id}`
  (a raw integer). Fields: `:id` (branded-string surface / `:bigint` column, `autogenerate: false`), `:title` (string),
  `:slug` (string — **keep**, existing), `:published` (boolean, default `false`), `timestamps(type: :utc_datetime)`. The
  `@derive {Jason.Encoder, only: [:id, :title, :slug]}` and the namespace Store keep working unchanged.
- **F6.3-D4** — `Course.changeset/2` as the parse-don't-validate boundary: `cast(attrs, [:title, :slug, :published])`,
  `validate_required([:title, :slug])`, `validate_length(:title, min: 3)`, `unique_constraint(:title)`; returns an
  `%Ecto.Changeset{}`. (`:slug` is a persisted catalog field cast at the boundary like `:title`, not derived.)
- **F6.3-D5** — `Portal.Error.from_changeset/1` extending F5.08: turn a failed changeset into a `%Portal.Error{}` with
  `code: :invalid` (or a mapped code) and `field:` set from the first changeset error, so callers receive the closed
  contract, never a raw changeset.
- **F6.3-D6** — composable queries: a `published/1` query value run with `Repo.all`, and `Repo.get(Course, id)` by the
  branded-string id (decoded to the `:bigint` column by the D3 type); query builders are plain functions returning an
  `Ecto.Queryable`.
- **F6.3-D7** — fill the `Portal.EventStore.Postgres` **adapter body** to implement the F5 `Portal.EventStore`
  behaviour: `append/2` via `Repo.insert_all(Event, rows)` and `read_stream/1` via `Repo.all(from e in Event, where:
  e.stream == ^stream, order_by: e.seq)`. The selecting `config :portal, :event_store, …` line **already exists**
  (built F5.8 — `Portal.EventStore.Postgres` in `config/prod.exs`, `…InMemory` in `config/{config,dev,test}.exs`); F6.3
  fills the adapter, not the config. This is the only module in the persistence layer that names `Repo` for the event
  log.
- **F6.3-D8** — the atomic-write pattern: any operation writing more than one row uses `Ecto.Multi` /
  `Repo.transaction`, returning `{:ok, _} | {:error, step, reason, _}`; `append/2` writes its event batch atomically.
  The pattern is established here for F6.4's multi-step context operations.
- **F6.3-D9** — verification: migrations run; a course round-trips through `changeset` + `Repo.insert` + `Repo.get`; an
  invalid changeset yields a `%Portal.Error{}` with a `field`; the Postgres adapter round-trips `append` →
  `read_stream` in order; switching `:event_store` config to `InMemory` changes no caller; DB tests run in the
  `Ecto.Adapters.SQL.Sandbox`; pure changeset/query builders are tested without the DB.

## Invariants

- **F6.3-INV1 (master, extended)** — `Ecto`, `Repo`, and schema modules appear only in the persistence/domain layer;
  no module under `lib/portal_web/` names `Repo`, an `Ecto.Schema`, or `Ecto.Query`. The web still calls only `Portal`.
- **F6.3-INV2 (parse, don't validate)** — untrusted attributes reach the domain only through a changeset that returns
  either a typed struct or a field-level `%Portal.Error{}`; no raw, unvalidated map is persisted or passed downstream.
- **F6.3-INV3 (identity)** — Snowflake ids are `:bigint` integers with `autogenerate: false`; the Portal mints them.
  The branded string is only the transport form (encode/decode at the boundary) and is **never a column**. The
  `Course` schema/struct `:id` **surface** remains the branded string (the frozen `Portal.ID` convention), bridged to
  the `:bigint` column by a custom `Ecto.Type` / boundary encode-decode.
- **F6.3-INV4 (atomicity)** — any operation that writes more than one row runs inside `Ecto.Multi` /
  `Repo.transaction`; a failure rolls back fully, so no partial write is ever observable.
- **F6.3-INV5 (port substitutability)** — `Portal.EventStore.Postgres` satisfies the exact `Portal.EventStore`
  behaviour; selecting it versus `InMemory` is a config change that alters no caller (the F5 boundary holds).
- **F6.3-INV6 (test determinism)** — pure changeset and query-building functions are tested with no database; every
  DB-touching test runs inside the Ecto sandbox, `async: true` wherever the sandbox allows.

## Control-flow note

Apply the decision matrix in [the specs approach](../specs.approach.md): a single-row context write composes with
`with` + tagged tuples; a multi-row or must-be-atomic write uses `Ecto.Multi` / `Repo.transaction`; untrusted input is
parsed by a `changeset` and bridged with `from_changeset/1`; a lost connection or programmer error is left to crash
under the supervisor. Expected failures never raise.

## Definition of Done

- [ ] `mix ecto.migrate` creates `:courses` and `:events` with `:bigint` Snowflake-id columns.
- [ ] A `Course` round-trips: `changeset` → `Repo.insert` → `Repo.get` by the **branded-string id** (decoded to the
      `:bigint` column) returns the struct with its branded-string `:id` surface (F6.3-INV3).
- [ ] An invalid changeset (missing/short title, duplicate) yields `%Portal.Error{}` with a `field` (F6.3-INV2).
- [ ] `Portal.EventStore.Postgres` round-trips `append/2` → `read_stream/1` in `:seq` order; switching config to
      `InMemory` changes no caller and the suite stays green (F6.3-INV5).
- [ ] A multi-row write demonstrably rolls back fully on failure (F6.3-INV4).
- [ ] `grep -rE "Repo|Ecto\.(Schema|Query|Changeset)" lib/portal_web/` is empty (F6.3-INV1).
- [ ] DB tests run in the sandbox; pure changeset/query builders have DB-free tests (F6.3-INV6).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f6.3.stories.md`).
