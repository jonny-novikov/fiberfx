# F6.4 · user stories

> Who the contexts serve, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.4.md`](f6.4.md).

## F6.4-US1 — One import for the web

As a **developer**, I want a single `Portal` facade to call, so that controllers depend on one surface and never reach
into a context or `Repo`.

Acceptance criteria
- Given the web layer, when it acts, then it calls only `Portal`.
- Given `Portal`, when inspected, then each public function is a `defdelegate` to a context and it owns no logic.
- Given a controller, when inspected, then it names no context, schema, or `Repo`.

INVEST — independent; testable by the call graph; encodes F6.4-INV2.
Priority: must · Size: 2 · Implements deliverables: F6.4-D4.

## F6.4-US2 — Catalog owns its data

As a **developer**, I want the `Catalog` context to own its schema and queries, so that the rest of the app reads
courses through a small public API and never touches `Repo`.

Acceptance criteria
- Given `Catalog`, when called, then `list_courses/0`, `get_course!/1`, `fetch_course/1`, and `create_course/1` return
  their documented shapes.
- Given the module, when inspected, then the `Course` alias, `Ecto.Query` import, and all `Repo.*` are private.
- Given a miss, when `get_course!/1` runs, then it raises; when `fetch_course/1` runs, then it returns `{:error,
  :not_found}`.

INVEST — independent; testable per function; encodes F6.4-INV1.
Priority: must · Size: 3 · Implements deliverables: F6.4-D1.

## F6.4-US3 — Enrollment over the port

As a **developer**, I want `Enrollment` to read and append through the `EventStore` port, so that it runs unchanged in
tests (in-memory) and production (Postgres).

Acceptance criteria
- Given `Enrollment`, when it persists or reads, then it uses the `Portal.EventStore` port, not the `Repo`.
- Given the in-memory adapter, when tests run, then `Enrollment` works without change.
- Given the Postgres adapter, when production runs, then the same code works without change.

INVEST — independent; testable by adapter substitution; encodes F6.4-INV4.
Priority: must · Size: 3 · Implements deliverables: F6.4-D2.

## F6.4-US4 — Contexts compose cleanly

As a **developer**, I want contexts to call each other only through public functions, so that boundaries hold and the
dependency graph stays acyclic.

Acceptance criteria
- Given `Enrollment.enroll/2`, when it needs a course, then it calls `Catalog.fetch_course/1` and branches on the
  public struct.
- Given any outcome, when it resolves, then it folds into the closed `%Portal.Error{}` set.
- Given the graph, when inspected, then edges are one-way (`Enrollment → Catalog`) with no cycles and no foreign schema
  access.

INVEST — independent; testable by the composition and the graph; encodes F6.4-INV3, INV5.
Priority: must · Size: 2 · Implements deliverables: F6.4-D5.

## F6.4-US5 — Enroll and welcome

As a **learner**, I want enrolling to also set up my welcome in one step, so that a single action does the right thing
across slices or fails cleanly.

Acceptance criteria
- Given `enroll_and_welcome/2`, when it runs, then it chains public calls across `Catalog`, `Enrollment`, and
  `Accounts` with `with`.
- Given a failure at any step, when it occurs, then the flow short-circuits to one closed `%Portal.Error{}`.
- Given success, when it completes, then each step's effect is applied.

INVEST — independent; testable by the orchestration; encodes F6.4-INV5, INV6.
Priority: should · Size: 2 · Implements deliverables: F6.4-D6.

## F6.4-US6 — No boundary leaks

As an **architect**, I want the dependency discipline enforced, so that consistency and encapsulation are not quietly
broken.

Acceptance criteria
- Given any context, when inspected, then it touches no other context's schema or `Repo`.
- Given an atomic multi-write, when written, then it uses `Ecto.Multi` within one context.
- Given cross-context consistency, when needed, then it uses a `with` chain, never one transaction across contexts.

INVEST — independent; testable by inspection; encodes F6.4-INV6.
Priority: must · Size: 1 · Implements deliverables: F6.4-D7.
