# F6.3 · user stories

> Who needs persistence, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.3.md`](f6.3.md).

## F6.3-US1 — Data survives restarts and deploys

As an **operator**, I want catalog and event data stored in PostgreSQL, so that nothing is lost when a node restarts or
a new release is deployed.

Acceptance criteria
- Given a running app with the Postgres adapter, when a course is inserted and the node restarts, then the course is
  still retrievable.
- Given the engine's event log in Postgres, when the engine restarts, then `init` replays the log from the database to
  the same state.
- Given the boot sequence, when children start, then `Portal.Repo` starts before the store and the engine.

INVEST — independent of UI; testable by insert + restart + read; encodes the supervision order.
Priority: must · Size: 3 · Implements deliverables: F6.3-D1, F6.3-D2, F6.3-D7.

## F6.3-US2 — Untrusted input is parsed at the boundary

As a **developer**, I want external attributes turned into a typed struct or a field-level error at the edge, so that
no invalid or unexpected data ever reaches the domain.

Acceptance criteria
- Given valid attributes, when they pass through `Course.changeset/2`, then a valid changeset (and on insert, a
  `%Course{}`) results.
- Given a missing or too-short title, when it passes through the changeset, then `Portal.Error.from_changeset/1`
  returns a `%Portal.Error{}` with the offending `field`.
- Given a duplicate title, when insert runs, then the `unique_constraint` produces a `%Portal.Error{}`, not a raw
  database error.

INVEST — valuable (the safety boundary); small; testable on the pure changeset with no DB for the validation cases.
Priority: must · Size: 3 · Implements deliverables: F6.3-D4, F6.3-D5.

## F6.3-US3 — Durable progress for learners

As a **learner**, I want my enrollment and progress stored durably, so that my streak and history are not lost between
sessions or deploys.

Acceptance criteria
- Given I have progress recorded, when I return after a restart, then my progress is intact (read back from Postgres
  via the engine's replayed log).
- Given a recorded action, when it is persisted, then it is written as an event in the `:events` table with a `:seq`
  ordering.

INVEST — valuable (underpins streaks); testable via the event-store round-trip; independent of the bot front-end.
Priority: must · Size: 2 · Implements deliverables: F6.3-D2, F6.3-D7.

## F6.3-US4 — Multi-step writes are atomic

As a **developer**, I want any operation that writes more than one row to be all-or-nothing, so that a failure never
leaves half-written state.

Acceptance criteria
- Given a multi-row write expressed as an `Ecto.Multi`, when one step fails, then the whole transaction rolls back and
  the database is unchanged.
- Given `append/2` writing a batch of events, when it runs, then all rows commit together or none do.
- Given a successful multi-step write, when it commits, then it returns `{:ok, _}`; on failure `{:error, step,
  reason, _}`.

INVEST — independent; testable by forcing a step failure; encodes F6.3-INV4.
Priority: must · Size: 2 · Implements deliverables: F6.3-D8.

## F6.3-US5 — Swappable storage, in-memory in tests

As an **operator**, I want the engine's event store chosen by config, so that production uses durable Postgres while
the test suite stays fast and in-memory.

Acceptance criteria
- Given `config :portal, :event_store, Portal.EventStore.Postgres`, when the app runs, then events persist to
  Postgres.
- Given `…InMemory` in the test environment, when the suite runs, then no database is required for the in-memory paths
  and no caller changes.
- Given either adapter, when the engine calls the port, then both satisfy `append/2` and `read_stream/1` identically.

INVEST — independent; testable by running the suite under each config; encodes F6.3-INV5.
Priority: must · Size: 2 · Implements deliverables: F6.3-D7.

## F6.3-US6 — The web layer stays free of the database

As a **developer**, I want no `Repo`, schema, or query in the web layer, so that the facade boundary that makes F6 safe
is preserved as persistence is introduced.

Acceptance criteria
- Given the web layer, when I search `lib/portal_web/`, then no module references `Repo`, `Ecto.Schema`, or
  `Ecto.Query`.
- Given a controller or LiveView needing data, when it reads, then it calls only the `Portal` facade.

INVEST — independent; testable by a static search; this is the master invariant under persistence.
Priority: must · Size: 1 · Implements deliverables: F6.3-D1, F6.3-D7.
