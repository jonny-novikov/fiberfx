## Requirements

- **F6.1-R1** — `PortalWeb.Endpoint` exists as the outermost plug with `Plug.Static`, `Plug.RequestId`,
  `Plug.Telemetry`, `Plug.Parsers` (urlencoded/multipart/json via `Jason`), `Plug.Session`, and `plug
  PortalWeb.Router` last; it declares `socket "/live", Phoenix.LiveView.Socket`. [US: F6.1-US1]
- **F6.1-R2** — `Portal.Application.start/2` supervises `[Portal.EventStore.adapter(), {Portal.Engine, []},
  PortalWeb.Endpoint]` with `strategy: :one_for_one`; the first two entries are byte-for-byte the F5 ones and precede
  the endpoint. [US: F6.1-US1, F6.1-US3, F6.1-US4]
- **F6.1-R3** — `PortalWeb.Router` defines a `:browser` pipeline (`fetch_session`, `fetch_live_flash`,
  `protect_from_forgery`, `put_secure_browser_headers`) and, inside `scope "/", PortalWeb` piping `:browser`, the route
  `get "/courses/:user_id", CourseController, :index`. [US: F6.1-US2]
- **F6.1-R4** — `PortalWeb.CourseController.index/2` calls only `Portal.courses_of/1`; on success it `render(conn,
  :index, courses: courses)`; on `{:error, %Portal.Error{} = e}` it `conn |> put_status(422) |> render(:error,
  error: e)`. No other module is called for domain data. [US: F6.1-US2, F6.1-US3, F6.1-US5]
- **F6.1-R5** — `PortalWeb.CourseHTML` embeds `course_html/index.html.heex` (and an `error` template) that render
  purely from `assigns`; no engine or repo symbol appears in any template. [US: F6.1-US2]
- **F6.1-R6** — `GET /health` returns `200` with body `ok` and performs no domain call. [US: F6.1-US1]
- **F6.1-R7** — no module under `lib/portal_web/` contains the strings `Portal.Engine`, `Repo`, or `GenServer.call`
  (the master invariant, statically checkable). [US: F6.1-US3]
- **F6.1-R8** — an expected domain failure yields a `422`, never a `500`; killing the endpoint restarts it under the
  supervisor. [US: F6.1-US4, F6.1-US5]

## Execution topology

**Runtime topology (what exists and how a request flows).**

```text
Portal.Application (Supervisor, :one_for_one)
├─ Portal.EventStore.adapter()        # F5 — unchanged, starts first
├─ {Portal.Engine, []}                # F5 — unchanged, starts second
└─ PortalWeb.Endpoint                 # F6.1 — the one new child, starts last
        │  (request)
        ▼
   Plug.Static → Plug.RequestId → Plug.Telemetry → Plug.Parsers → Plug.Session → PortalWeb.Router
        │
        ├─ pipeline :browser  → scope "/" → CourseController.index/2 ──calls──▶ Portal.courses_of/1 ──▶ Portal.Engine
        │                                         │
        │                                         └─ render CourseHTML :index | :error(422)   (assigns only)
        └─ GET /health → 200 "ok"                                              (no domain call)
```

The health route is the only path that does not reach the facade; every domain path goes through `Portal` and returns
either data or a `%Portal.Error{}`.

**Task topology (build order; each step leaves the app compiling).**

```text
T1 add Phoenix deps + PortalWeb entrypoint (lib/portal_web.ex: controller/html helpers) + Telemetry + endpoint config
      └─▶ T2 PortalWeb.Endpoint (plug stack + LiveView socket)            [R1]
            └─▶ T3 wire Endpoint into Portal.Application AFTER the engine [R2]
                  └─▶ T4 PortalWeb.Router (:browser pipeline + courses route + /health)  [R3, R6]
                        └─▶ T5 PortalWeb.CourseController.index/2 over Portal.courses_of/1 [R4, R8]
                              └─▶ T6 PortalWeb.CourseHTML + index.html.heex + error template [R5]
                                    └─▶ T7 verify: boot, render, /health, endpoint restart, invariant grep [R7, R8]
```

**Touched files.** `mix.exs`, `config/config.exs`, `config/runtime.exs`, `lib/portal_web.ex`,
`lib/portal_web/telemetry.ex`, `lib/portal_web/endpoint.ex`, `lib/portal/application.ex`, `lib/portal_web/router.ex`,
`lib/portal_web/controllers/course_controller.ex`, `lib/portal_web/controllers/course_html.ex`,
`lib/portal_web/controllers/course_html/index.html.heex`,
`lib/portal_web/controllers/course_html/error.html.heex`.

## Agent stories

Each agent story implements one user story and closes on a gate. Build them in task-topology order.

### F6.1-AS1 — Boot as a Phoenix app [implements F6.1-US1]
Directive: add Phoenix to `mix.exs`, create the `PortalWeb` entrypoint, telemetry, and endpoint config (T1); build
`PortalWeb.Endpoint` with the full plug stack and the LiveView socket (T2); add `PortalWeb.Endpoint` to
`Portal.Application`'s children **after** `{Portal.Engine, []}` (T3); add `GET /health → 200 "ok"` (part of T4).
Acceptance gate: `mix compile` clean; app boots; `GET /health` returns `200`; tree order is store → engine → endpoint.

### F6.1-AS2 — Render a user's courses [implements F6.1-US2]
Directive: add the `:browser` pipeline and `get "/courses/:user_id", CourseController, :index` (T4); implement
`CourseController.index/2` over `Portal.courses_of/1` (T5); build `CourseHTML` + `index.html.heex` rendering from
assigns (T6).
Acceptance gate: a known user renders their courses; no enrollments renders an empty state; the template references
only assigns.

### F6.1-AS3 — Hold the boundary [implements F6.1-US3]
Directive: ensure the controller calls only `Portal`, and that no `lib/portal_web/` module names `Portal.Engine`,
`Repo`, or `GenServer.call`.
Acceptance gate: `grep -rE "Portal\.Engine|Repo|GenServer\.call" lib/portal_web/` returns nothing; the supervision
tree shows exactly one new child.

### F6.1-AS4 — Self-heal and fail soft [implements F6.1-US4, F6.1-US5]
Directive: confirm `:one_for_one` restarts the endpoint; map `{:error, %Portal.Error{}}` to a `422` render of an
`error` template.
Acceptance gate: killing the endpoint restarts it and a later request succeeds; a malformed user id yields a `422`
(not `500`) with a message from the closed set.

## Execution plan — first two stories

A worked trace of how the framework turns a user story into a verified change, to confirm the path is short and leaves
no decision to the agent that the spec has not already fixed. Each plan is *story → agent story → tasks → files →
command → gate*; an agent runs the steps top to bottom and stops at the gate.

### Plan A — F6.1-US1 (serve the Portal as a Phoenix app) · via F6.1-AS1
1. Tasks T1 → T2 → T3 (foundation → endpoint → supervision).
2. Files: `mix.exs` (+ phoenix), `config/config.exs`, `config/runtime.exs`, `lib/portal_web.ex`,
   `lib/portal_web/telemetry.ex`, `lib/portal_web/endpoint.ex`, `lib/portal/application.ex`,
   `lib/portal_web/router.ex` (add `get "/health"`).
3. Command: `mix deps.get && mix compile && mix phx.server`, then `curl -i localhost:4000/health`.
4. Gate (closes US1): `200 ok` from `/health`; boot order is store → engine → endpoint; strategy `:one_for_one`.
   Requirements satisfied: F6.1-R1, F6.1-R2, F6.1-R6.

### Plan B — F6.1-US2 (see a user's courses) · via F6.1-AS2
1. Tasks T4 → T5 → T6 (browser pipeline + route → controller → view).
2. Files: `lib/portal_web/router.ex` (`get "/courses/:user_id"` in a `:browser` scope),
   `lib/portal_web/controllers/course_controller.ex`, `lib/portal_web/controllers/course_html.ex`,
   `course_html/index.html.heex`, `course_html/error.html.heex`.
3. Command: `mix test test/portal_web/controllers/course_controller_test.exs`, plus a manual
   `curl localhost:4000/courses/USR_known`.
4. Gate (closes US2): a known user renders their courses; no enrollments renders an empty state; a malformed id
   renders `422` from the closed error set (F6.1-R7 / INV4); `grep -rE "Portal\.Engine|Repo|GenServer\.call"
   lib/portal_web/` is empty (F6.1-R7 / INV1). Requirements satisfied: F6.1-R3, F6.1-R4, F6.1-R5, F6.1-R8.

Efficiency check: each story is two-to-six files behind one command and one gate, and every branch (empty state,
malformed id) is already specified — so the agent implements rather than decides.

## Comprehensive implementation prompt

> Paste into a coding agent (e.g. Claude Code) at the root of the `portal` app. It executes F6.1-AS1..AS4 in
> task-topology order and ends on the verification gates. Treat the Requirements and the Definition of Done in
> [`f6.1.md`](f6.1.md) as the acceptance checklist.

```text
You are implementing spec F6.1 (Bootstrap the Phoenix Portal) in the existing `portal` Elixir app. Read
specs/phoenix/f6.1.md, specs/phoenix/f6.1.stories.md, and build-guide/f6-01-lifecycle.md first. Do not change anything
under the Portal facade: the engine (Portal.Engine), the store, the contexts, and the Portal facade stay exactly as F5
left them. Honor the master invariant — the web layer calls only Portal and renders only %Portal.Error{}.

Build in this order, keeping the app compiling after each step:

1) Foundation (T1). Add Phoenix to mix.exs and fetch deps. Create lib/portal_web.ex exposing `controller/0` and
   `html/0` helper macros (import Phoenix.Controller, Phoenix.Component, verified routes). Add
   lib/portal_web/telemetry.ex. Configure the endpoint in config/config.exs and config/runtime.exs (http port,
   secret_key_base from env at runtime, render_errors with PortalWeb.ErrorHTML, pubsub_server: Portal.PubSub declared
   for later — do not start PubSub yet unless the endpoint requires it; if it does, add {Phoenix.PubSub, name:
   Portal.PubSub} to the tree before the endpoint).

2) Endpoint (T2 → R1). Create lib/portal_web/endpoint.ex `use Phoenix.Endpoint, otp_app: :portal` with, in order:
   socket "/live", Phoenix.LiveView.Socket; Plug.Static (from :portal, at "/assets"); Plug.RequestId; Plug.Telemetry;
   Plug.Parsers (:urlencoded, :multipart, :json with Jason); Plug.Session; and `plug PortalWeb.Router` last.

3) Supervision (T3 → R2). In lib/portal/application.ex, append PortalWeb.Endpoint to the children list AFTER
   {Portal.Engine, []} (and after the store). Keep strategy: :one_for_one. Do not reorder or alter the store/engine
   entries.

4) Router (T4 → R3, R6). Create lib/portal_web/router.ex `use PortalWeb, :router`. Define pipeline :browser with
   plug :fetch_session, :fetch_live_flash, :protect_from_forgery, :put_secure_browser_headers. In scope "/", PortalWeb
   pipe_through :browser and add get "/courses/:user_id", CourseController, :index. Add get "/health" returning
   send_resp(conn, 200, "ok") with no domain call (a tiny inline controller action or a function plug is fine).

5) Controller (T5 → R4, R8). Create lib/portal_web/controllers/course_controller.ex
   `use PortalWeb, :controller`. index(conn, %{"user_id" => user_id}) calls Portal.courses_of(user_id) ONLY. On a list
   or {:ok, courses} render(conn, :index, courses: courses); on {:error, %Portal.Error{} = e} do
   conn |> put_status(422) |> render(:error, error: e). Never call Portal.Engine, a repo, or GenServer.

6) View (T6 → R5). Create lib/portal_web/controllers/course_html.ex `use PortalWeb, :html` embedding templates from
   course_html/. Add course_html/index.html.heex listing @courses (and an empty-state when none) and
   course_html/error.html.heex showing @error.message. Templates read only assigns.

7) Verify (T7 → R7, R8 + DoD). Confirm: mix compile is clean; the app boots; GET /health is 200; GET /courses/:user_id
   renders a known user's courses, an empty state for none, and a 422 page for a malformed id; killing
   PortalWeb.Endpoint restarts it and a later request succeeds. Run `grep -rE "Portal\.Engine|Repo|GenServer\.call"
   lib/portal_web/` and confirm it is empty. Report each result against the F6.1 Definition of Done.

Deliverables to produce: the touched files listed in specs/phoenix/f6.1.llms.md, all compiling, with the app runnable
and every F6.1 invariant exercised. Stop when the Definition of Done in specs/phoenix/f6.1.md is fully checked.
```
