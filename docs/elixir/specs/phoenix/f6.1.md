# F6.1 ¬∑ Bootstrap the Phoenix Portal

> Stand the headless F5 engine up as a real Phoenix web application: `PortalWeb.Endpoint` joins the existing
> supervision tree, a browser-pipelined route reaches a thin controller that calls only the `Portal` facade and
> renders a HEEx view, and a liveness route proves the service is up ‚Äî with nothing below the facade changed.

## Goal

After F6.1, the Portal runs as a Phoenix app. A browser request travels through the endpoint's plug stack, the
router's `:browser` pipeline, and a thin `PortalWeb.CourseController` that calls `Portal.courses_of/1`, branches on the
closed `%Portal.Error{}` set, and renders a HEEx view. The F5 store, event-store adapter, and engine run unchanged
inside the same supervision tree; the single structural change is the `PortalWeb.Endpoint` child **replacing** the F5
`Bandit` front door (the net child count stays four).

## Rationale (5W)

- **Why** ‚Äî F5 produced a correct engine but no way for a human to reach it over HTTP. The platform needs a web front
  door, and it must be added without disturbing the engine or the boundary that protects it.
- **What** ‚Äî a running Phoenix endpoint wired into the existing supervision tree, serving one facade-backed page and a
  liveness route, with the request lifecycle visible end to end.
- **Who** ‚Äî Operators (run and serve the platform), Visitors (load a page and see real data), Developers (keep the
  domain core untouched while the web layer is introduced).
- **When** ‚Äî the first rung of the F6 value ladder; depends only on the F5.08 / F5.09 handoff (the `Portal` facade and
  the supervised engine), nothing else.
- **Where** ‚Äî a new `PortalWeb` web layer (`endpoint.ex`, `router.ex`, a controller, a view) under
  `apps/portal/lib/portal_web/` (extending the location of the F5.9 `EnrollmentLive` sketch ‚Äî unless a separate
  `apps/portal_web` umbrella app is pinned at build), plus the one tree line that swaps `Bandit` for the endpoint in
  `Portal.Application`. Nothing under the `Portal` facade is touched.

## Scope

- **In** ‚Äî the endpoint and its plug stack; the one supervision-tree change; a `:browser` pipeline; one read route to
  a thin controller over the facade; a HEEx view; a liveness route; verification of boot, a rendered request, and
  endpoint restart.
- **Out** ‚Äî multiple pipelines, protected scopes, and custom plugs (‚Üí F6.2); Ecto persistence (‚Üí F6.3); domain and
  context wiring beyond `courses_of/1` (‚Üí F6.4); rich templates, components, and forms (‚Üí F6.5); LiveView (‚Üí F6.6).

## Deliverables

- **F6.1-D1** ‚Äî `PortalWeb.Endpoint`: the outermost plug, with `Plug.Static`, `Plug.RequestId`, `Plug.Telemetry`,
  `Plug.Parsers` (`:urlencoded`, `:multipart`, `:json` via `Jason`), and `Plug.Session`, ending by calling
  `PortalWeb.Router`; a `socket "/live", Phoenix.LiveView.Socket` declared for later modules.
- **F6.1-D2** ‚Äî `Portal.Application` supervises `[Portal.Store, Portal.EventStore.adapter(), {Portal.Engine, []},
  PortalWeb.Endpoint]` under `:one_for_one`; `PortalWeb.Endpoint` **replaces** the F5 `Bandit` child (the net count
  stays four), and the store, adapter, and engine entries are byte-for-byte unmoved and start before the endpoint.
  Omitting `Portal.Store` ‚Äî the dual-write `%Enrollment{}` read model `courses_of/1` reads ‚Äî would crash every course
  render, so it is retained.
- **F6.1-D3** ‚Äî `PortalWeb.Router` with a `:browser` pipeline (`fetch_session`, `fetch_live_flash`,
  `protect_from_forgery`, `put_secure_browser_headers`) and a route `get "/courses/:user_id", CourseController,
  :index` inside a `scope "/"` that pipes `:browser`.
- **F6.1-D4** ‚Äî `PortalWeb.CourseController.index/2`: reads `user_id` from params, calls **only**
  `Portal.courses_of/1` (as-built `{:ok, [%Enrollment{}]}`), renders `:index` on `{:ok, courses}`, and renders an
  error view with HTTP status `422` on `{:error, %Portal.Error{}}`.
- **F6.1-D5** ‚Äî `PortalWeb.CourseHTML` plus `course_html/index.html.heex`, rendering the courses from `assigns` and
  calling no engine code.
- **F6.1-D6** ‚Äî a liveness route `get "/health", ‚Ä¶` returning `200` (a plain `"ok"`) without touching the domain, for
  operator health checks.
- **F6.1-D7** ‚Äî verification: the app boots under its supervisor; `GET /courses/:user_id` renders; `GET /health`
  returns `200`; killing the endpoint process triggers a supervised restart and requests succeed again.

## Invariants

- **F6.1-INV1 (master)** ‚Äî `PortalWeb` calls only the `Portal` facade and renders only the closed `%Portal.Error{}`
  set; no module under `lib/portal_web/` names `Portal.Engine`, a repo, or `GenServer.call`.
- **F6.1-INV2** ‚Äî the one structural change to the supervision tree is `PortalWeb.Endpoint` **replacing** the F5
  `Bandit` child; the `Portal.Store`, event-store adapter, and engine entries are unchanged and start before it; the
  strategy stays `:one_for_one`.
- **F6.1-INV3** ‚Äî the endpoint is the outermost plug and `PortalWeb.Router` is its last plug; the session plug runs
  before any controller reads the session.
- **F6.1-INV4** ‚Äî an expected domain failure renders as an HTTP status drawn from the closed error set (e.g. `422`),
  never a `500` or an unhandled crash.
- **F6.1-INV5 (parse, don't validate)** ‚Äî `user_id` from the URL is the only untrusted input; the controller hands it
  straight to `Portal.courses_of/1` and branches on the typed result, so an unknown or malformed id becomes a
  `%Portal.Error{}` (rendered `422`) rather than a raw value flowing deeper. The facade is the parser; the web layer
  trusts nothing it has not parsed.

## Definition of Done

- [ ] `mix compile` is clean and the app boots under `Portal.Application`.
- [ ] `GET /courses/:user_id` for a known user renders that user's courses; an unknown user renders an empty state.
- [ ] A malformed user id renders a `422` page with a message from the closed error set (F6.1-INV4).
- [ ] `GET /health` returns `200`.
- [ ] Killing `PortalWeb.Endpoint` restarts it and a subsequent request succeeds (F6.1-INV2).
- [ ] No `lib/portal_web/` module references `Portal.Engine`, a repo, or `GenServer.call` (F6.1-INV1).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f6.1.stories.md`).
