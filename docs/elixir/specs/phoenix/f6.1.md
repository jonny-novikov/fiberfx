# F6.1 · Bootstrap the Phoenix Portal

> Stand the headless F5 engine up as a real Phoenix web application: `PortalWeb.Endpoint` joins the existing
> supervision tree, a browser-pipelined route reaches a thin controller that calls only the `Portal` facade and
> renders a HEEx view, and a liveness route proves the service is up — with nothing below the facade changed.

## Goal

After F6.1, the Portal runs as a Phoenix app. A browser request travels through the endpoint's plug stack, the
router's `:browser` pipeline, and a thin `PortalWeb.CourseController` that calls `Portal.courses_of/1`, branches on the
closed `%Portal.Error{}` set, and renders a HEEx view. The F5 store, event-store adapter, and engine run unchanged —
each name byte-for-byte preserved below the facade. The web layer lands as a **new umbrella app** `apps/portal_web`
(OTP app `:portal_web`, namespace `PortalWeb`) that depends on `{:portal, in_umbrella: true}`, so the supervision tree
**splits into two app supervisors**: `Portal.Application` keeps the three F5 domain children and **drops** the F5
`Bandit` front door (four → three), while the new `PortalWeb.Application` owns `[PortalWeb.Telemetry,
PortalWeb.Endpoint]`. The boundary that protected the core is now an **app-level, compiler-enforced dependency arrow**:
Phoenix lives only in `apps/portal_web/mix.exs`, so `apps/portal/mix.exs` stays Phoenix-free and the master invariant
cannot be broken by accident.

## Rationale (5W)

- **Why** — F5 produced a correct engine but no way for a human to reach it over HTTP. The platform needs a web front
  door, and it must be added without disturbing the engine or the boundary that protects it.
- **What** — a running Phoenix endpoint wired into the existing supervision tree, serving one facade-backed page and a
  liveness route, with the request lifecycle visible end to end.
- **Who** — Operators (run and serve the platform), Visitors (load a page and see real data), Developers (keep the
  domain core untouched while the web layer is introduced).
- **When** — the first rung of the F6 value ladder; depends only on the F5.08 / F5.09 handoff (the `Portal` facade and
  the supervised engine), nothing else.
- **Where** — a **new umbrella app `apps/portal_web`** (OTP app `:portal_web`, namespace `PortalWeb`, dep
  `{:portal, in_umbrella: true}`): `apps/portal_web/mix.exs`, `apps/portal_web/lib/portal_web.ex`,
  `apps/portal_web/lib/portal_web/application.ex`, `apps/portal_web/lib/portal_web/telemetry.ex`,
  `apps/portal_web/lib/portal_web/endpoint.ex`, `apps/portal_web/lib/portal_web/router.ex`, the controller + view under
  `apps/portal_web/lib/portal_web/controllers/`, and endpoint config keyed to `:portal_web`. The F5.9
  `PortalWeb.EnrollmentLive` sketch **moves** from `apps/portal/lib/portal_web/live/` to
  `apps/portal_web/lib/portal_web/live/`. The only **core** touches are two lines: `apps/portal/lib/portal/application.ex`
  (drop the `Bandit` child) and `apps/portal/mix.exs` (drop the `bandit` dep). Nothing under the `Portal` facade is
  touched.

## Scope

- **In** — the endpoint and its plug stack; the one supervision-tree change; a `:browser` pipeline; one read route to
  a thin controller over the facade; a HEEx view; a liveness route; verification of boot, a rendered request, and
  endpoint restart.
- **Out** — multiple pipelines, protected scopes, and custom plugs (→ F6.2); Ecto persistence (→ F6.3); domain and
  context wiring beyond `courses_of/1` (→ F6.4); rich templates, components, and forms (→ F6.5); LiveView (→ F6.6).

## Deliverables

- **F6.1-D1** — `PortalWeb.Endpoint`: the outermost plug, with `Plug.Static`, `Plug.RequestId`, `Plug.Telemetry`,
  `Plug.Parsers` (`:urlencoded`, `:multipart`, `:json` via `Jason`), and `Plug.Session`, ending by calling
  `PortalWeb.Router`; a `socket "/live", Phoenix.LiveView.Socket` declared for later modules.
- **F6.1-D2** — the supervision tree **splits across two app supervisors**, each `:one_for_one`. `Portal.Application`
  supervises `[Portal.Store, Portal.EventStore.adapter(), {Portal.Engine, []}]` — the three F5 domain children
  byte-for-byte unmoved, with the F5 `Bandit` child **dropped** (four → three). The new `PortalWeb.Application`
  supervises `[PortalWeb.Telemetry, PortalWeb.Endpoint]`. Because `apps/portal_web` depends on
  `{:portal, in_umbrella: true}`, the umbrella boots `:portal` (its tree) before `:portal_web` (its tree), so the store,
  adapter, and engine are ready before the endpoint accepts traffic. Omitting `Portal.Store` — the dual-write
  `%Enrollment{}` read model `courses_of/1` reads — would crash every course render, so it is retained.
- **F6.1-D3** — `PortalWeb.Router` with a `:browser` pipeline (`fetch_session`, `fetch_live_flash`,
  `protect_from_forgery`, `put_secure_browser_headers`) and a route `get "/courses/:user_id", CourseController,
  :index` inside a `scope "/"` that pipes `:browser`.
- **F6.1-D4** — `PortalWeb.CourseController.index/2`: reads `user_id` from params, calls **only**
  `Portal.courses_of/1` (as-built `{:ok, [%Enrollment{}]}`), renders `:index` on `{:ok, courses}`, and renders an
  error view with HTTP status `422` on `{:error, %Portal.Error{}}`.
- **F6.1-D5** — `PortalWeb.CourseHTML` plus `course_html/index.html.heex`, rendering the courses from `assigns` and
  calling no engine code.
- **F6.1-D6** — a liveness route `get "/health", …` returning `200` (a plain `"ok"`) without touching the domain, for
  operator health checks.
- **F6.1-D7** — verification: the app boots under its supervisor; `GET /courses/:user_id` renders; `GET /health`
  returns `200`; killing the endpoint process triggers a supervised restart and requests succeed again.

## Invariants

- **F6.1-INV1 (master)** — `PortalWeb` calls only the `Portal` facade and renders only the closed `%Portal.Error{}`
  set; no module under `apps/portal_web/lib/` names `Portal.Engine`, a repo, or `GenServer.call`.
- **F6.1-INV2** — the supervision tree splits into two app supervisors: `Portal.Application` keeps the `Portal.Store`,
  event-store adapter, and engine entries unchanged and **drops** the F5 `Bandit` child (four → three), and the new
  `PortalWeb.Application` owns `[PortalWeb.Telemetry, PortalWeb.Endpoint]`; both strategies stay `:one_for_one`. The
  `:portal_web` → `:portal` app dependency orders the boots, so the three domain children are ready before the endpoint;
  that dependency arrow is the boundary, now compiler-enforced at the app level (Phoenix lives only in
  `apps/portal_web/mix.exs`).
- **F6.1-INV3** — the endpoint is the outermost plug and `PortalWeb.Router` is its last plug; the session plug runs
  before any controller reads the session.
- **F6.1-INV4** — an expected domain failure renders as an HTTP status drawn from the closed error set (e.g. `422`),
  never a `500` or an unhandled crash.
- **F6.1-INV5 (parse, don't validate)** — `user_id` from the URL is the only untrusted input; the controller hands it
  straight to `Portal.courses_of/1` and branches on the typed result rather than inspecting the raw value. At F6.1 the
  facade is total, so an unknown or malformed id yields `{:ok, []}` (the empty state); the `{:error, %Portal.Error{}}`
  arm is handled defensively (rendered `422`) and becomes the parser's response to a malformed id when id-validation
  lands in the facade a later rung. The facade is the parser; the web layer validates nothing itself.

## Definition of Done

- [ ] `mix compile` is clean and the app boots under `Portal.Application`.
- [ ] `GET /courses/:user_id` for a known user renders that user's courses; an unknown user renders an empty state.
- [ ] An unknown or malformed user id renders the empty state (a `200`), since `courses_of/1` is total at F6.1; the
      controller's `422` error-render path is unit-verified via an injected `%Portal.Error{}` (F6.1-INV4), the live
      request-driven `422` deferred to the facade-validates rung.
- [ ] `GET /health` returns `200`.
- [ ] Killing `PortalWeb.Endpoint` restarts it and a subsequent request succeeds (F6.1-INV2).
- [ ] No `apps/portal_web/lib/` module references `Portal.Engine`, a repo, or `GenServer.call` (F6.1-INV1).
- [ ] Every deliverable maps to a user story and every invariant is exercised by a check (see `./f6.1.stories.md`).
