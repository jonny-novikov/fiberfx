# F6.4 ¬∑ Contexts & domain on the web

> Put the Portal's real domain behind one web-facing surface. A `Catalog` context over the F6.3 Ecto `Course` schema
> (every `Repo` call private), an `Enrollment` context that reads and appends through the F5.9 `Portal.EventStore` port
> (not the `Repo`), an `Accounts` context, and the `Portal` facade that `defdelegate`s to all three. Contexts compose
> only through public functions, fold every outcome into `%Portal.Error{}`, and form a one-way dependency graph.

## Goal

After F6.4, the web reads and writes real domain through one facade. `Portal.Catalog` exposes `list_courses/0`,
`get_course!/1`, `fetch_course/1`, and `create_course/1` over the F6.3 `Course` schema, with the schema alias and all
`Repo` calls private to the module. `Portal.Enrollment` reads and appends through the `Portal.EventStore` port from
F5.9, so it runs unchanged against the in-memory adapter in tests and Postgres in production ‚Äî the catalog stays
Repo-backed, enrollment stays event-sourced. The `Portal` facade `defdelegate`s each public function to its context, so
the web imports one module and the slices stay separate behind it. Cross-context calls go through public functions
only, folding outcomes into `%Portal.Error{}`; a `with` orchestration, `enroll_and_welcome/2`, chains public calls
across `Catalog`, `Enrollment`, and `Accounts` and short-circuits to one closed error. The dependency graph is one-way
(`Enrollment ‚Üí Catalog`), with no cycles and no foreign schema or `Repo` access.

## Rationale (5W)

- **Why** ‚Äî Contexts are where a Phoenix app keeps its domain coherent and its boundaries honest: a context owns its
  schema and its `Repo` calls, exposes a small public API, and never reaches into another context's internals, so
  slices evolve independently and the web depends on one stable surface. Without the discipline, controllers reach into
  schemas and `Repo`, transactions sprawl across slices, and the boundary the F5 engine earned dissolves.
- **What** ‚Äî the `Catalog` context (Ecto-backed), the `Enrollment` context (EventStore-backed), the `Accounts`
  context, the `Portal` facade over all three, cross-context composition through public functions, a `with`
  orchestration, and the one-way dependency discipline.
- **Who** ‚Äî Developers (one import and clear slices) and Learners (the real domain ‚Äî courses and enrollment ‚Äî
  reachable from the web).
- **When** ‚Äî the fourth Phoenix rung; it sits on F6.3's `Course`/`Repo` and the F5.9 `EventStore` port and `Portal`
  facade, and it gives F6.5's views and F6.6's LiveViews a real domain to render.
- **Where** ‚Äî `lib/portal.ex` (facade), `lib/portal/catalog.ex` + `lib/portal/catalog/course.ex` (private schema),
  `lib/portal/enrollment.ex` + `lib/portal/enrollment/enrolled.ex`, `lib/portal/accounts.ex`, and `lib/portal/error.ex`.

## Scope

- **In** ‚Äî the `Catalog` public API and its private `Repo`/schema; the `Enrollment` context over the `EventStore` port;
  the `Accounts` context; the `Portal` facade as `defdelegate`s; cross-context composition (a context calling another's
  public function and folding to `%Portal.Error{}`); the `with` orchestration across contexts; and the one-way
  dependency graph.
- **Out** ‚Äî HEEx templates and forms (‚Üí F6.5); LiveView (‚Üí F6.6); PubSub (‚Üí F6.7); authentication and registration
  internals beyond the `Accounts` surface this rung needs; and any new domain feature beyond enroll plus catalog
  read/write.

## Deliverables

- **F6.4-D1** ‚Äî `Portal.Catalog` public API: `list_courses/0 :: [%Course{}]`, `get_course!/1 :: %Course{}` (raises on
  miss, for controllers), `fetch_course/1 :: {:ok, %Course{}} | {:error, :not_found}` (for composing contexts),
  `create_course/1 :: {:ok, %Course{}} | {:error, %Ecto.Changeset{}}`; `alias Portal.Catalog.Course`, `import
  Ecto.Query`, and every `Repo.*` are private to the module.
- **F6.4-D2** ‚Äî `Portal.Enrollment` reads and appends through the `Portal.EventStore` port (F5.9), not the `Repo`:
  `enroll/2` and the enrollment read it needs are expressed over the port, so the in-memory adapter serves tests and
  Postgres serves production unchanged; it exposes a published `%Enrolled{}` struct (never a schema).
- **F6.4-D3** ‚Äî `Portal.Accounts` exposes the small surface the web and the orchestration need (the learner the
  enroll-and-welcome flow touches), with its own internals private.
- **F6.4-D4** ‚Äî `Portal`, the facade: a thin module that `defdelegate`s each public function to its context (`Catalog`,
  `Enrollment`, `Accounts`); it owns no logic and is the only module the web calls.
- **F6.4-D5** ‚Äî cross-context composition: `Enrollment.enroll/2` calls `Catalog.fetch_course/1`, branches on the public
  struct (never a schema), and folds every outcome into the closed `%Portal.Error{}` set.
- **F6.4-D6** ‚Äî a `with` orchestration `enroll_and_welcome/2` chaining public calls across `Catalog`, `Enrollment`, and
  `Accounts`, short-circuiting to one closed `%Portal.Error{}` on any failure.
- **F6.4-D7** ‚Äî the dependency discipline: a one-way graph (`Enrollment ‚Üí Catalog`), no cycles, and no context touching
  another's schema or `Repo`; consistency uses a `with` chain across contexts and `Ecto.Multi` only within one.
- **F6.4-D8** ‚Äî verification: the web calls only `Portal`; no controller names a context, schema, or `Repo`; the
  `Course` schema and `Repo` are private to `Catalog`; `Enrollment` runs against both adapters unchanged; the
  orchestration returns one closed error on any failure.

## Invariants

- **F6.4-INV1 (encapsulated context)** ‚Äî each context owns its schema and `Repo` calls privately; no module outside a
  context names that context's schema or `Repo`.
- **F6.4-INV2 (one facade ‚Äî the master invariant)** ‚Äî the web calls only `Portal`, which `defdelegate`s to contexts and
  owns no logic; nothing in the web layer names a context, schema, `Repo`, or the engine.
- **F6.4-INV3 (composition through public APIs)** ‚Äî a context reaches another only through its public function, passing
  an id or a published struct, never a private schema; the dependency graph is one-way with no cycles.
- **F6.4-INV4 (storage-agnostic enrollment)** ‚Äî `Enrollment` reads and appends through the `Portal.EventStore` port, so
  it runs unchanged against the in-memory adapter (tests) and Postgres (production); the catalog stays Repo-backed.
- **F6.4-INV5 (closed errors at the seam)** ‚Äî cross-context and orchestration failures fold into the closed
  `%Portal.Error{}` set; `from/1` has no catch-all, so a new reason must be mapped explicitly.
- **F6.4-INV6 (transaction boundary)** ‚Äî atomic multi-writes use `Ecto.Multi` within a single context; consistency
  across contexts uses a `with` chain, never one transaction spanning contexts.

## Definition of Done

- [ ] `Catalog` exposes `list_courses/0`, `get_course!/1`, `fetch_course/1`, `create_course/1`; its `Course` schema and
      `Repo` calls are private (F6.4-INV1).
- [ ] `Enrollment` reads and appends through the `EventStore` port and runs unchanged on both adapters (F6.4-INV4).
- [ ] `Portal` `defdelegate`s to the contexts and owns no logic; the web calls only `Portal` (F6.4-INV2).
- [ ] Contexts compose only through public functions, one-way, with no cycles (F6.4-INV3).
- [ ] `enroll_and_welcome/2` chains contexts with `with` and short-circuits to one closed error (F6.4-D6).
- [ ] Atomic writes use `Ecto.Multi` within one context; cross-context consistency uses `with` (F6.4-INV6).
- [ ] Every deliverable maps to a user story and every invariant is checked (see `./f6.4.stories.md`).
