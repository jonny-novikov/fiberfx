## References

- `Phoenix.LiveView` `handle_info/2` and streams — <https://hexdocs.pm/phoenix_live_view/Phoenix.LiveView.html>.
  Folding broadcasts and a capped feed stream.
- `Phoenix.Presence` (`list/1`, `track/3`, `presence_diff`) — <https://hexdocs.pm/phoenix/Phoenix.Presence.html>. The
  cluster-correct viewer count.
- The `Portal` PubSub facade and the `"events"` topic — [`f6.7.md`](f6.7.md). The broadcast source the dashboard
  folds.
- The `Catalog`/`Enrollment` contexts (and their `count` functions) — [`f6.4.md`](f6.4.md). The seed.
- `live_session` and `on_mount` auth — [`f6.8.md`](f6.8.md). The protected route.

## Requirements

- **F6.9-R1** — `PortalWeb.DashboardLive` seeds `courses_count` (`Catalog.count_courses/0`) and `enrollments_count`
  (`Enrollment.count/0`) as assigns at mount, plus a capped `stream(:events, …)`. [US: F6.9-US1, F6.9-US2]
- **F6.9-R2** — on the connected mount it `Portal.subscribe("events")` and `Presence.track/3` in `"dashboard"`. [US:
  F6.9-US6]
- **F6.9-R3** — `handle_info/2`: `{:course_created, _}` bumps `courses_count` + `stream_insert(:events, row, at: 0)`;
  `{:enrolled, _}` bumps `enrollments_count` + prepends a row; `"presence_diff"` recomputes `viewers`. [US: F6.9-US1,
  F6.9-US3]
- **F6.9-R4** — `render/1` shows metric cards from assigns and the feed under `phx-update="stream"`; the page is
  read-only (no form, no write control). [US: F6.9-US2, F6.9-US4]
- **F6.9-R5** — `viewers = map_size(Presence.list("dashboard"))`, recomputed on `presence_diff`, cluster-correct. [US:
  F6.9-US3]
- **F6.9-R6** — `live "/dashboard", DashboardLive` in a `live_session` under F6.8 auth (`on_mount` requires a user).
  [US: F6.9-US5]
- **F6.9-R7** — the dashboard seeds via the contexts and subscribes via the facade; it names no engine, repo, or
  `GenServer.call`, and folds only existing broadcasts (the `{:enrolled, _}` broadcast applies the F6.7 ok-only
  discipline to the `Enrollment` context). [US: F6.9-US6]

## Execution topology

```text
The dashboard fold (seed once, then fold broadcasts):

   mount (connected) ─┬─ Catalog.count_courses/0      ─▶ assign :courses_count
                      ├─ Enrollment.count/0           ─▶ assign :enrollments_count
                      ├─ Portal.subscribe("events")
                      ├─ Presence.track(self(), "dashboard", id, %{})
                      └─ stream(:events, [])          # capped activity feed
                                  │
   handle_info/2:                 ▼
     {:course_created, course}  ─▶ update :courses_count + stream_insert(:events, row, at: 0)
     {:enrolled, enrollment}    ─▶ update :enrollments_count + stream_insert(:events, row, at: 0)
     %{event: "presence_diff"}  ─▶ assign :viewers = map_size(Presence.list("dashboard"))
                                  │
   render/1:                     ▼
     metric cards from @courses_count / @enrollments_count / @viewers   +   feed under phx-update="stream"
     (read-only — no form, no write control)

   route: live "/dashboard", DashboardLive  inside  live_session under F6.8 on_mount auth
```

```text
Task DAG (build order; each step leaves the app compiling):

  T1 add Catalog.count_courses/0 + Enrollment.count/0 + Enrollment broadcasts {:enrolled, _} on success
        └─▶ T2 DashboardLive mount (seed counts + subscribe("events") + track("dashboard") + stream)   [R1, R2]
              └─▶ T3 handle_info fold (counts + feed + viewers)                                          [R3, R5]
                    └─▶ T4 render (metric cards + feed stream, read-only)                                [R4]
                          └─▶ T5 route in a live_session under F6.8 auth                                 [R6]
                                └─▶ T6 verify: seed, fold, viewers, read-only, authed, capped feed       [R7]
```

Touched files: `lib/portal_web/live/dashboard_live.ex` (mount/render/handle_info), `lib/portal_web/router.ex`
(`live_session` under F6.8 auth), `lib/portal/catalog.ex` (`count_courses/0`), `lib/portal/enrollment.ex` (`count/0`
and the `{:enrolled, _}` broadcast on success), and the existing `Portal` PubSub facade + `PortalWeb.Presence` from
F6.7.

## Agent stories

- **F6.9-AS1** [implements F6.9-US1, F6.9-US6] — Directive: add `Catalog.count_courses/0` and `Enrollment.count/0`,
  make `Enrollment` broadcast `{:enrolled, enrollment}` on a successful enroll (F6.7 ok-only discipline), and build
  `DashboardLive.mount/3` to seed the counts, `Portal.subscribe("events")`, `Presence.track`, and start the feed
  stream. Acceptance gate: counts seed at mount; the connected mount subscribes and tracks.
- **F6.9-AS2** [implements F6.9-US1, F6.9-US3] — Directive: write `handle_info/2` to bump the right count and prepend a
  feed row for `{:course_created, _}` and `{:enrolled, _}`, and recompute `viewers` on `presence_diff`. Acceptance
  gate: each broadcast bumps the right count and prepends a row; the viewer count tracks connect/disconnect.
- **F6.9-AS3** [implements F6.9-US2, F6.9-US4] — Directive: write `render/1` with metric cards from assigns and the
  feed under `phx-update="stream"`; keep the page read-only. Acceptance gate: cards reflect the counts; the feed is a
  capped stream; there is no form or write control.
- **F6.9-AS4** [implements F6.9-US5] — Directive: add `live "/dashboard", DashboardLive` inside a `live_session` under
  the F6.8 auth (`on_mount` requiring a user). Acceptance gate: an anonymous user is redirected; an authenticated user
  reaches the dashboard.

## Execution plan — first two stories

1. **F6.9-AS1 — seed and subscribe.** Add `count_courses/0` to `Portal.Catalog` and `count/0` to `Portal.Enrollment`,
   and make `Enrollment`'s enroll broadcast `{:enrolled, enrollment}` only on `{:ok, _}` (reusing the F6.7 helper).
   Create `lib/portal_web/live/dashboard_live.ex`; `mount/3` assigns `courses_count`/`enrollments_count` from those
   counts, starts `stream(socket, :events, [])`, and — behind `connected?/1` — calls `Portal.subscribe("events")` and
   `PortalWeb.Presence.track(self(), "dashboard", socket.id, %{})`. Confirm the counts seed and the subscription is
   live.
2. **F6.9-AS2 — fold the events.** Add `handle_info/2`: `{:course_created, course}` → `update(socket, :courses_count,
   &(&1 + 1))` then `stream_insert(socket, :events, to_row(course), at: 0)`; `{:enrolled, enrollment}` → bump
   `enrollments_count` and prepend a row; `%{event: "presence_diff"}` → `assign(socket, :viewers,
   map_size(PortalWeb.Presence.list("dashboard")))`. Broadcast a create and an enroll and confirm the counts and feed
   update live.

## Comprehensive implementation prompt

```text
You are implementing spec F6.9 (the live dashboard capstone) in the existing `portal` Phoenix app. Read
specs/phoenix/f6.9.md, the F6.7 PubSub/Presence (the Portal PubSub facade and the "events" broadcasts), the F6.4
Catalog/Enrollment contexts, and the F6.8 auth (live_session + on_mount). The dashboard is READ-ONLY: it issues no
command and no write. Honor the master invariant: it seeds via the contexts and subscribes via Portal, and names no
engine, repo, or GenServer.call.

Build in this order, keeping the app compiling:

1) Seeds + enrollment broadcast (T1 -> R1, R7). Add Catalog.count_courses/0 and Enrollment.count/0. Make
   Enrollment's enroll broadcast {:enrolled, enrollment} on success only (reuse the F6.7 ok-only broadcast helper) so
   the dashboard has an event to fold.

2) Mount (T2 -> R1, R2). Create lib/portal_web/live/dashboard_live.ex. mount/3 assigns courses_count and
   enrollments_count from the context counts, starts stream(socket, :events, []) (capped), and — behind
   connected?(socket) — calls Portal.subscribe("events") and PortalWeb.Presence.track(self(), "dashboard", id, %{}).

3) Fold (T3 -> R3, R5). handle_info/2:
     {:course_created, course} -> update :courses_count (+1) and stream_insert(:events, row, at: 0)
     {:enrolled, enrollment}   -> update :enrollments_count (+1) and stream_insert(:events, row, at: 0)
     %{event: "presence_diff"} -> assign :viewers = map_size(PortalWeb.Presence.list("dashboard"))
   Seed once at mount; thereafter update only by folding (no re-query per event).

4) Render (T4 -> R4). render/1 shows metric cards from @courses_count, @enrollments_count, @viewers, and the feed under
   phx-update="stream". The page is read-only — no form, no write control.

5) Route (T5 -> R6). Add live "/dashboard", DashboardLive inside a live_session under the F6.8 auth
   (on_mount {PortalWeb.UserAuth, :ensure_authenticated}). An anonymous user is redirected.

6) Verify (T6 -> R7 + DoD). Confirm: the counts seed at mount; a {:course_created, _} and an {:enrolled, _} each bump
   the right count and prepend a feed row live; the viewer count tracks connect/disconnect across nodes; the page is
   read-only and requires auth; the feed is a capped stream. Run `grep -rE "Portal\.Engine|Repo|GenServer\.call"
   lib/portal_web/live/dashboard_live.ex` and confirm it is empty. Report against the F6.9 Definition of Done.

Constraints: read-only (no command, no write); seed via the contexts, subscribe via the Portal facade; the feed is a
capped stream; the route is under F6.8 auth and correct across the cluster. Out of scope: new domain features, new
event types, historical/time-series analytics, export, and role-based views.
```
