## Requirements

- **F6.2-R1** — `PortalWeb.Router` declares all four route kinds plus a public landing: `get "/", PageController,
  :home`; `get "/courses/:id", CourseController, :show`; `post "/enroll", EnrollmentController, :create`; `resources
  "/lessons", LessonController, only: [:show]`; `live "/enroll/:id", EnrollmentLive`. [US: F6.2-US1, F6.2-US6]
- **F6.2-R2** — a `:browser` pipeline (`fetch_session`, `fetch_live_flash`, `protect_from_forgery`,
  `put_secure_browser_headers`) and an `:api` pipeline (`accepts ["json"]`) exist; the `:api` pipeline carries no
  browser-only plugs. [US: F6.2-US1, F6.2-US5]
- **F6.2-R3** — a `:require_auth` pipeline runs `PortalWeb.RequireUser`; a protected `scope` does `pipe_through
  [:browser, :require_auth]`, in that order. [US: F6.2-US2, F6.2-US3, F6.2-INV5]
- **F6.2-R4** — `PortalWeb.RequireUser` implements `init/1` (returns opts at compile time) and `call/2` (reads
  `:user_id` from the session; on present, `assign(conn, :current_user_id, id)`; on absent, `conn |> put_flash(:error,
  …) |> redirect(to: ~p"/") |> halt()`). [US: F6.2-US2]
- **F6.2-R5** — a function plug `(conn, _opts)` is defined and wired into a pipeline, demonstrating the plug contract
  without a module. [US: F6.2-US3]
- **F6.2-R6** — every internal URL built in the app (links, the RequireUser redirect) uses a verified `~p` path; the
  project compiles, which proves the paths resolve. [US: F6.2-US4]
- **F6.2-R7** — public routes render with no session; a protected route redirects to `~p"/"` when unauthenticated and
  renders with `:current_user_id` assigned when a `:user_id` is in the session. [US: F6.2-US1, F6.2-US2]
- **F6.2-R8** — no plug or handler references `Portal.Engine`, a repo, or `GenServer.call`; `EnrollmentController` and
  `EnrollmentLive` reach the domain (when implemented) only through `Portal`. [US: F6.2-US3, master invariant]

## Execution topology

**Runtime topology (pipelines, scopes, and the two request paths).**

```text
PortalWeb.Endpoint (F6.1) → PortalWeb.Router
│
├─ pipeline :browser   = fetch_session, fetch_live_flash, protect_from_forgery, put_secure_browser_headers
├─ pipeline :api       = accepts ["json"]
└─ pipeline :require_auth = PortalWeb.RequireUser

scope "/", PortalWeb  (pipe_through :browser)            # PUBLIC
   get  "/"               PageController.home
   get  "/courses/:id"    CourseController.show           ──▶ Portal (facade)
   resources "/lessons"   LessonController, only: [:show] ──▶ Portal
   post "/enroll"         EnrollmentController.create      ──▶ Portal.enroll/2
   live "/enroll/:id"     EnrollmentLive                   ──▶ Portal (on the LiveView socket)

scope "/learn", PortalWeb  (pipe_through [:browser, :require_auth])   # PROTECTED
   <protected routes>
        │
        └─ RequireUser.call/2 : session :user_id?
                ├─ yes → assign :current_user_id → continue to action
                └─ no  → put_flash → redirect(~p"/") → halt   (action never runs)
```

Public requests pass `:browser` and reach their handler. Protected requests pass `:browser` then `:require_auth`;
`RequireUser` either assigns the user and continues or halts with a redirect before the action runs.

**Task topology (build order; each step leaves the app compiling).**

```text
T1 :browser + :api pipelines in the router                      [R2]
     └─▶ T2 public scope + public routes (/, /courses/:id, /lessons show)  [R1]
           └─▶ T3 PortalWeb.RequireUser plug (init/1 + call/2, redirect ~p"/")  [R4, R6]
                 └─▶ T4 :require_auth pipeline + protected scope pipe_through [:browser,:require_auth]  [R3]
                       └─▶ T5 write + live routes (post /enroll, live /enroll/:id) wired to thin targets  [R1, R8]
                             └─▶ T6 a function plug wired into a pipeline                                  [R5]
                                   └─▶ T7 verify: public vs protected access, :api JSON, a bad ~p fails    [R6, R7, R8]
```

**Touched files.** `lib/portal_web/router.ex`, `lib/portal_web/plugs/require_user.ex`,
`lib/portal_web/controllers/page_controller.ex`, `lib/portal_web/controllers/page_html.ex`,
`lib/portal_web/controllers/page_html/home.html.heex`,
`lib/portal_web/controllers/enrollment_controller.ex` (thin `create` over `Portal.enroll/2`),
`lib/portal_web/controllers/lesson_controller.ex` (thin `show`), `lib/portal_web/live/enrollment_live.ex` (thin mount),
and the F6.1 `course_controller.ex` (add a `show` action).

## Agent stories

Each agent story implements one user story and closes on a gate. Build in task-topology order.

### F6.2-AS1 — Public surface [implements F6.2-US1]
Directive: define `:browser` and `:api` pipelines (T1); add the public scope with `get "/"` (PageController.home),
`get "/courses/:id"` (CourseController.show), and `resources "/lessons", only: [:show]` (T2).
Acceptance gate: `mix compile` clean; `/`, `/courses/:id`, `/lessons/:id` render with no session.

### F6.2-AS2 — Authentication boundary [implements F6.2-US2]
Directive: build `PortalWeb.RequireUser` (`init/1` + `call/2`, redirect to `~p"/"` and `halt` when no session
`:user_id`) (T3); add the `:require_auth` pipeline and a protected scope `pipe_through [:browser, :require_auth]`
(T4).
Acceptance gate: a protected route redirects to `/` with a flash when unauthenticated, and renders with
`:current_user_id` assigned when a `:user_id` is in the session.

### F6.2-AS3 — Cross-cutting once, paths verified [implements F6.2-US3, F6.2-US4]
Directive: keep all session/CSRF/header/auth logic in pipelines (no action carries it); add a function plug into a
pipeline (T6); use verified `~p` paths for every internal URL (T2–T5).
Acceptance gate: no action contains cross-cutting code; the protected scope pipes `:browser` before `:require_auth`; a
deliberately wrong `~p` fails compilation and reverting it restores a clean build.

### F6.2-AS4 — JSON + write/live reachability [implements F6.2-US5, F6.2-US6]
Directive: ensure the `:api` pipeline negotiates JSON; wire `post "/enroll"` to a thin
`EnrollmentController.create` that calls `Portal.enroll/2`, and `live "/enroll/:id"` to a thin `EnrollmentLive` mount
(T5).
Acceptance gate: a JSON request to an `:api` route negotiates JSON; `POST /enroll` reaches `create` and issues the
facade command; `/enroll/:id` mounts the LiveView; nothing references `Portal.Engine`, a repo, or `GenServer.call`.

## Comprehensive implementation prompt

> Paste into a coding agent (e.g. Claude Code) at the root of the `portal` app, after F6.1 is in place. It executes
> F6.2-AS1..AS4 in task-topology order and ends on the verification gates. Treat the Requirements and the Definition of
> Done in [`f6.2.md`](f6.2.md) as the acceptance checklist.

```text
You are implementing spec F6.2 (Routing & the access surface) in the `portal` Phoenix app, on top of F6.1. Read
specs/phoenix/f6.2.md, specs/phoenix/f6.2.stories.md, and build-guide/f6-02-routing.md first. Do not change anything
below the Portal facade. Honor the master invariant — every route's handler and every plug calls only Portal and
renders only %Portal.Error{}; no Portal.Engine, repo, or GenServer.call anywhere in lib/portal_web/. Use verified ~p
paths for every internal URL.

Build in this order, keeping the app compiling after each step:

1) Pipelines (T1 → R2). In lib/portal_web/router.ex add pipeline :browser (plug :fetch_session, :fetch_live_flash,
   :protect_from_forgery, :put_secure_browser_headers) and pipeline :api (plug :accepts, ["json"]). The :api pipeline
   must not include browser-only plugs.

2) Public scope (T2 → R1). Add scope "/", PortalWeb with pipe_through :browser and routes: get "/", PageController,
   :home; get "/courses/:id", CourseController, :show; resources "/lessons", LessonController, only: [:show]. Create a
   thin PortalWeb.PageController with home/2 rendering a static PageHTML home.html.heex (no domain call). Add a show/2
   action to the F6.1 CourseController that calls only Portal (e.g. a public course read) and renders from assigns;
   add a thin LessonController.show/2 over Portal.

3) RequireUser plug (T3 → R4, R6). Create lib/portal_web/plugs/require_user.ex implementing the Plug behaviour:
   init(opts) returns opts; call(conn, _opts) reads get_session(conn, :user_id); if present, assign(conn,
   :current_user_id, id); if absent, conn |> put_flash(:error, "Please sign in.") |> redirect(to: ~p"/") |> halt().
   (F6.8 will later swap ~p"/" for the real login path from phx.gen.auth.)

4) Protected scope (T4 → R3). Add pipeline :require_auth (plug PortalWeb.RequireUser) and a scope "/learn", PortalWeb
   with pipe_through [:browser, :require_auth] in that order, containing at least one protected route to demonstrate
   the boundary.

5) Write + live routes (T5 → R1, R8). Add post "/enroll", EnrollmentController, :create and live "/enroll/:id",
   EnrollmentLive. Create a thin EnrollmentController.create/2 that calls Portal.enroll/2 and branches on
   %Portal.Error{} (full UI comes in F6.4/F6.6); create a thin EnrollmentLive with mount/3 that reaches the domain
   only via Portal.

6) Function plug (T6 → R5). Add a one-line function plug (conn, _opts) — e.g. assign a request marker — and
   pipe_through it in a pipeline, to show the plug contract without a module.

7) Verify (T7 → R6, R7, R8 + DoD). Confirm: mix compile is clean (proving all ~p paths resolve); /, /courses/:id,
   /lessons/:id render with no session; a protected route redirects to / with a flash when unauthenticated and renders
   with :current_user_id when a :user_id is in the session; an :api route negotiates JSON; a deliberately wrong ~p
   path fails compilation and reverting restores a clean build; `grep -rE "Portal\.Engine|Repo|GenServer\.call"
   lib/portal_web/` is empty. Report each result against the F6.2 Definition of Done.

Deliverables to produce: the touched files listed in specs/phoenix/f6.2.llms.md, all compiling, with public/protected
access working and every F6.2 invariant exercised. Stop when the Definition of Done in specs/phoenix/f6.2.md is fully
checked.
```
