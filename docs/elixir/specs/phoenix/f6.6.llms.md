## Requirements

- **F6.6-R1** — `CatalogLive` (`use PortalWeb, :live_view`): `mount/3` streams the catalog (`@streams.courses`, not a
  `@courses` assign — INV4) plus the `:query`/`:form` assigns; `render/1` reuses `course_card`; `handle_event/3` returns
  `{:noreply, socket}`. [US: F6.6-US5]
- **F6.6-R2** — the mount is two-stage; `connected?/1` guards side effects. [US: F6.6-US3]
- **F6.6-R3** — `phx-change="search"` filters through `Portal.search_courses/1` (the facade, not a direct `Catalog`
  call — `[RECONCILE]`, facade-only) and re-streams `:courses` with `reset: true` live (the list is a stream, never an
  assign — INV4). [US: F6.6-US1]
- **F6.6-R4** — `phx-submit="create"` calls `Portal.create_course/1`; flash + reset on `{:ok, _}`,
  `to_form(changeset)` inline errors on `{:error, _}`. [US: F6.6-US2]
- **F6.6-R5** — the list uses `stream/3` + `stream_insert/3` + `phx-update="stream"`; no list assign. [US: F6.6-US4]
- **F6.6-R6** — `live "/courses", CatalogLive` in the `:browser` scope, superseding F6.5's static `GET /courses` index;
  `resources "/courses", CourseController` reconciles to `only: [:show, :new, :create]`. [US: F6.6-US0]
- **F6.6-R7** — the view names only the facade and reuses the F6.5 `course_card`/`input` components. [US: F6.6-US5]
- **F6.6-R8** (the ONE ratified addition) — add `Portal.search_courses/1` → `Catalog.search_courses/1 :: [Course.t()]`,
  a read-only delegate named to match the `list_courses/0` siblings (`apps/portal/lib/portal.ex:41`,
  `catalog.ex:46`). `[RECONCILE]`-ratified in principle; **flagged for Director ratification**. Nothing else below the
  boundary changes. [US: F6.6-US1]

## Execution topology

```text
The socket lifecycle:

   first request ─▶ mount/3 (disconnected, HTTP)  ─▶ render/1  ─▶ first paint (indexable)
                          │
   socket connects ─▶ mount/3 (connected)  ── connected?/1 guards side effects ──┐
                          │                                                       │
                          ▼                                                       ▼
   user types ─▶ phx-change="search" ─▶ handle_event("search") ─▶ Portal.search_courses/1 ─▶ stream(reset: true) ─▶ diff/patch
   user submits ─▶ phx-submit="create" ─▶ handle_event("create") ─▶ Portal.create_course/1
                       ├─ {:ok, _}                 ─▶ put_flash + reset form + stream_insert(:courses, course, at: 0)
                       └─ {:error, %Ecto.Changeset{}} ─▶ assign to_form(cs)  # inline errors (F6.3)

   list:  stream(:courses, ...) + phx-update="stream"   # rows in the DOM, ids on the server, no list in memory
```

```text
Task DAG:
  T0 add Portal.search_courses/1 + Catalog.search_courses/1 (the one ratified delegate)
       └─▶ T1 route reconcile(live "/courses"; resources → show/new/create) ─▶ T2 mount(assigns, two-stage)
            ─▶ T3 render(course_card + input + stream) ─▶ T4 search event(Portal.search_courses/1)
            ─▶ T5 create event(ok/error) ─▶ T6 streams ─▶ T7 verify
```

## Execution plan — first stories

0. **F6.6-AS0 — the ratified delegate.** In `apps/portal/lib/portal/catalog.ex` add `search_courses/1 :: [Course.t()]`
   (a case-insensitive title filter over the Repo is sufficient); in `apps/portal/lib/portal.ex` add `defdelegate
   search_courses(query), to: Portal.Catalog`. This is the ONE addition below the web boundary — Director ratifies it.
1. **F6.6-AS1 — the LiveView and route reconcile.** Create `apps/portal_web/lib/portal_web/live/catalog_live.ex`,
   `use PortalWeb, :live_view`. `mount/3` assigns `courses` (stream) from `Portal.list_courses/0` and a `form` from
   `to_form(Portal.change_course())` — note **arity 0**; the context builds the base `%Course{}` internally because
   `@enforce_keys [:id, :title, :slug]` makes a `%Course{}` literal fail to compile. Guard any side effect with
   `if connected?(socket)`. `render/1` reuses `<.course_card>` and the local `<.input>`. In the router, add `live
   "/courses", CatalogLive` to the `:browser` scope and change `resources "/courses"` to `only: [:show, :new,
   :create]`. Confirm a disconnected then connected render.
2. **F6.6-AS2 — live search.** Add an input with `phx-change="search"` and `handle_event("search", %{"q" => q},
   socket)` that filters via `Portal.search_courses(q)` and re-streams/re-assigns `@courses`. Write a `LiveViewTest`
   using `render_change` to confirm the list narrows as the query changes.

## Comprehensive implementation prompt

```text
You are building the interactive catalog (rung F6.6) on top of F6.5. Read specs/phoenix/f6.6.md, the F6.5 course_card
+ input components (apps/portal_web/.../catalog_components.ex), and the as-built facade (apps/portal/lib/portal.ex:
list_courses/0, change_course/0 [arity 0], create_course/1). Below the boundary, ONE read-only addition is ratified
(search_courses/1); everything else (the create path, the F6.3 changeset) is UNCHANGED. The LiveView names ONLY
Portal. There is NO CoreComponents — use the local <.input> from PortalWeb.CatalogComponents (F6.5-INV8).

Build CatalogLive:

0) Search delegate (T0 -> R8, the ONE ratified addition). In apps/portal/lib/portal/catalog.ex add search_courses/1 ::
   [Course.t()] (a case-insensitive title filter over the Repo is enough); in apps/portal/lib/portal.ex add
   `defdelegate search_courses(query), to: Portal.Catalog`, beside list_courses/0. Nothing else below the boundary
   changes.

1) Route reconcile (T1 -> R6). In the router, add `live "/courses", CatalogLive` to the :browser scope and change the
   F6.5 `resources "/courses", CourseController` from `only: [:index, :show, :new, :create]` to `only: [:show, :new,
   :create]` — CatalogLive SUPERSEDES the static :index. One catalog resource, one URL; do NOT add a /catalog route.

2) Mount + render (T2,T3 -> R1,R2,R7). Create lib/portal_web/live/catalog_live.ex, `use PortalWeb, :live_view`.
   mount/3 runs in two stages: a disconnected HTTP render for first paint, then a connected mount; guard side effects
   with `if connected?(socket)`. Assign the list as a stream (`stream(socket, :courses, Portal.list_courses())`) and
   a form (`to_form(Portal.change_course())` — arity 0; the context builds the base %Course{} internally, since
   @enforce_keys [:id, :title, :slug] makes a %Course{} literal fail to compile). render/1 returns ~H reusing
   <.course_card> for each row and `phx-update="stream"` on the container. The view names only Portal — never the
   engine, a Repo, or GenServer.call.

3) Live search (T4 -> R3). Add an input with phx-change="search"; handle_event("search", params, socket) filters
   courses through Portal.search_courses/1 (NOT a direct Catalog call) and re-streams/re-assigns @courses. The list
   updates as the user types.

4) Live create (T5 -> R4). Add a form with phx-submit="create"; handle_event("create", params, socket) calls
   Portal.create_course/1. On {:ok, course}: put_flash, reset the form, and stream_insert(socket, :courses, course,
   at: 0). On {:error, %Ecto.Changeset{} = cs}: assign to_form(cs) so the F6.3 errors render inline. The view adds no
   validation of its own.

5) Streams (T6 -> R5). Use stream/3 + stream_insert/3 + phx-update="stream" so a large, append-only list is rows in
   the DOM with ids on the server and no list in socket memory.

6) Verify (T7). With Phoenix.LiveViewTest: typing filters the list (render_change); a valid create flashes and
   appears (render_submit); an invalid create shows inline errors; first paint renders over HTTP before the socket
   connects; the list is a stream (no list assign).

Constraints: state lives in socket assigns; reuse the F6.5 course_card (no duplicated row markup); verified ~p routes;
HEEx-escaped interpolation. Out of scope: PubSub/Presence and multi-client updates (F6.7), auth (F6.8), the dashboard
(F6.9).
```
