## Requirements

- **F6.6-R1** — `CatalogLive` (`use PortalWeb, :live_view`): `mount/3` assigns `@courses` and the search/form assigns;
  `render/1` reuses `course_card`; `handle_event/3` returns `{:noreply, socket}`. [US: F6.6-US5]
- **F6.6-R2** — the mount is two-stage; `connected?/1` guards side effects. [US: F6.6-US3]
- **F6.6-R3** — `phx-change="search"` filters through `Catalog` and re-assigns `@courses` live. [US: F6.6-US1]
- **F6.6-R4** — `phx-submit="create"` calls `Portal.create_course/1`; flash + reset on `{:ok, _}`,
  `to_form(changeset)` inline errors on `{:error, _}`. [US: F6.6-US2]
- **F6.6-R5** — the list uses `stream/3` + `stream_insert/3` + `phx-update="stream"`; no list assign. [US: F6.6-US4]
- **F6.6-R6** — `live "/catalog", CatalogLive` in a `:browser` scope. [US: F6.6-US6]
- **F6.6-R7** — the view calls only the facade/context and reuses the F6.5 component. [US: F6.6-US5]

## Execution topology

```text
The socket lifecycle:

   first request ─▶ mount/3 (disconnected, HTTP)  ─▶ render/1  ─▶ first paint (indexable)
                          │
   socket connects ─▶ mount/3 (connected)  ── connected?/1 guards side effects ──┐
                          │                                                       │
                          ▼                                                       ▼
   user types ─▶ phx-change="search" ─▶ handle_event("search") ─▶ Catalog filter ─▶ re-assign @courses ─▶ diff/patch
   user submits ─▶ phx-submit="create" ─▶ handle_event("create") ─▶ Portal.create_course/1
                       ├─ {:ok, _}                 ─▶ put_flash + reset form + stream_insert(:courses, course, at: 0)
                       └─ {:error, %Ecto.Changeset{}} ─▶ assign to_form(cs)  # inline errors (F6.3)

   list:  stream(:courses, ...) + phx-update="stream"   # rows in the DOM, ids on the server, no list in memory
```

```text
Task DAG:
  T1 route(live) ─▶ T2 mount(assigns, two-stage) ─▶ T3 render(course_card + stream) ─▶ T4 search event ─▶ T5 create event(ok/error) ─▶ T6 streams ─▶ T7 verify
```

Touched files: `lib/portal_web/live/catalog_live.ex`, the router (`live "/catalog", CatalogLive`), and the unchanged
`Portal` facade / `Portal.Catalog` context / `change_course/1` from F6.4–F6.5.

## Agent stories

- **F6.6-AS1** [implements F6.6-US5, F6.6-US6, F6.6-US3] — Directive: write `CatalogLive` (`use PortalWeb, :live_view`)
  with a two-stage `mount/3` assigning `@courses` (via the facade) and a form, guarding side effects with
  `connected?/1`; add `live "/catalog", CatalogLive` to a `:browser` scope. Acceptance gate: the page renders
  disconnected then connected; the view names only the facade/context.
- **F6.6-AS2** [implements F6.6-US1] — Directive: add a search box (`phx-change="search"`) and `handle_event("search",
  ...)` that filters through `Catalog` and re-assigns `@courses`. Acceptance gate: `render_change` filters the list
  live.
- **F6.6-AS3** [implements F6.6-US2] — Directive: add a create form (`phx-submit="create"`) and `handle_event("create",
  ...)` calling `Portal.create_course/1`, flashing and resetting on `{:ok, _}` and assigning `to_form(cs)` on
  `{:error, _}`. Acceptance gate: `render_submit` flashes on valid input and shows inline errors on invalid.
- **F6.6-AS4** [implements F6.6-US4] — Directive: render the list with `stream/3` + `phx-update="stream"` and insert new
  courses with `stream_insert/3`. Acceptance gate: no list assign; new rows appear via the stream.

## Execution plan — first two stories

1. **F6.6-AS1 — the LiveView and route.** Create `lib/portal_web/live/catalog_live.ex`, `use PortalWeb, :live_view`.
   `mount/3` assigns `courses` (stream) from `Portal.list_courses/0` and a `form` from
   `to_form(Portal.change_course(%Course{}))`; guard any side effect with `if connected?(socket)`. `render/1` reuses
   `<.course_card>`. Add `live "/catalog", CatalogLive` to the `:browser` scope. Confirm a disconnected then connected
   render.
2. **F6.6-AS2 — live search.** Add an input with `phx-change="search"` and `handle_event("search", %{"q" => q},
   socket)` that filters via the `Catalog` context and re-streams/re-assigns `@courses`. Write a `LiveViewTest` using
   `render_change` to confirm the list narrows as the query changes.

## Comprehensive implementation prompt

```text
You are building the interactive catalog (rung F6.6) on top of F6.5. Read specs/phoenix/f6.6.md, the F6.5 course_card
component and change_course/1, and the F6.4 Catalog facade (list_courses/0, create_course/1). The facade, context,
and changeset are UNCHANGED here; the LiveView calls only the facade/context.

Build CatalogLive:

1) Route (T1 -> R6). Add `live "/catalog", CatalogLive` to a :browser scope in the router.

2) Mount + render (T2,T3 -> R1,R2,R7). Create lib/portal_web/live/catalog_live.ex, `use PortalWeb, :live_view`.
   mount/3 runs in two stages: a disconnected HTTP render for first paint, then a connected mount; guard side effects
   with `if connected?(socket)`. Assign the list as a stream (`stream(socket, :courses, Portal.list_courses())`) and
   a form (`to_form(Portal.change_course(%Course{}))`). render/1 returns ~H reusing <.course_card> for each row and
   `phx-update="stream"` on the container. The view names only Portal/the Catalog context — never the engine, a Repo,
   or GenServer.call.

3) Live search (T4 -> R3). Add an input with phx-change="search"; handle_event("search", params, socket) filters
   courses through the Catalog context and re-streams/re-assigns @courses. The list updates as the user types.

4) Live create (T5 -> R4). Add a form with phx-submit="create"; handle_event("create", params, socket) calls
   Portal.create_course/1. On {:ok, course}: put_flash, reset the form, and stream_insert(socket, :courses, course,
   at: 0). On {:error, %Ecto.Changeset{} = cs}: assign to_form(cs) so the F6.3 errors render inline. The view adds no
   validation of its own.

5) Streams (T6 -> R5). Use stream/3 + stream_insert/3 + phx-update="stream" so a large, append-only list is rows in
   the DOM with ids on the server and no list in socket memory.

6) Verify (T7). With Phoenix.LiveViewTest: typing filters the list (render_change); a valid create flashes and
   appears (render_submit); an invalid create shows inline errors; first paint renders over HTTP before the socket
   connects; the list is a stream (no list assign).

Constraints: state lives in socket assigns; reuse the F6.5 course_card (no duplicated row markup); verified ~p routes;
HEEx-escaped interpolation. Out of scope: PubSub/Presence and multi-client updates (F6.7), auth (F6.8), the dashboard
(F6.9).
```
