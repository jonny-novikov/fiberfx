# F6.5.5 · user stories

> Who the styled Portal serves, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.5.5.md`](f6.5.5.md). Acceptance is the F0 TEXT PROXIES,
> not screenshots — containers / voice / links / clamp-spacing / degrade / token-fidelity (the `app.css` `:root` diff vs
> `f0.md` D1) / computed-style parity (`apps/e2e/tests/elixir-hero.spec.ts` under `E2E_BASE_URL`, `f0.roadmap.md` M5). The
> clamp-spacing proxy is the highest-leverage (the unspaced bug shipped invisibly across 204 static pages). Stories honor the
> [`f6.5.5.md`](f6.5.5.md) locked facts: the styling reaches the Portal over the facade, with a committed `app.css` served by
> the EXISTING `/assets` mount (no bundler, no endpoint change) and NO `CoreComponents`.

## F6.5.5-US0 — A styled front door

As a **Visitor**, I want the Portal to render a real page envelope, so that the catalog looks like the jonnify courses it
hosts instead of an unstyled fragment.

Acceptance criteria
- Given a request to `~p"/courses"` (or any browser route), when it renders, then the response is the F0 document envelope —
  `<!doctype html>` → `<html lang="en">` → a `<head>` carrying the four Google-Fonts `<link>`s and `<link rel="stylesheet"
  href={~p"/assets/app.css"}>` → `<body>` — NOT a bare `<h1>` fragment (per [`f6.5.5.md`](f6.5.5.md) D2, INV8;
  `f0.md`-D3).
- Given the envelope, when inspected, then it carries the two mandatory agile-shell rules: the clickable segmented
  `.route-tag` breadcrumb in a sticky header and the 3-column `.site-foot` footer (the look the
  `/course/agile-agent-workflow` pages use).
- Given `:4000`, when curled, then a 200 response includes `<head` and `app.css` in its body; a 200 with bare body markup
  (no head, no stylesheet link) is a REGRESSION (INV8).

INVEST — independent; testable by the liveness curl + an envelope grep; encodes F6.5.5-INV8 and the D2 envelope.
Priority: must · Size: 2 · Implements deliverables: F6.5.5-D1, D2.

## F6.5.5-US1 — The catalog in the design system

As a **Learner**, I want the catalog, cards, badge, and form to render in the dark-editorial design system, so that the
Portal reads as one coherent jonnify surface.

Acceptance criteria
- Given the catalog index, when it renders, then each row is the F0 `.mod` card (the restyled `course_card/1`), keeping its
  `~p"/courses/#{course.id}"` link and the `:if={course.published}` badge restyled to `.pill.live` (per
  [`f6.5.5.md`](f6.5.5.md) D5).
- Given the create form, when it renders, then its frame is the F0 `.fig` (the restyled `panel/1`), its fields are the F0
  input styling (the restyled `input/1`), and its submit buttons carry `.btn` / `.btn.ghost` (D6).
- Given the computed-style parity e2e (`apps/e2e/tests/elixir-hero.spec.ts`), when run against the styled Portal under
  `E2E_BASE_URL`, then it passes — the rendered computed styles + geometry match the F0 baseline, so "renders in the system"
  is a checked claim, not a resemblance (D7, INV7; `f0.roadmap.md` M5).

INVEST — independent; testable by the parity e2e + a class grep over the rendered markup; encodes F6.5.5-INV7 and the D5/D6
restyle.
Priority: must · Size: 3 · Implements deliverables: F6.5.5-D5, D6.

## F6.5.5-US2 — Tokens carried once, byte-for-byte

As a **Developer**, I want the F0 token palette to live exactly once in a committed `app.css` served by the mount that
already exists, so that the design system is carried faithfully with no bundler and no drift.

Acceptance criteria
- Given `priv/static/assets/app.css`, when served, then it is delivered verbatim at `~p"/assets/app.css"` by the EXISTING
  `Plug.Static` `/assets` mount (`endpoint.ex:28`, `static_paths/0` already lists `"assets"`) — with NO endpoint change, NO
  config change, and NO bundler (per [`f6.5.5.md`](f6.5.5.md) D4, INV5).
- Given `app.css`'s `:root` block, when diffed against the `f0.md`-D1 palette, then it matches BYTE-FOR-BYTE — the surfaces,
  text, accents, `--line`, the four font stacks, and `--measure: 68ch` — declared exactly once and never re-invented
  (INV2; `f0.md`-INV2). *(token-fidelity proxy.)*
- Given the palette, when rendered, then it is declared once in the single shared root `<head>` and never redeclared per page
  or per component (INV2).

INVEST — independent; testable by the token-fidelity diff + a `curl :4000/assets/app.css`; encodes F6.5.5-INV2 and the D4
no-bundler delivery.
Priority: must · Size: 2 · Implements deliverables: F6.5.5-D4.

## F6.5.5-US3 — Every clamp survives the parser

As a **Developer**, I want every `clamp()` in the design CSS to be correctly spaced, so that titles and body text render at
their fluid sizes instead of silently falling back to UA defaults.

Acceptance criteria
- Given `app.css`, when scanned, then every `clamp()` has spaces around its `+`/`-` operators (`clamp(2.7rem, 1.9rem + 4.2vw,
  5.1rem)`, NEVER `1.9rem+4.2vw`) — the unspaced form is invalid CSS, silently dropped to the UA default (h1 → 32px) (per
  [`f6.5.5.md`](f6.5.5.md) INV3).
- Given the clamp-spacing proxy (a parse / `CSS.supports` probe over `app.css`), when run, then it reports zero unspaced
  `clamp()` — this is the highest-leverage acceptance check (the bug shipped invisibly across 204 static pages).
- Given a rendered heading, when its computed `font-size` is read, then it is the fluid clamp value, not the 32px UA fallback
  (caught by the same parity e2e, INV7).

INVEST — independent; testable by the clamp-spacing parse + the parity e2e; encodes F6.5.5-INV3.
Priority: must · Size: 1 · Implements deliverables: F6.5.5-D4, D7.

## F6.5.5-US4 — Styling over the facade, nothing below it changes

As an **architect**, I want the styling to add zero boundary crossings and to leave everything below the facade
byte-unchanged, so that applying the design system cannot regress the domain.

Acceptance criteria
- Given the layout, `app.css`, and the restyled components, when inspected, then they name ONLY facade-rendered assigns and
  the `~p"/assets/app.css"` static route — no `Portal.Engine`, `Repo`, or `GenServer.call` (per [`f6.5.5.md`](f6.5.5.md)
  INV1).
- Given the change set, when reviewed, then NO file under `apps/portal` or `apps/echo_data`, no endpoint plug, and no config
  key is touched — the engine, contexts, store, ids, and `EventStore` port are byte-unchanged (INV6).
- Given the build references, when grepped, then NO `CoreComponents` module is created or referenced; the three LOCAL
  components carry the full F0 anatomy (INV4; `f6.5.md`-INV8 preserved).

INVEST — independent; testable by a boundary grep + a diff-scope review; encodes F6.5.5-INV1, INV4, INV6.
Priority: must · Size: 1 · Implements deliverables: F6.5.5-D5, D6 (and the D3 binding's facade-safety).

## F6.5.5-US5 — One envelope for static pages and LiveViews alike

As a **Developer**, I want the root layout bound on BOTH the controller render path and the LiveView render path, so that the
F6.5 server-rendered pages and the F6.6 live catalog share one styled envelope.

Acceptance criteria
- Given `portal_web.ex`, when inspected, then `layouts: []` (`:30`) has become `layouts: [html: PortalWeb.Layouts]` (the
  controller path) and the `:live_view` macro (`:52`) carries `layout: {PortalWeb.Layouts, :root}` (the LiveView dead-render
  path) (per [`f6.5.5.md`](f6.5.5.md) D3).
- Given an F6.5 controller page (e.g. the static `/courses/:id` show) and the F6.6 `CatalogLive`, when each renders, then BOTH
  wrap in the SAME `root.html.heex` envelope — the styled `<head>` and the `app.css` `<link>` appear for both (D2, D3).
- Given the live create's success flash, when it sets `@flash`, then the root layout this rung introduces is the surface a
  `<.flash_group>` would render into — closing F6.6's `[DEFERRED]` layout dependency (`f6.6.stories.md:51-53`). *(The
  rendered-flash assertion remains F6.6's, now UNBLOCKED by the root layout; painting the flash is the layout content's
  concern, [DEFERRED] to F6.6's follow-up, not re-claimed here.)*

INVEST — independent; testable by inspecting the binding + rendering both a controller page and the LiveView; encodes
F6.5.5-D3 and the predecessor-deferral close.
Priority: must · Size: 2 · Implements deliverables: F6.5.5-D3.
