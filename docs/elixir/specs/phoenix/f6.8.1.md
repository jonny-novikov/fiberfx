# F6.8.1 · Authentication — the honest door

> Give the Portal real sign-in. Port the static `login.html` into the Portal the F6.5.5 way — a
> `PageController.login/2` action rendering a complete-document HEEx template with its CSS/JS extracted to
> committed static assets — and back it with a NEW `Portal.Auth` facade (`sign_in/2`, `request_reset/1`) over
> the **existing** `%User{}`/`%Session{}` entities. Sign-in posts to `/auth/session`, mints a `%Session{}`, and
> writes the learner's id into the signed session cookie F6.1 already configured; reset-request posts to
> `/auth/reset` and always answers the same way. The interim `RequireUser` plug evolves into a session-backed
> `current_user` (a `%User{}`) gate with an `on_mount` hook, and the unauthenticated redirect target moves from
> `/` to the new `/login`. Registration is deferred. The web calls only `Portal.Auth`; the F5 engine is
> unchanged.

## Goal

After F6.8.1, a request to `GET /login` returns the static login page rendered in the F0 dark-editorial design
system — its own `<head>` (title, meta, Google-Fonts `<link>`, `<link>` to the extracted `login.css`), then its
body (the `.topbar`, the `.hero` with the `.auth-card` sign-in + reset forms and the SVG sign-in-flow, the
`#how` "how the door works" section, the `<footer>`), with `<script src={~p"/assets/login.js"}>` carrying its
progressive-enhancement JS. The page is STATIC content (no `Portal` facade call), parity-faithful to
`elixir/login.html`, exactly as F6.5.5's two indexes are to their masters.

The forms post to two NEW facade-backed endpoints:

- `POST /auth/session` — `SessionController.create/2` reads `identifier` + `password`, calls
  `Portal.Auth.sign_in(identifier, password)` → `{:ok, %Session{}} | {:error, :invalid_credentials}`. On
  `{:ok, session}` it writes the learner's id into the signed session cookie and answers a redirect to
  `/elixir` (the form's `data-redirect`); on `{:error, :invalid_credentials}` it answers a 401/403 — the SAME
  failure for a wrong name and a wrong password (the honest door: one bit on success, zero on failure).
- `POST /auth/reset` — `SessionController.request_reset/2` reads `email`, calls
  `Portal.Auth.request_reset(email)` → always `:ok`, and answers a 200 regardless of whether an account matches
  (no account enumeration — the difference plays out in an inbox, never in the reply).

`Portal.Auth` is a NEW facade module over the existing `Portal.Accounts` entities, exposing `sign_in/2` and
`request_reset/1`; the web auth (the controller + the gate plug) names only `Portal.Auth`, never `Accounts`
internals, `Portal.Engine`, a `Repo`, or `GenServer.call` (the master invariant). The interim `RequireUser`
gate evolves into `PortalWeb.UserAuth` — `fetch_current_user/2` loads a `current_user` (a `%User{}`) from the
session, `require_authenticated_user/2` redirects an anonymous request to `~p"/login"`, and `on_mount/4`
(`:ensure_authenticated`) gives F6.9's `live_session` its auth hook. Registration is deferred (the footer's
*"registration opens with the portal build. Soon."* is preserved as static copy). The F5 engine, store, contexts
below the new `Portal.Auth`/credential surface, and ids are unchanged.

## Rationale (5W)

- **Why** — A learning portal with no sign-in cannot hold per-learner progress, labs, and notes behind one
  door. Authentication gives a real identity and a protected area; doing it as a static-parity port of a page
  that already exists (and already names its own backend contract) ships the visible surface the F6.5.5 way —
  byte-faithful, no dynamic shell — while the credential machinery lands as a thin, honest `Portal.Auth` facade.
  The "honest door" discipline (one failure for both wrong-name and wrong-password; `:ok` for every reset
  request) is the same contract the rest of the course teaches: a function says what happened, not what the
  caller could probe for.
- **What** — the static login-page port (a `login/2` action + a complete-document `login.html.heex` + extracted
  `login.css`/`login.js` + a `GET /login` route); a NEW `Portal.Auth` facade (`sign_in/2`, `request_reset/1`)
  over the existing `%User{}`/`%Session{}` entities plus the net-new credential/hash + session-mint internals;
  two facade-backed controller actions (`POST /auth/session`, `POST /auth/reset`) writing/reading the signed
  session cookie; and the F6.2 `RequireUser` gate evolved into `PortalWeb.UserAuth` (the `current_user` plug +
  the `require_authenticated_user` plug + the `on_mount` hook), with the unauthenticated redirect moved to
  `/login`.
- **Who** — Learners (a real account, sign-in, and a one-step reset request), Operators (a real protected area
  gating the learner surface), and Developers (a facade-backed auth contract the web calls without reaching
  below the boundary).
- **When** — the auth half of the eighth Phoenix rung (`F6.8.1`), built on F6.4's `Accounts` entities, F6.1's
  configured session cookie, and F6.2's interim gate; it is the access substrate F6.9's protected dashboard
  rides (the `on_mount` + `current_user` forward-ref). Deployment (`F6.8.2`) follows.
- **Where** — a NEW `apps/portal/lib/portal/auth.ex` (the facade) + its credential/session-mint internals under
  `Portal.Accounts`; a NEW `apps/portal_web/lib/portal_web/controllers/page_html/login.html.heex` + the
  `login/2` action in `PageController`; a NEW `apps/portal_web/lib/portal_web/controllers/session_controller.ex`
  (the `/auth/*` actions); `apps/portal_web/lib/portal_web/router.ex` (the `/login` + `/auth/session` +
  `/auth/reset` routes); the F6.2 `RequireUser` plug evolved into
  `apps/portal_web/lib/portal_web/user_auth.ex`; and `apps/portal_web/priv/static/assets/login.css` +
  `login.js`.

## The persistence fork (RATIFIED — Option A)

Accounts is Store-backed today with no credential and no auth internals. F6.8.1 adds: a password credential +
hash verification, and session minting (`%Session{id: Portal.ID.new("SES"), user_id: user.id, token: …}`). The
Director ratified (2026-06-05) the thin vertical slice:

- **Option A (RATIFIED — the thin vertical slice).** Keep Accounts Store-backed. Extend the seeded user record
  with a private `password_hash`, verified with **`bcrypt_elixir`**; `Portal.Auth.sign_in/2` resolves the user
  by identifier, verifies the password against the stored hash, and on success mints a `%Session{}` (persisted
  in `Portal.Store` under `SES`, or held by the engine — see the brief). `request_reset/1` resolves-or-not and
  always returns `:ok`. **One net-new dependency**: `{:bcrypt_elixir, "~> 3.1"}` in `apps/portal/mix.exs`. This
  is a per-app DEP-GRAPH-VISIBILITY add — `Portal.Auth` lives in `apps/portal`, so the dep is declared there, not
  transitively assumed. No new schema, no new migration; the thinnest slice that makes the honest door real. The
  `bcrypt_elixir` add gets its own gate at build time; the Deliverables below are written against this option.

> **Documented alternative (not an open fork) — Option B, the durable slice.** Ecto `users` + `sessions` tables
> (F6.3 already added Ecto — `ecto_sql`/`postgrex` are locked and declared in `apps/portal/mix.exs deps`, so this
> needs a migration + schemas but **no new persistence dependency**), plus the same `bcrypt_elixir` dependency.
> It would migrate the Accounts entities off the Store stand-in — a larger move than the auth slice requires, and
> a schema reckoning the deploy rung (F6.8.2) or a later content rung is better placed to own. Recorded for the
> record should the credential later migrate off the Store stand-in; **F6.8.1 builds Option A.**

  the `login.js` reads the status (`r.status===401 || r.status===403`, `login.html:629`) and paints the honest
  failure. It names ONLY `Portal.Auth`, never `Accounts`/`Portal.Engine`/`Repo`/`GenServer.call`. [US:
  F6.8.1-US2, US5]

- **F6.8.1-D7 — `SessionController` over the facade — `POST /auth/reset`.** `SessionController.request_reset/2`:
  reads `email`, calls `Portal.Auth.request_reset(email)`, and answers a 200 regardless of the result (the
  `login.js` treats any `r.ok` as "if an account matches, a link is on its way", `login.html:664-667`). It names
  ONLY `Portal.Auth`. [US: F6.8.1-US2, US4]

- **F6.8.1-D8 — the routes.** In `apps/portal_web/lib/portal_web/router.ex`: a NEW `get("/login",
  PageController, :login)` in the public `scope "/", PortalWeb` block (`pipe_through :browser`, `:59-70`);
  and `post("/auth/session", SessionController, :create)` + `post("/auth/reset", SessionController,
  :request_reset)` (the public `:browser` scope — the sign-in cookie write needs `fetch_session`; CSRF handling
  for the JS `fetch` POST is resolved in the brief). The login page is public; the `/auth/*` endpoints are
  public (an anonymous learner signing in). [US: F6.8.1-US5, US6]

- **F6.8.1-D9 — the gate evolves: `PortalWeb.UserAuth` (`current_user` + `on_mount`), redirect → `/login`.** The
  F6.2 `PortalWeb.RequireUser` plug (`apps/portal_web/lib/portal_web/plugs/require_user.ex`) evolves into
  `apps/portal_web/lib/portal_web/user_auth.ex`, exposing: `fetch_current_user/2` (a plug that reads `:user_id`
  from the session and assigns `current_user`, a `%User{}` loaded via `Portal.Auth`, or `nil`);
  `require_authenticated_user/2` (a plug that admits a request with a `current_user` and redirects an anonymous
  one to `~p"/login"` — the redirect target moved from `~p"/"`, discharging the `RequireUser` moduledoc's owed
  change); and `on_mount/4` (`:ensure_authenticated`, the hook F6.9's `live_session` names as
  `{PortalWeb.UserAuth, :ensure_authenticated}`, `f6.9.stories.md:84`). F6.5's protected `/my/courses`
  (`router.ex:77-81`) reconciles to read the loaded `current_user`. `PortalWeb.UserAuth` calls ONLY `Portal.Auth`
  for the user load (a plug that loads a domain entity through the facade); it names no `Portal.Engine`, `Repo`,
  or `GenServer.call`. [US: F6.8.1-US5, US9]

- **F6.8.1-D10 — verification.** `GET /login` returns the styled login envelope (the master `<head>` + the
  `login.css` `<link>` + the hero `.auth-card`, not a bare body); every `clamp()` in `login.css` is spaced; the
  `login.css` `:root` matches `elixir/login.html:17-32`; the login page makes NO facade call and names no
  `CoreComponents`. `Portal.Auth.sign_in/2` returns `{:ok, %Session{}}` for valid credentials and the SAME
  `{:error, :invalid_credentials}` for a wrong name and a wrong password; `Portal.Auth.request_reset/1` returns
  `:ok` for both a matching and a non-matching email. `POST /auth/session` with valid credentials writes the
  session and redirects, and the protected `/my/courses` then admits; with either half wrong it answers the same
  401/403; an anonymous `GET /my/courses` redirects to `/login`. The web (`SessionController`, `UserAuth`, the
  login page) calls only `Portal.Auth`, never below the boundary. The suite + clean compile stay green; the F5
  engine/store/contexts-below-the-credential-surface/ids are unchanged. [US: F6.8.1-US1..US9]

## Invariants

- **F6.8.1-INV1 (auth over the facade — the master invariant).** `PortalWeb.SessionController`, the login page,
  and `PortalWeb.UserAuth` call ONLY `Portal.Auth`; the credential/hash + session-mint internals live under
  `Portal.Accounts` and are private to the context; no web module names `Portal.Engine`, a `Repo`, or
  `GenServer.call`. `Portal.Auth` is the NEW facade the web auth calls (preserves `f6.5.5.md`-INV1 /
  `f6.5.md`-INV7 / `f6.2.md`-INV1 / echo `CLAUDE.md` §2).

- **F6.8.1-INV2 (the login page is STATIC — no domain call, F6.5.5 parity).** `page_html/login.html.heex`,
  `login.css`, and `login.js`, and the `login/2` action, name only the `~p"/assets/…"` static routes, the
  in-Portal relative links (`/`, `/elixir`, the `/auth/*` POST targets), the config-sourced deep-link base, and
  literal external URLs; the page renders from NO assigns and makes NO `Portal` facade call. It carries its own
  complete `<head>` and single body header/footer (no root layout, no `CoreComponents`, no bundler) — judged at
  envelope + clamp-spacing + token-fidelity parity like the F6.5.5 indexes.

- **F6.8.1-INV3 (the honest door — one bit on success, zero on failure).** `Portal.Auth.sign_in/2` returns the
  IDENTICAL `{:error, :invalid_credentials}` for a non-existent user and for an existing user with a wrong
  password — no path, timing-shortcut, or message distinguishes which half failed; `SessionController.create/2`
  maps both to the same HTTP failure. `Portal.Auth.request_reset/1` ALWAYS returns `:ok` and
  `SessionController.request_reset/2` always answers the same 200 — a caller cannot use either endpoint as an
  account-existence oracle.

- **F6.8.1-INV4 (a session is minted with a branded id).** A successful sign-in mints
  `%Portal.Accounts.Session{id: Portal.ID.new("SES"), user_id: user.id, token: …}` — the `SES` namespace
  (`session.ex`), the canonical `Portal.ID.new/1` mint — and the learner's id is carried in the F6.1-configured
  signed session cookie, never in a URL or an unsigned cookie.

- **F6.8.1-INV5 (protected scopes require a loaded user, redirect to `/login`).** A request to a protected route
  without a session `current_user` is REDIRECTED to `~p"/login"` (never served) by
  `PortalWeb.UserAuth.require_authenticated_user/2`; a request WITH one is admitted with `current_user` (a
  `%User{}`) in `conn.assigns`; the LiveView `on_mount` (`:ensure_authenticated`) enforces the same before a
  `live_session` LiveView connects. F6.5's `/my/courses` reads the loaded `current_user`.

- **F6.8.1-INV6 (registration deferred).** F6.8.1 ships sign-in + reset-request ONLY; no register action, route,
  form, or `Accounts` registration internal is added — the login page's *"No account yet — registration opens
  with the portal build. Soon."* copy is preserved as static text (`login.html:448`).

- **F6.8.1-INV7 (the engine + the catalog/learning core are unchanged; the credential add is additive and
  below-facade-private).** The F5 engine, the `EventStore` port, the catalog/learning contexts, the store
  mechanism, and the ids are byte-unchanged; the ONLY domain change is the additive credential/session-mint
  surface under `Portal.Accounts` + the new `Portal.Auth` facade — and `%User{}`'s public `@derive` does NOT
  serialize the password hash (the credential never crosses the boundary to the web). No deploy concern
  (`runtime.exs`, release, cluster, deploy-bind, mint hardening) is touched here — those are F6.8.2.
