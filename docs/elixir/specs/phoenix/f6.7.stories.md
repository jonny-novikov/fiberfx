# F6.7 · user stories

> Who real-time serves, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.7.md`](f6.7.md).

## F6.7-US0 — An edit path for the update broadcast to fire from

As an **architect**, I want `update_course/2` defined with its facade delegate, so that the `{:course_updated, course}`
broadcast and the replace-in-place stream update have a real write to ride — instead of broadcasting from a path that
does not exist (the F6.7 [RECONCILE]).

Acceptance criteria
- Given F6.4/F6.5 ship `Catalog` list/get/fetch/create/change with no edit path, when this rung builds, then
  `Catalog.update_course/2` and its `Portal.update_course/2` delegate exist (the [RECONCILE] direction folded forward).
- Given `update_course/2`, when it returns `{:ok, _}`, then the context broadcasts `{:course_updated, course}`; when it
  returns `{:error, _}`, then it broadcasts nothing.
- Given the web layer, when it triggers an update, then it names only `Portal.update_course/2`, never the context or the
  engine directly.

INVEST — independent; testable by calling the new facade delegate; encodes F6.7-INV1, INV2.
Priority: must · Size: 2 · Implements deliverables: F6.7-D2.

## F6.7-US1 — See others' changes live

As a **learner**, I want the catalog to update when someone else changes it, so that what I see stays fresh without
reloading.

Acceptance criteria
- Given two connected sessions, when one creates a course, then the other shows it without a reload.
- Given `CatalogLive`, when it mounts connected (behind `connected?/1`), then it calls `Portal.subscribe("courses")`.
- Given a `{:course_created, _}` or `{:course_updated, _}` broadcast, when it arrives, then `handle_info/2` applies it
  and returns `{:noreply, socket}`.
- Given `CatalogLive` is referenced here by module, when routed, then it inherits F6.6's `/courses` namespace reconcile
  and adds no new route in this rung (the F6.7 [RECONCILE]).

INVEST — independent; testable with two LiveView processes; encodes F6.7-INV2, INV3.
Priority: must · Size: 3 · Implements deliverables: F6.7-D1, F6.7-D3.

## F6.7-US2 — Broadcasts are honest

As a **developer**, I want broadcasts to fire only after a successful write, so that clients never see a change that
did not happen.

Acceptance criteria
- Given a write, when it returns `{:ok, _}`, then the `broadcast/2` helper emits `{:course_created, course}` /
  `{:course_updated, course}`.
- Given a write, when it returns `{:error, _}`, then nothing is broadcast.
- Given both `create_course/1` and `update_course/2`, when inspected, then the helper is the single place writes
  broadcast from.

INVEST — independent; testable by a failing write; encodes F6.7-INV1.
Priority: must · Size: 2 · Implements deliverables: F6.7-D2.

## F6.7-US3 — Updates replace in place

As a **learner**, I want an edited course to update in my list, so that I see the new version without a duplicate.

Acceptance criteria
- Given `{:course_created, course}`, when handled, then `stream_insert(socket, :courses, course, at: 0)` prepends it.
- Given `{:course_updated, course}` (from US0's `update_course/2`), when handled, then `stream_insert(socket, :courses,
  course)` replaces the row.
- Given a re-delivered broadcast, when handled, then the row is not duplicated, because `stream_insert` keys on the
  course id.

INVEST — independent; testable by create and update broadcasts; encodes F6.7-INV6.
Priority: must · Size: 2 · Implements deliverables: F6.7-D4.

## F6.7-US4 — A live viewer count

As an **instructor**, I want to see how many people are viewing, so that I have a live sense of attention.

Acceptance criteria
- Given the connected mount, when it runs, then `PortalWeb.Presence.track/3` records the viewer.
- Given a `%{event: "presence_diff"}` message, when received, then the `viewers` count is recomputed.
- Given more than one node, when viewers connect across them, then the count is correct cluster-wide (CRDT-backed, not a
  single-node counter).
- Given the supervision tree, when inspected, then `PortalWeb.Presence` is a child of it.

INVEST — independent; testable by tracking and a diff; encodes F6.7-INV4.
Priority: should · Size: 3 · Implements deliverables: F6.7-D5.

## F6.7-US5 — A non-LiveView client (optional)

As a **developer**, I want a minimal channel, so that a non-LiveView client can receive catalog updates over the same
PubSub.

Acceptance criteria
- Given a channel module, when declared, then it implements `join/3` and `handle_in/3`.
- Given a `UserSocket`, when configured, then the channel is reachable.

INVEST — independent; small; testable by a channel join; encodes F6.7-D6.
Priority: could · Size: 1 · Implements deliverables: F6.7-D6.

## F6.7-US6 — Real-time over the facade, minimal new surface

As an **architect**, I want subscribe/broadcast to go through the facade, so that real-time adds no boundary leaks and
the substrate the F6.9 dashboard folds stays behind `Portal`.

Acceptance criteria
- Given the LiveView and context, when they do real-time, then they call `Portal.subscribe/1` / `Portal.broadcast/2`
  over `Portal.PubSub`.
- Given the tree, when inspected, then the only new children are `Phoenix.PubSub` and `PortalWeb.Presence`; everything
  below the facade is unchanged.
- Given any module, when inspected, then none names the engine, a `Repo`, or `GenServer.call`.

INVEST — independent; testable by inspection; encodes F6.7-INV2, INV5.
Priority: must · Size: 1 · Implements deliverables: F6.7-D1.
