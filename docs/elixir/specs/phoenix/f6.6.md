# F6.6 · LiveView

> Make F6.5's catalog interactive without full reloads. `CatalogLive` mounts `@courses` from the facade and renders the
> F6.5 `course_card`; a two-stage mount (disconnected first paint, then connected) guards side effects with
> `connected?/1`; a live search box filters as the user types; a live create form flashes on success and shows inline
> errors on failure; and a `stream` keeps a large list out of socket memory. The F6.3–F6.4 facade, context, and
> changeset are unchanged.

## Goal

After F6.6, the catalog is interactive without a page reload. `CatalogLive` (`use PortalWeb, :live_view`) assigns
`@courses` from the facade in `mount/3` and renders a `~H` template that reuses the F6.5 `course_card` component. The
mount runs in two stages — a disconnected HTTP render for first paint, then a connected mount — with `connected?/1`
guarding work that should run only once the socket is live. A live search box (`phx-change="search"` →
`handle_event("search", ...)`) filters through the `Catalog` context and re-assigns `@courses` as the user types. A
live create form (`phx-submit="create"` → `handle_event("create", ...)`) calls `Portal.create_course/1`, flashes and
resets on `{:ok, _}`, and re-assigns `to_form(changeset)` with inline errors on `{:error, _}`. A large, append-only
list uses `stream/3` + `stream_insert/3` + `phx-update="stream"`, so it lives as rows in the DOM with ids on the server
and no list in socket memory. A `live "/catalog", CatalogLive` route sits in the `:browser` scope.

## Rationale (5W)

- **Why** — LiveView is where Phoenix makes a page interactive without leaving the server: state lives in a socket, the
  client sends events, and the framework diffs and patches the DOM — so search-as-you-type and inline create work with
  no client framework and no second API. The two-stage mount keeps first paint fast and indexable while the connected
  socket carries interactivity, and streams keep large lists off the socket heap. Without LiveView, the same
  interactivity needs a client app and a parallel contract.
- **What** — the `CatalogLive` view, the two-stage mount with `connected?/1`, the live search, the live create, streams
  for the list, and the live route.
- **Who** — Learners (search-as-you-type and immediate create feedback) and Developers (interactivity over the same
  facade, with no separate frontend).
- **When** — the sixth Phoenix rung; it makes F6.5's static pages live over the same F6.4 context and F6.3 changeset,
  and it is the page F6.7 pushes across clients.
- **Where** — `lib/portal_web/live/catalog_live.ex`, the router (`live "/catalog", CatalogLive`), and the unchanged
  F6.3–F6.4 facade, `Portal.Catalog` context, and changeset.

## Scope

- **In** — `CatalogLive` (`mount/3`, `handle_event/3`, `render/1`); the two-stage mount with `connected?/1` guarding
  side effects; live search (`phx-change` → filter through the context → re-assign `@courses`); live create
  (`phx-submit` → `Portal.create_course/1` → flash and reset on success, `to_form(changeset)` inline errors on
  failure); streams (`stream/3` + `stream_insert/3` + `phx-update="stream"`); and the live route in `:browser`.
- **Out** — PubSub, Presence, and multi-client updates (→ F6.7); authentication (→ F6.8); the dashboard (→ F6.9); any
  new domain feature; and changes to the facade, context, or changeset (unchanged here).

## Deliverables

- **F6.6-D1** — `CatalogLive` (`use PortalWeb, :live_view`): `mount/3` assigns `@courses` from the facade plus the
  search and form assigns; `render/1` returns a `~H` template that reuses the F6.5 `course_card` component;
  `handle_event/3` returns `{:noreply, socket}`.
- **F6.6-D2** — a two-stage mount: a disconnected HTTP render for first paint, then a connected mount, with
  `connected?/1` guarding work that should run once the socket is live.
- **F6.6-D3** — a live search box: `phx-change="search"` → `handle_event("search", params, socket)` filters courses
  through the `Catalog` context and re-assigns `@courses`, updating as the user types.
- **F6.6-D4** — a live create form: `phx-submit="create"` → `handle_event("create", params, socket)` calls
  `Portal.create_course/1`; on `{:ok, _}` it `put_flash`es and resets the form; on `{:error, %Ecto.Changeset{} = cs}`
  it assigns `to_form(cs)` so the F6.3 errors appear inline.
- **F6.6-D5** — streams for the list: `stream/3` to initialize, `stream_insert/3` on create, and `phx-update="stream"`
  in the template, so a large, append-only list is rows in the DOM with ids on the server and no list in socket memory.
- **F6.6-D6** — the route: `live "/catalog", CatalogLive` in a `:browser` scope.
- **F6.6-D7** — verification: typing filters the list live; a valid create appears and flashes; an invalid create shows
  inline errors; the list is a stream (no list assign); first paint renders over HTTP before the socket connects; and
  the view calls only the facade and context.

## Invariants

- **F6.6-INV1 (interactivity over the facade — the master invariant)** — the LiveView calls only `Portal` and the
  `Catalog` context, never the engine, a `Repo`, or `GenServer.call`; it renders only assigns and the closed
  `%Portal.Error{}` set.
- **F6.6-INV2 (two-stage mount)** — `mount/3` runs disconnected (HTTP) for first paint, then connected; side effects
  are guarded by `connected?/1`, so they run once, on the live socket.
- **F6.6-INV3 (state in the socket)** — interactive state lives in socket assigns; the client holds no business logic;
  events drive every change.
- **F6.6-INV4 (streams for large lists)** — an append-only or large list uses `stream/3` + `stream_insert/3` +
  `phx-update="stream"`; the full list is not held in socket memory.
- **F6.6-INV5 (errors surface inline, parse boundary unchanged)** — a failed create re-assigns `to_form(changeset)` so
  the F6.3 errors show inline; the view adds no validation of its own, and `create_course/1`'s changeset is the parse
  boundary.
- **F6.6-INV6 (component reuse)** — `render/1` reuses the F6.5 `course_card` component; row markup is not duplicated in
  the LiveView.

## Definition of Done

- [ ] `CatalogLive` mounts `@courses` from the facade and renders by reusing `course_card` (F6.6-D1, INV6).
- [ ] The mount is two-stage and `connected?/1` guards side effects (F6.6-INV2).
- [ ] `phx-change="search"` filters through the context and re-assigns `@courses` live (F6.6-D3).
- [ ] `phx-submit="create"` flashes on success and shows inline errors on failure (F6.6-D4, INV5).
- [ ] The list uses `stream/3` + `stream_insert/3` + `phx-update="stream"`; no list assign (F6.6-INV4).
- [ ] `live "/catalog", CatalogLive` is reachable in the `:browser` scope (F6.6-D6).
- [ ] The view calls only the facade and context (F6.6-INV1); every deliverable maps to a story (see
      `./f6.6.stories.md`).
