# F6.6 ¬∑ LiveView

> Make F6.5's catalog interactive without full reloads. `CatalogLive` mounts `@courses` from the facade and renders the
> F6.5 `course_card`; a two-stage mount (disconnected first paint, then connected) guards side effects with
> `connected?/1`; a live search box filters as the user types via the new `Portal.search_courses/1`; a live create form
> flashes on success and shows inline errors on failure; and a `stream` keeps a large list out of socket memory. The
> F6.3 changeset and the F6.4 create path are unchanged; the facade and `Catalog` context gain only the one read-only
> `search_courses/1` (`[RECONCILE]`).

## [RECONCILE] ‚Äî folded forward from F6.5 (routes & components, `5a440fd`)

Knowns to resolve when this rung is built (made explicit per the F6.5 routing + component direction):

- **Route ‚Äî the live catalog shares the catalog namespace.** F6.5 made the catalog `resources "/courses"` (server-rendered index/show/new/create on `CourseController`). This rung's `live "/catalog", CatalogLive` predates that and addresses the SAME resource ‚Äî reconcile it into the `/courses` namespace (e.g. `live "/courses", CatalogLive` superseding the static `GET /courses` index), not a second `/catalog` surface. *(Why: one catalog resource, one URL; F6.5 claimed `/courses`.)*
- **Components ‚Äî reuse the LOCAL set, no `CoreComponents`.** `course_card`/`panel`/`input/1` live in `PortalWeb.CatalogComponents`, imported via `portal_web.ex` `html_helpers`; there is no `CoreComponents` (F6.5-INV8). The live create form uses the local `<.input>` and posts BOTH required fields (`title` + `slug`; `validate_required` rejects a title-only form). *(Why: F6.5 chose a minimal local component home.)*
- **Facade-only ‚Äî search through `Portal`, not `Catalog`.** D3/INV1's filter "through the `Catalog` context" must reconcile to a `Portal` facade call (a facade `search`/filter fn), since the web names only the facade (F6.5-INV7 reinforced the master invariant) ‚Äî or ratify a deliberate LiveView‚Üícontext relaxation. *(Why: web names only `Portal`.)*

## Goal

After F6.6, the catalog is interactive without a page reload. `CatalogLive` (`use PortalWeb, :live_view`) assigns
`@courses` from the facade in `mount/3` and renders a `~H` template that reuses the F6.5 `course_card` component. The
mount runs in two stages ‚Äî a disconnected HTTP render for first paint, then a connected mount ‚Äî with `connected?/1`
guarding work that should run only once the socket is live. A live search box (`phx-change="search"` ‚Üí
`handle_event("search", ...)`) filters through the new `Portal.search_courses/1` facade function (`[RECONCILE]`: the web
names only the facade) and re-assigns `@courses` as the user types. A live create form (`phx-submit="create"` ‚Üí
`handle_event("create", ...)`) calls `Portal.create_course/1`, flashes and resets on `{:ok, _}`, and re-assigns
`to_form(changeset)` with inline errors on `{:error, _}`. A large, append-only list uses `stream/3` +
`stream_insert/3` + `phx-update="stream"`, so it lives as rows in the DOM with ids on the server and no list in socket
memory. A `live "/courses", CatalogLive` route sits in the `:browser` scope, superseding F6.5's static `GET /courses`
index (`resources "/courses"` keeps only `[:show, :new, :create]`).

## Rationale (5W)

- **Why** ‚Äî LiveView is where Phoenix makes a page interactive without leaving the server: state lives in a socket, the
  client sends events, and the framework diffs and patches the DOM ‚Äî so search-as-you-type and inline create work with
  no client framework and no second API. The two-stage mount keeps first paint fast and indexable while the connected
  socket carries interactivity, and streams keep large lists off the socket heap. Without LiveView, the same
  interactivity needs a client app and a parallel contract.
- **What** ‚Äî the `CatalogLive` view, the two-stage mount with `connected?/1`, the live search, the live create, streams
  for the list, and the live route.
- **Who** ‚Äî Learners (search-as-you-type and immediate create feedback) and Developers (interactivity over the same
  facade, with no separate frontend).
- **When** ‚Äî the sixth Phoenix rung; it makes F6.5's static pages live over the same F6.4 context and F6.3 changeset,
  and it is the page F6.7 pushes across clients.
- **Where** ‚Äî `lib/portal_web/live/catalog_live.ex`, the router (`live "/courses", CatalogLive` superseding the static
  index), the `Portal` facade and `Portal.Catalog` context (each gains one read-only `search_courses/1`; the create
  path and changeset are otherwise unchanged from F6.3‚ÄìF6.4).

## Scope

- **In** ‚Äî `CatalogLive` (`mount/3`, `handle_event/3`, `render/1`); the two-stage mount with `connected?/1` guarding
  side effects; live search (`phx-change` ‚Üí `Portal.search_courses/1` ‚Üí re-assign `@courses`); live create
  (`phx-submit` ‚Üí `Portal.create_course/1` ‚Üí flash and reset on success, `to_form(changeset)` inline errors on
  failure); streams (`stream/3` + `stream_insert/3` + `phx-update="stream"`); the live route in `:browser` superseding
  the static `/courses` index; and the ONE read-only facade addition the search needs ‚Äî `Portal.search_courses/1`
  delegating to `Catalog.search_courses/1` (`[RECONCILE]`-ratified; named to match the `list_courses/0` siblings).
- **Out** ‚Äî PubSub, Presence, and multi-client updates (‚Üí F6.7); authentication (‚Üí F6.8); the dashboard (‚Üí F6.9); any
  new domain feature; and any change to the create path, the F6.3 changeset, or the contexts BELOW the one read-only
  `search_courses/1` delegate (the write boundary is unchanged here).

## Deliverables

- **F6.6-D1** ‚Äî `CatalogLive` (`use PortalWeb, :live_view`): `mount/3` assigns `@courses` from the facade plus the
  search and form assigns; `render/1` returns a `~H` template that reuses the F6.5 `course_card` component;
  `handle_event/3` returns `{:noreply, socket}`.
- **F6.6-D2** ‚Äî a two-stage mount: a disconnected HTTP render for first paint, then a connected mount, with
  `connected?/1` guarding work that should run once the socket is live.
- **F6.6-D3** ‚Äî a live search box: `phx-change="search"` ‚Üí `handle_event("search", params, socket)` filters courses
  through the new `Portal.search_courses/1` facade function (delegating to `Catalog.search_courses/1 :: [Course.t()]`,
  the one `[RECONCILE]`-ratified read-only addition ‚Äî the web names only the facade, F6.5-INV7) and re-assigns
  `@courses`, updating as the user types.
- **F6.6-D4** ‚Äî a live create form: `phx-submit="create"` ‚Üí `handle_event("create", params, socket)` calls
  `Portal.create_course/1`; on `{:ok, _}` it `put_flash`es and resets the form; on `{:error, %Ecto.Changeset{} = cs}`
  it assigns `to_form(cs)` so the F6.3 errors appear inline.
- **F6.6-D5** ‚Äî streams for the list: `stream/3` to initialize, `stream_insert/3` on create, and `phx-update="stream"`
  in the template, so a large, append-only list is rows in the DOM with ids on the server and no list in socket memory.
- **F6.6-D6** ‚Äî the route: `live "/courses", CatalogLive` in the `:browser` scope, superseding F6.5's static `GET
  /courses` index ‚Äî one catalog resource, one URL (`[RECONCILE]`, route). F6.5's `resources "/courses", CourseController,
  only: [:index, :show, :new, :create]` reconciles to `only: [:show, :new, :create]` (the `:index` action is replaced by
  `CatalogLive`).
- **F6.6-D7** ‚Äî verification: typing filters the list live; a valid create appears and flashes; an invalid create shows
  inline errors; the list is a stream (no list assign); first paint renders over HTTP before the socket connects; and
  the view calls only the facade and context.

## Invariants

- **F6.6-INV1 (interactivity over the facade ‚Äî the master invariant)** ‚Äî the LiveView names only `Portal` (the facade
  fronts the `Catalog` context ‚Äî `[RECONCILE]`, facade-only: search runs through `Portal.search_courses/1`, not a direct
  `Catalog` call), never the engine, a `Repo`, or `GenServer.call`; it renders only assigns and the closed
  `%Portal.Error{}` set.
- **F6.6-INV2 (two-stage mount)** ‚Äî `mount/3` runs disconnected (HTTP) for first paint, then connected; side effects
  are guarded by `connected?/1`, so they run once, on the live socket.
- **F6.6-INV3 (state in the socket)** ‚Äî interactive state lives in socket assigns; the client holds no business logic;
  events drive every change.
- **F6.6-INV4 (streams for large lists)** ‚Äî an append-only or large list uses `stream/3` + `stream_insert/3` +
  `phx-update="stream"`; the full list is not held in socket memory.
- **F6.6-INV5 (errors surface inline, parse boundary unchanged)** ‚Äî a failed create re-assigns `to_form(changeset)` so
  the F6.3 errors show inline; the view adds no validation of its own, and `create_course/1`'s changeset is the parse
  boundary.
- **F6.6-INV6 (component reuse)** ‚Äî `render/1` reuses the F6.5 `course_card` component; row markup is not duplicated in
  the LiveView.
