## Routing decision (resolves the `/courses` collision; ratified â€” apply, do not re-litigate)

F6.2 shipped `get "/courses/:user_id" â†’ CourseController.index`, meaning **a user's enrollments** (`router.ex:46`;
`course_controller.ex:41-44` reads `Portal.courses_of/1`). This F6.5 spec assumes `/courses` is the **catalog** â€” a
head-on collision. Resolution, per REST best practice (name each URL after the resource it returns) + the engine's two
bounded contexts, one controller per context:

| Resource | Context | Final routes |
|---|---|---|
| Course (catalog) | `Portal.Catalog` | `resources "/courses", CourseController, only: [:index, :show, :new, :create]` â†’ `GET /courses`, `GET /courses/:id`, `GET /courses/new`, `POST /courses` |
| A learner's enrolled courses | `Portal.Enrollment.courses_of/1` | `get "/my/courses", EnrollmentController, :index` (protected; reads `conn.assigns.current_user_id`) |

Three ratified sub-decisions:
1. `get "/courses/:user_id"` **and** the `/learn` scope BOTH retire into `get "/my/courses"` (one honest name for "a
   learner's courses").
2. The enrolled-courses read MOVES `CourseController` â†’ `EnrollmentController.index`. `CourseController` becomes the
   **pure catalog resource** (`index`/`show`/`new`/`create`), so each controller maps 1:1 to its context. The existing
   `course_html/index.html.heex` enrolled markup, the `:error` template, and the `render_outcome/2` railway MOVE with
   the read to the enrollment view/controller.
3. `EnrollmentController.create`'s post-enroll redirect changes `~p"/courses/#{user_id}"` â†’
   `~p"/courses/#{course_id}"` (`enrollment_controller.ex:26` â†’ the catalog **show** of the joined course; needs no
   session).

`:show` is in scope **because** `~p"/courses/#{course.id}"` (the index link, D1) and the post-enroll redirect must
compile under Verified Routes â€” an undeclared route fails the build. So `only:` includes `:show` and a minimal
`course_html/show.html.heex` ships (D8).

## Component-home decision (no `CoreComponents` exists)

This umbrella was hand-built across F6.1/F6.2 **without** `mix phx.gen.*`, so there is **no `CoreComponents` module**
(grep confirms). `Phoenix.Component` (reached via `portal_web.ex`'s `:html` â†’ `html_helpers`) provides `<.form>`,
`<.link>`, and `to_form/1`, but **not** `<.input>`/`<.label>`/`<.error>` â€” the `def input(assigns)` at
`deps/phoenix_live_view/lib/phoenix_component.ex:2304` is a **code example inside the `@doc` of `form/1`** ("generally
defined as part of your own application"), not an export. D4's `<.input field={@form[:title]} label="Title"/>`
therefore references a component that must be authored.

Decision: fold a **minimal `input/1`** (a text field + a per-field changeset error) into the **new
`PortalWeb.CatalogComponents`** (`catalog_components.ex`, already in the file set), NOT a generated `CoreComponents`
kitchen-sink. Rationale: process weight scales to risk; one create form needs one text input with inline errors, and a
component the rest of the app would not use does not earn a separate module. The F6.3 changeset errors reach the field
through `@form[:title]` (a `Phoenix.HTML.FormField` carrying `.errors`).

## Goal

After F6.5, the catalog is rendered from the server. `CourseController` is now the **pure catalog resource**
(`index`/`show`/`new`/`create`); the enrolled-courses read it previously held has moved to `EnrollmentController.index`
at `get "/my/courses"` (see the routing decision above). `course_html/index.html.heex` renders `@courses` with `:for`,
a `~p` verified link per course (`~p"/courses/#{course.id}"`, pointing at the catalog `:show`), escaped interpolation
for the title, and an `:if` empty state â€” with no `Repo` and no context call in the template. A `course_card` function
component (`attr :course, :map, required: true`, optional `:class`) replaces the per-row markup, and a slot-based
`panel` wrapper (`slot :inner_block`, `render_slot/1`) frames caller content. A changeset-backed form â€” `to_form/1`
over `Catalog.change_course/1` (the new wrapper, surfaced on the `Portal` facade as `change_course/1`), rendered with
`<.form>` and the locally-defined `<.input>` â€” drives the create action: params go to `Catalog.create_course/1`,
`{:ok, course}` redirects to `~p"/courses/#{course.id}"`, and `{:error, %Ecto.Changeset{}}` re-renders with
`to_form(changeset)` so the F6.3 changeset errors appear inline per field. Data access stays in the controller; the
templates only render from assigns.

## Rationale (5W)

- **Why** â€” HEEx is where Phoenix renders safely and composably: templates are compile-checked, interpolation is
  escaped by default, `~p` routes are verified, and function components turn repeated markup into a declared, reusable
  unit. The discipline that keeps data access in the controller and rendering in the template is what makes views
  testable and the markup free of surprise queries. Without it, templates make context calls, routes drift, and
  validation errors vanish.
- **What** â€” the index template over `@courses`, the `course_card` component, the slot-based `panel` wrapper, the
  changeset-backed form, the create action with inline errors, and the controller-fetches/template-renders discipline.
- **Who** â€” Visitors and Learners (a real rendered catalog and a working create form with clear inline errors).
- **When** â€” the fifth Phoenix rung; it renders what F6.3 and F6.4 make queryable and writable, and it is the static
  UI that F6.6 makes live.
- **Where** â€” `lib/portal_web/controllers/course_html/index.html.heex`, `course_html/show.html.heex`, and
  `course_html/new.html.heex`; `lib/portal_web/controllers/course_controller.ex` (repurposed to the catalog
  resource); `lib/portal_web/components/catalog_components.ex` (the `course_card`/`panel`/`input` home); the
  `Portal.Catalog.change_course/1` changeset wrapper + its `Portal` facade delegate; the enrolled-read MOVE into
  `enrollment_controller.ex` + `enrollment_html/index.html.heex` + `enrollment_html/error.html.heex`; the
  `router.ex` route changes; and the `enrollment_controller.ex:26` redirect fix.

## Scope

- **In** â€” the routing reconcile (catalog `resources "/courses"` `only: [:index, :show, :new, :create]`; the enrolled
  read moved to `get "/my/courses" â†’ EnrollmentController.index`; `/learn` + `/courses/:user_id` retired; the
  `enrollment_controller.ex:26` redirect fix); the catalog index template (`:for`, `~p` link to `:show`, escaped
  interpolation, `:if` empty state, no data access); a minimal `show.html.heex`; the `course_card` function component

- **Out** â€” LiveView and live validation (â†’ F6.6); PubSub (â†’ F6.7); a styling or design system beyond minimal classes;

## Invariants

- **F6.5-INV1 (templates render, controllers fetch)** â€” templates read only from assigns; all data access (the
  `Catalog` calls) is in the controller; no template names a `Repo` or a context.
- **F6.5-INV2 (safe by default)** â€” interpolation is HEEx-escaped (`{...}`); raw output is never used for user data.
- **F6.5-INV3 (verified routes)** â€” every link uses a `~p` verified route, so a path typo fails to compile.
- **F6.5-INV4 (declared components)** â€” function components declare their `attr`s (required where needed), so a missing
  required attr is a compile warning; slots are rendered with `render_slot/1`.
- **F6.5-INV5 (errors surface inline)** â€” a failed create re-renders the form with `to_form(changeset)`, so the F6.3
  changeset errors appear inline per field rather than being swallowed.
- **F6.5-INV6 (parse boundary unchanged)** â€” the form posts params to `Catalog.create_course/1`, which parses through
  the F6.3 `changeset/2`; the view adds no validation of its own.
- **F6.5-INV7 (one controller per context)** â€” `CourseController` calls only `Portal.Catalog` facade functions
  (`list_courses/0`, `get_course!/1`, `change_course/1`, `create_course/1`); `EnrollmentController` calls only the
  enrollment facade functions (`enroll/2`, `courses_of/1`). No URL maps to the wrong resource: `/courses*` is the
  catalog, `/my/courses` is the learner's enrollments. The web still names only the `Portal` facade (F6.2-INV1
  preserved through the move).
- **F6.5-INV8 (no `CoreComponents`)** â€” form-field components (`input/1`) are defined locally in
  `catalog_components.ex`; the build references no `CoreComponents` module and no unexported `Phoenix.Component`
  `input`/`label`/`error`.
