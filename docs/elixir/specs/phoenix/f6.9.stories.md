# F6.9 · user stories

> Who the dashboard serves, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.9.md`](f6.9.md).

## F6.9-US1 — Live metrics

As an **operator**, I want course and enrollment counts that update live, so that I see platform activity without
refreshing.

Acceptance criteria
- Given the mount, when the page loads, then `courses_count` and `enrollments_count` are seeded from the F6.4 contexts.
- Given a `{:course_created, _}` broadcast, when received, then `courses_count` increases.
- Given an `{:enrolled, _}` broadcast, when received, then `enrollments_count` increases — with no full re-query.

INVEST — independent; testable by seeding then broadcasting; encodes F6.9-INV2.
Priority: must · Size: 3 · Implements deliverables: F6.9-D1, F6.9-D3.

## F6.9-US2 — An activity feed

As an **instructor**, I want a live feed of recent activity, so that I can watch what is happening as it happens.

Acceptance criteria
- Given an event broadcast, when received, then a feed row is prepended via `stream_insert(:events, row, at: 0)`.
- Given the feed, when it grows, then it is a capped `stream` (no unbounded list in socket memory).
- Given the render, when drawn, then the feed is under `phx-update="stream"`.

INVEST — independent; testable by broadcasting several events; encodes F6.9-INV4.
Priority: must · Size: 2 · Implements deliverables: F6.9-D1, F6.9-D4.

## F6.9-US3 — Who is watching

As an **instructor**, I want a live viewer count, so that I know how many people are on the dashboard.

Acceptance criteria
- Given the connected mount, when it runs, then `Presence.track/3` records the viewer in `"dashboard"`.
- Given a `"presence_diff"`, when received, then `viewers` is recomputed from `Presence.list("dashboard")`.
- Given multiple nodes, when viewers connect across them, then the count is correct cluster-wide.

INVEST — independent; testable by tracking and a diff; encodes F6.9-INV5.
Priority: should · Size: 2 · Implements deliverables: F6.9-D5.

## F6.9-US4 — Read-only and safe

As an **operator**, I want the dashboard to be read-only, so that watching it cannot change anything.

Acceptance criteria
- Given the page, when rendered, then it has no form and no mutating control.
- Given the LiveView, when it runs, then it issues no command and no write.
- Given an event, when folded, then the dashboard only reads and updates its own read model.

INVEST — independent; testable by inspecting the page and the callbacks; encodes F6.9-INV1.
Priority: must · Size: 1 · Implements deliverables: F6.9-D4.

## F6.9-US5 — Protected access

As an **operator**, I want the dashboard behind auth, so that only signed-in users can reach it.

Acceptance criteria
- Given the route, when declared, then it is in a `live_session` under the F6.8 auth.
- Given an anonymous user, when they request `/dashboard`, then they are redirected to login.
- Given the LiveView, when it mounts, then `on_mount` requires an authenticated user.

INVEST — independent; testable by anonymous vs authenticated requests; encodes F6.9-INV5.
Priority: must · Size: 1 · Implements deliverables: F6.9-D6.

## F6.9-US6 — A capstone over the facade

As a **developer**, I want the dashboard to compose existing capabilities over the facade, so that the capstone adds no
new domain and no boundary leak.

Acceptance criteria
- Given the dashboard, when it seeds, then it reads the contexts' public counts; when it subscribes, then it uses the
  `Portal` PubSub facade.
- Given the LiveView, when inspected, then it names no engine, repo, or `GenServer.call`.
- Given the events it folds, when handled, then they are existing broadcasts — no new event type or write path.

INVEST — independent; testable by inspection; encodes F6.9-INV3, INV6.
Priority: must · Size: 1 · Implements deliverables: F6.9-D2.
