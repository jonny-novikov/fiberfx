# F6.9 · user stories

> Who the dashboard serves, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.9.md`](f6.9.md). Reconciled to the
> [`f6.9.md`](f6.9.md) `[RECONCILE]` callout: the new counts surface on the `Portal` facade (F6.5-INV7, the master
> invariant), the `{:enrolled, _}` broadcast is added on this rung (the F6.7 ok-only discipline applied to enrollment),
> and the `live_session` rides the F6.8 `UserAuth`/`current_user` auth (the interim `RequireUser`/`:require_auth`
> superseded; the `CoreComponents` decision come due — the dashboard renders over whichever surface F6.8 ratified).

## F6.9-US1 — Live metrics seeded then folded

As an **operator**, I want course and enrollment counts that seed once and then update live, so that I read platform
activity as it happens without ever re-querying.

Acceptance criteria
- Given the mount, when `DashboardLive` runs, then `courses_count` is seeded from `Portal.count_courses/0`
  (`Catalog.count_courses/0` on the facade) and `enrollments_count` from `Portal.count_enrollments/0`
  (`Enrollment.count/0` on the facade), both assigned as socket assigns.
- Given a `{:course_created, _}` broadcast, when `handle_info/2` folds it, then `courses_count` increases by exactly 1
  via `update(socket, :courses_count, &(&1 + 1))`.
- Given an `{:enrolled, _}` broadcast, when folded, then `enrollments_count` increases by exactly 1 — and neither fold
  re-runs `count_courses/0`/`count/0` (the count moves only by folding, not by a fresh query).

INVEST — independent; testable by reading the seed assigns then asserting each broadcast bumps the right count without
a re-query; encodes F6.9-INV2.
Priority: must · Size: 3 · Implements deliverables: F6.9-D1, F6.9-D3.

## F6.9-US2 — A capped activity feed

As an **instructor**, I want a live feed of recent activity, so that I can watch what is happening as it happens
without the page ever growing unbounded.

Acceptance criteria
- Given the connected mount, when it runs, then the feed is started as `stream(socket, :events, [])` (a stream, not a
  list assign).
- Given a folded `{:course_created, _}` or `{:enrolled, _}`, when handled, then a row is prepended via
  `stream_insert(socket, :events, row, at: 0)`.
- Given many broadcasts, when the feed fills, then it stays a capped `stream` — rows live in the DOM, with no unbounded
  list held in socket memory.
- Given the render, when drawn, then the feed container carries `phx-update="stream"`.

INVEST — independent; testable by starting the stream, broadcasting several events, and asserting capped DOM rows under
`phx-update="stream"`; encodes F6.9-INV4.
Priority: must · Size: 2 · Implements deliverables: F6.9-D1, F6.9-D4.

## F6.9-US3 — A cluster-correct viewer count

As an **operator**, I want a live count of who is watching that stays right across the cluster, so that the viewer
number is trustworthy even when watchers connect on different nodes.

Acceptance criteria
- Given the connected mount, when it runs, then `PortalWeb.Presence.track(self(), "dashboard", socket.id, %{})` records
  the viewer in the `"dashboard"` topic.
- Given a `%{event: "presence_diff"}` message, when received, then `viewers` is recomputed as
  `map_size(Presence.list("dashboard"))` and re-assigned.
- Given watchers connecting and disconnecting across multiple nodes, when each `presence_diff` arrives, then `viewers`
  reflects the cluster-wide count (Presence from F6.7/F6.8), not a per-node tally.

INVEST — independent; testable by tracking a viewer, simulating a `presence_diff`, and asserting `viewers ==
map_size(Presence.list("dashboard"))` across nodes; encodes F6.9-INV5.
Priority: should · Size: 2 · Implements deliverables: F6.9-D5.

## F6.9-US4 — Read-only by construction

As an **operator**, I want the dashboard to be strictly read-only, so that watching it can never change platform state.

Acceptance criteria
- Given the page, when rendered, then it draws metric cards from `@courses_count`, `@enrollments_count`, and `@viewers`
  with no form and no mutating control (no submit, no `phx-click` that writes).
- Given the LiveView, when any callback runs, then it issues no command and no write — it only seeds and folds
  broadcasts.
- Given a folded event, when handled, then the dashboard updates only its own socket read model and emits no new
  broadcast or domain event.

INVEST — independent; testable by inspecting the rendered markup and the callbacks for any write path; encodes
F6.9-INV1.
Priority: must · Size: 1 · Implements deliverables: F6.9-D4.

## F6.9-US5 — Protected access under the generated auth

As an **operator**, I want `/dashboard` behind the generated auth, so that only a signed-in user can reach it.

Acceptance criteria
- Given the route, when declared, then `live "/dashboard", DashboardLive` sits inside a `live_session` whose `on_mount`
  is the F6.8 `{PortalWeb.UserAuth, :ensure_authenticated}` (the generated `UserAuth`, superseding the interim
  `RequireUser`/`:require_auth`).
- Given an anonymous request to `/dashboard`, when it is handled, then the LiveView does not connect and the visitor is
  redirected to the generated login path.
- Given a signed-in request, when it mounts, then `current_user` (a `%User{}` from F6.8) is present and the dashboard
  renders.

INVEST — independent; testable by an anonymous request (redirect) versus an authenticated one (mounts) through the
`live_session`; encodes F6.9-INV5.
Priority: must · Size: 1 · Implements deliverables: F6.9-D6.

## F6.9-US6 — A subscription over the facade

As a **developer**, I want the dashboard to seed and subscribe only over the `Portal` facade, so that the capstone
names no engine, repo, or `GenServer.call`.

Acceptance criteria
- Given the connected mount, when it runs, then it `Portal.subscribe("events")` for the domain topic (the F6.7 `Portal`
  PubSub facade) and seeds via the contexts' facade counts (`Portal.count_courses/0` / `Portal.count_enrollments/0`).
- Given the LiveView, when inspected (`grep -rE "Portal\.Engine|Repo|GenServer\.call"` over `dashboard_live.ex`), then
  the match is empty — it renders only assigns and the stream.
- Given the `{:enrolled, _}` broadcast it folds, when traced, then it is the new ok-only enrollment broadcast added on
  this rung (the F6.7 discipline applied to `Enrollment`), not a new event type invented in the web layer.

INVEST — independent; testable by inspection plus the empty grep; encodes F6.9-INV3.
Priority: must · Size: 1 · Implements deliverables: F6.9-D2.

## F6.9-US7 — A composition that adds no new domain

As an **architect**, I want the capstone to compose only existing capabilities, so that it ships no new domain event
type and no write path.

Acceptance criteria
- Given the dashboard, when assembled, then it reuses contexts (the seed), the `Portal` PubSub facade (the stream),
  Presence (the count), streams (the feed), and the F6.8 auth (access) — adding no new context, topic, or surface the
  spec does not name.
- Given the events folded, when enumerated, then they are exactly `{:course_created, _}`, `{:enrolled, _}`, and
  `presence_diff` — all existing broadcasts; no new domain event type is introduced.
- Given the rung, when complete, then no write path exists from the dashboard (it composes and reads only).

INVEST — independent; testable by inspecting the touched modules and the folded-event set; encodes F6.9-INV6.
Priority: must · Size: 1 · Implements deliverables: F6.9-D2.

## F6.9-US8 — Provable from the page itself

As a **developer**, I want one end-to-end acceptance pass over the dashboard, so that the rung's done-ness is provable
from observed behaviour, not assumed.

Acceptance criteria
- Given a fresh mount, when it connects, then `courses_count`/`enrollments_count` show the seeded values from the
  contexts and the feed starts empty.
- Given a `{:course_created, _}` and an `{:enrolled, _}` broadcast, when each is received, then the matching count bumps
  by 1 and a feed row is prepended live (`at: 0`).
- Given a viewer connecting and then disconnecting (across nodes), when each `presence_diff` lands, then `viewers`
  tracks the change cluster-wide; and given the page, when inspected, then it is read-only, requires auth, and the feed
  is a capped stream (no unbounded list).

INVEST — independent; testable as a single seed→fold→presence→read-only/auth acceptance run; encodes F6.9-INV1, INV2,
INV4, INV5, INV6.
Priority: must · Size: 2 · Implements deliverables: F6.9-D7.
