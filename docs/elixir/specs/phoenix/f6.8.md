# F6.8 · Auth & deployment

> Turn the running app into a deployed product with real users. `mix phx.gen.auth` generates the `Accounts` context,
> a `User` and token schema, the login/registration LiveViews, and the session plumbing; protected scopes and
> `on_mount` guard access; `runtime.exs` reads secrets and host from the environment at boot; a `Portal.Release` task
> migrates a release that has no `mix`; and a libcluster topology makes PubSub and Presence span every node. The web
> still calls only the `Portal` facade, and the F5 engine is unchanged.

## [RECONCILE] — folded forward from F6.5 (the component & auth reckoning, `5a440fd`), then F6.6 (`3cf2480`)

This rung is where two F6.5 deferrals come due — made explicit so the build decides, not discovers:

- **Components — `phx.gen.auth` presumes `CoreComponents`; F6.5 has none.** The generated login/registration/reset LiveViews call `<.simple_form>`/`<.input>`/`<.button>`/`<.header>`/`<.flash>` from a `CoreComponents` this app never generated (F6.5-INV8 chose a minimal LOCAL `input/1`). Decide here: adopt the generated `CoreComponents` (superseding INV8 as the UI surface grows) OR port the auth templates to the local `PortalWeb.CatalogComponents` set. *(Why: F6.5 deferred a full component library; the auth UI forces the call.)*
- **Auth — the generated `UserAuth` supersedes the interim `RequireUser`.** `phx.gen.auth` brings `fetch_current_user` → `conn.assigns.current_user` (a `%User{}`) + `require_authenticated_user` + `on_mount`, replacing F6.2's hand-rolled `RequireUser` plug, the `:require_auth` pipeline, and the bare `:current_user_id` assign. Reconcile F6.5's `get "/my/courses"` (`EnrollmentController.index` reads `current_user_id`) to `current_user.id`, the protected pipeline to `require_authenticated_user`, and the unauthenticated redirect from `~p"/"` to the generated login path. *(Why: F6.5's protected `/my/courses` rides the interim auth this rung replaces.)*
- **Context — the generated `Accounts` is Repo-backed.** `phx.gen.auth Accounts User users` extends F6.4's Store-backed `Accounts` (`user/1`/`welcome/1`); reconcile the two surfaces (Store stand-in → Ecto `users`/tokens). *(Why: F6.4 `Accounts` is a Store stand-in.)*
- **Identity — the branded-id mint is unguarded under concurrency (F6.6).** `create_course/1` mints `Portal.ID.new("CRS")` and inserts directly (outside the single-writer engine), and `courses` has `unique_constraint(:title)` but NO `unique_constraint(:id)`; two same-millisecond seq-0 mints from fresh processes RAISE `Ecto.ConstraintError` instead of returning `{:error, changeset}`. Rate ~0 today, genuinely unguarded in production (sibling enroll/controller paths share it). As the id/deploy rung, resolve it: derive `worker_id` per machine (`echo/CLAUDE.md` §4 — `@node = 1` → "F6.8 derives it per machine"), and/or add `unique_constraint(:id)` + graceful handling, and/or route create through the single-writer engine. *(Why: F6.6's ≥100 determinism loop surfaced the collision; a multi-node deploy makes it live.)*
- **The live `/courses` shipped PUBLIC (F6.6).** `live "/courses", CatalogLive` sits in the `:browser` pipeline, unprotected. Decide here whether browsing stays public (likely) and whether the inline create is gated behind `require_authenticated_user`. *(Why: F6.6 added a public read+write surface the auth rung must classify.)*
- **Endpoint bind — `runtime.exs` hardcodes `127.0.0.1` (F6.6 liveness diagnosis).** `config :portal_web, PortalWeb.Endpoint, http: [ip: {127, 0, 0, 1}, port: port]` runs in EVERY non-test env (`config/runtime.exs`), so a deployed release binds loopback-only and the Fly edge proxy cannot reach it. This rung sets the deploy bind to `{0, 0, 0, 0}` (or `:any`) under release/`:prod` while keeping `127.0.0.1` for local dev. *(Why: the F6.6 liveness boot surfaced the loopback bind; deployment makes it load-bearing.)*

## Goal

After F6.8, the Portal has authenticated users and ships as a deployed, clustered release. `mix phx.gen.auth Accounts
User users` generates the `Accounts` context (extending F6.4's), a `User` schema, a token schema, the login,
registration, and reset LiveViews, and the session plumbing. The `:browser` pipeline runs `fetch_current_user`,
protected scopes use `require_authenticated_user`, and LiveViews enforce access with `on_mount` before they connect.
`config/runtime.exs` reads `DATABASE_URL`, `SECRET_KEY_BASE`, and `PHX_HOST` from the environment at boot, distinct
from compile-time `prod.exs`. A `Portal.Release` module migrates the database via `bin/portal eval`, since a release
has no `mix`. A libcluster topology lets `Phoenix.PubSub` and `PortalWeb.Presence` span every node. The deploy is build
→ migrate → boot, with the F5 supervision tree serving — and the web still calls only the `Portal` facade.

## Rationale (5W)

- **Why** — A platform with no users and no deploy story is a demo. Authentication gives protected areas and a real
  identity; a runtime/compile config split and a release make it deployable without `mix` on the server; clustering
  makes the F6.7 real-time features correct across nodes. These are the steps from "runs on my machine" to "serves
  real people."
- **What** — generated authentication (the `Accounts` context, `User`/token schemas, auth LiveViews), protected routes
  and `on_mount` enforcement, a config split by phase, a release migrate command, a libcluster topology, and the deploy
  sequence.
- **Who** — Learners (real accounts and protected areas) and Operators (a deployed, clustered, migratable release).
- **When** — the eighth Phoenix rung; it builds on F6.4's `Accounts` context and the F6.7 real-time features, and it is
  the auth and deploy substrate the F6.9 dashboard runs under.
- **Where** — the generated `Accounts` context plus `User`/token schemas and auth modules, `lib/portal_web/router.ex`,
  `config/runtime.exs` and `config/prod.exs`, `lib/portal/release.ex`, `Portal.Application` (Presence + libcluster
  children), and the deploy config.

## Scope

- **In** — generated authentication via `phx.gen.auth` (the `Accounts` context, `User`/token schemas, login/
  registration/reset LiveViews, session plumbing); `fetch_current_user` in `:browser`, `require_authenticated_user` on
  protected scopes, and `on_mount` for LiveViews; the runtime-vs-compile config split (`DATABASE_URL`,
  `SECRET_KEY_BASE`, `PHX_HOST` from env at boot); a `Portal.Release` migrate command via `bin/portal eval`; a
  libcluster topology for PubSub/Presence across nodes; the build → migrate → boot deploy sequence; and the facade
  fronting the auth queries the web needs.
- **Out** — the dashboard (→ F6.9); new domain features; a specific cloud provider's infrastructure-as-code; email
  delivery configuration beyond the generated flow; and role-based authorization beyond authenticated/anonymous.

## Deliverables

- **F6.8-D1** — authentication generated by `mix phx.gen.auth Accounts User users`: the `Accounts` context (extending
  F6.4's), a `User` schema, a token schema, the login/registration/reset LiveViews, and the session plumbing; the
  `Accounts` `Repo` calls and schemas are private to the context.
- **F6.8-D2** — protected routes: `fetch_current_user` added to the `:browser` pipeline (loading `current_user` into
  `conn.assigns` per request), `require_authenticated_user` on protected scopes, and `on_mount` enforcement for
  LiveViews before they connect.
- **F6.8-D3** — the facade fronts auth: `Portal` delegates the auth queries the web needs (for example
  `get_user_by_session_token/1` and `register_user/1`), so `PortalWeb.UserAuth` and the auth LiveViews call only
  `Portal`, preserving the master invariant.
- **F6.8-D4** — the config split: `config/runtime.exs` reads `DATABASE_URL`, `SECRET_KEY_BASE`, and `PHX_HOST` from the
  environment at boot; compile-time `config.exs`/`prod.exs` hold only fixed, committable settings.
- **F6.8-D5** — a release migrate command: a `Portal.Release` module with `migrate/0` (and `rollback/2`) run via
  `bin/portal eval "Portal.Release.migrate()"`, since a release has no `mix`.
- **F6.8-D6** — clustering: a libcluster topology in the supervision tree so `Phoenix.PubSub` and `PortalWeb.Presence`
  span every node, making broadcasts and viewer counts cluster-wide.
- **F6.8-D7** — the deploy sequence: build (`MIX_ENV=prod mix release`), migrate (`bin/portal eval`), boot
  (`bin/portal start`), with the F5 supervision tree serving.
- **F6.8-D8** — verification: registration and login work; a protected route redirects an anonymous user and admits an
  authenticated one; a LiveView enforces auth on connect; the release builds, migrates, and boots from env config;
  PubSub/Presence are correct across two nodes; and the web still calls only the facade.

## Invariants

- **F6.8-INV1 (auth over the facade — the master invariant)** — `PortalWeb.UserAuth`, the auth LiveViews, and
  protected scopes call only `Portal`; the `Accounts` context owns the `User`/token schemas and `Repo` calls privately;
  no web module names `Portal.Engine`, a repo, or `GenServer.call`.
- **F6.8-INV2 (protected scopes require a user)** — protected scopes require an authenticated user via
  `require_authenticated_user` (controllers) and `on_mount` (LiveViews); an anonymous request to a protected route is
  redirected, never served.
- **F6.8-INV3 (config by phase)** — secrets and host (`DATABASE_URL`, `SECRET_KEY_BASE`, `PHX_HOST`) are read at boot
  from `runtime.exs`; compile-time config holds no secret, and a missing required env var fails the boot loudly.
- **F6.8-INV4 (release-native migration)** — migrations run through `Portal.Release` via `bin/portal eval`, not `mix`
  (the release has none); the schema is migrated before the app serves.
- **F6.8-INV5 (cluster-correct real-time)** — with the libcluster topology, PubSub broadcasts and Presence counts are
  correct across all nodes; a single-node assumption is not baked in.
- **F6.8-INV6 (engine unchanged)** — the F5 engine, store, and facade are unchanged; F6.8 adds auth, config, a release
  task, and clustering as web and ops concerns, not domain changes.

## Definition of Done

- [ ] `phx.gen.auth` authentication works: register, log in, log out; `Accounts` owns `User`/token schemas privately
      (F6.8-D1, INV1).
- [ ] A protected route redirects an anonymous user; an authenticated user is admitted; LiveViews enforce via
      `on_mount` (F6.8-INV2).
- [ ] `PortalWeb.UserAuth` and the auth LiveViews call only `Portal` (F6.8-INV1).
- [ ] `runtime.exs` reads `DATABASE_URL`/`SECRET_KEY_BASE`/`PHX_HOST` at boot; a missing one fails loudly (F6.8-INV3).
- [ ] `Portal.Release.migrate/0` runs via `bin/portal eval` and migrates before serving (F6.8-INV4).
- [ ] PubSub and Presence are correct across two clustered nodes (F6.8-INV5).
- [ ] The F5 engine, store, and facade are unchanged (F6.8-INV6); every deliverable maps to a story (see
      `./f6.8.stories.md`).
