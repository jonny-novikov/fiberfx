# F6.7 · Real-time (PubSub & Presence)

> Push F6.6's per-socket state across every connected client. A facade `subscribe`/`broadcast` wrapper over
> `Portal.PubSub` (supervised); context broadcasts that fire only after a successful write; `CatalogLive` subscribing on
> its connected mount and applying `stream_insert/3` in `handle_info/2`; and `Presence` for a viewer count correct
> across the cluster. The only new supervision children are `Phoenix.PubSub` and `PortalWeb.Presence`; everything below
> the facade is unchanged.

## [RECONCILE] — folded forward from F6.5, then F6.6 (`3cf2480`)

- **Surface — `update_course/2` does not exist.** D2/Scope broadcast from `create_course/1` AND `update_course/2`, but F6.4/F6.5 ship no edit path (`Catalog` has list/get/fetch/create/change; F6.6 added only the read-only `search_courses/1`). Define `update_course/2` (+ its `Portal` facade delegate) when this rung builds, or drop the `:course_updated` broadcast until an edit flow exists. This is the one new **write** surface the rung adds below the facade — Venus pins it, the Director ratifies. *(Why: no update path is built yet.)*
- **Route inheritance — RESOLVED by F6.6.** `CatalogLive` is live at `live "/courses"` (F6.6 shipped `3cf2480`, superseding the static index); this rung references it by module and adds no route. *(Was the F6.6 route reconcile; now shipped — no longer open.)*
- **The `connected?/1` seam is built and waiting (F6.6).** F6.6's two-stage mount left `connected?/1` guarding a no-op `:connected?` marker *explicitly as the F6.7 seam*; D3's `Portal.subscribe("courses")` lands in exactly that branch — extend it, do not restructure the mount. *(Why: F6.6 built the seam for this rung.)*
- **Live search × broadcast insert — a new interaction F6.6 created.** F6.6's search re-streams on `@query` with `reset: true`; F6.7's `handle_info` `stream_insert` ignores `@query`, so a broadcast can insert a row outside an active filter. Decide at build: gate the insert on a `@query` match, or accept the interaction and record it. *(Why: F6.6 shipped a live filter the broadcast path must reckon with.)*
- **Styling — F6.5.5 did NOT style the live catalog; F6.7 owns CONSISTENCY, not inheritance.** F6.7's `stream_insert` prepends (create) / replaces (update) rows rendered by `PortalWeb.CatalogComponents.course_card/1`. [F6.5.5](f6.5.5.md) was reframed to ship **two STATIC parity pages** (`/` = `html/courses.html`, `/elixir` = `elixir/index.html`) via per-page **extracted CSS** — it added NO root layout, NO shared `app.css`, and did NOT restyle `CatalogComponents`, so `course_card` and the F6.6 `live "/courses"` `CatalogLive` it renders in are **still unstyled** after F6.5.5. The invariant F6.7 owns is therefore *consistency*: a broadcast-inserted row must render through the SAME `course_card/1` the mount uses, so it matches the mounted list — styled if a prior pass has styled the component, bare if not. Do not emit a divergent row in `handle_info/2`. The "style the live catalog" work (restyle `CatalogComponents` + a shared token stylesheet / root layout for the **dynamic** UI) is unbuilt after F6.5.5 and lands at the first dynamic styled page (F6.8, or an earlier dedicated pass) — F6.7 does not depend on it, only on rendering the broadcast row identically to the mount. *(Why: F6.5.5 styled static documents, not the LiveView component F6.7 streams through; the broadcast path must match the mount, whatever its styled state.)*

## Goal

After F6.7, the catalog updates live across every connected client. The `Portal` facade gains a PubSub wrapper —
`subscribe/1` and `broadcast/2` over `Portal.PubSub`, started in the supervision tree. The `Catalog` context broadcasts
`{:course_created, course}` and `{:course_updated, course}` after a successful write, via a `broadcast/2` helper that
fires only on `{:ok, _}`. `CatalogLive` subscribes to `"courses"` on its connected mount and handles those events in
`handle_info/2`, calling `stream_insert/3` (prepend on create, replace on update), so every connected client's list
updates without a reload. `Presence` (`track/3` on the connected mount plus a `"presence_diff"` handler) recomputes a
live viewer count correct across the cluster. Optionally, a minimal channel (`join/3` + `handle_in/3`) serves a
non-LiveView client. Everything still calls only the facade; the only new tree children are `Phoenix.PubSub` and
`PortalWeb.Presence`.

## Rationale (5W)

- **Why** — PubSub is how the BEAM turns one user's write into every user's update: a context broadcasts a fact after
  it commits, and each subscribed LiveView patches its own DOM — no polling, correct across a cluster. Presence adds an
  accurate, distributed count of who is here. Broadcasting only on `{:ok, _}` keeps the stream of facts honest —
  clients learn of a change only after it happened. Without PubSub, multi-client freshness needs polling or a bespoke
  socket protocol.
- **What** — the facade `subscribe`/`broadcast` wrapper (supervised), context broadcasts on successful writes, a
  subscribed LiveView applying stream updates, `Presence` for a live viewer count, and an optional channel.
- **Who** — Learners (a catalog that updates as others change it) and Instructors (a live viewer count and immediate
  propagation of edits).
- **When** — the seventh Phoenix rung; it pushes F6.6's per-socket state across all clients over the same facade, and
  it is the real-time substrate the F6.9 dashboard folds.
- **Where** — the `Portal` facade and `Portal.Catalog` (broadcast helper), `Portal.Application` (`Phoenix.PubSub` +
  `PortalWeb.Presence` children), `lib/portal_web/live/catalog_live.ex`, `lib/portal_web/presence.ex`, and optionally a
  channel module plus a `UserSocket`.

## Scope

- **In** — `Portal.subscribe/1` and `Portal.broadcast/2` over `Portal.PubSub` (supervised); broadcasts from
  `create_course/1` and `update_course/2` firing only on `{:ok, _}`; `CatalogLive` subscribing to `"courses"` on the
  connected mount and handling events in `handle_info/2` into `stream_insert/3` (prepend on create, replace on update);
  `Presence` `track/3` + `presence_diff` recomputing a live viewer count across the cluster; and optionally a minimal
  channel (`join/3` + `handle_in/3`) for a non-LiveView client.
- **Out** — authentication and registration (→ F6.8); the dashboard (→ F6.9); cluster setup and the deployed release
  (→ F6.8); any new domain feature; and changes below the facade.

## Deliverables

- **F6.7-D1** — the facade PubSub wrapper: `Portal.subscribe(topic)` calling `Phoenix.PubSub.subscribe(Portal.PubSub,
  topic)` and `Portal.broadcast(topic, message)` calling `Phoenix.PubSub.broadcast(Portal.PubSub, topic, message)`;
  `{Phoenix.PubSub, name: Portal.PubSub}` added to `Portal.Application`.
- **F6.7-D2** — broadcasts from the context: `create_course/1` and `update_course/2` emit `{:course_created, course}` /
  `{:course_updated, course}` after a successful write, via a `broadcast/2` helper that fires only on `{:ok, _}` (a
  failed write broadcasts nothing).
- **F6.7-D3** — a subscribed LiveView: `CatalogLive` subscribes to `"courses"` on its connected mount (behind
  `connected?/1`) and handles `{:course_created, _}` / `{:course_updated, _}` in `handle_info/2`.
- **F6.7-D4** — stream updates from broadcasts: `handle_info/2` calls `stream_insert(socket, :courses, course, at: 0)`
  on create and `stream_insert(socket, :courses, course)` on update, so every connected client's list updates without
  a reload.
- **F6.7-D5** — `Presence`: `PortalWeb.Presence` (`use Phoenix.Presence`), `track/3` on the connected mount and a
  `%{event: "presence_diff"}` handler recomputing a `viewers` count correct across the cluster; the `Presence` child is
  added to the tree.
- **F6.7-D6** — (optional) a minimal channel: a `join/3` + `handle_in/3` module (and a `UserSocket` entry) for a
  non-LiveView client.
- **F6.7-D7** — verification: a create in one session appears in another without reload; an update replaces the row in
  every client; the viewer count reflects connect and disconnect across nodes; a failed write broadcasts nothing; the
  only new tree children are PubSub and Presence; and the LiveView still calls only the facade.

## Invariants

- **F6.7-INV1 (broadcast only on success)** — a broadcast fires only after a successful write (`{:ok, _}`); a failed
  write broadcasts nothing, so clients only ever learn of facts.
- **F6.7-INV2 (real-time over the facade — the master invariant)** — `subscribe`/`broadcast` go through `Portal` over
  `Portal.PubSub`; the LiveView and context name no transport beyond the facade wrapper, and nothing names the engine,
  a `Repo`, or `GenServer.call`.
- **F6.7-INV3 (per-client patching)** — each subscribed client patches its own DOM from the broadcast via
  `stream_insert`; no client holds the full list, and no reload occurs.
- **F6.7-INV4 (cluster-correct presence)** — `Presence` `track` + `presence_diff` yield a viewer count correct across
  the cluster (CRDT-backed), not a single-node counter.
- **F6.7-INV5 (minimal new surface)** — the only new supervision children are `Phoenix.PubSub` and `PortalWeb.Presence`;
  everything below the facade is unchanged (the F5→F6 handoff invariant holds).
- **F6.7-INV6 (idempotent stream identity)** — `stream_insert` keys on the course id, so a create prepends once and an
  update replaces in place; a re-delivered broadcast does not duplicate a row.
