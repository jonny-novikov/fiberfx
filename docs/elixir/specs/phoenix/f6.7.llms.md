## Requirements

- **F6.7-R1** — `Portal.subscribe/1` and `Portal.broadcast/2` wrap `Portal.PubSub`; `{Phoenix.PubSub, name:
  Portal.PubSub}` is supervised. [US: F6.7-US1, F6.7-US5]
- **F6.7-R2** — `create_course/1` and `update_course/2` broadcast `{:course_created, _}` / `{:course_updated, _}` via a
  helper that fires only on `{:ok, _}`. [US: F6.7-US2]
- **F6.7-R3** — `CatalogLive` subscribes to `"courses"` on the connected mount and handles the events in
  `handle_info/2`. [US: F6.7-US1]
- **F6.7-R4** — `handle_info/2` applies `stream_insert(:courses, course, at: 0)` on create and `stream_insert(:courses,
  course)` on update, keyed by id. [US: F6.7-US3]
- **F6.7-R5** — `PortalWeb.Presence` tracks on the connected mount; a `presence_diff` handler recomputes a cluster-correct
  `viewers` count. [US: F6.7-US4]
- **F6.7-R6** — (optional) a minimal channel (`join/3`, `handle_in/3`) plus a `UserSocket`. [US: F6.7-US6]
- **F6.7-R7** — real-time goes through the facade; the only new tree children are PubSub and Presence; nothing names
  the engine, a `Repo`, or `GenServer.call`. [US: F6.7-US5]

## Execution topology

```text
The fan-out (one write → every client):

   create_course/1 ─{:ok, course}─▶ broadcast("courses", {:course_created, course})   # ok-only
                                            │  (Portal.broadcast over Portal.PubSub)
              ┌─────────────────────────────┼─────────────────────────────┐
              ▼                             ▼                             ▼
        client A handle_info        client B handle_info          client C handle_info
        stream_insert(at: 0)        stream_insert(at: 0)          stream_insert(at: 0)
        └─ patches own DOM          └─ patches own DOM            └─ patches own DOM     (no reload, no list in memory)

Presence:
   connected mount ─▶ Presence.track(self(), "courses", id, %{}) ─▶ "presence_diff" ─▶ recompute viewers (cluster-correct)
```

```text
Task DAG:
  T1 facade wrapper + supervise PubSub ─▶ T2 context broadcast (ok-only) ─▶ T3 LV subscribe (connected) ─▶ T4 handle_info stream updates ─▶ T5 Presence(track + diff) ─▶ T6 (optional) channel ─▶ T7 verify
```

Touched files: the `Portal` facade, `Portal.Catalog` (broadcast helper), `Portal.Application` (PubSub +
`PortalWeb.Presence` children), `lib/portal_web/live/catalog_live.ex`, `lib/portal_web/presence.ex`, and optionally a
channel module plus `UserSocket`.

## Agent stories

- **F6.7-AS1** [implements F6.7-US5, F6.7-US1] — Directive: add `Portal.subscribe/1` and `Portal.broadcast/2` over
  `Portal.PubSub`, and add `{Phoenix.PubSub, name: Portal.PubSub}` to the tree. Acceptance gate: a subscriber receives
  a manually broadcast message; the only new child is PubSub.
- **F6.7-AS2** [implements F6.7-US2] — Directive: add a `broadcast/2` helper in `Portal.Catalog` and call it from
  `create_course/1` and `update_course/2` only on `{:ok, _}`. Acceptance gate: a successful write broadcasts; a failed
  write broadcasts nothing.
- **F6.7-AS3** [implements F6.7-US1, F6.7-US3] — Directive: subscribe `CatalogLive` to `"courses"` on the connected
  mount and handle the events in `handle_info/2` with `stream_insert/3` (prepend on create, replace on update).
  Acceptance gate: a create in one LiveView process appears in another; an update replaces the row without a
  duplicate.
- **F6.7-AS4** [implements F6.7-US4] — Directive: add `PortalWeb.Presence`, `track/3` on the connected mount, and a
  `presence_diff` handler recomputing `viewers`. Acceptance gate: connecting and disconnecting changes the count;
  correct across nodes.

## Execution plan — first two stories

1. **F6.7-AS1 — the facade wrapper and supervision.** Add to `lib/portal.ex`: `def subscribe(topic), do:
   Phoenix.PubSub.subscribe(Portal.PubSub, topic)` and `def broadcast(topic, message), do:
   Phoenix.PubSub.broadcast(Portal.PubSub, topic, message)`. Add `{Phoenix.PubSub, name: Portal.PubSub}` to
   `Portal.Application`. Confirm a subscriber receives a broadcast message.
2. **F6.7-AS2 — context broadcasts on success.** In `Portal.Catalog`, add a private `broadcast(topic, message)` helper
   and wrap the write results so `create_course/1` emits `{:course_created, course}` and `update_course/2` emits
   `{:course_updated, course}` only on `{:ok, _}`. Confirm a failing changeset broadcasts nothing.

## Comprehensive implementation prompt

```text
You are adding real-time to the catalog (rung F6.7) on top of F6.6. Read specs/phoenix/f6.7.md, the F6.6 CatalogLive
(streams, two-stage mount), and the F6.4 Catalog facade. Everything still calls only the facade; the only new
supervision children are Phoenix.PubSub and PortalWeb.Presence.

Build real-time:

1) Facade wrapper (T1 -> R1). In lib/portal.ex add:
     def subscribe(topic),            do: Phoenix.PubSub.subscribe(Portal.PubSub, topic)
     def broadcast(topic, message),   do: Phoenix.PubSub.broadcast(Portal.PubSub, topic, message)
   Add `{Phoenix.PubSub, name: Portal.PubSub}` to Portal.Application's children.

2) Context broadcasts (T2 -> R2). In Portal.Catalog add a `broadcast/2` helper and call it from create_course/1 and
   update_course/2 ONLY on {:ok, _}: emit {:course_created, course} and {:course_updated, course} respectively. A
   failed write (an {:error, changeset}) broadcasts nothing.

3) LiveView subscribe (T3 -> R3). In CatalogLive, on the connected mount (behind connected?/1) call
   Portal.subscribe("courses").

4) handle_info stream updates (T4 -> R4). Add handle_info/2:
     {:course_created, course} -> {:noreply, stream_insert(socket, :courses, course, at: 0)}
     {:course_updated, course} -> {:noreply, stream_insert(socket, :courses, course)}
   stream_insert keys on the course id, so a create prepends once and an update replaces in place; a re-delivered
   broadcast does not duplicate a row.

5) Presence (T5 -> R5). Create lib/portal_web/presence.ex (`use Phoenix.Presence`), add its child to the tree, call
   Presence.track(self(), "courses", id, %{}) on the connected mount, and handle %{event: "presence_diff"} in
   handle_info to recompute a `viewers` count. The count must be correct across the cluster (CRDT-backed).

6) Channel (T6 -> R6, optional). Add a minimal channel (join/3, handle_in/3) and a UserSocket entry for a
   non-LiveView client.

7) Verify (T7). A create in one session appears in another without reload; an update replaces the row everywhere; the
   viewer count reflects connect/disconnect across nodes; a failed write broadcasts nothing; the only new tree
   children are PubSub and Presence; the LiveView still calls only the facade.

Constraints: broadcast only on {:ok, _}; subscribe/broadcast go through Portal over Portal.PubSub; nothing names the
engine, a Repo, or GenServer.call. Out of scope: auth (F6.8), the dashboard (F6.9), cluster setup/deploy (F6.8).
```
