# F6.2 · user stories

> Who needs the routing surface, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.2.md`](f6.2.md).

## F6.2-US1 — Browse public pages without signing in

As a **visitor**, I want to reach public pages without an account, so that I can explore the Portal before committing.

Acceptance criteria
- Given no session, when I request `GET /` (the landing), then it renders.
- Given no session, when I request `GET /courses/:user_id` or `GET /lessons/:id`, then it renders.
- Given any public route, when it renders, then no sign-in is required and no redirect occurs.

INVEST — independent; small; testable with unauthenticated requests to the public routes.
Priority: must · Size: 2 · Implements deliverables: F6.2-D1, F6.2-D2.

## F6.2-US2 — Protect pages behind authentication

As a **learner**, I want protected pages to require me to be signed in, so that my learning area is not open to
anyone.

Acceptance criteria
- Given no `:user_id` in the session, when I request a protected route, then I am redirected to the public landing
  with a flash, and the protected action does not run.
- Given a `:user_id` in the session, when I request the same route, then it renders and `:current_user_id` is
  assigned.

INVEST — valuable (the access boundary); small; testable by toggling the session and checking redirect vs render.
Priority: must · Size: 3 · Implements deliverables: F6.2-D3, F6.2-D4.

## F6.2-US3 — Declare cross-cutting concerns once

As a **developer**, I want session, CSRF, headers, and auth declared in pipelines, so that I never copy that logic
into individual actions.

Acceptance criteria
- Given the pipelines, when I add a protected route, then I only `pipe_through [:browser, :require_auth]`.
- Given any action, when I read it, then it contains no session/CSRF/header/auth code.
- Given the protected scope, when it is defined, then `:browser` is piped before `:require_auth`.

INVEST — independent; testable by inspecting the router and actions; encodes F6.2-INV2 and F6.2-INV5.
Priority: must · Size: 2 · Implements deliverables: F6.2-D2, F6.2-D3, F6.2-D5.

## F6.2-US4 — Compile-time-safe links

As a **developer**, I want every internal URL verified at compile time, so that a renamed or mistyped route is a build
error, not a broken link in production.

Acceptance criteria
- Given a link or redirect, when I build a URL, then I use a verified `~p` path.
- Given a deliberately wrong `~p` path, when I compile, then the build fails; when I fix it, the build passes.

INVEST — small; testable by introducing and reverting a bad `~p`; encodes F6.2-INV4.
Priority: should · Size: 1 · Implements deliverables: F6.2-D6.

## F6.2-US5 — A JSON surface for clients

As an **operator**, I want a JSON pipeline for programmatic clients, so that non-browser callers get content
negotiated as JSON.

Acceptance criteria
- Given the `:api` pipeline, when a client requests an `:api` route with a JSON accept header, then the response is
  negotiated as JSON.
- Given the `:api` pipeline, when it is defined, then it uses `accepts ["json"]` and does not carry browser-only plugs.

INVEST — independent of the browser routes; small; testable with a JSON request.
Priority: could · Size: 1 · Implements deliverables: F6.2-D2.

## F6.2-US6 — Reach write and live routes

As a **learner**, I want enroll actions and a live enroll page to be reachable, so that the platform can act on my
intent (even as their targets are completed in later rungs).

Acceptance criteria
- Given the router, when I `POST /enroll`, then the request reaches `EnrollmentController.create` (which will issue a
  facade command).
- Given the router, when I visit `GET /enroll/:id`, then `EnrollmentLive` mounts over the endpoint's LiveView socket.
- Given `resources "/lessons", only: [:show]`, when I `GET /lessons/:id`, then it reaches `LessonController.show` and
  no other RESTful action is generated.

INVEST — independent of the targets' completion; testable by route resolution; the write/live bodies are thin here.
Priority: should · Size: 2 · Implements deliverables: F6.2-D1.
