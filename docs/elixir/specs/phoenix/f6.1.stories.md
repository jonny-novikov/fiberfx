# F6.1 · user stories

> Who wants the bootstrap, what they need, and how we will know it works. Connextra stories with Given/When/Then
> acceptance criteria; together they cover every deliverable in [`f6.1.md`](f6.1.md).

## F6.1-US1 — Serve the Portal as a Phoenix app

As an **operator**, I want the Portal to boot as a Phoenix application, so that I can run and serve it with the
standard Elixir/Phoenix toolchain.

Acceptance criteria
- Given the application, when it starts, then `PortalWeb.Endpoint` listens for HTTP and the supervision tree is
  running.
- Given the running app, when I request `GET /health`, then I receive `200` with body `ok`.
- Given the boot sequence, when children start, then the store and engine start before the endpoint.

INVEST — independent of page content; testable by a boot check and a health request; encodes F6.1-INV2, INV3.
Priority: must · Size: 3 · Implements deliverables: F6.1-D1, F6.1-D2, F6.1-D6.

## F6.1-US2 — See a user's courses

As a **visitor**, I want to open a course page for a given user, so that I can see the courses that user is enrolled
in.

Acceptance criteria
- Given a known user id with enrollments, when I request `GET /courses/:user_id`, then the page renders that user's
  courses.
- Given a known user id with no enrollments, when I request the page, then it renders an empty state (not an error).
- Given the page, when it renders, then the markup comes only from `assigns` (no engine call in the template).

INVEST — valuable on its own (the first real page); testable by rendered output for the three cases;
encodes F6.1-INV1.
Priority: must · Size: 3 · Implements deliverables: F6.1-D3, F6.1-D4, F6.1-D5.

## F6.1-US3 — Keep the domain core untouched

As a **developer**, I want the F5 engine to run unchanged inside the Phoenix app, so that introducing the web layer
does not risk the domain core.

Acceptance criteria
- Given the supervision tree, when the app boots, then `Portal.Application` keeps the `Portal.Store`, event-store
  adapter, and engine entries unchanged and drops the F5 `Bandit` child (four → three), and the new
  `PortalWeb.Application` owns `[PortalWeb.Telemetry, PortalWeb.Endpoint]`; nothing below the `Portal` facade changes.
- Given the web layer, when I search `apps/portal_web/lib/`, then no module references `Portal.Engine`, a repo, or
  `GenServer.call`.
- Given a controller action, when it needs domain data, then it calls only the `Portal` facade.

INVEST — independent; testable by a tree assertion and a static search; encodes F6.1-INV1, INV2.
Priority: must · Size: 2 · Implements deliverables: F6.1-D2, F6.1-D4.

## F6.1-US4 — A self-healing front door

As an **operator**, I want the web endpoint to recover on its own, so that a transient crash does not take the
platform down.

Acceptance criteria
- Given the running app, when the `PortalWeb.Endpoint` process is killed, then the supervisor restarts it.
- Given the restart, when I issue a request afterwards, then it succeeds.

INVEST — independent; testable by killing the process and re-requesting; encodes F6.1-INV2.
Priority: should · Size: 2 · Implements deliverables: F6.1-D2, F6.1-D7.

## F6.1-US5 — A friendly error, not a stack trace

As a **visitor**, I want an invalid request to show a clear error page, so that I am not shown a server error or a
stack trace for an expected failure.

Acceptance criteria
- Given an unknown or malformed user id, when I request `GET /courses/:user_id`, then the total facade
  (`courses_of/1`, success-only at F6.1) returns no rows and the page renders the empty state — a clean `200`, never a
  `500` or a stack trace.
- Given the error-render path itself, when an injected `%Portal.Error{}` reaches the controller's error branch, then it
  renders a `422` page whose message comes from the closed `%Portal.Error{}` set (unit-verified). That branch becomes
  live-reachable from a request when the facade gains id-validation (a later F6 rung), at which point a malformed id
  produces a `%Portal.Error{}` and the same path renders the `422`.

INVEST — valuable for trust; testable today by the unknown-id empty-state case and the injected-error render;
encodes F6.1-INV4, INV5.
Priority: should · Size: 2 · Implements deliverables: F6.1-D4, F6.1-D7.
