# F6.8 · agent brief (llms)

## References

- `mix phx.gen.auth` — <https://hexdocs.pm/phoenix/mix_phx_gen_auth.html>. Generates the `Accounts` context, `User`
  and token schemas, auth LiveViews, and `PortalWeb.UserAuth`.
- Phoenix auth plugs and `on_mount` — <https://hexdocs.pm/phoenix/Phoenix.Router.html> and
  <https://hexdocs.pm/phoenix_live_view/Phoenix.LiveView.html#on_mount/1>. `fetch_current_user`,
  `require_authenticated_user`, and LiveView enforcement.
- Elixir releases and `Mix.Release` — <https://hexdocs.pm/mix/Mix.Tasks.Release.html>. Building and running a release
  with no `mix` on the host.
- Runtime configuration (`config/runtime.exs`) — <https://hexdocs.pm/elixir/Config.html>. Reading env at boot.
- libcluster — <https://hexdocs.pm/libcluster/readme.html>. A clustering topology for PubSub and Presence.
- Upstream: the `Accounts` context and facade [`f6.4.md`](f6.4.md), the PubSub/Presence real-time
  [`f6.7.md`](f6.7.md), and the F5 engine handoff [`../pragmatic/f5.9.md`](../pragmatic/f5.9.md).

## Requirements

- **F6.8-R1** — `mix phx.gen.auth Accounts User users` generates the `Accounts` context, `User`/token schemas, auth
  LiveViews, and session plumbing; `Accounts` `Repo` calls and schemas are private. [US: F6.8-US1]
- **F6.8-R2** — `fetch_current_user` runs in `:browser`; protected scopes use `require_authenticated_user`; LiveViews
  enforce via `on_mount`. [US: F6.8-US2]
- **F6.8-R3** — `Portal` delegates the auth queries the web needs (`get_user_by_session_token/1`, `register_user/1`,
  …); `PortalWeb.UserAuth` and the auth LiveViews call only `Portal`. [US: F6.8-US1, F6.8-US6]
- **F6.8-R4** — `config/runtime.exs` reads `DATABASE_URL`, `SECRET_KEY_BASE`, `PHX_HOST` at boot; compile-time config
  holds no secret; a missing required var fails the boot. [US: F6.8-US3]
- **F6.8-R5** — `Portal.Release` provides `migrate/0` (and `rollback/2`) run via `bin/portal eval`. [US: F6.8-US4]
- **F6.8-R6** — a libcluster topology in the tree makes PubSub and Presence cluster-wide. [US: F6.8-US5]
- **F6.8-R7** — the deploy is `MIX_ENV=prod mix release` → `bin/portal eval "Portal.Release.migrate()"` →
  `bin/portal start`; the F5 engine, store, and facade are unchanged. [US: F6.8-US4, F6.8-US6]

## Execution topology

```text
Auth request flow (every request loads or requires a user):

   request ─▶ :browser pipeline ─▶ fetch_current_user (Portal.get_user_by_session_token/1) ─▶ conn.assigns.current_user
                                          │
              ┌───────────────────────────┴───────────────────────────┐
              ▼                                                         ▼
   public scope (served)                            protected scope: require_authenticated_user (controllers)
                                                    protected LiveView: on_mount {PortalWeb.UserAuth, :ensure_authenticated}
                                                          └─ anonymous ─▶ redirect to login

Deploy sequence:
   MIX_ENV=prod mix release  ─▶  bin/portal eval "Portal.Release.migrate()"  ─▶  bin/portal start
                                          (runtime.exs reads DATABASE_URL / SECRET_KEY_BASE / PHX_HOST)

Tree additions: {Cluster.Supervisor, libcluster topology} so Phoenix.PubSub + PortalWeb.Presence span nodes.
```

```text
Task DAG (build order; each step leaves the app compiling):

  T1 mix phx.gen.auth Accounts User users ─▶ T2 facade delegations (auth queries) ─▶ T3 protected routes + on_mount
        └─▶ T4 runtime.exs (env at boot) ─▶ T5 Portal.Release (migrate/rollback) ─▶ T6 libcluster topology ─▶ T7 deploy + verify
```

Touched files: the generated `lib/portal/accounts.ex` + `lib/portal/accounts/user.ex` + token schema +
`lib/portal_web/user_auth.ex` + auth LiveViews, `lib/portal.ex` (facade delegations), `lib/portal_web/router.ex`,
`config/runtime.exs` and `config/prod.exs`, `lib/portal/release.ex`, `lib/portal/application.ex` (libcluster child),
and `mix.exs` (`:libcluster`, release config).

## Agent stories

- **F6.8-AS1** [implements F6.8-US1] — Directive: run `mix phx.gen.auth Accounts User users`, migrate, and confirm
  register/login/logout; keep `Accounts` `Repo`/schemas private. Acceptance gate: the auth flow works end to end.
- **F6.8-AS2** [implements F6.8-US1, F6.8-US6] — Directive: add facade delegations for the auth queries the web needs
  and re-point `PortalWeb.UserAuth` and the auth LiveViews at `Portal`. Acceptance gate: no web module names a context,
  the engine, a repo, or `GenServer.call`.
- **F6.8-AS3** [implements F6.8-US2] — Directive: add `fetch_current_user` to `:browser`, `require_authenticated_user`
  on protected scopes, and `on_mount` on protected LiveViews. Acceptance gate: anonymous is redirected; authenticated
  is admitted; the LiveView enforces on connect.
- **F6.8-AS4** [implements F6.8-US3, F6.8-US4, F6.8-US5] — Directive: write `config/runtime.exs` (env at boot),
  `Portal.Release` (`migrate/0`, `rollback/2`), and a libcluster topology in the tree; document the deploy sequence.
  Acceptance gate: the release builds, migrates via `bin/portal eval`, boots from env, and clusters so PubSub/Presence
  span nodes.

## Execution plan — first two stories

1. **F6.8-AS1 — generate authentication.** Run `mix phx.gen.auth Accounts User users`, then `mix ecto.migrate`. This
   generates `Accounts`, the `User` and token schemas, the auth LiveViews, `PortalWeb.UserAuth`, and the session
   plumbing. Confirm registration, login, and logout work in the browser; confirm `Accounts` keeps its schemas and
   `Repo` calls private.
2. **F6.8-AS2 — front auth with the facade.** Add `defdelegate` entries in `lib/portal.ex` for the auth queries the web
   touches (`get_user_by_session_token/1`, `register_user/1`, and the few `UserAuth` needs), then change
   `PortalWeb.UserAuth` and the auth LiveViews to call `Portal` rather than `Accounts` directly. Run `grep -rE
   "Portal\.Engine|Repo|GenServer\.call|Portal\.Accounts" lib/portal_web/` and confirm the web names only `Portal`.

## Comprehensive implementation prompt

```text
You are implementing spec F6.8 (Auth & deployment) in the existing `portal` Phoenix app. Read specs/phoenix/f6.8.md,
the F6.4 Accounts/facade, and the F6.7 PubSub/Presence. Do not change the F5 engine, store, or the Portal facade's
domain functions. Honor the master invariant: the web calls only Portal and names no engine, repo, or GenServer.call.

Build in this order, keeping the app compiling:

1) Authentication (T1 -> R1). Run `mix phx.gen.auth Accounts User users` and `mix ecto.migrate`. This generates the
   Accounts context, User and token schemas, login/registration/reset LiveViews, PortalWeb.UserAuth, and the session
   plumbing. Keep Accounts' Repo calls and schemas private to the context.

2) Facade delegations (T2 -> R3). In lib/portal.ex add `defdelegate` entries for the auth queries the web needs
   (get_user_by_session_token/1, register_user/1, and the others PortalWeb.UserAuth calls). Re-point PortalWeb.UserAuth
   and the auth LiveViews to call Portal instead of Accounts directly, so the web calls only the facade.

3) Protected routes (T3 -> R2). Add plug :fetch_current_user to the :browser pipeline. Put protected controller routes
   behind require_authenticated_user, and protected LiveViews in a live_session with
   on_mount {PortalWeb.UserAuth, :ensure_authenticated}. An anonymous request to a protected route redirects to login.

4) Runtime config (T4 -> R4). Write config/runtime.exs to read DATABASE_URL, SECRET_KEY_BASE, and PHX_HOST from the
   environment at boot (fail loudly if a required one is missing). Keep compile-time config.exs/prod.exs free of
   secrets.

5) Release migration (T5 -> R5). Create lib/portal/release.ex with migrate/0 (load the app, run Ecto migrations for
   each repo) and rollback/2, runnable as `bin/portal eval "Portal.Release.migrate()"`. Add release config to mix.exs.

6) Clustering (T6 -> R6). Add :libcluster to deps and a Cluster.Supervisor with a topology to Portal.Application so
   Phoenix.PubSub and PortalWeb.Presence span every node.

7) Deploy + verify (T7 -> R7 + DoD). Document and run: MIX_ENV=prod mix release; bin/portal eval
   "Portal.Release.migrate()"; bin/portal start. Confirm: register/login/logout; a protected route redirects anonymous
   and admits authenticated; a LiveView enforces on connect; the release boots from env; two nodes cluster and
   PubSub/Presence are correct across them. Run `grep -rE "Portal\.Engine|Repo|GenServer\.call|Portal\.Accounts"
   lib/portal_web/` and confirm the web names only Portal. Report against the F6.8 Definition of Done.

Constraints: Accounts owns User/token schemas and Repo privately; the web calls only Portal; secrets only in
runtime.exs; migrations via the release task, not mix. Out of scope: the dashboard (F6.9), role-based authorization,
and a specific cloud provider's infrastructure-as-code.
```
