## Mode — Flat-L2 lead-team, review gates

Real `Agent` spawns + SendMessage, Director-in-loop — not a `Workflow` fan-out. Risk is **moderate** — but the
risk class is **gate-blindness, not domain logic**: F6.8.2 changes nothing below the facade except a
runtime-derived `worker_id`, yet **almost none of what it ships is exercised by `mix test`**. The defining
feature is that the gate inverts F6.8.1's:

- **The artifacts are BUILD-LOCAL-judged, NOT suite-judged.** A `Dockerfile`, a `fly.toml`, a prod-only
  `runtime.exs` branch (gated out under `:test`), and an OTP `release` are invisible to `mix test`. The proof is
  the **build-local recipe**: `MIX_ENV=prod mix release portal` assembles (and lists the 3 apps, NOT
  `echo_bot`) → `bin/portal eval "Portal.Release.migrate()"` applies the migration → `fly config validate`
  passes → a local prod boot answers `GET /health` **200**. A green suite is necessary hygiene, not the gate.
- **The one id-touching change (D7) DOES earn the ≥100 loop.** `Portal.ID`'s `worker_id` derivation is the
  same-millisecond branded-id collision path (`echo/CLAUDE.md §4`; `id.ex:13`), so the **≥100 determinism loop
  applies to it** — and it must **OWN the machine** (the `IDTest.at/1` ~1h wall-clock skew is a *pre-existing,
  load-induced* flake, NOT a D7 defect; do not chase it, and do not run a wasteful third full loop). D7 is
  behavior-preserving single-node (one node → one `worker_id`), so the loop is a regression guard, not a new
  hazard.

So the gate is: **the build-local recipe (release + migrate + `fly config validate` + a prod-boot `/health`
200) + the ≥100 loop for the id change + the boundary/below-facade grep.** All peers Opus (LAW-2); the Director
coordinates + ratifies (LAW-1a); one commit at close (LAW-4) — and **`libcluster` is the one new dep**,
confirmed at the Stage-5 gate (added to `apps/portal_web/mix.exs` + `mix.lock` at build, not before).

### The build-local boot is a one-shot proof — NOT a standing server (the F6.8.1 inversion)

F6.8.1's liveness was a **standing** property (the Portal had to be left RUNNING for the Operator to probe
`/login`), so `boot→curl→kill` was forbidden. **F6.8.2 is the opposite**: it ships *artifacts*, and the
standing server is the **Operator's `fly deploy`**, downstream of this rung. So `boot→curl /health 200→stop`
**IS** the correct build-local proof here — the falsifiable thing is the *repeatable recipe*, not a live node.
A spawned agent MAY run the one-shot boot-and-curl and report it (it need not leave a node up). The build-local
boot needs a reachable **local Postgres** (the `Portal.Repo` in the supervision tree + the migrate); if none is
available, the gate degrades honestly — the release still assembles, `fly config validate` still runs, and the
boot/migrate become a config dry-review **stated as reviewed, not claimed as run** ("a check counts only if it
RUNS").

## Why the prompts are short (the evolved process)

The discipline lives in the agent definitions; the spawn carries only the rung delta. F6.8.2's one process
delta is the **inversion of F6.8.1's gate**: it is the FIRST rung where `mix test` proves almost nothing — the
deliverables are a `Dockerfile`, a `fly.toml`, a prod-only config branch, and an OTP release, none exercised by
the suite — so the gate moves wholesale to the **BUILD-LOCAL recipe** (assemble → migrate → `fly config
validate` → a prod-boot `/health` 200), and the lone code change the suite *does* touch (D7's per-machine
`worker_id`) is the one piece the ≥100 loop guards. Two corollaries Mars + Apollo must not collapse: the
build-local boot is a **one-shot proof, not a standing server** (the opposite of F6.8.1 — `boot→curl→stop` is
correct because the live server is the Operator's deploy), and the libcluster child must carry an **empty
`:test` topology** or the suite clusters and flakes. Cheap acceptance — the Given/When/Then stories Mars +
Apollo verify against — is the multiplier; here the "tests" are mostly *read-checks* of artifacts (`releases/0`,
`runtime.exs`, the `fly.toml` keep/drop), with the build-local recipe as the executable proof.
