## One equality, three names for it

Stream order == id sort == mint order. Three names for the same fact: the order the writer minted records in, the
order their ids sort in, and the order `XRANGE` reads them back. Because they are one fact, the log is replayable
in the order things happened **without a second index** — no separate timestamp column, no sort step on read.

And it is not a property the writer hopes holds at run time. It is a **theorem proven by construction** in
`EchoMQ.Stream.Id`. The proof is two steps.

## Step 1 — branded byte order == snowflake integer order

A branded id is `namespace ++ base62₁₁(snowflake)`. `EchoData.Base62` is the order-preserving fixed-width-11 codec
— its alphabet is `0-9A-Za-z` in ascending byte order, so the lexical order of the eleven characters equals the
numeric order of the snowflake they encode. For a **fixed namespace**, the leading three bytes are constant, so the
byte order of the whole 14-byte string equals the numeric order of the snowflake.

The qualifier is load-bearing: byte order equals integer order **only within one namespace**, because the namespace
bytes compare first. Two different brands would sort by their letters, not their snowflakes. That is exactly why the
writer's kind door (`evt?/1`) admits **one brand per stream** — it keeps step 1 sound.

```elixir
# echo_mq — EchoMQ.Stream.Id
# evt?/1: the kind predicate the writer calls before any wire. One brand per
# stream (EVT) is what keeps "byte order == integer order" sound — base62 byte
# order equals int order only WITHIN one namespace.
def evt?(branded) when is_binary(branded) do
  match?({:ok, "EVT", _snow}, BrandedId.parse(branded))
end

def evt?(_), do: false
```

## Step 2 — the XADD id is an order-preserving image of the snowflake

Write the snowflake as `snow = ms_part <<< 22 ||| tail22`. The A1 mapping emits the pair `(ms_part + epoch_ms,
tail22)`. Because the snowflake packs the timestamp **high** and the `node|seq` tail in the **low 22 bits** with no
overlap, `snow_a < snow_b` exactly when `(ms_part_a, tail22_a) < (ms_part_b, tail22_b)` lexicographically — which is
precisely the pair `XADD` compares when it orders two entry ids. The `+ epoch_ms` is a constant monotone shift, so
it preserves order. Therefore `snow_a < snow_b ⇔ xadd_id(a) < xadd_id(b)`.

Steps 1 and 2 join: for two records of one namespace, branded sort order, snowflake order, and stream position all
agree. The two architecture decisions — one brand per stream (the kind door) and the field-correspondence id (the
A1 mapping) — are not independent; they are joined precisely so the theorem holds.

## Why a single writer never sees a rejection

`XADD` with an explicit id rejects any id that is equal to or smaller than the stream's current top item. For a
single writer that is impossible to trigger: the mint cell is a strictly-monotone `:atomics` counter, so successive
mints are strictly increasing snowflakes, which (by the theorem) are strictly increasing entry ids, which always
exceed the stream top. The writer never has to retry, never has to fall back to a `*` autogenerated id.

So when does the rejection appear? Only when the mint-order invariant is actually broken upstream — two writers on
one stream, or a misconfiguration. The writer does not paper over it. It surfaces the truth:

```elixir
# echo_mq — EchoMQ.Stream (inside append_id/5)
# The ONLY error mapped to :nonmonotonic is Valkey's verbatim "id ≤ top"
# rejection. It is the wire telling the truth that an upstream mint-order
# violation happened — NEVER swallowed, NEVER retried with a "*" autogen id
# (which would hide the violation and break the order theorem). Every OTHER
# error_reply (e.g. WRONGTYPE) passes through verbatim.
case Connector.command(conn, parts) do
  {:ok, id} when is_binary(id) -> {:ok, branded}
  {:ok, {:error_reply, @id_too_small}} -> {:error, :nonmonotonic}
  {:ok, {:error_reply, msg}} -> {:error, {:error_reply, msg}}
  {:error, _} = err -> err
end
```

`@id_too_small` is the verbatim Valkey message: `"ERR The ID specified in XADD is equal or smaller than the target
stream top item"`. Mapping it to a typed `{:error, :nonmonotonic}` keeps the failure honest and recognizable — a
caller learns the ordering was violated, not that "the write failed for some reason."

## Pattern & implementation

- **The pattern (Redis Patterns Applied).** A log orders entries by their id; readers replay in id order. The
  ordering is usually a runtime property of how ids are assigned.
- **The implementation (echo_mq).** The ordering is a **compile-time theorem**, not a runtime hope: the
  order-preserving Base62 codec plus the high-timestamp / low-tail snowflake packing make stream position a
  provable image of mint order — within one namespace, which the kind door enforces. A real violation is surfaced
  as `:nonmonotonic`, never hidden behind a `*` retry.

## Recap

The order theorem is two joined steps: branded byte order equals snowflake order within one namespace (the kind
door), and the entry id is an order-preserving image of the snowflake (the A1 mapping). A single monotone writer
can never be rejected; a true violation surfaces as `{:error, :nonmonotonic}`. The next dive reads the log back and
shows the branded id is stored as the entry's own field.
