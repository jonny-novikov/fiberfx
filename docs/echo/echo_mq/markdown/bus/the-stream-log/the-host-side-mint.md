> Route: `/echomq/bus/the-stream-log/the-host-side-mint` · module 02, dive 1.
> Surface: `EchoMQ.Stream.append/4`, `append_id/5`, `EchoMQ.Stream.Id.xadd_id/1`, `kind/0`.
> No Lua — `XADD` is issued direct. Extract-and-annotate the Elixir only.

## The append in four moves

To append to the stream is to run `append/4`, and the whole point of the writer fits in four moves:

1. **Mint host-side.** `EchoData.Snowflake.next_branded("EVT")` mints an `EVT`-branded record id in the writer's
   own process. The writer owns the mint, so the id cannot be supplied by a caller and cannot be spoofed — there
   is no "give me an id and I'll trust it" path on the happy route.
2. **Derive the entry id.** `Stream.Id.xadd_id/1` maps the branded id to an explicit `XADD` entry id by field
   correspondence: `"<ms>-<tail22>"` — the real Unix-millisecond high, the 22-bit `node|seq` tail low.
3. **Append direct.** `XADD <key> <xadd_id> id <branded> <fields…>` through `EchoMQ.Connector.command/3`. No Lua,
   no new script: the stream tier is a no-new-script rung. The 14-byte branded string is stored as the entry's
   `id` field.
4. **Return the receipt.** `{:ok, branded}`. The branded id is the receipt the caller keeps; `{:error,
   :nonmonotonic}` is the `id≤top` rejection surfaced; any other connector or server fault passes through verbatim.

## The handle

```elixir
# echo_mq — EchoMQ.Stream
# append/4: mint an EVT id host-side, then delegate to the kind-checked append.
# The writer OWNS the mint (next_branded/1) — nothing to spoof. fields is a
# claims-only flat pair list ([{k, v}, …] or a pre-flattened [k, v, …]).
def append(conn, queue, name, fields)
    when is_binary(queue) and is_binary(name) and is_list(fields) do
  branded = EchoData.Snowflake.next_branded(Id.kind())   # Id.kind() == "EVT"
  append_id(conn, queue, name, branded, fields)
end
```

`append/4` mints, then routes through `append_id/5`. Because its id is freshly minted of the admitted namespace, a
wrong-kind id on this path would be a programming error that cannot occur — the kind door below is for a caller who
supplies the id directly.

```elixir
# append_id/5: the host-side KIND DOOR (INV2). The kind check runs FIRST —
# a malformed or wrong-namespace id RAISES before any wire, symmetric with the
# job-key builder: policy before existence before write.
def append_id(conn, queue, name, branded, fields)
    when is_binary(queue) and is_binary(name) and is_list(fields) do
  case Id.xadd_id(branded) do
    {:ok, xadd_id} ->
      key = stream_key(queue, name)
      # XADD <key> <xadd_id> id <branded> <fields…> — issued DIRECT, no Lua.
      # The branded id is stored as the entry's "id" field (claims-only).
      parts = ["XADD", key, xadd_id, "id", branded | flatten(fields)]

      case Connector.command(conn, parts) do
        {:ok, id} when is_binary(id) -> {:ok, branded}                  # the receipt
        {:ok, {:error_reply, @id_too_small}} -> {:error, :nonmonotonic} # id ≤ top, surfaced
        {:ok, {:error_reply, msg}} -> {:error, {:error_reply, msg}}     # WRONGTYPE etc., verbatim
        {:error, _} = err -> err
      end

    {:error, :kind} ->
      raise ArgumentError,
            "EchoMQ.Stream admits one brand per stream (#{Id.kind()}); got: #{inspect(branded)}"

    {:error, :malformed} ->
      raise ArgumentError, "EchoMQ.Stream.append_id requires a valid branded id; got: #{inspect(branded)}"
  end
end
```

## The A1 mapping — branded id to XADD id

The entry id is not a fresh `*` autogenerated id. The writer derives an **explicit** id from the branded record id,
so the position in the stream is a function of the mint, not of when the wire happened to accept the write:

```elixir
# echo_mq — EchoMQ.Stream.Id
# xadd_id/1: the branded record id → its explicit XADD entry id, by field
# correspondence over the 63-bit snowflake. Pure and total; the writer raises
# on a refusal, the function RETURNS it.
def xadd_id(branded) when is_binary(branded) do
  case BrandedId.parse(branded) do
    {:ok, "EVT", snow} -> {:ok, "#{Snowflake.unix_ms(snow)}-#{snow &&& 0x3FFFFF}"}
    {:ok, _other_ns, _snow} -> {:error, :kind}       # well-formed, wrong namespace
    :error -> {:error, :malformed}                    # not a branded id at all
  end
end
```

The `<ms>` field is the **real Unix-millisecond** of the mint (`Snowflake.unix_ms/1`), not the raw epoch-relative
field — that is what lets a later wall-clock read (`read_window/6`) map a `DateTime` to a bound that lands on the
right entries. The `<tail22>` is the full 22-bit `node|seq` tail: carrying the **node** keeps the coordination-free
"mint on any node" law collision-free on the wire — two writers in the same millisecond with the same sequence stay
distinct by node.

The real doctest vectors, verbatim:

```
EchoMQ.Stream.Id.xadd_id("EVT000xY9Wvvcd")  →  {:ok, "1704117200000-1620567"}
EchoMQ.Stream.Id.xadd_id("ORD000xY9Wvvcd")  →  {:error, :kind}
EchoMQ.Stream.Id.xadd_id("not-branded")     →  {:error, :malformed}
```

## Pattern & implementation

- **The pattern (Redis Patterns Applied).** Append-only logging writes each event to a stream and lets readers
  replay; the entry id orders the log. `/redis-patterns/streams-events` teaches the append-only log and event
  sourcing.
- **The implementation (echo_mq).** `append/4` mints an `EVT` id host-side, derives the explicit `XADD` id by
  field correspondence, and stores the branded id as the entry's `id` field — so the log position is a function of
  the mint, and the receipt the caller keeps is the canonical id, not an opaque stream offset.

## Recap

`append/4` mints `EVT` host-side, derives the explicit entry id with `xadd_id/1`, `XADD`s it direct, and returns
`{:ok, branded}` — the receipt. `append_id/5` is the kind door, raising before the wire on a malformed or
wrong-namespace id. The next dive proves why the order the writer mints in is the order the stream reads back.
