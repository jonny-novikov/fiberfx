# BCS · The law in three clauses

<show-structure depth="2"/>

Part I — Ideas Behind opens here, with the law itself. The preface ([`bcs.preface.md`](bcs.preface.md)) earned one sentence through sixty years of history: encapsulation boundaries are drawn around systems, not objects, and the only thing an object keeps is its name. This chapter unpacks that sentence into its three working clauses — **systems own their state and behavior; only identities and messages about identities cross boundaries; identity is a typed, ordered, placed contract** — and pairs each clause with the failure mode that appears, by necessity rather than bad luck, wherever the clause is dropped. The clauses are not independent virtues. Each one makes the next checkable, and the chapter closes on that composition, because it is where the architecture stops being a style and becomes something a compiler can enforce.

## Clause one — systems own their state and behavior

A system is the unit that encapsulates: a property store, the code that is allowed to touch it, and a boundary that everything else stays behind. Ownership is total or it is not ownership — the AI system's alertness table has exactly one writer and one reader, and the renderer's opinion about alertness is expressed, if at all, as a message to the system that owns it. State and behavior live together not because objects were right all along, but because the *sweep* — the engine iterating its own homogeneous table — is the unit of behavior, and a sweep over somebody else's table is a data race wearing a method's clothing.

The failure mode is **the reach-through**: any path by which one system reads or writes state another system owns. The inherited models institutionalized it — the fat struct made every system a reader of every field, and deep inheritance moved the same promiscuity into base classes — and its distributed forms are worse, because a reach-through across processes is a consistency protocol nobody designed. The tell is always the same: a change to one system's representation breaks code in another system's file. When the clause holds, representation changes are invisible past the boundary, which is the property the whole series spends — the CHAMP store, the ETS timelines, and the queue's keyspace all change shape in later parts without any consumer noticing, precisely because no consumer was ever allowed to look.

## Clause two — only identities, and messages about identities, cross

If systems own everything, something must still connect them, and the second clause names the entire permitted vocabulary: identities, and messages that talk about identities. A message may say *generate an image for `IMG0NnJYzZ19t2` with this prompt* or *the entity this id names has burned*; it may never carry the entity itself, a pointer to it, or a live structure entangled with one. The capstone pipeline is the working example: the job payload is the id, the prompt, and the model name; the status URL *is* the id; the worker's result is keyed by the id and annotated with its placement hash. Nothing shaped like an object ever crossed the HTTP, queue, or storage boundary — and so nothing on either side ever held a reference into the other's memory.

The failure mode is **the traveling object**: an entity, handle, or pointer that escapes its owner. Inside one process it creates lifetime coupling — the receiver can keep the object alive past its owner's intent, or watch it mutate mid-use. Across processes it forces a choice between two bad serializations: deep-copy the object graph (and fork the truth into diverging replicas) or fake a remote reference (and rebuild distributed shared memory, the protocol with all of the costs and none of the guarantees). The clause dissolves the choice. An identity is the one value that can be copied freely without forking truth, because it *names* state rather than carrying it; whoever needs the current truth asks the owner. Every queue payload, cache key, stream entry, and API parameter in this project is an application of that sentence.

## Clause three — identity is a typed, ordered, placed contract

The first two clauses concentrate all inter-system traffic into one value, which is why the third clause exists: the value everything crosses on cannot be the weakest value in the program. The contract ([`../../contract/contract.md`](../../contract/contract.md)) gives identity four load-bearing properties, and each one retires a specific failure that otherwise reappears, system by system, as bespoke machinery.

**Typed, or the silent join.** Without an in-value, in-type discriminant, an id of one entity kind reaches another kind's table and the program compiles anyway; the bug surfaces as a runtime miss if the system is lucky and as cross-entity data if it is not. With the namespace in the first three bytes and in the type parameter, the wrong-table join dies twice before runtime: the cross-runtime article's deliberate negative test is one compiler error — `BrandedId<"USR">` is not assignable where `BrandedId<"CRS">` is required — and at the HTTP boundary the same mismatch is a schema-layer rejection, the measured gate row reading 200 / 400 / 400 / 404 with the wrong-namespace 400 issued before any handler runs.

**Ordered, or the second clock.** Without mint order in the value, every system that needs chronology bolts on a parallel scheme — a timestamp column, a sort step, an index on `created_at` — and the id and the clock drift into separate sources of truth. The order theorem collapses them: string order is time order, so a property table keyed by branded id is already a timeline, a feed is a descending string sort, and a time window is two synthetic cursors. The structures article builds five stores on nothing else, and the storage decision record extends it into stream entries — a ten-millisecond window addressed purely by id arithmetic returning exactly its predicted 40960 entries.

**Placed, or the routing table.** Without a placement function in the contract, locating data means asking — a directory service, a rendezvous step, a coordination protocol — and every runtime asks differently. With hash32 in the contract, placement is arithmetic any holder of the id performs identically: the same function routes a CHAMP slot, a cache shard, and a hash partition, and it costs under a nanosecond in pure Go (0.9586 ns) because it is five integer instructions. The reference vector — 234878118 — has now been computed, identically, by Elixir, Rust, C, TypeScript, wasm, Go, and SQL in this project's committed outputs.

**A contract, or the dialect.** Without a canon, each runtime grows its own slightly-wrong identity library, and the boundaries between systems become translation layers. The canon — one Rust source, one C reference, one vector file, a conformance suite per runtime — makes "which language is this system in" a deployment detail. That is the property the second clause quietly depends on: identities can be the only thing that crosses *because* they mean the same thing on every side.

## The composition

Read forward, the clauses are a proof sketch. Clause one makes the crossings *enumerable*: a system has a finite set of ports, not an object graph's unbounded reference surface. Clause two makes every crossing *typeable*: if only identities cross, then every port's signature is expressible in namespaces. Clause three makes the typing *checkable before anything runs*: the namespace is a compile-time discriminant, so the flow of identities between systems — which is now the entire inter-system architecture — is verified by the type checker, the schema compiler, and the storage domain. This is the preface's closing claim made mechanical: the compile-time hierarchy of encapsulation matches the domain model, because the domain vocabulary is the namespace set and each system's boundary declares the subset it admits.

Read backward, the composition shows why dropping any clause re-breaks the others. Drop ownership and the typed identities are decoration — reach-throughs bypass the ports the types guard. Drop the identities-only rule and the contract has nothing to bind — traveling objects carry their own untyped references past every gate. Drop the contract and the first two clauses still hold in spirit while failing in practice — the silent join, the second clock, the routing table, and the dialect each reopen, one per missing property, and the boundary that systems kept is crossed by a value too weak to police it. Three clauses, one law.

## Boundaries

This chapter argues; it does not yet build. The OTP shape of a system, the property stores behind the boundary, archetypes, and relations are Part II's work, and the bus the messages ride is Part III's. The persistence question stays deferred per the standing decisions in [`bcs.toc.md`](bcs.toc.md). And the law is stated for systems that adopt it whole — the composition section is the argument for why partial adoption returns partial failure modes, not partial benefit.
