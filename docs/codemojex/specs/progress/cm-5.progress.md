### T-4 — [postgres] The relational-lens derivation: two facts drive the whole redesign

(1) Pool-is-a-column → the buy-in is the app's first CROSS-ENTITY transaction (players + games). Both are Postgres, so ONE Repo.transaction spans them — no Ecto.Multi needed (L-13); the convert_to_keys paired-txn! idiom (wallet.ex:118) extends to a debit + an atomic update_all(inc: [prize_pool: …]). The atomic SQL `+` (never an app-side RMW) is the lost-update guard for N concurrent buy-ins (D-13).
(2) Ledger-is-authority → exactly-once must live IN Postgres, not in Valkey. L-10's headline leak: a Valkey SET …:paid NX and the Postgres debit are in DIFFERENT stores and are NOT atomic, so a crash diverges them (paid-but-never-debited = a pool money-leak; or double-debit). The fix is a partial unique index on transactions(player,ref) WHERE reason='buy_in' — putting the exactly-once guard in the same store as the debit + pool `+`, crash-safe by construction. The Valkey paid-set degrades to a fast-path HINT. The refund mirrors this with WHERE reason='buy_in_refund'.
COROLLARY for the migration: the index `where:` predicate is the SOLE source of truth for the wallet on_conflict conflict_target fragment (the cm.4 byte-match lockstep migration2:11-12 ↔ wallet.ex:73-76). Three schema-shape questions remain genuinely OPEN (the pool denomination, the gather-deadline column, the gold_multiplier migration mechanism) — framed as forks F-1/F-2/F-3, not decided.

## {cm-5-learnings} Learnings

### L-1 — [postgres] Cross-store exactly-once belongs in the SAME store as the money move — the partial unique index, not a Valkey NX

The relational corollary of L-10: when an exactly-once guard and the mutation it guards live in DIFFERENT stores (a Valkey SET …:paid NX vs a Postgres debit+pool-`+`), they are not atomic and a crash diverges them — a pool money-leak (paid-but-never-debited) or a double-debit. The redesign therefore makes the guard a partial UNIQUE index on transactions(player,ref) WHERE reason='buy_in' (and 'buy_in_refund'), co-located with the debit in ONE Repo.transaction, so the op is exactly-once crash-safe BY CONSTRUCTION; the Valkey paid-set degrades to a fast-path hint. HOW TO APPLY: on any money op spanning Valkey + Postgres, put the idempotency marker in the ledger (the durable store of the value move), and treat the cache/Valkey guard as a non-authoritative hint. The byte-matched on_conflict conflict_target fragment ↔ index `where:` predicate (the cm.4 lockstep) is the second half: a drift there is gate-invisible on a single writer and only a concurrent/crash test catches it. Aimed at the relational-design contract (the store-boundary idempotency remit).

### L-4

L-R1 — [postgres] A tiered-by-ordinal money model introduces a per-game-count race; the games-row FOR UPDATE fixes it AND subsumes the start race

The revised economy (D-7) makes the entry-fee allocation a function of the MEMBER ORDINAL (1..start_threshold = deposit-recovery → platform; start_threshold+1..+first_movers = first-mover split → pool; beyond = pure revenue). The ordinal = the count of buy_in TXNs for the game. The shipped wallet locks ONLY the player row FOR UPDATE (wallet.ex:195) — which does NOT serialize a per-GAME count, so two concurrent different-player buy-ins could read the same count and BOTH mis-tier (e.g. both think ordinal 11) — a money MISALLOCATION (the pool/platform split is wrong). The fix: lock the GAMES row FOR UPDATE at the top of buy_in. buy_in already writes games.prize_pool, so the game row is the natural serialization point — and it ALSO serializes the gather count + the :gathering→:open start trigger, subsuming the L-7 cross-store start race (the Valkey :started NX degrades to belt-and-suspenders). HOW TO APPLY: when a money op's behavior depends on a COUNT/ORDINAL over a parent entity (not just the actor's own row), lock the PARENT row FOR UPDATE (or a per-parent advisory xact lock) so the count is consistent — the per-actor row lock is insufficient. One lock then covers the ordinal, the count-triggered transition, AND the parent mutation. The guess→pool increment, by contrast, needs NO lock (a bare atomic update_all(inc:) — additive, no ordinal). Aimed at the relational-design contract (the per-parent-count serialization remit).
