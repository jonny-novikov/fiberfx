### L-3 — Postgres SUM over :bigint returns Decimal, not integer (the cm.6 read hazard)

The reconciliation reads (house_balance/1, revenue_breakdown/1) use `sum(r.delta)` over the :bigint `delta` column. Postgres SUM() over a bigint returns arbitrary-precision numeric → Postgrex maps it to a %Decimal{} struct, NOT an Elixir integer. The first suite run failed 6 stories with `(ArithmeticError) Decimal.new("8") + 0` on `Map.values() |> Enum.sum()`. The cm.5 reads (buy_in_count, paid_count) never hit this because they use count(), which returns a plain integer.
FIX (realization-over-literal, in PRODUCTION not the test): wrap the SUM in `type(sum(r.delta), :integer)` in both reads, so finance + the BNK bank + the property tests read a plain integer. Justified by D-2: keys is integer-exact (the split floors in keys before the ×10), so the cast is lossless. This is the keys-unit-exactness ruling enforced at the read boundary.
The per-row reads in the test (`select: r.delta` for a single row) are unaffected — only the aggregate SUM needs the cast.
