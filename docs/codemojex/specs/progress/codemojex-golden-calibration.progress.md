So Venus-B IS sweep-immune — but ONLY by silent BEAM term-ordering, NOT by an explicit type guard.

- Venus-A (:gathering): immune because the guard matches status==:open and a gathering game is :gathering — immune by an EXPLICIT status check, the most legible+robust form.
ADJUDICATION INPUT: Venus-B's "the engine already no-ops on a nil ends_ms, no new guard" is HALF-TRUE — expired? is robust, close_if_expired is robust only by accident of term ordering. ### D-14 — [economy] close_void's buy-in refund: refund EVERY buy-in (read the paid set / the buy_in TXN rows), exactly-once BELOW the close lock, crash-retry-safe via per-player idempotent refund markers; refund ONLY the buy-ins, NOT the per-guess fees (per-guess fees are platform revenue, Operator-ruled)

### L-7

L-A3 — THE NEVER-FILLS INTEGRITY PROBE (high-risk money op, probed adversarially) + UN-PROMPTED #3: the gathering→open transition has NO exactly-once guard in EITHER design

### L-10

L-A5 — UN-PROMPTED (economy) #2: the buy-in entry-guard has the SAME 3-non-atomic-ops race as the gather start-lock — and a deeper ordering hazard: paid-NX vs the pool-credit are in DIFFERENT stores

The design guards the buy-in with SET cm:<game>:paid:<player> NX checked BEFORE buy_in/3 (§3.2). This is the SAME class as the gathering→open start-lock I flagged in the start-mechanic eval (L-A3): the NX-check and the buy_in Ecto.Multi are TWO ops in TWO stores (Valkey NX, then Postgres Multi), NOT atomic. The failure mode is ASYMMETRIC and worse than the start-lock:
(1) NX succeeds (marks paid) then the buy_in Multi CRASHES/rolls back → the player is marked paid in Valkey but NEVER debited and the pool NEVER credited → on retry the NX returns 0 (already paid) → the player plays FREE and the pool is short. A money LEAK (the pool is underfunded; the house or the winners eat it).
(2) Reverse order (Multi first, then NX-mark): the Multi debits+pools, then the NX-mark CRASHES → on retry NX succeeds again → DOUBLE buy-in (player debited twice, pool credited twice). A player OVERCHARGE.
NEITHER ordering is exactly-once across the store boundary. The CORRECT design (which the brief must mandate): make the paid-guard LEDGER-AUTHORITATIVE like the refund (§6.1), NOT Valkey-NX. The buy_in Multi itself should be idempotent on a `buy_in` TXN existing for (player, ref=game) — a partial unique index on transactions(player,ref) WHERE reason='buy_in' (mirroring the buy_in_refund index + the shipped tg_user_id pattern), so the buy_in is exactly-once IN POSTGRES (the same store as the debit+pool), and the Valkey paid-set becomes a cheap fast-path HINT, not the source of truth. This makes the entry charge crash-safe by construction. The design's Valkey-NX entry-guard is the ONE money-integrity gap in the economy model that a build would ship as-specified and only a cross-store crash test would catch. THIS IS THE HEADLINE RESIDUAL RISK — sharper than the design's own §10 (which names the Multi atomicity + the paid-NX but does NOT see that the NX-across-stores is itself non-atomic with the Multi).
