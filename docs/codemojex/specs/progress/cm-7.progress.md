## {cm-7-alternatives} Alternatives

### V-1 — F1 the RAIL abstraction: a first-class rail entity VS a bare currency-discriminator column

ARM A — a first-class payment-rail/method entity (a `rails` lookup table or an Elixir config-module enum of %{rail, currency, min_units, decimals, store_fee_pct, provider}). ARM B — a bare `currency`/`rail` string column on the order, no entity; the rail is just a tagged value, its facts live in a pure code module.

STEELMAN A: rails carry real per-rail facts (decimals, min-unit, store-fee, the exactly-once key shape, the provider adapter) — a first-class entity is the honest home for them and lets an operator add a rail (a new stablecoin) without a deploy. STEELMAN B: there are exactly FOUR rails (Stars/TON/USDT/RUB), known at compile time, changing on the order of months — a DB lookup table for four near-static rows is ceremony; a pure `Codemojex.KeyShop.Rails` module (the rail facts as module data, pattern-matched) is correct, testable, zero-DDL, and the brand on the order (currency string) already type-distinguishes.

RANK (product/minimal-surface lens): ARM B — a bare `currency` string on the order + ORD/OTX, with the rail FACTS as a pure code module (`KeyShop.Rails`), NO `rails` table. The four rails are a closed, code-known set; the revenue_ledger ALREADY keys revenue by a free `currency` string (house_balance GROUPs by it, wallet.ex:329) — adding a parallel `rails` table duplicates that dimension and invites drift. STEELMAN A: the decimals are a PURE FUNCTION of the currency (a closed, code-known set); the conservation invariant (cm.6) is WITHIN-currency (house_balance GROUPs BY currency, wallet.ex:329), so cross-currency summing never happens at the row level and the units never need reconciling in one SUM — a code module is the honest, drift-free home; a column duplicates a code fact.
RANK (product/minimal-surface lens): ARM A — store each amount in its standard minor unit (the integer smallest unit) inside the :bigint delta, tag by the `currency` string, carry the decimals/min-unit/standard-name in the pure `KeyShop.Rails` module (F1 Arm B's home). NO per-currency unit column. The :bigint is sized for it (nanoTON of a 130-TON whale = 1.3e11 ≪ bigint 9.2e18; T-1/T-3). house_balance's per-currency buckets stay integer-exact (the cm.6 D-2 keys-exactness generalizes: each currency is exact in its own minor unit).
PRE-EMPT the payments-integrity objection ("a reader needs a decimals column to render/import a row"): the (currency string → decimals) map is a single pure function over a closed set — the row's `currency` + the module IS the scale; a column stores a derivable constant and invites drift (a row's decimals diverging from the currency's true scale). For an external finance export, the convention is published (the cm.6 forward-compat reconcile note + the KeyShop.Rails module data) — a documented, versioned mapping, not a per-row copy. If a currency ever needed a per-instance scale (it does not — minor units are fixed by the currency), THAT would justify a column; the closed fixed-scale set does not.

### L-1 — The roadmap §Commerce "OTX separate from TXN" vs cm.6 D-5 "book revenue to the same RVL" is NOT a contradiction — it is three rows about one event

SURPRISE worth recording for the Director synthesis + the cm.7 build: a reader could see codemojex.roadmap.md §Commerce L272 ("A payment ledger OTX for Stars, kept SEPARATE from the currency ledger TXN") and cm.6 D-5/S11 ("book gross purchase revenue into the SAME revenue_ledger RVL") as conflicting homes for purchase money.
