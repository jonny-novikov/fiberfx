### D-3

## {cm-7-alternatives} Alternatives

### V-1 — F1 the RAIL abstraction: a first-class rail entity VS a bare currency-discriminator column

ARM A — a first-class payment-rail/method entity (a `rails` lookup table or an Elixir config-module enum of %{rail, currency, min_units, decimals, store_fee_pct, provider}). ARM B — a bare `currency`/`rail` string column on the order, no entity; the rail is just a tagged value, its facts live in a pure code module.

STEELMAN A: rails carry real per-rail facts (decimals, min-unit, store-fee, the exactly-once key shape, the provider adapter) — a first-class entity is the honest home for them and lets an operator add a rail (a new stablecoin) without a deploy. STEELMAN B: there are exactly FOUR rails (Stars/TON/USDT/RUB), known at compile time, changing on the order of months — a DB lookup table for four near-static rows is ceremony; a pure `Codemojex.KeyShop.Rails` module (the rail facts as module data, pattern-matched) is correct, testable, zero-DDL, and the brand on the order (currency string) already type-distinguishes.

RANK (product/minimal-surface lens): ARM B — a bare `currency` string on the order + ORD/OTX, with the rail FACTS as a pure code module (`KeyShop.Rails`), NO `rails` table. The four rails are a closed, code-known set; the revenue_ledger ALREADY keys revenue by a free `currency` string (house_balance GROUPs by it, wallet.ex:329) — adding a parallel `rails` table duplicates that dimension and invites drift. The minor-unit/decimals/store-fee facts belong in the pure pricing module (F2/F5), not a row.

PRE-EMPT the payments-integrity objection ("you under-modeled — a rail needs provider config, an enable flag, audit"): the rail's IDENTITY on the money trail IS captured — every OTX row carries its rail + the external charge id (the audit anchor), every RVL row carries currency=the rail (the revenue audit). What ARM B omits is a *mutable operator-tunable rail registry* — and that is a deliberate cm.8+ deferral (a rail is enabled by shipping its adapter, not toggling a row), not an audit gap. If the Operator wants runtime rail-toggling, that is a named forward (a `rails` table beside `packages`), and the order's currency string extends to it with no reshape.

### V-2 — F2 PACKAGE PRICING: per-rail price columns VS one base price + a rate-derived per-rail conversion

ARM A — the `packages` row carries a price PER RAIL (stars_price, ton_price, usdt_price, rub_price columns), each set/edited by the operator. ARM B — the `packages` row carries ONE base price (keys + a base Stars price, the economy.packages.md ladder) and the per-rail price is DERIVED at checkout from the base via a live rate (TON/USDT/RUB ← the rate surface, F3).

STEELMAN A: each rail's price is then exact, operator-controlled, and stable (no rounding surprises, no live-rate dependency at display); a round-number RUB price reads better than a rate-converted one; the discount tiers (economy.packages.md, 0..50%) live once, in the row. STEELMAN B: four price columns × N packages is a maintenance multiplier — a rate move means re-pricing every cell by hand (the very drift economy.packages.md warns of: "the USD value fluctuates daily with TON's market price"); one base price + a derived conversion keeps a single source of truth and the discounts in ONE place (the base ladder), the rails computing off it.

RANK (pricing-module lens): ARM B as the SPINE — one base price (the Stars ladder is the canonical face, economy.packages.md; keys is the product), the per-rail amount DERIVED by the pure `KeyShop` module from the base + the F3 rate, WITH an optional per-rail price OVERRIDE column (nullable) for the rails an operator wants pinned to a round number (RUB especially). The discounts live ONCE in the base ladder (the package's stars/keys ratio). This is the economy.packages.md model precisely: "a configurable TON rate so the USD figures recompute from a live stars_usd_rate rather than the baked-in $0.013."

PRE-EMPT the payments-integrity objection ("a derived price means the amount you charge ≠ the amount you booked — a rounding/timing seam"): the derivation happens ONCE, at order creation, and the DERIVED rail amount is PINNED on the order row (the amount_minor + the rate snapshot, F3/F4) — the charge, the OTX receipt, and the RVL revenue all read that ONE pinned figure, never re-derive. The live rate is an INPUT to pinning, never read again after the order is created. So there is no charge-vs-book drift; the override column simply lets the pin be operator-set instead of rate-derived for chosen rails.

### V-3 — F3 the RATE SURFACE: a snapshotted rates table VS a live provider VS config; WHERE the rate-at-purchase is pinned

ARM A — a `rates` table (rows {pair, rate, source, fetched_at}), refreshed by a poller, read at order creation, and the rate-at-time SNAPSHOTTED onto the order. ARM B — a live provider call at checkout (hit a price API per order). ARM C — config (a static rate in runtime.exs, operator-edited).

STEELMAN A: a rates table is auditable (every rate has a fetched_at + source), decoupled from the provider's uptime (the order reads the last-good row, not a live API on the hot path), and the snapshot onto the order is the audit pin finance needs. STEELMAN B: a live call is always-fresh, zero staleness. STEELMAN C: config is the smallest possible surface — no table, no poller — and for a soft-launch the operator can hand-set the TON rate daily.

RANK (minimal-surface lens, but money-honest): the PIN LOCATION is non-negotiable and lens-independent — the rate MUST be captured ON THE ORDER ROW at creation (a `rate_used` + `rate_pair` + `rate_at` pinned per order), because TON floats and booked revenue must be reproducible (the cm.6/economy.packages.md discipline: "revenue recognized ≠ cash in hand"). Given the pin is on the order, the SOURCE is the cheap part: rank ARM C (config rate, read at order creation, pinned) for cm.7's launch — a single `key_shop_rates` config map (stars_usd, ton_usd, usdt_usd, rub_usd) in runtime.exs, operator-tunable, with the value SNAPSHOTTED onto every order. NAME ARM A (the `rates` table + poller) as the cm.8 forward (the withdrawal rung needs live floating rates for cash-out anyway — build the rates table THERE, where the volume + the regulatory audit justify it). ARM B (live-on-hot-path) is rejected — a provider outage must never block a purchase.

PRE-EMPT the payments-integrity objection ("config rates go stale — you book revenue at a wrong rate"): the rate is ALWAYS pinned on the order, so whatever rate was config'd AT THAT MOMENT is the auditable truth for THAT order — a later config change never rewrites a booked order. Staleness is an operator-freshness concern (set the config daily, the economy.packages.md cadence), NOT a correctness-of-the-ledger concern. The pin makes the books reproducible regardless of the source's freshness; that is the property that matters. The forward rates table (cm.8) upgrades the SOURCE without touching the pin's shape.

### V-4 — F4 the ORDER/PAYMENT model: one orders table + a rail discriminator VS per-rail payment tables; the exactly-once key per rail

ARM A — ONE `orders` table (ORD) with a `rail`/`currency` discriminator + a state machine (created→pending→paid→fulfilled / failed / refunded), and ONE `order_transactions` table (OTX) holding the external payment record per rail (the charge id / tx hash / processor id), the exactly-once anchor; one `webhooks` (WHK) idempotency table for inbound rail events. ARM B — per-rail payment tables (a stars_payments, a ton_payments, a fiat_payments), each shaped to its rail's fields.

STEELMAN A: the order LIFECYCLE (a player buys package X, it is pending, it is paid, keys are minted) is rail-INDEPENDENT — one orders table with a discriminator is the honest model; the rail-SPECIFIC bytes (charge id vs tx hash vs processor ref) are a single OTX row keyed by (rail, external_id) UNIQUE, which IS the exactly-once guard per rail; WHK dedupes the inbound event before it ever touches ORD/OTX. Three tables, the roadmap's exact ORD/OTX/WHK split. STEELMAN B: each rail's payment fields differ (Stars has a telegram_charge_id + a currency=XTR amount; TON has a tx_hash + a from_address + a confirmation count; fiat has a processor + a settlement id), so per-rail tables type each rail's fields exactly with no nullable sprawl.

RANK (minimal-surface lens): ARM A decisively — the ORD/OTX/WHK split the roadmap already names (lines 61-64, 269-274). ONE orders table (the rail-independent lifecycle + a `currency` discriminator), ONE order_transactions table (OTX: {id, order_id, rail, external_id, amount_minor, raw_payload, status}; the external_id the rail's charge id / tx hash / processor id), with a UNIQUE(rail, external_id) — THAT index is the exactly-once key per rail (the named fix for today's "stars"-literal double-mint). ONE webhooks table (WHK: {id, rail, event_id UNIQUE, processed_at}) for inbound idempotency. The rail-specific bytes live in OTX's external_id + a jsonb raw_payload, NOT in separate tables. The keys MINT (the wallet TXN credit) fires exactly-once gated on OTX's UNIQUE — the cm.5 buy_in exactly-once pattern (a partial unique index = the exactly-once authority) applied to purchases.

PRE-EMPT the payments-integrity objection ("a single nullable-heavy OTX under-types each rail; a jsonb payload is not an audit trail"): the exactly-once-CRITICAL field is UNIFORM across rails — every rail HAS exactly one external charge identifier (charge id / tx hash / processor ref), so UNIQUE(rail, external_id) is well-typed and sufficient as the dedup key; the rail-VARYING fields (from_address, confirmations, settlement id) are AUDIT-context, not exactly-once keys, and a jsonb raw_payload preserves the FULL provider receipt verbatim (a stronger audit trail than hand-picked columns — nothing is dropped). The state machine + the (rail, external_id) UNIQUE + the verbatim payload IS the audit trail; per-rail tables would fork the lifecycle logic three ways for no exactly-once gain. cm.7 BUILDS the Stars rail end-to-end (the Telegram invoice flow) and SHAPES the ORD/OTX/WHK rows for TON/USDT/RUB (their adapters land as each rail's webhook verifier ships — the order/OTX/WHK shape is rail-stable).

### V-5 — F1 the RAIL abstraction (VenusPG / payments-integrity lens): a CURRENCY DISCRIMINATOR column on the order, NOT a first-class payment-rail entity

## {cm-7-learnings} Learnings

### L-1 — The roadmap §Commerce "OTX separate from TXN" vs cm.6 D-5 "book revenue to the same RVL" is NOT a contradiction — it is three rows about one event

SURPRISE worth recording for the Director synthesis + the cm.7 build: a reader could see codemojex.roadmap.md §Commerce L272 ("A payment ledger OTX for Stars, kept SEPARATE from the currency ledger TXN") and cm.6 D-5/S11 ("book gross purchase revenue into the SAME revenue_ledger RVL") as conflicting homes for purchase money. They are not. A single purchase produces THREE rows in THREE tables, each a distinct concern:
- OTX (order_transactions) = the EXTERNAL PAYMENT RECEIPT — the rail's charge (Telegram charge id / on-chain tx hash / processor settlement id); the exactly-once gate UNIQUE(rail, external_id). This is what L272 means by "separate from TXN" — the rail receipt is NOT the player's currency ledger.
- TXN (transactions, as-built) = the PLAYER'S WALLET CREDIT — the N keys minted (the cm.5 ledger).
- RVL (revenue_ledger, cm.6) = the PLATFORM'S REVENUE RECOGNITION — the gross rail amount as house income.
So OTX IS separate from TXN (L272 holds) AND revenue books to RVL (D-5 holds). The resolution is in cm.7.md §2 ("Three rows, three concerns, one event"). This matters because a naive build might try to put purchase revenue on OTX (wrong — OTX is the receipt, not the revenue balance) or skip RVL (wrong — finance reads house_balance, which only sees RVL). The exactly-once gate (OTX UNIQUE) makes BOTH the TXN mint and the RVL booking fire once — they ride the gate, they are not the gate.

SECONDARY: the as-built cm.6 build sits UNCOMMITTED in the working tree (wallet.ex/rooms.ex M, revenue_ledger.ex + the migration untracked) — exactly as the brief stated. The Director must keep this in view when committing cm.7-phase docs (LAW-4 pathspec): the cm.6 code + the cm.7 docs are TWO concerns; do not entangle them in one commit. (This is a design phase — no commit by me; flagging for the eventual ratify.)
