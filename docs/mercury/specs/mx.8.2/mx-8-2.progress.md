### L-2 — `storybook/test` is type-invisible from a package-owned story under pnpm strict; a runtime require.resolve does NOT prove tsc resolution

mx.8.2 is the first slice to import `storybook/test` (mx.8.1 deferred K-4). The 3 spy edits compiled at runtime but `sb:typecheck` failed TS2307 on all three. Root cause: the story files physically live under `packages/mercury-ui/`, but `storybook` (providing `fn`) is symlinked ONLY under `apps/storybook/node_modules` (pnpm strict, not hoisted). `tsc` resolves bare specifiers by walking up from the IMPORTING FILE's dir — which never reaches the app's node_modules — so `storybook/test`'s types are unreachable, even though `moduleResolution: Bundler` and the `./test` export declares its `.d.ts`. This is the identical situation the tsconfig already solved for `@storybook/react-vite` via an explicit `paths` entry (apps/storybook/tsconfig.json:12).

The Director's pre-ruling "verified via require.resolve from apps/storybook" was GREEN while tsc was RED — because Node's runtime resolver resolves from the consuming APP dir (or an explicit {paths}), but tsc resolves from the SOURCE FILE dir. A "check counts only if it RUNS" case at the type layer: require.resolve is not a substitute for the host tsc on a cross-package story import.

Fix (zero-dep, realization-over-literal): one `paths` entry `"storybook/test": ["./node_modules/storybook/dist/test/index.d.ts"]` mirroring line 12. It adds no dependency (INV-6 preserved — no package.json/pnpm-lock/main.ts change; fn still resolves to SB core) and is group-agnostic: apps/storybook/tsconfig includes ALL `../../packages/mercury-ui/src/**/*.stories.tsx`, so every future interactive slice (mx.8.3 selection, inputs, …) importing `storybook/test` inherits the fix. Fold forward: a brief that mandates a new SB-core subpath import from a package-owned story must budget the matching host-tsconfig `paths` entry, and its acceptance must run the host tsc (not just require.resolve).
