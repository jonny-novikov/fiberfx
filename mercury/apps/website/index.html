01 values and principles

A design system serving products, not itself.

Mercury is built around three principles. Each one rules out a different kind of waste — waste of time, waste of effort, waste of quality — and together they keep the system honest.

01 · efficiency

Efficiency, not consistency.

Consistency is a result of efficiency, not the other way around. When the system is intuitive and empowering to use, consistency becomes an automatic by-product of adoption.

"A design system isn't a project. It's a product serving products." — Nathan Curtis, EightShapes
02 · agility

Dynamic capability + stability.

Agility needs two things. A dynamic capability — speed, nimbleness, responsiveness — and a stable foundation of things that don't change. Mercury is both.

"The more decisions you put off, and the longer you delay them, the more expensive they become." — Craig Villamor, Google Maps
03 · quality

Quality over quantity.

Focus on the best solutions. Quality emerges as the consolidation of Agility and Efficiency — and compounds as each component iterates.

"Styles come and go. Good design is a language, not a style." — Massimo Vignelli, Italian designer
02 why mercury

Same, same — but different.

One system powering many brands. Mercury ships a shared language of components, then lets each product express itself through theming.

Consistent among different brands

Same interaction and UI across every module — checkout, sidebar, onboarding, inputs. Themed per brand without forking the system.

Reducing development costs

Devs from different teams don't re-implement the same components. Designers don't redraw them. Less alignment, fewer bugs.

Faster UI development

One source of truth for components and icons. Non-frontend developers can still build good-looking UI.

Validating ideas for stakeholders

Build your own UI (rapid prototype) faster, not from scratch. A real benefit during hackathons, pitches and internal MVPs.

Performance

Best practices baked in. Lightweight by design — less JS, more performance.

03 foundations

Tokens first. Components second.

Every style — colour, shadow, radius, spacing — lives in a token. Themes swap tokens; components never change.

05 getting started

Install. Import. Ship.

Mercury works with React today, and with design tokens in any framework. No CLI, no config, no peer-dependency matrix to chase.

Install
npm i @mercury/core
npm i @mercury/icons
npm i @mercury/themes
Use
import { Button } from '@mercury/core';

<Button variant="primary">
  Ship it
</Button>
