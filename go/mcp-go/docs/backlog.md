## §1. API design rough edges (sourced from `docs/rough_edges.md`)

Nine items enumerated from `docs/rough_edges.md` (weave-autogenerated from
`internal/docs/rough_edges.src.md`). All classified **[V2]** because the
upstream file header declares: *"These issues can't be addressed without
breaking backward compatibility, but we'll revisit them for v2."* Each entry
restates the rough edge, its current workaround (if any), migration surface,
and the SDK-v2 action item.

### 1.1. [V2] `EventStore.Open` is unnecessary

- **Summary:** `Open` method on `EventStore` is a vestigial artifact from an
  earlier SDK iteration where event persistence and delivery were combined.
- **Current workaround:** Implementers may provide a no-op `Open`
  implementation with no observable effect.
- **Breaking-change impact:** Any embedder with a custom `EventStore`
  implementation must remove the `Open` method signature from the interface.
  Impact surface: custom event-store authors; negligible for default in-memory
  implementation consumers.
- **SDK-v2 action item:** Remove `Open` from the `EventStore` interface.
  Document the removal in v1→v2 migration notes.

### 1.2. [V2] `Event` should not have been exported; `Name` field is a misnomer

- **Summary:** `Event` is an internal implementation detail of the SSE and
  streamable transports. Further, its `Name` field is misnamed — the payload
  semantic is `event`, not "name".
- **Current workaround:** None. Field accessible externally but intended as
  internal.
- **Breaking-change impact:** Consumers reaching into `Event` by name must
  switch to the unexported replacement (or use higher-level transport APIs).
- **SDK-v2 action item:** Unexport `Event`; rename `Name` → `Event` in the
  internal replacement.

### 1.3. [V2] Missed opportunity to panic on invalid tool names

- **Summary:** Tool-name validation per upstream
  [SEP-986](https://github.com/modelcontextprotocol/modelcontextprotocol/issues/986)
  landed after the SDK reached v1.0. The SDK currently logs an error rather
  than panicking on invalid tool names.
- **Current workaround:** Error-logging stays visible but does not fail-fast.
  Registering an invalid tool name produces a log entry and the tool is
  skipped.
- **Breaking-change impact:** Any caller registering tools with non-conforming
  names will begin panicking at registration time in v2.
- **SDK-v2 action item:** Replace error-log-on-invalid-tool-name with panic;
  document pre-flight name validation as a migration step.

### 1.4. [V2 — SHIPPED] Notification param-type naming normalized

- **Summary:** Two notification param types previously carried the word
  "Notification" in their type names even though most sibling notification
  param types omit it. Renamed in v2 for consistency.
  - `ResourceUpdatedNotificationParams` → `ResourceUpdatedParams`
  - `ProgressNotificationParams` → `ProgressParams`
- **Status:** Shipped in v2. Consumer-facing breaking change; migration is a
  mechanical rename.
- **Request aliases preserved:** `ProgressNotificationServerRequest`,
  `ProgressNotificationClientRequest`, `ResourceUpdatedNotificationRequest` retain
  their names (scope discipline — alias rename deferred to a future wave).

### 1.5. [V2 — SHIPPED] `AudioContent.MarshalJSON` pointer receiver

- **Summary:** `AudioContent.MarshalJSON` now uses a pointer receiver,
  consistent with peer content types. Previously a value receiver.
- **Status:** Shipped in v2. All existing callsites already used pointer form
  (`&AudioContent{...}` or `new(AudioContent)`), so no downstream churn.
- **Breaking-change impact:** Only a hypothetical value-form regression
  (`[]Content{AudioContent{...}}`) would fail to compile; none present in-tree.

### 1.6. [V2] `ClientCapabilities.Roots` should be a distinguished struct pointer

- **Summary:** Tracked at upstream
  [#607](https://github.com/fiberfx/mcp-go/issues/607). `Roots` was not modeled
  as a distinguished struct pointer, unlike other capability fields.
- **Current workaround:** Use `ClientCapabilities.RootsV2`, which aligns with
  the semantics of other capability fields.
- **Breaking-change impact:** Consumers currently reading `Roots` must switch
  to `RootsV2` pre-v2, then `RootsV2` becomes `Roots` at v2. Two-step
  migration.
- **SDK-v2 action item:** Remove legacy `Roots`; promote `RootsV2` to `Roots`.

### 1.7. [V2] Default capabilities should have been empty

- **Summary:** Servers currently default to advertising `logging` and clients
  default to advertising `roots` with `listChanged: true`. A nil
  `Capabilities` field therefore does not mean "no capabilities", which is
  semantically confusing.
- **Current workaround:** Set `ServerOptions.Capabilities` or
  `ClientOptions.Capabilities` to an empty `&ServerCapabilities{}` or
  `&ClientCapabilities{}` respectively to advertise no capabilities.
- **Breaking-change impact:** Consumers relying on the implicit defaults must
  explicitly opt in to `logging` (servers) or `roots.listChanged` (clients)
  during v2 migration.
- **SDK-v2 action item:** Change default capabilities to empty; document
  required explicit opt-in.

### 1.8. [V2] `CreateMessageResult.Content` and `SamplingMessage.Content` should be `[]Content`

- **Summary:** The 2025-11-25 MCP spec permits `content` to be a single block
  or an array (the array form supports parallel tool calls and multiple
  `tool_result` blocks in a single user message). The SDK modeled `Content` as
  singular.
- **Current workaround:** `CreateMessageResultWithTools` (with
  `Content []Content`) was added as a workaround, matching the TypeScript
  SDK's approach.
- **Breaking-change impact:** All `CreateMessageResult` consumers re-type
  `Content` from `Content` to `[]Content`. Similarly for `SamplingMessage`.
- **SDK-v2 action item:** Change both types to use `[]Content` directly;
  retire `CreateMessageResultWithTools` workaround.

### 1.9. [V2] `CallToolParams` / `CallToolParamsRaw` redundancy

- **Summary:** `CallToolParams` and `CallToolParamsRaw` co-exist because of a
  perceived need to differentiate custom-Go-type unmarshalling from raw-map
  unmarshalling. The `mcp.AddTool` convenience wrapper still must first
  unmarshal into `map[string]any` for server-side required-field validation,
  so the split is functionally redundant.
- **Current workaround:** None; both types continue to be exported.
- **Breaking-change impact:** Consumers of `CallToolParamsRaw` migrate to a
  unified `CallToolParams` carrying a `map[string]any`.
- **SDK-v2 action item:** Collapse both into a single `CallToolParams` with
  `map[string]any` arguments.

## §4. Protocol invariants to preserve

Contract-level guarantees the Research Preview fork must not drift from
during opportunistic refactors. Each invariant carries a regression-risk
rating and a recommended protection mechanism.

### 4.1. [INVARIANT] Strict struct-field casing on JSON-RPC decode

- **Location:** `apps/mcp-go/internal/json/json.go:22`
- **Contract:** `NewDecoder` calls
  `dec.DontMatchCaseInsensitiveStructFields()` on every decoder it wraps.
  This disables the `segmentio/encoding` case-insensitive fallback so that
  only the exact case — matching MCP JSON-RPC 2.0 field names — decodes
  successfully.
- **Why it exists:** MCP JSON-RPC 2.0 §5 mandates strict field casing
  (`method`, `params`, `result`, `error`, `id`). A lenient decoder would
  silently accept `{"Method": ...}` and mask real interop bugs in third-party
  MCP clients/servers. Cross-SDK compatibility depends on this strict-match
  guarantee.
- **Regression risk:** **HIGH.** A single line change (dropping the
  `DontMatchCaseInsensitiveStructFields()` call) would silently soften
  protocol compliance with no observable test failure unless a targeted
  strict-casing test exists.
- **Recommended protection mechanism:**
  1. Add (or confirm existence of) a strict-casing unit test that asserts
     `{"Method": "initialize"}` fails to decode into a
     `JSONRPCRequest`-shaped struct (field missing, not populated by the
     case-insensitive fallback).
  2. Add a CODEOWNERS or lint-rule pattern flagging any edit to
     `internal/json/json.go` for heightened review.
  3. Cross-reference: FTR-006 state.yaml Decision D-15.

### 4.2. [INVARIANT] OAuth RFC 9110 case-INsensitivity (cross-reference hazard)

- **Location:** `apps/mcp-go/oauthex/resource_meta.go:100`, `:109`, `:112`,
  `:290`, `:302`; test case at `oauthex/resource_meta_test.go:137`.
- **Contract:** OAuth 2.0 / RFC 9110 requires that scheme identifiers and
  parameter keys be parsed **case-INsensitively**; parsed values are
  normalized to lower-case.
- **Why it exists:** RFC 9110 (HTTP Semantics) mandates this for
  authentication-scheme + parameter-key handling. OAuth 2.0 builds on 9110.
- **Regression risk: MEDIUM → HIGH (cross-contamination).** The direct
  regression risk (accidentally folding back to case-sensitive) is low —
  existing tests assert case-insensitive behaviour. The elevated risk is
  **cross-contamination with §4.1**: the two subsystems carry
  directly-opposite casing contracts (strict-sensitive JSON-RPC vs
  strict-insensitive OAuth). An engineer familiar with one may apply the
  wrong rule to the other during refactor or bug-fix work.
- **Recommended protection mechanism:**
  1. Cross-reference comment at both `internal/json/json.go:22` AND
     `oauthex/resource_meta.go:100` explicitly naming the other subsystem
     and stating "do not conflate."
  2. Preserve the case-insensitive test at
     `oauthex/resource_meta_test.go:137`; add a negative-assertion
     companion: a test that would fail if someone introduced case-sensitive
     parsing to `oauthex`.

### 4.3. [INVARIANT] Path-traversal containment on resource URIs

- **Location:** `apps/mcp-go/mcp/resource.go:88` — prose contract: *"It
  must not try to escape its directory"*.
- **Contract:** Resource URI handlers must not resolve paths that escape
  the configured resource root (classic directory-traversal guard).
- **Why it exists:** Security invariant. MCP server exposes filesystem-like
  resource trees to clients; allowing escape would leak arbitrary host
  filesystem content.
- **Regression risk: HIGH (security).**
- **Recommended protection mechanism:**
  1. Promote prose contract to enforced test — a positive assertion
     (valid paths resolve) plus a negative-set assertion (traversal
     attempts `..`, `/../`, URL-encoded `%2e%2e/`, Windows `\\..\\`,
     symlink-escape) all reject.
  2. Consider code-level enforcement via a dedicated `func
     assertWithinRoot(root, path string) error` helper; consolidate any
     bypass-prone manual prefix checks into that single helper.
  3. Apply archetype `feedback_invariants_as_code` pattern — prose
     contract converted to test + type + helper.

### 5.6. [DOC] Fork-parity gofmt policy — directive required

**Context:** §3.3 lists 24 unformatted files. These are upstream
fiberfx/mcp-go parity state — `gofmt` was never run at the upstream repo.
Running `gofmt -w` locally diverges from upstream and creates
merge-conflict debt on future sync.

**Three options (trade-offs):**

- **(A) Run `gofmt -w` locally + accept upstream-diff debt.**
  Pros: aligns with Go community norm, eliminates per-read noise, and
  hygiene tooling thereafter shows clean.
  Cons: every upstream sync produces a non-trivial rebase or merge
  conflict on these 24 files until upstream fixes their own CI.
- **(B) Preserve upstream parity + document as known drift.**
  Pros: upstream sync stays mechanical; no merge debt.
  Cons: `gofmt -l` will continue to report non-clean state; CI would need
  explicit allow-list for these 24 files.
- **(C) File upstream PR to enforce `gofmt` in fiberfx/mcp-go CI.**
  Pros: root-cause fix for both this fork and upstream; community benefit.
  Cons: dependent on upstream maintainer responsiveness; sync-burden
  continues until merged upstream.

**Recommendation:** **Option C → Option B as interim**. Option C is the
root-cause fix and the lowest-overhead end-state; Option B is the
lowest-churn interim while awaiting upstream. Option A creates
monotonically-growing merge debt and is not recommended.

**Cross-link:** §3.3 (file list); `apps/mcp-go/docs/contributing.md`
(potential landing spot for the documented policy).
