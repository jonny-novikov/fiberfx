# Codemojex · Valkey node: version 9, the allocator, one gigabyte

Codemojex keeps a small, hot slice of its state in Valkey and the durable rest in Postgres, so the Valkey node is sized not by how much it holds but by how steady its latency stays under a single command thread. This article settles that node: Valkey 9.x, the bundled jemalloc allocator, and a Fly `shared-cpu-2x` machine with one gigabyte of memory. It reads the real key usage in the `codemojex` app, derives why one core and one allocator decide the floor, maps what version 9 brings to this specific game, names a memory hazard already latent in the close path, and ships a tuned `valkey.conf`, `Dockerfile`, and `fly.toml` to match.

## Scope and method

The runtimes in frame are Valkey `9.1` (the official image, on Debian trixie or Alpine) over its bundled jemalloc, reached from the `codemojex` umbrella app whose pinned build is Elixir 1.18.4 on OTP 28; the deploy target is a Fly `shared-cpu-2x` machine, two shared vCPUs and one gigabyte, on a private 6PN.

What is decided here versus shown: the version, allocator, image, and machine size are decisions, argued from the workload and from cited platform behaviour. The performance statements are of two kinds and no third — external figures carry a numbered reference to the source that published them, and sizing statements about Codemojex are derivations from the committed source, marked as such. No benchmark was run for this article and none is quoted; there is no committed `.out`, so any order-of-magnitude figure about the game is reasoning, not measurement. The companion files are `valkey/conf/valkey.conf`, `valkey/Dockerfile`, and `valkey/fly.toml`. Out of frame: the Phoenix machine sizing, the connector internals, and the EchoMQ stream protocol, each its own document.

## What the node holds

The live state is small because the money is not in Valkey. Player balances — keys, clips, diamonds — live in Postgres behind a row lock and a non-negative check; Valkey holds the per-game surface and the in-flight queue, both of which die with the round or the job.

Read the game keys in `board.ex`. A scored guess updates a player's running base score in the hash `cm:<game>:base`, accrues first-mover tier bonuses in `cm:<game>:bonus`, claims each tier's first crack atomically with `HSETNX` against `cm:<game>:tierfirst`, and writes the effective score into the sorted set `cm:<game>:board`. The score worker bumps `cm:<game>:attempts`. Position locks sit in a per-player hash `cm:<game>:lock:<player>`. One global counter, `cm:total_won`, persists across rounds. A round in flight is a sorted set, four hashes, and a counter — a few kilobytes. Two hundred concurrent rounds with fifty players each is single-digit megabytes.

The hot path is short and pipelined. The four background consumers — `cm_score`, `cm_settle`, `cm_notify`, `cm_commands` — open the connector at port `6390` with `protocol: 3`, so reads, writes, and the out-of-band push for cache invalidation share one RESP3 connection per consumer rather than a wide pool. Scoring one guess touches the half-dozen to a dozen hash and sorted-set operations listed above; the queue rides Redis streams, where `EchoMQ.Stream.trim/4` bounds each stream with `XTRIM` over a keep-newest window or a Snowflake-derived minimum id. The shape that matters for sizing: many small operations on small values, issued in pipelined bursts over a few long-lived connections.

## Why the allocator is the storage engine

On an in-memory store the memory allocator is part of the storage engine, and Valkey's allocation pattern is hard on a general-purpose one. Every sorted-set node, every hash field, every stream entry, every key name is a small allocation, and they churn continuously as rounds open and close and streams trim. A general allocator fragments under that pattern — freed blocks of one size cannot satisfy a request of another, freed memory is slow to return to the kernel, and resident memory drifts well above the live data. For a process whose resident size is the bill and the trigger for an out-of-memory kill, that drift is the whole problem.

Valkey's answer is to ship jemalloc and make it the default on Linux precisely because it has shown fewer fragmentation problems than the C library allocator [7]. jemalloc rounds allocations into size classes and packs same-size objects together, so a freed slot is reused by the next allocation of that size, and its decay logic returns unused pages to the kernel so resident memory tracks live data. Three ties to the conf beside this article. Active defragmentation — relocating live objects out of sparse runs so empty pages can be purged — works only against jemalloc, so `activedefrag yes` is meaningful only here. The memory accounting in `INFO`, the `mem_fragmentation_ratio` the deploy will watch, is jemalloc's introspection. And the version is pinned: the official container builds the bundled jemalloc from the dependency tree, patching its build to match the platform's page size [9], which is why the benchmarking bench targets that same allocator rather than a system one. On the one-gigabyte box this is what makes the 512-megabyte working-set guardrail hold in practice rather than in theory.

## One core runs the commands

Command execution in Valkey is single-threaded. The benchmark documentation states it plainly: the server runs commands sequentially, and for workloads that need multiple cores the recommended path is multiple instances, not a bigger one [10]. The latency documentation says the same from the other side — only one command executes at any instant, and a slow command makes every other client wait [11]. The I/O threading added in Valkey 8 does not change this: those threads read and parse requests, write replies, poll sockets, and free memory, so the main thread spends more of its time executing, but the execution itself stays on the one thread [13]. A proposal to run read commands in parallel across slot dictionaries was floated for the 9.0 cycle, and its own framing concedes that offloading only I/O gives limited gain when processing is the constraint [14]; it did not ship as a 9.0 or 9.1 headline.

Two consequences set the machine. First, moving from one vCPU to two earns its keep through the background work, not the command rate. Forking for an AOF rewrite runs in the main thread and can stall it [11]; the `everysec` AOF fsync runs on a separate thread [11]; jemalloc's purge and the lazy frees in the conf are background threads. A second vCPU lets those run alongside command execution instead of contending head-to-head, which is what trims the latency tail on a single-threaded server. Second, moving from two vCPUs to four buys nothing at this size, because the command thread is the ceiling; the way to spend more cores is horizontal — a cluster sharded by hash tag over the branded Snowflake ids — not a wider single node.

Version 9 does not move this line. Its throughput work is memory prefetching for pipelined commands, reported up to forty percent higher throughput, plus zero-copy replies and SIMD for a few commands [3]; the prefetch raises the single thread's ceiling by hiding the memory-fetch stall, it does not add a second execution thread. As an external bound, public benchmarks put a single Valkey instance's simple-operation throughput in the high hundreds of thousands to low millions per second on server hardware [13][3] — orders of magnitude above Codemojex's guess rate, where each scored guess is a short pipelined burst of sub-microsecond operations [11]. The node will be bound by its latency tail and its memory guardrail long before its command rate.

## What 9 brings this game

Set against that floor, the 9.x line carries five things that touch Codemojex directly, and one that does not yet.

**Hash-field expiration** is the headline of 9.0 [2] and the closest fit. The `HEXPIRE` family gives each field inside a hash its own time to live, reclaimed by an active expiry scan rather than a per-read check, so the hot hash commands keep their constant time [4]. Valkey's maintainer frames the feature with a lobby — a top-level key for a game, a field per player, each field expiring on its own clock, the lapsed ones dropped [5]. That is Codemojex's presence and idle-timeout shape. It is also the clean tool for the hazard in the next section.

**`DELIFEQ`** is a native guarded delete: release a key only if it still equals the token you wrote, an operation that previously needed a Lua script [6]. Candidly, it does not replace anything in the close path today — the exactly-once close in `rooms.ex` is a `SET cm:<game>:closed "1" NX`, a tombstone that is never released, and the position locks in `locks.ex` clear a single field with `HDEL`, not a whole keyed lease. `DELIFEQ` is the primitive to reach for when a leased lock is released by token, keeping that release off the scripting path that serialises on the one command thread. It earns its place as the lock surface grows, not before.

**Pipeline memory prefetching** is the 9.0 change that amplifies the dial Codemojex already turns. The connector's lever is pipeline depth, not pool width; the up-to-forty-percent figure is a single-thread gain from prefetching the keys a pipelined batch will touch [3], which is exactly how the scoring burst is issued.

The **9.1** point release adds two fixes on paths this code uses and one operational metric. It reduces latency spikes during hash-table rehashing by releasing pages incrementally [1] — a tail-latency win on the single thread, where the keyspace and the round hashes rehash as they grow. It fixes a null-pointer crash in stream-trim delta calculation [1], directly on the `XTRIM` path that `EchoMQ.Stream.trim/4` drives. And it adds `used_active_time_main_thread` to `INFO`, a truer headroom signal than CPU percentage, because with I/O threads the server busy-waits and pins CPU near full even with spare capacity [12] — the metric to alert on for this node.

The one that does not apply yet is **atomic slot migration**: 9.0 moves whole cluster slots in one snapshot-based handoff with no key-by-key window [11]. It matters only past the single node, but it is precisely the safety the horizontal path above depends on, so it future-proofs the one way Codemojex will add cores.

The cost side: `9.1.0` is the first stable release on the 9.1 line and is marked low upgrade urgency [1]. For a money game upgrading a live dataset, that caution argues for the more patch-mature 9.0 line; for a fresh deploy with nothing to migrate, the caution is small and the 9.1 fixes are worth pinning. Either way the three security fixes in `9.1.0`, including a use-after-free during a full sync over a yielding script [1], belong in any money build.

## A memory hazard already in the close path

The close path leaks Valkey keys, and on a one-gigabyte node under `noeviction` that is the failure that will arrive first. Read `do_close` in `rooms.ex`: it computes the effective pool, pays the winners, bumps `cm:total_won`, marks the round closed in Postgres, and resets the room. It never deletes or expires the round's Valkey keys. So every closed round leaves its sorted set, its four hashes, its attempts counter, the `:closed` tombstone, and a lock hash per player resident with no time to live, and they accumulate without bound. With eviction off — the right policy for a money game, since silent dropping of game state is worse than a loud write rejection — a long-running node fills toward the guardrail and then begins refusing writes.

The fix is small and does not need version 9: set a time to live on the round keys at close, or delete them outright once the settle has paid out and the leaderboard has been read for the last time. Where finer control helps — a presence hash whose player fields should lapse on their own clocks while the hash lives — `HEXPIRE` is the 9.x tool for it. The point is that the guardrail and the leak interact: the 512-megabyte cap is far above the real working set and will rarely bind, unless this leak is left in, in which case the cap is the only thing standing between the node and an out-of-memory kill.

## The image and the machine

Use the official `valkey/valkey:9.1` image rather than a from-source build for the deploy. It is built with the bundled jemalloc [9], it carries the current security patches, and its tags cover Debian trixie and Alpine [8]. Two operational reasons reinforce it. Its entrypoint runs as root, chowns the mounted volume, and drops to the `valkey` user — which a hand-rolled image that sets a non-root user cannot do against a freshly mounted Fly volume, since the volume mounts root-owned. And take the base image, not the bundle variant, which preloads JSON, Bloom, Search, and LDAP modules this game does not use [8]. One caveat the image creates: it turns protected mode off by default for container networking [8], which is why the conf re-asserts `protected-mode yes` and the password is injected at boot.

The machine is `shared-cpu-2x` with one gigabyte. One gigabyte is a valid size only on a shared machine: Fly's minimum memory is 256 megabytes per shared vCPU but 2048 per performance vCPU [15], so a one-gigabyte node cannot be a performance machine at all. The shared model is a burstable quota — each shared vCPU has a baseline near six percent of a period and accrues a balance to burst to full, and a two-vCPU machine shares one combined quota across both vCPUs [16]. That is why the conf keeps `io-threads 1`: extra I/O threads busy-wait, pinning CPU near full [12] and draining a shared machine's burst balance even while idle, whereas the background fsync and purge threads sleep and are the right tenants for the second vCPU. On a performance machine you would flip I/O threads up; on this one you do not. The size, along with a SIGTERM kill signal and a window long enough for the AOF to flush before a forced exit, are set in the machine's `fly.toml` [17].

Networking is private by construction. The `fly.toml` carries no `[http_service]` and no `[[services]]`, so Fly Proxy assigns no public address; binding the conf to the IPv6 wildcard makes the server reachable over the org's 6PN with no service definition and no provisioned IP [19]. Private networking over the 6PN is on by default for apps in an organization [18], so the Phoenix app reaches this node at `codemojex-valkey.internal:6390` and nothing outside the organization can. The volume is sized well above the AOF working set and lives on one host in one region — not network storage, one volume to one machine [20] — which is the single point of failure to retire with a replica before the game carries real money.

## Boundaries

This article measures nothing; its figures about Valkey throughput and the forty-percent and security details are cited to their publishers, and its figures about Codemojex's footprint and operation count are derivations from the committed source, not benchmarks. The allocator reasoning is specific to the Linux build with bundled jemalloc; a macOS source build defaults to the C library allocator and will not reproduce the resident-memory behaviour. The sizing assumes the live dataset stays small because balances live in Postgres; if that moves into Valkey, the cap and the policy both need revisiting. The machine is a single node in a single region with a single volume — adequate for launch, not for the availability a money game eventually needs, where the next steps are a replica in the primary region, then splitting the Phoenix web and worker process groups, then a cluster shard by hash tag when a sustained guess rate finally pressures the one command thread.
