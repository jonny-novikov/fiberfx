`docs/elixir/toolkit/` is a **self-contained, portable authoring kit** for the /elixir course — meant to be zipped and uploaded into a fresh Claude.ai session to continue authoring (see its `CONTINUE-PROMPT.md` / `SKILL.md`: "Work in `/home/claude/elixir-course/`"). It is its OWN Python build system (`build_page.py`, stdlib-only, **no Go / no GOWORK** — distinct from the [[jonnify-cms-toolchain]] Go gates), with `content/` body fragments, a `_head.html`, and the 9 Apollo gates. It is committed to the repo as a backup/reference, but the authoring historically happened in a separate sandbox.

**It drifts from the live course and must be re-synced.** The kit was frozen ~F4.05 (May 2026) while the live `elixir/**/*.html` completed F0–F6. The **live pages are the source of truth**; the kit's `build_page.py` manifest had genuinely diverged (not just lagged): F5 planned a phantom `mix/releases/telemetry` syllabus but the BUILT F5 is `foundations→domain→tracer-bullets→contracts→cqrs→state→testing→boundaries→engine-lab`; F4 grew from 9 to **12 modules**; slugs renamed (`index`→`indexing`). module_count() is now **57**, not the SKILL's stale "54".

**The sync (done 2026-06-02, committed by user as `elixir: partials`, 204 fragments + manifest):**
- A **fragment = inverse of `_assemble()`**: `page = _head.html + "\n<body>\n" + FRAGMENT + "\n" + BOOTSTRAP + "\n</body>\n</html>\n"`. Extract = strip head up to first `<body>`, cut from the `<script>` opening the marker `/* progressive enhancement: mark JS on, then reveal-on-scroll */`, swap the minted stamp `id="stampId"`/`id="st-ts"` back to `{{BUILD_ID}}`/`{{BUILD_TS}}`. Fragments carry NO `<style>` (CSS is in `_head.html`). Only `/elixir` + `/elixir/course` also embed `{{CONTENTS}}/{{MODULE_COUNT}}/{{CHAPTERS_JSON}}` (irreversible → chrome-refresh those 2 in place).
- **`elixir/<chapter>/llms.txt` is the authoritative numbering source** (`F5.01`, `F5.01.1`, `F5.0.1` front-matter). But numbering is inconsistent across chapters (F3 dives all reuse the parent num `F3.04`), so derive dive numbers from the **parent module's num + sibling order**, not the dive's own tag.
- **Manifest gate = `build_page.py routes` must equal the live route set exactly** (computed by globbing `elixir/**/*.html` → routes). Dives are linkable ONLY via `SUBPAGES[module_n]`; front-matter via `CHAPTER_SUBPAGES[cid]`; everything in `PAGES`. This objective gate is what makes a 204-entry regeneration safe.
- Scripts (deterministic, dry-run+--apply): `/tmp/elixir_toolkit_sync.py` (extractor+plan JSON), `/tmp/sync_specials.py`.

**`_head.html` / `HEAD_CSS` alignment — DONE (2026-06-02, committed `7591471 "elixir partials"`).** Live pages bake the design system (incl. `.hero-lede` + `.foot-nav` family) into each per-page `<head>` `<style>`, stripped during extraction; the kit's shared `_head.html` lacked them (0 `hero-lede`/0 `foot-nav`). Fixed by appending one accent-neutral block to `HEAD_CSS` in `build_page.py`, then `python3 build_page.py extract-head` to regenerate `_head.html` (the builder reads `_head.html`, NOT live HEAD_CSS — edit HEAD_CSS then extract-head, or changes won't appear). Key subtlety: the two-column hero (`.hero{grid}`) must NOT be global — it would scramble dive/landing heroes whose children have no `grid-area`. Live scopes it per-page; the shared head scopes it with **`.hero:has(.hero-copy)`** (grid fires only on hub fragments). `.foot-nav` needs no scoping (all 204 carry it).
